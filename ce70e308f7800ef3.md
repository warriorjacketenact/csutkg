# LinkVault 聚合资源索引系统

LinkVault 是一个面向技术社区与内容创作者的轻量级外链资源聚合与导航平台。该项目定位于帮助开发者、研究员与技术写作人员高效组织、分类、检索与共享大量分散的外部链接资源，尤其适用于处理批次化、大规模、多来源的 URL 数据整理任务。LinkVault 本身不存储任何第三方内容，仅提供结构化索引与元数据描述框架，使资源管理从杂乱无章的收藏夹转变为可维护、可扩展、可协作的知识库。

本项目针对的核心痛点包括：浏览器书签无法承载复杂分类与备注信息；手动维护 Markdown 列表缺乏统一的字段约束与校验；多人协作时链接重复、失效、分类冲突难以追踪；以及缺乏从原始 URL 文本到结构化 YAML/JSON 数据集的自动化转换管线。LinkVault 通过约定优于配置的目录体系、基于正则表达式的智能分类器以及可插拔的链接状态检查中间件，为上述问题提供开箱即用的解决方案。

## 功能概览

**批量导入与去重**：支持从纯文本、CSV 及常见书签导出格式（HTML/Netscape）批量导入 URL，自动检测并合并重复条目，保留首次出现的分类标签与备注信息。

**智能分类引擎**：基于 URL 路径关键词、域名特征与可自定义的正则规则库，将链接自动归类至预设的 12 个顶级目录（如文档、教程、工具、API、博客、视频、代码仓库、学术论文、行业资讯、社区讨论、招聘、其他）。

**元数据增强管线**：通过可配置的抓取器尝试从目标页面提取标题、描述、最后修改时间与主要语言，生成统一的元数据摘要，并支持人工覆写与补充标注。

**链接健康度巡检**：内置定时任务与手动触发接口，对已收录链接进行 HTTP 状态码检查，标记失效、重定向及超时节点，生成健康度报告并支持导出异常列表。

**多格式数据导出**：支持将整个索引库导出为 Markdown 表格、JSON Schema、YAML 清单或 HTML 静态导航页，适配不同发布场景（如 README 嵌入、静态站点生成、数据交换）。

**标签与全文检索**：基于标题、描述、自定义标签及路径片段构建轻量级倒排索引，提供命令行与 Web 简易界面两种检索入口，支持模糊匹配与布尔组合查询。

**版本化变更追踪**：每次增删改操作均记录操作日志与快照差异，支持回滚至任意历史版本，便于团队协作审计与误操作恢复。

**权限分级管理**：支持只读访客、编辑者与管理员三级角色，管理员可配置字段可见性与导出策略，适合内部知识库与对外公开发布两种模式切换。

## 应用场景

**技术团队内部文档中心**：研发团队可将 LinkVault 部署为内部技术周报、项目复盘文档、新人 onboarding 指南的底层链接底座。每当成员分享一篇有价值的博文、工具站或论文，只需提交 URL 与简短摘要，系统自动完成分类与去重，文档编写者通过 API 拉取指定标签下的最新链接列表，动态生成周报或技术雷达页面。

**开源项目外部资源附录**：开源项目维护者通常需要在 README 中列出大量相关依赖、学习资料、社区论坛与扩展项目链接。随着链接数量增长，手动维护极易出错。LinkVault 提供从原始 URL 批次到规范化 Markdown 资源列表的完整流水线，可针对不同分支（如 stable 与 dev）维护独立的资源快照，并在 CI 流程中自动检查所有外链可达性，确保发布版本不含死链。

**个人知识管理（PKM）工作流增强**：对于频繁使用 Obsidian、Notion 或 Logseq 进行知识管理的用户，LinkVault 可作为外部链接的中转缓存层。用户将碎片化阅读中积累的 URL 统一投递至 LinkVault 收件箱，系统完成初步清洗与分类后，用户定期通过导出功能将已处理的链接清单合并入个人笔记库，并附带系统生成的元数据摘要，极大降低手工整理成本。

**内容聚合与 Newsletter 生产**：独立博主或小型内容团队可利用 LinkVault 管理每周要闻的素材池。运营人员将候选链接录入系统，编辑审核时直接查看每个链接的自动摘要与健康状态，择优采用并一键生成带有分类导航的 Markdown 草稿，后续渲染为邮件或网页即可发布。

## 快速开始

以下命令演示在 Linux/macOS 环境下从源码部署 LinkVault 核心服务。Windows 用户建议使用 WSL2 或 Docker 环境。

```bash
git clone https://github.com/linkvault/linkvault-core.git
cd linkvault-core

# 创建并激活 Python 虚拟环境
python3 -m venv venv
source venv/bin/activate

# 安装核心依赖与开发工具
pip install --upgrade pip
pip install -r requirements.txt
pip install -r requirements-dev.txt

# 初始化配置文件与本地数据库
cp .env.example .env
python scripts/init_db.py --force

# 运行内置的示例导入任务（使用项目自带样例数据）
python scripts/batch_import.py --input samples/url_list.txt --category demo

# 启动开发服务器（默认监听 127.0.0.1:8000）
python app.py
```

访问 `http://127.0.0.1:8000` 可进入 Web 仪表盘。首次启动会自动创建管理员账户，用户名 `admin`，初始密码打印在终端日志中，请及时修改。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.10 及以上 | 核心运行环境，低于 3.10 将无法使用 match 语句与类型注解新特性 |
| SQLite | 3.35.0 及以上 | 内置数据库，用于存储链接元数据、分类树与操作日志，生产环境建议迁移至 PostgreSQL |
| Redis | 6.2 及以上 | 可选，用于缓存健康检查结果与加速检索查询，若无 Redis 则降级为内存缓存 |
| Node.js | 18.17.0 及以上 | 仅用于前端资产构建，若只使用命令行接口可跳过此依赖 |
| Nginx | 1.22 及以上 | 生产部署推荐反向代理，用于静态文件服务与负载均衡，非强制 |
| Git | 2.30 及以上 | 用于版本化变更追踪中的差异对比功能，若无 Git 则使用内置 diff 算法降级 |
| curl | 7.68 及以上 | 用于链接健康度检查中的 HTTP 探测，系统将自动检测并回退至 Python 的 urllib |
| cron | 系统自带 | 用于定时巡检任务调度，非必需但建议配置 |

## 贡献指南

1. 阅读开发者文档与代码风格规范，确保本地开发环境已安装所有依赖并成功通过测试套件（`pytest tests/`）。所有新增功能必须附带对应的单元测试与集成测试，测试覆盖率不低于 85%。

2. 从 GitHub 仓库派生（fork）主分支，创建以 `feature/` 或 `fix/` 为前缀的命名分支。提交信息遵循 Conventional Commits 格式（`type(scope): subject`），例如 `feat(classifier): add Bayesian fallback for unknown domains`。

3. 编写或修改代码时，保持与项目现有抽象层次一致。若引入新的第三方库，需在 PR 说明中给出充分理由，并确认该库的许可证与项目兼容（MIT、BSD、Apache 2.0 优先）。

4. 提交 Pull Request 前，执行 `pre-commit run --all-files` 进行代码格式化（black、isort）与静态检查（flake8、mypy）。PR 描述中需明确阐述解决的问题、实现方案以及可能的性能影响或兼容性风险。

5. 项目维护者将在 5 个工作日内评审 PR，可能要求补充测试用例或调整接口设计。合并后，贡献者名称将自动记录在 `CONTRIBUTORS.md` 文件中，并欢迎加入核心开发者邮件列表。

## 常见问题

**Q：LinkVault 是否存储目标页面的完整内容或快照？**

A：默认情况下，LinkVault 仅存储从目标页面 `<title>` 与 `<meta name="description">` 中提取的摘要信息，不保存完整 HTML 或资源文件。用户可在配置中开启全文抓取功能，但该选项需明确启用且仅限私有部署环境使用，以规避版权与存储膨胀风险。

**Q：如何迁移已有的大量书签数据？**

A：LinkVault 提供三种迁移路径：其一，从 Chrome/Firefox 导出的 HTML 书签文件直接导入（保留文件夹层级映射为分类）；其二，通过 CSV 模板批量导入，支持自定义字段映射；其三，调用 REST API 的 `/batch` 端点以 JSON 数组形式提交。迁移后系统自动生成去重报告，用户可在仪表盘核对并手动合并重复条目。

**Q：健康度检查对目标服务器会造成压力吗？**

A：巡检模块默认采用单线程顺序请求，间隔为 500 毫秒，并遵守 `robots.txt` 中的爬取延迟指令。对于大规模索引库（超过 5000 条链接），建议配置 cron 在低峰时段（如凌晨 2:00-5:00）执行，并可设置最大并发数（默认 5）以避免误判为目标服务器攻击。所有检查请求的 User-Agent 均标识为 `LinkVault-HealthCheck/2.0`，便于站点管理员识别。
