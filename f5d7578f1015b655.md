# ResourceForge

ResourceForge 是一个面向开发者与技术研究人员的开源外链资源归集与导航系统。该项目不生产内容，而是以结构化方式对互联网上的高质量技术文档、工具站点、交互演示与代码示例进行整理和索引，帮助用户在海量信息中快速定位有价值的参考材料。项目本身采用静态站点生成方式，所有资源条目均以可编辑的标记文件存储，便于社区贡献和版本追踪。

ResourceForge 主要面向两类使用者：一类是日常需要查阅大量技术文档和代码示例的软件工程师、架构师与运维人员，另一类是希望建立个人或团队知识库的技术管理者。项目通过统一的条目模板、标签体系和引用关系，将零散的浏览器书签转化为可检索、可协作、可审计的资产目录。每一批资源收录均经过格式校验和去重处理，确保目录的整洁与可用性。

## 功能概览

**结构化资源目录** 所有外链按类别、主题和适用场景进行多级分类，支持快速筛选与排序。

**标签与全文检索** 每条资源可附加多个标签，配合标题与摘要的文本索引，实现毫秒级查找。

**条目模板校验** 新增资源需遵循预定义的 YAML 前置声明格式，包含标题、链接、描述、标签和收录批次字段。

**版本化变更记录** 每次增删改操作均生成变更日志，便于回溯资源目录的演进历史。

**批量导入与去重** 支持从 CSV 或纯文本列表批量导入 URL，自动检测并剔除重复条目。

**静态站点导出** 项目可构建为完全静态的 HTML 站点，无需数据库或后端服务即可部署到任意 Web 服务器。

## 应用场景

**技术团队内部知识库建设** 团队技术负责人可将 ResourceForge 作为团队文档门户的补充，集中存放日常开发中遇到的优质外链，包括第三方 API 文档、开源库主页、调试工具和性能分析平台。新成员入职时可通过该目录快速了解团队常用技术栈的参考来源。

**个人开发者的研究笔记整理** 独立开发者或研究员在研究某一技术领域（如分布式系统、前端框架、数据库内核）时，可使用 ResourceForge 记录阅读过的论文链接、实验环境地址和代码仓库。配合标签系统，可以按研究阶段或难度等级组织学习路径。

**技术社区资源共建** 开源社区或技术兴趣小组可以利用 ResourceForge 的贡献流程，由多位成员共同维护一份主题资源清单，例如"Rust 异步运行时生态""Kubernetes 可观测性工具集"等。项目内置的变更记录功能可清晰展示每次更新的提交者与时间。

**技术文档站点的外部引用管理** 当企业文档站点或项目 README 需要引用大量外部链接时，可将所有外链交由 ResourceForge 统一管理，主文档仅引用内部资源编号，降低链接失效或域名变更带来的维护成本。

## 快速开始

以下命令序列演示从克隆代码仓库到运行本地开发服务的完整流程。

```bash
git clone https://github.com/resourceforge/resourceforge.git
cd resourceforge
pip install -r requirements.txt
python manage.py build --watch
```

执行上述命令后，开发服务器默认启动于本地 8000 端口。打开浏览器访问 http://localhost:8000 即可查看资源目录首页。若需导入用户提供的原始 URL 列表，可将链接保存为 plaintext 文件，然后执行导入命令：

```bash
python manage.py import --batch 876 --source ./urls.txt
```

## 项目结构

项目采用模块化组织方式，核心代码与资源数据分离，便于独立更新和扩展。以下为项目根目录下的主要目录与文件结构。

```
resourceforge/
├── core/                          # 核心引擎模块
│   ├── __init__.py                # 包初始化，导出主要接口
│   ├── loader.py                  # 资源加载器，解析 YAML 与 Markdown
│   ├── indexer.py                 # 索引构建器，生成倒排索引
│   └── validator.py               # 条目格式校验器
├── cli/                           # 命令行接口模块
│   ├── __init__.py
│   ├── main.py                    # 入口命令分发器
│   ├── build.py                   # 构建静态站点命令
│   └── import.py                  # 批量导入命令
├── templates/                     # Jinja2 页面模板
│   ├── base.html                  # 基础骨架模板
│   ├── index.html                 # 资源目录首页
│   └── detail.html                # 单条资源详情页
├── static/                        # 静态资源输出目录（构建后生成）
│   ├── css/                       # 样式表文件
│   ├── js/                        # 前端交互脚本
│   └── assets/                    # 图片等媒体文件
├── data/                          # 资源数据存储目录
│   ├── batches/                   # 按批次存放的原始条目文件
│   │   └── 876.yml                # 当前批次的 YAML 条目
│   ├── tags/                      # 标签索引文件
│   └── changelog/                 # 变更日志记录
├── tests/                         # 单元测试与集成测试
│   ├── test_loader.py
│   ├── test_validator.py
│   └── fixtures/                  # 测试用固定数据集
├── docs/                          # 项目文档源文件
│   ├── user/                      # 用户手册
│   ├── contributing/              # 贡献指南
│   └── ops/                       # 运维参考
├── requirements.txt               # 生产环境依赖列表
├── requirements-dev.txt           # 开发环境额外依赖
├── pyproject.toml                 # 项目元数据与构建配置
└── README.md                      # 项目说明文档（本文件）
```

## 贡献指南

ResourceForge 欢迎社区贡献，包括但不限于新增资源条目、修复解析逻辑、优化模板样式和完善文档。请按照以下步骤提交贡献。

第一，在 GitHub 上 Fork 本仓库，并克隆到本地开发环境。创建新的功能分支，分支名称应简要描述变更内容，例如 `feature/add-batch-877` 或 `fix/validator-encoding-issue`。

第二，新增资源条目时，请将链接整理为 YAML 格式并放置于 `data/batches/` 目录下。每条条目必须包含 title、url、description、tags 和 batch 字段。提交前运行 `python manage.py validate` 确保格式合规。

第三，若修改核心解析或索引逻辑，需同步更新对应的单元测试文件，确保测试覆盖率达到 90% 以上。运行 `pytest tests/` 验证所有测试用例通过。

第四，提交 Pull Request 前，请确保代码已使用 black 格式化，并更新相关文档章节以反映变更。PR 描述中应清晰说明修改目的、影响范围以及测试结果。

## 常见问题

**问：导入大量 URL 时出现重复条目如何处理？**

导入命令默认启用去重机制，基于 URL 的规范化字符串进行精确匹配。若发现重复，系统会在日志中输出警告并跳过该条目，同时记录重复次数。用户可在导入后查看 `data/changelog/duplicates.log` 文件获取详细列表。对于近似重复（如协议不同或尾部斜杠差异），目前需手动检查并合并。

**问：静态站点生成后，部分链接无法访问怎么办？**

ResourceForge 本身不负责验证外部链接的可达性。若生成站点中的链接失效，建议在资源条目中更新 url 字段或标注状态。项目提供了 `check --dead-links` 命令，可对已收录链接进行 HTTP 状态探测，输出失效链接报告供维护者参考。该功能仅做探测，不自动删除条目。

**问：如何将已有浏览器书签批量导入项目？**

项目未直接集成浏览器书签解析器，但提供了通用导入接口。用户可从浏览器导出书签为 HTML 文件，然后使用第三方工具（如 `bookmarks-parser`）将其转换为 CSV 格式，再通过 `python manage.py import --format csv --source bookmarks.csv` 完成导入。若书签数量较大，建议分批导入并利用批次编号进行管理。
