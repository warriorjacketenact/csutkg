# LinkVault Core

LinkVault Core 是一个面向技术团队与内容运营者的外链资源归集与分发系统。项目定位于将散落在各类内容平台、文档站点与内部知识库中的外部链接进行统一抓取、分类存储、状态监控与结构化输出，从而解决技术资源链接易失效、难检索、无法批量管理的问题。LinkVault Core 并非一个简单的书签管理器，而是一个具备元数据提取、链接健康度检查、访问频次统计和只读 API 输出能力的轻量级后端服务。该项目的目标用户包括开源社区文档维护者、技术内容聚合站点运营方、企业内部开发者门户管理员，以及需要从海量外链中构建稳定知识索引的自动化流程开发者。

## 功能概览

- 批量链接抓取与元数据提取：支持从给定入口页面批量抓取链接，自动提取页面标题、元描述、内容类型标签及最后修改时间，形成结构化记录存入本地 SQLite 数据库。

- 链接健康状态定时巡检：内置基于 HTTP HEAD 请求的链接可用性检查器，可配置检查间隔与超时阈值，自动标记 4xx、5xx 及超时链接，并生成异常报告。

- 分类标签与多级目录映射：允许用户为每个链接自定义命名空间与标签组合，支持按项目、领域、文档层级或批次进行多维筛选，便于在大型资源库中快速定位。

- 只读 API 与静态站点生成：提供 RESTful 风格的只读查询接口，支持按标签、批次号、域名过滤输出；同时支持将链接列表渲染为静态 Markdown 表格或 JSON 清单，方便集成到 CI/CD 流程或文档站点中。

- 导入导出与批次管理：支持 CSV 与 JSON 格式的链接批量导入，内置批次版本管理功能，每个导入批次自动生成唯一批次号（如 653/1241），允许回滚或对比不同批次间的链接变更差异。

- 访问热度分析：记录每个外部链接的被访问次数与最后访问时间，生成简单热度排行，辅助判断哪些资源在团队内具有更高参考价值。

## 应用场景

1. 开源项目文档站外链维护：开源项目维护者可以使用 LinkVault Core 管理 README、Wiki 或官网中引用的所有外部参考链接。定时巡检功能可提前发现失效的参考文档或已迁移的博客文章，避免用户点击后遇到 404 页面，提升文档质量与用户体验。

2. 技术博客聚合站点的资源索引构建：面向特定技术领域（如云原生、前端工程化、数据库内核）的垂直内容聚合站点，可使用该项目将分散在多个视频教程、代码示例仓库和技术专栏中的外链按主题归类，并通过 API 动态渲染到站点侧边栏或资源导航页中。

3. 企业开发者门户的知识资产沉淀：企业内部的开发者门户通常汇集大量内部工具文档、部署指南和外部依赖说明。LinkVault Core 可作为后台链接治理工具，帮助平台工程团队统一管理这些外链的权限描述、维护责任人和更新周期，降低知识碎片化带来的协作成本。

4. 自动化测试用例中的链路依赖监控：在端到端测试或 API 集成测试场景中，部分测试用例依赖外部 Mock 数据或 Schema 定义文件的临时地址。通过将这类外部依赖链接纳入 LinkVault Core 管理，测试流水线可在执行前预先检查所有依赖链接的可访问性，从而减少因外部资源不可用导致的测试失败误报。

## 快速开始

以下操作适用于 Linux/macOS 环境，Windows 用户建议使用 WSL2 或 Git Bash。

```bash
# 克隆代码仓库
git clone https://github.com/your-org/linkvault-core.git
cd linkvault-core

# 安装项目依赖（使用 Python 3.9+ 与 pip）
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 初始化数据库并运行内置示例导入（批次 653/1241）
python manage.py init-db
python manage.py import-batch --source ./samples/links_653.csv --batch 653/1241
python manage.py health-check --batch 653/1241
python manage.py render --batch 653/1241 --output ./output/links_653.md
```

执行上述命令后，系统会在 `./output/` 目录下生成当前批次的链接状态报表，同时启动本地 API 服务（默认监听 8000 端口）。访问 `http://127.0.0.1:8000/api/v1/links?batch=653/1241` 即可获得 JSON 格式的链接数据。

| 依赖组件 | 必需版本 | 说明 |

| Python | 3.9 或更高 | 核心运行环境，建议使用 3.11+ 以获得更好的性能 |
| SQLite | 3.35 或更高 | 内置数据库，用于存储链接元数据与巡检记录，无需额外安装 |
| requests | 2.28.0 或更高 | 用于发送 HTTP HEAD/GET 请求进行链接健康检查与元数据抓取 |

| pytest | 7.2.0 或更高 | 仅开发测试时需要，用于执行单元测试与集成测试用例 |
| click | 8.1.0 或更高 | 提供命令行交互界面，用于执行管理命令与批量操作 |

## 贡献指南

1. 首先在 GitHub 上 Fork 本仓库，并基于 main 分支创建以 feature/ 或 fix/ 为前缀的开发分支。对于较大功能改动，建议先在 Issues 中创建讨论帖，说明设计方案与预期影响范围，避免重复劳动。

2. 本地开发时请确保通过所有现有单元测试：执行 `pytest tests/` 应全部通过。新增功能必须附带对应的单元测试用例，覆盖率不低于 80%。对于涉及外部 HTTP 请求的模块，需使用 pytest-mock 或 responses 库模拟网络交互。

3. 提交代码前运行代码风格检查与静态分析工具：项目使用 black 作为格式化标准，isort 管理导入顺序，flake8 检查语法风格。执行 `make lint` 或对应命令确保无错误输出。

4. 所有与链接处理相关的核心逻辑变更，需在 docs/operations.md 中同步更新命令行参数说明或配置项描述。新增 API 字段或修改返回结构时，必须同步更新 docs/api-reference.md 中的示例请求与响应。

5. 提交 Pull Request 时，请清晰描述改动内容、测试结果以及是否影响已有批次数据的兼容性。合并前需要至少一位项目维护者审阅通过。对于修复链接巡检逻辑或元数据解析缺陷的 PR，建议附带可复现的失败案例链接。

## 常见问题

**Q: 导入包含大量链接的 CSV 文件时，提示部分行解析失败，应该如何处理？**

A: 检查 CSV 文件是否包含 BOM 头（尤其是 Windows 下保存的文件）。项目默认期望 UTF-8 without BOM 编码。同时确认每一行是否符合 `url,title,tags` 的列顺序。导入失败的行会被记录到 `logs/import_errors.log` 中，建议先修正这些行后重新导入，或使用 `--skip-errors` 参数跳过异常行。

**Q: 健康检查任务运行时间过长，是否有优化方式？**

A: 默认情况下健康检查采用串行请求。对于超过 100 条链接的大批次，建议调整 `config/settings.py` 中的 `CHECKER_CONCURRENCY` 参数（例如设置为 10），以使用并发请求。同时注意目标站点的限流策略，合理设置 `REQUEST_INTERVAL` 以避免被目标服务器封禁。

**Q: 如何迁移数据库到另一台服务器？**

A: SQLite 数据库文件默认位于 `./data/linkvault.db`。直接复制该文件到目标服务器相同路径即可。若需迁移至 PostgreSQL 等外部数据库，可参考 `docs/advanced.md` 中的配置指引，修改 `core/storage.py` 中的连接串，并执行 `manage.py migrate` 初始化表结构。
