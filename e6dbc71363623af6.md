# ResourceLink 统一资源外链聚合系统

ResourceLink 是一个面向技术团队、内容创作者与数字化运营人员的开源外链资源聚合与管理平台。该项目并非传统的网页爬虫或书签管理器，而是一套以人工精选与结构化分类为核心的静态资源导航生成工具。它解决了中大型项目文档中外部链接分散、失效风险高、引用格式混乱以及合规追溯困难等痛点，帮助用户将大量分散的参考链接、案例入口与扩展阅读材料整合为可维护、可审计、可快速检索的标准化资源清单。

ResourceLink 的目标用户包括技术文档撰写者、开源项目维护者、企业培训部门以及研究机构信息管理人员。通过本项目提供的模板化分类体系与自动化校验脚本，用户能够将原本杂乱的 URL 集合转化为结构清晰、注释完备、持续集成的资源目录，显著降低跨团队知识传递过程中的信息损耗。

## 功能概览

**批量链接导入与去重** 支持从纯文本文件、CSV 表格或直接粘贴的 URL 列表中批量导入链接，系统自动识别重复条目并保留首次出现的记录，同时标记可能存在的域名变体。

**分类标签与多级目录绑定** 每个资源条目可绑定一个或多个自定义分类标签，并支持将标签映射至项目结构中的特定目录层级，实现资源与代码模块或文档章节的强关联。

**链接可达性定时检测** 内置基于 Headless 浏览器的轻量级检测模块，定期对已收录链接发起请求，记录响应状态码与重定向链，自动标记失效或内容变更的条目。

**元数据扩展字段管理** 除基础标题与 URL 外，支持为每个资源添加维护人、所属项目批次、收录日期、摘要描述与优先级评分等扩展字段，满足企业级审计需求。

**多格式导出与嵌入** 支持将资源列表导出为 Markdown 表格、JSON API 结构或纯文本枚举，方便嵌入至项目 README、静态站点生成器或内部知识库系统中。

**变更历史与回滚能力** 每次对资源列表的增删改操作均记录操作日志与快照，允许用户回溯至任意历史版本，防止误删除或批量覆盖造成的数据丢失。

## 应用场景

技术文档站点的外部引用规范化。大型开源项目的文档通常包含数十个指向外部规范、依赖仓库或社区讨论的链接。使用 ResourceLink 统一管理这些引用后，文档编写者只需在文档中插入资源标识符，由构建过程自动拉取最新 URL，彻底解决硬编码链接维护困难的问题。

企业内部培训资料的知识库构建。培训团队经常需要整理大量在线课程、工具官网与案例演示地址。通过 ResourceLink 的分类标签功能，可以为不同岗位角色定制专属资源视图，并利用可达性检测提前发现因外部站点调整导致的链接失效。

开源项目治理与合规审计。法务或合规部门需要定期审查项目文档中引用的外部资源是否涉及许可证冲突或内容变更。ResourceLink 的元数据扩展字段与变更历史功能为每一条链接提供了完整的追踪记录，极大简化了审计准备工作。

## 快速开始

以下命令演示了如何从 GitHub 克隆项目、安装依赖并启动本地开发服务。

```bash
git clone https://github.com/resourcelink/core.git resourcelink
cd resourcelink
pip install -r requirements.txt
python manage.py migrate
python manage.py loaddata initial_resources.json
python manage.py runserver 0.0.0.0:8000
```

访问本地 8000 端口即可进入管理仪表板。首次启动时，系统会自动创建默认管理员账户，用户名与密码均为 admin，请在首次登录后立即修改。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.10 及以上 | 核心运行时环境，推荐使用 3.11 长期支持版本 |
| Django | 4.2 LTS | Web 框架，用于提供管理界面与 RESTful API 接口 |
| PostgreSQL | 14.0 及以上 | 生产环境推荐数据库，用于存储资源条目、分类与变更日志 |
| Redis | 6.2 及以上 | 可选依赖，用于缓存可达性检测结果与加速频繁查询 |
| Node.js | 18.0 及以上 | 仅用于前端静态资源构建，运行时可忽略 |
| Git | 2.30 及以上 | 用于版本管理及后续的钩子脚本集成 |
| Headless Browser | Chromium 110+ | 用于链接可达性检测，可通过环境变量禁用该功能 |

## 贡献指南

我们欢迎社区开发者以多种形式参与 ResourceLink 项目的改进。请遵循以下步骤提交贡献。

第一，查阅问题追踪器中的待办事项。访问 GitHub Issues 页面，筛选标签为 "help wanted" 或 "good first issue" 的任务，这些任务通常具有明确的范围与较低的进入门槛。

第二，创建功能分支并编写测试。在本地仓库中从 main 分支切出新分支，命名格式为 feature/简述改动内容。对于新增功能或修复，必须编写对应的单元测试用例，确保代码覆盖率不低于 85%。

第三，遵循代码风格规范。Python 代码严格遵循 PEP 8 标准，使用 Black 与 isort 进行自动格式化。前端代码使用 ESLint 与 Prettier 统一风格。提交前请运行 lint 脚本确保无格式警告。

第四，提交拉取请求并描述变更。推送分支至远程仓库后，通过 GitHub 界面发起 Pull Request。请求描述中需清晰说明解决的问题、改动范围以及手动测试步骤。至少需要一位核心维护者审阅通过后方可合并。

第五，更新相关文档。如果改动涉及用户可见的功能变更或配置选项调整，请同步更新 docs 目录下对应的 Markdown 文件，确保文档与代码保持同步。

## 常见问题

Q: 导入大量链接时系统响应变慢甚至超时，应该如何优化？

A: 建议将导入操作拆分为多个小批次，每批不超过 100 条。您可以使用 management/commands/import_links.py 命令并指定 --batch-size 参数控制单次处理数量。对于超过 5000 条的大型导入，推荐在非高峰时段通过 Celery 异步任务队列执行，并在 Django 配置中调整数据库连接池大小。

Q: 链接可达性检测是否会对目标站点造成过大压力？

A: 系统默认并发请求数为 5，且每个请求间隔至少 500 毫秒。检测模块会自动识别并跳过已被目标站点 robots.txt 明确禁止的路径。您可以在 settings/production.py 中通过 HEALTH_CHECK_CONCURRENCY 与 HEALTH_CHECK_DELAY 变量调整请求频率，以适应不同外部站点的承受能力。

Q: 如何将本系统与现有的静态站点生成器（如 Hugo 或 MkDocs）集成？

A: 您可以使用系统提供的导出 API 端点 /api/export/markdown/ 获取完整的资源列表 Markdown 表格，并通过 CI/CD 流水线在站点构建前自动拉取并嵌入至指定文档页面。具体集成示例可参考 docs/export_api.md 中的 Hugo 与 MkDocs 配置片段。
