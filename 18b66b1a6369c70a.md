# LinkVault Resource Aggregator

LinkVault 是一个面向开发者和技术研究人员的轻量级外链资源聚合与导航系统。该项目不存储任何实际文件内容，而是作为结构化索引层，对分散在互联网各处的技术文档、工具站点、开源镜像、API 参考与社区讨论进行统一收束与分类管理。项目定位为"技术资源的入口而非终点"，适用于需要高频查阅外部参考资料、维护私有知识外链体系、或构建团队共享书签库的用户。LinkVault 通过静态配置与自动化校验机制，确保所有收录链接的有效性与可访问性，降低因链接失效导致的信息损耗风险。

## 功能概览

批量链接导入与结构化存储：支持通过 CSV 或 YAML 格式批量导入外部链接，自动根据域名、路径关键词与自定义标签完成归类，并生成 JSON 索引文件供前端或 CLI 工具消费。

链接存活状态周期性检测：内置基于 HTTP HEAD 请求的链接可用性检查器，支持配置超时阈值与重试策略，检测结果以状态标记形式写入索引，便于后续清理或告警。

多维度标签与全文检索：每条链接可附加多个层级标签（如 language、framework、region、author），并提供基于 MiniSearch 或 SQLite FTS5 的轻量级全文检索接口，支持模糊匹配与字段权重调整。

自定义视图与渲染模板：提供可扩展的 Jinja2 模板系统，支持按标签、域名或日期范围生成静态 HTML 导航页，也可输出为纯 Markdown 表格或 JSON Schema，便于嵌入其他文档体系。

外链变更监控与快照比对：对已收录链接的响应内容特征（如 Title、Description、Content-Length）进行定期采样，当检测到显著变化时生成差异报告，辅助判断资源是否发生实质更新或迁移。

权限分级与私有标记：支持多用户只读/可写权限划分，允许团队内部对同一链接添加私有备注、评分或过期提醒日期，私有数据与公开索引分离存储。

CLI 与 HTTP API 双模式交互：提供完整的命令行工具用于日常维护操作，同时暴露 RESTful API 供 CI/CD 流水线或第三方服务集成，API 响应格式兼容 JSON 与 YAML。

## 应用场景

技术团队内部知识库外链管理：开发团队可将项目依赖的官方文档、镜像源、CI/CD 日志链接、监控面板地址统一收录至 LinkVault，配合标签体系区分环境（生产/预发/测试），避免浏览器书签混乱与人员变动导致的信息断层。

开源项目 README 与文档站点外链维护：开源维护者可使用 LinkVault 生成稳定的外链列表，用于项目主页的"相关资源"章节，通过自动化检测避免引用链接在数月后变为死链，提升文档质量与用户信任度。

个人技术研究文献与工具集索引：研究人员可将 arXiv 论文链接、GitHub 仓库、在线实验环境、数据可视化工具等按研究方向分类存储，借助全文检索快速回溯历史参考材料，减少重复查找时间。

自动化链接巡检与告警系统：运维或 SRE 团队可将依赖的外部服务状态页、云厂商健康看板、第三方 API 文档等纳入 LinkVault 监控范围，当检测到链接不可用或页面标题异常变更时，通过 Webhook 触发告警通知。

## 快速开始

以下命令演示了从源码克隆到本地运行完整流程，默认使用 SQLite 作为存储后端，并监听 127.0.0.1:8080。

```bash
git clone https://github.com/example/linkvault.git
cd linkvault
pip install -r requirements.txt
python -m linkvault init --db ./data/linkvault.db
python -m linkvault serve --host 127.0.0.1 --port 8080
```

首次启动后，可通过 `linkvault import --file ./samples/links.yaml` 导入示例数据，或通过 Web 界面手动添加。后台检测任务默认每 24 小时执行一次，可在 `config/production.toml` 中调整 `check_interval_hours` 参数。

## 常见问题

Q：链接检测任务执行缓慢或超时，如何优化？

A：检测性能受网络环境与目标服务器响应速度影响。建议优先调整 `config/production.toml` 中的 `checker.timeout`（默认 10 秒）与 `checker.concurrent_limit`（默认 50）参数。对于大规模索引，可启用 `checker.head_only` 选项仅获取响应头而非完整内容，并配置 `checker.cache_ttl_hours` 缓存成功状态以减少重复请求。若仍无法满足要求，可考虑部署多个检测节点或使用外部监控服务作为补充。

Q：如何将私有备注或标签与公开索引同步？

A：LinkVault 默认将私有数据存储于独立的 SQLite 表 `link_private_meta` 中，与公开索引表 `link_public` 通过 `link_id` 外键关联。公开索引导出或 API 查询默认不包含私有字段，仅在 Web 界面登录后或 API 请求携带有效 JWT 且具备 `read:private` 权限时才会附加。若需完全分离，可在配置中设置 `storage.private_db_path` 指向不同的数据库文件。

Q：项目是否支持 PostgreSQL 替代 SQLite？

A：当前稳定版本仅内置 SQLite 后端，但 `storage/base.py` 定义了完整的数据库抽象接口，理论上可扩展 PostgreSQL 或 MySQL 适配器。社区已有一个实验性的 `storage/postgres_backend.py` 位于 `contrib/` 目录，基于 asyncpg 实现，但尚未经过大规模生产验证。如需在生产环境中使用 PostgreSQL，建议等待官方后续版本或自行基于接口实现，并充分测试事务隔离级别与连接池配置。
