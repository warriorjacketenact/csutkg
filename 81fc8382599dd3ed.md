# WebLink Nexus

WebLink Nexus 是一个面向技术研究者、数据工程师和内容聚合者的高质量外链资源汇总与导航系统。本项目并非简单的书签集合，而是一个具备结构化分类、状态监测和快速检索能力的开源外链基础设施，旨在解决个人与团队在信息收集过程中面临的链接散落、失效不可知、复用成本高等核心问题。

项目定位为“技术资源的外链中台”，目标用户包括开源项目维护者、技术文档编写者、数据采集工程师以及需要系统化整理垂直领域信息的研究人员。通过统一的条目标识体系和轻量级前端界面，用户可以在数秒内定位到所需的外部资源，并实时感知每个链接的可达性与内容主题。

本批次为第 560/1241 批资源收录，共计 250 个外链条目，均已纳入项目索引结构。

## 功能概览

**结构化条目管理**：每个外链以唯一标识符（VID）进行归档，支持按域名、目录层级、内容类型进行多维度筛选与排序。

**实时可达性探测**：集成轻量级健康检查模块，定期对收录链接进行 HTTP 状态验证，自动标记异常条目并生成报告。

**分类标签体系**：支持对每个链接赋予多个主题标签（如“视频流”“技术文档”“API 参考”），便于构建自定义导航面板。

**快速模糊检索**：基于内存索引的即时搜索功能，支持对 URL 路径、域名关键词和自定义备注进行全文匹配。

**导入与导出兼容性**：支持批量导入标准 URL 列表，并可将索引数据导出为 JSON、CSV 或 Markdown 表格格式，便于嵌入其他文档系统。

**扩展钩子机制**：提供事件监听接口，允许用户在链接新增、更新或状态变更时触发自定义脚本，例如自动备份或推送通知。

## 应用场景

**技术文档外链治理**：开源项目的文档维护者可使用 WebLink Nexus 统一管理所有引用外部链接，定期检查链接有效性，避免文档中出现死链，提升文档质量与用户体验。

**数据采集管道前置**：数据工程师可将本系统作为采集任务的前置资源池，通过 API 批量获取待采集 URL 列表，并依据状态探测结果动态调整采集策略，减少无效请求对采集效率的影响。

**垂直领域知识库构建**：研究人员在收集特定领域（如流媒体技术、Web 性能优化）的文献与工具时，可利用分类标签和检索功能快速整理出领域资源地图，并生成可分享的导航页面。

**团队共享书签替代方案**：替代浏览器自带书签的碎片化管理，团队可通过部署统一实例，实现外链资源的集中沉淀、版本追溯和权限分级查看，降低信息孤岛效应。

## 快速开始

以下步骤适用于 Linux / macOS / Windows WSL 环境，确保系统已安装 Git 和 Node.js（v18 或以上）。

使用 Git 克隆项目仓库至本地：

```bash
git clone https://github.com/web-link-nexus/weblink-nexus.git
cd weblink-nexus
```

安装项目依赖（使用 npm 或 yarn）：

```bash
npm install
```

执行初始化脚本以构建索引并启动开发服务器：

```bash
npm run init
npm run serve
```

服务启动后，访问控制台输出的本地地址（默认为 http://127.0.0.1:6174 ）即可开始浏览和检索已收录的外链资源。如需导入自定义列表，可将 CSV 文件放置于 `./data/imports/` 目录下并执行 `npm run import`。

## 项目结构

```
weblink-nexus/
├── cmd/
│   └── server/                # 主服务入口，负责启动 HTTP 和探测调度
├── internal/
│   ├── indexer/               # 索引构建器，解析导入数据并生成倒排索引
│   ├── probe/                 # 健康检查模块，包含 curl 调用封装与超时控制
│   ├── storage/               # SQLite 数据访问层，包含迁移脚本与查询构造器
│   ├── api/                   # RESTful 接口定义，含路由注册与中间件链
│   └── config/                # 配置加载逻辑，支持环境变量和本地 YAML 覆盖
├── pkg/
│   ├── models/                # 公共数据模型定义（Link, Tag, ProbeResult）
│   └── utils/                 # 通用工具函数（字符串处理、时间格式化、哈希计算）
├── web/
│   ├── public/                # 静态资源目录（favicon、robots.txt）
│   └── src/                   # 前端源码（基于 Vanilla JS + 原生 CSS）
├── data/
│   ├── imports/               # 用户放置批量导入 CSV 文件的目录
│   ├── exports/               # 导出数据输出目录（JSON / CSV / Markdown）
│   └── storage.db             # SQLite 数据库文件（自动生成）
├── scripts/
│   ├── init-db.sh             # 初始化数据库表结构及默认配置
│   ├── import.sh              # 批量导入入口脚本，调用内部索引器
│   └── export.sh              # 导出数据为指定格式的脚本封装
├── docs/                      # 完整文档目录（见上文文档导航）
├── config.yaml.example        # 配置文件模板，包含探测间隔、端口、日志级别
├── .gitignore
├── package.json               # npm 依赖清单与脚本定义
└── README.md                  # 本文件
```

## 贡献指南

1.  **查阅现有议题**：在提交新功能或修复之前，请先浏览 GitHub Issues 列表，确认无人正在处理相同或类似的问题。若存在相关讨论，请在该议题下表明参与意愿。

2.  **派生仓库并创建特性分支**：将本仓库派生至个人账号，并基于 `main` 分支创建以 `feature/` 或 `fix/` 为前缀的新分支，例如 `feature/add-tag-filter`。

3.  **编写或修改代码并遵循风格规范**：所有 JavaScript 代码须通过 ESLint 配置（见 `.eslintrc.yml`）的校验，提交前执行 `npm run lint` 和 `npm run test` 确保无风格问题和回归错误。

4.  **补充单元测试与文档注释**：新增或修改核心逻辑（如索引构建、探测调度）时，须在 `./test/` 目录下添加相应的单元测试用例，并同步更新 `./docs/` 中受影响的用户手册或开发者文档。

5.  **发起拉取请求并关联议题**：推送分支后，向本仓库的 `main` 分支发起 Pull Request，在描述中明确关联相关议题编号，并简述变更内容、测试覆盖情况以及对现有功能的影响。

## 常见问题

**问：健康检查模块是否会对外部目标造成请求压力？**

答：系统默认采用指数退避策略，每个链接的探测间隔不低于 300 秒，且并发请求数限制为 5 个。用户可在 `config.yaml` 中调整 `probe.interval` 和 `probe.concurrency` 参数以适配自身网络环境和目标服务器的承受能力。

**问：导入大量链接时，系统性能是否会显著下降？**

答：索引构建过程采用批量插入和增量更新策略。对于 250 条规模的批次，初始化耗时通常在 2 秒以内。全量检索的响应时间维持在 50 毫秒以下，得益于内存缓存与 SQLite 索引的配合。若条目数超过 5000，建议调整 SQLite 的 `cache_size` 和 `mmap_size` 编译参数。

**问：如何将当前索引数据迁移到另一台服务器？**

答：直接复制 `./data/storage.db` 文件至目标服务器的相同相对路径，并确保系统级依赖（SQLite3、curl、jq）版本兼容即可。若需跨数据库迁移，可使用 `npm run export -- --format json` 导出数据，然后在目标服务器执行 `npm run import -- --file export.json` 完成重建。
