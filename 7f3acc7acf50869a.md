# Tancheng Legal Resource Aggregator

Tancheng Legal Resource Aggregator is a specialized technical documentation and external resource indexing system designed for legal professionals, academic researchers, and policy analysts who require structured access to distributed legal case archives and regulatory materials. The system provides a unified query interface over a curated collection of legal episode records, enabling efficient retrieval of case precedents, judicial interpretations, and procedural documentation from the Tancheng legal knowledge base.

This project addresses the critical challenge of navigating large-scale legal document repositories where episode identifiers follow non-intuitive alphanumeric patterns. By implementing a lightweight indexing layer with metadata extraction capabilities, the aggregator transforms raw URL-based episode references into a searchable, filterable, and cross-referenced knowledge graph. The target audience includes legal tech developers building case management systems, law librarians maintaining digital archives, and compliance officers conducting regulatory research across multiple jurisdictional boundaries.

## 功能概览

**Episode Identifier Normalization** – Automatically parses and validates the alphanumeric episode codes from URL paths, extracting embedded metadata patterns and normalizing them into consistent record keys for internal indexing.

**Bulk URL Ingestion Pipeline** – Supports batch processing of episode URLs through a configurable crawler that respects robots.txt directives and implements exponential backoff retry logic for reliable large-scale data acquisition.

**Full-Text Search Over Case Summaries** – Provides inverted-index search capabilities across downloaded episode content, with support for boolean operators, phrase matching, and proximity searches over judgment texts and procedural histories.

**Metadata Extraction And Tagging** – Applies rule-based natural language processing to extract entity names, court levels, judgment dates, and statutory citations from each episode document, generating facet tags for faceted navigation.

**RESTful Query API** – Exposes a JSON-based API endpoint for programmatic access to the indexed episode collection, supporting pagination, sorting, and field-filtering parameters for integration with external legal workflow tools.

**Export And Reporting Module** – Generates structured reports in CSV, JSON, and Markdown table formats, enabling offline analysis, citation network construction, and compliance audit trail documentation.

**Versioned Change Tracking** – Maintains an append-only log of all ingestion operations, allowing administrators to replay updates, rollback erroneous imports, and monitor the evolution of the episode corpus over time.

## 应用场景

Legal research firms conducting retrospective analyses of judicial decision patterns can leverage the aggregator to compile customized datasets spanning multiple episode categories. Researchers input a list of episode identifiers, and the system retrieves corresponding full-text documents along with extracted metadata fields for quantitative analysis of ruling tendencies.

Academic institutions teaching advanced legal methodology courses utilize the aggregator as a pedagogical tool for demonstrating large-scale document retrieval concepts. Students learn to construct complex search queries, evaluate precision-recall tradeoffs, and interpret metadata distributions across the episode collection.

Corporate compliance departments monitoring regulatory changes maintain a local mirror of relevant episode records using the aggregator's scheduled synchronization feature. Compliance teams configure daily or weekly update cycles to ensure their internal policy databases reflect the most recent judicial interpretations and administrative rulings.

Independent legal aid organizations operating in low-bandwidth environments deploy the aggregator as an offline reference appliance. The system's lightweight indexing engine runs on commodity hardware and provides full-text search capabilities without requiring persistent internet connectivity, ensuring access to critical legal information in underserved regions.

Government archives and public record repositories integrate the aggregator into their document management workflows to enhance discoverability of historical case files. The system's metadata extraction pipeline converts unstructured episode texts into structured records suitable for integration with national legal information systems.

## 快速开始

```bash
# Clone the repository from the official source
git clone https://github.com/tancheng-legal/aggregator.git

# Navigate to the project root directory
cd aggregator

# Install runtime dependencies using pip with virtual environment
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# Initialize the local index database and run the ingestion worker
python manage.py migrate
python manage.py ingest --source episodes.csv
python manage.py runserver --host 0.0.0.0 --port 8080
```

## 安装要求

| 依赖 | 必需版本 | 说明 |
|------|----------|------|
| Python | 3.9 - 3.11 | Core runtime interpreter; 3.12 not yet fully validated |
| PostgreSQL | 13.0 或更高 | Primary relational database for metadata and index storage |
| Redis | 6.2 或更高 | Caching layer and task queue broker for asynchronous ingestion |
| Elasticsearch | 7.17.x | Full-text search engine; 8.x supported but requires configuration changes |
| Git | 2.30 或更高 | Version control system for repository management and patch application |
| Node.js | 16.0 或更高 | Required only for building the administrative dashboard frontend |
| wkhtmltopdf | 0.12.6 | Optional dependency for PDF report generation from HTML templates |
| OpenJDK | 11 | Required for running Apache Tika, used in document content extraction |
| Nginx | 1.18 或更高 | Recommended production reverse proxy for load balancing and TLS termination |

## 项目结构

```
aggregator/
├── cmd/
│   ├── crawler/                  # 独立爬虫可执行程序入口
│   │   ├── main.go               # 爬虫主流程，包含 URL 队列管理和并发控制
│   │   └── worker_pool.go        # 工作池实现，管理 HTTP 请求的并发数与超时重试
│   ├── api/                      # REST API 服务入口
│   │   ├── server.go             # HTTP 服务器初始化，注册路由和中间件
│   │   └── handlers/             # 各端点处理器：搜索、元数据、导出等
│   └── cli/                      # 命令行工具入口，用于运维和调试
│       ├── ingest.go             # 批量导入 CSV 或 JSON 格式的 URL 列表
│       └── index.go              # 手动触发索引重建和优化操作
├── internal/
│   ├── core/                     # 核心业务逻辑，不对外暴露
│   │   ├── episode.go            # Episode 结构体定义，包含 URL 解析和验证方法
│   │   ├── metadata.go           # 元数据提取器：从 HTML 中抽取 court, date, citations
│   │   └── search.go             # 搜索索引的构建与查询实现，基于倒排索引
│   ├── storage/
│   │   ├── postgres/             # PostgreSQL 适配器，实现记录持久化
│   │   ├── elastic/              # Elasticsearch 适配器，实现全文检索
│   │   └── cache/                # Redis 缓存适配器，存储高频查询结果
│   ├── queue/
│   │   ├── rabbitmq/             # RabbitMQ 适配器，用于分布式任务分发
│   │   └── memory/               # 开发环境使用的内存队列实现
│   └── parser/
│       ├── html/                 # HTML 解析器，基于 goquery 提取正文和元标签
```

## 常见问题

**Q: 系统如何处理 URL 不可访问或返回非 200 状态码的情况?**

A: 爬虫组件实现了三层重试机制：首次失败后等待 5 秒重试，第二次失败后等待 30 秒，第三次失败后将该 URL 记录到失败队列并跳过。所有失败记录均写入 logs/failed_ingest.log，管理员可手动重试。对于永久性不可访问的 URL，系统会保留其元数据占位符，并标记为"unreachable"状态以供审计。

**Q: 能否在完全离线环境下部署和使用本系统?**

A: 可以。系统设计为支持离线部署模式。首次安装时需在联网环境中下载依赖包和基础索引模板，之后可将整个容器镜像或虚拟环境迁移至离线网络。所有外部资源链接在系统内仅作为索引键存储，实际内容需通过爬虫预先获取并存入本地存储。离线环境下，搜索功能、元数据查询和报告生成均不受影响，但实时爬取和更新功能不可用。

**Q: 搜索结果的排序逻辑是什么，能否自定义权重?**

A: 默认排序基于 BM25 算法计算的相关性分数，同时结合文档新鲜度（按收录日期递减）进行轻微加权。用户可以通过查询参数 sort_by 指定按 date, relevance, 或 case_id 排序。高级用户可在 configs/search.yaml 中调整字段权重系数，例如提高标题字段的匹配权重或降低正文长度对分数的归一化因子。自定义排序函数可以通过扩展 internal/core/search.go 中的 Scorer 接口实现。
