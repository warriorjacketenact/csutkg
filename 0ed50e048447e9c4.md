# ResourceBridge

ResourceBridge 是一个面向技术团队与内容运营者的外链资源归集与管理平台，专注于解决多源、多批次、多类型外部链接的集中存储、分类索引与状态监控问题。项目定位于中大规模资源链接的规范化管理场景，适用于需要长期维护数百至数千条外部 URL 的技术文档站、导航站或知识库系统。

ResourceBridge 不提供爬虫或自动采集功能，而是提供标准化的资源录入接口、链接健康检查机制、分类标签体系以及多格式导出能力。目标用户包括开源项目维护者、技术内容运营人员、企业内部知识管理团队以及需要定期整理外部参考链接的研究人员。

本项目属于第 439/1241 批次资源整合计划的一部分，当前批次共包含 250 个外链条目，均已完成基础校验与分类标记。

## 功能概览

**批量链接导入**：支持通过 CSV、JSON 或纯文本列表形式批量导入外部 URL，自动解析协议与域名层级，完成基础格式校验。

**链接健康状态监控**：内置 HTTP 状态码检查器，可定期探测每个链接的可达性，自动标记失效、重定向或超时链接，并生成异常报告。

**分类标签体系**：支持为每条链接添加多级分类标签与自定义备注字段，便于按主题、来源、优先级或批次进行筛选与排序。

**多格式数据导出**：支持将资源列表导出为 Markdown 表格、HTML 目录页、JSON 结构化数据或纯文本清单，满足文档嵌入、静态站点生成或二次开发需求。

**历史版本追溯**：每次导入或编辑操作均记录时间戳与操作者信息，支持回滚至任意历史版本的链接集合状态。

**权限分级管理**：内置只读访客、编辑者、管理员三级权限模型，适用于多人协作的资源维护场景。

**检索与过滤**：提供基于关键词、标签、状态码、批次号的组合检索功能，支持正则表达式过滤规则。

**开放 API 接口**：提供 RESTful API 用于与其他内部系统集成，支持链接的增删改查及状态查询操作。

## 应用场景

技术文档站的外链维护：技术文档中往往包含大量参考链接、API 文档链接和第三方工具链接。随着时间推移，这些链接可能失效或变更。ResourceBridge 可定期检查文档站中嵌入的所有外部链接，自动生成失效链接报告，帮助文档维护者快速定位并修复问题。

企业内部知识库的资源整合：企业内部知识库通常分散存储于 Confluence、Notion、GitLab Wiki 等多个平台，外部参考链接分散在各处。使用 ResourceBridge 可以建立一个统一的外部资源注册中心，各平台通过 API 查询链接有效性，避免重复维护和链接冗余。

开源项目的社区资源导航：开源项目 README 或官方文档中常包含社区插件、教程文章、视频课程等外部资源列表。ResourceBridge 可以帮助项目维护者批量管理这些链接，按版本发布周期导出静态链接清单，确保每次发版时附带的外部资源均有效且分类清晰。

学术研究的参考文献链接管理：研究人员在撰写论文或技术报告时需要引用大量在线资源。ResourceBridge 提供批次管理功能，可将某一研究方向的所有引用链接归为一个批次，导出为标准引用格式，并持续监控链接可用性，防止发表后引用链接失效。

## 快速开始

以下命令演示如何从 GitHub 克隆项目、安装依赖并启动开发服务。

```bash
git clone https://github.com/resourcebridge/resourcebridge.git
cd resourcebridge
npm install
npm run dev
```

生产环境构建与启动：

```bash
npm run build
npm start
```

Docker 方式启动：

```bash
docker build -t resourcebridge:latest .
docker run -p 3000:3000 -d resourcebridge:latest
```

## 贡献指南

欢迎社区开发者参与 ResourceBridge 的改进与维护。请遵循以下流程提交贡献。

1. 查阅 Issues 列表，选择标记为 "help wanted" 或 "good first issue" 的问题，或在开始较大规模改动前先创建新 Issue 说明意图，避免重复劳动。

2. Fork 本仓库至个人账号，在本地新建功能分支，分支命名格式为 `feature/功能简述` 或 `fix/问题简述`，确保分支基于最新的 main 分支。

3. 编写代码时遵循项目已配置的 ESLint 与 Prettier 规则，所有新增或修改的 API 接口需同步更新 docs/api-reference.md 中的对应字段说明，新增工具函数需包含单元测试，测试覆盖率不低于 80%。

4. 提交代码时使用语义化提交信息格式，如 `feat: 添加链接批量标签更新接口` 或 `fix: 修复健康检查超时未正确记录的问题`。提交前运行 `npm run test` 确保所有现有测试通过。

5. 发起 Pull Request 至 main 分支，PR 描述中需明确关联对应的 Issue 编号，并简要说明改动内容与测试结果。PR 将由项目维护者进行 Code Review，通过后将合并至主分支并同步更新 CHANGELOG。

## 常见问题

**Q：导入的链接数量较大时，健康检查是否会阻塞其他操作？**

健康检查服务以独立的后台任务队列方式运行，默认配置下并发检查数为 10，超时时间为 5000 毫秒。检查过程不会阻塞链接的 CRUD 操作或 API 请求。用户可在配置文件中调整 concurrency 与 timeout 参数以适配自身服务器性能。检查结果会异步写入数据库，并通过日志系统输出进度。

**Q：如何迁移已有的大量外链数据到 ResourceBridge？**

项目提供两种迁移路径。若已有数据为 CSV 或 JSON 格式，可直接使用批量导入接口或 CLI 命令 `npm run import -- --file=data.json` 完成导入。若数据分散在多个 Markdown 文档中，建议先使用脚本提取所有 URL，再通过 importService 的文本解析功能按行导入。导入后系统会自动去重并生成批次编号，方便后续统一管理。

**Q：导出为 Markdown 表格时，是否支持自定义表头与分组？**

导出模块支持模板定制功能。用户可在 config/export-templates 目录下放置自定义 Handlebars 模板文件，模板中可引用链接对象的任意字段（url、status、tags、batchId 等）。导出时通过 `--template=custom.hbs` 指定模板即可生成符合文档风格的表格结构。默认模板按状态码分组展示，并标记失效链接。
