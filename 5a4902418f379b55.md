# EON Resource Catalog

EON Resource Catalog 是一个面向技术研究、数据分析与知识管理领域的结构化外链资源汇总系统。本项目并非传统意义上的内容聚合平台，而是一套基于严格分类体系与持久化标识符管理的资源导航工具，专为需要高频查阅特定领域参考资料、技术文档或数据源的研发团队、学术研究者以及个人知识工作者设计。

项目定位为轻量级资源索引层，不直接存储或托管任何第三方内容，而是通过稳定的视图标识符系统，将分散于网络各处的高价值信息入口进行统一归集与语义化标注。用户可通过本项目快速定位至经筛选的外部资源，显著降低信息检索过程中的时间损耗与认知负担。

本批次资源收录工作属于项目整体建设计划的第 1114 至 1241 批次，共计纳入 250 个独立资源链接。所有资源链接均保留原始交付格式与协议头，确保跳转路径的完整性与可追溯性。项目维护者定期对链接有效性进行自动化校验，并对失效条目执行标记与替换流程。

## 功能概览

**结构化资源索引**：基于多维分类体系对全部外链进行主题聚类，支持按技术领域、内容类型、更新频率等维度快速筛选。

**永久视图标识系统**：每个资源条目均分配唯一视图标识符，形成稳定的内部引用链，便于项目内交叉索引与版本追溯。

**轻量化快速部署**：项目本身不依赖数据库或外部服务进程，克隆即用，所有资源数据以纯文本格式存储，兼容各类静态托管环境。

**分类目录动态生成**：提供自动化脚本，可根据资源元数据实时生成分类目录树与标签云，保持导航结构随资源增删动态更新。

**多格式数据导出**：支持将资源列表导出为 JSON、CSV 及 HTML 书签格式，方便用户将数据迁移至其他工具链或进行离线分析。

**链接健康度监控**：内置链接状态检查工具，可定时探测各资源端点的可访问性，并生成异常报告，辅助维护者及时处理失效链接。

**上下文备注扩展**：每条资源记录支持附加补充说明字段，用于记录资源用途、关键内容摘要或使用注意事项，丰富索引信息的参考价值。

## 应用场景

技术团队内部知识库构建：研发团队可利用本项目搭建私有技术文档导航站，将日常使用的 API 参考手册、开源库主页、技术博客等分散资源统一整理，新成员入职时可快速了解团队常用工具链与信息来源。

学术文献与数据源管理：科研人员在开展文献综述或数据分析工作时，可将项目作为个人研究资源工作台，集中存放预印本服务器、数据集下载页、统计年鉴入口等链接，配合分类功能实现按研究课题的分组管理。

运维监控与文档归集：运维工程师可将本项目作为内部监控面板、日志查询平台、配置管理文档的导航入口，通过统一的视图标识符快速切换不同环境或组件的管理后台，减少手动输入 URL 或翻阅收藏夹的时间。

个人知识体系外延：知识工作者可将项目作为个人知识管理系统（如 Obsidian、Notion）的外部资源补充层，将网络上的散落高价值信息通过本项目进行结构化整理，形成与本地笔记互为参照的完整知识网络。

## 快速开始

以下命令序列适用于 Linux 与 macOS 环境。Windows 用户可使用 Git Bash、WSL 或 PowerShell 中兼容的终端执行。

```bash
# 克隆项目仓库至本地
git clone https://github.com/eon-resource/eon-catalog.git
cd eon-catalog

# 安装项目依赖（基于 Python 3.8+，使用 pip 安装工具链）
pip install -r requirements.txt

# 执行资源索引构建与本地预览
python build_index.py --input data/raw_links_1114_1241.txt --output dist/
python serve_preview.py --port 8080
```

执行上述命令后，打开浏览器访问 http://localhost:8080 即可查看本批次资源的分类导航页面。若需生成静态站点文件，可运行 `python build_static.py --output public/` 将全部页面输出至指定目录，随后托管至任意 HTTP 服务器。

## 安装要求

项目运行与开发环境依赖如下表所列。所有依赖均可通过 Python 包管理工具 pip 直接安装。推荐使用虚拟环境隔离项目依赖。

## 项目结构

项目目录遵循模块化设计原则，将原始数据、处理脚本、配置定义与生成输出分离。核心目录结构如下所示。

```
eon-catalog/
├── data/                           # 原始资源数据存储目录
│   ├── raw_links/                  # 按批次存放的原始链接列表
│   │   └── batch_1114_1241.txt     # 本批次 250 条链接原始文本
│   ├── metadata/                   # 资源补充元数据定义
│   │   └── annotations.yaml        # 视图标识符对应的分类与备注信息
│   └── schemas/                    # 数据校验规则定义
│       └── link_schema.json        # 链接格式与字段约束 JSON Schema
├── scripts/                        # 可执行工具脚本集合
│   ├── build_index.py              # 主构建脚本，解析输入并生成导航索引
│   ├── serve_preview.py            # 本地预览服务启动脚本
│   ├── build_static.py             # 静态站点全量生成脚本
│   └── check_links.py              # 链接健康度批量检测工具
├── templates/                      # Jinja2 HTML 模板文件
│   ├── base.html                   # 全站基础布局模板
│   ├── index.html                  # 资源总览首页模板
```

## 贡献指南

本项目欢迎外部贡献者参与资源补充、分类优化及工具链改进。请遵循以下流程提交贡献。

第一，查阅现有分类体系与资源列表，确认拟新增或修改的资源尚未被收录，且不属于项目明确排除的内容类型（如动态接口地址、需登录授权的内部系统链接、临时性活动页面等）。

第二，在 data/metadata/annotations.yaml 文件中按既有格式添加或修改资源条目，需提供视图标识符、目标 URL、建议分类及简要说明。若为新增资源，请使用项目提供的辅助脚本 `scripts/generate_id.py` 生成符合规范的新标识符。

第三，执行本地构建与预览流程，通过 `python build_index.py --validate-only` 运行数据校验，确保新增条目标格式正确、无重复标识符。随后启动预览服务，在浏览器中检查分类归属与备注渲染效果是否符合预期。

第四，提交拉取请求至项目主仓库。请求说明中应明确列出本次变更涉及的资源标识符列表、变更类型（新增/修改/删除）以及必要的变更理由。若涉及链接替换或删除操作，需附上链接失效的检测日志或相关说明。

第五，等待项目维护者审阅。审阅期间可能要求补充资源用途说明或调整分类归属。合并后，新增资源将随下一轮静态站点构建自动发布。

## 常见问题

问：项目本身不存储任何第三方内容，但访问资源链接时出现 404 或连接超时，应该如何处理？

答：项目维护者会定期运行 `scripts/check_links.py` 对全部链接进行状态检测，并将检测结果记录于 `data/link_status.log`。用户若发现特定链接不可用，可通过 GitHub Issues 提交报告，附带视图标识符与访问时间。维护者核实后将标记该资源为失效状态，并在后续版本中尝试寻找替代链接或将其移出活跃索引。用户也可在本地运行检测脚本自行生成报告。

问：如何将本项目生成的资源导航页面部署到公网或内网服务器？

答：执行 `python build_static.py --output /path/to/webroot` 即可生成完整的静态 HTML 文件集。将该目录下的所有文件上传至任意支持静态托管的 HTTP 服务器（如 Nginx、Apache、Caddy 或对象存储静态网站功能）即可直接访问。项目不依赖服务端动态语言或数据库，因此也可部署至 GitHub Pages、Cloudflare Pages 等平台。

问：项目中的视图标识符是否有特定语义含义？能否通过标识符反推资源内容？

答：视图标识符为随机生成的字母组合，长度固定，不包含任何语义信息。该设计旨在避免标识符变更时因内容主题调整导致引用断裂，同时降低外部猜测有效标识符的可能性。资源的分类与内容描述完全通过 `annotations.yaml` 中的元数据字段管理，标识符仅作为内部引用的唯一键值。
