# ResourceBridge

ResourceBridge 是一个面向技术文档工程师、开发者关系维护者以及开源社区运营者的外链资源归集与导航系统。该项目并非简单的书签管理工具，而是一套具备分类索引、关系图谱与状态监控能力的资源网关。

ResourceBridge 的核心定位在于解决技术团队在维护外部参考链接时面临的三大痛点：链接失效不可感知、资源分类混乱难以检索、以及上下文信息缺失导致重复劳动。本项目通过结构化的元数据定义与自动化的健康检查流程，帮助团队将零散的浏览器收藏夹转化为可共享、可继承、可审计的组织知识资产。

目标用户包括编写技术白皮书与集成指南的文档工程师、整理 API 示例库的开发者布道师、以及需要为内部培训体系搭建外部阅读清单的技术管理者。ResourceBridge 不依赖特定云服务，可独立部署于内网或公共服务器，所有资源数据以纯文本格式存储，便于版本控制与协同编辑。

## 功能概览

**批量链接导入与解析**：支持从 Markdown 表格、CSV 文件或原始 URL 列表批量导入资源条目，自动提取域名、路径层级与查询参数，生成标准化的资源对象。

**自定义分类标签体系**：允许用户创建多级分类与自由标签，同一资源可归属多个分类，支持基于分类路径的快速筛选与聚合视图。

**链接可达性周期性巡检**：内置轻量级调度器，可按每日、每周或每月周期对已收录链接发起 HEAD 请求，标记异常状态并生成变化日志。

**资源元数据扩展字段**：每条记录支持标题、作者、发布日期、所属系列、摘要描述与关联项目等十余个扩展属性，满足企业级知识库的细粒度管理需求。

**全文检索与高级过滤器**：基于倒排索引实现标题与描述的快速搜索，结合分类、标签、状态码、更新时段等多维度过滤条件，精准定位目标资源。

**数据导入导出标准接口**：提供 JSON、YAML 与 Markdown 三种导出格式，支持与静态站点生成器、文档平台或数据可视化看板进行对接。

**访问热度统计与排序**：记录每条资源的点击次数与最后访问时间，支持按热度、新增时间或字母顺序动态排列列表，辅助识别高频参考材料。

**协作审核工作流**：针对新增或编辑的资源条目，提供“草稿 - 审核 - 已发布”三态状态机，支持多人协作场景下的内容质量控制。

## 应用场景

技术文档部门的参考资料库建设：文档团队在编写系统架构说明或 SDK 使用指南时，需要引用大量外部规范、博客文章与社区讨论。ResourceBridge 可为每篇文档生成专属的外部链接附录，并在版本发布前自动检查所有引用链接的有效性，避免文档上线后出现死链。

开源社区的外部贡献指南聚合：开源项目维护者通常在 README 或贡献文档中列出行为准则、编码规范、讨论论坛与相关工具链的地址。ResourceBridge 能够将这些分散的链接统一管理，并为社区贡献者提供一个可视化的资源门户，减少新成员在信息查找上耗费的时间。

技术培训课程的阅读清单管理：企业技术培训部门每季度会为不同岗位的学员准备课外阅读材料。ResourceBridge 的分类标签功能可以将材料按“基础概念”、“实战案例”、“性能调优”、“安全合规”等维度组织，培训讲师可随时增删条目并推送更新通知给学员。

个人技术研究的知识外脑：研究人员或资深工程师在调研新兴技术方向时，往往会积累数百个临时标签页。ResourceBridge 提供快速记录与后续整理能力，帮助将临时收藏转化为长期可检索的个人知识索引。

## 快速开始

以下指令适用于 Linux 与 macOS 环境，Windows 用户可使用 WSL2 或 Git Bash 执行。

```bash
# 克隆项目仓库至本地
git clone https://github.com/resourcebridge/resourcebridge.git

# 进入项目根目录
cd resourcebridge

# 安装依赖项（使用 npm，若使用 yarn 可替换为 yarn install）
npm install

# 以开发模式启动服务，默认监听 127.0.0.1:3000
npm run dev
```

启动成功后，在浏览器中访问 http://127.0.0.1:3000 即可进入 ResourceBridge 的仪表盘界面。首次启动会自动生成示例数据与默认管理员账户，请根据控制台输出的提示完成初始配置。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | 18.x LTS 或更高 | 运行时环境，推荐使用 nvm 管理多版本 |
| npm | 9.x 或更高 | 包管理器，随 Node.js 一同安装 |
| SQLite3 | 3.40.0 或更高 | 嵌入式数据库，用于存储资源元数据与巡检记录，无需单独安装 |
| Git | 2.30.0 或更高 | 版本控制工具，用于克隆仓库与获取更新 |
| 系统内存 | 最低 512 MB，推荐 1 GB | 内存占用主要取决于链接总数与巡检并发数 |
| 磁盘空间 | 最低 100 MB | 用于存储数据库文件与日志，每万条链接约占用 15 MB |
| 操作系统 | Linux / macOS / Windows (WSL2) | 生产环境推荐使用 Ubuntu 20.04 或更新版本 |
| 网络访问 | 外网出站可达 | 巡检功能需要向目标域名发起请求，内网部署需配置代理 |
| 时钟同步 | NTP 服务正常运行 | 用于巡检时间戳与日志排序，偏差超过 5 秒将影响调度准确性 |

## 常见问题

Q: 巡检功能对目标服务器会造成压力吗？如何控制请求频率？

A: ResourceBridge 的巡检模块默认采用指数退避策略，并发请求数上限为 5，且两次请求之间至少间隔 200 毫秒。对于每个目标域名，系统会自动限制每秒不超过 2 次请求。用户可在配置文件中调整 maxConcurrency 与 rateLimitPerDomain 参数以满足内部网络策略。此外，巡检仅发送 HEAD 请求，不下载响应体，对目标服务器的资源消耗极低。

Q: 部署在内网环境无法访问外网链接时，巡检功能如何处理？

A: 当 ResourceBridge 运行于隔离内网时，巡检模块会记录超时错误并将链接状态标记为 UNREACHABLE。用户可通过配置 httpProxy 字段指定出站代理地址，或通过 skipHealthCheckForPrivateIP 选项跳过对内网地址的检查。对于必须验证的外网链接，建议部署一台位于边界网络的巡检代理节点，通过 API 将结果回传至主服务。

Q: 如何从旧版本的 ResourceBridge 迁移数据到新版本？

A: 项目在发布每个主要版本时都会提供对应的迁移脚本，存放于 src/db/migrations 目录。迁移流程为：备份数据库文件，停止旧版本服务，执行新版本的迁移脚本（如 node scripts/migrate.js --to v2），然后启动新版本服务。迁移脚本仅修改表结构，不会删除或修改已有数据。建议在预发布环境先验证迁移流程，再应用于生产环境。
