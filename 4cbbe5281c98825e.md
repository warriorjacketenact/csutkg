# LinkForge

LinkForge 是一个面向技术研究者、开源贡献者以及内容创作者的轻量级外链资源汇集与导航系统。该项目定位于将分散在各类技术社区、文档站点与媒体平台中的高价值外部链接进行结构化整理，通过分类索引与快速检索机制，帮助用户在海量信息中精准定位所需资源。LinkForge 本身不存储任何第三方内容，仅提供元数据描述与链接跳转功能，适用于个人知识管理、团队技术栈沉淀以及公开文档站的外链增强场景。

## 功能概览

- **分类索引引擎**：支持按技术领域、资源类型、来源平台等多维度对链接进行自动或手动归类，提供层级分明的浏览路径。
- **全文检索与过滤**：内置基于标题、描述、标签的轻量级搜索组件，支持模糊匹配与多条件组合过滤，实现秒级结果返回。
- **链接健康状态监测**：周期性对收录的 URL 发起可用性探测，自动标记失效或响应超时的链接，并在管理面板中生成异常报告。
- **批量导入与导出**：支持通过 CSV、JSON 或纯文本列表形式批量新增链接，亦可按选定分类导出链接集合，便于迁移或备份。
- **访问统计与热度排序**：记录每个外部链接的点击次数与最后访问时间，支持按热门度、新增时间或字母序对列表进行动态排序。
- **自定义元数据扩展**：允许用户为每个链接添加自定义字段（如阅读时长、适用水平、关联项目），满足个性化管理需求。
- **响应式前端界面**：提供适配桌面与移动设备的浏览界面，确保在不同屏幕尺寸下均可获得良好的可读性与操作反馈。
- **开放 API 接口**：提供 RESTful 风格的只读 API，支持第三方应用以 JSON 格式获取链接列表及详情，便于集成到其他工具链中。

## 应用场景

- **技术团队内部知识库外链管理**：开发团队可将项目中涉及的官方文档、依赖库主页、技术方案参考等外部链接统一录入 LinkForge，并按照微服务、前端、运维等模块分类，新成员入职时可快速了解技术生态全貌。
- **开源项目 README 外链补充**：开源维护者通常需要在 README 中列出相关教程、社区论坛或扩展工具，但受限于文档长度无法详尽罗列。通过 LinkForge 生成的外部链接页面，可以在 README 中仅放置一个入口链接，同时保持资源的持续更新与扩充。
- **技术博客与内容聚合站的外链附录**：博客作者或新闻聚合站点在发布专题文章时，可将文中引用的所有参考资料、数据来源及延伸阅读链接整理至 LinkForge 的对应分类下，读者可一键访问原始出处，提升内容的可追溯性与可信度。
- **个人学习路径的资源收藏夹**：自学者在学习新技术栈时，往往需要收集大量视频教程、在线练习平台、API 参考手册等。利用 LinkForge 的分类与搜索功能，可以构建一套结构清晰、可随时增删的个人学习资源库，替代浏览器自带的扁平化书签系统。

## 快速开始

以下命令演示如何在本地环境中获取 LinkForge 源码、安装依赖并启动开发服务器。

```bash
# 克隆代码仓库
git clone https://github.com/linkforge/linkforge.git
cd linkforge

# 安装项目依赖（使用 npm）
npm install

# 启动开发模式，默认监听 3000 端口
npm run dev
```

执行上述步骤后，在浏览器中访问 `http://localhost:3000` 即可看到 LinkForge 的主界面。首次启动时会自动生成默认的分类模板与示例链接数据，便于快速体验核心功能。

## 安装要求

LinkForge 采用 Node.js 作为运行时环境，数据存储层使用 SQLite 嵌入式数据库，无需额外安装数据库服务。以下为必需的依赖项与说明：

| 依赖 | 必需 | 说明 |
|------|------|------|
| Node.js 18.x 或更高 | 是 | 项目运行时环境，推荐使用 LTS 版本以保证稳定性 |
| npm 9.x 或更高 | 是 | Node.js 包管理器，用于安装项目依赖与执行脚本 |
| SQLite 3 | 是 | 嵌入式关系型数据库，用于存储链接元数据与分类信息 |
| Git 2.x | 否 | 仅用于从版本仓库克隆源码，生产环境部署可省略 |
| 现代浏览器（Chrome/Firefox/Edge） | 否 | 用于访问前端界面，支持 ES6 及 CSS Grid/Flexbox 特性 |
| 网络连接 | 否 | 仅在首次启动时需下载 npm 包，之后可离线运行 |

## 项目结构

项目采用分层架构设计，前端界面与后端服务分离，数据层通过 ORM 访问 SQLite 数据库。以下为核心目录与文件说明：

```

│   ├── controllers/                # 请求控制器，处理路由逻辑
│   │   ├── linkController.js       # 链接增删改查及搜索接口
│   │   └── categoryController.js   # 分类管理接口
│   ├── models/                     # 数据模型定义
│   │   ├── Link.js                 # 链接表结构及实例方法
│   │   ├── Category.js             # 分类表结构及层级关系
│   │   └── HealthCheck.js          # 健康检查记录模型
│   ├── routes/                     # 路由注册与中间件配置
│   │   ├── api.js                  # RESTful API 路由聚合
│   │   └── web.js                  # 服务端渲染页面路由
│   ├── services/                   # 业务逻辑层
│   │   ├── linkService.js          # 链接索引、过滤、排序核心逻辑
│   │   ├── healthService.js        # 定时健康监测任务调度
│   │   └── importService.js        # 批量导入解析与校验
│   ├── utils/                      # 通用工具函数
│   │   ├── validator.js            # URL 格式与自定义字段校验
│   │   └── logger.js               # 日志记录与输出格式化
```

## 贡献指南

LinkForge 遵循开源社区协作规范，欢迎任何形式的贡献，包括但不限于新增功能、修复缺陷、完善文档或补充示例数据。请按照以下步骤提交贡献：

1. 在 GitHub 上 Fork 本仓库，并克隆到本地开发环境中。确保本地 Node.js 版本符合安装要求，执行 `npm install` 安装所有依赖。
2. 新建一个以 `feature/` 或 `fix/` 为前缀的分支，例如 `feature/add-import-export`。在该分支上进行代码修改，并确保代码风格符合项目 ESLint 配置。
3. 若新增或修改了后端 API 或前端组件，请同步更新对应的单元测试文件，并确保所有测试用例通过（执行 `npm run test`）。对于文档类改动，需在 `/docs` 目录下调整相应章节。
4. 提交代码时，使用清晰且语义化的 commit message，格式遵循 `type(scope): subject` 规范，例如 `feat(backend): add batch delete API`。
5. 推送分支到远程仓库后，通过 GitHub 界面发起 Pull Request，填写 PR 模板中的改动概述、测试情况及关联 issue。项目维护者会在 3 个工作日内进行 Review，并给出合并或修改意见。

## 常见问题

**Q：LinkForge 是否支持 HTTPS 访问？**  
A：LinkForge 本身不强制要求 HTTPS，但推荐在生产环境中通过反向代理（如 Nginx 或 Caddy）启用 TLS 终止，以保护传输中的数据安全。开发环境下默认使用 HTTP 协议，可通过配置环境变量 `USE_HTTPS=true` 并提供证书路径来启用原生 HTTPS 支持。

**Q：如何迁移或备份已收录的链接数据？**  
A：所有链接元数据均存储在项目根目录下的 `data/linkforge.db` SQLite 文件中。您可以直接复制该文件进行备份。若需迁移至其他服务器，同样只需携带该数据库文件，并在新环境中通过 `npm run migrate` 命令执行表结构升级（如有版本变更）。

**Q：健康监测任务是否会影响页面响应速度？**  
A：健康监测任务运行在独立的后台进程（`healthCheckRunner.js`）中，与主应用进程分离。监测周期默认配置为每 24 小时执行一次，且采用异步非阻塞 I/O，不会阻塞前端请求的处理。您可以在 `backend/services/healthService.js` 中调整监测超时时间与并发数。
