# EON Resource Gateway

EON Resource Gateway 是一个面向技术社区与开发者的外链资源汇总与导航系统。该项目定位于对分散于网络各处的高价值技术文档、工具页面、参考手册及社区讨论进行结构化收录与分类管理，解决开发者在信息检索过程中面临的海量链接分散、检索效率低下以及跨站点资源难以统一维护等实际问题。

本项目并非一个传统意义上的前端应用或后端框架，而是一套以资源目录为核心的轻量级信息枢纽。它适用于个人开发者、技术团队、开源社区运营者以及企业知识管理部门，帮助其以最低成本建立可维护、可扩展、可审计的外部参考资源池。通过约定式的链接收录规范与清晰的项目结构，EON Resource Gateway 可被快速部署为内部文档门户的补充模块，也可作为公开技术导航站的底层数据框架。

## 功能概览

结构化链接收录：以扁平化视图与分类标签方式收录全部外部资源链接，支持按来源域名、内容类型与收录批次进行筛选与定位。

静态资源导航：所有收录链接均以纯静态列表形式呈现，无需依赖后端数据库或动态页面渲染，降低维护成本与安全风险。

多维度分类索引：支持按文档类型、应用场景、技术领域、所属项目批次等多重维度对资源进行分组与展示，便于不同角色的使用者快速定位所需内容。

全量链接清单导出：提供完整的外部链接清单，支持一键复制与批量检查，便于运维人员进行可用性审计与失效链接修复。

零依赖核心引擎：项目核心不依赖任何第三方库或运行时环境，仅需标准 Markdown 阅读器与文本编辑器即可完成全部操作。

批次管理支持：内置批次标识（第 1103/1241 批）与资源编号追踪能力，方便大规模长期运营场景下的增量更新与回溯审计。

纯文本可读性设计：所有资源信息以人类可读的纯文本与表格形式组织，避免复杂的 JSON 结构或 YAML 配置，降低新手贡献门槛。

即克隆即用部署：从代码仓库克隆后无需构建或编译步骤，直接通过任意静态文件服务器即可提供服务，实现零配置上线。

## 应用场景

技术团队内部文档导航：研发团队可将 EON Resource Gateway 部署为内部知识库的入口页面，集中存放常用 API 文档、设计规范、运维手册及团队博客链接，替代浏览器书签栏的零散管理方式。新成员入职时只需访问该网关即可获得全部必要参考资源。

开源社区资源聚合站：开源项目维护者可使用本项目作为社区贡献指南的补充模块，收录外部教程、视频讲解、相关工具链以及生态依赖项目链接，帮助社区贡献者更快理解项目全貌与周边生态。

个人开发者知识库门户：独立开发者或技术博主可利用该项目整理自己长期积累的技术书签、学习笔记参考链接以及项目灵感来源，形成个人化的技术资产目录，并可通过版本控制工具同步多设备使用。

企业合规与审计资源索引：对于需要遵守外部标准或监管要求的企业团队，可使用 EON Resource Gateway 收录标准文档下载页、合规检查清单、审计工具官网以及行业报告原文链接，实现合规资源的集中受控管理。

技术培训与教学辅助材料导航：培训机构或高校讲师可将项目作为课程资源中心，收录实验环境入口、在线编译器、参考书籍官网以及往期项目示例链接，学员通过单一日志即可访问全部教学配套资源。

## 快速开始

以下步骤可帮助您在本地环境中快速启动 EON Resource Gateway 实例，完成从克隆到运行的全部流程。

```bash
# 克隆项目仓库至本地
git clone https://github.com/eon-resource-gateway/eon-gateway.git

# 进入项目根目录
cd eon-gateway

# 使用任意静态服务器启动服务（以 Python 内置模块为例）
python3 -m http.server 8080
```

启动成功后，在浏览器中访问 `http://localhost:8080` 即可查看资源导航主页。若您希望将项目部署至生产环境，可将整个目录上传至任何支持静态文件托管的 Web 服务器或对象存储服务中。

## 安装要求

EON Resource Gateway 对运行环境无特殊依赖，但建议在以下条件中运行以获得最佳体验。

| 依赖项 | 必需性 | 说明 |
|--------|--------|------|
| 现代 Web 浏览器（Chrome / Firefox / Edge 最新版） | 必需 | 用于渲染导航页面，支持标准 HTML5 与 CSS3 特性 |
| Python 3.6 及以上（或任意静态服务器） | 可选 | 仅用于本地快速预览，生产环境可替换为 Nginx / Apache 等 |
| Git 2.0 及以上 | 可选 | 用于克隆仓库，若通过压缩包下载则无需安装 |
| 文本编辑器（VS Code / Sublime / Vim 等） | 可选 | 用于查看或修改资源列表与文档内容 |
| Markdown 阅读器 | 推荐 | 用于本地阅读 README 及贡献指南等 Markdown 文件 |
| 网络连接 | 必需 | 用于访问资源列表中收录的外部链接，本地导航页本身无需联网 |
| 文件系统权限（读取与写入） | 必需 | 用于克隆仓库及后续更新资源列表文件 |
| 静态文件托管环境（如 GitHub Pages / Cloudflare Pages） | 可选 | 用于公开部署，非本地运行所必需 |
| 域名与 DNS 配置 | 可选 | 用于生产环境自定义访问地址，非本项目核心依赖 |

```
eon-gateway/
├── README.md                  # 项目总览与快速入门文档
├── CONTRIBUTING.md            # 贡献者操作指南与流程规范
├── LICENSE                    # MIT 许可证全文
├── index.html                 # 导航门户主页面（静态入口）
├── resources/                 # 核心资源目录
│   ├── index.md               # 资源总索引，含批次信息与统计
│   ├── 1103_batch/            # 第 1103 批次资源子目录
│   │   ├── manifest.md        # 本批次资源清单（含全部 URL）
│   │   └── tags.md            # 本批次资源分类标签定义
│   └── archive/               # 历史批次归档目录
│       └── 1102_batch/        # 上一批次资源备份
├── docs/                      # 项目文档与运维资料
│   ├── operations.md          # 日常运维操作手册
```

## 贡献指南

EON Resource Gateway 欢迎来自社区的各种形式的贡献，包括但不限于新增资源链接、更新失效地址、改进文档内容以及优化项目结构。请按照以下步骤参与贡献。

第一步：阅读项目文档与收录规范。在提交任何变更之前，请仔细阅读本 README 文件以及 docs/linking-policy.md 中的链接收录标准，确保新增或修改的资源符合项目定位与质量要求。

第二步：创建 Issue 讨论变更。对于新增资源链接或批量更新操作，建议先在 GitHub Issues 中创建一个类型为 resource_request 的工单，说明拟变更的内容、理由以及参考资料，以便维护者与其他贡献者进行讨论与评审。

第三步：克隆仓库并创建特性分支。在获得初步反馈后，请克隆项目仓库至本地，并基于 main 分支创建一个新的特性分支，分支命名建议采用 add-resource-{批次号} 或 fix-link-{描述} 的格式。

第四步：执行变更并本地验证。在特性分支中完成资源列表或文档的修改后，请运行 tools/check-links.sh 脚本对新增或修改的链接进行可用性检查，同时使用 tools/validate-urls.py 校验 URL 格式是否符合原样输出规则。确保所有变更不会破坏页面渲染或索引结构。

第五步：提交 Pull Request。将特性分支推送至远程仓库后，创建一个 Pull Request，并在描述中关联对应的 Issue 编号，详细说明变更内容、验证结果以及影响范围。维护者将在 3 个工作日内进行审核与合并。

## 常见问题

问：EON Resource Gateway 是否提供搜索或过滤功能？

答：当前版本以静态资源导航为核心，不内置动态搜索或过滤引擎。但所有资源链接均以结构化 Markdown 表格与分类小节形式组织，用户可通过浏览器页面查找（Ctrl+F 或 Command+F）快速定位关键词。未来版本计划基于静态站点生成器构建轻量级索引页，实现基础分类筛选能力。

问：如何批量更新资源列表中的链接？

答：建议使用 tools/generate-index.py 脚本辅助生成新批次的资源清单。该脚本接受一个纯文本 URL 列表作为输入，自动按照约定格式生成对应的 Markdown 索引块。对于失效链接的批量替换，可结合 tools/check-links.sh 的输出结果进行筛选与编辑，随后提交 Pull Request 进行合并。

问：本项目是否支持多语言文档？

答：当前版本仅提供中文文档与界面。但项目本身为纯静态 Markdown 与 HTML 结构，不涉及硬编码语言逻辑。社区贡献者可自由创建其他语言版本的 README 与导航页面，通过独立的文件后缀（如 README.zh-CN.md、README.en-US.md）进行维护。项目维护团队将在未来考虑引入多语言切换支持。
