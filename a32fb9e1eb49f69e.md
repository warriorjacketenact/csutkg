# LinkVault Core

LinkVault Core 是一个面向技术团队与独立开发者的外链资源聚合与管理工具。它并非简单的书签管理器，而是一个具备元数据索引、批量导入、全文检索以及健康状态检测能力的轻量级资源中台。项目定位服务于需要维护大量外部技术文档、教程、API 参考或社区讨论链接的维护者，帮助其从无序的收藏行为转向结构化的资源治理。

本项目的核心目标用户包括开源文档编写者、技术社区运营、DevOps 工程师以及知识库管理员。通过统一的外链录入规范与可扩展的解析引擎，LinkVault Core 能够将散落在各处的零散 URL 转化为可查询、可分类、可监控的资产，有效降低链接失效与内容漂移带来的维护成本。

## 功能概览

**批量链接导入** 支持通过文本文件、CSV 或直接粘贴方式一次性导入大量 URL，自动去重并识别已有条目。

**元数据自动抓取** 对每条链接发起轻量级 HEAD 与 GET 请求，提取页面标题、描述、内容类型及最后修改时间，填充至索引字段。

**全文检索与过滤** 基于倒排索引实现标题、描述及自定义标签的快速检索，支持按域名、文件类型、状态码等多维度过滤。

**健康状态监控** 周期性对已存储链接发起可用性检查，记录响应时间与 HTTP 状态码，对失效或重定向链接发出标记。

**标签与分类体系** 允许用户自定义多级标签树，支持一条链接关联多个标签，便于按主题或项目维度组织资源。

**导入与导出兼容** 提供 JSON、YAML 及 Markdown 列表格式的导出功能，便于与静态站点生成器或其他文档工具链集成。

## 应用场景

**技术文档站的外链资产管理** 文档团队在编写教程或 API 参考时，往往需要引用大量外部资源。LinkVault Core 可集中管理这些引用链接，在构建前自动检查有效性，避免文档中出现死链影响读者体验。

**开源项目 README 与 Wiki 维护** 开源项目维护者通常在 README 中列出相关工具、生态项目或学习资料。使用 LinkVault Core 可以批量维护这些外链，定期同步更新状态，并在版本发布时快速导出为格式规范的 Markdown 列表。

**个人技术博客的参考链接库** 技术博主在撰写多篇文章时会累积大量参考资料。通过 LinkVault Core 统一归档，配合标签系统按主题分类，可在写作新文章时快速检索相关背景材料，提升内容产出效率。

**社区资源导航站的构建** 技术社区或学习小组常需维护一份外部资源导航页。LinkVault Core 可以作为后台数据管理端，维护链接元数据与分类，并通过导出功能生成静态导航站点的源文件，降低手动更新成本。

## 快速开始

以下操作以 Linux / macOS 环境为例，Windows 用户可使用 WSL 或 Git Bash 执行。

```bash
# 克隆仓库
git clone https://github.com/your-org/linkvault-core.git
cd linkvault-core

# 安装依赖（使用 Python 3.10+ 与 pip）
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 初始化数据库并启动开发服务
python manage.py migrate
python manage.py runserver 0.0.0.0:8000
```

服务启动后，访问 http://localhost:8000 进入 Web 管理界面。首次使用请通过命令行创建管理员账户：

```bash
python manage.py createsuperuser
```

## 安装要求

| 依赖 | 必需 | 说明 |
|---|---|---|
| Python 3.10 或更高版本 | 是 | 核心运行时环境，建议使用 3.11 以上以获得性能优化 |
| PostgreSQL 14+ | 是 | 生产环境推荐，用于存储链接元数据与索引，支持 JSONB 字段 |
| Redis 7+ | 否 | 用于缓存与周期性任务队列，不配置则降级为同步模式 |
| Node.js 18+ | 否 | 仅当启用前端资产构建或使用预渲染功能时需要 |
| Nginx 1.24+ | 否 | 生产环境反向代理与静态文件服务推荐，非必需 |
| sqlite3（开发）/ psycopg2（生产） | 是 | Python 数据库驱动，开发默认 sqlite3，生产需安装 libpq-dev |
| requests >= 2.31.0 | 是 | HTTP 请求库，用于元数据抓取与健康检查 |
| beautifulsoup4 >= 4.12.0 | 是 | HTML 解析，用于提取页面标题与描述 |
| django >= 4.2.0 | 是 | Web 框架，提供 ORM、管理界面与命令行工具 |
| django-celery-beat >= 2.6.0 | 否 | 如果启用 Redis 定时任务，用于周期性监控调度 |
| pytest >= 8.0.0 | 否 | 仅测试环境需要，用于运行单元测试与集成测试 |

## 贡献指南

本项目的贡献流程遵循标准的 Fork-Pull Request 模型，所有代码变更需通过持续集成检查。具体步骤如下：

1. 在 GitHub 上 Fork 本仓库，并克隆至本地开发环境。请确保你的开发分支基于最新的 main 分支创建，命名格式建议为 feature/描述或 fix/描述。

2. 完成代码或文档变更后，运行完整的测试套件以确保无回归问题。对于新增功能，需补充对应的单元测试与文档说明。

3. 提交 commit 时请遵循 Conventional Commits 规范，使用 feat、fix、docs、refactor 等类型前缀，并附上简要的变更说明。

4. 推送至你的远程仓库后，通过 GitHub 界面发起 Pull Request。PR 描述中需明确说明解决的问题、变更范围以及测试覆盖情况。

5. 项目维护者将在 3 个工作日内进行 Review。如有修改意见，请及时更新 PR 分支。通过后由维护者合并至主分支。

## 常见问题

**问：导入大量链接时，元数据抓取会阻塞界面吗？**

答：默认情况下，批量导入接口采用异步任务模式。当您提交一批链接后，系统会立即返回任务 ID，后续的 HTTP 请求与页面解析均在后台队列中执行。您可以通过任务状态接口查询进度，也可在管理界面中查看实时处理日志。若未配置 Redis 与 Celery，系统会降级为同步模式，此时建议单次导入不超过 100 条。

**问：健康状态监控对目标服务器会造成压力吗？**

答：健康检查默认使用 HEAD 请求，仅获取响应头信息，不下载完整页面内容。检查间隔默认为 7 天，且每次检查会随机抖动 ±12 小时以避免流量尖峰。对于大规模实例（超过 5000 条链接），建议调整并发数，并在配置文件中设置请求超时与重试上限。

**问：如何迁移现有书签或收藏夹数据？**

答：LinkVault Core 内置了导入适配器，支持常见书签导出的 HTML 格式（如 Chrome、Firefox 的收藏夹文件），也支持 CSV 列映射（需包含 URL 与标题字段）。您可以通过管理界面的导入向导完成映射配置，或使用命令行脚本批量处理，具体操作请参考 docs/workflows/migration.md 中的示例。
