# ResourceHub

ResourceHub 是一个面向开发者与技术研究人员的结构化外链资源汇总平台。项目定位为高质量技术资源导航与检索系统，专注于收集、分类和索引互联网中的优质技术文档、工具站点、数据源与学术资源。ResourceHub 不生成原创内容，而是通过人工审核与自动化校验相结合的方式，为特定技术领域的从业者提供高信噪比的资源发现路径。

本项目面向系统架构师、运维工程师、后端开发者、机器学习工程师以及技术决策者。ResourceHub 解决的核心痛点是技术信息过载与资源分散问题，通过统一的数据模型、标签体系与版本追踪机制，帮助用户在特定技术栈或业务场景下快速定位到经过验证的外部资源，显著降低信息筛选成本。

## 功能概览

- **多维度资源分类**：支持按技术领域、应用场景、资源类型、成熟度等级四个正交维度对每条外链进行标记，允许用户通过组合筛选快速缩小范围。

- **资源健康度监控**：内置周期性链接可用性检查机制，对返回 HTTP 4xx/5xx 状态码、SSL 证书过期或 DNS 解析失败的资源自动标记异常状态，并提供历史可用性趋势视图。

- **标签化检索系统**：每条资源支持最多 15 个自定义标签，标签体系采用层级结构（如 "database/relational/postgresql"），支持前缀模糊匹配与标签交集检索。

- **版本变更追踪**：对资源的目标 URL、标题、描述、分类等元数据的每次修改均记录变更日志，支持按时间轴回溯任意历史版本状态。

- **使用统计与热度排序**：记录每个资源的点击次数、最近访问时间、收藏数等行为数据，支持按综合热度、新增时间、名称字母序等多种排序策略。

- **结构化元数据模板**：针对常见资源类型（如官方文档、API 参考、论文预印本、视频教程、交互式实验室）预置不同的元数据模板，强制填写必要字段以保证信息完整性。

- **开放数据导出**：支持将资源列表以 JSON、YAML 或 Markdown 表格形式导出，便于集成到其他文档系统或静态站点生成器中。

- **用户自定义集合**：允许注册用户创建私有或公开的资源集合，支持集合内资源的备注标注与集合级别的标签继承。

## 应用场景

- **技术选型调研**：当团队需要评估多个中间件或框架时，可以使用 ResourceHub 的分类与标签功能快速聚合特定领域（如消息队列、API 网关、ORM 框架）的所有相关资源，包括官方站点、性能对比报告、社区案例与安全公告，大幅缩短调研周期。

- **项目文档库构建**：技术负责人可以为新项目建立专属资源集合，将涉及的所有外部依赖文档、运维手册、监控面板链接、日志查询入口统一收录，并为每条资源添加内部备注说明（如 "生产环境使用版本 v2.3.1"），使团队知识库保持同步更新。

- **离线环境资源准备**：在网络隔离或受限环境中工作的人员，可以通过 ResourceHub 的批量导出功能预先获取所有必需资源的 URL 列表与元数据，结合自动化下载脚本完成离线包准备，避免因遗漏依赖导致部署中断。

- **技术社区内容推荐**：技术博客作者或社区运营人员可使用 ResourceHub 的标签检索与热度排序功能发现近期新增或高关注度的资源，作为每周技术周刊、推荐阅读列表或 FAQ 引用的素材来源，保证推荐内容的新鲜度与权威性。

- **持续集成流水线依赖检查**：在 CI/CD 流水线中集成 ResourceHub 的 API 接口，对项目配置文件中声明的外部资源 URL 进行可用性预检，若检测到资源不可用则提前告警，避免构建或部署阶段因外部资源访问失败而中断。

## 快速开始

以下步骤帮助您在本地环境中快速启动 ResourceHub 服务。

```bash
# 克隆代码仓库
git clone https://github.com/resourcehub/resourcehub.git
cd resourcehub

# 安装项目依赖（使用 pip 和虚拟环境）
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 执行数据库初始化与种子数据加载
python manage.py migrate
python manage.py loaddata initial_resources.json

# 启动开发服务器
python manage.py runserver --host 0.0.0.0 --port 8080
```

访问 http://localhost:8080 即可进入 ResourceHub 首页。默认管理员账号为 admin，密码在首次启动时输出至控制台日志中。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.10 或更高 | 核心运行环境，推荐使用 3.11 以获取性能改进 |
| PostgreSQL | 14.0 或更高 | 主数据存储库，使用 JSONB 字段存储元数据与标签 |
| Redis | 6.2 或更高 | 用于缓存热点资源列表与统计计数器，可选但强烈推荐 |
| Node.js | 18.0 或更高 | 仅用于前端静态资源构建，生产环境可预编译后移除 |
| Nginx | 1.22 或更高 | 生产环境反向代理与静态文件服务，开发环境可使用内置服务器替代 |
| Elasticsearch | 8.5 或更高 | 可选组件，用于支持全文检索与标签模糊匹配的高级搜索 |
| Prometheus | 2.40 或更高 | 可选组件，用于采集资源健康度检查指标与系统运行状态 |
| Docker | 20.10 或更高 | 容器化部署方案的支持组件，非强制但推荐生产环境使用 |
| Git | 2.30 或更高 | 用于版本控制与自动化部署脚本中的仓库更新操作 |

## 项目结构

```
resourcehub/
├── api/                                  # RESTful API 模块，处理资源 CRUD 与检索
│   ├── endpoints/                        # 各功能路由端点实现
│   │   ├── resources.py                  # 资源列表、详情、创建、更新、删除操作
│   │   ├── collections.py                # 用户集合管理及集合内资源关联
│   │   ├── tags.py                       # 标签层级管理与检索接口
│   │   └── health.py                     # 资源健康度状态查询与报告
│   ├── serializers/                      # 请求/响应数据序列化与校验
│   │   ├── resource_schema.py            # 资源元数据模板验证器
│   │   └── collection_schema.py          # 集合数据结构序列化
│   └── middleware/                       # 认证、限流、日志中间件
│       ├── auth.py                       # JWT 令牌验证与角色权限控制
│       └── ratelimit.py                  # 基于 Redis 的滑动窗口限流
├── core/                                 # 核心业务逻辑层
│   ├── models/                           # 数据模型定义（SQLAlchemy ORM）
│   │   ├── resource.py                   # 资源主表、元数据 JSONB、变更日志
│   │   ├── collection.py                 # 集合与资源关联表
│   │   └── tag.py                        # 标签树与资源标签映射表
│   ├── services/                         # 业务服务实现
```

## 贡献指南

ResourceHub 欢迎并鼓励社区贡献，包括新增资源推荐、分类体系优化、代码缺陷修复和文档改进。请按照以下流程参与项目：

1.  **提交资源推荐**：在提交新资源前，请先查阅现有分类与标签体系，确保新资源不与已有记录重复。推荐时需提供资源标题、目标 URL、所属分类、至少三个标签以及 50-200 字的摘要说明。资源推荐通过 GitHub Issue 提交，并使用 "resource-request" 标签。审核团队将在 5 个工作日内评估并反馈是否收录。

2.  **代码贡献流程**：所有代码变更必须通过 Pull Request 提交。请先将主仓库 fork 至个人账户，在功能分支上开发，并确保所有单元测试通过且新增代码的测试覆盖率不低于 80%。PR 标题格式为 "[类型] 简要变更描述"，类型包括 feat、fix、docs、refactor、perf。PR 描述中需关联相关 Issue 编号，并附上变更动机与影响范围的说明。

3.  **分类与标签体系维护**：若发现现有分类或标签定义模糊、重叠或缺失，请提交详细的改进提案。提案需包含当前问题分析、建议的新结构、迁移路径以及对现有资源的影响评估。此类提案应使用 "taxonomy" 标签提交 Issue，经维护者讨论达成共识后实施变更。

4.  **文档改进与翻译**：文档使用 Markdown 编写，位于 docs/ 目录下。若发现文档错误、歧义或缺失部分，可直接提交 PR 修正。欢迎提供非中文语言的翻译版本，但需在主文档中保留原始中文版本作为权威参考，翻译版本置于独立的语言子目录中。

5.  **问题报告与讨论**：使用 GitHub Issues 报告 Bug 或提出功能请求。报告 Bug 时请提供完整的复现步骤、预期行为与实际行为、环境信息（操作系统、Python 版本、浏览器版本）以及相关的日志或截图。功能请求请说明使用场景与期望的接口行为，以便评估实现优先级。

## 常见问题

**Q: ResourceHub 的资源更新频率是怎样的？如何保证链接不过期？**

A: 系统默认每 24 小时对所有已收录资源执行一次可用性检查，检查包括 HTTP 状态码、响应时间、SSL 证书有效期和 DNS 解析结果。检查失败的资源会在前端界面中标记为 "异常" 状态，并记录失败原因。管理员会每周审核异常资源列表，对于连续 7 天不可用的资源将其移至归档状态，同时尝试通过搜索引擎查找新的有效 URL 进行替换。用户也可通过 "报告链接失效" 按钮主动反馈问题。

**Q: 我可以将 ResourceHub 部署在完全离线的内网环境中吗？**

A: 可以。ResourceHub 的核心功能（资源展示、分类检索、集合管理、健康度检查）均不依赖外部互联网访问。唯一需要离线环境特别处理的是资源元数据自动抓取服务（crawler），该服务在离线部署时应禁用或替换为手动录入模式。您需要在部署前将初始资源数据通过种子文件（JSON 格式）导入数据库，并配置内网可访问的 PostgreSQL 与 Redis 实例即可完整运行。

**Q: 如何将我现有的书签或收藏夹批量导入到 ResourceHub 中？**

A: 项目提供了一组导入脚本，位于 scripts/importers/ 目录下，目前支持从 Chrome 书签 HTML 导出文件、Firefox JSON 备份文件以及 Pinboard API 格式的数据导入。您需要将书签文件放置在指定目录下，然后运行 `python scripts/importers/import_bookmarks.py --source chrome --file bookmarks.html` 命令执行导入。导入过程中脚本会尝试自动补全缺失的元数据字段，并标记需要人工审核的条目。对于其他格式的书签数据，您可参考现有导入器实现编写自定义转换脚本。
