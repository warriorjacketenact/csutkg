# Assisted Transition Resource Catalog

Assisted Transition Resource Catalog (ATRC) 是一个面向技术研究者、内容索引工程师和信息分析从业者的结构化外链资源汇总系统。该项目并非简单的书签集合，而是一套带有版本控制、分类标签和元数据追踪的开放式资源导航工具，旨在解决分散信息源难以批量归档、检索和共享的痛点。

ATRC 定位于中大型技术团队的知识库基础设施。当团队需要维护一份跨项目、跨领域的可信外部链接清单，并希望该清单能够像代码一样接受 PR 审查、变更记录追踪和自动化校验时，ATRC 提供了一套开箱即用的目录规范与脚手架工具。项目本身不存储任何实际内容，仅保留 URL 的规范化元数据，所有资源指向均为原始第三方服务。

## 功能概览

**结构化资源登记**：基于 YAML 前置数据块对每条 URL 进行上下文标注，包括但不限于资源类型、语种、所属垂直领域及收录日期，便于后续自动化分类统计。

**批量链接健康检查**：集成定时任务与 GitHub Actions 工作流，周期性探测每条外链的 HTTP 状态码及响应时长，自动标记失效或重定向链接并生成报告。

**多维度标签体系**：支持一级分类（如技术文档、视频教程、工具库、行业报告）与二级自由标签的组合，允许用户通过标签表达式进行组合筛选。

**Markdown 原生渲染**：所有资源列表和元数据直接以 Markdown 表格和列表形式呈现，与 GitHub、GitLab 等主流代码托管平台的渲染引擎完全兼容，无需额外前端服务。

**变更历史审计**：每次增删改操作均通过 Pull Request 流程完成，合并后自动更新 CHANGELOG 文件，确保资源池的每一次变动均有记录可查。

**插件化导出适配器**：内置 JSON、CSV、HTML 书签文件三种导出格式，并预留 WebHook 接口，可对接企业内部知识中台或第三方笔记软件。

## 应用场景

**技术文档团队维护外部参考库**：当编写系统设计文档或 API 说明时，需要频繁引用外部标准规范、官方 SDK 仓库或社区最佳实践文章。ATRC 可作为单一可信源，统一管理这些引用链接，避免文档中出现重复、过期或拼写错误的 URL。

**数据标注与内容审核流水线**：对于需要人工复核的视频片段、文章详情页或产品快照，审核人员可通过 ATRC 生成的清单快速跳转至目标页面，同时利用元数据字段记录审核结论，形成闭环的质量管理台账。

**开源社区贡献者入门导航**：大型开源项目通常依赖大量第三方依赖库、学习资料和讨论区。ATRC 可将这些资源按角色（新人、贡献者、维护者）分类，降低社区成员的认知负担，提升协作效率。

**离线归档与迁移前评估**：在进行系统重构或数据迁移之前，运维团队可利用 ATRC 的全量链接清单，预先分析外部依赖的可用性与协议兼容性，制定灰度切换或缓存降级策略。

## 快速开始

以下指令适用于 Linux/macOS 环境，Windows 用户建议通过 WSL2 或 Git Bash 执行。

```bash
# 克隆项目仓库至本地
git clone https://github.com/assisted-transition/atrc.git
cd atrc

# 安装项目依赖（需要 Node.js 18+ 与 npm 9+）
npm install

# 运行资源目录构建脚本，生成最新的资源列表文档
npm run build
```

执行完上述三步后，`/dist` 目录下将生成包含全部资源分类汇总的 `catalog.md` 文件，以及用于健康检查的 `links.json` 元数据缓存。

## 贡献指南

本项目采用 GitHub Flow 协作模型，所有资源变更均通过 Pull Request 合并至 main 分支。

1.  **Fork 仓库并创建功能分支**：从 main 分支创建以 `add-`、`update-` 或 `remove-` 为前缀的分支名，例如 `add-2026-q2-links`。确保分支名称简洁描述变更意图。

2.  **编辑 YAML 源文件**：在 `/src/links/` 目录下找到对应批次的 YAML 文件，按 JSON Schema 定义添加或修改资源条目。每个条目必须包含 `id`、`url`、`title`、`category` 和 `tags` 字段，并确保 `url` 字段与资源列表中的原始链接完全一致。

3.  **本地验证**：在提交之前，运行 `npm run validate` 检查元数据格式是否符合规范，运行 `npm run build` 确认构建流程能够正常生成 `catalog.md`。如果验证失败，请根据终端提示修正数据。

4.  **提交变更并推送**：提交信息请遵循约定式提交格式，如 `feat: add 659 batch resources` 或 `fix: correct url for id 14876`。推送至你的远程分支。

5.  **发起 Pull Request**：向本仓库的 main 分支发起 PR，在 PR 描述中引用本次变更对应的批次号（第 659/1241 批）并简要说明变更原因。至少需要一位维护者审核通过后方可合并。

## 常见问题

**Q: 资源列表中的链接访问时返回 404，我应该如何处理？**

A: 请先在目标网站确认该资源是否已被迁移或下架。若确认失效，请在 `/src/links/` 对应批次文件中将该条目的 `status` 字段修改为 `deprecated`，并在 `notes` 字段中记录失效日期。若资源仅为 URL 结构变化，请更新 `url` 字段为新地址，并在 `notes` 中注明重定向原因。

**Q: 如何查找某一特定资源是否已经收录？**

A: 你可以使用 `npm run search -- --keyword="关键词"` 命令在本地元数据缓存中进行全文检索，该命令会匹配 `title`、`tags` 和 `notes` 字段。另外，你也可以直接查看 `/dist/catalog.md` 文件，利用编辑器的页面内查找功能进行定位。

**Q: 我是否可以提交非技术类的资源链接？**

A: 是的。ATRC 对资源类型不作技术限制，但要求每个条目必须填写 `category` 一级分类（可选值包括 `documentation`、`video`、`tool`、`report`、`community`、`other`）。如果资源涉及敏感或成人内容，请在 `tags` 中明确标注 `nsfw`，以便生成时进行过滤或标识。
