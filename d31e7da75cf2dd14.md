# LinkVault Core

LinkVault Core 是一个面向开发者与技术研究团队的外链资源聚合与管理平台，专注于对海量外部技术链接进行结构化收录、分类索引与状态监控。项目定位于中大型技术团队的知识资产沉淀场景，解决因外链散落、失效、不可追溯而导致的技术调研成本升高问题。通过标准化采集流程与自动化校验机制，LinkVault Core 帮助用户将分散的外部资源转化为可检索、可追踪、可审计的内部知识库。

## 功能概览

- **批量链接入库**：支持一次性导入数以千计的原始 URL，自动完成协议解析与去重，生成唯一资源标识。
- **多维度标签分类**：允许用户为每个链接自定义标签体系，支持按项目、领域、来源、优先级等维度进行精细化管理。
- **可用性主动探测**：内置分布式探测引擎，定时对已收录链接发起 HEAD/GET 请求，自动标记异常状态并生成告警通知。
- **元数据智能补全**：调用公共解析服务，自动提取页面标题、描述、关键词及最后修改时间，减少人工录入负担。
- **资源变更追踪**：记录每次探测返回的响应码、响应时长与内容摘要哈希，形成资源变更历史，便于回溯分析。
- **权限与审计日志**：提供基于角色的访问控制（RBAC），所有查询与修改操作均记录审计日志，满足企业内部合规要求。
- **开放 API 接口**：提供 RESTful API，支持第三方工具或脚本批量查询资源状态、拉取分类清单，便于集成至现有工作流。
- **数据导出与报告**：支持将资源列表及探测结果导出为 CSV、JSON 或 Markdown 格式，方便生成周报或迁移至其他系统。

## 应用场景

1. **技术调研与竞品分析**：技术选型或竞品分析过程中，调研人员需频繁查阅大量外部文档、博客、开源仓库及官方公告。LinkVault Core 可将这些分散链接统一收录，并自动监控页面变动，确保调研依据始终指向最新有效内容。

2. **项目文档外链管理**：大型项目通常有多份设计文档、API 参考和运维手册，其中引用大量外部依赖地址。使用 LinkVault Core 集中管理这些外链，可在依赖地址变更或失效时快速定位影响范围，避免文档中的引用变成“死链”。

3. **安全合规资源巡检**：安全团队需要定期检查各类威胁情报源、漏洞数据库及补丁公告页面的可访问性与内容更新情况。系统自动化的探测与变更追踪能力可显著降低人工巡检工作量，并生成可追溯的检查记录。

4. **知识库自动化构建**：企业知识管理团队可将分散在各类临时笔记、邮件、即时通讯中的技术参考链接导入 LinkVault Core，结合元数据补全与分类功能，快速搭建起结构清晰、可持续维护的技术资源知识库。

## 快速开始

以下步骤帮助您在本地环境中快速启动 LinkVault Core 服务。

```bash
# 克隆项目仓库
git clone https://github.com/linkvault/core.git linkvault-core
cd linkvault-core

# 安装项目依赖（使用 pip 与虚拟环境）
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 初始化数据库并启动开发服务器
python manage.py migrate
python manage.py init_admin --username admin --password your_secure_password
python manage.py runserver --host 0.0.0.0 --port 8080
```

启动成功后，访问 `http://localhost:8080` 即可进入管理控制台，使用初始化的管理员账户登录。

| Nginx | 1.20+ | 生产环境反向代理与静态文件服务，可选但推荐部署 |

## 项目结构

项目采用分层架构，各模块职责清晰，便于扩展与维护。

```
linkvault-core/
├── cmd/                                # 入口命令行工具
│   ├── server/                         # 主服务启动入口
│   │   └── main.go                     # 初始化路由、数据库与中间件
│   └── worker/                         # Celery 工作进程入口
│       └── main.py                     # 启动探测与补全任务消费者
├── internal/                           # 内部核心逻辑（不对外暴露）
│   ├── crawler/                        # 链接探测与元数据抓取模块
│   │   ├── checker.py                  # 可用性探测核心逻辑
│   │   └── parser.py                   # 页面标题、描述等元数据解析
│   ├── storage/                        # 数据库访问层
│   │   ├── models.py                   # SQLAlchemy 实体定义
│   │   └── repository.py               # CRUD 与复杂查询封装
│   ├── scheduler/                      # 定时任务调度
│   │   └── tasks.py                    # Celery 任务定义（探测、补全、清理）
│   └── auth/                           # 认证与权限控制
│       ├── rbac.py                     # 角色与权限矩阵
│       └── jwt_handler.py              # JWT 签发与验证
├── api/                                # 对外 RESTful API 定义
│   ├── routes/                         # 路由分组
│   │   ├── v1/                         # API 版本 v1
│   │   │   ├── links.py                # 链接资源接口
│   │   │   └── reports.py              # 报告与统计接口
│   └── middleware/                     # 请求中间件（日志、限流、鉴权）
│       └── auth_middleware.go
├── web/                                # Web 管理控制台前端
│   ├── static/                         # 静态资源（CSS、JS、图片）
│   └── templates/                      # Jinja2 页面模板
│       ├── dashboard.html              # 总览面板
```

## 常见问题

**问：系统能承受多大的链接管理规模？**

系统设计上支持管理百万级链接资源。实际承载能力受限于部署环境配置，尤其是数据库与 Redis 内存。生产环境下建议为 PostgreSQL 配置至少 4GB 内存，并启用 Redis 持久化。对于超过 50 万链接的场景，推荐使用分区表并按创建时间分库。

**问：探测任务是否会影响目标站点的正常访问？**

系统默认采用保守策略，探测请求间隔至少为 60 秒，且同一域名下的并发请求数限制为 2 个。请求头模拟主流浏览器 User-Agent，不携带任何恶意参数。若需调整探测频率或并发数，可在 Celery 配置中修改 `CHECK_INTERVAL` 与 `CONCURRENT_PER_DOMAIN` 变量。

**问：如何迁移或备份已收录的链接数据？**

系统内置数据导出命令 `python manage.py export_links --format json --output backup.json`，可导出所有链接元数据及最新探测结果。导入时使用 `python manage.py import_links --file backup.json` 即可完成恢复。跨版本迁移前，建议先执行数据库 schema 比对工具确保兼容性。
