# ResourceWatch 技术资源索引系统

ResourceWatch 是一个面向技术社区的开源外链资源聚合与导航系统，专为需要持续追踪大量外部技术文档、工具站点、API 参考与代码示例的开发者与团队设计。本项目并非一个传统的爬虫或采集器，而是一个基于人工精选与社区贡献的轻量级资源目录框架，提供标准化的资源录入模板、标签体系与版本追踪能力。

项目定位为技术团队内部的知识中台辅助组件，亦可作为个人开发者的书签管理替代方案。通过结构化的元数据描述与统一的访问入口，ResourceWatch 帮助用户从海量零散链接中构建可检索、可审计、可协作的外部知识图谱。项目默认收录的链接均经过初步分类，用户可根据自身需求自由增删改查，并利用内置的静态站点生成能力将资源列表渲染为可公开访问的文档门户。

## 功能概览

**资源条目模板化录入** 提供基于 YAML 的前置元数据定义，每条资源可附带标题、描述、标签、收录日期与维护状态，便于长期管理。

**多级标签分类体系** 支持无限层级的标签嵌套，允许用户按技术领域、难易度、适用场景或团队项目名称对资源进行多维度标记。

**链接可用性周期检测** 集成基于 HTTP 状态码的周期性检查脚本，可输出失效链接报告，辅助用户及时清理或更新条目。

**静态文档门户生成** 内置基于 Markdown 的渲染流水线，可将全部资源列表与分类目录输出为纯静态 HTML 页面，适合部署于 GitHub Pages 或内部服务器。

**全文检索与快速过滤** 借助客户端搜索引擎 lunr.js 或 minisearch，为生成的文档站点提供标题与描述的实时搜索能力。

**导入导出兼容性** 资源数据支持导出为 CSV、JSON 与 OPML 格式，便于与其他书签工具或知识管理软件进行数据交换。

**社区协作审核机制** 提供基于 Pull Request 的资源新增与变更流程，每条资源的增删改均需经过至少一名维护者审核，保证条目质量。

**版本化的资源变更日志** 每次对资源列表的修改都会生成变更记录，支持按时间回溯查看资源的添加、移除与元数据更新历史。

## 应用场景

技术团队内部知识库建设：开发团队可将 ResourceWatch 作为技术选型参考库的载体，集中存放各类中间件官网、性能对比报告、安全公告与迁移指南链接，新成员入职时可快速了解团队所依赖的外部生态。

个人开发者学习路径管理：正在系统学习某一技术栈的开发者可按阶段标记资源，例如将官方教程标记为 primary，将进阶博客标记为 advanced，将视频课程标记为 video，形成可量化的学习进度追踪基线。

技术峰会与沙龙资料归档：参会者可在活动结束后将演讲幻灯片链接、官方代码仓库地址、相关论文预印本与新闻稿统一录入 ResourceWatch，并为每条资源附加会议名称与时间戳，构建可回溯的事件档案。

离线文档镜像辅助索引：对于已将部分核心文档内网镜像的团队，可使用 ResourceWatch 记录内网地址与官方源的对应关系，并在官方源更新时通过检测脚本获得同步提醒。

## 快速开始

以下指令适用于 Linux 与 macOS 环境，Windows 用户建议通过 WSL2 或 Git Bash 执行。

```bash
# 克隆项目仓库至本地
git clone https://github.com/resourcewatch/resourcewatch.git
cd resourcewatch

# 安装项目依赖（基于 Node.js 22 LTS 与 npm 10）
npm install

# 构建初始资源索引并启动本地预览服务器
npm run build
npm run serve
```

执行上述步骤后，访问命令行输出中提示的本地地址（默认为 http://localhost:8787）即可查看示例资源门户。如需立即导入用户提供的链接清单，请将项目根目录下的 data/sources.yaml 替换为包含所有 URL 的合法 YAML 文件，或通过 `npm run import -- --format=urls --source=urllist.txt` 命令批量导入。

## 常见问题

问：导入大量链接后，静态站点生成速度明显下降，应当如何优化？

答：建议将 sources.yaml 按照标签拆分为多个文件，并通过 config/default.yaml 中的 sources 数组按需加载。同时可启用 `build --optimize` 选项，该选项会压缩生成的 HTML 与 JavaScript 产物，并移除未使用的 CSS 规则。对于超过 5000 条资源的用例，推荐配合 CI 流程仅在资源变更时触发完整构建，日常预览可使用增量构建模式。

问：链接可用性检测脚本误报率为多少？如何处理被误判为失效的链接？

答：检测脚本默认使用 HEAD 请求并遵循重定向，超时时间设为 5000 毫秒。对于响应较慢但内容正常的服务，可在 config/default.yaml 的 healthCheck 字段中调整 timeout 与 retry 参数。若某链接始终被标记为失效但浏览器访问正常，请为该条目添加 `health: skip` 标记以跳过自动检测，并在描述中注明原因。

问：能否将 ResourceWatch 部署为团队内部的私有服务，并限制访问权限？

答：项目核心功能为静态站点生成，并不包含服务端鉴权逻辑。若需权限控制，建议将生成的静态文件部署至支持基础认证的 Web 服务器（如 Nginx 的 auth_basic 模块），或使用 Cloudflare Access 等零信任访问方案。资源数据本身仍以 YAML 形式存储在仓库中，请根据团队策略决定仓库的可见性。
