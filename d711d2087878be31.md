# VideoLink 聚合导航系统

VideoLink 是一个面向技术内容创作者、在线教育平台运营者及数字资源管理者的高性能外链聚合与导航系统。该项目并非视频托管平台，而是专注于对分散于各处的视频资源链接进行统一收录、分类索引与状态监控的中台系统。其核心目标在于解决因链接分散导致的资源遗忘、失效不可知以及跨平台检索效率低下等问题，帮助用户构建个人或团队级别的可维护、可扩展的媒体资源知识库。

## 功能概览

**批量链接导入与自动解析** 系统支持通过文本文件或API接口批量导入链接，并自动请求目标地址以获取页面标题、描述及关键元数据，减少人工录入成本。

**多维度分类与标签管理** 用户可为每条记录分配自定义标签及二级分类，支持基于分类树与标签组合的复杂筛选逻辑，满足精细化资源管理需求。

**链接健康状态主动侦测** 内置异步任务队列，按可配置周期对所有存储链接发起HEAD请求，实时标记状态码异常（如404、502）或响应超时的记录，并生成可视化健康报表。

**全文检索引擎** 基于倒排索引技术对链接标题、描述及自定义备注建立搜索索引，支持模糊匹配与短语精确查询，显著提升海量记录下的定位速度。

**数据快照与版本回溯** 每日凌晨自动生成全量数据快照，用户可手动触发快照创建，并在系统设置中浏览历史快照列表，支持一键回滚至任意历史状态。

**权限分级与协作支持** 内置基于角色的访问控制模型，区分管理员、编辑者与访客三种角色，允许团队协作编辑资源库的同时保护核心配置不被误操作。

## 应用场景

**个人技术博主的资源仓库** 独立技术博主在撰写教程或制作视频时，需引用大量外部参考链接。VideoLink可作为私有外链数据库，按主题（如Go语言并发、React性能优化）分类存储，在输出内容时可快速检索并生成引用列表，避免重复搜索。

**在线教育机构的课程素材管理** 教育机构教务人员可将分散在不同视频平台（如B站、YouTube、Vimeo）的课程相关视频链接统一录入系统，通过标签关联至具体课程编号。讲师在备课阶段可直接通过课程编号筛选出全部辅助素材，无需逐个平台手动查找。

**数字营销团队的活动资源追踪** 营销团队在策划线上活动时，需收集竞品案例、行业报告解读视频及产品演示录像。系统可对上述链接进行状态监控，当某案例视频因版权下架时自动发出告警，便于团队及时替换活动方案中的引用材料。

**开源项目文档中的参考链接索引** 大型开源项目维护者可将社区内优秀的入门讲解视频、架构演进分享录像链接纳入项目附属资源库，并在项目文档中引用该资源库的查询接口，降低新贡献者的学习曲线陡峭程度。

## 快速开始

以下步骤将引导您在本地环境中快速启动VideoLink服务。

```bash
# 克隆项目仓库至本地
git clone https://github.com/videolink/videolink-core.git

# 进入项目根目录
cd videolink-core

# 使用pip安装项目依赖（建议先创建并激活Python虚拟环境）
pip install -r requirements.txt

# 执行数据库迁移脚本，初始化SQLite数据库表结构
python manage.py migrate

# 创建超级管理员账户，用于后续登录管理后台
python manage.py createsuperuser

# 启动开发服务器，默认监听8000端口
python manage.py runserver
```

服务启动后，访问 http://127.0.0.1:8000/admin 即可进入管理后台进行链接的增删改查操作。

## 安装要求

生产环境部署前，请确保基础设施满足以下最低依赖要求。

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.9 及以上 | 核心运行时环境，低于此版本将无法解析类型注解语法 |
| Django | 4.2 LTS | Web框架及ORM层，LTS版本保证长期安全更新支持 |
| Celery | 5.3.x | 异步任务队列，用于链接健康状态侦测及快照生成等后台作业 |
| Redis | 6.2 及以上 | 作为Celery的Broker与Result Backend，同时用于缓存热点查询结果 |
| SQLite / PostgreSQL | SQLite 3.35+ / PostgreSQL 13+ | 默认使用SQLite，生产环境强烈建议切换至PostgreSQL以支持高并发写入 |
| Node.js | 18.x | 仅用于前端静态资源构建，运行时环境不依赖Node.js |

## 项目结构

项目采用Django标准应用布局，并结合领域驱动设计思想对核心业务模块进行了垂直划分。

```
videolink-core/
├── manage.py                         # Django项目管理入口
├── requirements.txt                  # Python依赖声明
├── config/                           # 项目全局配置模块
│   ├── settings/                     # 多环境配置拆分
│   │   ├── base.py                   # 基础配置（通用）
│   │   ├── development.py            # 开发环境配置（启用调试）
│   │   └── production.py             # 生产环境配置（关闭调试、启用缓存）
│   ├── urls.py                       # 根URL路由映射
│   └── asgi.py                       # ASGI应用入口（支持WebSocket）
├── apps/                             # 所有业务应用存放目录
│   ├── links/                        # 链接管理核心模块
│   │   ├── models.py                 # Link、Tag、Category数据模型
│   │   ├── services.py               # 链接解析、状态更新等业务服务层
│   │   └── admin.py                  # 后台管理界面定制
│   ├── health/                       # 健康侦测模块
│   │   ├── tasks.py                  # Celery异步任务定义
│   │   ├── checker.py                # HTTP状态码检查器实现
│   │   └── scheduler.py              # 周期调度配置
│   ├── search/                       # 全文检索模块
│   │   ├── indexes.py                # 倒排索引构建逻辑
│   │   └── querier.py                # 查询解析与排序
│   ├── snapshot/                     # 快照与版本管理模块
│   │   ├── dumper.py                 # 数据序列化导出
│   │   └── loader.py                 # 快照恢复加载
│   └── accounts/                     # 用户与权限模块
│       ├── models.py                 # 扩展用户Profile
│       └── permissions.py            # 角色权限校验逻辑
├── static/                           # 收集后的静态资源（CSS/JS）
├── media/                            # 用户上传文件存储目录
├── templates/                        # Django模板文件
│   ├── admin/                        # 后台模板覆盖
│   └── web/                          # 前台展示模板
├── tests/                            # 单元测试与集成测试
```

## 常见问题

Q: 系统支持导入的链接数量上限是多少？
A: 系统本身不设硬性上限，实际承载能力取决于部署环境的硬件资源。使用默认SQLite配置时，单表记录数建议控制在50万条以内以保证查询性能。若预期管理规模超过此阈值，强烈建议切换至PostgreSQL，并配置合适的索引策略。经实测，在4核16GB内存、PostgreSQL 14环境下，系统可稳定管理200万条链接记录。

Q: 健康侦测任务是否会因为目标网站响应慢而阻塞整体任务队列？
A: 不会。健康侦测模块为每个链接的HTTP请求设置了独立的超时控制（默认连接超时5秒，读取超时10秒）。同时，Celery工作进程采用预取多任务模式，单个任务的阻塞不会影响其他任务的执行。若目标站点频繁超时，系统会自动将该链接标记为疑似不稳定，并降低其后续侦测频率以节省资源。

Q: 如何从旧版本迁移数据至新版本？
A: 项目提供了版本迁移工具集。对于小版本升级（如v1.1.x至v1.2.x），通常仅需执行 python manage.py migrate 即可自动完成数据库模式变更。对于大版本升级（如v1.x至v2.x），建议先停止服务，在旧版本环境下执行 python manage.py dumpdata --exclude=contenttypes --exclude=auth.permission > data.json 导出数据，再启动新版本实例并执行 python manage.py loaddata data.json 完成导入。具体迁移步骤会随每个大版本发布提供详细说明文档。
