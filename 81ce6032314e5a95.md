# VueBest 渠道资源聚合平台

VueBest 渠道资源聚合平台是一个面向 Vue.js 生态系统开发者的高质量外链与资源导航系统。该项目系统性地收集、分类并索引了与 Vue.js 相关的优质技术内容、组件库、工具链、教程案例及社区讨论渠道，帮助开发者从海量信息中快速定位所需资源。

本项目主要服务于 Vue.js 全栈工程师、前端团队负责人、技术选型决策者以及从 React 或 Angular 转向 Vue 生态的迁移开发者。通过结构化的渠道标识符体系，用户能够以可预测的 URL 模式访问经过人工筛选与社区验证的技术资源，显著降低信息检索成本，提升开发效率与学习路径的清晰度。

## 功能概览

- **渠道化资源索引**：每个资源条目均分配唯一渠道标识符，形成稳定的永久链接体系，便于书签管理与团队内部分享。

- **多维度分类筛选**：资源按组件库、工具插件、性能优化、状态管理、测试框架、SSR 方案、移动端适配等维度进行标签化组织。

- **社区驱动的内容更新**：采用投票与评论机制，允许用户对资源质量进行反馈，确保列表中的每一项均经过实际使用验证。

- **版本兼容性标注**：针对每个资源清晰标注其支持的 Vue 2.x / 3.x 版本以及所需的 Node.js 运行时最低要求。

- **快速预览与跳转**：在列表页面提供资源的简短描述、GitHub Star 数量、周下载量等元数据，支持一键直达目标站点。

- **响应式搜索与过滤**：提供实时模糊搜索功能，支持按渠道 ID、资源名称、描述关键词或分类标签进行动态过滤。

- **RSS 订阅与更新通知**：提供频道更新订阅源，开发者可通过 RSS 阅读器跟踪新增资源或重要版本变更。

## 应用场景

- **技术选型与组件库评估**：当团队需要为项目选择 UI 组件库、表格组件或图表库时，可通过本平台快速对比不同渠道中的候选资源，查看其维护活跃度、文档完整度及社区评价，辅助决策。

- **新人入职培训路径规划**：前端团队负责人可利用本平台的教程与最佳实践类渠道，为新入职的 Vue 开发者构建系统化的学习路线，从基础响应式原理到高级组合式 API 应用，覆盖完整知识图谱。

- **性能问题排查与优化参考**：当应用出现渲染卡顿或内存泄漏时，开发者可检索性能监控、分析工具或优化策略相关的渠道，获取经过实践验证的解决方案与第三方库使用经验。

## 快速开始

以下命令帮助您在本地环境中快速启动 VueBest 资源聚合平台的开发实例。

```bash
# 克隆代码仓库至本地
git clone https://github.com/vuebest/vuebest-channel-hub.git

# 进入项目根目录并安装依赖
cd vuebest-channel-hub
npm install

# 启动开发服务器，默认监听 3000 端口
npm run dev
```

## 贡献指南

1. 渠道资源新增或更新

   如果希望向平台添加新的 Vue.js 相关资源或更新现有渠道的目标链接与描述，请先查阅 docs/guide/contribution-criteria.md 中的收录标准。确认资源符合条件后，在 src/api/channelService.js 中的 MOCK_CHANNELS 数组尾部追加新条目对象，并确保 id 字段采用三段式随机字母数字组合（如 abc12）。提交前需运行 npm run test:unit 验证数据结构完整性。

2. 分类标签体系优化

   分类标签定义位于 src/stores/tagStore.js 中的 TAXONOMY 常量对象。若需新增或重命名标签，请同步更新 docs/tag-guidelines.md 中的分类说明文档，并确保所有已有渠道的 tags 数组引用了正确的标签键值。新增标签后，执行 npm run build 检查无控制台警告。

3. 文档翻译与国际化

   项目文档默认采用简体中文。若希望贡献英文或其他语言版本，请在 docs/ 目录下创建对应语言子目录（如 docs/en/）。翻译时需保持 Markdown 结构一致，并更新 router/index.js 中的 locale 路由前缀配置。翻译完成后需至少两位已有贡献者进行审校。

4. 缺陷报告与功能请求

   所有问题请通过 GitHub Issues 提交。缺陷报告需包含复现步骤、预期行为与实际行为对比，并附上浏览器控制台报错截图或网络请求日志。功能请求需清晰描述使用场景与期望的交互流程，并标注是否愿意参与实现。

5. 代码提交规范

   Git 提交信息请遵循 Conventional Commits 格式（如 feat: 新增渠道详情页骨架、fix: 修复搜索防抖失效问题、docs: 更新安装要求表格）。提交前需通过 ESLint 检查与单元测试全量回归。Pull Request 目标分支为 main，合并前需至少一位 Maintainer 批准并通过 CI 流水线。

## 常见问题

Q: 渠道标识符（如 rpk94.html）的生成规则是什么？是否可以自定义？

A: 渠道标识符由三位随机字母与两位随机数字组合而成，由系统在资源首次收录时自动生成，以确保唯一性且避免与已有 ID 冲突。该标识符不可手动修改，旨在维护永久链接的稳定性。如需变更，建议停用旧渠道并新建条目，同时在旧页面通过 meta refresh 或 HTTP 301 重定向至新地址。

Q: 平台是否支持私有化部署或离线使用？

A: 本项目完全开源，支持私有化部署。您可以将仓库克隆至内网服务器，并修改 .env 中的 VITE_API_BASE_URL 指向内部数据源。静态资源与渠道数据均可预构建为纯静态文件，适合在内网环境或离线文档场景中使用。但请注意，第三方外链资源的可用性取决于目标站点的网络可达性。

Q: 如何获取渠道资源的更新动态，例如新增渠道或某渠道的文档链接发生变更？

A: 您可以在首页底部订阅 RSS 源（/feed.xml），该源会在渠道列表发生变更时推送通知。同时，项目维护团队会在每周一通过 GitHub Discussions 发布周报，汇总本周新增、更新及下架的资源条目。对于企业用户，支持通过 Webhook 将变更事件推送至钉钉或飞书群组。
