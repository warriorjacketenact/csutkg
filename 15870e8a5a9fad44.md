# ResourceVault

ResourceVault 是一个面向技术团队与个人开发者的外链资源归集与导航系统。该项目并不直接托管内容，而是提供结构化、可检索的技术资源索引机制，帮助使用者从海量散落的外部链接中快速定位高价值信息。ResourceVault 适用于需要系统化管理技术文档、API 参考、开源组件、运维手册及学习资料等外链资源的各类场景，解决资源散落、遗忘、重复查找等常见问题。

## 功能概览

**多级标签分类体系**：支持为每条外链赋予多个自定义标签，并基于标签树进行过滤与聚合检索，便于按技术栈、业务域或文档类型组织资源。

**全文元数据检索**：基于链接标题、描述、标签及自定义备注字段进行快速关键词匹配，支持模糊搜索与精确匹配两种模式。

**自动快照与可用性检测**：定期对已收录链接进行 HTTP 状态检测，标记失效或重定向链接，并提供最近一次成功访问的时间记录。

**导入与导出机制**：支持批量导入链接列表（CSV、JSON、Markdown 列表格式），同时可将当前索引导出为静态 HTML 或 JSON 结构，便于离线查阅。

**访问统计与热度排序**：记录每个链接的点击次数与最近访问时间，支持按热度、添加时间或字母序对资源列表进行动态排序。

**协作批注与版本记录**：允许团队成员在链接条目下添加批注或阅读笔记，并保留每次修改的历史版本，方便追溯信息变更过程。

**外部数据源同步**：可通过配置 RSS 订阅或 GitHub Watch 列表，自动拉取指定源的新增链接并归入待审核队列，减少手动录入成本。

## 应用场景

技术团队内部知识库构建：团队可将日常开发中涉及的依赖库主页、技术规范文档、内部运维面板地址等统一录入 ResourceVault，替代浏览器书签的分散管理方式，新成员入职时可直接查阅全量技术资源地图。

开源项目文档关联：开源维护者可在项目 README 中引用 ResourceVault 生成的资源列表页，将外部参考、插件生态、社区教程等链接统一归集，避免在多个地方重复维护链接集合。

个人学习路径管理：学习者可按阶段（基础、进阶、实战）或主题（前端、后端、运维、算法）对收藏的教程、视频、代码示例链接进行分类，并结合批注功能记录学习心得与完成状态。

技术选型评估辅助：在进行中间件或框架选型时，可将候选项目的官网、GitHub 仓库、性能测评报告、社区讨论帖等链接集中收录，通过标签和备注对比各选项的优劣势，形成结构化的评估记录。

离线文档中心构建：对于网络受限的内网环境，可将官方文档站点、镜像源、内部 Git 仓库等链接统一导出为静态页面，配合内网 Web 服务器提供本地化的资源导航入口。

## 快速开始

以下命令演示如何从 GitHub 克隆项目、安装依赖并启动开发服务器。请确保已安装 Node.js 18.x 及以上版本。

```bash
git clone https://github.com/your-org/resourcevault.git
cd resourcevault
npm install
npm run dev
```

若需要构建生产环境静态文件，请执行：

```bash
npm run build
npm run start
```

项目默认监听 3000 端口，启动后可通过浏览器访问 http://localhost:3000 使用界面。首次启动时系统会自动创建 SQLite 数据库文件并生成示例分类数据。

## 安装要求

| 依赖名称 | 必需版本 | 说明 |
|----------|----------|------|
| Node.js | 18.x 或 20.x LTS | 运行时环境，用于执行服务端代码及构建脚本 |
| npm | 9.x 或以上 | 包管理器，用于安装项目依赖及执行 scripts 命令 |
| SQLite3 | 3.40 或以上 | 嵌入式数据库，用于存储链接元数据与分类结构 |
| Python | 3.9 或以上 | 仅在运行外部数据源同步脚本时需要，用于解析 RSS 及 XML 数据 |
| Git | 2.30 或以上 | 用于克隆仓库及后续拉取更新，非强制但建议安装 |
| curl | 7.68 或以上 | 用于可用性检测模块中的 HTTP 请求探测，系统自带或需手动安装 |

## 贡献指南

1. 阅读开发文档：在提交任何代码或文档变更之前，请先阅读 docs/development/ 目录下的架构说明与编码规范，确保您的修改与项目整体设计方向一致。

2. 选择或创建 Issue：请在 GitHub Issues 中查找已有的待办任务或功能提案。若无相关条目，请新建一个 Issue 描述您要解决的问题或希望增加的特性，并等待维护者确认。

3. 分支开发流程：从 main 分支切出新的特性分支，命名格式为 feature/功能简述 或 fix/问题描述。所有提交信息应遵循 Conventional Commits 规范，便于自动生成变更日志。

4. 编写测试用例：对于新增的功能或修复的缺陷，请在 tests/ 对应目录下补充相应的单元测试或集成测试，确保测试覆盖率达到 80% 以上。运行 npm test 验证所有用例通过。

5. 提交 Pull Request：推送分支至 GitHub 仓库后，创建 Pull Request 并填写模板中的检查清单。维护者将在 2 个工作日内进行 Review，若有修改意见请及时响应。合并后您的贡献将被列入项目贡献者列表。

## 常见问题

Q: 项目启动后界面显示空白，控制台报错 "Cannot find module 'sqlite3'"？

A: 这是由于 SQLite3 原生依赖未正确编译所致。请先确认系统已安装 Python 3.x 及构建工具（Windows 需安装 Visual Studio Build Tools，Linux 需安装 build-essential），然后执行 npm rebuild sqlite3 重新编译该模块。若仍失败，可尝试使用 npm install --force 强制重新安装所有依赖。

Q: 如何将已有浏览器书签批量导入 ResourceVault？

A: 目前支持两种方式：其一，将书签导出为 HTML 文件，再通过界面上的 "导入书签" 功能上传；其二，将书签整理为 CSV 格式（每行包含 title, url, tags 三列），使用命令行脚本 scripts/import-csv.js 进行导入。具体步骤参见 docs/user-guide/import-export.md 文档。

Q: 可用性检测模块会消耗大量网络带宽或导致目标站点被频繁请求吗？

A: 检测模块默认采用指数退避策略，每个链接的检测间隔不少于 24 小时，且并发请求数限制为 3 个。您可以在 config/default.json 中调整 CHECK_INTERVAL_HOURS 和 MAX_CONCURRENT_CHECKS 参数来控制检测频率与并发度，避免对源站造成压力。
