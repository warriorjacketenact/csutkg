# LinkBridge 技术资源导航站

LinkBridge 是一个面向开发者与技术研究人员的结构化外链资源聚合平台，专注于对分散在各类技术社区、论坛与文档站点的优质外部链接进行系统性收集、分类与索引。项目定位为技术资源的中转枢纽，帮助用户在海量信息中快速定位所需的技术文档、讨论串、代码示例与最佳实践参考。

本项目不直接托管内容，而是通过精细化的元数据标注与链接状态监控，为技术团队、开源贡献者与独立研究者提供高可用性的外链检索与回溯服务。LinkBridge 适用于需要长期维护技术知识图谱、追踪社区讨论脉络或构建内部技术文库的各类组织。

## 功能概览

**结构化外链入库** 对每个收录的 URL 进行自动化的标题抓取、来源站点识别与内容类型分类，支持论坛帖子、文档页面、代码片段等多种内容形态。

**多维度标签体系** 基于链接来源域名、路径结构与参数模式自动生成技术领域、话题热度与更新活跃度标签，便于后续过滤与检索。

**链接可用性监控** 定期对已收录的 URL 执行 HTTP 状态检查，自动标记失效链接并生成告警通知，保障资源库的健康度。

**全文元数据检索** 支持对链接标题、来源站点、收录时间与自定义备注字段进行组合查询，提供精确与模糊两种匹配模式。

**分类目录导航** 按站点来源、内容类型与主题领域组织资源视图，用户可通过层级目录快速浏览同类资源。

**批量导入与导出** 提供 JSON 与 CSV 格式的数据交换接口，支持将外部链接列表批量导入系统，或导出当前索引库供其他工具使用。

## 应用场景

**技术团队内部知识库建设** 技术负责人可使用 LinkBridge 将团队日常参考的社区讨论帖、官方文档镜像与解决方案文章统一收录，形成可共享的团队知识资产，避免重复查找与信息丢失。

**开源项目文档外链管理** 开源项目维护者可将项目 README 或官方网站中引用的所有外部链接托管于 LinkBridge，通过统一的监控面板掌握每个引用链接的可用性，及时替换失效资源。

**技术社区内容聚合与推荐** 社区运营人员可利用 LinkBridge 的批量导入功能，将历史精华帖与优质问答链接进行结构化整理，生成按主题或时间排列的推荐阅读列表，提升社区内容利用率。

**个人技术研究资源整理** 独立研究者或学习者可将日常阅读的技术文章、API 参考与案例代码链接存入 LinkBridge，通过标签与分类快速回溯，构建个人化的学习路径。

## 快速开始

以下指令适用于 Linux 与 macOS 环境，Windows 用户可使用 WSL 或 Git Bash 执行。

```bash
# 克隆代码仓库至本地
git clone https://github.com/linkbridge/linkbridge.git

# 进入项目目录
cd linkbridge

# 安装项目依赖（使用 Python 3.10+ 与 pip）
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 初始化本地数据库并启动开发服务器
python manage.py migrate
python manage.py runserver
```

访问终端输出的本地地址即可进入 LinkBridge 实例首页，默认端口为 8000。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.10 至 3.12 | 核心运行环境，低于 3.10 将无法兼容异步 IO 模块 |
| PostgreSQL | 14.x 或 15.x | 主数据库，用于存储链接元数据与索引信息 |
| Redis | 7.x | 缓存与任务队列后端，用于链接状态监控调度 |
| Node.js | 18.x 或 20.x | 前端资源构建工具链依赖，仅开发模式需要 |
| Nginx | 1.24 或更新 | 生产环境推荐反向代理服务器，用于静态资源服务与负载均衡 |

## 贡献指南

项目采用 GitHub Flow 协作模型，所有贡献均通过 Pull Request 进行。请按照以下步骤提交变更。

第一，在 GitHub 上 Fork 本仓库至个人账户，然后克隆 Fork 后的仓库至本地开发环境。确保本地已安装 Go 1.21+ 与 golangci-lint 工具。

第二，新建功能分支，分支命名规则为 feature/简述功能 或 fix/简述问题。分支名称使用小写字母与连字符，例如 feature/add-tag-filter。

第三，编写代码时遵循项目已有的代码风格，运行 make lint 检查语法问题，运行 make test 确保所有已有测试用例通过。新增功能必须附带对应的单元测试。

第四，提交变更时使用语义化的 commit message 格式，即 type(scope): subject 模式，类型包括 feat、fix、docs、refactor、test 等。scope 填写影响的模块名称。

第五，向本仓库的 main 分支发起 Pull Request，在 PR 描述中清晰说明变更目的、实现方式与测试覆盖情况。至少一位项目维护者审核通过后方可合并。

## 常见问题

Q: 系统支持同时管理多个批次的资源吗？

A: 支持。LinkBridge 内置批次管理功能，每个链接入库时均可指定批次编号与总批次数。当前运行中的批次为 281/1241，用户可通过管理界面切换查看不同批次的数据，或按批次进行整体导出与状态监控。

Q: 链接监控的频率是多少，是否可以自定义？

A: 默认每 24 小时对所有已收录链接执行一次状态检查。用户可在 configs/production.yaml 中修改 check_interval 字段的值，单位为小时。监控进程会分布式地拉取 Redis 队列中的任务，检查结果会更新至数据库的 last_checked_at 与 status_code 字段。

Q: 如何迁移现有的书签或收藏夹数据到 LinkBridge？

A: 项目提供了 scripts/import_links.py 脚本，支持从 Netscape HTML 书签导出格式、CSV 文件或纯文本 URL 列表中导入链接。导入时可通过命令行参数指定目标批次号与默认标签。详细用法请参考 scripts/import_links.py --help 输出的帮助信息。
