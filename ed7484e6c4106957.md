# Tokyo Mirai Video Resource Aggregator

Tokyo Mirai Video Resource Aggregator 是一个面向技术内容创作者、研究机构与视频归档工程师的开源外链管理与资源索引系统。项目定位为高可用、可扩展的视频资源元数据收集与展示平台，专门用于处理来自 Tokyo Mirai 网络节点的大规模视频文件外链。目标用户包括数据工程师、自动化运维人员以及需要批量管理视频资源的开发团队。

本项目解决了视频资源链接分散、缺乏统一索引、难以批量验证可用性以及无法快速生成可读报告等实际问题。通过结构化的资源清单与自动化的元数据抽取脚本，用户能够以编程方式获取资源状态、生成目录快照，并集成至现有的 CI/CD 或数据流水线中。

## 功能概览

**批量外链导入与解析** 支持一次性导入数百条原始视频链接，自动识别文件扩展名与路径结构，生成标准化内部标识符。

**资源状态健康检查** 内置异步 HTTP 探活模块，可配置超时与重试策略，定期检测每个资源 URL 的可访问性并输出 JSON 格式报告。

**元数据自动补全** 基于 URL 路径中的日期与随机串，自动推算视频采集时间、生成短代码标签，并支持人工打标覆盖。

**分类目录生成** 根据资源来源域名、路径层级及文件类型自动归类，输出树状结构视图，便于人工浏览与审计。

**CLI 与 API 双接口** 提供命令行工具用于日常运维，同时暴露 RESTful API 供前端面板或第三方系统调用。

**增量更新与变更日志** 记录每次资源列表的增删改操作，生成可追溯的变更历史，支持回滚至任意历史版本。

**导出格式多样化** 支持将资源索引导出为 Markdown 表格、CSV 电子表格或纯文本清单，适配不同下游工具需求。

**访问控制与密钥管理** 内置基于环境变量的 API 密钥认证机制，限制敏感操作仅对授权用户开放。

## 应用场景

**自动化监控看板搭建** 运维工程师可定时运行健康检查脚本，将结果推送至 Prometheus 或 Grafana，实时监控所有视频资源的在线率与响应延迟，及时发现失效链接。

**数据迁移辅助工具** 在将资源从 Tokyo Mirai 节点迁移至其他存储服务时，本项目的导出功能可生成完整的源地址列表，配合 wget 或 rsync 实现批量迁移，同时保留原始路径结构。

**学术研究数据索引** 研究机构收集大规模视频样本时，利用本项目维护资源清单，记录每个样本的采集时间、来源节点与文件哈希值，确保实验可复现性。

**内容审核流水线集成** 内容审核团队可将本项目的 API 接入内部工作流，自动拉取待审资源列表，审核完成后回调更新状态字段，实现闭环管理。

## 快速开始

以下步骤适用于 Linux / macOS / Windows WSL 环境，确保已安装 Git 与 Node.js 18.x 以上版本。

```bash
# 克隆仓库至本地
git clone https://github.com/tokyo-mirai/video-aggregator.git
cd video-aggregator

# 安装项目依赖
npm install

# 复制环境变量模板并填写必要配置
cp .env.example .env

# 运行初始资源导入与健康检查（首次运行约需 2-5 分钟）
npm run import:resources
npm run health:check
```

执行完成后，`output/` 目录下将生成 `resource-index.json` 与 `health-report.json` 两个文件，分别包含完整资源清单与当前可用性状态。

## 贡献指南

1. 查阅 `docs/developer/` 目录下的开发指南，了解项目架构、编码规范与测试要求。所有新功能需提供对应的单元测试用例，测试覆盖率不低于 80%。

2. 从 GitHub Issues 中选择标记为 `good-first-issue` 或 `help-wanted` 的任务，在评论区说明认领意向，避免多人重复工作。对于较大特性，建议先创建讨论 Issue 并与维护者沟通设计方案。

3. 派生项目仓库至个人账户，基于 `main` 分支创建特性分支，分支命名遵循 `feature/功能简述` 或 `fix/问题编号` 格式。提交信息采用 Conventional Commits 规范，例如 `feat: 添加批量导入进度显示`。

4. 完成开发后，确保所有现有测试通过且新增测试覆盖所改代码。运行 `npm run lint` 与 `npm run format` 统一代码风格，然后发起 Pull Request 至 `main` 分支，详细描述变更内容、测试结果与影响范围。

5. PR 合并前需要至少一位核心维护者的 Code Review，如有修改意见需及时响应。合并后相关贡献者将被列入 `CONTRIBUTORS.md` 文件致谢。

## 常见问题

**问：导入 250 个资源链接时出现超时或部分失败怎么办？**

答：系统默认并发数为 10，超时时间为 30 秒。若网络环境较差，可在 `config/default.json` 中调低 `checker.concurrency` 至 5 或 3，同时增大 `checker.timeout` 至 60000 毫秒。失败记录会写入 `output/failed.log`，可通过 `npm run retry` 命令重试。

**问：如何自定义导出格式或增加新的元数据字段？**

答：导出器模块位于 `src/core/exporter/`，实现了策略模式。您可继承 `BaseExporter` 类并实现 `format(data)` 方法，随后在 `cli/commands/export.ts` 中注册新格式名称。元数据字段扩展需同步修改 `src/types/resource.ts` 中的 `ResourceMetadata` 接口，并调整解析器逻辑以填充新字段。

**问：API 服务如何启用 HTTPS 并配置域名？**

答：生产环境建议使用 Nginx 或 Caddy 作为反向代理终止 TLS，将请求转发至本机 API 端口（默认 3000）。如需 Node.js 原生支持 HTTPS，可在 `.env` 中设置 `SSL_ENABLED=true`，并配置 `SSL_KEY_PATH` 与 `SSL_CERT_PATH` 指向 PEM 格式密钥与证书文件，然后重启服务。
