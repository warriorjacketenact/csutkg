# HyperLink Hub

HyperLink Hub 是一个面向技术研究者与内容聚合者的高密度外链资源导航站。该项目不生产原始内容，而是通过结构化方式收录来自互联网的高价值信息页面，解决技术文档查阅、历史快照追溯与专题资料整理中的链接分散问题。目标用户包括开源社区维护者、技术博主、渗透测试人员以及需要长期保存研究线索的数字取证分析师。

本项目本质上是一个大型外链索引仓库，所有收录的 URL 均保持原始页面完整定位信息，不进行重定向代理或内容转储。通过分类标签与层级目录，用户可快速定位到特定主题下的历史文章、技术笔记或事件记录。第 1186/1241 批次收录工作聚焦于 huangliang.org 域下的 250 个深度页面链接，涵盖多个技术子领域与时间跨度。

## 功能概览

- **裸链接原样存储**：所有收录的 URL 严格按照原始字符串保存，不添加协议头、不补全域名、不修改大小写，确保引用路径与原始发布环境完全一致。
- **批次化导入机制**：支持按批次（每批 200-300 条）批量导入链接数据，自动生成导入日志与重复检测报告。
- **分类标签系统**：每条链接可关联多个自定义标签，支持按技术领域、发布时间、内容类型等多维度过滤。
- **只读镜像视图**：提供对已收录链接的只读列表视图，支持 Markdown 表格导出与 JSON 格式序列化。
- **完整性校验工具**：内置链接可达性检查模块，可定期检测收录 URL 的响应状态码并标记异常条目。
- **元数据扩展字段**：每条链接支持记录收录时间、批次号、简短备注说明，便于团队协作维护。
- **全文检索接口**：基于 URL 路径中的数字 ID 与备注内容提供基础关键词搜索能力。
- **变更历史追踪**：每次增删改操作记录操作时间与操作人，支持回滚至任意历史版本。

## 应用场景

- **技术历史资料归档**：研究人员可通过该仓库回溯 huangliang.org 上发布的过往技术讨论，避免原站点内容迁移或下线导致的引用失效。
- **漏洞分析线索串联**：安全分析人员可在同一域下批量检索特定 ID 段的文章，快速建立漏洞披露时间线或补丁演进图谱。
- **博客站点备份参考**：内容创作者可将该索引作为个人博客备份计划的补充参考，了解同类站点的文章结构及永久链接策略。
- **开源项目文档外链整理**：开源项目维护者可将本仓库作为项目 README 或 Wiki 的外部参考资源池，统一管理分散的引用链接。
- **网络调研数据采集**：市场或学术调研人员可基于本批次链接进行定向内容抓取，分析特定时段内的技术舆论热点。

## 快速开始

以下命令演示如何将本仓库克隆至本地，安装基础依赖并启动本地索引服务。

```bash
# 克隆仓库到本地
git clone https://github.com/your-org/hyperlink-hub.git
cd hyperlink-hub

# 安装 Python 依赖（推荐使用虚拟环境）
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 初始化 SQLite 数据库并导入第 1186/1241 批次链接
python manage.py init-db
python manage.py import-batch --batch 1186 --source ./data/batch_1186.txt

# 启动本地 Web 索引服务（默认端口 8080）
python manage.py runserver --port 8080
```

## 贡献指南

1. 克隆本仓库并切换到开发分支，所有新增批次导入或元数据修改需在独立分支上进行，分支命名格式为 `batch/{批次号}` 或 `fix/{简要描述}`。
2. 新增批次导入时，在 `data/batches/` 目录下创建对应批次号文本文件，每行一个裸 URL，确保 URL 格式与原始来源完全一致，不添加额外空白字符。
3. 运行 `make test` 执行全部单元测试与链接格式校验，确保新增数据未破坏现有导入逻辑，测试通过后方可提交。
4. 提交 Pull Request 时需附上变更说明，包括批次号、链接总数、是否有重复或异常条目。仓库维护者将在 48 小时内完成审查。
5. 若发现已收录链接失效或内容明显偏离技术主题，请在 `metadata/tags.yaml` 中对应条目下添加 `status: dead` 或 `status: off-topic` 标记，并备注发现日期。

## 常见问题

**问：本项目是否存储原始页面的副本或缓存内容？**
答：不存储。本项目仅收录 URL 字符串及其基础元数据（收录时间、批次号、标签）。原始页面的内容、样式及可用性完全依赖于源站点。用户访问收录链接时，请求将直接导向原始服务器。

**问：如何报告某个收录链接已永久失效或指向恶意内容？**
答：请在本仓库的 Issues 区提交问题，标题注明 `[Broken Link]` 及对应 URL 的末尾数字 ID。维护团队会通过链接检查工具进行复核，确认后会在元数据中标记该链接状态，但不会从列表中物理删除，以保证批次记录的完整性。

**问：导入批次时遇到重复 URL 会如何处理？**
答：导入引擎会基于完整 URL 字符串进行去重判断。若检测到与已存在记录完全相同的 URL，新条目将被跳过并记录在导入日志的 `duplicates` 字段中，不会覆盖原有记录的收录时间与备注信息。
