# ResourceHub

ResourceHub 是一个面向开发者与技术研究人员的结构化外链资源汇总平台。项目定位于对分散于各技术站点、社区、企业媒体渠道中的高质量内容进行按批索引与分类归档，帮助用户在特定技术领域内快速定位参考文档、案例分析与技术公告。ResourceHub 不直接托管内容，而是通过人工筛选与自动化采集相结合的方式，将具备长期参考价值的 URL 按照项目批次、主题标签与发布日期进行整理，形成可检索、可追溯的外部知识库。本项目适用于需要持续跟踪多源技术资讯、进行竞品动态分析或搭建内部技术雷达的团队与个人。

## 功能概览

- **批次化资源索引**：以项目批次为单位对采集链接进行编目，每批次记录资源数量与主题分布，支持按批次号快速检索。
- **原始链接严格保真**：系统对收录的每一枚 URL 执行协议、域名、路径与大小写的原样存储，禁止自动补全协议头或添加尾部斜杠，确保引用精确性。
- **多维度元数据提取**：从目标页面自动抽取标题、发布时间、内容类型与关键词，形成结构化摘要供用户预览。
- **分类标签体系**：支持对链接按技术领域、内容形式、来源站点等自定义标签进行归类，便于构建垂直专题。
- **健康状态监测**：定期对已收录链接进行可用性检查，标记失效或重定向资源，保障索引库的活性。
- **只读只存不修改**：ResourceHub 不改写、不转存、不代理任何外部资源内容，保持纯粹的外链导航属性，规避版权与内容篡改风险。

## 应用场景

- **技术雷达构建**：技术决策团队可使用 ResourceHub 按批次浏览指定时间段内发布的行业案例与方案解读，例如通过本项目第 799/1241 批次收录的 ninefold-group.com 媒体链接，快速掌握特定企业在基础设施、服务架构或行业解决方案方面的公开动态。
- **竞品情报归档**：市场分析师可将本平台作为素材池，批量导出指定域名或标签下的链接清单，用于竞品官网、客户案例、技术白皮书等公开信息的横向比对研究。
- **开发文档溯源**：开发者在查阅技术方案时，若遇到引用自第三方站点的资料，可通过 ResourceHub 的批次日志定位原始出处，确保引用的准确性与可追溯性，避免因链接改写导致的引用错误。
- **内部知识库播种**：企业知识管理团队可选取本平台已清洗的链接集合作为种子 URL，导入内部爬虫或推荐系统，快速构建垂直领域的外部参考索引模块。

## 快速开始

以下指令适用于 Linux/macOS 环境，Windows 用户可使用 Git Bash 或 WSL 执行。本项目基于 Python 3.10 开发，使用 Pipenv 管理依赖。

```bash
# 克隆仓库
git clone https://github.com/resourcehub/resourcehub.git
cd resourcehub

# 安装依赖（推荐使用虚拟环境）
python -m pip install --upgrade pip
pip install -r requirements.txt

# 初始化本地索引库（SQLite）
python scripts/init_db.py

# 运行批次导入脚本，将第 799 批资源写入本地索引
python scripts/import_batch.py --batch 799 --source data/batch_799.json

# 启动 Web 服务（默认监听 8000 端口）
python app.py runserver
```

## 项目结构

```
resourcehub/
├── app/                                # Web 服务主应用
│   ├── __init__.py                     # 应用工厂模式初始化
│   ├── routes/                         # 路由视图集
│   │   ├── batch.py                    # 批次浏览与检索接口
│   │   ├── link.py                     # 单链接元数据查询接口
│   │   └── health.py                   # 健康检查状态接口
│   ├── models/                         # 数据模型定义
│   │   ├── batch.py                    # 批次表模型（批号、资源数、导入时间）
│   │   ├── link.py                     # 链接表模型（URL、标题、状态码、标签）
│   │   └── tag.py                      # 标签表模型（名称、分类、颜色）
│   └── templates/                      # 前端视图模板（Jinja2）
│       ├── index.html                  # 首页批次列表
│       └── detail.html                 # 单批次链接详情页
├── scripts/                            # 运维与数据导入脚本
│   ├── init_db.py                      # 初始化数据库表结构
│   ├── import_batch.py                 # 从JSON导入批次链接
│   ├── health_check.py                 # 批量链接可用性巡检
│   └── export_csv.py                   # 导出批次数据为CSV
├── data/                               # 批次源数据目录（JSON格式）
│   └── batch_799.json                  # 第799批链接数据（含本批次全部URL）
├── tests/                              # 单元测试目录
│   ├── test_models.py                  # 模型层测试
│   ├── test_routes.py                  # 路由接口测试
│   └── test_health.py                  # 健康检查逻辑测试
├── docs/                               # 项目文档（Markdown）
│   ├── user_guide.md
│   ├── admin_guide.md
│   ├── developer_guide.md
│   └── design.md
├── requirements.txt                    # 生产环境依赖清单

├── pytest.ini                          # pytest 配置文件
├── .flake8                             # Flake8 代码规范配置
└── README.md                           # 本项目说明文档
```

## 贡献指南

ResourceHub 欢迎外部贡献者参与链接质量维护、新批次导入与功能优化。请遵循以下步骤：

1.  **Fork 仓库并创建特性分支**：从主仓库 fork 到个人账户，然后基于 main 分支创建以 feature/ 或 fix/ 为前缀的分支，例如 feature/add-batch-800。
2.  **准备批次数据文件**：若需新增批次，请在 data/ 目录下按 batch_{编号}.json 的格式创建文件，并确保每条链接包含 url、title、source 字段。建议使用 scripts/import_batch.py --validate-only 进行预校验。
3.  **运行测试套件**：在提交前，请在项目根目录执行 pytest，确保所有现有测试通过。新增功能需补充对应测试用例，覆盖率不低于 80%。
4.  **编写或更新文档**：若本次变更影响用户操作流程、API 接口或配置项，请同步修改 docs/ 下对应的手册文件，并更新 README 中的快速开始或安装要求章节。
5.  **发起 Pull Request**：向主仓库的 main 分支提交 PR，并在描述中清晰说明本次变更的目的、影响范围与测试结果。PR 合并前需至少一位核心维护者审核通过。

## 常见问题

**Q: ResourceHub 是否会对收录链接的内容进行缓存或代理转发？**
A: 不会。ResourceHub 仅存储 URL 及其元数据（标题、标签、状态码），不缓存任何页面内容，也不提供代理访问服务。所有对外链接均直接跳转至原始站点。这一设计保证了不侵犯任何站点的版权，也避免了内容同步带来的合规风险。

**Q: 链接失效后，ResourceHub 会如何处理？**
A: 系统后台每日执行一次健康检查任务，对状态码非 200 或返回重定向的链接标记为“异常”。异常链接仍保留在索引中，但会在前端界面以灰色标注并附上最近检查时间。批次管理员可通过 admin_guide.md 中描述的方式手动清理或更新失效记录。

**Q: 我能否将 ResourceHub 集成到我自己的应用中？**
A: 可以。ResourceHub 提供只读的 REST API，支持按批次、标签、关键字检索链接列表及单链接元数据。详细 API 端点、请求/响应格式与鉴权方式请参阅 developer_guide.md。所有 API 均返回 JSON 格式，方便前端或服务端程序调用。
