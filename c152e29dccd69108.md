# ResourceHub

ResourceHub 是一个面向开发者与技术研究人员的结构化外链资源聚合平台。该项目不生产内容，而是以人工筛选与自动化校验结合的方式，对互联网上优质的技术文档、教程、工具站、参考手册及社区讨论进行索引与分类管理。项目定位为技术决策支持层，帮助用户在浩瀚的信息海洋中快速定位高价值外部资源，减少重复搜索成本。

ResourceHub 适用于日常开发查错、技术选型调研、架构设计参考以及新领域入门学习等多种工作流。项目本身不存储任何第三方内容，仅提供链接与元数据描述，所有版权与内容责任归属原始发布方。当前批次为第 968/1241 批资源收录，累计已处理超过十二万条外链。

## 功能概览

**多维度资源分类**  
每个资源条目均按技术领域、内容类型、适用层级（入门/进阶/专家）进行标签化组织，支持快速过滤。

**链接可用性健康检查**  
系统每日定时对收录 URL 进行 HTTP 状态码探测，自动标记失效链接并生成报告，保证资源列表的鲜活度。

**元数据自动抽取**  
对目标页面进行可配置的元数据抓取，包括页面标题、描述、内容长度、最后修改时间等字段，用于排序与搜索。

**全文检索与高级过滤**  
基于 SQLite FTS5 或 Elasticsearch 可选后端，提供对资源标题、描述、标签的模糊搜索，并支持按域名、状态码、收录时间等维度组合筛选。

**收藏夹与自定义标签**  
用户可在本地或服务端（自部署）为资源添加自定义标签、备注与星标等级，构建个人知识索引。

**导入导出标准格式**  
支持 JSON、CSV 和 Markdown 列表格式的批量导入导出，便于与其他知识管理工具（如 Obsidian、Notion）进行数据交换。

## 应用场景

技术选型与方案调研  
当团队需要从多个开源库或商业方案中做出选择时，可通过 ResourceHub 快速检索相关对比文章、官方文档、社区评价及替代品列表，将调研时间从数天压缩至数小时。

新语言或框架的入门学习  
针对 Golang、Rust、Kubernetes 等新技术栈，项目收录了官方教程、最佳实践指南、示例项目及常见陷阱解析，为初学者提供一条由浅入深的阅读路径。

故障排查与错误码定位  
开发过程中遇到罕见错误信息时，通过资源库的全文搜索匹配已有社区讨论或官方 issue 记录，快速获取上下文与解决方案。

架构设计参考  
在微服务拆分、数据库选型、缓存策略设计等阶段，利用项目聚合的架构案例与性能评测报告，获取业界真实生产环境的经验数据。

自动化文档监控  
运维或文档工程师可配置定期扫描特定资源列表，当外部文档发生版本更新或链接变更时，系统发送通知，确保内部引用始终保持同步。

## 快速开始

以下步骤将在本地环境启动 ResourceHub 的基础实例，包含资源索引服务与 Web 管理界面。

```bash
# 克隆代码仓库
git clone https://github.com/resourcehub/resourcehub.git
cd resourcehub

# 安装项目依赖（使用 pip 与虚拟环境）
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 初始化 SQLite 数据库并导入示例资源
python scripts/init_db.py
python scripts/import_batch.py --batch 968 --source ./data/batch_968.json

# 启动开发服务器
python app.py --host 0.0.0.0 --port 8080
```

访问 http://127.0.0.1:8080 即可浏览当前批次资源列表。生产环境部署请参考 `docs/deployment.md`。

## 贡献指南

我们欢迎并鼓励社区贡献，包括但不限于新增优质资源链接、修正失效 URL、改进分类标签体系以及完善文档。所有贡献需遵循以下流程：

1.  Fork 本仓库至个人账户，并克隆到本地开发环境。确保本地 Python 版本与依赖符合安装要求，运行 `pytest` 确认所有测试通过后再进行修改。

2.  新建以 `feature/` 或 `fix/` 为前缀的分支，例如 `feature/add-batch-969-resources`。对于资源新增，请按照 `data/batches/batch_template.json` 格式准备条目，至少包含 `url`、`title`、`description`、`tags` 四个字段。对于代码修改，请保持与现有代码风格一致（遵循 PEP 8，使用 black 格式化）。

3.  提交变更时，使用清晰且有意义的 commit message，推荐 Conventional Commits 规范（如 `feat: add new resource parser for hnrtzs domain`）。每个 commit 应保持原子性，避免混合不相关改动。

4.  向本仓库主分支发起 Pull Request，并在 PR 描述中明确说明变更目的、测试覆盖情况以及是否影响现有功能。若新增了外部依赖，需同步更新 `requirements.txt` 与 `docs/developer` 中的说明。

5.  等待维护者进行 Code Review。对于资源新增类 PR，维护者会进行链接存活验证与内容相关性评估；对于代码类 PR，会检查测试覆盖率与性能影响。通过后由维护者合并入主分支，并自动触发 CI 流水线更新生产索引。

## 常见问题

**Q: 如果某个链接失效了，系统会如何处理？**  
A: 健康检查任务每天凌晨 2:00 对所有收录 URL 发送 HEAD 请求，连续 3 次失败（超时或非 2xx/3xx 状态）的链接将被标记为 `unreachable`，并在 Web 界面中置灰显示。标记后第 7 天若仍未恢复，则自动移入 `archive` 表并从主列表中隐藏。用户可通过后台「失效链接管理」面板手动复核或更新为新 URL。

**Q: 如何导入我自己整理的链接列表？**  
A: 项目支持通过 CLI 脚本 `scripts/import_batch.py` 导入标准 JSON 或 CSV 格式文件。您需要准备一个包含 `url`、`title`、`description`、`tags` 字段的数组，并指定 `--batch` 编号。导入前系统会进行格式校验与去重检查（基于 URL 标准化后的域名+路径）。导入完成后，新资源会自动触发一次实时健康检查。

**Q: 项目是否支持多用户与权限管理？**  
A: 当前开源版本为单机单用户模式，适用于个人开发者或小型团队内部部署。若需多用户支持（如团队协作标注、审核流程），建议参考 `docs/ops/multi-user.md` 中的二次开发方案，或使用社区提供的基于 Flask-Login 的扩展分支。生产环境中，我们推荐通过反向代理（如 nginx）配合基本认证（HTTP Basic Auth）实现简单访问控制。
