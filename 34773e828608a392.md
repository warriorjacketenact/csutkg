# LinkVault

LinkVault 是一个面向技术团队与内容创作者的轻量级外链资源汇总与导航系统。该项目旨在解决分散在各处的外部链接难以统一管理、检索与共享的问题，通过结构化的资源聚合能力，帮助用户将零散的 URL 转化为可复用、可检索、可协作的知识资产。

LinkVault 适用于需要频繁引用外部资料的技术文档编写者、开源项目维护者、课程讲师以及研究助理。其核心定位不是又一个书签管理器，而是一个以纯文本可版本化的方式存储、分类并呈现外链资源的静态站点生成工具。用户可通过命令行快速导入链接、自动提取元信息、生成分类索引，并输出为可供团队内部或公开访问的 HTML 仪表板。

## 功能概览

**批量链接导入**：支持从 CSV、JSON 或纯文本列表批量导入 URL，自动去重并校验可访问性。

**自动元信息提取**：对每个导入的链接自动抓取页面标题、描述与关键词，生成摘要索引。

**分类与标签系统**：允许用户为每条资源分配多级分类和自由标签，支持基于分类树的浏览。

**全文检索与过滤**：内置基于标题、描述、标签和 URL 路径的全文搜索，支持按分类和状态过滤。

**版本化快照**：每次导入或更新操作均生成时间戳快照，支持回滚到任意历史资源集状态。

**多格式导出**：支持导出为静态 HTML 仪表板、Markdown 清单或 JSON 结构化数据，便于嵌入其他系统。

**链接健康检查**：定期对已收录链接进行 HTTP 状态检查，标记失效或重定向的资源，并生成报告。

**权限分级访问**：支持基于 API 密钥的读写分离权限，适合团队协作场景，可配置公开只读与私有写入。

## 应用场景

**技术文档团队的外链统一管理**：技术文档中常包含大量参考链接、API 文档地址和第三方库主页。LinkVault 可为文档团队提供一个集中维护的外链仓库，文档编写者仅需引用仓库中的资源 ID 而非完整 URL，当外部链接变更时只需在 LinkVault 中更新一处即可全局生效。

**开源项目的外部资源聚合页**：开源项目维护者可使用 LinkVault 快速生成项目相关的生态资源导航页，包括教程、视频、插件列表、社区论坛等，方便新用户快速找到周边资源，同时降低维护单独 Wiki 页面的成本。

**课程与培训材料的链接索引**：讲师在准备课程材料时需引用大量论文、工具站点和数据源。LinkVault 可将这些链接按课程章节、主题或难度分级整理，生成学员可访问的在线资源目录，并支持在学期中动态增删。

**研究文献与数据源的组织**：研究人员在文献调研阶段收集的论文链接、数据集仓库和工具代码库可通过 LinkVault 进行系统化整理，配合标签和检索功能，在撰写综述或实验对比时可快速定位特定领域的所有相关资源。

## 快速开始

以下步骤帮助您在五分钟内完成 LinkVault 的安装并导入第一批链接。

```bash
# 克隆仓库
git clone https://github.com/linkvault/linkvault.git
cd linkvault

# 安装依赖
pip install -r requirements.txt

# 运行初始化设置
python linkvault.py init

# 导入示例资源列表（支持从纯文本文件导入）
python linkvault.py import --file sample_links.txt --category "starter"

# 启动本地预览服务器
python linkvault.py serve --port 8080
```

## 安装要求

LinkVault 基于 Python 3.9 以上版本开发，依赖以下核心库与工具：

| 依赖 | 必需 | 说明 |
|------|------|------|
| Python 3.9+ | 是 | 运行环境基础，建议使用 3.10 及以上版本以获得更好的类型支持 |
| pip 21.0+ | 是 | Python 包管理器，用于安装项目依赖 |
| requests 2.28+ | 是 | 用于发起 HTTP 请求以抓取链接元信息和进行健康检查 |
| beautifulsoup4 4.12+ | 是 | HTML 解析库，用于从目标页面提取标题和描述元数据 |
| click 8.1+ | 是 | 命令行接口框架，提供子命令和参数解析能力 |
| python-dotenv 1.0+ | 是 | 环境变量管理，用于配置文件路径和 API 密钥等敏感信息 |
| pytest 7.0+ | 否 | 单元测试框架，仅在开发环境中需要运行测试用例 |
| black 23.0+ | 否 | 代码格式化工具，仅在贡献代码时用于保持风格统一 |

## 项目结构

linkvault/
├── linkvault.py            # 主入口 CLI 程序，注册所有子命令
├── requirements.txt        # 生产环境依赖清单
├── setup.py                # 安装包配置，定义 entry_points 控制台脚本
├── config/                 # 配置目录
│   ├── default.yaml        # 默认配置项（存储路径、请求超时、并发数等）
│   └── schema.json         # 配置文件的 JSON Schema 校验定义
├── core/                   # 核心业务逻辑模块
│   ├── importer.py         # 导入引擎：支持 CSV/JSON/纯文本解析，含去重与校验
│   ├── indexer.py          # 索引引擎：构建倒排索引与分类树，管理标签映射
│   ├── snapshot.py         # 快照管理：创建、列表、回滚、清理历史版本
│   └── health.py           # 健康检查调度器：异步并发检测 HTTP 状态，生成失效报告
├── extractors/             # 元信息提取器插件目录
│   ├── base.py             # 抽象提取器基类，定义 extract 接口
│   ├── html_parser.py      # 基于 BeautifulSoup 的通用 HTML 元数据提取
│   └── open_graph.py       # Open Graph 协议专用提取器（og:title, og:description）
├── exporters/              # 导出格式插件目录
│   ├── html_export.py      # 生成响应式静态 HTML 仪表板，含搜索与分类过滤
│   ├── markdown_export.py  # 输出为按分类分组的 Markdown 清单
│   └── json_export.py      # 导出完整结构化 JSON，含元数据与时间戳
├── tests/                  # 测试套件
│   ├── unit/               # 单元测试：覆盖导入、索引、快照等核心函数
│   └── integration/        # 集成测试：端到端导入导出流程与 CLI 命令行为
├── docs/                   # 文档源码（Markdown 格式，由 MkDocs 构建）

## 贡献指南

LinkVault 欢迎社区贡献，所有贡献者需遵守项目行为准则。建议的贡献流程如下：

1. 查阅问题追踪器中的待办事项，选择未被认领且与自身技能匹配的任务。对于新功能提议或重大变更，建议先在问题追踪器中创建讨论议题，与维护者确认方向可行性后再投入开发。

2. 派生项目仓库至个人账户，在派生仓库中创建以功能或修复命名的主题分支。分支命名建议使用 feat/ 或 fix/ 前缀，例如 feat/add-notion-export 或 fix/health-timeout。

3. 开发过程中请遵循项目现有的代码风格（使用 black 进行自动格式化）并补充对应的单元测试。新增提取器或导出器插件时，需在 tests/unit/ 下添加相应的测试用例，确保代码覆盖率不低于 80%。

4. 提交前运行完整的测试套件确保无回归问题。提交信息请采用约定式提交格式，即使用 type(scope): subject 的结构，例如 feat(importer): support wildcard pattern matching。

5. 向主仓库的 main 分支发起拉取请求，并在描述中详细说明变更内容、测试结果以及用户文档的更新情况。拉取请求至少需要一位维护者的审阅批准后方可合并。

## 常见问题

**Q: LinkVault 与传统的书签管理器或收藏夹工具有何本质区别？**

A: 传统书签管理器通常与浏览器深度绑定，存储格式不开放，且缺乏批量操作、元信息提取和版本化能力。LinkVault 将所有数据以纯文本（JSON/YAML）形式存储在本地，完全可版本控制（Git），支持命令行批量导入导出，并且内置了链接健康检查与快照回滚，更适合技术团队作为基础设施而非个人工具使用。

**Q: 对于需要登录或反爬虫机制的页面，LinkVault 能否正常提取元信息？**

A: 默认情况下，LinkVault 仅发送标准的 GET 请求并解析 HTML 头部元数据，不执行 JavaScript，也不携带 Cookie 或会话状态。对于需要登录的页面，建议用户在配置文件中设置自定义请求头（如 User-Agent、Referer）或使用环境变量注入访问令牌，部分场景下可配合 --no-metadata 选项仅存储 URL 而不尝试抓取内容，以避免被目标站点拦截。

**Q: 如何处理收录链接中的失效或内容变更问题？**

A: LinkVault 内置了定期健康检查调度器，可配置为每日或每周对所有链接执行 HEAD 或 GET 请求并记录状态码。检查结果会生成失效链接报告，用户可根据报告决定是否更新或移除资源。对于内容变更但 URL 未变的情况，LinkVault 不支持自动检测内容差异，但可手动触发重新抓取以更新元信息快照。
