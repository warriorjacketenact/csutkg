# LinkVault 资源聚合引擎

LinkVault 是一个面向技术团队、内容创作者与运维人员的轻量级外链资源聚合与导航系统。项目定位于解决多源异构链接分散、难以统一管理与检索的问题，提供一套可私有化部署、可扩展的链接元数据采集与展示框架。

目标用户包括需要维护大量项目文档的研发团队、负责技术社区内容运营的编辑人员、以及希望构建内部知识导航体系的系统管理员。LinkVault 通过标准化链接条目模型与可插拔的元数据解析管道，将原始链接转化为可分类、可标签化、可全文检索的结构化资源条目，从而降低信息碎片化带来的认知负担。

## 功能概览

**结构化链接条目模型**：基于 YAML Front Matter 定义链接的标题、标签、所属领域、访问频次与状态备注，支持自定义扩展字段。

**多源批量导入接口**：提供 CSV 与 JSON 批量导入端点，支持将现有链接清单快速迁移至系统内，并自动触发元数据补全流程。

**元数据智能补全管道**：通过可配置的解析器链（支持 OpenGraph、DOM 标题提取、WHOIS 查询等），自动填充链接标题、描述与站点归属信息。

**标签化分类与多级目录绑定**：支持创建无限级标签树，并将链接与多个标签关联，实现按项目、按技术栈、按业务域的灵活组织。

**全文检索引擎集成**：基于 SQLite FTS5 或 Elasticsearch 适配器，提供对链接标题、描述、标签组合的高速检索，并支持模糊匹配与权重排序。

**访问状态健康检查**：内置定时任务，支持对已收录链接进行 HTTP 状态码探测，自动标记失效或重定向链接，并提供异常报告导出。

**权限分级与审计日志**：支持管理员、编辑者、访客三级角色划分，记录所有链接增删改操作，满足内部合规要求。

**响应式管理面板与公共门户主题**：提供两套界面主题，分别用于后台管理与前台公开导航展示，且均适配移动端布局。

## 应用场景

**研发团队内部文档导航**：团队可将分散在 Confluence、GitHub Wiki、语雀及各类监控面板中的链接统一录入 LinkVault，通过标签体系按项目或微服务模块归类，新成员入职时可快速获取完整上下文访问路径。

**技术社区内容聚合站**：技术博客或资讯站点可使用 LinkVault 作为外链资源后台，编辑人员批量导入参考链接后，系统自动提取文章标题与摘要，前台门户动态生成分类导航页，降低手动维护 HTML 目录的成本。

**运维故障处置知识库**：运维团队将常见故障排查手册、日志查询入口、云厂商状态页、内部工单系统链接集中托管，通过健康检查定时监控各入口可用性，当检测到异常时触发告警通知，缩短故障响应时间。

**企业合规审计资源索引**：法务与合规部门可将外部法规数据库、政府公告平台、行业标准文档来源链接统一收录，利用审计日志功能记录何人于何时访问或修改了特定资源链接，便于在合规审查中提供完整的访问追溯链。

## 快速开始

以下指令适用于 Linux 与 macOS 环境，Windows 用户可使用 WSL2 或 Git Bash 执行。

```bash
# 克隆项目仓库
git clone https://github.com/linkvault/linkvault.git
cd linkvault

# 安装依赖（使用 Python 3.10+ 与 pipenv）
pipenv install --deploy --ignore-pipfile
pipenv shell

# 初始化数据库并启动开发服务器
python manage.py migrate
python manage.py loaddata fixtures/initial_tags.json
python manage.py runserver 0.0.0.0:8000
```

生产环境部署请参考 `deploy/production` 目录下的 Docker Compose 与 systemd 示例文件。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---------|---------|------|
| Python | 3.10 及以上 | 核心运行环境，建议使用 3.11 或 3.12 以获得性能优化 |
| SQLite | 3.35 及以上 | 默认内置数据库，用于存储链接条目与标签关系；生产环境建议切换至 PostgreSQL |
| pipenv | 2023.x 及以上 | 依赖管理与虚拟环境隔离工具，用于锁定精确依赖版本 |
| Node.js | 18.x 及以上 | 仅在前端资源构建时必需，用于编译管理面板的静态资产 |
| Redis | 6.2 及以上 | 可选组件，用于缓存元数据解析结果与分布式任务队列后端 |
| Elasticsearch | 7.x 或 8.x | 可选组件，当启用高级全文检索引擎时替代 SQLite FTS5 |
| Nginx | 1.20 及以上 | 生产环境推荐的反向代理服务器，用于承载静态文件与负载均衡 |

## 贡献指南

欢迎社区开发者提交 Pull Request 或 Issue。为保障代码质量与协作效率，请遵循以下流程：

1. 阅读 `docs/development/contribution_agreement.md` 并签署开发者贡献者许可协议（CLA），首次提交时需在 PR 评论中注明已阅读并同意。

2. 从 `main` 分支切出功能分支，命名格式为 `feature/功能简述` 或 `fix/问题简述`，确保分支名称清晰反映变更内容。

3. 开发前运行 `pipenv install --dev` 安装开发依赖，并执行 `pre-commit install` 启用代码风格预检查钩子（基于 Black、isort、Flake8）。

4. 提交代码时编写符合 Conventional Commits 规范的提交信息，且每个提交应保持逻辑独立；新功能或修复必须附带对应的单元测试用例，测试覆盖率不得低于 85%。

5. 推送分支后在 GitHub 上创建 Pull Request，填写 PR 模板中的检查清单，并至少请求一位项目维护者进行代码审查。审查通过且所有 CI 流水线（测试、构建、安全扫描）均为绿色后方可合并。

## 常见问题

**问：系统是否支持 PostgreSQL 作为生产数据库？如何切换？**

答：完全支持。项目在 `backend/core/settings/production.py` 中预置了 PostgreSQL 适配配置。用户只需在部署环境的 `.env` 文件中设置 `DB_ENGINE=django.db.backends.postgresql`，并提供 `DB_NAME`、`DB_USER`、`DB_PASSWORD`、`DB_HOST` 和 `DB_PORT` 环境变量。系统将自动读取并切换数据库后端，初始迁移命令与 SQLite 版本完全一致。

**问：元数据智能补全管道在解析某些网站时失败或超时，应如何调整？**

答：解析管道默认超时时间为 10 秒，并启用指数退避重试策略（最多重试 3 次）。若特定站点频繁失败，可在 `backend/libs/parsers/base.py` 中为对应域名配置自定义超时或禁用特定解析器。此外，可启用 Redis 缓存层避免对同一 URL 重复解析，缓存 TTL 默认为 7 天，可根据更新频率在 `settings/base.py` 的 `PARSER_CACHE_TTL` 变量中调整。

**问：健康检查任务发现大量链接返回 5xx 或超时，如何区分是目标站点问题还是网络环境问题？**

答：健康检查模块支持配置代理检测模式。在 `.env` 中设置 `HEALTH_CHECK_PROXY=http://your-proxy:port` 可让请求经由代理发出；同时可配置 `HEALTH_CHECK_ALTERNATIVE_ENDPOINT` 备用探测端点（如 HEAD 请求）。检查结果会记录响应时间、状态码与异常类型，管理员可在管理面板的「健康报告」页面按错误类型分组查看，辅助定位是否为网络出口或目标服务器问题。
