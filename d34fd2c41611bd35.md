# RYMFSS — 技术资源与外部链接聚合索引系统

RYMFSS 是一个面向技术研究者、开源贡献者和内容策展人的外部资源链接聚合与索引管理平台。该项目专注于对分散在各类技术社区、官方文档、博客与工具站中的高质量外链进行系统性归集、分类与版本追踪，帮助用户在海量信息中快速定位有效资源。

项目定位为轻量级链接网关与元数据管理中间件，不存储资源实体内容，仅维护链接地址、快照指纹、可用性状态与分类标签。目标用户包括技术文档编写者、开源项目维护人、以及需要批量管理外部引用链接的工程团队。通过提供统一的访问入口与状态监控，RYMFSS 有效解决了链接失效、来源不可考、重复收录与检索困难等常见问题。

## 功能概览

- **链接聚合收录**：支持批量导入与手动添加两种方式，自动去重并生成内部短码映射关系，单次批次最高可处理 250 条外部链接。

- **可用性健康检查**：定时对已收录链接发起 HEAD 请求，记录响应状态码与延迟时间，标记失效链接并生成告警日志。

- **分类标签管理**：允许用户为每条链接添加多级标签（如 official-doc、blog、tool、community），支持按标签组合过滤检索。

- **快照指纹记录**：记录链接对应页面内容的 ETag 或最后修改时间，辅助判断资源是否发生更新，避免内容不一致。

- **访问统计与热力图**：统计每条链接的被引用次数、点击频次与时序分布，为资源价值评估提供数据支撑。

- **批量导出与迁移**：支持将链接列表导出为 JSON、CSV 或 Markdown 格式，方便迁移至其他文档系统或静态站点生成器。

- **权限分级控制**：支持多用户协同维护，区分管理员、编辑者与只读访客角色，操作日志全程审计。

- **Webhook 事件通知**：当链接状态变更或新增高危失效链接时，向配置的端点推送通知，便于及时响应。

## 应用场景

- **技术文档库的外部引用管理**：大型开源项目的文档体系通常包含数百个外部参考链接，RYMFSS 可集中管理这些链接，定期检测失效情况，避免文档中出现死链影响用户体验。

- **个人知识库的资源索引**：研究员或技术博主在积累资料时，可将散落在浏览器书签、笔记软件中的链接统一导入 RYMFSS，利用标签与搜索功能构建可复用的个人资源导航。

- **团队内部工具链的公共依赖声明**：开发团队可将项目依赖的第三方库文档、API 参考、镜像站等链接纳入 RYMFSS，作为团队知识库的基础设施组件，确保新成员快速获取权威资料。

- **静态网站的外链自动化维护**：使用 Hugo、VuePress 等工具生成的静态站点，可通过 RYMFSS 提供的 API 动态渲染外部链接列表，实现链接数据的集中更新而无需重新构建整个站点。

## 快速开始

以下操作基于 Ubuntu 22.04 / macOS 13 环境，确保系统已安装 Git、Node.js 18.x 与 pnpm。

使用 Git 克隆项目仓库至本地，进入项目目录并安装依赖，最后启动开发服务器。

```bash
git clone https://github.com/rymfss/rymfss-core.git
cd rymfss-core
pnpm install
pnpm run dev
```

生产环境部署建议使用 Docker Compose 方式，详见 `deploy/docker-compose.yml` 示例文件。

## 贡献指南

我们欢迎并感谢所有形式的贡献，包括但不限于代码提交、文档改进、问题反馈与功能建议。请遵循以下步骤参与项目协作。

1. 在 GitHub 上 Fork 本仓库，并基于 `develop` 分支创建您的特性分支，分支命名请使用 `feature/功能简述` 或 `fix/问题简述` 格式。

2. 本地开发时请确保通过全部单元测试与 ESLint 检查，提交前运行 `pnpm run test` 与 `pnpm run lint`。新增功能需附带对应的测试用例与文档更新。

3. 提交 Pull Request 前请将您的分支与上游 `develop` 分支同步，解决所有冲突。PR 描述中需清晰说明改动内容、影响范围与测试覆盖情况。

4. 核心模块的变更需要至少一位项目维护者进行 Code Review，确认通过后由维护者合并至 `develop` 分支。重大版本更新将经由 `main` 分支发布。

5. 文档贡献者可直接在 `docs/` 目录下修改 Markdown 文件并提交 PR，无需经过严格的代码审查流程，但需确保格式与现有风格一致。

## 常见问题

**Q: 系统如何处理链接失效或内容变更的情况？**

A: 系统内置了定时健康检查服务，默认每 24 小时对所有活跃链接发起 HEAD 请求。若连续三次检查返回非 2xx 状态码，该链接将被标记为 `unstable` 并触发 Webhook 通知。同时，系统会记录每次检查的响应头 ETag 与 Last-Modified 值，当这些指纹发生变更时，会在管理后台提示内容可能已更新，建议人工复核。

**Q: 能否将 RYMFSS 作为静态站点生成器的数据源使用？**

A: 可以。系统提供了只读的公共 API 接口，支持按标签、分类或关键词检索链接列表，并以 JSON 格式返回。您可在静态站点的构建脚本中调用这些接口，将外链数据注入页面模板。推荐配合构建缓存机制，避免频繁请求影响性能。对于大型站点，可配置 API 的限流阈值以满足高并发读取需求。

**Q: 如何迁移或备份已收录的链接数据？**

A: 管理员可通过后台面板的一键导出功能，将当前全量链接数据（含标签、状态、统计信息）导出为 JSON 或 CSV 格式。同时，系统每日凌晨自动生成数据库备份文件并存储于配置的持久化目录中。若需迁移至新实例，可使用 `scripts/seed.ts` 脚本导入之前导出的 JSON 文件，系统会自动处理外键关联与去重逻辑。
