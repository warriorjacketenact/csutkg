# LinkVault

LinkVault 是一个面向开发者、技术研究人员以及内容创作者的轻量级外链资源汇总与导航系统。该项目定位于帮助用户从海量分散的网络链接中快速筛选、分类并访问高价值的技术文档、视频教程、工具站点及社区讨论。LinkVault 不生产内容，而是致力于提供一套清晰、可扩展的链接组织方案，解决个人收藏夹混乱、团队知识共享效率低以及技术资源发现成本高的问题。通过简单的命令行接口和 Markdown 驱动的数据管理方式，用户可以便捷地维护自己的资源索引，并支持一键启动本地预览服务。

## 功能概览

**批量链接导入与清洗**：支持从文本文件、CSV 或直接粘贴的原始数据中自动识别并提取 URL，去除重复项，并依据域名或路径特征进行初步分类。

**多级分类与标签系统**：允许用户为每个链接分配多个标签（如 `#video`、`#tutorial`、`#api`），并创建无限层级的目录结构，实现从宏观领域到具体技术点的精细组织。

**全文检索与快速过滤**：内置基于标题、描述、标签和 URL 关键字的实时搜索功能，支持按分类、来源站点或添加时间范围进行组合过滤。

**本地静态站点生成**：通过内置的模板引擎，可将链接数据渲染为响应式 HTML 静态网站，适合部署到 GitHub Pages、Nginx 或内部服务器，便于团队共享。

**数据备份与迁移支持**：所有链接数据存储为纯文本 YAML 或 JSON 文件，用户可直接使用版本控制系统（如 Git）管理历史变更，并支持导入/导出为书签 HTML 或 Markdown 列表格式。

**链接可用性健康检查**：提供可配置的定时任务或手动触发命令，对存储的链接发起 HTTP 请求，检测状态码并生成失效链接报告。

## 应用场景

个人技术知识库构建：开发者可以将日常浏览中发现的优质博客、API 参考、开源项目地址集中收录到 LinkVault，配合标签和备注记录阅读心得，逐步形成个人专属的技术学习路径。

团队文档中心资源整合：技术团队可将分散在 Confluence、Notion、飞书文档以及外部 Wiki 中的参考链接统一导入 LinkVault，并按照前端、后端、运维、产品等维度建立共享分类，新成员入职时可快速了解团队常用工具栈。

在线课程与视频资料索引：教育机构或独立讲师可以将一系列教学视频、配套讲义链接和练习平台入口整合到一个 LinkVault 项目中，分发给学生作为课程导航门户，替代杂乱的传统书签文件。

技术峰会与黑客松活动资源站：活动组织方可将报名入口、日程表、直播链接、代码仓库和文档合集整理为 LinkVault 静态站点，在活动期间提供统一入口，活动结束后可直接归档保存。

## 快速开始

以下命令演示了从克隆项目到启动本地服务的完整流程。请确保系统已安装 Git 和 Node.js（v16 及以上版本）。

```bash
git clone https://github.com/your-username/linkvault.git
cd linkvault
npm install
npm run dev
```

执行 `npm run dev` 后，终端将输出本地预览地址（通常为 `http://localhost:3000`）。此时访问该地址即可查看包含示例数据的导航界面。若要使用自己的链接数据，请将原始 URL 列表按 `data/links.yaml` 格式替换，或运行 `npm run import -- --file ./my-links.txt` 进行批量导入。

## 项目结构

```text
linkvault/
├── bin/                            # 可执行命令行入口
│   └── cli.js                      # 主命令调度器，处理 import、serve、check 等子命令
├── src/
│   ├── core/                       # 核心业务逻辑
│   │   ├── loader.js               # 从 YAML/JSON 加载链接数据，负责解析与校验
│   │   ├── classifier.js           # 基于规则和标签的自动分类引擎
│   │   └── health.js               # 链接可用性检查器，支持并发请求与超时控制
│   ├── server/                     # 本地预览服务
│   │   ├── index.js                # Express 应用初始化与路由挂载
│   │   └── middleware.js           # 日志、静态文件缓存、错误处理中间件
│   ├── generator/                  # 静态站点生成器
│   │   ├── renderer.js             # 使用 EJS 模板渲染 HTML 页面
│   │   └── assets/                 # 内置 CSS 主题与 JavaScript 交互脚本
│   └── utils/                      # 通用工具函数
│       ├── file.js                 # 文件读写、目录创建工具
│       └── validator.js            # URL 格式验证、去重与路径规范化
├── data/                           # 用户数据存储目录（可被 git 忽略）
│   ├── links.yaml                  # 主链接数据库文件，用户手动编辑或由 import 生成
│   └── schema.json                 # JSON Schema 用于校验 links.yaml 结构
├── config/                         # 项目配置文件
│   ├── default.json                # 默认端口、分类规则、健康检查间隔等
│   └── custom.json                 # 用户自定义配置（覆盖 default.json）
├── docs/                           # 完整文档目录，包含用户手册与开发指南
├── test/                           # 单元测试与集成测试用例
│   ├── unit/                       # 针对 loader、classifier 的隔离测试
│   └── integration/                # 端到端测试，模拟命令行调用和页面渲染
├── .gitignore                      # 忽略 node_modules、data/backups 等目录
├── package.json                    # npm 依赖清单与脚本定义
└── README.md                       # 项目概览、快速开始与核心文档入口
```

## 贡献指南

1. 阅读项目 `docs/development.md` 了解架构设计、代码规范以及插件接口定义。所有新功能或修复应遵循现有的 ESLint 和 Prettier 配置，确保代码风格一致。

2. 在 Issue 列表中认领任务或提交新提案。对于较大改动（如新增数据源适配器、修改分类算法），请先在 Discussion 中发起设计讨论，获得核心维护者反馈后再着手编码。

3. 派生项目仓库，在本地功能分支（如 `feat/add-json-export`）上进行开发。提交信息请遵循 Conventional Commits 格式（类型(范围): 简短描述），并确保所有测试用例通过。

4. 编写或更新相应的单元测试及文档。对于新增的命令行选项或配置项，需同步更新 `docs/user-guide.md` 和 `docs/admin-guide.md` 中的相关章节。

5. 发起 Pull Request 到主仓库的 `main` 分支。描述中请关联相关 Issue，并附上手动测试的步骤或截图。PR 将通过 CI 流水线检查（包括测试覆盖率、构建成功率和链接有效性）。

## 常见问题

**Q: LinkVault 是否支持从浏览器书签（如 Chrome 或 Firefox 的 HTML 导出文件）直接导入？**

A: 目前版本通过 `npm run import -- --browser=chrome --file=./bookmarks.html` 支持解析 Chrome 和 Firefox 导出的标准书签 HTML 文件。系统会自动提取书签名称、URL 和文件夹层级，并转换为 LinkVault 的分类与标签结构。对于其他格式的书签文件，建议先转换为上述两种格式之一，或使用通用 CSV 导入。

**Q: 链接健康检查功能会频繁访问外部站点，是否会影响被检查站点的正常服务？**

A: 项目默认采用保守的并发策略（并发数 3，超时 10 秒，间隔 500 毫秒），并在请求头中标识 User-Agent 为 `LinkVault-HealthCheck/1.0`。检查任务默认每周执行一次，且仅对状态码非 200 的链接进行重试（最多 2 次）。用户可在配置文件中调整检查频率和并发数，建议在内部网络或非生产高峰期运行完整扫描。
