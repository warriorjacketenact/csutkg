# VueBest 外链资源聚合平台

VueBest 是一个面向 Vue.js 生态系统的外链资源导航与聚合平台，专为前端开发者、架构师和技术决策者设计。该项目系统性地收集、分类和索引了与 Vue.js 相关的优质外部资源，涵盖组件库、工具链、性能优化、设计系统和教学资源等多个维度，帮助开发团队在技术选型、项目启动和日常开发中快速定位高价值参考材料。

本平台定位为技术资源的中立索引枢纽，不直接托管代码或内容，而是通过人工筛选与社区反馈机制，维护一份高质量的外部链接清单。目标用户包括正在评估技术栈的团队负责人、希望拓展组件选择范围的资深开发者，以及需要系统化学习路径的初级工程师。通过本平台，用户能够将信息检索时间从数小时压缩至数分钟，显著提升研发效率。

## 功能概览

**按频道分类的资源索引** 每个外部链接均归属至特定频道（如组件、工具、性能、设计等），便于按领域浏览。

**原始链接直出模式** 所有资源链接均以原始 URL 呈现，不附加任何追踪参数或中间跳转页，确保访问路径透明。

**轻量级元数据标注** 每个链接条目附带简要说明标签，标注资源类型、维护状态和适用场景。

**无依赖静态部署** 平台本身为纯静态站点，无需后端服务或数据库，可托管于任何 Web 服务器或 CDN。

**社区提交与投票机制** 用户可通过 GitHub Issue 提交新资源链接，经维护者审核后纳入索引，社区投票帮助筛选优质内容。

**版本化资源快照** 每次索引更新均生成版本记录，方便回溯历史资源状态，应对链接失效问题。

**多维度搜索筛选** 支持按频道名称、资源类型、标签关键词进行组合筛选，快速定位目标资源。

**响应式浏览体验** 适配桌面、平板与移动设备，在各类屏幕尺寸下保持清晰的导航与阅读布局。

## 应用场景

技术选型与方案评审：技术负责人需要在短时间内评估 Vue.js 生态中可用的 UI 组件库或状态管理方案。通过 VueBest 的频道分类，可以横向对比多个同类资源的外部文档、GitHub 仓库和社区讨论，大幅缩短调研周期。

项目快速原型开发：前端开发者在启动新功能原型时，需要寻找现成的 Vue 组件解决通用需求（如表格、图表、表单验证）。VueBest 提供的链接索引可以直接导航至对应组件的官方文档和示例代码，加速原型搭建。

学习路径系统化构建：初级和中级前端开发者希望系统了解 Vue.js 生态的全貌，但缺乏发现优质学习材料的渠道。VueBest 汇总的教学资源、案例分析和性能优化指南，为开发者提供了一条结构化的知识获取路径。

社区贡献与知识共享：开源贡献者希望将自己维护的 Vue 组件或工具推荐给更广泛的受众。通过向 VueBest 提交资源链接，贡献者可以增加项目的曝光度，同时丰富平台的资源覆盖面，形成良性循环的共享生态。

## 快速开始

以下指令帮助您在本地环境中快速启动 VueBest 静态站点。

```bash
# 克隆项目仓库至本地
git clone https://github.com/vuebest/vuebest-index.git

# 进入项目根目录
cd vuebest-index

# 安装项目依赖（使用 npm 或 yarn）
npm install

# 启动本地开发服务器，默认监听端口 5173
npm run dev
```

执行完成后，在浏览器中访问控制台输出的本地地址即可预览站点。若需构建生产环境静态文件，请执行 `npm run build`，生成的产物默认位于 `dist` 目录。

## 安装要求

| 依赖 | 必需 | 说明 |
|------|------|------|
| Node.js | 是 | 版本要求 18.0 或更高，用于运行构建工具和开发服务器 |
| npm | 是 | 版本要求 8.0 或更高，或使用 yarn 1.22+ 作为替代包管理器 |
| Git | 是 | 用于克隆仓库和版本控制操作 |

| Vue 3 | 是 | 前端框架运行时依赖，版本 3.2 或更高 |
| TypeScript | 是 | 开发时依赖，版本 4.9 或更高，用于类型检查 |
| ESLint | 否 | 可选依赖，用于代码风格检查，推荐但非强制 |
| Playwright | 否 | 可选依赖，用于端到端测试，仅在 CI 环境中使用 |
| HTTP 服务器 | 否 | 生产部署时需要任意静态文件服务器（如 Nginx、Apache、Caddy） |

## 项目结构

```
vuebest-index/
├── index.html                         # 主入口 HTML，定义根挂载点与基础 meta
├── package.json                       # 项目依赖清单、脚本命令与版本信息
├── vite.config.ts                     # Vite 构建配置，含别名、代理与插件声明
├── tsconfig.json                      # TypeScript 编译选项，指定严格模式与目标版本
├── .eslintrc.cjs                      # ESLint 规则配置，基于 Vue 3 推荐规范
├── public/                            # 静态资源目录，不经过构建处理
│   └── favicon.ico                    # 站点图标文件
├── src/                               # 源代码主目录
│   ├── main.ts                        # 应用入口文件，初始化 Vue 实例
│   ├── App.vue                        # 根组件，定义整体布局与路由出口
│   ├── assets/                        # 静态资产，如样式、字体与图片
│   │   ├── styles/                    # 全局样式文件，含 CSS 变量与重置
│   │   └── images/                    # 图片资源，按功能模块分类存放
│   ├── components/                    # 可复用 Vue 组件目录
│   │   ├── common/                    # 通用组件，如按钮、输入框、卡片
│   │   ├── layout/                    # 布局组件，如头部、底部、侧边栏
│   │   └── resource/                  # 资源相关组件，含链接列表与筛选器
│   ├── composables/                   # 组合式函数，封装响应式逻辑与副作用
│   │   ├── useResourceFilter.ts       # 资源筛选与搜索逻辑
│   │   └── useChannelGroup.ts         # 频道分组与聚合逻辑
```

## 贡献指南

我们欢迎社区成员为 VueBest 贡献新的资源链接、改进现有文档或提交代码优化。请遵循以下步骤确保贡献流程顺畅。

**提交新资源链接** 通过 GitHub Issue 提交新资源建议，标题格式为 `[Resource Request] 资源名称 - 频道名称`，正文中需包含原始 URL、资源简介（100 字以内）以及推荐理由。维护者将在 5 个工作日内审核并回复。

**编辑现有条目** 若发现资源链接失效、分类错误或描述不准确，请提交 Pull Request 修改 `src/data/resources.json` 文件。PR 描述中需注明修改原因和验证依据，并确保 JSON 格式通过 `npm run validate` 检查。

**参与投票与讨论** 在 GitHub Discussion 板块中参与资源推荐投票，对已提交的候选链接进行 +1 或评论。高票资源将优先纳入下个版本索引。讨论时请遵守行为准则，保持友善和建设性。

**代码贡献与测试** 如需贡献前端功能或修复缺陷，请基于 `main` 分支创建 `feature/` 或 `fix/` 前缀的新分支。提交前运行 `npm run lint` 和 `npm run test:unit` 确保代码风格与测试通过。PR 需要至少一名维护者批准后合并。

## 常见问题

**问：VueBest 与 Vue.js 官方文档和生态系统有什么关系？**

答：VueBest 是一个第三方社区项目，与 Vue.js 核心团队及官方组织无直接隶属关系。本平台独立收集和索引互联网上的公开资源，旨在为 Vue 开发者提供便捷的导航参考。所有列出的链接均为外部资源，其内容和可用性由原始维护者负责。我们鼓励用户同时参考 Vue.js 官方文档和 Awesome Vue 等官方认可的资源列表，VueBest 可作为这些官方渠道的补充。

**问：如果某个链接失效了怎么办？**

答：链接有效性是 VueBest 维护的重点。我们会在每次版本更新前运行自动化链接检查脚本，标记失效链接。用户也可以通过 GitHub Issue 报告失效链接，标题请注明 `[Broken Link]`。维护者会及时核实并更新或移除失效条目。历史失效链接会记录在 `docs/archive/` 目录中，方便追溯。

**问：如何请求新增一个频道类别？**

答：频道分类体系旨在保持适度的粒度，避免过细或过粗。如果您认为现有分类不足以容纳某一类重要资源，请通过 GitHub Issue 提交频道新增请求，标题格式为 `[Channel Request] 新频道名称`。正文中需说明新频道的定位、预期容纳的资源类型以及至少 5 个候选链接示例。维护者将评估请求的合理性并组织社区讨论，通过后会在下个版本中新增频道。
