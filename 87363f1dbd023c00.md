# Tokyo Mirai Video Resource Aggregator

Tokyo Mirai Video Resource Aggregator 是一个面向技术内容创作者、开源社区运营者和分布式视频索引系统的外链资源归集工具。项目定位于将分散在 Tokyo Mirai 平台上的视频资源条目进行结构化整理，提供统一的访问入口与元数据管理能力，便于开发者构建自定义的视频聚合服务、内容分析管道或自动化归档工作流。

目标用户包括视频平台运维工程师、数据采集与清洗工程师、独立开发者以及需要定期获取 Tokyo Mirai 平台视频资源索引的研究人员。项目本身不存储任何视频文件，仅维护资源定位符集合，并通过脚本化的方式支持批量解析、可用性检测和分类导出，从而解决人工收集效率低下、链接零散难以追踪、重复性手工操作易出错等实际问题。

## 功能概览

**资源链接批量导入** 支持一次性导入数百条视频资源 URL，自动完成格式校验与去重，并提供导入日志记录。

**分类标签自动生成** 基于 URL 路径结构与文件名哈希特征，自动为每条资源打上日期、类型与来源标签，便于后续筛选。

**可用性定时检测** 内置 HTTP 状态码探测模块，可定期检查每个资源链接的可访问性，输出不可用链接报告。

**结构化元数据导出** 支持将资源列表导出为 JSON、CSV 与 YAML 格式，满足不同下游系统的数据接入需求。

**灵活的过滤查询** 提供基于日期范围、文件名模式、资源长度等条件的过滤查询接口，快速定位特定资源。

**增量更新机制** 支持新链接的增量追加与旧链接的批量替换，适应资源列表的动态变化场景。

**Docker 一键部署** 提供官方 Docker 镜像，支持容器化运行，降低环境配置成本。

**Prometheus 监控集成** 暴露监控指标端点，可接入 Prometheus 对资源检测任务执行情况进行可视化监控。

## 应用场景

视频资源归档与备份规划
数据工程师可将本项目的资源列表作为输入，编写自动化脚本定期拉取 Tokyo Mirai 平台的视频资源索引，生成归档清单，辅助制定增量备份策略，防止资源链接失效后无法追溯。

内容聚合站点的数据源构建
独立开发者或小型内容团队可利用本项目导出的结构化数据，快速构建自己的视频推荐站点或导航页面，避免重复编写爬虫逻辑，直接聚焦于前端展示与用户体验优化。

平台资源变化趋势分析
研究人员可对项目输出的历史资源列表进行时间序列分析，统计视频数量变化、文件命名规律、资源活跃度等指标，为平台运营决策提供数据支撑。

CI/CD 流水线中的资源检查环节
将本项目集成至持续集成流水线，在每次构建时自动检测指定资源列表的可用性，当不可用链接比例超过阈值时触发告警，确保生产环境引用的外链始终有效。

## 快速开始

以下命令演示了从代码仓库克隆、安装依赖到运行基础资源列表导出功能的完整流程。

```bash
# 克隆项目仓库
git clone https://github.com/tokyo-mirai/tm-resource-aggregator.git
cd tm-resource-aggregator

# 安装依赖（使用 Python 虚拟环境）
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 运行资源列表导出（默认输出 JSON 格式至 ./output/ 目录）
python cli.py export --input resources.txt --output ./output/ --format json
```

我们欢迎所有形式的贡献，包括但不限于代码提交、文档改进、问题报告与功能建议。请遵循以下步骤参与项目。

首先在 GitHub 上 fork 本仓库，并将 fork 后的仓库克隆到本地开发环境中。建议在开发前阅读 docs/developer-guide.md 了解项目架构与编码规范。

然后创建一个新的功能分支，分支名称应简洁描述本次变更内容，例如 feature/add-filter-by-duration 或 fix/http-timeout-error。请确保所有代码变更都附带相应的单元测试，并且测试覆盖率不低于现有水平。

完成开发后，提交代码并推送到远程 fork 仓库，随后通过 GitHub 界面发起 Pull Request 到主仓库的 main 分支。PR 描述中应清晰说明变更目的、实现方式以及测试结果。所有 PR 需要经过至少一名维护者的代码审查，并在 CI 流水线全部通过后方可合并。

若发现安全问题或重大缺陷，请直接通过项目邮件列表或 GitHub Issues 提交报告，避免在公开渠道讨论漏洞细节。文档类变更可直接提交 PR，无需关联 Issues。

## 常见问题

Q: 导入资源列表时提示格式校验失败，如何定位具体错误行？

A: 校验失败时程序会输出首个错误行的行号与内容。可使用 cli.py 的 --validate-only 参数单独运行校验模式，该模式会输出全部错误行而不执行导入。同时可开启 --verbose 获取更详细的调试信息，包括具体的校验规则命中情况。

Q: 可用性检测任务执行时间过长，如何优化？

A: 检测模块默认使用 10 个并发工作线程，可通过 --workers 参数调整并发数。若网络环境受限，建议降低并发数至 3 或 5。此外可通过 --timeout 参数设置单次请求超时阈值，默认 5 秒，可根据实际网络状况适当缩短。对于大规模资源列表，推荐使用 --incremental 模式仅检测上次未通过或新增的资源。

Q: 导出的 JSON 文件中包含哪些字段？

A: 每条资源导出为 JSON 对象，包含以下字段：url（原始链接字符串）、source（固定为 "tokyo-mirai"）、batch（批次号，格式为 YYYYMMDD）、tags（自动生成的标签数组）、last_check（最近一次可用性检测的时间戳，ISO 8601 格式）、status（可用性状态，值为 "available"、"unavailable" 或 "pending"）。完整字段定义见 docs/api-reference.md 中的 Exporter 模块说明。
