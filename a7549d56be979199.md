# WebIndex Core

WebIndex Core 是一个面向技术社区的开源外链资源汇总与导航系统，专为开发者、技术内容策展人以及开源项目维护者设计。该项目并非传统意义上的搜索引擎或爬虫框架，而是一个高度结构化的技术资源索引库，旨在解决技术文档分散、优质外链难以追溯以及项目依赖关系不透明等问题。通过将分散于各类技术博客、官方文档、视频教程以及社区讨论中的有价值链接进行集中管理与分类，WebIndex Core 帮助用户快速定位到特定技术栈的权威资料，从而显著降低信息检索的时间成本。

本项目适用于需要频繁查阅外部技术资源的人群，包括但不限于全栈工程师、架构师、技术文档撰写者以及开源项目贡献者。WebIndex Core 本身不提供视频转码、数据存储或代理服务，其核心价值在于构建了一个可维护、可扩展的链接索引框架，并附带了一批经过初步整理的参考链接资源。这批资源涵盖了 2026 年 6 月期间收录的多项技术参考视频与文档地址，用户可基于此框架自行增删、标注或二次分发链接数据。

## 功能概览

- **结构化链接索引**：提供统一的链接条目格式，支持按来源域名、收录日期、内容类型进行多维度筛选与排序。

- **批次化资源管理**：内置批次标识系统，每批资源均带有唯一的批次号与总量统计，便于用户追踪数据更新历史。

- **多协议透明支持**：索引系统原生支持 HTTP 与 HTTPS 混合协议链接，保留原始 URL 的协议头与域名格式，不进行任何自动转换或规范化改写。

- **轻量级部署架构**：无需独立数据库或缓存服务，基于纯静态 Markdown 与 JSON 数据文件即可运行，兼容主流静态站点生成器。

- **可定制分类体系**：允许用户根据自身需求调整资源分类层级，支持一级分类与二级标签的组合过滤。

- **快速检索接口**：提供基于文件名、路径关键词以及 URL 片段的内容过滤能力，适配 grep、jq 等命令行工具链。

- **版本化更新日志**：每次资源增删或链接变更均记录于变更日志文件中，确保索引数据的可审计性。

- **零外部依赖核心库**：核心索引解析模块不依赖任何第三方 Python 包或 Node.js 模块，仅使用标准库完成 URL 解析与格式校验。

## 应用场景

**技术文档站点外链整合**  
技术博客或开源项目文档站通常需要引用大量外部资源，如规范文档、API 参考或视频讲解。WebIndex Core 可作为这些站点的子模块，将分散的外链统一纳入索引体系，避免文档中直接散落裸链接，提升维护效率。

**离线资源镜像站前置索引**  
在网络受限的内网开发环境中，团队往往需要搭建离线镜像站。WebIndex Core 可提前整理好所需外链的元数据清单，供镜像工具批量拉取，从而减少人工整理链接的时间开销。

**技术培训课程资料包**  
技术培训讲师可将 WebIndex Core 作为课程资料包的组成部分，学员通过浏览索引即可获取所有课堂涉及的延伸阅读链接，无需手动从课件中复制粘贴。

**开源项目依赖关系溯源**  
当开源项目需要引用第三方库的文档或示例代码时，WebIndex Core 可以帮助维护者清晰记录每个外部引用的来源与更新时间，便于后续进行链接有效性检查与版本升级。

## 快速开始

以下步骤演示如何获取 WebIndex Core 项目代码并启动本地索引浏览服务。

```bash
# 克隆项目仓库
git clone https://github.com/webindex/core.git webindex-core

# 进入项目目录
cd webindex-core

# 安装本地预览工具（基于 Python 内置 HTTP 服务器）
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 启动索引预览服务（默认端口 8080）
python serve.py --port 8080
```

启动成功后，在浏览器中访问 `http://localhost:8080` 即可查看当前索引库的概览页面，并可通过侧边栏筛选特定批次的资源链接。

## 贡献指南

WebIndex Core 欢迎社区贡献者提交链接增补、分类优化以及文档改进。请遵循以下步骤参与贡献：

1. 复刻项目仓库至个人账号，并在本地创建功能分支。分支命名建议采用 `feature/batch-xxx` 或 `fix/category-yyy` 格式，以便于维护者识别变更意图。

2. 在 `data/batches/` 目录下新建或修改批次 JSON 文件。每个链接条目必须包含 `url`、`title`、`content_type` 以及 `tags` 四个字段。提交前请运行 `scripts/validate_urls.py` 确保所有 URL 格式符合预期。

3. 若新增资源链接，请同步更新 `data/indexes/by_domain.json` 与 `data/indexes/by_date.json` 索引文件。可使用 `scripts/generate_index.py` 自动生成索引，避免手动编辑出错。

4. 提交 Pull Request 时，请在描述中注明本次变更涉及的批次编号、新增链接数量以及是否影响现有分类体系。维护者将在 5 个工作日内进行审核。

5. 对于大规模链接增删（超过 50 条），建议先在 Issue 中提出变更提案，获得核心维护者确认后再执行批量修改，以避免索引冲突。

## 常见问题

**问：为什么每个链接的 URL 必须保留原始协议和域名格式，不能统一改成 HTTPS 或去掉 www？**  
答：WebIndex Core 的设计原则之一是"索引内容与原始来源严格一致"。部分视频资源或技术文档站点可能仅支持 HTTP 协议访问，或者其 HTTPS 证书配置不完整。自动改写协议会导致链接不可用，且违背了索引的可追溯性目标。因此，所有 URL 均按原始收录时的形态原样保留，由访问端自行决定是否升级协议。

**问：我如何快速检查本批次 250 个链接中哪些已经失效？**  
答：项目提供了 `scripts/check_availability.sh` 脚本，该脚本基于 curl 对每个链接发送 HEAD 请求，并记录非 200/301/302 状态的响应。运行方式为 `bash scripts/check_availability.sh data/batches/batch_026.json`，执行后将生成 `unavailable_links.log` 文件供查阅。

**问：WebIndex Core 是否支持自动抓取链接内容或生成网页快照？**  
答：不支持。WebIndex Core 严格限定自身为索引与导航工具，不包含任何爬虫、内容存储或快照生成功能。用户若需要离线访问或内容备份，应使用专门的镜像工具或浏览器插件，本项目不提供相关支持。
