# ResourceBridge

ResourceBridge 是一个面向技术团队与独立开发者的外链资源聚合与导航系统。该项目定位于解决分散在各处的优质技术文档、工具站点、学习材料难以统一管理和快速检索的问题，通过结构化的资源索引与轻量级的元数据标注，帮助用户在复杂的信息环境中高效定位所需内容。

ResourceBridge 适用于需要维护大量外链资源的个人知识库、团队文档站、开源项目推荐页等场景。项目本身不存储实际资源内容，而是以链接为第一公民，提供分类、标签、状态标记等辅助信息，便于后续扩展为自动化监测或推荐系统。当前批次为第 236/1241 批，共收录 250 个资源链接。

## 功能概览

**链接收录与去重**：支持批量导入 URL，自动检测重复条目，保留首次收录时间与来源标记。

**分类标签管理**：允许为每个链接添加多个自定义标签，支持按标签过滤和组合查询。

**状态监控标记**：可手动标记链接的可访问状态、更新频率、内容语言等属性，便于定期巡检。

**快速检索接口**：提供基于关键词、标签、时间范围的命令行查询接口，支持正则表达式匹配。

**导入导出兼容**：支持 JSON、CSV、Markdown 列表等多种格式的批量导入与导出，便于与其他工具链集成。

**批次追溯记录**：每个链接均记录所属批次号与收录序号，方便回溯数据来源与更新历史。

**只读只写分离**：核心数据库支持只读副本查询，写入操作通过事务保证一致性，适合多用户协作场景。

## 应用场景

技术团队内部知识库建设：团队可以将日常开发中遇到的优秀博客、API 文档、调试工具等链接统一收录至 ResourceBridge，并按项目或技术栈打标，新成员入职时可快速浏览团队推荐的外部资源。

开源项目 README 资源汇总：开源维护者可使用 ResourceBridge 管理项目依赖的相关生态链接，如社区论坛、示例项目、视频教程等，定期导出为 Markdown 列表嵌入文档，保持外链部分的结构化与可维护性。

个人学习路径组织：学习者可按阶段（入门、进阶、实战）或主题（前端、后端、运维）分类收藏在线课程、交互式练习平台、技术电子书等，通过标签筛选制定每周学习计划。

自动化监测前置层：运维或安全团队可将待监测的第三方服务状态页、版本发布日志、安全公告等链接统一录入，配合外部定时任务读取 ResourceBridge 的导出数据，实现集中化监测配置。

## 快速开始

以下命令演示了从克隆代码库到启动服务的完整流程。

```bash
# 克隆代码仓库
git clone https://github.com/resourcebridge/resourcebridge.git
cd resourcebridge

# 安装依赖（使用 pip 与虚拟环境）
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 初始化数据库并运行服务
python scripts/init_db.py --config config/default.yaml
python app.py --host 0.0.0.0 --port 8080
```

## 项目结构

```
resourcebridge/
├── app.py                      # 主入口，启动 HTTP 服务与 CLI 调度
├── config/
│   ├── default.yaml            # 默认配置：端口、数据库路径、批次参数
│   └── schema.json             # 配置字段校验规则
├── core/
│   ├── database.py             # SQLite 连接池与基础 CRUD 操作
│   ├── models.py               # 链接、标签、批次的数据类定义
│   └── exceptions.py           # 自定义异常类（重复收录、格式错误等）
├── cli/
│   ├── import_cmd.py           # 导入子命令：支持 json/csv 格式
│   ├── export_cmd.py           # 导出子命令：支持 md/json/csv 格式
│   └── query_cmd.py            # 查询子命令：标签过滤、关键词搜索
├── scripts/
│   ├── init_db.py              # 初始化数据库表结构与索引
│   └── migrate_v1_to_v2.py     # 版本迁移脚本（如有 schema 变更）
├── tests/
│   ├── test_database.py        # 数据库操作单元测试
│   ├── test_import.py          # 导入功能测试用例
│   └── fixtures/               # 测试用样本数据
├── docs/                       # 完整文档目录，含用户手册与 API 参考
├── logs/                       # 运行时日志输出目录（默认按日滚动）
├── requirements.txt            # 生产环境依赖列表
└── README.md                   # 项目概述与快速入门
```

## 贡献指南

1. 查阅问题跟踪器中的待办事项，选择未被认领的任务或提出新的改进建议。建议先通过议题讨论变更方案，避免重复劳动。

2. 派生代码仓库至个人账号，在派生副本中创建特性分支，分支命名采用 feat/描述 或 fix/描述 格式，例如 feat/tag-export。

3. 编写或更新对应的单元测试，确保测试覆盖率达到现有标准，所有测试用例在本地通过后方可提交。

4. 提交变更时遵循约定式提交规范，提交信息首行简明概括变更内容，正文补充设计思路或影响范围。

5. 向主仓库发起合并请求，并在请求描述中关联相关议题编号，等待维护者审阅。审阅通过后由维护者合并至主分支。

## 常见问题

问：导入链接时提示重复收录，但确认之前未导入过该链接，如何排查？

答：系统基于完整 URL 字符串做去重校验，请检查链接末尾是否包含多余斜杠或锚点。可以在导入命令中添加 --force 参数强制写入，或使用 --update 参数更新已有条目的标签与备注信息。

问：批量导出为 Markdown 列表时，能否自定义输出格式？

答：当前版本支持通过 --template 参数指定 Jinja2 模板文件路径，用户可自行设计列表样式，例如添加序号、分类标题或状态徽章。默认模板生成无序列表。

问：数据库文件能否迁移至 PostgreSQL 或 MySQL？

答：核心 SQL 语句遵循标准 SQL 规范，仅使用了少量 SQLite 特有函数（如 julianday）。迁移至其他关系型数据库时需调整日期函数与分页语法，项目提供了一份迁移指南文档 docs/ops/migration.md 供参考。
