# ResourceForge

ResourceForge 是一个面向技术研究者、教育工作者和内容创作者的轻量级外链资源聚合与导航工具。该项目定位于将分散于网络各处的优质技术文档、学术资料、开发指南和多媒体资源进行集中收录、分类索引和快速检索，帮助用户在特定技术领域内高效定位所需的外部学习材料。

ResourceForge 并非一个传统的爬虫或搜索引擎，而是一个由人工精选与社区贡献共同驱动的资源门户口袋。其核心目标用户包括：需要维护技术文档索引的团队负责人、整理教学素材的讲师、撰写技术博客的开发者，以及任何希望系统化组织个人学习书签库的进阶用户。通过 ResourceForge，用户可以将原本零散的书签文件夹转化为具备可读性、可维护性和可共享性的结构化资源清单。

## 功能概览

**结构化资源收录**：基于 YAML 前文配置，将外部链接按照预定义的分类体系（如文档、视频、代码库、工具站）进行标记和分组，支持多级标签系统。

**批量导入与去重**：支持从 CSV、JSON 或纯文本列表批量导入 URL，并自动进行语法规范校验与重复项检测，确保资源清单的整洁性。

**静态站点生成**：内置模板引擎，可将资源列表渲染为静态 HTML 站点，便于部署至 GitHub Pages、Netlify 等托管服务，实现零后端的外链门户发布。

**自定义元数据扩展**：每个资源条目除标题和 URL 外，可附加描述字段、维护状态（活跃/存档）、适用技术栈、阅读时长估算等自定义属性，满足不同领域的细粒度管理需求。

**多格式数据导出**：支持将资源列表导出为 Markdown 表格、JSON API 格式或 ATOM 订阅源，方便与其他文档工具或自动化流程集成。

**模糊搜索与快速过滤**：内置基于 Fuse.js 的前端模糊搜索引擎，用户可按标题、描述、分类或标签对全量资源进行实时筛选，提升大列表下的查找效率。

**资源可用性检查**：提供可选的链接存活巡检功能（需配置定时任务），定期对已收录 URL 发起 HEAD 请求，标记异常链接，辅助维护者清理失效资源。

## 应用场景

**技术团队内部知识库索引**：开发团队可使用 ResourceForge 维护一份“团队推荐技术读物”或“常用第三方服务文档”列表，统一团队的技术信息获取入口，避免重复寻找资料的时间浪费。

**开源项目辅助文档站**：开源项目维护者可将项目中引用的所有外部依赖文档、API 参考、社区教程等链接通过 ResourceForge 构建为独立的“外部资源附录”，随项目文档一同发布，提升项目的完整性与易用性。

**在线课程辅助材料汇总**：讲师或在线教育内容创作者可将视频课程中提及的全部参考链接、延伸阅读材料、作业相关工具站点通过 ResourceForge 整合为单页资源目录，作为课程网站的一个固定栏目供学员随时查阅。

**个人知识体系的公开展示**：技术博主或数字研究员可将个人长期积累的领域书签（例如“机器学习必读论文”或“云原生故障排查手册”）通过 ResourceForge 整理为对外开放的精选列表，建立个人专业品牌。

## 快速开始

以下指令适用于 Linux / macOS / Windows（WSL 或 Git Bash）环境。

```bash
# 克隆项目仓库至本地
git clone https://github.com/resourceforge/resourceforge.git
cd resourceforge

# 安装项目依赖（基于 Node.js 22 LTS）
npm install

# 构建资源索引并启动本地开发服务
npm run build
npm start
```

执行 `npm start` 后，服务默认在 `http://localhost:3000` 启动。访问该地址即可看到示例资源列表页面。若要生成可供部署的静态文件，执行 `npm run generate`，输出目录为 `./dist`。

## 安装要求

| 依赖 | 必需 | 说明 |
|------|------|------|
| Node.js | 22.x LTS 或更高 | 项目运行时与构建工具链基础，推荐使用 nvm 管理版本 |
| npm | 10.x 或更高 | 随 Node.js 一同安装，用于管理项目依赖包 |
| Git | 2.30 或更高 | 用于克隆仓库以及后续通过 pull 方式更新资源数据 |
| YAML 解析器（内置） | 否 | 项目通过 js-yaml 库内置支持，无需额外安装 |
| HTTP 客户端（可选） | 否 | 仅当启用链接存活巡检功能时，需保证系统网络可访问目标 URL |

## 贡献指南

ResourceForge 欢迎社区贡献。无论是提交资源收录建议、报告缺陷还是完善文档，请遵循以下流程：

1. 在 GitHub 仓库的 Issue 区域搜索现有议题，确认没有重复提交。若为新议题，请使用提供的模板明确标注议题类型（增强功能、缺陷修复、资源请求或文档更新）。

2. 从仓库主分支（main）创建新的功能分支，分支命名格式为 `feature/简要描述` 或 `fix/问题编号`。本地开发时，请运行 `npm run test` 确保现有测试用例全部通过，并为新增功能补充对应的单元测试。

3. 对于资源清单的更新（即 `data/resources.yml` 文件），请严格按照 `docs/data-schema.md` 中定义的字段格式提交变更，并附带说明每个新增资源的收录理由或来源上下文。

4. 提交 Pull Request 前，请确保代码风格与项目 ESLint 配置一致，并执行 `npm run build` 验证构建流程无报错。PR 描述中需清晰列出变更内容、影响范围以及相关的 Issue 编号。

5. 项目维护者会在收到 PR 后的 5 个工作日内进行审阅。如有修改意见，会以评论形式反馈。合并后，您的贡献者信息将自动列入项目主页的贡献者名单（需在 PR 描述中确认贡献者署名方式）。

## 常见问题

**问：ResourceForge 是否会自动抓取每个 URL 对应的页面内容或截图？**

答：不会。ResourceForge 定位为轻量级外链索引工具，只存储用户提供的元数据（标题、描述、分类等），不主动抓取目标页面的正文内容或生成缩略图。这样做旨在保持项目简单、快速，并避免潜在的版权或网络负载问题。如需链接可用性提示，可配置可选的存活巡检功能，但也仅检测 HTTP 状态码，不解析页面内容。

**问：我能否将 ResourceForge 生成的静态站点部署到支持 HTTPS 的平台上？**

答：完全可以。ResourceForge 生成的 dist 目录包含纯静态 HTML、CSS 和 JavaScript 文件，可部署至任何支持静态文件托管的服务（如 Vercel、Netlify、Cloudflare Pages 或任何标准 Web 服务器）。部署后，站点本身的访问协议（HTTP 或 HTTPS）完全由托管平台决定，与 ResourceForge 内部配置无关。项目中收录的外部链接保持其原始 URL 协议不变，不会自动改写。

**问：资源清单中的 URL 是否可以包含非 learning-society.net 域名的链接？**

答：可以。ResourceForge 本身不限制外部链接的域名来源。本项目 README 资源列表章节所展示的 URL 均为示例批次数据，实际使用中，您完全可以在 `data/resources.yml` 中收录任意合法域名的链接，包括但不限于 GitHub、MDN、arXiv、YouTube 等。项目仅对 URL 格式进行基本的协议（http/https）校验，不进行域名白名单过滤。
