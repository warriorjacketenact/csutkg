# StreamGate 统一资源索引系统

StreamGate 是一个面向技术团队与内容运营者的轻量级外链资源聚合与导航系统，专注于将分散于多源头的技术文档、数据流接口、动态配置端点与知识库入口，通过结构化索引方式统一呈现。本系统不替代原有数据源，而是提供稳定、可审计、可快速检索的资源定位层，适用于内部开发者门户、技术文档中心或数据管道监控面板的补充路由层。

目标用户包括运维工程师、技术文档维护者、数据平台开发人员以及需要频繁引用外部技术参考的架构师。StreamGate 解决的核心问题在于：当组织内存在大量动态生成或按批次交付的资源端点时，手工记录与口头传递极易导致链接失效、权限混淆与版本错位，本系统通过固定的索引规范，将所有端点按批次、类别与状态进行固化登记，确保引用路径的可复现性。

## 功能概览

**统一资源登记** 提供标准化的资源条目录入格式，每个资源可附加状态标签、归属批次与最后验证时间，确保索引与实际内容保持同步。

**多维度分类视图** 支持按资源类型、业务域、数据格式或地域标签进行筛选，方便不同角色的用户快速定位所需端点。

**批量导入与校验** 内置链接可达性检查工具，可定期对已登记资源发起 HEAD 请求，标记异常状态并生成告警日志。

**版本历史追踪** 记录每次资源条目的新增、修改与删除操作，支持回溯特定时间点的索引快照，满足审计合规要求。

**只读镜像输出** 可将索引数据渲染为静态 Markdown 或 JSON 格式，便于嵌入现有文档站点或 CI 流程中作为外部参考。

**访问统计摘要** 展示各资源被引用或点击的频次，帮助团队识别高频依赖与潜在废弃端点。

## 应用场景

**技术文档中心嵌入** 在组织内部的技术文档站点侧边栏或附录中，挂载 StreamGate 生成的资源索引页面，使开发者无需离开文档即可获取相关依赖服务地址、Schema 定义文件或示例数据集链接。

**数据管道配置管理** 数据工程师在编写 ETL 任务或调度脚本时，将 StreamGate 中登记的资源端点作为配置源，通过环境变量或配置文件引用索引中的 URL，实现数据源地址的统一变更管理。

**新员工入职参考** 为新加入的团队成员提供一份经过整理的资源地图，涵盖开发环境常用工具入口、日志查询界面、监控面板与代码仓库模板，降低初期摸索成本。

**外部合作方接口交付** 当与第三方合作方共享 API 清单时，通过 StreamGate 生成一份只读的、带版本号的资源列表，避免逐一发送邮件造成的版本混乱与权限过度暴露。

## 快速开始

以下操作将引导您在本地环境中部署 StreamGate 索引服务的基础版本。

```bash
# 克隆项目仓库
git clone https://github.com/streamgate/streamgate-core.git
cd streamgate-core

# 安装依赖（使用 pip 管理 Python 后端与验证脚本）
pip install -r requirements.txt

# 执行本地索引服务（默认监听 8080 端口）
python app.py --port 8080 --batch 497
```

启动后访问 http://localhost:8080 即可查看当前批次的资源索引面板，系统会自动加载 `data/batch_497.json` 中的初始条目。

## 项目结构

```text
streamgate-core/
├── app.py                      # Flask 应用入口，注册路由与启动服务
├── requirements.txt            # Python 依赖清单，固定版本号
├── config/
│   ├── default.yaml            # 默认配置（端口、校验间隔、日志级别）
│   └── batch_497.yaml          # 第 497 批次的专属分类映射规则
├── data/
│   ├── batch_497.json          # 当前批次资源条目的主存储文件
│   └── archive/                # 历史批次 JSON 存档目录
├── src/
│   ├── loader.py               # 加载并解析 JSON 与 YAML 数据源
│   ├── validator.py            # 执行资源可达性 HEAD 校验
│   ├── renderer.py             # 将索引数据渲染为 HTML / Markdown
│   └── watcher.py              # 定时触发校验任务并记录状态变更
├── tests/
│   ├── test_loader.py          # 测试数据加载与格式转换逻辑
│   ├── test_validator.py       # 模拟 HTTP 响应测试校验器
│   └── test_renderer.py        # 检查静态输出是否包含必填字段
├── templates/
│   └── index.html              # 默认首页模板，展示资源卡片列表
├── static/
│   ├── style.css               # 基础响应式样式与打印样式
│   └── script.js               # 前端筛选与状态标记交互脚本
└── docs/                       # 完整文档目录，包含用户与运维手册
```

## 贡献指南

1. 在 GitHub 上 fork 本项目，并克隆到本地开发环境，确保使用 Python 3.9 及以上版本创建独立虚拟环境。

2. 在 `data/` 目录下新建或修改对应批次的 JSON 文件，遵循 `schema/entry.schema.json` 中定义的字段规范，包括 `id`、`url`、`title`、`category` 与 `status`。

3. 运行 `pytest tests/` 执行全部单元测试，确保新条目未破坏加载与渲染流程，同时使用 `src/validator.py --batch 497` 手动校验新增资源可达性。

4. 提交 pull request 前，请更新 `CHANGELOG.md` 记录本次变更摘要，并在 `docs/user-guide.md` 中补充相关使用示例（若涉及新增功能）。

5. 等待项目维护者进行 Code Review，合并后将自动触发 CI 重新生成静态镜像并同步至发布分支。

## 常见问题

Q: 批次 ID 的生成规则是什么，是否必须按顺序使用？
A: 批次 ID 由系统根据导入时间戳自动计算，格式为 `YYYYMMDD` 加当日序号，例如 `497` 代表第 497 个登记批次。使用者无需手动生成，但可沿用已有批次号进行增量更新。

Q: 如果某个资源端点失效，系统如何处理？
A: 校验器会在每次定时任务中标记失效状态，并在前端界面显示警示图标。连续三次校验失败后，系统会将该条目移入 `deprecated` 状态，但仍保留历史记录以便追溯。

Q: 是否支持将索引数据导出为 PDF 或 Excel 格式？
A: 当前版本原生支持 Markdown 与 JSON 导出，PDF 和 Excel 可通过扩展工具（如 pandoc 或 pandas）二次转换。我们计划在后续版本中集成直接导出功能。
