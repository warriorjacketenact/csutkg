# RY MFS External Resource Aggregator

RY MFS External Resource Aggregator 是一个面向技术研究、数据分析和内容挖掘场景的轻量级外链资源汇总与导航系统。该项目定位于为开发者、数据分析师、安全研究员以及信息收集人员提供结构化的外部资源索引能力，通过统一的访问前缀与路径规范，将分散在互联网各处的有价值资源进行集中管理与快速检索。

项目本身不存储任何第三方内容，仅提供引用与路由机制。用户可通过该项目快速定位到目标资源，并结合自定义标签与分类体系完成资源的再组织。该项目适用于需要频繁切换外部数据源、维护研究素材库或构建私有导航页面的技术团队与个人研究者。

## 功能概览

**统一路由前缀管理**：所有外部资源均通过 /v/ 前缀进行映射，支持路径参数解析与重定向，便于集成到现有网关或代理服务中。

**资源分类标签系统**：每个外部链接可关联一个或多个分类标签，支持按标签过滤、检索以及批量导出，方便构建垂直领域的资源集合。

**批量导入与去重机制**：支持从文本文件、CSV 或 JSON 格式批量导入链接列表，自动检测重复条目并生成冲突报告，减少人工维护成本。

**访问统计与健康检查**：记录每个外链的访问次数、最后访问时间以及响应状态码，定期执行可用性探测，标记失效链接以便及时清理或更新。

**只读镜像与缓存模式**：对于频繁访问的资源链接，可开启只读缓存以降低目标服务器的请求压力，同时支持配置缓存过期策略。

**开放 API 接口**：提供 RESTful API 用于查询、新增、删除和更新资源条目，便于与其他内部系统或自动化脚本进行集成。

**用户权限分级管理**：支持管理员、编辑者和访客三种角色，管理员可管理全部资源与用户，编辑者可增删改资源，访客仅拥有查询与访问权限。

**自定义元数据扩展**：每条资源记录支持自定义元数据字段，例如来源描述、入库时间、关联项目编号等，满足不同业务场景的个性化需求。

## 应用场景

**技术研究文献收集**：研究人员在阅读技术博客、论文或官方文档时，可将零散的重要链接通过该系统统一收录，并按照研究主题添加分类标签，后续撰写综述或报告时可快速检索引用来源。

**渗透测试与安全审计**：安全团队在执行外部资产收集或信息枚举任务时，可通过该系统集中管理目标域名、IP 相关的情报页面、漏洞数据库条目以及第三方威胁情报源，提高协同工作效率。

**数据爬虫与采集调度**：数据工程师可将频繁访问的数据接口、公开数据集页面以及备用数据源链接注册至该系统，结合访问统计功能分析调用频率与可用性，从而优化采集任务的路由策略。

**团队知识库外链整合**：企业内部的 Wiki 或知识管理平台可将该项目作为外链中间层，当外部资源地址发生变更时，仅需在系统中更新映射关系，无需逐一修改所有文档中的引用链接。

## 快速开始

以下操作演示如何在本地环境中快速启动该项目。

```bash
# 克隆代码仓库至本地
git clone https://github.com/example/ry-mfs-aggregator.git
cd ry-mfs-aggregator

# 安装项目依赖
pip install -r requirements.txt

# 执行数据库迁移并启动服务
python manage.py migrate
python manage.py runserver 0.0.0.0:8000
```

服务启动后，访问 http://localhost:8000 即可进入管理界面。默认管理员账号为 admin，密码在首次启动时由系统生成并打印至控制台日志中。

| systemd | 234 及以上 | Linux 系统下的服务守护与管理工具 |

## 常见问题

**问：外链资源无法访问或响应超时如何处理？**

答：系统内置的异步健康检查任务会每隔 24 小时自动探测所有注册链接的可用性。若某链接连续三次探测失败，该条目会被标记为异常状态并记录至日志。管理员可在后台管理界面查看异常列表，手动确认是否为临时故障或资源已迁移。若资源永久失效，可通过编辑功能更新目标地址或直接删除该条目。

**问：是否可以导入其他来源的链接集合？**

答：支持通过管理后台的批量导入功能上传 CSV 或 JSON 格式文件。CSV 文件需包含 url、category、tags 三列，JSON 文件需为对象数组且每个对象包含相同字段。导入过程中系统会自动校验 URL 格式并过滤无效条目，最终生成导入报告反馈成功与失败的具体数量。

**问：该项目的 API 接口是否支持跨域请求？**

答：默认情况下 API 接口不开启跨域资源共享，以确保内部使用场景的安全性。若需要从前端页面直接调用 API，可在配置文件中将 cors.enabled 设置为 true，并指定 allowed_origins 白名单。生产环境建议通过 Nginx 反向代理统一处理跨域策略，避免直接暴露内部端口。
