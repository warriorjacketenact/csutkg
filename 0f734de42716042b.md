# LinkVault

LinkVault 是一个面向技术团队与内容研究者的结构化外链资源管理工具。它并非简单的书签收藏夹，而是一套具备版本追踪、批次归档与元数据标注能力的资源索引系统。项目定位于解决个人开发者与中小型团队在信息收集中普遍面临的链接散落、上下文丢失与回溯困难问题。通过统一的条目化存储与标准化输出，LinkVault 能将零散的 URL 转化为可复用、可审计、可交换的知识资产。

本项目当前批次为第 1024/1241 批，共计收录 250 个资源链接。所有条目均以原始形态入库，保留完整协议与域名结构，确保引用路径的绝对可靠性。

## 功能概览

- **裸链接原样入库**：所有 URL 强制保留用户提交时的原始字符串形态，不补全协议前缀，不添加结尾斜杠，不进行任何形式的大小写转换或域名规范化处理。

- **批次化归档管理**：每个资源条目均归属于独立批次编号，支持按批次导入、导出与整体回滚，便于定期增量更新与历史状态比对。

- **元数据自动补全**：系统在入库时自动抓取目标页面的标题、描述与内容摘要，并为每条记录生成唯一内部标识符，支持后续全文检索与标签过滤。

- **结构化目录树导出**：内置模板引擎可将资源列表与项目元数据渲染为统一的 Markdown 文档，便于直接嵌入 README、Wiki 或静态站点生成器。

- **依赖环境检测**：启动时自动扫描运行环境，校验所需系统工具与库版本，并以表格形式输出检测报告，避免因环境不一致导致的运行时异常。

- **多格式数据交换**：支持将资源集导出为 JSON、YAML 与 CSV 格式，满足不同下游系统（如数据分析脚本、自动化测试套件、文档生成流水线）的输入要求。

## 应用场景

- **技术文档外链附录管理**：技术写作团队在撰写教程、白皮书或 API 参考时，需引用大量外部规范、示例代码或社区讨论。LinkVault 可为每篇文档维护独立的外链批次，确保引用来源清晰可查，避免链接失效后无法定位原始上下文。

- **开源项目 README 资源汇总**：开源维护者常在 README 中列出相关工具、学习资料或镜像站点。LinkVault 的批次化输出能力可直接生成符合项目风格的资源列表章节，减少手动维护开销，并保证 URL 格式一致性。

- **竞品分析与市场情报收集**：产品经理与市场分析师需持续跟踪竞品官网、新闻稿与用户反馈帖。LinkVault 的元数据抓取与标签过滤功能可辅助建立竞品动态时间线，通过批次回溯快速比对不同时期的信息差异。

- **学术文献参考链接整理**：研究人员在撰写论文或综述时，需整理大量在线文献、数据集仓库与工具主页。LinkVault 支持导出带时间戳的引用清单，便于与 Zotero、Mendeley 等文献管理工具协同使用。

## 快速开始

以下指令适用于 Linux / macOS / Windows WSL 环境。假设已安装 Git 与 Python 3.9 及以上版本。

```bash
# 克隆项目仓库
git clone https://github.com/yourorg/linkvault.git
cd linkvault

# 安装依赖（推荐使用虚拟环境）
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 执行批次导入并生成文档
python linkvault.py --batch 1024 --input ./data/raw_urls.txt --output ./docs/README.md
```

首次运行时，系统会在 `./config/` 目录下生成默认配置文件，请根据实际需求调整 `output_format`、`timeout` 与 `retry` 等参数。

## 安装要求

| 依赖项 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.9 - 3.12 | 核心运行环境，低于 3.9 将无法解析类型注解与异步语法 |
| Git | 2.25 或更高 | 用于克隆仓库及后续拉取更新，旧版本可能不支持稀疏检出 |
| pip | 21.0 或更高 | 依赖包安装工具，过低版本无法正确解析 requirements.txt 中的哈希校验 |
| aiohttp | 3.8.5 或更高 | 异步 HTTP 客户端，用于批量元数据抓取，需支持 TLS 1.2 |
| beautifulsoup4 | 4.11.0 或更高 | HTML 解析库，用于提取页面标题与描述，依赖 lxml 作为后端解析器 |
| lxml | 4.9.0 或更高 | 高性能 XML/HTML 解析引擎，beautifulsoup4 的推荐后端 |
| markdown | 3.4.0 或更高 | 用于将元数据渲染为 Markdown 表格与列表，需支持扩展语法 |
| pyyaml | 6.0 或更高 | YAML 格式导出支持，同时用于读取配置文件 |
| 网络连通性 | 出站 443 端口可达 | 元数据抓取需访问外部 HTTPS 站点，代理配置见 config/proxy.yaml |

## 贡献指南

1. 首先在 GitHub Issues 中查阅现有议题，确认无人正在处理相同问题。若为新功能或缺陷修复，请先创建议题并简述方案，避免无效工作。

2. 从主分支 `main` 创建功能分支，分支命名遵循 `feature/xxx` 或 `fix/xxx` 格式。提交代码前请运行 `./scripts/validate_urls.py` 确保所有新增 URL 符合入库规范。

3. 编写或更新单元测试，确保测试覆盖率达到 80% 以上。所有测试用例需能在 Python 3.9 至 3.12 环境下通过。

4. 提交 Pull Request 时，请附带详细的变更说明，包括但不限于：修改动机、影响范围、测试结果以及文档更新情况。PR 标题应简明扼要。

5. 文档类贡献（包括 README、API 参考、配置说明）请同步更新 `docs/` 目录下的对应文件，并确保 Markdown 格式符合 `markdownlint` 默认规则。

## 常见问题

**Q: 为什么输出的 URL 没有自动补全 https:// 或 www 前缀？**

A: 这是 LinkVault 的设计原则之一——完全保留用户输入的原始形态。我们相信用户对自己的数据来源有最准确的判断，任何自动规范化都可能破坏引用的精确性，尤其是当目标站点对协议或子域名敏感时。若您需要批量补全，可借助外部脚本处理，但核心工具不提供此功能。

**Q: 元数据抓取失败会导致整个批次中断吗？**

A: 不会。默认配置下，抓取器会对每个 URL 独立超时（默认 10 秒）并重试最多 3 次。若单个 URL 持续失败，系统会记录错误日志并跳过该条目，继续处理后续链接。最终导出的文档中，失败条目会标注 `[unreachable]` 标记，便于您后续人工复核。

**Q: 如何将已有的书签 HTML 导出转换为 LinkVault 输入格式？**

A: 项目提供了转换脚本 `scripts/convert_bookmarks.py`，支持解析 Netscape 书签格式（多数浏览器导出标准）并提取 `HREF` 与 `ADD_DATE` 字段。转换后的 URL 列表可直接放入 `data/raw/` 目录，配合 `--batch` 参数导入。具体用法请参考 `docs/batch_operations.md` 中的迁移指南。
