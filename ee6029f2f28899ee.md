# ResourceWatch 技术资源聚合平台

ResourceWatch 是一个面向开发者与技术研究人员的开放式外链资源聚合系统，专注于对互联网上的技术文档、工具链、实践指南与参考手册进行结构化收录与索引。该项目定位于解决个人开发者与团队在信息检索过程中面临的来源分散、质量参差、回溯困难等问题，通过统一的外链管理机制，将高价值资源以可追踪、可分类、可共享的方式呈现。

项目不提供内容存储或代理服务，仅作为外部资源的导航层与组织层，帮助用户以最低成本建立自己的技术参考库。ResourceWatch 适用于日常开发查阅、技术选型调研、团队知识沉淀、项目文档外链备份等多种工作流。

## 功能概览

**外链统一收录** 提供标准化的外链入库格式，所有资源以原始 URL 形式存储，保证链接可溯源、可复查。

**分类索引机制** 支持按技术领域、资源类型、适用层级等维度对链接进行标签化分类，便于快速筛选定位。

**快照元数据记录** 每条链接可附加录入时间、收录批次、简要描述等元信息，方便后期维护与失效检测。

**只读访问模式** 所有收录链接以只读方式对外展示，不修改目标站点内容，确保资源原始完整性。

**批量导入导出** 支持批次化导入链接集合，适用于项目迁移、镜像部署或定期同步场景。

**状态监控接口** 提供链接可访问性检查的辅助功能，帮助管理员识别失效或变更的外部资源。

## 应用场景

技术团队内部知识库构建 团队可将日常开发中参考的官方文档、博客文章、API 手册通过 ResourceWatch 统一收录，形成共享的外链池，减少重复搜索成本，同时避免成员各自收藏导致的信息孤岛。

技术选型与调研资料整理 在进行框架对比、工具选型或方案评估时，将相关的评测文章、性能报告、社区讨论链接集中收录，便于后续编写调研报告时快速引用和回溯原文。

项目文档外链管理 项目 README、设计文档、运维手册中常需引用大量外部参考资料。使用 ResourceWatch 对外链进行集中管理，可避免文档中散落过长 URL，提升文档可读性，同时便于统一更新失效链接。

个人开发环境资源备份 开发者可将日常高频访问的技术资源（如语言规范、库文档、调试工具页）通过本系统建立本地索引，配合浏览器书签工具使用，形成轻量级个人知识检索入口。

## 快速开始

以下步骤适用于 Linux / macOS / Windows WSL 环境。

```bash
# 克隆项目仓库
git clone https://github.com/resourcewatch/resourcewatch.git
cd resourcewatch

# 安装依赖（使用 pip 与虚拟环境）
python -m venv venv
source venv/bin/activate  # Windows 下使用 venv\Scripts\activate
pip install -r requirements.txt

# 初始化本地数据库并启动服务
python manage.py migrate
python manage.py runserver 0.0.0.0:8000
```

服务启动后，访问 http://localhost:8000 即可进入资源列表主界面。默认管理后台路径为 /admin，初始管理员账号需通过 `python manage.py createsuperuser` 创建。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---------|---------|------|
| Python | 3.9 至 3.12 | 核心运行环境，低于 3.9 不支持类型注解，高于 3.12 需验证第三方库兼容性 |
| Django | 4.2.x LTS | Web 框架，用于提供管理界面与 API 接口 |
| SQLite | 3.35+ | 默认数据库，用于存储链接元数据与分类信息，生产环境可切换至 PostgreSQL |
| requests | 2.31.x | 用于链接可用性检查与状态码获取 |
| pytest | 7.4.x | 单元测试与集成测试框架，仅在开发环境中使用 |
| git | 2.25+ | 用于版本管理与代码克隆，部署时需确保 git 可访问私有仓库 |

## 贡献指南

ResourceWatch 遵循开源社区协作规范，欢迎任何形式的贡献。请按以下流程操作：

1. 复刻仓库并创建功能分支。从主仓库 fork 到个人账户后，基于 main 分支创建新的特性分支，分支命名采用 `feature/描述` 或 `fix/描述` 格式。

2. 编写或修改代码时遵循项目现有风格。Python 代码须通过 flake8 与 black 格式化检查，提交前运行 `pytest` 确保所有测试用例通过。新增功能需附带对应的单元测试。

3. 提交拉取请求前更新文档。若本次改动涉及用户可见的功能变化，请同步更新 docs 目录下的对应文档，并在 PR 描述中说明改动范围与测试情况。

## 常见问题

Q: 收录的链接如果失效了怎么办？
A: ResourceWatch 本身不托管外部内容，链接的有效性由目标站点维护。项目提供了链接状态检查脚本 `scripts/check_links.py`，可定期运行以识别失效链接。管理员可通过后台标记失效链接或将其移出活跃列表。

Q: 如何导入自定义的链接集合？
A: 使用 `scripts/batch_import.py` 工具，传入 CSV 或 JSON 格式的链接文件即可。文件需包含 url、category、description 字段。具体格式示例请参考 docs/user-guide.md 中的导入章节。

Q: 生产环境是否可以更换数据库？
A: 可以。项目使用 Django ORM，支持 PostgreSQL、MySQL、SQLite 等主流数据库。在生产环境中推荐使用 PostgreSQL，只需在 settings/prod.py 中修改 DATABASES 配置并安装对应的数据库驱动即可。
