# ResourceLink Collective

ResourceLink Collective 是一个面向开发者、技术研究人员与内容创作者的轻量级外链资源聚合与导航系统。该项目定位于将分散于各处的优质技术文档、工具站点、参考手册与社区讨论进行结构化整理，通过统一的索引机制与分类体系，帮助用户快速定位所需的外部资源，减少重复搜索与信息过滤成本。

项目采用纯静态生成方案，不对外部链接进行代理或缓存，仅提供元数据标注与分类导航能力。目标用户包括需要维护大量外部引用链接的技术文档撰写者、开源项目维护人、技术社区运营人员以及日常需要频繁查阅多源技术资料的一线研发工程师。

## 功能概览

**多级分类索引**：支持按技术领域、资源类型、适用人群等多维度对链接进行归类，每一条外链均可挂载多个标签，便于交叉筛选。

**元数据标注系统**：每个资源条目可记录标题、描述、维护状态、语言、最后检查时间等字段，为链接质量评估提供数据基础。

**批量导入与校验**：支持从 CSV、JSON 及纯文本列表批量导入链接，并提供自动化可达性检测与失效链接标记功能。

**静态站点生成**：基于配置文件一键生成完整的静态 HTML 导航页面，无需后端服务即可部署到任意 Web 服务器或 CDN。

**自定义主题与布局**：提供多套预设主题模板，并支持通过 CSS 变量与布局配置调整页面展示风格，适配不同项目品牌需求。

**链接状态监控**：集成定时检查任务，可配置检查周期与通知方式，及时获知外部资源访问异常情况。

**全文检索支持**：基于标题、描述与标签构建轻量级倒排索引，支持模糊查询与多关键字组合搜索。

**访问统计看板**：记录各链接的点击频次与趋势，辅助识别高频资源与沉寂资源，指导链接库维护决策。

## 应用场景

**技术文档库的外部参考管理**：当项目文档中需要引用大量第三方规范、教程或工具主页时，可通过 ResourceLink Collective 统一维护这些引用，并在文档生成时自动嵌入最新的链接列表，避免硬编码导致的链接漂移问题。

**开源项目生态资源导航页**：开源项目维护者可为自己的项目搭建专属的生态资源导航站，集中展示相关插件、扩展工具、学习资料与社区渠道，降低新贡献者的上手门槛。

**企业技术团队的知识库索引**：企业内部技术团队可将常用内外部系统入口、技术规范文档、培训视频链接等统一纳入索引体系，配合标签与检索能力，显著提升团队信息获取效率。

**技术社区的内容聚合导航**：技术社区运营方可利用该工具快速搭建社区推荐资源板块，按照专栏、作者、主题等维度组织外部优质内容，丰富社区内容生态。

## 快速开始

以下操作基于 Linux/macOS 环境，Windows 用户可使用 WSL 或 Git Bash 执行。

```bash
# 克隆代码仓库
git clone https://github.com/resourcelink/collective.git
cd collective

# 安装依赖（项目使用 Python 3.10+ 与 pipenv 管理依赖）
pip install pipenv
pipenv install --dev

# 进入虚拟环境并执行初始化构建
pipenv shell
python build.py --init
python build.py --generate
```

执行完成后，生成的静态站点位于 `_site` 目录下，可直接使用任意 HTTP 服务器启动预览：

```bash
python -m http.server 8000 --directory _site
```

访问 `http://localhost:8000` 即可查看导航首页。

## 常见问题

**问：项目是否会对收录的外部链接进行内容缓存或代理转发？**

答：不会。本项目仅存储链接的元数据（标题、描述、分类、标签等），不缓存任何外部页面的内容，也不提供代理转发服务。所有链接均以原生形式展示，用户点击后直接跳转至原始地址。链接可达性检查仅发送 HEAD 请求验证响应状态，不抓取页面正文。

**问：如何批量导入已有的大量链接数据？**

答：项目提供了 `scripts/import.py` 辅助脚本，支持从 CSV 文件和 JSON 数组格式批量导入。具体用法请参考 `docs/link-management.md` 中的批量操作章节。导入前建议先使用 `--dry-run` 参数进行模拟运行，确认字段映射正确后再执行实际导入。对于超过 1000 条链接的批量操作，建议分批次执行以避免超时。

**问：链接状态检查发现失效链接后如何处理？**

答：检查结果会输出到日志文件 `logs/check.log` 中，同时可在生成的导航页面中查看标记为失效的链接列表。推荐的处理流程为：首先人工访问确认是否临时不可达，若为永久失效则从 `config/links.yaml` 中移除或替换为有效地址；若为临时故障可暂不处理，系统会在下一检查周期自动重试。对于长期失效链接，项目支持配置自动标记规则，可设定连续失败 N 次后自动添加 deprecated 标记。
