# VOD Resource Aggregator

VOD Resource Aggregator 是一个面向视频内容运营团队、技术研究人员与数据分析师的高效外链资源整合工具。该项目将大量分散的媒体资源标识符统一收录于可检索、可维护的结构化体系之中，用于支撑自动化内容采集、播放源验证、质量监控与数据分析等核心业务场景。

本项目并非传统意义上的播放器或视频网站，而是一个资源定位符管理与调度系统，目标用户为具备基础技术能力的运营人员、爬虫开发者、数据中台工程师以及学术研究者。通过统一的资源索引框架，用户可以快速获取已归类的外部视频源节点，并将其用于自身业务逻辑之中。项目当前收录第 1003 至 1241 批次的资源链接，共计 250 个有效节点标识，所有链接均保留原始格式与域名结构，确保与上游数据源严格一致。

## 功能概览

**批量资源收录** - 支持一次性导入数百个外部视频资源链接，并按批次、编号、域名自动归类，便于后续批量处理与调度。

**原始链接透传** - 所有收录的 URL 保持原始协议、域名、路径与查询参数，不做任何自动补全或规范化改写，确保与上游采集系统完全兼容。

**多维度筛选检索** - 基于资源标识符特征、域名来源、批次编号等元数据字段提供快速过滤能力，帮助用户在海量链接中定位特定节点。

**结构化元数据导出** - 支持将资源列表导出为 JSON、CSV 或纯文本格式，方便对接外部自动化脚本、监控系统或数据分析流水线。

**健康状态标记** - 集成简单的可用性探测接口，可对每个资源节点进行 HTTP 状态校验，并记录响应时间与状态码，辅助运维人员排查失效节点。

**版本增量更新** - 每一批次资源以独立版本号进行管理，用户可清晰追踪每批次的添加时间、链接数量与变更说明，便于回滚与审计。

**权限分级控制** - 内置基于角色的访问控制雏形，区分只读查看、数据导出、资源编辑与管理员四级权限，适用于团队协作场景。

## 应用场景

**视频源质量巡检** - 运维团队可定期通过本系统获取全部资源链接列表，配合自动化检测脚本对每个节点进行连通性测试与播放协议验证，快速发现超时、404 或协议不兼容的异常节点，生成日报并推送告警。

**内容采集管道数据对接** - 数据中台团队可将本系统作为资源索引上游，通过 API 批量拉取指定批次的链接，注入分布式爬虫队列或流式处理框架，用于后续内容元数据抓取与结构化存储。

**学术研究与网络测量** - 研究人员可利用本项目提供的结构化资源清单，开展视频内容分发网络性能对比、地域可用性分析、运营商缓存策略评估等学术课题，所有链接均保留原始标识符便于复现实验。

**运营活动素材池管理** - 运营人员可按批次整理活动所需的外部视频素材链接，通过本系统的分类标记功能为不同活动项目分配专属资源子集，避免链接混乱与重复引用。

## 快速开始

以下指令适用于 Linux / macOS / Windows WSL 环境，需提前安装 Git 与 Node.js 18 及以上版本。

```bash
# 克隆项目仓库至本地
git clone https://github.com/example/vod-resource-aggregator.git
cd vod-resource-aggregator

# 安装项目依赖（使用 npm 或 yarn）
npm install

# 启动本地开发服务器，默认监听端口 3000
npm run start
```

访问 http://localhost:3000 即可进入资源管理面板。首次启动时系统会自动加载 `data/batch_1003_1241.json` 中的初始资源条目。

## 常见问题

Q: 导入的链接为什么不做协议补全或规范化处理？

A: 本项目定位为原始数据透传与索引系统，而非用户代理或播放器。上游数据源可能依赖特定的协议标识（包括 http 与 https 混合）以及路径格式，任何自动改写都可能导致下游系统无法正确解析或访问。因此我们严格保留用户提供的原始 URL 字符串，仅做字符串级别的存储与展示。如需统一协议，用户可在导出后自行使用脚本处理。

Q: 如何更新或新增一批资源链接？

A: 管理员可通过 API 接口 `POST /api/batches` 创建新批次，并随请求体传入 JSON 格式的链接数组。系统会自动校验每一条 URL 的格式合法性，并生成唯一的批次编号。此外，您也可以将 JSON 文件放置于 `data/` 目录下，然后执行 `npm run import -- --file=your-file.json` 命令完成批量导入。导入完成后可通过 `GET /api/batches` 验证新增记录。

Q: 健康检查功能是否会频繁请求外部服务器，导致对方封禁 IP？

A: 健康检查服务默认采用指数退避策略，每次检查间隔不低于 60 秒，且并发请求数限制为 5 个。检查仅发送 HTTP HEAD 请求以获取状态码，不会下载实际媒体内容。用户可在配置文件中调整 `CHECK_INTERVAL` 与 `CONCURRENCY` 参数以降低请求频率。对于敏感目标源，建议在检查请求头中设置合理的 User-Agent 并遵守 robots.txt 规则。
