# LinkVault Resource Aggregator

LinkVault 是一个面向技术研究者和内容策展人的轻量级外链资源汇总平台，专注于对分散在网络各处的垂直领域信息源进行结构化收录与分类导航。项目本身不生产原始内容，而是通过人工筛选与自动化校验相结合的方式，构建高可读性的外链索引目录，帮助用户在特定主题下快速定位有价值的参考页面。

项目定位为“技术资源的入口层”，目标用户包括开源社区维护者、技术文档撰写人、数据分析师以及需要持续追踪多源信息的产品经理。LinkVault 不对收录链接做深度爬取或内容转储，而是以稳定、清晰、可审计的链接清单为核心交付物，配合版本记录与可用性检测机制，降低外链失效带来的信息损耗风险。当前批次为第 109/1241 批，共收录 250 个资源链接，覆盖影视技术、媒体制作、行业资讯等多个子方向。

## 功能概览

- **按批次导入与去重**：支持批量粘贴原始 URL 列表，系统自动识别重复条目并生成批次编号（如当前 109/1241），方便后续增量更新。

- **自定义分类标签**：每个链接可关联一个或多个分类标签（如 /film/ 目录下的内容统一归入“影视技术”类），支持用户级分类体系扩展。

- **可用性定时检测**：后台调度任务每 24 小时对已收录链接发起 HEAD 请求，标记状态码异常或超时的条目，并在管理面板高亮提示。

- **Markdown 清单导出**：一键将当前批次或全量数据导出为符合 README 规范的 Markdown 列表，保留原始 URL 格式，不做协议补全或域名改写。

- **结构化的文档导航**：内置“文档 - 资源 - 项目”三层目录，用户可通过表格视图快速定位安装指南、API 说明、贡献规范等模块。

- **版本化变更日志**：每次新增或移除链接时自动生成变更记录，包含操作时间、操作人（本地模式下为设备标识）和受影响条目数量。

- **搜索与过滤**：基于链接路径关键字（如 .html 后缀、目录名）进行快速模糊匹配，支持按检测状态（正常/异常）过滤列表。

## 应用场景

**技术文档中的参考链接附录**：开源项目维护者在撰写手册或论文时，可将 LinkVault 生成的链接清单直接作为附录嵌入，确保所有外链格式统一且可追溯，避免手工编辑导致的协议遗漏或大小写错误。

**媒体资源追踪工作台**：影视制作公司或数字内容团队可将 /film/ 目录下的系列链接导入 LinkVault，结合自定义标签区分“素材来源”“技术规范”“发行参考”等类别，每周批量检测失效链接，及时更新合作方接口。

**个人知识库的外链管理插件**：使用 Obsidian、Logseq 等本地知识管理工具的用户，可将 LinkVault 作为外链中转站，定期导入浏览器书签导出的 HTML 文件，输出规范化列表后嵌入笔记，减少手动整理时间。

**开源社区资源聚合页构建**：技术社区运营者利用 LinkVault 的多批次管理能力，为不同活动主题（如黑客松、技术沙龙）创建独立的资源批次，生成专属 README 页面，供参与者一站式获取所有相关外链。

## 快速开始

以下步骤适用于 macOS / Linux / Windows WSL 环境，确保系统已安装 Git 和 Node.js 18.x 或以上版本。

```bash
# 克隆项目仓库
git clone https://github.com/your-org/linkvault.git
cd linkvault

# 安装依赖（使用 npm）
npm install

# 启动开发服务器，默认监听 3000 端口
npm run dev
```

若需要执行一次性链接导入并生成当前批次的 Markdown 清单，可使用 CLI 模式：

```bash
npm run import -- --batch 109 --source ./data/raw_urls_109.txt --output ./docs/batch_109.md
```

导入完成后，输出文件位于 `./docs/batch_109.md`，可直接复制到项目 README 的“资源列表”章节。

| SQLite 3（内置） | 否 | 项目内嵌 SQLite 驱动，无需额外安装 |
| Docker 20.x（可选） | 否 | 若使用容器化部署方式则需安装 |

│   └── exporter.js                  # 导出为 Markdown / JSON / HTML 格式
├── data/                            # 数据存储目录（SQLite 数据库与批次原始文件）
│   ├── linkvault.db                 # 主数据库文件，存储链接、标签、批次、检测记录

├── docs/                            # 项目文档目录
│   ├── user-guide.md                # 用户使用手册

## 贡献指南

1. 分叉本仓库至个人账号，克隆到本地开发环境。建议在 feature/xxx 分支上进行修改，避免直接操作 main 分支。

2. 如需新增链接批次，在 `./data/` 目录下创建 `raw_urls_<批次号>.txt` 文件，每行一个 URL，确保 URL 不包含空格或不可见字符。随后运行 `npm run import -- --batch <批次号>` 执行导入。

3. 提交变更前，运行 `npm run test` 确保所有单元测试通过。若新增了分类标签或修改了导出模板，请同步更新 `./docs/user-guide.md` 中的对应说明。

4. 提交 Pull Request 时，请在描述中注明本次新增或移除的链接数量、涉及的分类目录，以及是否触发过可用性检测。PR 至少需要一名项目维护者审核。

5. 若发现收录链接失效或内容严重偏离分类主题，请先在 Issues 中标记 `broken-link` 或 `misclassified` 标签，等待维护者确认后再提交移除或重新分类的 PR。

## 常见问题

**Q：为什么所有 URL 都保留原始格式，不统一补充 https:// 或去掉 www？**

A：LinkVault 的设计原则之一是“最小化信息篡改”。用户提供的 URL 可能包含特定的协议偏好（例如某些内网环境仅支持 http）、子域名依赖（如 www 与不带 www 可能指向不同服务）或路径大小写敏感性。自动补全或改写会引入不可预期的访问失败，因此项目严格按原始字符串存储和输出，仅在检测时做标准化处理。

**Q：可用性检测显示某个链接超时，但浏览器中可以正常打开，是什么原因？**

A：检测器使用 HEAD 请求且默认超时时间为 5 秒，某些服务器可能对 HEAD 方法响应较慢或直接拒绝，但 GET 请求正常。此情况属于误报，用户可手动在管理面板将该链接标记为“已验证”，系统将在后续检测中跳过该条目。同时建议在检测配置中调整超时阈值或启用 GET 回退策略。

**Q：如何处理同一批次中重复的 URL？**

A：导入器默认开启去重功能，基于完整 URL 字符串进行精确匹配。若发现重复条目，系统会在日志中输出警告，并仅保留首次出现的记录。重复的 URL 不会计入批次总数，也不会写入数据库。如需强制保留重复项，可在导入命令中添加 `--no-dedup` 标志。
