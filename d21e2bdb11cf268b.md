# CQBJ Resource Navigator

CQBJ Resource Navigator 是一个面向技术研究人员与内容聚合场景的轻量级外链资源导航系统。项目定位于将分散在多个子目录下的结构化资源条目进行集中索引与分类展示，帮助用户快速定位特定主题的参考资料、工具站点或文档入口，适用于个人知识库管理、团队技术栈导航、项目文档外部引用整理等场景。

本项目不提供具体业务功能实现，而是围绕 URL 资源的元信息管理构建标准化的目录树与引用清单。通过统一的资源列表输出格式，解决多源链接分散、难以追溯、缺乏上下文说明的痛点。目标用户包括文档撰写者、技术运营人员、开源项目维护者以及需要批量管理外部链接的研发团队。

## 功能概览

**批量资源导入**：支持以固定格式条目批量录入外部链接，自动识别 URL 结构与来源域名。

**分类目录索引**：根据资源所属业务领域或文件路径前缀，自动生成多级分类视图，支持按字母序与时间序排列。

**原样链接输出**：严格保留用户提供的 URL 原始格式，不添加协议前缀或域名标准化处理，确保目标地址可直达。

**资源状态标记**：为每条资源标注收录批次编号与入库时间戳，便于追踪资源的新旧程度与更新频率。

**项目结构展示**：通过 ASCII 目录树可视化呈现项目文件组织方式，帮助贡献者快速理解代码布局。

**多格式导出**：支持将资源列表导出为 Markdown 表格、纯文本清单或 JSON 结构化数据，适配不同下游工具链。

**变更历史记录**：维护资源增删改的操作日志，支持回滚至历史版本，保障资源清单的可靠性。

**跨平台兼容**：基于标准 Python 3 环境构建，无需额外图形界面依赖，可在 Linux、macOS、Windows 终端中直接运行。

## 应用场景

技术文档外部参考整理：在撰写系统设计文档或技术方案时，需要引用大量外部规范、论文或开源仓库链接。CQBJ Resource Navigator 可帮助将这些分散的引用链接统一管理，并为每条链接附加业务标签与收录说明，避免链接丢失或遗忘。

团队共享资源库建设：研发团队内部常积累各类工具官网、镜像站、API 文档入口等实用链接。通过本项目构建共享的资源导航文件，新成员可快速获得完整的工具链访问路径，减少沟通成本。

开源项目 README 外链维护：开源项目在 README 中通常需要列出依赖项目、参考文档、社区论坛等外部链接。随着项目发展，这些链接可能频繁变动。使用本项目的结构化资源清单，可以清晰追踪每条链接的添加原因与更新记录，便于 reviewer 审核。

个人书签系统迁移：从浏览器书签导出的大量 URL 往往缺乏分类层次。本项目提供基于目录树的重分类机制，可将扁平化的书签列表重构为层级分明的资源索引，并生成可版本控制的纯文本清单。

## 快速开始

以下步骤适用于 Linux 与 macOS 环境，Windows 用户建议使用 WSL 或 Git Bash。

```bash
# 克隆项目仓库至本地
git clone https://github.com/example/cqbj-resource-navigator.git
cd cqbj-resource-navigator

# 安装项目依赖（推荐使用虚拟环境）
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 运行资源索引构建脚本，生成当前批次的资源清单
python build_index.py --batch 1129 --output resources_1129.md
```

执行完成后，将在项目根目录下生成包含全部 250 个资源链接的 Markdown 文件，同时输出统计摘要信息。

## 项目结构

```
cqbj-resource-navigator/
├── build_index.py            # 主入口脚本，解析资源条目并生成输出文件
├── config.yaml               # 全局配置文件，包含批次参数、输出路径与分类映射
├── requirements.txt          # Python 依赖清单，锁定所需第三方库版本
├── src/                      # 核心源码目录
│   ├── parser/               # URL 解析与标准化模块
│   │   ├── url_reader.py     # 从输入源读取原始 URL 列表，支持文件与标准输入
│   │   └── category_mapper.py # 根据路径前缀或关键词将资源映射到分类标签
│   ├── formatter/            # 输出格式化模块
│   │   ├── markdown_gen.py   # 生成 Markdown 格式的资源列表与统计信息
│   │   └── json_exporter.py  # 导出 JSON 格式数据，供其他工具消费
│   ├── validator/            # 链接校验模块
│   │   ├── syntax_check.py   # 检查 URL 是否符合基本格式规范
│   │   └── reachability.py   # 可选网络探测，标记可能失效的链接
│   └── core/                 # 核心数据模型
│       ├── resource.py       # 定义 Resource 类，包含 url、category、batch_id 等字段
│       └── index.py          # 管理资源集合的增删改查与批次迭代
├── tests/                    # 单元测试目录
│   ├── test_parser.py        # 针对解析模块的测试用例
│   └── test_formatter.py     # 针对格式化输出的测试用例
├── docs/                     # 文档目录
│   ├── quickstart.md         # 快速入门指南
│   ├── usage.md              # 详细操作手册
│   ├── development.md        # 开发者指南与 API 说明
```

## 贡献指南

我们欢迎并鼓励社区贡献者参与项目改进。请遵循以下步骤提交变更：

1. 阅读 docs/development.md 了解项目整体架构与核心模块职责，确认您的改动方向与现有设计一致。对于较大规模的重构或新功能提议，建议先提交 issue 进行讨论，避免无效工作。

2. 从 main 分支创建新的功能分支，分支命名遵循 feature/xxx 或 fix/xxx 格式。确保本地开发环境已安装所有开发依赖（包括 pytest、flake8、pre-commit），并执行 pre-commit run --all-files 检查代码风格。

3. 在对应模块中实现您的改动，并为新增或修改的代码编写相应的单元测试。测试用例需覆盖正常路径与边界条件，保证代码覆盖率不低于 80%。

4. 提交 commit 时使用清晰且语义化的消息格式，例如 "fix(parser): 修复空行导致的解析异常" 或 "feat(formatter): 增加表格对齐选项"。每个 commit 应聚焦于单一逻辑变更。

5. 推送分支至远程仓库并创建 Pull Request。PR 描述中需说明改动目的、实现方式以及测试结果。项目维护者将在三个工作日内进行 review，并根据反馈进行后续调整。

## 常见问题

**Q: 项目是否会对收录的 URL 进行内容变更或代理转发？**

A: 不会。本项目仅为链接的元数据管理系统，不对目标资源的内容进行任何形式的抓取、缓存、修改或代理。所有 URL 在输出时保持原始格式，用户访问时直接跳转至原始站点。项目自身不存储任何外部资源副本。

**Q: 如何处理资源列表中已经失效的链接？**

A: 项目提供了可选的 reachability 校验模块，但默认关闭以避免网络请求影响构建速度。维护者可通过配置文件启用该功能，在生成清单时标记超时或返回 4xx/5xx 状态的链接。标记后的链接会在输出中附加提醒注释，但不会自动删除，最终处理由用户决策。

**Q: 能否自定义资源分类方式？**

A: 可以。分类逻辑集中在 src/parser/category_mapper.py 中，用户可根据 URL 中的路径关键词或完整域名修改映射规则。同时也支持在 config.yaml 中定义自定义分类表，无需修改源码即可调整分类结果。
