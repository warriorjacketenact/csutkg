# Diannaodian Episode Link Aggregator

Diannaodian Episode Link Aggregator is a structured, community-maintained index of episode-level resource references sourced from the Diannaodian platform. This project is designed for technical researchers, content curators, and developers who require a stable, queryable, and version-controlled catalog of episode identifiers mapped to underlying materials, discussions, or downloadable assets.

The project does not host the referenced content itself. Instead, it provides a curated manifest of URLs that can be consumed programmatically for batch processing, archival validation, or deep-link integration into downstream systems. The primary goal is to reduce friction when navigating episode-based resources by offering a single, well-documented entry point with dependency isolation and reproducible build steps.

## 功能概览

**Ephemeral Identifier Mapping** – Each episode is assigned a stable, human-readable short code that maps directly to the original Diannaodian URL structure, enabling persistent referencing across documentation and tooling.

**Batch Validation Pipeline** – A built-in verification routine checks the HTTP status of each listed episode URL, flagging redirects, timeouts, and 404 responses for manual review.

**Metadata Extraction Templates** – Predefined scraping schemas allow users to extract title, description, and publication timestamp from each episode page using CSS selectors or XPath rules, with sample implementations in Python and JavaScript.

**Versioned Manifest Export** – The episode list is maintained as a JSON Lines (JSONL) file, with each entry containing the original URL, a local cache key, and a last-verified timestamp. This manifest is updated on every release tag.

**Programmatic Query Interface** – A lightweight CLI tool supports filtering episodes by pattern matching on the URL path segment, returning matching entries in table or CSV format for integration with external analytics tools.

**Dependency-Free Core** – The core manifest parser and validator are written in POSIX-compliant shell script, requiring no external libraries beyond standard utilities like curl, jq, and grep.

**CI-Friendly Health Checks** – GitHub Actions workflows execute daily health probes against all listed URLs, generating a status badge and a detailed report artifact with failure categorization.

**Extensible Plugin System** – Third-party plugins can register custom validators, notification handlers, or export formatters via a simple hook directory, allowing organizations to adapt the aggregator to their internal compliance policies.

## 应用场景

**Content Archive Maintenance** – Digital preservation teams use the aggregator to periodically verify that episode resources remain accessible. When a URL becomes unreachable, the pipeline generates an alert, enabling administrators to update or remove the entry before downstream consumers encounter broken links.

**Automated Documentation Generation** – Technical writers embed the episode manifest into build systems that generate internal wikis or API reference portals. The structured URL list ensures that all external references in documentation are consistently formatted and can be refreshed without manual editing.

**Data Pipeline Bootstrapping** – Data engineering teams load the episode URLs into ETL workflows that fetch and parse the underlying HTML content for natural language processing or trend analysis. The stable identifier scheme allows incremental processing without re-scanning unchanged episodes.

**Compliance Auditing** – Organizations subject to regulatory requirements use the validation pipeline to produce attestation reports that confirm all referenced external resources are operational and point to expected content types. The timestamped manifest provides an immutable audit trail.

**Personal Knowledge Base Integration** – Individual researchers and developers incorporate the episode list into their note-taking systems, using the short codes as cross-reference keys across multiple projects and notebooks, ensuring that external references remain resolvable over time.

## 快速开始

Clone the repository, install the minimal dependencies, and run the initial validation on the episode manifest.

```bash
git clone https://github.com/diannaodian/episode-aggregator.git
cd episode-aggregator
make setup
make validate
```

The `make setup` command creates a local virtual environment for Python-based tools if Python 3.8+ is detected, and installs the `jq` utility if missing. The `make validate` command reads the `manifest.jsonl` file, checks each URL with a 5-second timeout, and writes a validation report to `reports/validation-$(date +%Y%m%d).log`.

To generate a CSV export of all episode identifiers with their validation status, run:

```bash
make export format=csv
```

The output file `exports/episodes.csv` contains columns: `id`, `url`, `status`, `last_checked`, and `content_type`.

## 安装要求

The aggregator is designed to run on Linux, macOS, and Windows Subsystem for Linux (WSL). The following dependencies are required for full functionality.

| 依赖 | 必需 | 说明 |
|------|------|------|
| curl | 是 | Used for HTTP health checks and content retrieval. Version 7.68.0 or higher recommended. |
| jq | 是 | JSON processor for parsing manifest entries and formatting output. Version 1.6 or higher. |
| GNU make | 是 | Build automation and task orchestration. Version 4.2.1 or higher. |
| bash | 是 | Core script interpreter. Version 4.4 or higher. Required for array handling and associative arrays. |
| Python 3.8+ | 否 | Required only for the metadata extraction template and plugin development. Not needed for basic validation. |
| git | 否 | Required only for cloning the repository and contributing changes. Not needed for read-only usage. |
| docker | 否 | Optional for running the validation pipeline in an isolated container environment. |
| grep | 是 | Used for pattern matching and log filtering. Standard POSIX implementation. |
| sed | 是 | Used for manifest transformation and report generation. Standard POSIX implementation. |
| awk | 否 | Used in advanced export formatting. Standard POSIX implementation, optional but recommended. |

## 项目结构

The repository follows a modular layout that separates source code, configuration, data, and output artifacts. Each directory serves a distinct purpose in the validation and export pipeline.

```
episode-aggregator/
├── bin/                              # Executable scripts and entry points
│   ├── validate                      # Main validation script (bash)
│   ├── export                        # Export formatter (bash + jq)
│   └── health-check                  # CI health probe wrapper
├── lib/                              # Shared library functions
│   ├── http.sh                       # curl wrapper with retry and timeout logic
│   ├── manifest.sh                   # JSONL parsing and filtering utilities
│   └── report.sh                     # Report generation and colorized output
├── etc/                              # Configuration files
│   ├── manifest.jsonl                # Primary episode manifest (JSON Lines)
│   ├── ignore-list.txt               # URLs excluded from validation (user-maintained)
│   └── plugins.rc                    # Plugin enablement and ordering rules
├── plugins/                          # Extensible hook directory
│   ├── available/                    # Plugin source files (one per plugin)
│   │   ├── slack-notify.sh
│   │   ├── csv-advanced.awk
│   │   └── html-summary.xslt
│   └── enabled/                      # Symlinks to active plugins
├── docs/                             # Project documentation
│   ├── getting-started.md
│   ├── operations.md
│   ├── development.md
│   └── architecture.md
├── exports/                          # Generated output files
│   ├── episodes.csv                  # CSV export (created by make export)
│   └── episodes.json                 # JSON export (created by make export-json)
├── reports/                          # Validation reports and logs
│   ├── validation-YYYYMMDD.log       # Daily validation log
│   └── failures-YYYYMMDD.json        # Detailed failure records for debugging
├── tests/                            # Unit and integration test suite
│   ├── test-http.sh                  # HTTP function tests
│   ├── test-manifest.sh              # Manifest parsing tests
│   └── fixtures/                     # Test data (mock manifests and responses)
├── Makefile                          # Primary task runner
├── .github/                          # GitHub Actions workflows
│   └── workflows/
│       ├── daily-validation.yml      # Scheduled health check
│       └── release-manifest.yml      # Manifest packaging on tags
├── .gitignore                        # Git ignore patterns
├── LICENSE                           # MIT License text
└── README.md                         # This document
```

## 贡献指南

Contributions to the episode aggregator are welcome. Please follow the process outlined below to ensure consistency and reliability.

**Fork the repository and create a feature branch** – Start by forking the main repository to your GitHub account. Create a new branch with a descriptive name, such as `feat/add-episode-batch` or `fix/validate-timeout`, and make all changes within this branch.

**Update the manifest file** – Add new episode URLs to `etc/manifest.jsonl` using the existing format: one JSON object per line, with fields `id` (derived from the URL path), `url` (the original full URL), and `source` (fixed to "diannaodian"). Run `make validate` locally to confirm that the new entries are reachable and correctly formatted.

**Write or update tests** – For any changes to the core validation logic or export functionality, add corresponding test cases in the `tests/` directory. Use the fixture data under `tests/fixtures/` to mock external HTTP responses and avoid making real network calls during unit testing.

**Run the full test suite** – Execute `make test` from the repository root. This command runs all shell script tests, checks for lint warnings via shellcheck, and validates the manifest schema. Ensure that all tests pass before submitting your changes.

**Submit a pull request** – Push your feature branch to your fork and open a pull request against the `main` branch of the upstream repository. ## 常见问题

**Why are some episode URLs returning 404 or redirecting to a different path?**

The Diannaodian platform may periodically reorganize its content, retire old episodes, or move them to new URLs. The validation pipeline marks these entries as `unreachable` in the report. Users should manually verify the intended resource and update the manifest entry if a new permanent URL is available. The project does not automatically rewrite URLs to avoid unintended redirect chains.

**How often is the manifest updated, and how do I receive notifications about changes?**

The manifest is updated on a best-effort basis with each release tag, typically once per month. The CI pipeline runs daily validation but does not automatically update the manifest with new episodes. Users who wish to stay informed about manifest changes should watch the GitHub repository for releases. For notification of validation failures, enable the Slack plugin by creating a `plugins/enabled/slack-notify.sh` symlink and setting the `SLACK_WEBHOOK` environment variable with a valid incoming webhook URL.

**Can I use this aggregator with offline or air-gapped environments?**

Yes, but additional preparation is required. The validation pipeline relies on external HTTP requests, which are not available in air-gapped networks. In such environments, we recommend running `make export` on a connected machine to generate a static CSV or JSON snapshot of the manifest, then transferring that snapshot to the offline system. The exported files contain all metadata needed for documentation or reference purposes, but live health checks will not function without network access.
