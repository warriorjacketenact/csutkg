# Theta Resource Gateway

Theta Resource Gateway is a curated, machine-readable index of technical reference materials, simulation archives, and structured dataset pointers sourced from distributed theater-level repositories. It is not a search engine, nor a traditional bookmark manager. The project provides a deterministic, version-tracked gateway to a fixed corpus of 250 external resource entries, organized for automated ingestion, batch validation, and compositional analysis.

The primary audience includes research engineers, data pipeline architects, and technical writers who require stable, unmodified external references for documentation generation, regression testing of URL availability, or metadata enrichment pipelines. Every entry is preserved exactly as provided, without normalization, redirection, or semantic rewriting, ensuring byte-level reproducibility across integration environments.

## 功能概览

**Deterministic Resource Listing** – Exposes all 250 source URLs as plain-text entries with no additional HTML wrappers or Markdown link transformations, suitable for grep, awk, and custom parser integration.

**Batch ID Extraction** – Each resource entry includes a discrete theater-level identifier derived from the URL path, enabling secondary indexing and cross-referencing with external logs or audit trails.

**Status Code Monitoring** – The gateway distribution includes a companion shell script that performs HEAD requests against each URL and generates a CSV report of HTTP status codes, response times, and content-length headers.

**Metadata Template Injection** – Provides a YAML frontmatter template for each entry, allowing users to attach custom tags, expiration dates, or internal priority flags without modifying the original URL string.

**Change Detection Placeholder** – The project structure reserves a `deltas/` directory for future implementation of content-hash comparison, alerting maintainers when upstream resources return unexpected payloads.

**Offline Mirror Manifest** – Generates a `wget`-compatible mirror list that preserves the original protocol and hostname, facilitating local caching or air-gapped deployment scenarios.

**CI/CD Ready Outputs** – All listing files are output in both Markdown and newline-delimited JSON (NDJSON) formats, enabling direct consumption by GitHub Actions, GitLab CI, or Jenkins pipelines.

## 应用场景

**Documentation Validation Suites** – Technical writing teams integrate the resource list into their CI pipelines to automatically verify that all external references cited in product manuals remain accessible before each release build. The gateway provides a stable baseline for comparing link rot across versions.

**Data Provenance Tracking** – Researchers working with simulation outputs use the gateway to record the exact source URLs of input parameters, training datasets, or benchmark configurations. The unaltered URL strings serve as cryptographic-like identifiers in experiment logs.

**Proxy Whitelist Generation** – Network administrators deploy the gateway to populate allowlists for outbound traffic from isolated development environments, ensuring that only pre-authorized theater resources are reachable during build processes.

**Content Aggregation Feeds** – Curators of technical newsletters or knowledge bases consume the NDJSON output to populate daily digest templates, automatically pulling titles or descriptions via secondary enrichment services without manual copy-paste.

## 快速开始

Clone the repository, install the minimal runtime dependencies, and generate the master resource index.

```bash
git clone https://github.com/theta-resource-gateway/trg-core.git
cd trg-core
pip install -r requirements.txt
python generate_index.py --input urls/master.txt --output docs/index.md --format markdown
```

For batch validation, run the included status checker:

```bash
bash scripts/check_status.sh docs/index.md > reports/status_$(date +%Y%m%d).csv
```

To produce the NDJSON variant for automated systems:

```bash
python generate_index.py --input urls/master.txt --output data/index.ndjson --format ndjson
```

## 贡献指南

Contributions to Theta Resource Gateway are accepted in the form of corrections to the master URL list, improvements to the generation scripts, and enhancements to the status checking tooling. All contributions must preserve the invariant that the 250 URLs remain unmodified in their exact string form.

1. Fork the repository and create a feature branch from `main`. Use a descriptive branch name such as `fix/url-typo` or `feat/status-retries`.

2. If correcting a URL in `urls/master.txt`, ensure that the replacement string is copied directly from the source material without adding or removing protocol prefixes, subdomains, or trailing slashes. Submit a separate commit for the correction.

3. Run the test suite locally using `pytest tests/` and confirm that all existing tests pass. For new functionality, add corresponding unit tests under the appropriate `tests/` subdirectory.

4. Update the documentation if your change affects user-facing behavior, such as adding a new output format or modifying the status checker's retry logic. Include the change in the relevant `docs/` section.

5. Submit a pull request with a clear title and a detailed description of the change. Reference any related issue numbers. The pull request must include a passing CI build; the maintainers will review within five business days.

## 常见问题

**Why are the URLs not normalized to a consistent format, such as adding https:// or removing www.?**

The project explicitly preserves the original URL strings to maintain byte-level fidelity with the source manifest. Normalization would introduce unintended semantic changes, potentially breaking downstream systems that rely on exact string matching for audit trails or cryptographic verification. Users are expected to handle normalization at the consumption layer if required.

**How do I handle a URL that returns a 404 or timeout during batch validation?**

The status checker reports all non-200 responses in the CSV output with the corresponding status code and error message. Users are encouraged to review these reports regularly. If a URL is confirmed to be permanently moved or removed, contributors may submit a correction pull request to update the entry, provided the new URL is sourced from an official redirect or verified alternative. The project does not automatically remove or rewrite failing URLs.

**Can I add additional URLs beyond the 250 entries in the master list?**

The master list is fixed for batch 671/1241. Additional URLs are not accepted in this batch to preserve the deterministic scope. For future batches, a separate manifest file will be created, and the tooling will support multiple batch manifests. Users who require a larger or custom corpus are encouraged to fork the repository and modify the manifest accordingly.
