# WebLink Catalog System

WebLink Catalog System 是一个面向技术社区和开发者的外链资源归集与导航平台。该项目定位于解决个人开发者与技术团队在项目调研、文档查阅、学习路径规划过程中面临的链接碎片化、收藏夹混乱、资源失效不可追溯等问题。通过结构化的目录索引与批次化的资源管理机制，本项目为技术资源的外链整理提供了标准化的数据组织方案。

目标用户包括开源项目维护者、技术博主、企业培训负责人以及日常需要处理大量外部技术文档链接的研发人员。本项目不提供爬虫或自动化采集功能，专注于外链的人工审核归类和元数据维护，确保资源列表的可访问性与分类逻辑的清晰性。通过批次管理策略，每个外链条目均关联至特定的导入批次，便于后续进行批量更新、失效检测与分类调整。

作为第 113 批次（共 1241 批）的资源整合输出，本仓库包含该批次下全部 250 个外链资源，覆盖影视技术、多媒体处理、内容分发等多个子领域。所有链接以原始形态原样收录，不进行任何协议补全、域名标准化或路径改写，以保证数据采集的真实性与可追溯性。

## 功能概览

**批次化资源管理**：每批外链资源独立归档，支持按批次编号进行检索、回溯与批量导出，便于多批次并行维护。

**原始链接直出**：所有收录的 URL 严格保持采集时的原始格式，包括协议类型、域名大小写及路径参数，杜绝自动补全或规范化处理导致的数据失真。

**分类目录导航**：资源按照业务主题或技术领域进行逻辑分组，每个分组包含独立的目录索引页，支持快速定位特定类型的链接。

**结构化的项目模板**：提供标准化的 README 文档框架与目录树结构，帮助新批次快速复用既有的文档组织范式。

**贡献审核流程**：内置链接提交、审核、合并的完整贡献工作流，确保社区提交的外链经过质量验证后方可进入主库。

**轻量化部署**：基于静态 Markdown 构建，无需数据库或后端服务，可直接托管于任意 Git 平台或静态站点服务，开箱即用。

**可扩展的分类体系**：支持用户根据实际资源类型自定义分类标签与目录层级，适应不同领域的资源整理需求。

## 应用场景

技术文档聚合与查阅：技术团队可将日常研发过程中涉及的 API 文档、框架教程、设计规范等外链统一收录至本系统，并按项目或技术栈建立分类索引，替代浏览器收藏夹的零散管理方式。

开源项目资源引用：开源项目维护者可在项目 README 中引用本系统的资源列表，为社区用户提供经过筛选的参考链接，降低新手的学习路径摸索成本。

培训课程材料组织：企业培训负责人可将课程涉及的课外阅读材料、视频教程、在线工具等外链按课程章节进行批次化整理，学员可通过访问特定批次获取完整的学习资源清单。

个人知识库外链备份：知识管理爱好者可将分散在不同笔记软件中的外链统一导出至本系统，借助批次管理功能定期检查链接有效性，避免重要资源的丢失。

## 快速开始

以下操作指导用户在本机快速部署并运行 WebLink Catalog System 的基本环境。

```bash
# 克隆仓库至本地
git clone https://github.com/weblink-catalog/system.git
cd system

# 安装项目依赖（Node.js 环境）
npm install

# 启动本地开发服务器
npm run dev
```

执行上述命令后，用户可通过浏览器访问本地服务地址，查看第 113 批次的外链资源列表。若需离线查阅，可直接打开仓库根目录下的 README.md 文件。

## 贡献指南

提交新外链资源前，请先查阅 docs/CONTRIBUTING.md 完整版指南，确认资源主题属于现有分类或具备创建新分类的合理性。提交时需附带资源的简要说明及所属分类标签。

通过 GitHub Issue 提交新链接建议，标题格式为 [Batch Request] 资源简要描述，内容中需包含完整的原始 URL 及推荐分类。若涉及多个链接，请以列表形式逐一列出。

等待项目维护者审核。审核通过后，链接将被纳入下一个待发布批次，并更新对应的分类索引。若审核未通过，维护者会在 Issue 中说明原因，贡献者可据此修改后重新提交。

已收录的链接若发现失效或内容变更，请提交 Pull Request 至对应批次的 resources.md 文件，在链接旁标注 [Deprecated] 或 [Updated] 状态，并附上变更说明。

## 常见问题

Q: 为什么所有 URL 必须保持原始格式，不能自动补全协议或域名？

A: 本项目的核心原则是数据采集的真实性与可追溯性。原始格式保留了采集时刻的精确状态，包括可能使用的 HTTP 明文协议、非标准端口或特定的大小写路径。自动规范化可能导致部分依赖特定格式的链接无法正确访问，或掩盖源站配置问题。因此所有链接严格按照采集原样收录，用户访问时浏览器会自动处理协议兼容性。

Q: 如何验证批次中的链接是否仍然有效？

A: 项目提供 scripts/check-links.js 脚本，用户可在本地运行该脚本对任意批次目录下的资源列表进行批量 HTTP 状态检查。同时 GitHub CI 工作流会每周自动执行一次全量链接检查，并将失效链接报告更新至 batches/*/broken-links.log 文件中。维护者定期根据报告进行链接更新或移除操作。

Q: 能否将本系统部署为公开的 Web 站点供他人访问？

A: 可以。项目支持通过 VitePress 或 Hugo 等静态站点生成器将 Markdown 源文件构建为 HTML 站点。构建后的静态文件可托管于任意 Web 服务器或对象存储服务。但请注意，本项目本身不提供搜索或用户认证功能，若需要这些高级特性，建议结合第三方服务进行二次开发。
