# LinkVault Resource Aggregator

LinkVault 是一个面向开发者、技术研究人员与信息分析师的轻量级外链资源汇总与导航工具。它不提供内容存储，而是以可维护、可检索、可版本化的方式对分散于互联网各处的技术文档、社区讨论、代码示例与项目主页进行集中索引与管理。

LinkVault 定位于中大型技术团队内部的知识库辅助组件，或独立开发者个人工作流中的资源备忘系统。其核心价值在于将零散的浏览器书签、临时收藏夹与聊天记录中散落的 URL 转化为可长期维护的结构化数据，并通过元数据标签、状态标记与定期校验机制，降低资源失效与信息碎片化带来的认知负担。

## 功能概览

批量导入与去重：支持从 CSV、JSON 或纯文本列表中批量导入 URL，自动检测并合并重复条目，保留首次导入时间与来源标记。

自动元数据补全：对每个提交的链接执行后端异步任务，抓取页面标题、描述、关键词与主要语言，填充至资源记录的对应字段。

自定义标签与分类体系：允许用户创建无限层级的标签树，每个资源可关联多个标签，支持按标签组合进行快速筛选与聚合。

存活状态定期校验：内置轻量级健康检查器，按可配置周期对已收录链接发送 HEAD 请求，标记异常状态码、超时或证书错误，生成变动报告。

全文检索与高级过滤器：基于倒排索引实现标题与描述的全字段搜索，支持按添加时间、最后校验时间、标签与状态码等多维度组合过滤。

数据导出与分享：支持将筛选结果导出为 Markdown 表格、JSON 或 CSV 格式，便于嵌入技术文档、周报或团队知识库。

RESTful API 接口：所有核心操作均提供 JSON API，便于与其他内部系统（如 CI/CD 流水线、监控看板、自动化脚本）集成。

用户与权限管理：提供基于角色的访问控制，区分管理员、编辑员与只读观察员，适用于多人协作的资源池维护场景。

## 应用场景

技术团队内部知识库的资源层：开发团队可将 LinkVault 作为文档站的前置依赖，集中管理所有引用的外部 API 文档、SDK 主页、社区示例与故障排查参考链接，当外部资源变动时通过校验报告及时通知文档维护者。

个人开发者的技术栈备忘录：独立开发者可使用 LinkVault 记录不同项目中依赖的第三方库主页、配置教程与性能对比文章，通过标签区分前端、后端、数据库、运维等维度，避免重复搜索浪费时间。

技术调研与竞品分析的信息采集：分析师在进行技术选型或竞品功能对比时，可批量收集相关产品官网、技术白皮书、用户论坛与版本发布公告，利用自定义字段记录调研结论与评分，形成结构化评估报告。

开源项目的外部依赖索引：开源项目维护者可将项目文档中引用的所有外部链接统一托管至 LinkVault，并在 CI 流程中增加校验步骤，确保文档中引用的示例代码源、数据下载地址与参考文章长期可用。

## 快速开始

以下指令适用于 Linux / macOS / Windows WSL 环境，假设系统已安装 Git、Node.js 18+ 与 npm 9+。

```bash
# 克隆代码仓库
git clone https://github.com/linkvault/linkvault-core.git
cd linkvault-core

# 安装项目依赖
npm install

# 复制环境变量模板并填充数据库连接字符串
cp .env.example .env

# 运行数据库迁移与种子数据填充
npm run migrate:up
npm run seed:init

# 以开发模式启动服务（默认监听 3000 端口）
npm run dev
```

启动后访问 http://localhost:3000 即可进入 Web 控制台。首次登录使用默认管理员账号 admin@linkvault.local / changeme，并建议立即修改密码。

| Git | 2.30 或更高 | 版本控制，用于拉取更新与贡献代码 |

## 贡献指南

提交问题报告前，请先查阅现有 Issue 列表与故障排查文档，确认当前行为是否符合预期设计。若确定存在缺陷或改进空间，请提供最小可复现步骤、环境信息与相关日志片段。

代码贡献需遵守 TypeScript 严格模式与 ESLint 规则集。所有新增功能应附带对应的单元测试或集成测试，确保核心流程覆盖率不低于 85%。提交 Pull Request 时请填写标准模板，说明变更动机、实现方式与潜在影响。

文档改进同样被视作重要贡献。若发现 API 描述不清晰、示例代码过期或翻译缺失，欢迎直接编辑 docs 目录下的 Markdown 文件并提交 PR。文档变更无需通过全部测试流水线，但需通过拼写检查与链接有效性校验。

本地开发环境可通过 docker-compose up -d 一键启动所有依赖服务，随后运行 npm run dev 启动应用。建议使用 Volta 或 nvm 锁定 Node.js 版本，避免因运行时差异导致的不确定行为。

对于重大功能提案或架构调整，请先通过 Discussion 发起设计讨论，获得核心维护者初步共识后再投入实现，以减少返工成本。

## 常见问题

Q: 批量导入大量 URL 时页面出现超时或 504 错误，应如何解决？

A: 导入操作默认采用同步处理模式，对于超过 200 条记录的批次，建议改用异步导入方式：通过 CLI 脚本 scripts/import-csv.js 执行，或调用 API 的 /api/imports 端点并传递 async=true 参数。异步模式下任务进入队列，可通过 /api/imports/{id}/status 轮询进度，导入完成后系统会向配置的通知渠道发送结果摘要。

Q: 存活状态校验报告显示大量链接为 403 或 429 状态，是否意味着链接已失效？

A: 不一定。许多技术文档站点和代码托管平台对自动化请求有频率限制或反爬策略。LinkVault 的校验器默认使用浏览器 UA 头并跟随重定向，但仍可能被服务端拦截。建议在环境变量中配置 CUSTOM_USER_AGENT 和 REQUEST_TIMEOUT，并启用 REQUEST_PROXY 以更换出口 IP。同时可以在标签体系中为特定域名设置忽略校验标记，或调整校验间隔避免密集请求。

Q: 如何将现有浏览器书签或 Notion 数据库中的链接迁移至 LinkVault？

A: 主流浏览器支持将书签导出为 HTML 格式，Notion 数据库可导出为 CSV。LinkVault 提供 scripts/import-bookmarks.js 和 scripts/import-notion.js 两个转换脚本，分别处理上述格式。转换后生成标准化 JSON 文件，再通过 Web 控制台的导入向导或 API 接口完成入库。迁移过程中所有原始标签、添加时间与备注字段均会被保留，并映射至 LinkVault 的自定义属性体系。
