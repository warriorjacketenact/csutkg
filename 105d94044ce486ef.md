# ResourceBridge

ResourceBridge 是一个面向技术团队与内容研究者的外链资源归集与导航系统。该项目定位于解决分散在各类文档、邮件、聊天记录中的高价值外部链接难以统一管理和检索的问题，通过结构化索引与轻量化元数据标注，帮助用户将零散的书签转化为可维护、可共享、可追溯的知识资产。

本项目适用于需要长期跟踪特定领域信息源的技术负责人、文档工程师、研究型开发者以及开源社区维护者。ResourceBridge 不替代浏览器书签，而是提供一套基于项目根目录的资源描述规范与辅助工具链，使得外链资源能够随代码仓库一同版本化，并支持生成静态导航页面以供团队内部使用。

## 功能概览

**资源条目化录入** 支持以 YAML 或 JSON 格式定义单条资源记录，包含标题、原始 URL、标签、采集时间与简短摘要，便于后续检索与分类。

**标签体系与多级分类** 内置标签聚合引擎，允许用户为每条资源赋予多个标签，并基于标签自动生成分类索引视图，支持无限层级嵌套。

**批量导入与去重检测** 提供从 CSV 文件、浏览器书签导出文件（HTML 格式）批量导入资源的功能，导入过程中自动执行 URL 规范化与重复项检测，避免冗余存储。

**静态站点生成模式** 内置模板渲染器，可将资源数据转换为纯静态 HTML 导航页面，无需数据库或后端服务即可部署至任何 Web 服务器或 GitHub Pages。

**资源状态监控** 周期性检查已收录资源的可访问性，对返回 HTTP 错误状态码或连接超时的链接进行标记，辅助用户清理失效资源。

**全文检索与过滤** 基于标题、标签、摘要字段提供轻量级全文搜索能力，支持按标签、状态（有效/失效）、时间范围进行组合过滤。

**协作编辑与版本控制** 资源定义文件采用纯文本格式，与 Git 等版本控制系统天然集成，支持多人分支协作、变更历史追溯与冲突合并。

## 应用场景

技术文档团队维护外部参考链接库 文档撰写过程中需引用大量第三方技术博客、官方 API 文档、学术论文及开源项目主页。ResourceBridge 可帮助团队将分散的参考链接统一录入项目仓库，并在文档构建流程中自动生成链接附录，确保引用来源清晰可溯。

开源社区收集生态相关资源 开源项目维护者可使用 ResourceBridge 建立社区资源页面，收录插件、工具、示例项目、视频教程等周边生态链接。通过标签分类与静态站点生成，访客可快速定位所需资源，降低社区参与门槛。

个人研究者构建领域知识索引 研究人员在文献调研或技术学习过程中积累大量网页书签。利用 ResourceBridge 的标签体系与全文检索功能，可对海量链接进行主题归类与快速回溯，显著提升信息查阅效率。

企业内部合规性资源归档 企业法务或信息安全团队需定期归档外部监管政策、行业标准及安全公告的发布页面。ResourceBridge 的资源状态监控功能可及时提醒链接变更或失效，辅助完成合规性审计的材料留存要求。

## 快速开始

以下指令适用于 Linux 与 macOS 环境。Windows 用户建议通过 WSL 2 或 Git Bash 执行。

```bash
# 克隆项目仓库
git clone https://github.com/resourcebridge/resourcebridge.git
cd resourcebridge

# 安装依赖（项目使用 Python 3.10+ 与 pipenv 管理依赖）
pip install pipenv
pipenv install --dev

# 运行初始化向导，创建示例资源库并启动本地预览服务
pipenv run resourcebridge init
pipenv run resourcebridge serve --port 8080
```

执行上述命令后，打开浏览器访问 http://localhost:8080 即可查看生成的导航页面。默认示例包含约 30 条预置资源记录，可供快速体验核心功能。

## 安装要求

| 依赖 | 必需 | 说明 |
|------|------|------|
| Python 3.10 或更高版本 | 是 | 核心运行环境，需包含 pip 和 venv 模块 |
| pipenv | 是 | 用于管理项目虚拟环境与依赖包版本锁定 |
| Git 2.25 或更高版本 | 是 | 用于克隆仓库及后续版本控制操作 |

| Docker 20.10+ | 否 | 用于容器化部署方式，生产环境可选 |
| Node.js 16+ | 否 | 仅当启用高级主题定制（需编译前端资源）时必需 |

│   └── commands/                   # 各子命令实现
│       ├── init.py                 # 初始化资源库结构
│       ├── add.py                  # 添加单条资源条目
│       ├── import.py               # 批量导入（CSV / 书签HTML）
│       ├── serve.py                # 启动本地预览服务

│
├── internal/                       # 内部核心逻辑（不对外暴露）
│   ├── resource/                   # 资源数据模型与管理
│   │   ├── model.py                # Resource 数据类定义
│   │   ├── store.py                # 文件存储读写（YAML/JSON）
│   │   └── validator.py            # URL 规范化与去重验证
│   ├── indexer/                    # 标签索引与全文检索引擎
│   │   ├── tag_tree.py             # 标签层级树构建
│   │   ├── inverted_index.py       # 倒排索引实现
│   │   └── search.py               # 查询解析与匹配
│   ├── monitor/                    # 资源状态监控模块
│   │   ├── http_client.py          # 异步 HTTP 请求池
│   │   ├── status_tracker.py       # 状态记录与变更通知
│   │   └── reporter.py             # 生成监控报告
│   └── renderer/                   # 静态站点渲染引擎
│       ├── template.py             # Jinja2 模板加载与缓存
│       ├── page_generator.py       # 页面生成流水线

## 贡献指南

ResourceBridge 欢迎各类贡献，包括但不限于功能建议、缺陷报告、文档改进和代码提交。请遵循以下步骤参与项目开发。

第一步，在 GitHub 上 Fork 本仓库至个人账户，并将 Fork 后的仓库克隆至本地开发环境。建议在新建分支上进行修改，分支命名采用 `feature/简述` 或 `fix/简述` 格式。

第二步，运行 `pipenv install --dev` 安装所有开发依赖，并执行 `pre-commit install` 启用代码风格检查钩子。项目使用 Black 进行 Python 代码格式化，使用 flake8 进行静态检查，所有提交必须通过 CI 流水线校验。

第三步，若新增功能或修复缺陷，请在 `tests/` 目录下补充对应的单元测试或集成测试用例，确保测试覆盖率达到 80% 以上。运行 `pytest` 验证所有测试通过后方可提交。

第四步，编写或更新相关文档。用户可见的功能变更需同步更新 `docs/user-guide/` 中的对应章节，API 变动需更新 `docs/api/` 下的接口说明。文档采用 Markdown 格式撰写，要求语言简洁、结构清晰。

第五步，提交 Pull Request 至主仓库的 `main` 分支。PR 描述中需清晰说明变更目的、实现方式及测试情况，并关联相关 Issue 编号。项目维护者将在 3 个工作日内进行评审，必要时会提出修改意见。

## 常见问题

**问：ResourceBridge 与普通浏览器书签管理器有何本质区别？**

答：浏览器书签管理器通常以扁平列表或简单文件夹形式存储链接，缺乏版本控制、协作编辑、状态监控和静态站点生成等能力。ResourceBridge 将资源定义为代码仓库中的文本文件，所有变更均可通过 Git 追溯，支持多人并行维护，并可通过自动化工具检查链接有效性。此外，生成的静态导航页面便于团队内部共享，无需依赖特定浏览器或账户体系。

**问：导入浏览器书签时是否支持保留原有文件夹结构？**

答：支持。导入模块会解析浏览器导出的 HTML 书签文件中的文件夹层级，并将其映射为 ResourceBridge 的标签树结构。每个书签条目以原始 URL、标题和添加时间作为基础字段，文件夹路径则转换为以 `/` 分隔的标签路径。若导入过程中发现重复 URL，系统会提示用户选择跳过、覆盖或合并标签。

**问：静态站点生成后能否部署到 GitHub Pages 或 Nginx？**

答：可以。`resourcebridge build` 命令会在输出目录（默认为 `_site/`）生成完整的静态文件集合，包含 HTML 页面、CSS 样式表和 JavaScript 交互脚本。用户可将该目录内容直接推送至 GitHub Pages 仓库的 `gh-pages` 分支，或将其复制到任意 Nginx、Apache、S3 静态托管服务中，无需额外配置后端环境。
