# ResourceHub

ResourceHub 是一个面向开发者与技术研究人员的结构化外链资源汇总平台。项目定位于高质量技术资料、工具链文档、开源库与学习路径的索引中枢，通过人工筛选与社区贡献相结合的方式，将散落在互联网各处的优质技术资源按照领域、用途与难度进行系统化整理，帮助用户在海量信息中快速定位所需内容。项目本身不托管资源文件，仅提供外部链接的元数据描述与导航服务，适用于个人知识管理、团队技术选型参考以及开源项目依赖调研等场景。

## 功能概览

**分级资源索引**：按照初级入门、中级进阶与高级专题三个层次对收录的外链进行标签化分类，支持用户根据自身技术栈阶段快速筛选。

**多维度筛选体系**：提供按编程语言、应用框架、基础设施、算法与数据结构、设计模式、运维监控、安全审计等七个维度组合查询的能力。

**外链健康度监测**：定期对收录的 URL 进行可达性检查与响应时间统计，自动标记失效链接与慢速资源，保证索引库的可用性。

**社区投票与评论**：允许注册用户对资源链接进行质量投票与使用体验评论，优质资源自动置顶展示，低质资源经审核后降权或移除。

**批量导入与导出**：支持通过 CSV 与 JSON 格式批量导入外部链接清单，也支持将当前索引库按选定分类导出为 Markdown 表格或结构化数据文件。

**RSS 订阅与更新通知**：为每个分类目录生成独立的 RSS 订阅源，用户可实时跟踪新增资源与已有资源的变更动态。

**API 查询接口**：提供 RESTful API 用于外部程序化查询，支持按关键词、分类标签、域名白名单等条件检索资源条目，便于与其他工具链集成。

**自定义标签系统**：用户可为资源打上私有标签，实现个性化分类管理，私有标签不影响全局索引结构。

## 应用场景

团队技术选型调研时，新项目启动前需要评估可用开源方案与第三方服务。ResourceHub 的框架与库分类下收录了大量官方文档、社区评测与性能对比报告链接，技术负责人可在半小时内完成从需求梳理到候选方案筛选的全流程，显著缩短选型周期。

个人开发者学习新技术栈时，经常面临优质教程分散、官方文档入口难以记忆的问题。通过在 ResourceHub 中按语言或框架标签检索，可获得经过社区验证的学习路径资源清单，包括官方指南、最佳实践文章与典型项目源码仓库，避免在低质量内容上浪费时间。

开源项目维护者需要为项目文档添加依赖说明或相关工具推荐时，可在 ResourceHub 中快速查找同类项目的参考链接与外部引用规范，作为自身文档中"相关资源"章节的素材来源，同时通过 API 接口实现文档站点与资源索引的自动化同步。

技术社区运营方在组织线上分享或黑客松活动时，可将 ResourceHub 作为活动页面的外链素材库，按活动主题筛选对应的技术文档、视频教程与在线工具链接，快速生成活动资源包，降低内容整理的人力成本。

## 快速开始

以下命令演示了从 GitHub 克隆项目、安装依赖并启动本地开发服务器的完整流程。

```bash
git clone https://github.com/resourcehub/resourcehub.git
cd resourcehub
npm install
npm run dev
```

执行上述命令后，开发服务器默认监听 3000 端口。访问 http://localhost:3000 可浏览本地索引库首页，访问 http://localhost:3000/admin 进入后台管理界面，使用默认管理员账户 admin / admin123 进行首次登录，登录后请立即修改密码。生产环境部署请参考 docs/deployment.md 文档。

## 安装要求

| 依赖 | 必需 | 说明 |
|---|---|---|
| Node.js 18.x 或更高版本 | 是 | 项目运行时环境，推荐使用 LTS 版本 |
| PostgreSQL 14.x 或更高版本 | 是 | 主数据库，用于存储资源条目、用户信息与投票数据 |
| Redis 7.x 或更高版本 | 是 | 缓存层与 Session 存储，用于提升 API 响应速度 |
| Nginx 1.20 或更高版本 | 否 | 生产环境反向代理与静态资源服务，开发环境可省略 |
| Elasticsearch 8.x | 否 | 全文搜索引擎，启用后可增强关键词检索精准度 |
| Prometheus 2.x + Grafana 9.x | 否 | 监控与可视化组件，用于系统运行状态观测 |
| Docker 20.10+ 与 Docker Compose 2.x | 否 | 容器化部署方案，便于快速搭建完整开发环境 |
| Git 2.30 或更高版本 | 是 | 版本控制工具，用于克隆仓库与提交贡献 |
| pnpm 7.x 或更高版本 | 是 | 包管理器，推荐使用 pnpm 以节省磁盘空间与加速安装 |

## 项目结构

├── apps/

│   │   ├── src/app/                 # 页面路由与布局组件
│   │   ├── src/components/          # 可复用 UI 组件（shadcn/ui 风格）
│   │   └── src/lib/                 # 服务端工具函数与数据获取层
│   └── api/                         # RESTful API 服务（Fastify 独立进程）
│       ├── routes/                  # 路由定义文件，按资源类型拆分
│       ├── middleware/              # 认证、日志与限流中间件
│       └── services/               # 业务逻辑层，含索引更新与检测服务
├── packages/
│   ├── types/                       # 共享 TypeScript 类型定义与 Zod Schema
│   ├── database/                    # Prisma ORM 模型定义、迁移文件与种子数据
│   ├── crawler/                     # 外链健康度检测爬虫（Playwright 实现）
│   └── config/                      # 环境变量解析与全局配置入口
├── docker/
│   ├── compose/                     # 开发与生产环境 Docker Compose 编排文件
│   ├── postgres/                    # PostgreSQL 初始化脚本与扩展配置
│   └── redis/                       # Redis 持久化与缓存策略配置
├── docs/                            # 完整项目文档（详见文档导航表格）
├── scripts/                         # 工具脚本（数据迁移、种子填充、批量导入）
├── tests/
│   ├── unit/                        # 单元测试（Vitest）
│   ├── integration/                 # 集成测试（测试数据库与 API 交互）
│   └── e2e/                         # 端到端测试（Playwright）
├── .github/

## 常见问题

Q: 项目中的外链资源由谁维护？如何保证链接的有效性？

A: 所有外链资源由项目维护者与社区贡献者共同维护。系统后台每 72 小时自动执行一次全量链接可达性检测，检测结果通过邮件摘要发送给核心维护团队。对于连续三次检测失败的链接，系统会自动将其标记为"疑似失效"并移出主索引，移至待审核队列。社区用户也可以通过页面上的"报告失效"按钮主动提交反馈，运营人员会在 24 小时内人工复核。

Q: 能否将 ResourceHub 部署到内网环境，仅用于公司内部的技术资源管理？

A: 可以。ResourceHub 支持完全离线部署，所有依赖镜像均已上传至阿里云容器镜像仓库，内网环境可通过配置私有镜像代理拉取。部署时需关闭 RSS 订阅与外部 API 回调功能，具体配置方法见 docs/deployment.md 中的"内网部署模式"章节。项目不强制要求外网访问，所有核心功能（资源增删改查、标签管理、投票评论）均可在内网正常运行。

Q: 索引库中的数据如何备份与迁移？

A: 数据备份通过 PostgreSQL 的 pg_dump 工具实现，备份脚本位于 scripts/backup.sh，支持全量与增量备份两种模式。迁移数据时，使用 Prisma 的 db push 与 db seed 命令可快速重建表结构与填充基础数据。完整的数据迁移流程包括导出 JSON 格式的资源条目、用户信息与投票记录三部分，具体命令详见 docs/administration.md 中的"数据迁移"小节。建议每周执行一次全量备份，备份文件保留最近 30 个版本。
