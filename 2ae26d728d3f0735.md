# LinkVault Core

LinkVault Core 是一个面向技术团队与内容运营者的轻量级外链资源聚合与管理平台。该项目定位于解决分散链接难以追踪、缺乏结构化描述、无法快速检索与复用的问题，适用于需要批量维护外部参考资料、文档映射、第三方工具索引或内容协作场景的团队。LinkVault Core 不依赖复杂的前端框架，基于纯静态生成与标记分类机制，使链接集合既可被机器读取，也能以人类可读的方式呈现。

## 功能概览

**批量链接导入与标准化清洗**  
支持从 CSV、Markdown 列表、原始文本块中批量抽取 URL，并自动完成去重、协议归一化（保留原样但标记异常）、路径格式校验。

**多层级标签与分类引擎**  
允许为每条链接赋予多个标签（如 `#documentation`、`#api`、`#blog`、`#tool`），并基于标签组合生成动态分类视图。

**自定义元数据扩展字段**  
每条链接可附加备注、状态（有效/待审/失效）、归属项目、更新日期与责任人，便于团队协作审计。

**只读静态站点生成模式**  
内置模板引擎可将链接数据渲染为纯 HTML 目录页，无需数据库，适合部署在任意 Web 服务器或对象存储上。

**RESTful 查询接口**  
提供只读 JSON API，支持按标签、域名前缀、更新时间范围过滤，便于与其他内部系统集成。

**链接健康状态定期检查**  
集成基于 HEAD 请求的可用性探测，自动标记异常链接，并生成异常报告（仅在本地运行模式中启用）。

**全文检索与模糊匹配**  
基于倒排索引实现标题、备注、标签的快速检索，支持中文分词与拼音首字母匹配。

**导入导出兼容性**  
支持导入 Firefox/Chrome 书签导出文件（HTML 格式），导出为 JSON、YAML 或符合 RFC 的纯文本列表。

## 应用场景

**技术文档中心的外部引用管理**  
技术文档团队可将分散在各篇文档中的外部参考链接统一迁移至 LinkVault Core，每条链接标注适用产品版本与维护人，文档中仅保留短引用码，更新链接时无需逐个修改文档。

**开源项目 README 资源索引**  
开源维护者可使用 LinkVault Core 生成外链资源页面，将社区提交的教程、视频、第三方工具链接集中展示，避免 README 正文过于冗长，同时保留完整的来源追溯。

**数据采集管道中的 URL 暂存池**  
数据工程团队在爬取或采集外部数据前，可将候选 URL 导入 LinkVault Core，附加优先级、频次限制与解析模板，作为采集任务的上游配置输入。

**合规审计外链备案**  
法务或合规部门可使用 LinkVault Core 记录所有对外引用的链接，附加审核状态、授权凭证编号与到期提醒，满足行业监管对引用来源的留痕要求。

## 快速开始

以下命令将在本地启动 LinkVault Core 的开发实例，默认监听 8787 端口。

```bash
git clone https://github.com/linkvault/core.git
cd core
npm install
npm run start
```

首次启动后，访问 http://127.0.0.1:8787 即可进入初始化向导，导入示例链接数据。使用 `npm run build` 可生成静态站点输出至 `dist/` 目录。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---------|---------|------|
| Node.js | 18.x LTS 或更高 | 运行时环境，推荐使用官方预编译二进制 |
| npm | 9.x 或更高 | 包管理器，用于安装依赖与执行脚本 |
| SQLite3 | 系统自带或由 better-sqlite3 绑定 | 内置数据库引擎，无需额外安装，但编译依赖需系统支持 |
| Python | 3.9+（仅构建时可选） | 用于部分高级导入脚本与分词模型编译 |
| Git | 2.30+ | 版本控制工具，用于克隆仓库与拉取更新 |
| 磁盘空间 | 至少 200 MB | 用于存放源代码、依赖包及生成的数据文件 |
| 内存 | 最低 512 MB，推荐 1 GB | 运行时内存占用取决于链接总量，每 10 万条链接约消耗 300 MB |

## 贡献指南

1. 查阅开发者文档（`docs/dev/`）了解项目架构与编码规范，确保本地开发环境满足安装要求，并运行 `npm run test` 验证现有测试全部通过。

2. 在 GitHub Issues 中搜索是否存在相关议题或正在进行的工作，若无则新建议题描述要解决的问题或建议的新功能，等待维护者确认后再开始编码。

3. 派生项目仓库，在派生副本中创建功能分支（命名格式为 `feature/简短描述` 或 `fix/问题编号`），提交代码时遵循 Conventional Commits 规范（`feat:`、`fix:`、`docs:`、`refactor:` 等前缀）。

4. 为新增代码编写单元测试，确保覆盖率不低于当前主干分支水平，并在提交前运行 `npm run lint` 检查代码风格，运行 `npm run build` 确认静态生成无异常。

5. 发起合并请求（Pull Request）至主仓库的 `main` 分支，在描述中关联相关议题编号，维护者将在 5 个工作日内完成评审，必要时会要求补充测试用例或调整接口设计。

## 常见问题

Q: 导入大型书签文件（超过 5000 条）时进程卡顿或内存占用过高如何处理？

A: 建议使用 `--batch` 模式分批导入，每批 1000 条并设置 500 毫秒间隔。也可在配置文件中调整 `importer.chunkSize` 参数。若仍无法解决，请切换至 Node.js 的 `--max-old-space-size=4096` 参数增加内存上限。

Q: 生成的静态站点搜索功能无法匹配中文短语，仅能匹配单个汉字。

A: 默认分词器基于二元组（bigram）索引，对于常见中文词组已覆盖。若需更精确的分词，请在配置中启用 `search.useJieba` 选项（需额外安装 `nodejieba` 依赖），并重新构建索引。注意该选项会增加内存占用与构建时间。

Q: 链接健康检查报告显示大量超时，但浏览器访问正常。

A: 健康检查默认超时时间为 3000 毫秒，且不跟随重定向。建议检查目标服务器是否屏蔽了非浏览器 User-Agent，或网络环境存在防火墙限制。可在配置中调整 `healthCheck.timeout` 与 `healthCheck.followRedirect` 选项，或使用 `--proxy` 参数指定代理。
