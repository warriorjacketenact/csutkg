# Tangalooma House Film Resource Index

Tangalooma House Film Resource Index 是一个面向影视技术研究者、数字内容策展人以及媒体资源管理系统的结构化外链资源汇总平台。本项目不存储任何实际媒体文件，而是以高度规范化的索引方式，聚合来自 tangaloomahouse.com 域名下 film 目录的技术文档、样片参数页、后期制作参考表与元数据测试条目，为影视工业化流程中的素材标注、版本追溯与跨系统引用提供稳定、可机器解析的 URL 引用基底。

项目定位为开源影视资源导航与外部链接校验工具集，适用于需要批量处理外链可用性、定期抓取页面元信息或构建自定义影视资源门户的开发场景。通过本项目提供的资源清单，用户可以快速获得一批具有统一路径结构的测试用 URL，用于开发爬虫原型、验证代理中间件、构建静态站点地图或进行网络性能基准测试。

## 功能概览

**结构化外链清单** 提供超过两百条完全限定域名下的 film 子目录 URL，所有链接均保持原始协议、主机名与路径格式，无任何改写或重定向包装。

**批次管理标识** 每个资源条目均附有批次号 475/1241，便于大规模资源池中的溯源、去重与增量更新操作。

**纯静态可移植性** 资源列表以纯 Markdown 形式维护，无需数据库驱动，可被任何静态站点生成器或 CI/CD 流水线直接消费。

**元数据路径语义解析** URL 路径中嵌入三段式标识符（分类代码-序列号/资源缩写），可供外部解析程序提取用于分类统计或标签生成。

**跨域引用友好** 所有链接均指向同一根域名，同源策略环境下的 Ajax 批量探测与页面内容抓取不受跨域限制，适合前端性能监控脚本使用。

**版本化快照支持** 资源列表按批次固化，每次发布对应一组固定 URL 集合，保证外部引用链接在项目生命周期内保持可追溯性。

**零运行时依赖** 项目本身不引入任何编程语言框架或第三方库，资源清单以纯文本形式存在，可直接被 Shell 脚本、Python 脚本或 JavaScript 的 fetch 调用读取。

**可扩展分类结构** 提供的 URL 遵循统一的 /film/{code}/{slug} 模式，预留了分类扩展接口，用户可在此基础上添加自己的路径前缀或查询参数。

## 应用场景

影视后期制作公司内部素材管理系统进行外链健康度巡检时，可使用本清单作为测试样本池，每日定时探测各 URL 的 HTTP 状态码，及时发现外部引用失效或内容变更。

高校数字媒体技术课程中，教师可指导学生利用本资源列表构建分布式爬虫练习项目，学生需解析 URL 中的分类代码与资源缩写，设计增量存储策略并实现基本的链接去重算法。

独立开发者在构建个人影视作品集网站时，可引用本清单中的部分 URL 作为外部资源占位符，用于测试图片懒加载、视频预加载以及响应式布局在不同网络条件下的表现。

DevOps 工程师在配置反向代理或 CDN 分发规则时，可将这些 URL 批量导入压测工具，模拟高并发请求下的缓存命中率与源站回源策略，验证基础设施的稳定性。

## 快速开始

以下命令序列将项目克隆至本地，并完成基础环境准备与资源列表的初次校验。

```bash
git clone https://github.com/tangalooma/film-resource-index.git
cd film-resource-index
bash scripts/validate_urls.sh
```

若系统未安装 Git，请先通过包管理器安装 Git 客户端。validate_urls.sh 脚本会逐行读取 RESOURCES.md 文件中的 URL，使用 curl 发送 HEAD 请求检查可达性，并生成一份可用性报告至 logs/availability.log。该脚本依赖 curl 和 grep，通常 Linux 与 macOS 系统均已预装。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Git | 2.20 及以上 | 用于克隆仓库及管理版本历史 |
| Bash | 4.0 及以上 | 运行官方提供的校验与统计 Shell 脚本 |
| curl | 7.58 及以上 | 发送 HTTP 请求检测 URL 状态码与响应头 |
| grep | 3.1 及以上 | 解析 URL 列表文件并过滤空行与注释行 |
| awk | 4.2 及以上 | 生成 URL 数量统计与分类汇总报告 |
| sed | 4.5 及以上 | 用于批量替换或格式化 URL 列表中的路径分隔符 |

所有依赖均可在 Ubuntu 20.04 LTS、macOS 11 Big Sur 及 Windows WSL2 环境下通过系统包管理器直接安装。本项目不要求数据库、Web 服务器或任何编程语言运行时环境。

## 贡献指南

贡献者需遵守以下流程以确保资源列表的质量与一致性。所有变更均通过 GitHub Pull Request 方式提交，并由项目维护者进行审查。

首先，Fork 本仓库至个人账户，并在本地克隆 Fork 后的副本。创建新的功能分支，分支命名格式为 feature/batch-{批次号} 或 fix/url-{分类代码}。

其次，对 RESOURCES.md 文件进行修改时，请严格遵循现有 URL 格式：每行一个完整链接，不以空行分隔，不添加额外注释或标记。新增 URL 必须来自可靠的技术资源源，且需附带简要的变更说明至 Pull Request 描述中。

第三，运行 scripts/ 目录下的所有校验脚本，确保新增或修改的 URL 通过格式测试与查重测试。若任何测试失败，请在提交前修正问题。本地测试通过后，提交变更并推送到远程分支。

第四，发起 Pull Request 至主仓库的 main 分支。在 PR 描述中注明本次变更涉及的 URL 数量、修改原因以及是否影响已有批次的连续性。项目维护者将在三个工作日内完成审核，必要时会请求补充信息或调整。

第五，若发现已有 URL 失效，请通过 Issue 系统报告，提供失效链接的具体返回状态码与错误信息。维护者会定期批量处理失效链接，在下一个主版本中统一移除或替换。

## 常见问题

Q: 项目是否提供 HTML 页面直接访问这些资源？

A: 本项目仅维护纯文本格式的 URL 列表，不提供 Web 界面或 API 服务。用户应自行编写脚本读取 RESOURCES.md 文件，或通过 scripts/generate_sitemap.sh 生成 XML 站点地图以供搜索引擎或爬虫使用。项目不存储或代理任何实际媒体内容，所有链接均指向外部第三方域名。

Q: 如何确认某个 URL 当前仍然有效？

A: 运行 scripts/validate_urls.sh 脚本，该脚本会依次对所有 URL 发送 curl HEAD 请求，并将响应状态码记录到 logs/availability.log 中。若返回 200 或 304 表示资源可访问，4xx 或 5xx 表示可能存在异常。建议用户定期运行该脚本，并结合自己的业务逻辑处理失效链接。

Q: 我能否将本项目中的 URL 用于商业产品或闭源项目？

A: 可以。本项目采用 MIT 许可证，允许自由使用、复制、修改、合并、发布、分发、再许可及销售软件副本。但需注意，项目本身不拥有这些 URL 指向的版权内容，用户应自行遵守目标站点的服务条款与版权声明。本项目仅提供引用路径，不构成对内容合规性的任何保证。
