# EMHA Programme Resource Aggregator

EMHA Programme Resource Aggregator 是一个面向技术研究者、内容策展人和数据挖掘工程师的轻量级资源外链汇总工具。项目定位为程序化可读的 URL 资产清单系统，专注于对大规模、多批次、异构来源的 Web 资源进行结构化采集、分类存储与快速检索。

本项目不对资源内容本身进行爬取或二次分发，而是作为一份机器可读且人工可审阅的外部链接索引仓库。核心目标用户包括需要维护批量外链清单的技术文档维护者、进行网络资源图谱分析的研究人员，以及构建自定义导航页面的前端开发者。通过本仓库，用户可以获得一份经过格式统一、批次标注、按原始路径完整保留的 URL 资产文件，并配合辅助脚本进行可用性检查、分类统计和格式校验。

## 功能概览

**批量 URL 清单导入**：支持一次性导入本批次 250 条原始节目资源链接，所有链接均保留原始协议头与域名路径，不做任何自动化改写或跳转压缩。

**按原始路径严格输出**：资源列表章节强制保持用户提供的原始 URL 字符串，禁止补全协议、禁止增减 www、禁止改变大小写、禁止添加尾随斜杠，确保与上游数据源完全一致。

**程序化目录树结构**：项目提供清晰的 ASCII 目录树，展示脚本、配置、测试、文档等子目录的层级关系与职责说明，便于贡献者快速理解仓库布局。

**多维度资源分类展示**：资源列表按功能或主题划分为若干小节，如节目核心链接、辅助信息链接、扩展资源链接等，方便用户按类别浏览。

**Markdown 技术文档骨架**：完整的 README 模板包含功能概览、应用场景、安装要求、文档导航、项目结构、贡献指南和常见问题，满足开源项目文档化标准。

**依赖与运行环境表格化**：通过表格明确列出运行时依赖、必需组件与说明信息，降低部署门槛。

**文档导航分层设计**：按层面（用户、开发者、维护者、管理者）提供不同目录入口，并对应回答不同类型的问题，提升文档的可发现性。

**快速开始三步流程**：提供标准化的克隆、安装、运行命令代码块，用户可直接复制执行。

## 应用场景

技术文档库的外链资产维护：技术写作团队可使用本仓库作为外部参考链接的集中存储库，每次新增或更新资源时仅需修改清单文件，并通过 CI 流程自动校验链接可达性。

网络资源图谱的批量导入：研究人员可将本仓库输出的 URL 清单批量导入到网络爬虫、链接分析工具或知识图谱构建管道中，作为种子链接集合进行深度遍历。

自定义导航页面的数据源：前端开发者可 fork 本仓库，通过脚本解析资源列表并生成静态导航站点，将 250 条链接按分类渲染为卡片或列表布局，便于内部团队使用。

批处理脚本的测试样本：开发链接检查器或 URL 规范化工具时，可将本仓库的 250 条真实 URL 作为测试数据集，验证工具在协议处理、路径解析和重定向跟踪方面的正确性。

## 快速开始

```bash
# 克隆仓库到本地
git clone https://github.com/emha-programme/resource-aggregator.git

# 进入项目目录
cd resource-aggregator

# 安装依赖（基于 Python 3.10+）
pip install -r requirements.txt

# 运行资源清单校验脚本
python scripts/validate_urls.py --input resources/urls_1148.txt

# 生成分类汇总报告
python scripts/generate_report.py --batch 1148 --output reports/
```

## 贡献指南

提交新批次资源清单：在 resources/ 目录下创建新的 urls_{批次号}.txt 文件，每行一条原始 URL，确保不包含空行或额外空格，然后运行 validate_urls.py 进行格式校验。

更新分类映射：如需对新增 URL 进行主题分类，请编辑 resources/categories.json 文件，按照已有键值对格式添加路径前缀与分类名称的映射关系，并确保 JSON 结构合法。

完善文档或修复错误：对于 README、文档或脚本中的拼写错误、链接失效或逻辑缺陷，请先在 issue 中描述问题，获得确认后提交 Pull Request，并确保通过所有测试用例。

运行测试与 lint：提交前在项目根目录执行 pytest tests/ 和 flake8 scripts/，确保所有测试通过且代码风格符合 PEP 8 规范。对于文档修改，需执行 markdown-link-check README.md 确认内部链接有效。

## 常见问题

问：资源列表中的链接为什么不做成超链接格式，而是纯文本 URL？

答：本项目定位为程序化可读的资产清单系统。纯文本格式的 URL 便于脚本直接解析、去重和批量操作，同时避免 Markdown 渲染器对链接进行额外转义或截断。用户可直接复制整段 URL 用于自己的工具链。

问：如何验证某个 URL 是否在当前批次或历史批次中已存在？

答：可以运行 scripts/check_duplicates.py 脚本，它会扫描 resources/ 目录下所有 urls_*.txt 文件并输出跨批次重复项。对于单批次内的重复，validate_urls.py 会在校验阶段直接报错。

问：如果原始资源域名或路径发生变化，项目如何处理？

答：本项目不维护资源内容本身，仅作为外链索引。如果用户发现某个链接永久失效或域名变更，可以提交 issue 说明情况，维护者将在下一个批次更新时剔除或替换对应条目。项目不会自动重写或代理任何 URL。
