# LinkHub 资源聚合引擎

LinkHub 是一个面向开发者与技术研究人员的开源外链资源聚合与导航系统。该项目并非传统意义上的内容管理系统，而是一个高度结构化的技术资源索引中间件，旨在解决个人知识管理过程中优质外链分散、失效及检索效率低下的问题。目标用户包括开源贡献者、技术博主、运维工程师以及任何需要长期跟踪和管理大量技术阅读清单的从业者。

LinkHub 通过轻量级的元数据标注机制，将大量原始 URL 转化为可分类、可追溯、可共享的结构化知识库。项目本身不存储任何侵权内容，仅提供公开信息的整理与导航服务，帮助用户从信息过载中解脱出来，快速定位到真正有价值的技术文档、教程及解决方案。

## 功能概览

- **海量外链归一化收录**：支持批量导入原始 URL，自动去重并生成永久存档标识，确保资源不丢失。
- **智能分类与标签系统**：基于 URL 特征及自定义规则，对链接进行技术栈、难度等级和应用场景的自动打标。
- **全文元数据检索**：不仅检索 URL 本身，更支持对页面标题、摘要及自定义备注的快速模糊查询。
- **状态监控与健康检查**：定期巡检已收录链接的有效性，自动标记死链并生成报告，保证资源库的鲜活度。
- **结构化目录树导出**：支持将任意分类下的链接集合导出为 Markdown 格式的目录树，便于嵌入文档或 Wiki。
- **私有化部署与共享**：提供完整的 Docker 化部署方案，用户可在内网搭建专属团队知识导航站。
- **API 接口支持**：提供 RESTful API 用于第三方工具的集成，实现自动化收录与查询。

## 应用场景

- **技术团队内部知识库构建**：团队 Leader 可将长期积累的技术踩坑记录、官方文档镜像及优秀博客集中收录于 LinkHub，新成员入职时通过访问导航站即可快速了解团队技术栈与常用工具链，显著降低上手成本。
- **个人开发者的阅读清单管理**：日常浏览过程中发现的有价值文章或 GitHub 仓库可一键添加至 LinkHub，系统自动归类至“待读”、“深度阅读”或“参考手册”等自定义分区，避免浏览器书签的杂乱无章。
- **开源项目文档关联**：开源维护者可将项目相关的设计文档、社区讨论帖、性能测试报告等外链统一整理至项目的 README 或 Wiki 中，利用 LinkHub 生成的稳定短链避免原始链接变更导致的文档失效问题。
- **技术沙龙与会议资料归档**：参加完技术大会后，将演讲嘉宾分享的参考链接、PPT 下载地址及视频回放链接批量导入系统，按年份和主题归档，形成个人专属的技术会议回溯数据库。

## 快速开始

以下步骤将指导您在本地环境快速启动 LinkHub 服务。

```bash
# 1. 克隆项目仓库
git clone https://github.com/linkhub-dev/linkhub-core.git
cd linkhub-core

# 2. 安装项目依赖（使用 pnpm 或 npm）
npm install -g pnpm
pnpm install

# 3. 配置环境变量（复制示例配置并修改数据库连接）
cp .env.example .env

# 4. 初始化数据库结构并填充基础分类数据
pnpm run migrate
pnpm run seed

# 5. 启动开发服务器（默认监听 3000 端口）
pnpm run dev
```

访问 `http://localhost:3000` 即可看到导航站首页。生产环境部署请参考下文文档导航中的《生产部署指南》。

## 安装要求

LinkHub 基于 Node.js 生态构建，使用 PostgreSQL 作为主要数据存储。请在安装前确保您的环境满足以下要求：

| 依赖组件 | 必需版本 | 说明 |
| :--- | :--- | :--- |
| Node.js | >= 18.0.0 且 < 21.0.0 | 推荐使用 LTS 版本（Hydrogen 或 Iron） |
| pnpm | >= 8.0.0 | 项目包管理器，禁止使用 npm 或 yarn 进行依赖安装 |
| PostgreSQL | >= 14.0 | 主数据库，需启用全文检索（pg_trgm 扩展） |
| Redis | >= 7.0 | 用于缓存聚合数据及 Session 存储（可选，但强烈推荐） |
| Docker | >= 24.0 | 仅在使用容器化部署方式时需要 |
| Git | >= 2.40 | 用于版本管理与源码克隆 |

## 项目结构

LinkHub 采用模块化分层架构设计，核心目录结构与功能说明如下：

```
linkhub-core/
├── apps/                                   # 应用程序入口层
│   ├── web/                                # 用户端 Web 界面 (Next.js 14 App Router)
│   │   ├── app/                            # 页面路由与布局
│   │   └── components/                     # 可复用 UI 组件库
│   └── api/                                # RESTful API 服务 (Fastify 框架)
│       ├── routes/                         # 路由定义与控制器
│       └── schemas/                        # 请求/响应数据校验
├── packages/                               # 可复用业务逻辑与核心库
│   ├── core/                               # 核心数据模型与 ORM 映射 (Prisma)
│   │   ├── models/                         # 数据库模型定义 (Link, Category, Tag)
│   │   └── migrations/                     # 数据库迁移脚本
│   ├── crawler/                            # 链接状态巡检与元数据抓取模块
│   │   ├── checkers/                       # 各协议健康检查实现
│   │   └── parsers/                        # HTML 标题与描述解析器
│   ├── shared/                             # 公共类型定义与工具函数
│   │   ├── constants/                      # 系统级常量配置
│   │   └── utils/                          # 通用工具 (日期、加密、URL 处理)
│   └── config/                             # 环境变量解析与全局配置中心
├── tests/                                  # 单元测试与集成测试
│   ├── unit/                               # 各模块单测用例
│   └── e2e/                                # 端到端测试脚本 (Playwright)
├── docker/                                 # 容器化部署资源
│   ├── Dockerfile                          # 多阶段构建定义
```

## 贡献指南

LinkHub 遵循开源社区协作规范，欢迎各类形式的贡献。无论是修复文档中的错别字，还是提交新功能的实现，我们都将认真评审并给予反馈。请遵循以下流程：

1.  **查阅议题与项目看板**：访问 GitHub Issues 页面，查找标记为 `help wanted` 或 `good first issue` 的议题。在开始工作前，请在该议题下评论说明您打算认领，以避免多人同时处理同一任务。
2.  **派生仓库并创建功能分支**：将主仓库派生至您的个人账户下，并基于 `main` 分支创建一个新的功能分支。分支命名请遵循 `feat/功能简述` 或 `fix/问题简述` 的格式，例如 `feat/support-json-export`。
3.  **编写代码与测试**：在您的分支上进行开发。所有新增功能必须包含相应的单元测试，并确保现有测试套件能够全部通过。请使用 `pnpm run lint` 和 `pnpm run format` 命令保持代码风格的一致性。
4.  **提交变更并签署 DCO**：提交信息请采用语义化提交规范（如 `feat: 添加批量导入进度条` 或 `fix: 修复分类排序接口异常`）。提交时需签署开发者原创证书，即 `git commit -s -m "your message"`。
5.  **发起拉取请求**：推送分支至您的派生仓库，然后向主仓库的 `main` 分支发起 Pull Request。请在 PR 描述中清晰说明变更的背景、实现思路及测试情况，并关联相关的 Issue 编号。

## 常见问题

**问：LinkHub 与普通浏览器书签管理器有何本质区别？**

答：书签管理器通常以扁平列表或简单文件夹形式存储链接，缺乏对链接有效性的主动监控、全文检索以及结构化分类能力。LinkHub 定位为知识导航中间件，它提供了基于标签的交叉检索、自动化的死链检测、以及可供二次开发的 API 接口。当您收录的链接数量超过数百条时，LinkHub 的检索与分类优势将显著体现。

**问：项目收录的 51hualala.com 链接是否涉及版权或敏感内容？**

答：本项目仅收录和整理公开互联网上可自由访问的 URL 地址，本身不复制、存储或传播任何受版权保护的完整内容。所有链接指向的信息均由第三方网站提供，LinkHub 仅作为导航索引存在。如果权利人认为某链接指向的内容侵犯了其合法权益，请直接联系目标网站的管理员进行处理，或通过项目仓库的 Issue 通道向我们提交删除索引的请求。

**问：如何升级 LinkHub 到最新版本并保证数据不丢失？**

答：我们严格遵循语义化版本规范。在升级前，请务必通过 `pg_dump` 命令备份您的 PostgreSQL 数据库。升级时，拉取最新代码后执行 `pnpm install` 更新依赖，随后运行 `pnpm run migrate` 应用新的数据库迁移文件。启动新版本服务前，请检查 `CHANGELOG.md` 文件中关于破坏性变更的说明。若使用 Docker 部署，只需拉取最新镜像并重新创建容器，挂载的卷（Volume）中的数据将自动保留。
