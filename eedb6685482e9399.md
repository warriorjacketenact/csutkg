# LinkFlow Core

LinkFlow Core 是一个面向技术团队与内容运营者的轻量级外链资源聚合与导航系统。该项目不提供传统的内容管理后台，而是基于静态 Markdown 与结构化数据，将大量分散的技术文档、教程、工具站点、行业报告与社区讨论链接进行统一归集、分类索引与版本追踪。LinkFlow Core 的核心价值在于帮助开发者、技术决策者与研究人员在信息过载的环境中快速定位高质量外部资源，降低重复检索成本，提升技术调研与知识管理的效率。

本项目定位于技术资源的外链汇总层，而非内容生产层。它通过约定优于配置的目录结构与元数据规范，将海量 URL 资源转化为可维护、可审计、可分享的知识索引。LinkFlow Core 适用于个人知识库构建、团队技术雷达维护、开源项目文档的外部参考整理，以及企业技术培训的材料准备等场景。项目本身不依赖数据库，所有资源记录以纯文本形式存储，支持 Git 版本控制，便于多人协作与变更回溯。

## 功能概览

- **批量链接导入与去重**：支持通过脚本批量导入原始 URL 列表，自动识别并移除重复条目，生成标准化的资源记录文件。
- **多维度分类索引**：每个资源可归属至技术领域、内容类型、适用人群、成熟度等级等多个分类维度，支持灵活的组合筛选。
- **元数据增强标注**：为每条外链补充标题、摘要、作者/来源机构、发布日期、内容质量评分与标签系统，提升资源的可发现性。
- **状态生命周期管理**：对资源标记有效、失效、待审、更新中、已归档等状态，支持定期健康检查与失效链接自动报警。
- **结构化文档生成**：根据元数据动态生成面向不同受众的导航文档，包括新手入门路径、高级专题汇总、快速查阅卡片等。
- **变更历史审计**：所有增删改操作均记录在版本控制系统中，支持按时间、操作者、资源 ID 回溯历史变更，满足团队协作审计需求。
- **自定义输出模板**：允许用户根据自身需求定制资源列表的输出格式，例如生成 JSON API 供前端调用、生成 PDF 报告或生成 Slack 通知片段。

## 应用场景

- **技术团队周报与月报素材整理**：技术负责人可每周将团队关注的博客、提案、发布公告链接录入 LinkFlow Core，系统自动生成带摘要的周报素材库，减少重复编写工作。
- **开源项目外部参考索引**：开源项目维护者可在项目仓库的 docs 目录下维护一份 external-references.md，通过 LinkFlow Core 管理所有引用的 RFC、论文、替代实现与社区讨论，确保引用可溯。
- **技术选型调研资料库**：在进行中间件、数据库、前端框架等选型时，调研人员可批量录入官方文档、性能对比文章、生产案例与安全通告，利用分类标签快速对比不同选项的生态成熟度。
- **企业合规与安全知识库**：安全团队可将零散的 CVE 通告、厂商安全公告、合规指南链接集中托管，配合状态管理标记已处理与待处理项，满足审计合规要求。
- **个人技术学习路线图构建**：开发者可将日常阅读的优质教程、视频课程、项目源码与问题排查帖子按学习阶段分类，形成个人专属的学习地图，避免遗忘与重复搜索。

## 快速开始

以下命令将在本地克隆项目、安装依赖并启动开发预览服务，默认监听 8787 端口。

```bash
git clone https://github.com/linkflow-core/linkflow-core.git
cd linkflow-core
npm install
npm run dev
```

启动后，访问 http://localhost:8787 可查看默认的资源索引首页。若需导入自定义链接数据，请将包含 URL 列表的纯文本文件放入 ./data/raw/ 目录，然后执行 `npm run import`，系统将自动解析并生成对应的资源记录文件，存放在 ./data/entries/ 目录下。

| 依赖 | 必需 | 说明 |
|------|------|------|
| Node.js 18.x 或更高版本 | 是 | 项目运行时环境，建议使用 LTS 版本 |
| npm 9.x 或更高版本 | 是 | 用于安装项目依赖与管理脚本 |
| Git 2.30 或更高版本 | 是 | 用于版本控制与协作，若仅预览可忽略 |

| curl 或 wget（健康检查脚本） | 否 | 若使用命令行检测工具，需预装其中之一 |

## 项目结构

```
linkflow-core/
├── bin/                                 # 命令行工具入口脚本
│   ├── linkflow-cli.js                  # 主 CLI 程序，整合 import、check、export 子命令
│   └── health-check.js                  # 独立链接健康检测脚本，可被 cron 调用
├── config/                              # 项目配置目录
│   ├── default.toml                     # 默认配置，包含分类体系、输出格式、检查超时等
│   └── custom.taxonomy.json             # 用户自定义分类标签，可覆盖默认分类
├── data/                                # 所有运行时数据存储
│   ├── entries/                         # 单条资源记录，每个文件对应一个 URL，含元数据
│   ├── indexes/                         # 按分类、标签、状态生成的预计算索引快照
│   ├── raw/                             # 原始导入文件暂存区，用于批量处理
│   └── audit.log                        # 变更操作审计日志，按行追加
├── docs/                                # 项目文档
│   ├── getting-started.md               # 入门指南
```

## 贡献指南

1. 在 GitHub 上 Fork 本仓库，并在本地 clone 已 Fork 的版本。创建新的功能分支，分支命名遵循 `feat/功能简述` 或 `fix/问题简述` 格式，例如 `feat/auto-tagging`。
2. 若新增资源条目，请将原始 URL 放入 `./data/raw/` 目录下的对应分类子目录，并确保文件为纯文本格式，每行一个 URL。运行 `npm run import -- --classify` 触发自动分类与元数据生成。
3. 若修改核心解析逻辑或分类规则，需同步更新 `./test/unit/` 下对应的测试用例，并确保所有测试通过（执行 `npm test`）。提交前请运行 `npm run lint` 进行代码风格检查。
4. 提交 Pull Request 时，请清晰描述变更目的、涉及的功能模块以及是否影响现有数据兼容性。若为新增分类标签，需在 `./docs/taxonomy.md` 中补充说明。
5. 文档类贡献（如修正拼写错误、补充示例、翻译等）可直接在 `./docs/` 目录下修改并提交，无需额外测试，但需确保 Markdown 格式正确。

## 常见问题

**问：导入大量 URL 后，系统如何避免重复条目？**

答：系统在导入阶段会根据 URL 的标准化形式（去除末尾斜杠、统一为小写协议等）计算唯一指纹，并与已有条目的指纹进行比对。若发现重复，系统默认跳过该条目并在日志中记录警告，同时支持 `--overwrite` 参数强制覆盖已有条目的元数据。

**问：链接健康检查是否会影响外部站点正常访问？**

答：健康检查模块默认使用 HEAD 请求方法，仅获取响应头状态码，不会下载页面完整内容。检查间隔默认不低于 24 小时，且支持配置随机延迟与并发限制，避免对目标服务器造成压力。对于部分限制 HEAD 请求的站点，可配置为 GET 请求并限定超时时间。

**问：项目是否支持多用户协同编辑？**

答：由于所有数据以文件形式存储在 Git 仓库中，多用户协同通过 Git 分支与 Pull Request 流程实现。不同贡献者可在各自分支上增删改资源，通过合并请求将变更汇总至主分支。项目不提供内置的 Web 编辑界面，推荐配合 GitHub/GitLab 的 Web IDE 或本地编辑器使用。
