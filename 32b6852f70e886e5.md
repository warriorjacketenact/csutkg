# LinkVault 聚合资源导航系统

LinkVault 是一个面向技术社区与内容创作者的轻量级外链资源聚合与导航平台。该项目定位于帮助开发者、研究员与技术写作人员快速构建结构化的外部资源目录，将分散于各处的优质 URL 进行集中管理、分类展示与版本追踪。LinkVault 不提供爬虫或自动化采集功能，而是通过人工维护的 YAML 数据源驱动静态站点生成，确保每一条外链的准确性与可溯源性。

本项目适用于需要维护大量外部链接的技术文档站点、开源项目 Wiki、个人知识库以及团队内部资源收藏夹。LinkVault 内置了基于路径规则的链接校验中间件，能够自动检测失效链接并生成报告，同时提供灵活的标签过滤与全文检索接口，方便用户在海量外链中快速定位目标资源。

## 功能概览

**层级化目录结构** 支持无限层级的分类嵌套，每个分类可独立配置描述、图标与访问权限，满足从简单收藏到复杂文档体系的不同需求。

**URL 原样输出引擎** 严格保留用户提交的每一个 URL 原始格式，不自动补全协议头、不添加或移除 www 前缀、不改变大小写、不追加尾部斜杠，确保目标地址的精确可达性。

**自动链接健康检查** 每日定时任务对全部外链进行 HEAD 请求探测，标记响应码非 200 的记录，并通过邮件或 Webhook 发送异常汇总通知。

**标签与全文检索** 基于 Bleve 构建的轻量级索引，支持按标签、标题、路径片段进行组合检索，检索响应时间控制在 200 毫秒以内。

**Markdown 驱动的内容管理** 所有分类说明、链接注释与页面元数据均以 Markdown 格式存储，与 Git 工作流无缝集成，支持版本差异对比与协作审阅。

**响应式前端界面** 基于 Bootstrap 5 构建的自适应布局，在桌面端与移动端均提供清晰的目录树导航与面包屑指引，无需额外配置即可适配主流屏幕尺寸。

**导入导出兼容性** 支持从 CSV、JSON 及主流书签 HTML 格式批量导入链接，同时提供 JSON API 供第三方系统消费目录数据。

## 应用场景

技术文档站点的外链附录管理。当开源项目文档中包含大量参考链接、依赖仓库地址或第三方教程时，LinkVault 可以作为独立的导航子系统运行，通过 iframe 或 API 集成的方式嵌入现有文档页面，避免主文档仓库因频繁更新外链而产生大量无意义的提交记录。

个人研究人员的知识收藏夹。研究人员在阅读论文或浏览技术博客时积累的大量 URL 可通过 LinkVault 进行分类标注，配合内置的标签系统实现按主题、按时间、按重要性的多维筛选，替代传统浏览器书签的扁平化管理方式。

团队内部资源共享平台。开发团队、运营团队或教研组可以将常用设计资源、内部工具地址、运维监控面板等链接统一录入 LinkVault，通过简单的权限分级控制不同成员的可视范围，减少重复询问和口头传递链接的成本。

静态网站生成器的数据源组件。使用 Hugo、Jekyll 或 VuePress 构建静态站点的用户可以将 LinkVault 作为独立的数据服务，通过定时构建流程拉取最新的外链数据集，生成符合站点主题的导航页面，实现内容与展示的解耦。

## 快速开始

以下指令适用于 Linux / macOS / Windows WSL 环境，确保系统已安装 Git 与 Go 1.21 以上版本。

```bash
# 克隆项目仓库
git clone https://github.com/linkvault/linkvault.git
cd linkvault

# 安装依赖模块
go mod download
make build-deps

# 以开发模式运行服务
go run cmd/linkvault/main.go --config configs/dev.yaml --port 8080
```

服务启动后访问 http://localhost:8080 即可查看默认的示例目录。首次运行时会自动在 data/ 目录下生成示例分类与链接数据文件。

## 贡献指南

欢迎各类形式的贡献，包括但不限于新增功能、修复缺陷、完善文档与补充示例数据。请遵循以下流程以确保协作顺畅。

首先，在 GitHub 上 fork 本仓库并克隆到本地开发环境，建议使用 feature/xxx 或 fix/xxx 格式的命名分支进行开发。提交代码前请运行 make test 确保所有已有测试用例通过，并为新增功能补充对应的单元测试或集成测试。

其次，对于数据文件（data/ 目录下所有 YAML 内容）的修改，请严格遵循 docs/schema/catalog-schema.md 中定义的字段规范，并在提交信息中注明数据来源或变更理由。新增外部链接时建议附带简要的描述字段，便于其他用户理解该资源的用途。

第三，若涉及前端界面的改动，请确保在 Chrome 与 Firefox 最新稳定版中完成兼容性验证，移动端视口适配同样需要测试通过。修改静态资源后需执行 make build-assets 重新生成压缩文件，并将编译后的产物一并提交。

第四，文档类贡献请将内容放置于 docs/ 下对应的子目录，使用 Markdown 格式编写，遵循中文文案排版指北中的空格与标点规范。提交前运行 make lint-docs 检查文档中的链接有效性。

最后，所有 Pull Request 需要至少一名维护者审阅通过后方可合并。较大的功能改动建议在 issues 中先进行设计讨论，避免大量返工。对于首次贡献者，可在 PR 描述中标注 first-contribution 标签，维护者将提供更细致的指导。

## 常见问题

**Q：启动服务后访问页面，所有链接都显示为裸 URL 且无法点击跳转，这是否正常？**

A：此现象属于预期行为。LinkVault 默认启用「原样输出」模式，所有 URL 在页面渲染时以纯文本形式展示，不自动添加 a 标签或任何可点击包装。这是为了防止自动补全协议头或改动域名大小写导致目标地址不可达。若需启用可点击链接，请将配置文件中 display.clickable_links 设置为 true，但需自行承担因 URL 格式不规范导致跳转失败的风险。

**Q：如何批量导入已有的书签文件？**

A：LinkVault 提供了 import 子命令，支持从 Chrome 书签 HTML 导出文件、标准 CSV 以及 JSON 数组格式进行导入。具体用法为 ./linkvault import --source bookmarks.html --format chrome --target data/links/imported.yaml。导入过程中会自动去重并生成分类映射建议，导入完成后请人工复核数据文件中的分类归属与描述字段是否准确。

**Q：健康检查报告中的链接状态为 403 或 429 时应该如何处理？**

A：403 状态通常表示目标服务器拒绝了 LinkVault 的请求头，可在配置文件中调整 health.user_agent 字段模仿主流浏览器标识。429 状态表示触发频率限制，请增大 health.interval 配置项的值（默认 24 小时）。对于持续异常的链接，建议手动访问确认是否已失效，确认后从数据文件中删除或标记为 archived 状态。
