# ResourceBridge

ResourceBridge 是一个面向技术社区的内容聚合与导航系统，专注于将分散在网络各处的优质技术文档、学习资料和参考资源进行结构化整理与分类检索。项目定位为技术资源的外链元数据库，不对资源内容进行二次托管，而是通过标准化的元数据描述和分类体系，帮助开发者高效定位所需信息。

项目主要服务于技术文档编写者、开源项目维护者、技术社区运营人员以及需要频繁查阅外部参考资料的软件工程师。通过 ResourceBridge，用户可以获得一个可定制、可扩展的资源索引框架，用于构建自己的技术知识导航站。

## 功能概览

**流式资源采集**：支持从指定源地址批量获取资源链接，自动去重并生成标准化的资源条目。

**分类标签系统**：每条资源可关联多个分类标签，支持按技术领域、内容类型、适用层级等维度进行筛选。

**元数据提取**：自动解析资源页面的标题、摘要、关键字等元信息，生成统一的资源描述记录。

**检索与过滤**：提供基于关键词、标签、时间范围的组合检索能力，支持结果排序与分页。

**外部链接校验**：定期对已收录的外链进行可用性检查，标记失效或变更的资源地址。

**数据导出接口**：支持将资源列表导出为 JSON、CSV、Markdown 表格等多种格式，便于集成到其他系统。

**用户订阅机制**：允许用户订阅特定标签或分类的更新，当有新资源入库时通过邮件或 Webhook 通知。

**访问统计面板**：展示资源的点击热度、分类分布、新增趋势等统计数据，辅助内容优化决策。

## 应用场景

技术博客与文档站点的外链管理。技术作者在撰写文章时常常需要引用大量外部资料，ResourceBridge 可以提供统一的外链库，避免重复搜索和链接失效问题。作者只需在系统中检索所需主题，即可获得经过预筛选的参考链接列表。

开源项目 README 与 Wiki 的资源整理。开源项目维护者可以使用 ResourceBridge 构建项目的资源导航页，将相关的教程、API 参考、社区讨论、扩展工具等链接分类陈列，降低新贡献者的学习门槛。

技术社区的内容聚合中心。技术社区或论坛运营方可以部署 ResourceBridge 作为内容聚合后端，将用户推荐的优质资源统一收录并展示在社区的资源频道，提升社区内容资产的利用率。

企业内部技术知识库的补充索引。企业研发团队可以将 ResourceBridge 作为内部知识库的外链补充模块，将与团队技术栈相关的官方文档、最佳实践、问题排查资料等统一管理，方便团队成员快速查阅。

## 快速开始

以下命令演示如何从 GitHub 克隆项目仓库、安装依赖并启动开发服务器。

```bash
git clone https://github.com/resourcebridge/resourcebridge.git
cd resourcebridge
npm install
npm run dev
```

执行上述命令后，开发服务器默认运行在 http://localhost:3000 。访问该地址即可进入 ResourceBridge 的管理界面，开始配置资源源和分类体系。生产环境部署请参考 `docs/deployment.md` 中的详细说明。

## 安装要求

| 依赖 | 必需 | 说明 |
|------|------|------|
| Node.js 18.x 或更高 | 是 | 运行时环境，建议使用 LTS 版本 |
| npm 9.x 或更高 | 是 | 包管理器，用于安装项目依赖 |
| PostgreSQL 14.x 或更高 | 是 | 主数据库，存储资源元数据和用户信息 |
| Redis 7.x 或更高 | 是 | 缓存与任务队列后端，用于提升检索性能和异步任务 |
| Elasticsearch 8.x | 否 | 可选组件，启用后提供全文搜索能力，建议生产环境部署 |
| MinIO / S3 兼容存储 | 否 | 可选组件，用于存储资源快照和导出文件 |

## 项目结构

```
resourcebridge/
├── src/
│   ├── core/                         # 核心模块：资源模型、分类引擎、检索接口
│   ├── collectors/                   # 采集器模块：各类资源源的适配器实现
│   │   ├── http-fetcher.ts           # HTTP 资源抓取与解析
│   │   └── stream-processor.ts       # 流式资源批量处理
│   ├── services/                     # 业务服务层：元数据管理、链接校验、统计聚合
│   ├── api/                          # RESTful API 路由：资源 CRUD、检索、订阅
│   └── utils/                        # 工具函数：日志、配置、缓存、队列
├── tests/
│   ├── unit/                         # 单元测试用例
│   └── integration/                  # 集成测试（数据库、缓存、外部服务）
├── docs/                             # 完整文档：入门、使用、开发、运维
├── scripts/                          # 运维脚本：数据迁移、种子数据、健康检查
├── config/
│   ├── default.yaml                  # 默认配置文件
│   └── production.yaml.example       # 生产环境配置示例
├── docker-compose.yml                # 本地开发环境容器编排（PostgreSQL + Redis）
├── Dockerfile                        # 生产环境容器镜像构建文件
├── package.json                      # 项目依赖与脚本定义
├── tsconfig.json                     # TypeScript 编译配置
└── README.md                         # 项目说明文档（本文件）
```

## 常见问题

问：ResourceBridge 是否存储资源内容的副本？

答：不存储。ResourceBridge 仅保存资源的外链地址和元数据描述（标题、摘要、标签、采集时间等），不缓存或托管任何第三方内容。所有资源访问均直接跳转至原始链接。链接可用性校验仅做状态检测，不保存页面内容。

问：如何添加自定义的资源源？

答：可以在 `src/collectors/` 目录下新建采集器类，继承 `BaseCollector` 并实现 `fetch` 和 `parse` 方法。之后在配置文件中注册该采集器并指定调度周期即可。具体开发指南请参考 `docs/developer-guide.md` 中的采集器扩展章节。

问：生产环境下 Elasticsearch 是否必须启用？

答：非必须。未配置 Elasticsearch 时，系统会自动降级为 PostgreSQL 的全文检索功能，性能在百万级资源以内仍可接受。当资源数量超过百万或需要复杂聚合查询时，建议启用 Elasticsearch 以获得更好的响应速度和查询灵活性。
