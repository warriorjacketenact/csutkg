# ResourceBridge

ResourceBridge 是一个面向技术团队与内容运营人员的结构化外链资源汇总与管理平台。该项目解决的是多源、多批次、大规模外链资源在采集、归档、展示与复用过程中缺乏统一元数据标准与可视化管理工具的问题。ResourceBridge 定位于中大型开源项目文档站、技术知识库维护者以及需要长期维护外部引用资源列表的开发团队，提供从资源录入、分类存储、版本追踪到前端展示的完整工作流。

本项目当前处于第 121 批资源导入阶段，总计包含 250 个经过初筛的外部链接，覆盖影视、技术文档、行业资讯等多个子类。ResourceBridge 本身不生产内容，也不对链接指向的第三方站点做可用性担保，但提供完整的资源清单导出、链接状态检测与分类标记能力，帮助用户高效管理大批量外链资产。

## 功能概览

- **批次化资源导入**：支持按批次编号（如第 121/1241 批）批量录入 URL 列表，自动记录导入时间戳与批次元数据，便于后续按批次回溯与增量更新。

- **多维度分类标签**：每个资源可赋予多个分类标签（如 film、tech-doc、reference），支持基于标签的快速筛选与统计，解决大批量链接无序分散的问题。

- **原样输出与格式校验**：系统严格保留用户输入的原始 URL 格式，不自动补全协议前缀、不修改域名大小写、不添加尾部斜杠，确保链接指向的精确性。

- **链接状态健康检查**：内置异步 HTTP 状态检测任务，可定期检查资源链接的可访问性，标记失效或重定向链接，辅助资源维护者及时清理或更新。

- **结构化文档生成**：基于导入的资源数据，自动生成符合开源项目规范的 Markdown 格式 README 与资源列表章节，减少手动整理文档的人力开销。

- **目录树可视化**：提供项目文件结构的 ASCII 树形展示，附带注释说明，方便贡献者快速理解代码组织方式与各模块职责。

## 应用场景

- **开源项目文档站的外链管理**：技术文档往往需要引用大量外部规范、教程或 API 参考链接。ResourceBridge 帮助文档维护者统一管理这些外链，并在版本迭代时快速确认链接有效性，避免文档中出现死链。

- **技术社区的内容聚合与分享**：社区运营人员可将用户推荐、编辑精选的优质外部资源通过 ResourceBridge 进行批次导入，生成公开的资源汇总页，供社区成员查阅与引用。

- **企业内部知识库的引用源登记**：企业内部的 Confluence 或 Wiki 知识库中常混有大量外部参考链接。使用 ResourceBridge 可建立独立的外部链接登记表，与内部分类体系对接，降低知识库中链接散落、难以追踪的风险。

- **个人技术博主的素材库管理**：技术博主在写作过程中积累的大量参考资料链接，可通过 ResourceBridge 按主题、项目或时间批次进行分类存储，便于写作时快速检索与引用。

## 快速开始

以下操作基于 Linux/macOS 环境，Windows 用户可使用 WSL 或 Git Bash 执行。

```bash
# 克隆项目仓库
git clone https://github.com/resourcebridge/resourcebridge.git
cd resourcebridge

# 安装项目依赖（使用 Python 3.10+ 与 pip）
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 运行本地开发服务器（默认监听 8000 端口）
python manage.py runserver
```

启动后访问 http://127.0.0.1:8000 可进入资源管理面板，首次使用需通过 admin/ 路径创建管理员账户。资源导入功能位于 "Batch Import" 菜单，支持粘贴 URL 列表或上传 CSV 文件。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---------|---------|------|
| Python | 3.10 及以上 | 核心运行环境，低于此版本将无法解析类型注解与新语法特性 |
| Django | 4.2 LTS | Web 框架，用于提供管理界面与 API 接口 |
| PostgreSQL | 14 及以上 | 生产环境推荐数据库，用于存储资源元数据与批次信息 |
| Redis | 6.2 及以上 | 异步任务队列后端，用于链接状态检测等耗时操作 |
| Celery | 5.3 及以上 | 分布式任务调度器，配合 Redis 执行周期性健康检查任务 |
| Git | 2.30 及以上 | 版本控制工具，用于克隆仓库与提交贡献 |
| Node.js | 18 及以上 | 仅用于前端资源构建（非必需，可使用预编译静态文件） |

生产部署时建议额外配置 Nginx 作为反向代理与静态文件服务器，以及 Supervisord 用于管理 Celery Worker 进程。开发环境可使用 SQLite 替代 PostgreSQL，但功能测试建议与生产环境保持一致。

## 贡献指南

欢迎社区开发者参与 ResourceBridge 的改进与维护。请遵循以下步骤提交贡献：

1. 在 GitHub 上 fork 本仓库至个人账户，并 clone 到本地开发环境。建议在 dev 分支上基于最新 main 分支创建特性分支，命名格式为 feature/功能简述 或 fix/问题描述。

2. 本地开发时请确保已安装所有依赖，并运行单元测试确认现有功能未被破坏。新增功能或修复缺陷必须附带对应的测试用例，测试覆盖率不低于原有水平。

3. 提交代码前执行代码风格检查工具（flake8 与 black），确保代码符合 PEP 8 规范。提交信息请采用约定式提交格式，如 feat: 添加批次导入进度条、fix: 修复 URL 校验时尾部斜杠被误删的问题。

4. 向 main 分支发起 Pull Request，并在描述中清晰说明改动目的、涉及模块以及手动测试结果。PR 至少需要一名项目维护者审核通过后方可合并。

5. 如需引入新的第三方依赖，请在 PR 中详细说明必要性、许可证兼容性以及版本选型理由，避免盲目增加依赖体积。

## 常见问题

**问：导入资源时提示 URL 格式校验失败，但我的链接明明是有效的。**

答：ResourceBridge 默认启用了严格的原样输出校验规则。系统会检查 URL 是否包含非法空白字符、是否带有非 ASCII 编码的未转义符号，以及是否以常见协议（http/https）开头。如果你的链接包含特殊中文参数或未编码的空格，请先进行 URL 编码后再提交。同时请注意，系统不会自动补全协议头，裸域名不会被接受，必须显式写入 http:// 或 https://。

**问：链接健康检查报告大量超时，是否意味着这些资源全部失效？**

答：不一定。健康检查采用默认的 5 秒超时阈值与 3 次重试机制，部分海外站点或响应较慢的资源可能被误判为不可达。建议先手动访问确认，若确认为有效链接，可在管理后台将该资源标记为 "忽略检测" 或手动调整检测超时配置。生产环境中可根据网络环境适当调整 tasks.py 中的超时参数。

**问：如何将当前批次的数据迁移到另一台服务器？**

答：ResourceBridge 提供了数据库级别的导入导出能力。使用 Django 的 dumpdata 命令可导出 resources 应用的全部数据为 JSON 格式，再在目标服务器执行 loaddata 命令恢复。注意跨环境迁移时需确保目标服务器的数据库表结构版本一致，建议先运行 migrate 同步模型后再导入数据。
