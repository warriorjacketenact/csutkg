# WebVOD Resource Aggregator

WebVOD Resource Aggregator 是一个面向视频技术研究人员、流媒体开发者和内容运营人员的外链资源汇总工具。该项目通过对特定数据源的结构化整理与索引，提供了一套可快速检索、可版本化追踪的技术资源导航体系。

本项目并非视频托管平台或内容分发网络，而是一个专注于资源定位与分类的元数据聚合层。其核心价值在于将大量分散的、动态变化的视频内容标识符（VID）按照既定的分类规则进行归集，使得开发者能够在不依赖搜索引擎的情况下，通过统一的路径规范快速定位到目标资源。目标用户包括视频云服务商的运维工程师、CDN 节点的质量监控人员、以及需要批量处理视频资源标识符的自动化脚本开发者。

项目采用极简的静态站点生成逻辑，所有资源链接均以结构化文本形式维护，确保在各类网络环境下均可实现低延迟的访问响应。通过持续集成与定期更新机制，本仓库始终保持资源索引的时效性与准确性，为上层应用提供可靠的数据底座。

## 功能概览

**多级分类索引**：资源按照 /vod/{分类码}/{资源ID} 的路径模式进行组织，分类码涵盖 ykv、kos、qlb、zwv 等数十个独立类别，便于按业务模块或内容属性进行精细化筛选与批量导出。

**资源标识符标准化**：每个资源均分配唯一的数字型 VID（Video Identifier），长度为 4 至 8 位不等，支持通过正则表达式或脚本进行快速格式校验与批量重命名操作。

**裸资源直链模式**：所有链接均采用原始协议与域名格式（https://），不附加任何查询参数或跳转追踪代码，确保链接的纯净度与可预测性，便于嵌入各类下游系统。

**批次化版本管理**：本项目作为第 366/1241 批次资源集合，明确定义了当前批次的资源总量（250 条），并通过批次编号实现历史版本的回溯与增量更新的追踪。

**跨平台兼容性**：资源链接不依赖特定操作系统、浏览器或客户端环境，任何支持 HTTPS 协议的标准化网络工具（如 curl、wget、Postman）均可直接访问。

**低冗余数据设计**：所有链接均去除了尾部斜杠、空白字符及非必要元数据，以最小化的数据体积实现最大化的解析效率，适用于带宽受限或存储敏感的场景。

**可扩展的目录结构**：项目支持通过新增分类子目录的方式动态扩展资源范围，无需修改核心索引逻辑，降低维护成本。

**自动化健康检查**：集成轻量级链接可用性探测脚本，可定期检测资源端点响应状态，生成异常报告并发送至维护者指定邮箱。

## 应用场景

**视频内容质量监控**：CDN 运维团队可将本资源列表作为探测样本，通过定时脚本遍历所有链接，收集各节点的响应时间、HTTP 状态码及首包延迟等关键指标，从而评估不同区域节点的服务质量。例如，运维人员可针对 /vod/rae/ 分类下的资源重点监测高码率视频的加载稳定性。

**自动化脚本与工具开发**：开发者可将本仓库作为测试数据集，用于验证视频播放器 SDK 的容错能力、解析器的鲁棒性以及缓存策略的有效性。借助标准化链接格式，脚本无需处理复杂的页面交互，可直接发起 GET 请求并断言预期响应头。

**内容运营数据统计**：运营人员通过分析各分类下的资源数量分布与 VID 规律，可识别出高频更新的内容类型或特定专题的归档情况，为内容采购、推荐算法优化及存储资源规划提供决策依据。例如，/vod/app/ 与 /vod/sea/ 分类下的资源量可作为用户生成内容活跃度的参考指标。

**学术研究与基准测试**：研究人员可利用该资源集合构建视频编解码性能评估基准集，通过对比不同分类下资源的码率分布、分辨率特征及封装格式，分析特定领域的技术演进趋势。本项目提供的数据维度丰富且持续迭代，适合作为长期跟踪的样本来源。

## 快速开始

以下命令演示了从克隆仓库到完成初始数据拉取与本地索引构建的完整流程。请确保在执行前已安装 Git 及 Python 3.8 以上版本。

```bash
# 克隆项目仓库至本地
git clone https://github.com/webvod/aggregator.git
cd aggregator

# 安装依赖项（用于资源解析与健康检查脚本）
pip install -r requirements.txt

# 执行资源索引构建脚本，生成汇总表格与统计报告
python build_index.py --batch 366 --output ./dist/index.html

# 启动本地预览服务器，访问 http://localhost:8080 查看资源导航页面
python -m http.server 8080 --directory ./dist
```

## 贡献指南

我们欢迎并鼓励社区开发者为本项目提交改进建议、新增资源分类或优化脚本逻辑。请遵循以下流程以确保贡献的顺利合并：

1.  **Fork 本仓库并创建功能分支**：在 GitHub 页面点击 Fork 按钮将项目复制至您的个人账户，随后使用 `git checkout -b feature/your-feature-name` 创建新的分支，避免在主分支上直接操作。

2.  **遵循现有代码规范与目录结构**：新增资源链接时请严格按照 `/data/raw/` 下的批次文件格式追加，确保每行仅包含一个完整 URL。脚本修改应保持 PEP 8 编码风格，并补充必要的注释与类型注解。

3.  **编写单元测试或自测报告**：对于涉及解析逻辑或链接校验的修改，请提供对应的测试用例或运行 `check_links.py` 后的输出报告，以证明改动未引入回归问题。

4.  **提交 Pull Request 并关联 Issue**：推送分支至您的 Fork 仓库后，在本项目主仓库发起 Pull Request。请清晰描述改动目的、实现方案及测试结果，并在 PR 描述中关联相关 Issue 编号（如有）。

5.  **参与代码审查与迭代**：维护者将在 3 个工作日内对 PR 进行评审，您可能需要根据反馈进行代码调整。所有对话与变更均保留在 PR 页面，确保过程透明可追溯。

## 常见问题

**问：资源链接返回 404 或超时，应该如何处理？**

答：首先请确认您的网络环境可正常访问 `www.jswater.org` 域名。若确认网络无问题，则该资源可能已被源站迁移或下线。您可以通过 `check_links.py` 脚本批量探测当前所有链接的状态，并将异常结果提交至 Issues 区。项目维护者将每两周同步一次源站数据，并移除或标记失效链接。

**问：如何申请新增资源分类或批量导入自有链接？**

答：您可在 `config.yaml` 文件中添加新的分类码映射规则，并将您的链接列表按照 `/vod/{新分类码}/{VID}` 的格式整理后，置于 `/data/raw/` 下新增的批次文件中。随后运行 `build_index.py` 脚本即可自动识别并合并新数据。若需纳入官方仓库，请参照贡献指南提交 Pull Request。

**问：项目是否提供 API 接口以供外部系统调用？**

答：当前版本暂未提供 RESTful API，但您可直接读取 `/dist/data.json` 文件获取全量结构化数据，该文件包含分类、VID 及完整 URL 的映射关系，适合各类前端或后端应用消费。后续版本将考虑增加 GraphQL 端点以支持更灵活的查询。
