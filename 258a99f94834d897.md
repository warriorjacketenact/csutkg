# ResourceHub

ResourceHub 是一个面向技术研究、学术资料收集与知识管理场景的轻量级外链资源汇总系统。项目定位于为个人研究者、技术团队、内容策展人提供一套结构清晰、可快速部署的 URL 资源索引框架，用于集中管理和分类展示大量分散的外部链接。系统核心功能围绕资源目录树的组织、可视化呈现与快速检索展开，目标用户包括开源社区文档维护者、技术博客作者、在线课程运营方以及内部知识库管理员。项目本身不依赖数据库，采用纯静态文件生成方式，所有资源链接以 Markdown 格式存储，便于版本控制和跨平台迁移。通过本项目，用户可以将数百甚至数千个零散的外部链接转化为具备层级导航和分类标签的可浏览资源站，显著提升信息查找效率与资源复用价值。

## 功能概览

**层级目录导航**：支持无限层级的目录树结构，用户可根据主题、领域或使用场景自由划分资源分类，每个分类下可挂载独立的外链列表。系统自动生成侧边栏索引，方便在多级目录间快速跳转。

**批量链接导入**：提供基于 Markdown 列表的链接导入语法，用户只需按照约定格式编写列表文件，系统即可自动解析并生成对应的资源页面。支持同时导入数百条链接，适用于大批量资源迁移场景。

**资源标签过滤**：每条链接可附带一个或多个标签（如 "video"、"tutorial"、"paper"、"tool"），系统提供按标签组合筛选资源的功能，用户可通过 URL 参数或下拉菜单快速定位特定主题下的所有相关链接。

**外部资源快照**：对于重要的外部链接，系统支持配置快照镜像地址，当原始资源不可访问时，用户可通过快照链接查看存档版本。快照功能通过集成第三方缓存服务实现，降低资源失效风险。

**响应式展示布局**：前端模板针对桌面端和移动端分别优化，资源卡片在宽屏下采用网格布局，窄屏下自动切换为列表视图。每个资源项包含标题、简介、来源域名和添加时间元数据。

**全文检索支持**：集成轻量级客户端搜索引擎，用户可在当前目录范围内对资源标题和简介进行关键词模糊匹配。检索结果实时高亮显示，无需刷新页面即可完成筛选。

**目录导出与备份**：支持将当前资源目录导出为标准 Markdown 或 JSON 格式文件，便于用户进行离线备份、二次加工或迁移至其他静态站点生成器。导出文件保留完整的层级结构和元数据字段。

## 应用场景

**技术团队内部知识库**：研发团队可使用 ResourceHub 整理项目相关的技术文档链接、API 参考手册、开源组件仓库地址以及内部培训视频资源。通过层级分类将前后端、运维、测试等不同领域的外链分开管理，新成员入职时可快速浏览团队积累的外部参考资料体系。

**在线教育课程资料索引**：讲师或课程运营人员可将每节课的拓展阅读材料、视频讲解链接、在线编程练习平台整理成 ResourceHub 资源站。学生通过统一入口即可访问全课程周期的所有外部资源，无需在多个聊天记录或邮件中反复查找。目录结构可按章节、难度或主题灵活调整。

**开源项目文档生态建设**：大型开源项目往往依赖大量外部生态工具和社区资源，ResourceHub 可作为官方文档的补充模块，集中列出插件列表、第三方集成指南、社区博客合集和视频教程索引。项目维护者可通过 pull request 方式协作更新链接清单，确保文档生态始终保持最新状态。

**个人研究文献管理**：研究人员可将阅读过的论文预印本、数据集下载页、实验工具官网以及同行实验室主页按研究方向分门别类存放。配合标签功能，可同时从主题和资源类型两个维度检索已有收藏，有效解决浏览器书签杂乱无章、难以批量操作的问题。

## 快速开始

以下命令演示了从 GitHub 克隆项目、安装依赖并启动本地开发服务器的完整流程。

```bash
git clone https://github.com/your-org/resourcehub.git
cd resourcehub
npm install
npm run dev
```

执行上述命令后，项目将在 http://localhost:3000 启动开发服务器。首次启动时，系统会自动生成示例目录结构和演示数据，用户可在此基础上编辑或替换为自己的资源链接列表。生产环境部署请参考 `docs/deployment.md` 文档使用 `npm run build` 生成静态文件。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | 18.x 或更高 | 项目运行时环境，用于执行构建脚本和开发服务器 |
| npm | 9.x 或更高 | 依赖包管理器，用于安装项目所需第三方库 |
| Git | 2.30 或更高 | 版本控制工具，用于克隆仓库和管理配置变更 |
| Markdown 解析器 | 内置 | 项目内嵌 commonmark 库，无需额外安装，用于解析资源列表文件 |
| 现代浏览器 | Chrome 90+ / Firefox 88+ / Edge 90+ | 前端界面运行环境，支持 ES6 语法和 CSS Grid 布局 |
| 磁盘空间 | 至少 50 MB | 用于存放项目源码、生成的静态资源以及缓存文件 |
| 操作系统 | Linux / macOS / Windows (WSL2 推荐) | 跨平台支持，开发和生产环境均经过上述系统验证 |

## 项目结构

```
resourcehub/
├── src/                                 # 核心源代码目录
│   ├── core/                            # 核心功能模块
│   │   ├── parser.js                    # Markdown 链接列表解析器，将列表转换为内部数据结构
│   │   ├── tree.js                      # 目录树生成器，负责构建层级索引和父子关系映射
│   │   └── validator.js                 # URL 格式校验与去重模块，确保导入链接的合法性
│   ├── server/                          # 开发服务器与构建工具
│   │   ├── dev.js                       # 本地开发服务器，支持热重载和实时预览
│   │   ├── build.js                     # 生产环境静态文件生成器，输出优化后的 HTML/CSS/JS
│   │   └── watcher.js                   # 文件系统监听器，监控资源目录变更并触发增量更新
│   ├── templates/                       # 前端模板引擎
│   │   ├── layout.hbs                   # 主布局模板，包含 HTML 骨架和公共样式引用
│   │   ├── index.hbs                    # 首页模板，展示根目录下的所有一级分类与资源计数
│   │   ├── category.hbs                 # 分类页面模板，渲染特定目录下的子目录和链接卡片
│   │   └── partials/                    # 可复用模板片段，包括导航栏、侧边栏、搜索框等组件
```

## 贡献指南

欢迎社区开发者参与 ResourceHub 项目的改进与扩展。请按照以下步骤提交贡献。

第一步，查阅项目 Issue 列表和 Project Board，了解当前开发计划和待办任务。建议优先选择标记为 "good first issue" 或 "help wanted" 的条目着手，此类任务通常具有明确的范围和较低的入门门槛。在开始实现之前，请在对应 Issue 下回复认领，避免多人同时处理相同任务。

第二步，从主仓库 Fork 项目到个人账户，并将 Fork 后的仓库克隆至本地开发环境。创建新的功能分支，分支命名应遵循 `feature/描述` 或 `fix/描述` 格式，例如 `feature/add-json-export`。确保分支基于最新的 main 分支创建，且提交记录保持整洁。

第三步，完成代码修改后，运行项目内置的测试套件以确保变更未引入回归问题。测试命令为 `npm test`，该命令会执行全部单元测试和 lint 检查。新增功能需附带对应的测试用例，测试覆盖率不应低于现有水平。同时更新或补充文档，确保代码变更与文档描述保持一致。

第四步，提交代码时使用规范的提交信息格式，推荐采用 Conventional Commits 标准，例如 `feat: 添加按标签过滤功能` 或 `fix: 修复嵌套目录下链接显示异常`。提交信息应简明扼要地概括变更内容，便于后续版本发布时生成变更日志。

第五步，向主仓库发起 Pull Request，PR 标题和描述应清晰说明本次变更的目的、实现方式以及测试结果。关联相关的 Issue 编号。项目维护者将在 3 个工作日内进行 Code Review，并根据反馈意见进行迭代修改。合并前需通过所有自动化检查（包括 CI 构建和测试流水线）。

## 常见问题

Q: 项目是否支持动态数据库（如 PostgreSQL 或 MongoDB）作为数据源？

A: 当前版本默认采用纯静态 Markdown 文件存储方式，不依赖外部数据库。这种设计旨在降低部署复杂度并提升页面加载速度，适合资源数量在数千条以内的场景。如果确实需要动态数据源，可通过扩展 `src/core/parser.js` 模块实现自定义数据适配器，项目提供基于文件系统的接口抽象层，可替换为 HTTP API 或数据库查询逻辑。相关扩展方案请参考 `docs/advanced-customization.md`。

Q: 如何将现有浏览器书签（HTML 导出格式）批量迁移至 ResourceHub？

A: 项目自带的 `scripts/import.js` 工具支持从 Netscape Bookmark 格式（HTML）和标准 CSV 文件导入链接数据。具体操作为：将浏览器书签导出为 HTML 文件，然后运行 `node scripts/import.js --from bookmarks.html --target data/links/` 命令。工具会自动解析书签文件夹层级结构，映射为 ResourceHub 的目录分类，并将每个书签条目转换为链接列表中的一项。导入后请检查分类命名和链接标题是否符合预期，如有需要可手动调整。

Q: 生产环境部署后，如何更新资源列表而不重新构建整个站点？

A: ResourceHub 提供两种增量更新方式。第一种是使用 `npm run update` 命令，该命令会扫描 `data/` 目录下的变更，仅重新生成受影响的页面文件，适用于小规模增删改操作。第二种是启用自动同步模式，在启动生产服务器时附带 `--watch` 参数，系统会监听文件变化并在后台自动更新缓存，此模式适合内容频繁变动的场景，但会占用少量额外内存。对于绝大多数静态部署方案（如 Nginx 托管），建议使用第一种方式配合定时脚本完成周期性更新。
