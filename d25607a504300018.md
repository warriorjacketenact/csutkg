# Tangalooma Resource Index

Tangalooma Resource Index 是一个面向技术研究人员的结构化外链资源汇总系统，专用于批量管理和分类呈现来自互联网的深度技术参考资料。本项目不提供具体的内容托管，而是作为一个高可读性的索引门户，将分散于 Tangalooma House 域名下的电影与媒体技术文档链接进行集中整理，便于团队内部或开源社区进行统一查阅与引用。

本项目适用于需要频繁处理大量外部链接的技术文档维护者、数据标注团队以及知识库构建工程师。通过标准化的章节结构，使用者可以快速定位所需资源，并依据项目提供的分类框架进行自定义扩展。项目本身采用纯静态 Markdown 构建，零依赖，低维护成本，适合作为任何技术文档站点的导航层。

## 功能概览

**结构化资源索引** 提供多级分类体系，将所有外链按内容特征划分为电影编号、技术专题等维度，支持快速筛选。

**零运行时依赖** 本项目不引入任何 JavaScript 或后端服务，所有内容以 Markdown 形式呈现，可直接托管于任意静态站点服务。

**批量链接录入** 支持一次性导入大量 URL，并自动保持原始地址格式，避免因协议或域名改写导致的访问失效。

**分类导航表格** 内置文档导航表格，从文档层面、目录层面和问题层面三个角度组织内容，降低新用户的认知负担。

**ASCII 目录树注释** 项目结构采用带注释的 ASCII 树形图展示，开发者可在三秒内理解各个目录的职能边界。

**贡献友好流程** 提供清晰的贡献指南，包含分支管理、提交规范以及链接验证步骤，降低外部贡献者的参与门槛。

**常见问题内置** 预置高频问题与解答，覆盖部署、编码和链接有效性等常见运维场景。

**MIT 许可证** 采用宽松的 MIT 协议，允许任意商业与非商业使用，仅需保留版权声明。

## 应用场景

**技术文档站的导航侧栏替代方案** 当技术文档站点拥有超过一百篇独立文章时，传统侧栏导航往往难以承载。Tangalooma Resource Index 可作为独立的导航页面，以表格和列表形式展示所有外链资源，用户通过浏览器页面查找功能即可快速定位目标链接。

**数据标注团队的参考源管理** 数据标注团队在标注视频或图像数据时，常需要查阅外部编码规范或样例库。本项目可将这些参考链接统一收录，并按照项目批次（如第 471/1241 批）进行分类，确保团队成员始终访问正确的版本。

**开源项目的贡献者入门指引** 许多开源项目在 README 中堆砌大量外部链接，导致可读性下降。使用本项目的结构，可以将所有外链移至独立的“资源列表”章节，主文档仅保留核心介绍和快速开始步骤，提升新贡献者的阅读体验。

**静态网站生成器的内容源** 基于 Hugo 或 Jekyll 构建的技术博客，可以将本项目作为数据文件导入，利用 Markdown 的兼容性直接渲染资源列表，避免手动维护 HTML 代码。

## 快速开始

以下命令可在任何安装了 Git 和标准 Shell 环境的操作系统上执行，用于获取项目副本并启动本地预览。

```bash
# 克隆项目仓库
git clone https://github.com/tangalooma/resource-index.git

# 进入项目目录
cd resource-index

# 安装依赖（本项目仅依赖 markdownlint-cli 用于格式检查，若无需检查可跳过）
npm install -g markdownlint-cli

# 运行本地预览（使用任何静态服务器，例如 Python 内置模块）
python3 -m http.server 8000
```

执行上述命令后，在浏览器中访问 http://localhost:8000 即可查看 README 渲染后的完整页面。若需要修改内容，直接编辑 README.md 文件，保存后刷新浏览器即可生效。

## 贡献指南

1. 首先在 GitHub 上 Fork 本仓库至个人账户，然后克隆到本地开发环境。请确保本地 Git 配置了正确的 user.name 和 user.email，以便提交记录可追溯。

2. 新建一个功能分支，分支名称格式为 `feature/batch-{批次号}` 或 `fix/url-{域名}`。例如，针对第 471 批资源，分支名应为 `feature/batch-471`。所有改动均应在分支中完成，禁止直接向 main 分支提交。

3. 修改资源列表时，请严格保持 URL 原样，不得添加或删除协议头、www 前缀或尾部斜杠。新增链接请按照原列表顺序追加，并在提交信息中注明来源或变更原因。修改后使用 `scripts/check-urls.sh` 验证所有链接是否有效，若返回非零状态码则需修复后再提交。

4. 提交信息应遵循 Conventional Commits 规范，格式为 `<type>(<scope>): <subject>`。其中 type 可选 `docs`、`fix`、`feat`，scope 填写 `readme` 或 `resource`。提交前运行 `markdownlint README.md` 确保无格式错误。

5. 推送分支至个人远程仓库后，通过 GitHub 界面发起 Pull Request 至主仓库的 main 分支。PR 描述中需列出本次改动的资源数量、涉及章节以及测试结果。至少需要一位维护者审核通过后方可合并。

## 常见问题

**问：为什么所有 URL 都必须保持原样输出，不能添加或修改协议前缀？**

答：这是因为目标服务器可能配置了严格的域名绑定或协议重定向规则。例如，某些资源仅允许 HTTP 访问，若改为 HTTPS 会导致证书不匹配或连接超时。此外，部分内网环境或老旧浏览器对协议头敏感。保持原样可最大程度保证链接的可用性与兼容性，避免因格式转换造成访问失败。

**问：如何批量验证资源列表中的 URL 是否仍然有效？**

答：项目提供了 `scripts/check-urls.sh` 脚本，内部使用 curl 发送 HEAD 请求并检查状态码是否为 200。执行命令为 `bash scripts/check-urls.sh`，脚本会遍历资源列表中的所有链接，输出每个 URL 的状态。若某个链接返回 4xx 或 5xx 错误，脚本会在标准错误中显示警告，并最终返回失败计数。建议每次修改后运行该脚本，对于失效链接应尝试更新为最新地址或标注为过期。

**问：项目能否部署到 GitHub Pages 或类似的静态托管服务？**

答：完全可以。由于本项目仅为纯 Markdown 文件，无需构建或编译步骤，直接上传至任何静态托管服务即可。若使用 GitHub Pages，可将仓库命名为 `<username>.github.io`，或在仓库设置中启用 Pages 功能并指向 main 分支。访问页面时，README.md 会被自动渲染为 HTML。若需要自定义主题，可添加 `_config.yml` 文件，但本项目默认不提供，以保持最低复杂度。
