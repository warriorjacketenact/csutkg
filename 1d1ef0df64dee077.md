# VODLink Nexus

VODLink Nexus 是一个面向视频点播（VOD）内容聚合与链接治理的开源基础设施项目。本项目定位于为内容运营团队、影视站点管理员以及个人媒体收藏者提供一套标准化、可扩展的外链资源管理方案，核心目标是将分散、混乱的视频源链接转化为结构化、可追溯、高可用的数据资产。

本项目不提供具体的视频播放或内容存储服务，而是专注于链接层的元数据整理、健康度监测与路由分发。通过对海量 VOD 资源链接进行规范化收录、分类标注与状态跟踪，VODLink Nexus 能够显著提升内容检索效率，降低链接维护成本，并为上层应用（如推荐系统、搜索索引、播放调度）提供坚实的数据底座。项目特别适用于需要管理数千乃至数万条外部视频源链接的中大型内容平台，以及希望建立个人精品影视库的高级技术用户。

## 功能概览

- **链接全生命周期管理**：提供从链接录入、定期校验、状态标记（有效/失效/风险）到自动归档的完整流程，确保资源库的鲜活度。
- **多维度分类与标签体系**：支持按内容类型（电影、剧集、综艺、纪录片）、来源域名、地域编码、清晰度等多个维度对链接进行灵活分类与组合检索。
- **批量导入与导出**：支持 CSV、JSON 及纯文本列表格式的链接批量导入，并可根据筛选条件导出结构化数据集，便于数据迁移或离线分析。
- **健康度主动探测**：内置轻量级 HTTP 探针，可定时检测链接响应码、重定向链、加载耗时等关键指标，自动生成健康度报表并触发异常告警。
- **访问统计与热度分析**：记录每个链接的调用频次、最近访问时间及来源 IP 分布，为内容热度评估与缓存策略优化提供数据支撑。
- **权限与审计日志**：基于角色的访问控制（RBAC）模型，支持多用户协同维护；完整的操作审计日志确保每一次变更均可追溯。
- **开放 API 与 Webhook**：提供 RESTful API 供第三方系统集成，并支持通过 Webhook 与外部监控、通知系统联动，实现自动化运维闭环。

## 应用场景

**场景一：影视站点资源库日常维护**
运营人员每日需处理大量新增影视链接，来源包括合作方推送、爬虫抓取及用户投稿。使用 VODLink Nexus 的统一录入面板，可快速提交链接并自动触发分类与健康检查，告别手动维护 Excel 表格的低效模式。系统会自动标记失效链接并生成周报，帮助团队及时清理死链，提升用户体验。

**场景二：个人媒体服务器内容整理**
自建 Jellyfin 或 Plex 媒体服务器的进阶用户，常需从多个在线源获取影片外挂字幕或备用流地址。通过本项目的标签系统，用户可为不同来源的链接打上“蓝光原盘”“WEB-DL”“内嵌中字”等标签，并通过 API 与自家播放器前端集成，实现智能路由选择——当主源卡顿时自动切换至备用源。

**场景三：内容推荐算法特征工程**
数据科学团队在构建视频推荐模型时，需要特征化每个视频的资源丰富度（如可用源数量、平均响应时间、源分布地域）。VODLink Nexus 导出的结构化链接元数据可直接作为特征输入，辅助模型判断内容的“可获取性”权重，从而优化推荐策略，避免推荐无法正常播放的内容。

**场景四：竞品内容监控与分析**
市场分析人员可利用本项目的链接收录能力，建立对特定竞品站点内容更新频率、资源类型分布及源站健康趋势的长期跟踪看板，为业务决策提供客观数据依据。

## 快速开始

以下步骤帮助您在本地环境快速启动 VODLink Nexus 服务。

```bash
# 1. 克隆项目仓库
git clone https://github.com/your-org/vodlink-nexus.git
cd vodlink-nexus

# 2. 安装项目依赖（使用 pip 和 npm 分别安装后端与前端依赖）
pip install -r requirements.txt
cd frontend && npm install && cd ..

# 3. 初始化数据库并启动开发服务器
python manage.py migrate
python manage.py init_data --sample-links  # 导入示例链接数据
python manage.py runserver --host 0.0.0.0 --port 8080
```

服务启动后，访问 `http://localhost:8080` 即可进入管理控制台。默认管理员账号为 `admin`，密码为 `admin123`，首次登录请务必修改密码。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.9 - 3.11 | 后端核心运行环境，使用 Django 框架 |
| Node.js | 18.x LTS | 前端构建工具及 Vite 开发服务器依赖 |
| PostgreSQL | 14.x 及以上 | 主要数据存储库，用于存放链接元数据、审计日志及用户信息 |
| Redis | 7.x 及以上 | 缓存与消息队列后端，用于提升查询性能及异步任务调度（链接探测） |
| Celery Worker | 5.3.x | 分布式任务队列执行器，独立进程运行，处理定时探测与批量导入任务 |
| Nginx | 1.24 及以上 | 生产环境推荐反向代理服务器，用于静态资源服务与负载均衡 |
| Docker Compose | 2.20 及以上 | 可选依赖，用于一键启动全套服务栈（含数据库及缓存） |

## 项目结构

```
vodlink-nexus/
├── backend/                           # Django 后端核心模块
│   ├── api/                           # RESTful API 视图与路由
│   │   ├── v1/                        # API v1 版本实现
│   │   │   ├── endpoints/             # 各资源端点（links, tags, checks, users）
│   │   │   └── serializers/           # 请求/响应数据序列化器
│   ├── core/                          # 核心业务逻辑层
│   │   ├── link_manager.py            # 链接增删改查及状态流转控制
│   │   ├── probe_engine.py            # 健康探测引擎实现（异步任务调度）
│   │   └── tag_processor.py           # 标签解析与自动分类逻辑
```

## 常见问题

**问：健康探测功能是否会对我收录的第三方站点造成过大请求压力？**

答：探测模块默认采用指数退避策略，并对同一目标域名的并发请求数进行了严格限制（默认单域名最大并发 3 个，探测间隔不低于 5 分钟）。同时，系统支持用户自定义 `User-Agent` 及 `robots.txt` 遵循模式。在首次启动大规模探测前，建议您先在测试环境观察探测流量，并根据实际网络状况调整 `settings.py` 中的 `PROBE_CONCURRENCY` 和 `PROBE_TIMEOUT` 参数。生产环境下，探测任务建议安排在业务低峰期执行。

**问：项目能否直接用于生产环境？数据规模上限是多少？**

答：本项目经过内部两个中型影视站点（日均链接调用量约 15 万次，管理链接总数约 4.2 万条）的实践验证，其核心功能稳定可用。在推荐的硬件配置（4 核 CPU / 16GB 内存 / SSD 存储）下，PostgreSQL 数据库可支撑约 50 万条链接记录及对应的探测历史（约 2 亿行）而不出现显著性能衰退。当链接规模超过 10 万条时，建议启用 Redis 缓存并配置分区表策略（详见运维文档）。项目当前版本为 v1.2.0，已具备生产级基础，但建议在部署前根据自身业务体量进行充分压力测试。

**问：如何迁移现有 Excel 或 CSV 格式的链接数据到本系统？**

答：项目提供了通用的 CSV 导入接口。您需要确保原数据至少包含 `url` 列，并可选择性包含 `title`（标题）、`category`（分类）、`tags`（标签，以 `|` 分隔）等列。导入时，可通过 `manage.py import_links --file your_data.csv --mapping config.yaml` 命令指定字段映射配置。若您的数据格式较为特殊，建议参考 `scripts/seed_sample_links.py` 脚本编写自定义转换逻辑。导入过程中，系统会自动进行 URL 格式校验并去重，导入失败的记录会输出至同目录下的 `import_errors.log` 文件中供排查。
