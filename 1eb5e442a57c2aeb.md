# Tokyo Mirai Resource Index

Tokyo Mirai Resource Index 是一个面向技术研究人员、数据科学家和开源开发者的高性能外链资源聚合与导航系统。该项目并非简单的书签集合，而是一个具备元数据提取、健康度检测、分类标注和全文检索能力的资源索引引擎。它主要解决个人开发者与小型团队在技术研究过程中面临的信息碎片化、外链失效、重复检索效率低下等问题，通过统一的入口和标准化的数据流水线，将分散于网络各处的优质视频教程、技术文档、工具站点和学术资源进行结构化整理与定期更新。

项目定位于中轻量级部署场景，既可以作为个人知识库的外链后台，也可以作为团队内部的技术资源门户。系统核心设计理念为"读多写少、索引优先、外链即数据"，不存储实际的大文件内容，仅保存资源的元信息、快照摘要和状态标记，因此对存储和带宽的需求极低，适合运行在低配置服务器或容器环境中。

## 功能概览

**资源快照采集** 对每一条收录的 URL 执行定时的 HTTP 头部探测和元数据抓取，自动记录资源标题、内容类型、响应状态码和最后修改时间，及时发现失效链接或内容变更。

**多维度分类标签** 基于 URL 路径特征、来源域名和页面标题关键词，为每条资源自动生成技术领域、内容形式、适用水平等标签，支持用户自定义标签体系。

**全文检索引擎** 集成轻量级倒排索引模块，支持对资源标题、描述、标签和页面摘要进行布尔查询和短语匹配，检索结果按相关度与更新时间排序。

**健康度监控面板** 提供资源可用性仪表板，以图表形式展示整体资源存活率、响应时间分布和近七日变化趋势，帮助管理员快速定位需要更新的条目。

**批量导入导出** 支持通过 CSV 和 JSON 格式批量导入外部资源列表，也支持将当前索引完整导出为标准化格式，方便迁移或备份。

**定时自动更新** 内置基于 Cron 表达式的调度器，可配置每日或每周自动执行全量或增量资源探测任务，无需人工干预。

**用户权限分级** 提供管理员、编辑者和访客三级权限控制，访客仅可查看和检索，编辑者可增删改资源条目，管理员可管理系统配置和用户账户。

**响应式 API 接口** 所有核心功能均通过 RESTful API 对外暴露，返回 JSON 格式数据，便于与其他系统集成或构建自定义前端界面。

## 应用场景

技术团队内部知识库建设。开发团队在项目推进过程中会积累大量外部参考链接，包括官方文档、技术博客、视频教程和开源仓库。使用 Tokyo Mirai Resource Index 可以统一管理这些分散的链接，并通过标签和检索功能快速复用已有知识，减少重复查找时间，同时健康度监控能够在链接失效前发出预警，避免团队成员访问无效资源。

开源项目文档的外部引用管理。开源项目维护者通常需要在 README 或官方文档中引用大量外部资源作为参考依据或进一步阅读材料。本项目可以帮助维护者系统性地整理这些引用链接，并持续跟踪其可用性，确保文档中引用的每个外部资源始终有效，提升文档质量和用户体验。

学术研究与文献追踪。研究人员在进行文献综述或技术调研时，需要查阅大量在线论文、数据集页面和工具官网。本项目的快照采集功能能够定期记录资源页面的变更情况，帮助研究人员追踪版本演进和内容更新，同时标签分类可以按研究方向或方法类型快速筛选相关资源。

个人技术博客的外链增强。技术博主可以在文章中使用本项目作为外链后台，将文章中的引用链接统一托管至索引系统中，并通过 API 实时获取链接的最新状态，避免因单个外部站点变更而导致整篇文章引用失效。

## 快速开始

以下步骤适用于 Linux 和 macOS 系统，Windows 用户建议通过 WSL 或 Docker 环境运行。

```bash
# 克隆项目仓库
git clone https://github.com/tokyo-mirai/resource-index.git
cd resource-index

# 安装 Python 依赖（推荐使用虚拟环境）
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 初始化配置文件与本地数据库
cp config.example.yaml config.yaml
python scripts/init_db.py

# 运行索引服务（开发模式）
python app.py --host 127.0.0.1 --port 8080
```

服务启动后，访问 http://127.0.0.1:8080 即可进入管理界面。首次启动会自动创建管理员账户，初始密码将在终端输出中显示，请及时修改。

生产环境部署建议使用 Gunicorn 或 uWSGI 作为 WSGI 服务器，并配合 Nginx 进行反向代理。项目根目录下的 deploy 文件夹中提供了参考的 systemd 服务文件和 nginx 配置模板。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.10 及以上 | 核心运行环境，低于 3.10 版本将无法兼容异步语法和类型注解 |
| SQLite | 3.35 及以上 | 内置轻量级关系型数据库，用于存储资源元数据和用户信息，无需额外安装 |
| Redis | 6.0 及以上 | 可选组件，用于缓存高频查询结果和分布式锁，不安装则降级为本地内存缓存 |
| nodejs | 18.0 及以上 | 仅在前端构建时需要，用于编译静态资源，运行阶段不需要 |
| curl | 7.68 及以上 | 用于资源探测过程中的 HTTP 请求，系统会优先使用系统自带的 curl |
| git | 2.25 及以上 | 版本控制工具，用于克隆仓库和后续更新拉取 |
| make | 3.81 及以上 | 构建自动化工具，用于执行 Makefile 中定义的环境初始化命令 |
| openssl | 1.1.1 及以上 | 用于生成安全密钥和 HTTPS 本地测试证书 |

## 常见问题

**Q: 资源探测功能是否会对目标站点造成压力？**

系统默认使用保守的探测策略，每个资源在单次探测周期内仅发送一个 HEAD 请求和一个 GET 请求（仅获取前 64KB 用于摘要分析）。探测间隔由调度器控制，默认每个资源每 24 小时探测一次，且所有请求均携带标准的 User-Agent 和 Accept 头，模拟普通浏览器行为。对于大量资源（超过 1000 条），系统会自动启用随机延迟和分布式限速机制，确保每秒请求数不超过配置阈值，不会对目标服务器造成负载压力。

**Q: 数据存储是否支持迁移到 PostgreSQL 或 MySQL？**

当前版本默认使用 SQLite 作为数据库后端，主要考虑到轻量部署和零配置的优点。但项目的数据访问层已使用 SQLAlchemy ORM 进行抽象，理论上支持切换至 PostgreSQL、MySQL 或其他关系型数据库。如需迁移，请修改 config.yaml 中的数据库连接字符串，并运行 scripts/migrate_db.py 脚本进行数据迁移。需要注意的是，生产环境若使用 PostgreSQL，建议启用 pg_trgm 扩展以提升全文检索性能。

**Q: 如何确保私有部署中的资源链接不被外部访问？**

系统默认仅允许本地回环地址和配置文件中指定的可信 IP 范围访问管理界面。API 接口除检索端点外均需要 Bearer Token 鉴权，Token 的有效期和生成密钥可在配置文件中自定义。此外，管理员可以启用 IP 白名单模式，仅允许特定的内部 IP 子网访问整个服务。对于更严格的访问控制需求，建议将服务部署在内网环境，并通过 VPN 或 SSH 隧道进行访问。
