# LinkTower Resource Aggregator

LinkTower 是一个面向开发者、技术研究人员与内容策展人的轻量化外链资源归集与导航系统。该项目定位于将大量分散的、具有潜在技术参考价值的 URL 进行结构化存储、分类展示与快速检索，解决个人或团队在浏览过程中产生的链接散落、遗忘、难以复用等问题。LinkTower 不提供爬虫或自动化采集功能，而是作为一个人工精选与机器辅助索引相结合的中间层，帮助用户构建属于自己的高质量外链仓库。

项目目标用户包括：需要维护技术周报或月刊的编辑者、进行竞品调研或市场分析的产品经理、跟踪前沿论文与工程实践的研究人员，以及希望系统化管理个人知识库的开发者。LinkTower 通过统一的条目格式、标签化分类和基础检索能力，将原本杂乱的 URL 列表转化为可探索、可维护的知识资产。

## 功能概览

- **原始链接归集入库**：支持通过命令行或 Web 表单批量导入原始 URL，自动提取域名、路径参数与扩展名信息，生成标准化条目。
- **自动分类与标签建议**：基于 URL 路径特征与域名主体，对链接进行初步的主题聚类，并提供人工复核与标签编辑界面。
- **多维度检索过滤**：支持按域名、路径关键词、导入批次、自定义标签和日期范围进行组合筛选，快速定位目标资源。
- **批次管理视图**：以项目批次（如第 95/1241 批）为单位展示链接集合，便于按周期进行内容审核与导出。
- **结构化导出能力**：支持将选定链接导出为 Markdown 列表、JSON 元数据或 CSV 表格，用于嵌入文档、生成报告或迁移至其他系统。
- **链接可用性检查**：内置轻量级 HTTP 状态检测模块，可对已入库链接进行定期可达性验证，标记失效或重定向条目。
- **元数据扩展字段**：每条记录支持备注、重要等级（1-5 星）和阅读状态（未读/已读/已归档），适配个人工作流。

## 应用场景

- **技术周报素材整理**：内容编辑在浏览 Hacker News、GitHub Trending 或技术博客时，将值得引用的文章地址即时录入 LinkTower。每周固定时间通过批次视图筛选当周新增链接，一键生成周报的参考文献列表，大幅减少手动整理成本。
- **竞品动态监控**：产品团队将竞争对手的官方公告、版本发布页、招聘信息栏等 URL 统一归入 LinkTower，并标注“竞品”标签。每周执行一次链接可用性检查，同时人工浏览更新内容，形成竞品动态简报，为产品决策提供信息输入。
- **论文与预印本跟踪**：研究人员将 arXiv、OpenReview 等平台上与自身方向相关的论文链接存入系统，利用备注字段记录阅读笔记或核心结论。通过批次管理回顾某一时间段的研究热点，辅助文献综述写作。
- **开源项目生态导航**：社区维护者将围绕某一开源项目的教程、插件列表、迁移案例、第三方工具等外部资源整理至 LinkTower，生成生态资源页面的初始素材，降低新进贡献者的信息获取门槛。

## 快速开始

以下操作基于 Linux / macOS 环境，Windows 用户可使用 WSL2 或 Git Bash 执行。

```bash
# 克隆仓库
git clone https://github.com/linktower/linktower-core.git
cd linktower-core

# 安装依赖（使用 Python 3.10+ 和 pip）
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 初始化本地数据库并导入示例链接
python manage.py initdb
python manage.py import --batch 95 --file samples/urls_95.txt
python manage.py serve --host 127.0.0.1 --port 8080
```

执行上述命令后，LinkTower Web 界面将在本地 8080 端口启动。浏览器访问 `http://127.0.0.1:8080` 即可查看第 95 批次的链接列表，并开始进行标签编辑与导出操作。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.10 或更高 | 核心运行环境，推荐使用 3.11 以获取性能优化 |
| SQLite | 3.35 或更高 | 嵌入式数据库，用于存储链接元数据与标签，无需额外安装 |
| pip | 21.0 或更高 | Python 包管理器，用于安装 requirements 中的依赖 |
| Git | 2.30 或更高 | 用于克隆仓库和后续版本更新 |
| 操作系统 | Linux / macOS / Windows (WSL2) | 生产环境推荐 Debian 12 或 Ubuntu 22.04 LTS |
| 浏览器 | 现代浏览器（Chrome 110+ / Firefox 120+） | 仅用于 Web 管理界面，API 模式无需浏览器 |
| 网络连接 | 出站 HTTPS 可达 | 用于链接可用性检测功能，内网环境可关闭该功能 |
| 磁盘空间 | 至少 200 MB | 用于存放数据库文件和日志，链接数超过 10 万条时建议 1 GB 以上 |

## 项目结构

```
linktower-core/
├── cmd/                            # 命令行入口包
│   ├── server/                     # Web 服务启动命令
│   │   └── main.go                 # 入口函数，解析 flags 并初始化路由
│   └── cli/                        # 命令行工具集合
│       ├── import.go               # 导入链接子命令，支持 txt/csv 格式
│       ├── export.go               # 导出链接子命令，支持 json/md/csv 格式
│       └── check.go                # 链接可用性检测子命令
├── internal/                       # 内部私有包，不对外暴露
│   ├── storage/                    # 数据库操作层
│   │   ├── sqlite.go               # SQLite 连接池与基础 CRUD 实现
│   │   └── migration.go            # 数据库迁移脚本（版本 1-8）
│   ├── model/                      # 数据模型定义
│   │   ├── link.go                 # Link 结构体，包含 URL、批次、标签等字段
```

## 贡献指南

LinkTower 遵循开源社区协作规范，欢迎各类形式的贡献，包括但不限于功能提案、代码提交、文档改进和问题反馈。

1. 查阅现有 Issue 与 Project Board。在提交新功能或修复之前，请先访问 GitHub Issues 页面，确认无人正在处理同类问题。如无相关记录，请创建一个新的 Issue 并选择对应的模板（Bug 报告 / 功能请求 / 文档改进）。

2. 派生仓库并创建功能分支。从主仓库派生至个人账户后，基于 main 分支新建一个描述性的分支名称，例如 `feature/batch-tag-editor` 或 `fix/import-timeout`。请确保分支名称与 Issue 编号关联。

3. 编写代码并执行测试套件。所有 Go 源码需通过 `gofmt` 格式化，并补充对应的单元测试文件（`*_test.go`）。提交前运行 `make test` 确保全部测试通过，同时使用 `make lint` 进行静态检查。

4. 更新相关文档。若变更涉及用户可见行为（如新增配置项、修改 API 响应格式），请同步更新 `docs/` 目录下对应的手册文件，并在 README 的「文档导航」表格中注明关联章节。

5. 提交 Pull Request。推送分支后，在主仓库发起 PR，填写模板中的变更摘要、测试结果和影响范围。至少需要一位维护者进行 Code Review。PR 合并后，分支将被自动删除。

## 常见问题

**Q1：LinkTower 是否支持私有化部署，以及是否需要联网才能使用？**

A1：LinkTower 完全支持私有化部署。整个系统除链接可用性检查功能需要向外发出 HTTP 请求之外，其余所有操作（导入、分类、检索、导出）均在本地完成，不向任何第三方服务发送数据。如果运行在完全隔离的内网环境，可以在配置文件中将 `checker.enabled` 设为 `false` 以关闭检测功能，系统仍可正常使用所有管理能力。

**Q2：导入的链接数量达到万级或十万级时，检索和分类性能是否会明显下降？**

A2：LinkTower 默认使用 SQLite 数据库，并为 `url`、`batch`、`created_at` 等字段建立了 B-Tree 索引。在十万条记录规模下，基于关键词的检索响应时间通常在 200 毫秒以内。自动分类操作采用批处理机制，每次最多处理 1000 条，避免长事务锁定。如果数据量超过百万级，建议迁移至 PostgreSQL 后端（项目提供 PG 适配器，详见运维手册）。

**Q3：能否自定义自动分类的规则？规则文件格式是什么？**

A3：可以。分类规则存储在 `config/classifier_rules.yaml` 文件中，用户可直接编辑该文件而无需修改源码。规则采用 YAML 格式，每条规则包含 `domain`（域名匹配）、`path_keywords`（路径关键词列表）和 `assigned_tags`（命中后添加的标签数组）。系统启动时自动加载该文件，修改后执行 `manage.py reload-classifier` 即可热更新，无需重启服务。
