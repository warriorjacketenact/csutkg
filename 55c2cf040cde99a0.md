# Water Resource Data Aggregator

Water Resource Data Aggregator (WRDA) is a specialized technical resource aggregation system designed for hydrology researchers, water conservancy engineers, and environmental data scientists. The project serves as a unified access point for water resource monitoring data, providing structured routing and query capabilities for distributed hydrological datasets. WRDA addresses the critical challenge of fragmented water quality and quantity data sources by implementing a standardized URL routing layer that enables consistent access to diverse monitoring endpoints across multiple water conservancy districts. The system targets professionals working with real-time water level readings, flow rate measurements, precipitation records, and water quality parameters, offering a reliable gateway for automated data collection pipelines and analytical workflows. By consolidating access to numerous regional water monitoring stations through a predictable routing interface, WRDA reduces integration complexity and accelerates the development of water resource management applications.

## 功能概览

**路由解析引擎** - Implements a three-level routing mechanism (cbf/sfj/ful) that maps request patterns to corresponding water monitoring data endpoints with configurable parameter transformation.

**数据缓存层** - Provides in-memory and disk-based caching for frequently accessed hydrological records, reducing upstream API load and improving response times for repeat queries.

**批量请求处理** - Supports concurrent data retrieval across multiple monitoring stations with configurable concurrency limits and retry policies for failed requests.

**数据格式标准化** - Converts heterogeneous data responses from various water conservancy sources into a unified JSON schema with consistent field naming and timestamp formatting.

**健康检查端点** - Exposes monitoring endpoints for tracking upstream service availability, response latency, and data completeness metrics.

**配置热加载** - Allows runtime modification of routing tables and data source credentials without service restart through file-based configuration watching.

**访问日志审计** - Records all request patterns with timestamp, source IP, requested route, and response status for compliance and usage analysis.

**数据导出适配器** - Provides pluggable exporters for common data formats including CSV, Parquet, and GeoTIFF for integration with GIS platforms.

## 应用场景

**区域水情监测系统集成** - Municipal water authorities can deploy WRDA as a middleware layer to aggregate data from multiple upstream monitoring stations along river basins, providing unified access for flood warning systems. The routing engine intelligently distributes queries to appropriate data sources based on geographic coordinates and time ranges.

**学术研究数据采集** - Hydrological research teams utilize WRDA to automate the retrieval of historical water quality parameters across different seasons and weather conditions. The batch processing capability enables efficient collection of large-scale datasets for machine learning model training on water pollution prediction.

**灌溉调度决策支持** - Agricultural water management districts employ WRDA to obtain real-time soil moisture and irrigation water availability data from distributed sensor networks. The standardized data format allows seamless integration with existing farm management dashboards and automated irrigation controllers.

**环境合规报告生成** - Environmental consulting firms leverage WRDA's data export adapters to compile water quality compliance reports for regulatory submissions. The audit logging feature ensures traceability of all data sources used in report generation.

## 快速开始

Clone the repository, install dependencies, and start the aggregator service with default configuration:

```bash
git clone https://github.com/water-resource-aggregator/wrda.git
cd wrda
pip install -r requirements.txt
python -m wrda.server --config config/default.yaml --port 8080
```

For production deployment with systemd service registration:

```bash
sudo cp contrib/wrda.service /etc/systemd/system/
sudo systemctl enable wrda
sudo systemctl start wrda
```

Docker deployment option:

```bash
docker build -t wrda:latest .
docker run -d -p 8080:8080 -v /etc/wrda:/config wrda:latest
```

## 贡献指南

**代码风格与规范** - All contributions must adhere to the project's coding standards defined in the pyproject.toml file. Run black and isort before submitting any pull request to ensure consistent formatting. Type hints are mandatory for all function definitions.

**测试覆盖率要求** - New features and bug fixes must include corresponding unit tests. The overall test coverage should not decrease below 85%. Run pytest with the coverage plugin to verify your changes meet the requirement before opening a pull request.

**提交信息格式** - Follow the conventional commit specification for all commit messages. Use the format type(scope): description where type is one of feat, fix, docs, style, refactor, perf, test, or chore. Include the issue number in the commit body when applicable.

**拉取请求流程** - Create a feature branch from the main branch with a descriptive name. Open a draft pull request early to facilitate discussion and receive feedback. Ensure all CI checks pass before marking the pull request as ready for review. Request at least one maintainer approval before merging.

**文档同步更新** - Any change that affects user-facing functionality or configuration must include corresponding updates to the documentation in the docs/ directory. API changes require updates to the API reference documentation with example requests and responses.

## 常见问题

**Q: 系统如何处理上游数据源响应超时或不可用的情况？**

A: The aggregator implements a configurable retry mechanism with exponential backoff for failed upstream requests. The default configuration retries failed requests up to three times with initial delay of 100ms, doubling after each retry. When all retries are exhausted, the system returns a 503 Service Unavailable response with detailed error context in the response body, including which upstream endpoint failed and the last error message. Administrators can adjust retry policy parameters through the configuration file without restarting the service using the hot-reload feature. Additionally, the health check endpoint tracks upstream availability and can trigger alerting when error rates exceed configurable thresholds.

**Q: 如何验证新增的路由规则是否正确映射到预期的数据源？**

A: WRDA provides a route validation endpoint at /_validate that accepts a test routing configuration and returns the resolved upstream endpoint without actually executing the request. This allows administrators to verify routing logic before deploying changes to production. The validation endpoint also checks for circular dependencies, duplicate route definitions, and parameter compatibility issues. For comprehensive testing, the system includes a route simulation mode that can be enabled via the --dry-run command-line flag, which logs all routing decisions without making external requests. Integration tests in the tests/integration/ directory provide additional validation through actual HTTP requests against staging endpoints, ensuring that routing changes do not break existing data retrieval workflows.

**Q: 部署高可用集群时需要特别注意哪些配置项？**

A: For clustered deployments, the most critical configuration items are the cache backend selection and the shared state management. The in-memory cache is not suitable for multi-instance deployments because each instance maintains its own isolated cache. Redis must be configured as the cache backend to ensure cache consistency and invalidation across all nodes. The session affinity setting for the load balancer should be configured to forward requests from the same client to the same backend instance when sticky sessions are required. The health check endpoint should be configured to return detailed status information including cache connectivity and upstream availability, allowing the load balancer to make informed routing decisions. Administrators must also ensure that the configuration files are synchronized across all nodes, either through a shared network mount or a configuration management system, to prevent configuration drift.
