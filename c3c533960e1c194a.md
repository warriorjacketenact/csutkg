# ResourceNavigator

ResourceNavigator 是一个面向技术开发者与内容研究者的结构化外链资源聚合平台。该项目不生成任何原创内容，专注于对互联网上已有高质量技术资源进行人工筛选、分类归档与索引展示，解决开发者在信息过载环境中难以高效定位有效参考链接的痛点。

项目定位为“技术资源的导航枢纽”，目标用户包括全栈工程师、运维工程师、技术文档撰写者、开源项目维护者以及计算机科学领域的研究人员。通过统一的目录结构与标签体系，ResourceNavigator 将散落在各独立站点上的教程、案例、工具文档与社区讨论整合为可检索、可追溯的静态资源池，显著降低信息查找的时间成本。

## 功能概览

- **按主题分类索引** 系统预设后端架构、前端工程、运维监控、数据库调优、安全审计等五大技术领域分类，每个资源链接依据内容特征自动归入相应分类节点。

- **多维度标签系统** 每个资源条目附带技术栈标签、难度等级标签与内容类型标签，支持快速筛选与交叉检索。

- **静态站点生成** 项目采用纯静态架构，所有资源数据以 Markdown 与 YAML 混合格式存储，构建时生成完整 HTML 页面，无需后端服务即可部署。

- **资源状态检测** 内置链接可用性检查脚本，定期对收录的 URL 进行 HTTP 状态码探测，自动标记失效链接并生成报告。

- **全文检索支持** 集成轻量级客户端搜索组件，用户可在索引范围内对资源标题、描述、标签与来源域名进行关键词匹配。

- **版本化资源库** 每次收录更新均以 Git 提交记录形式保存，支持回溯历史版本与对比资源列表变更差异。

- **自定义分类模板** 高级用户可通过修改配置文件新增分类节点或调整现有分类规则，无需改动核心代码。

- **导入导出接口** 支持以 CSV 与 JSON 格式批量导入资源列表，亦可导出完整索引供第三方工具使用。

## 应用场景

- **技术选型调研** 团队在引入新框架或工具时，可通过 ResourceNavigator 快速查阅相关领域的案例链接与社区讨论，辅助决策过程。所有链接均来自实际生产环境或权威社区，具备较高参考价值。

- **文档编写参考收集** 技术文档工程师在撰写教程或操作手册时，可利用本项目的分类索引收集外部引用素材，确保引用来源的多样性与可靠性，同时避免重复收集已归档链接。

- **开源项目 README 外链整理** 开源项目维护者可将 ResourceNavigator 作为外部资源引用池，在项目文档中直接引用本平台收录的稳定链接，减少各自维护外链列表的工作量。

- **离线知识库构建** 用户可克隆完整仓库，结合自定义脚本批量下载资源页面内容，构建内部离线知识库，适用于网络受限的开发环境。

- **定期技术资讯阅读** 订阅本项目更新日志的开发者可定期获得新增资源列表，以系统化方式跟踪技术社区动态，避免依赖碎片化的社交媒体信息流。

## 快速开始

以下步骤帮助您在本地环境中完成 ResourceNavigator 的克隆、依赖安装与静态站点构建。

```bash
# 克隆项目仓库至本地
git clone https://github.com/resourcenavigator/resourcenavigator.git

# 进入项目根目录
cd resourcenavigator

# 安装项目依赖（基于 Node.js 环境）
npm install

# 执行构建脚本，生成静态站点至 dist 目录
npm run build
```

构建完成后，您可以通过任意 HTTP 服务器预览生成的站点，例如使用 Python 内置模块：

```bash
python3 -m http.server 8080 --directory dist
```

随后在浏览器中访问 `http://localhost:8080` 即可查看本地构建结果。

## 贡献指南

开源项目的持续发展依赖社区贡献者的支持。请遵循以下步骤提交资源收录或代码改进。

1. 复刻项目仓库至个人账户，并在本地克隆复刻后的版本。所有修改均应在独立的功能分支上进行，分支命名遵循 `feature/描述` 或 `fix/描述` 格式。

2. 新增资源链接时，在 `data/resources/` 相应来源目录下创建新的 YAML 文件，填写完整的资源条目信息，包括原始 URL、标题、简短描述、所属分类与标签列表。提交前需运行 `npm run validate` 确保格式合规。

3. 提交代码前执行完整的构建流程 `npm run build`，验证本地站点生成无报错。若涉及模板或样式修改，需确保在主流浏览器中渲染正常。

4. 发起 Pull Request 至主仓库的 `main` 分支，在 PR 描述中清晰说明本次变更的目的、涉及的新增或修改资源条目数量，以及是否涉及破坏性变更。

5. 项目维护者将在两个工作日内审阅 PR，可能提出修改意见。合并后，新增资源将在下一次站点部署中自动上线。

## 常见问题

**问：项目是否提供在线演示站点？**

答：当前版本未提供官方托管演示站点。用户可按照快速开始步骤在本地构建并预览完整站点。项目设计为纯静态架构，用户亦可自行部署至任意支持静态文件托管的平台，如 GitHub Pages、Cloudflare Pages 或 Netlify。

**问：如何批量更新资源链接的状态？**

答：项目根目录下提供了 `scripts/check-links.sh` 脚本，执行该脚本可对 `data/resources/` 中记录的所有 URL 发起 HTTP 请求检测。检测结果输出至 `reports/link-status.csv`，包含每个链接的状态码与响应时间。用户可将此脚本配置为定时任务实现自动化监控。

**问：能否在不重新构建全站的情况下新增单个资源？**

答：项目采用全量构建策略，任何资源条目的增删改均需重新执行构建流程。对于仅需测试新增条目的场景，可使用 `npm run build -- --filter=新条目ID` 进行局部构建预览，但生产部署仍建议执行完整构建以确保索引一致性。
