# ResourceBridge

ResourceBridge 是一个面向开发者与技术研究人员的结构化外链资源汇总平台。项目定位于技术文档、工具链、开源库、学习资料与运维手册等外部链接的集中式索引与管理，帮助团队和个人在信息过载的环境中建立可维护、可追溯的知识资源库。

ResourceBridge 不存储任何实际内容，仅提供链接的元数据描述与分类导航。项目面向日常开发、技术决策、文档查阅、新人入职培训等场景，通过统一的管理规范将分散的优质资源转化为团队可复用的知识资产。项目适合任何规模的开发团队、开源社区贡献者以及希望系统化管理技术链接的个人研究者。

## 功能概览

**结构化资源索引** 支持按技术领域、应用场景、目标受众对资源链接进行多级分类，并提供统一的元数据字段（名称、描述、标签、收录日期）用于快速检索。

**链接状态监测** 周期性对收录的 URL 发起可用性检查，自动标记失效或响应异常的链接，并在管理面板中生成状态报告，确保资源库的长期有效性。

**标签与全文检索** 基于标签体系和资源描述字段提供多维度筛选能力，支持模糊匹配与精确过滤，可在数百条链接中快速定位目标资源。

**权限与协作管理** 内置基于角色的访问控制，支持管理员、编辑者、只读观察员三种权限级别，便于团队协作维护资源库，并记录每次变更的操作日志。

**批量导入与导出** 支持通过 CSV、JSON 格式批量导入外部链接列表，亦可按筛选条件导出为 Markdown 表格或结构化数据文件，便于迁移或归档。

**自定义分类模板** 允许用户根据自身技术栈创建分类模板（如前端工程、后端架构、运维监控、AI 模型等），并将链接批量关联至指定模板，实现个性化导航。

**资源备注与版本标记** 每条链接支持附加维护备注、版本号或快照时间戳，记录该资源在项目中所处的上下文或适配的软件版本，减少信息断层。

## 应用场景

**团队技术选型与决策支持** 在引入新框架或工具时，团队可通过 ResourceBridge 快速查阅已收录的官方文档、社区评测、性能对比报告等链接，减少重复搜索成本，确保决策基于充足且可追溯的信息源。

**新人入职培训知识库** 将项目文档、编码规范、部署流程、常用工具站等链接统一收纳至 ResourceBridge，并分配只读权限，新人可按分类浏览或通过关键词检索，在短时间内掌握团队所依赖的外部资源体系。

**开源项目文档站外链接管理** 开源项目维护者可将项目依赖的第三方库主页、API 参考手册、协议解读文章等外链集中管理，避免在 README 或 Wiki 中散落大量裸链接，提升文档的可维护性和整洁度。

**个人学习路线与收藏夹同步** 开发者可将日常阅读的技术博客、视频教程、在线课程、实验环境入口等链接导入 ResourceBridge 并按主题分类，同时利用导出功能将选中的资源列表转换为个人知识库的参考资料章节。

## 快速开始

以下指令演示了如何在本地环境中部署 ResourceBridge 开发实例。

```bash
# 克隆代码仓库
git clone https://github.com/resourcebridge/resourcebridge.git
cd resourcebridge

# 安装项目依赖（使用 pip 和 requirements.txt）
pip install -r requirements.txt

# 初始化数据库并运行开发服务器
python manage.py migrate
python manage.py runserver
```

完成上述步骤后，在浏览器中访问 http://127.0.0.1:8000 即可进入本地实例。首次启动将自动创建默认管理员账号，控制台会输出初始登录凭证，请及时修改密码。

## 安装要求

ResourceBridge 基于 Python 3.10 及以上版本开发，使用 Django 作为 Web 框架，并依赖 PostgreSQL 作为生产环境数据库。以下为完整的依赖清单与必需环境说明。

| 依赖项 | 必需版本 | 说明 |
|--------|----------|------|
| Python | 3.10 至 3.12 | 运行时环境，低于 3.10 版本将无法解析类型注解语法 |
| PostgreSQL | 14.0 及以上 | 生产环境主数据库，用于存储链接元数据、用户信息与操作日志 |
| Redis | 6.0 及以上 | 用于缓存查询结果和会话存储，非必需但强烈推荐以提升性能 |
| Node.js | 18.0 及以上 | 仅用于前端静态资源构建，后端运行不依赖 Node 运行时 |
| pip | 22.0 及以上 | Python 包管理器，用于安装 requirements.txt 中声明的依赖库 |
| Git | 2.30 及以上 | 用于版本控制管理，以及后续拉取更新和贡献代码 |
| 系统时区 | UTC+8 或 UTC | 需在 settings.py 中明确配置 TIME_ZONE，确保链接时间戳记录一致 |
| 环境变量 | 见 .env.example | 需配置 SECRET_KEY、DATABASE_URL、REDIS_URL 等关键变量 |
| 磁盘空间 | 至少 1 GB | 用于存储静态文件、日志以及 SQLite 本地开发数据库（如使用） |

## 项目结构

项目采用 Django 标准应用布局，并结合前端静态资源分离管理。以下为源码目录树及其功能注释。

```
resourcebridge/
├── manage.py                      # Django 命令行入口，用于运行开发服务器与数据库迁移
├── requirements.txt               # Python 依赖声明文件，包含 Django、psycopg2、redis 等核心库
├── .env.example                   # 环境变量模板文件，供部署前复制并填充实际密钥
├── .gitignore                     # Git 忽略规则，排除缓存、日志、本地数据库等临时文件
```

## 贡献指南

ResourceBridge 欢迎社区贡献，包括但不限于代码修复、功能增强、文档完善和资源分类建议。请遵循以下步骤参与贡献。

第一步：查阅贡献者文档与项目看板
访问 docs/contributing/ 目录下的代码风格指南和测试规范，同时在 GitHub Issues 中查看当前未分配的任务列表，选择适合自身能力的问题进行认领。

第二步：拉取代码并创建功能分支
从主分支 main 拉取最新代码，并基于 main 创建以 feature/ 或 fix/ 为前缀的分支名称，例如 feature/add-batch-tag-editor。禁止直接在 main 分支上提交。

第三步：编写代码并添加对应测试
所有新增功能或修复缺陷均需在 tests/ 目录下补充相应的测试用例，确保测试覆盖率达到 80% 以上。运行 pytest 验证所有现有测试不受影响。

第四步：提交 Pull Request 并填写变更摘要
推送分支至远程仓库后，通过 GitHub 发起 Pull Request 到 main 分支。PR 描述中需清晰说明变更动机、实现方式、测试结果以及关联的 Issue 编号。PR 至少需要一名项目维护者审查通过方可合并。

第五步：签署贡献者许可协议
首次提交 PR 时，需在 PR 评论中声明已阅读并同意项目贡献者许可协议，该协议文本位于 docs/contributing/cla.md。合并后的代码将以 MIT 许可证发布。

## 常见问题

Q：ResourceBridge 是否存储链接指向的实际文件内容或页面快照？
A：不存储。ResourceBridge 仅保存链接的元数据（标题、描述、分类、标签以及收录时间），不抓取或缓存目标页面的任何内容。链接状态监测仅通过 HTTP 响应头判断可达性，不解析页面主体。用户点击链接时将直接跳转至原始目标站点。

Q：如何将现有书签或收藏夹批量导入 ResourceBridge？
A：项目支持通过 CSV 格式批量导入。首先在管理面板中下载导入模板，按列填写链接 URL、标题、描述和标签，然后通过“批量导入”功能上传文件。系统会校验 URL 格式并自动去重，导入完成后生成导入结果报告，包含成功数量、失败记录及原因说明。

Q：链接状态监测是否会因为请求频率过高而被目标站点限制？
A：系统默认将状态监测任务的并发数限制为 5，且每次请求间隔至少 200 毫秒，并在 User-Agent 中标识为 ResourceBridge/1.0 加项目主页 URL。对于已知的高敏感站点，管理员可在配置中将该域名加入白名单，系统将跳过对该域名的自动监测，转为人工确认。
