# ResourceBridge 技术资源导航站

ResourceBridge 是一个面向开发者、技术研究员与开源爱好者的轻量级技术资源导航与聚合平台。该项目定位于将分散在互联网各处的优质技术文档、工具链页面、社区讨论与项目样板聚合为统一的索引体系，帮助用户快速定位所需信息，减少重复检索成本。

本项目并非传统意义上的爬虫或镜像站，而是一个人工筛选与社区驱动相结合的资源目录系统。其核心受众包括正在学习新技术栈的工程师、需要快速查找参考实现的技术负责人、以及希望跟进特定领域动态的研究人员。通过结构化的分类索引与简洁的呈现方式，ResourceBridge 致力于成为每个技术团队内部可私有化部署的知识起点。

## 功能概览

- **分类资源索引**：按编程语言、框架、基础设施、运维监控、算法与数据结构等维度对资源链接进行归类，支持快速筛选。

- **全文搜索与标签过滤**：内置基于倒排索引的轻量级搜索功能，支持对资源标题、描述、标签字段进行模糊匹配与多标签交集过滤。

- **自定义资源订阅源**：允许用户将外部 RSS 或 Atom 订阅源绑定至特定分类，系统自动抓取并更新资源列表状态。

- **访问状态健康检查**：定时对已收录的 URL 发起 HEAD 请求，标记不可达链接并生成可用性报告，便于维护者清理失效条目。

- **资源导入与导出**：支持通过 CSV 与 JSON 格式批量导入外部链接，并可将当前索引完整导出为静态站点数据，用于离线浏览或迁移。

- **用户贡献与投票机制**：注册用户可提交新资源链接，经基础审核后进入待收录池；社区成员可对资源进行有用性投票，高票资源自动提升排序权重。

- **访问统计与热度排行**：记录各资源链接的点击次数与最近访问时间，按日、周、月维度生成热度榜单，辅助发现流行趋势。

## 应用场景

- **技术团队内部知识库搭建**：团队 Leader 可将 ResourceBridge 部署在内网服务器，将团队常用的依赖镜像源、内部文档入口、CI/CD 控制台地址统一收录，新成员入职时仅需访问该导航页即可获取所有必要工具入口。

- **开源项目依赖梳理**：开源项目维护者可以使用本项目的分类索引功能，整理项目所依赖的第三方库主页、官方文档、社区论坛及迁移指南，将分散的外部链接集中管理，便于贡献者查阅。

- **技术选型调研辅助**：架构师在进行技术选型时，可通过 ResourceBridge 建立竞品对比资源集合，将各候选方案的官网、性能测试报告、GitHub 仓库、Stack Overflow 讨论帖归入同一分类，实现一站式对照调研。

- **在线教育课程配套资源**：培训机构或高校教师可将课程各章节涉及的参考资料、实验环境地址、代码示例仓库链接按教学周次分类收录，学生通过导航站即可获取所有课外阅读材料，无需在课件与浏览器书签间反复切换。

## 快速开始

以下步骤适用于在 Linux 或 macOS 开发环境下从源码启动项目。

```bash
# 克隆代码仓库
git clone https://github.com/resourcebridge/resourcebridge.git
cd resourcebridge

# 安装项目依赖（使用 pip 与虚拟环境）
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 初始化数据库并导入默认分类与示例资源
python manage.py migrate
python manage.py loaddata initial_categories.json
python manage.py loaddata initial_resources.json

# 启动开发服务器
python manage.py runserver --host 0.0.0.0 --port 8080
```

启动成功后，访问控制台输出中显示的本地地址即可进入资源导航首页。如需生产环境部署，请参考 `docs/deployment.md` 中的说明配置 WSGI 服务器与反向代理。

## 安装要求

本项目基于 Python 3.10 及以上版本开发，依赖以下关键组件与运行环境。建议在 Ubuntu 20.04 LTS 或同等稳定的操作系统上部署。

| 依赖项 | 必需版本 | 说明 |
|--------|----------|------|
| Python | 3.10 ~ 3.12 | 核心解释器，低于 3.10 将无法解析类型注解中的新语法特性 |
| PostgreSQL | 13.0 及以上 | 生产环境推荐使用 PostgreSQL 作为主数据库，支持 JSONB 字段与全文检索 |
| Redis | 6.0 及以上 | 用于缓存热点查询结果与存储临时会话数据，非必需但强烈建议启用 |
| Node.js | 18.0 及以上 | 仅用于前端静态资源构建（CSS 与 JavaScript 压缩），运行时可忽略 |
| Nginx | 1.20 及以上 | 生产环境反向代理与静态文件服务，开发环境可使用内置服务器替代 |
| Docker | 20.10 及以上 | 若采用容器化部署方式，需安装 Docker Engine 与 Docker Compose |
| Git | 2.25 及以上 | 用于版本控制与从仓库拉取更新，源码安装方式必需 |
| gunicorn | 20.1.0 及以上 | Python WSGI HTTP 服务器，生产环境用于运行应用进程 |
| virtualenv | 20.0 及以上 | 用于创建隔离的 Python 依赖环境，推荐但不强制 |

## 贡献指南

我们欢迎社区以多种形式参与 ResourceBridge 的建设。请遵循以下流程以确保贡献过程顺畅高效。

1. 查阅议题列表与项目看板。访问 GitHub Issues 页面，浏览现有议题或新建议题描述您发现的问题或建议的新功能。对于较大规模的功能改动，建议先通过议题与维护者沟通设计思路，避免重复工作或方向偏离。

2. 派生代码仓库并创建功能分支。将主仓库 Fork 至个人账户下，使用 `git checkout -b feature/your-feature-name` 创建独立分支。分支命名建议遵循 `feature/`、`fix/`、`docs/` 前缀规范。

3. 编写代码与测试用例。所有新增逻辑应包含对应的单元测试，测试覆盖率不得低于原有水平。提交前请运行 `make test` 确保全部测试通过，并执行 `make lint` 检查代码风格是否符合项目规范（gofmt 与 golangci-lint）。

4. 签署开发者原创声明。在 Pull Request 描述中明确确认您的贡献基于当前最新主分支开发，且未侵犯任何第三方知识产权。对于引入的新依赖，需在 PR 中说明其许可证兼容性。

5. 提交 Pull Request 并参与评审。将分支推送到您的派生仓库后，向主仓库的 `main` 分支发起 PR。至少两名项目维护者将进行代码审查，若有修改意见请在原分支上追加提交，评审通过后由维护者执行合并操作。

## 常见问题

**Q：启动时提示数据库连接失败，应该如何排查？**

A：首先检查 `configs/development.yaml` 或环境变量中的 `DATABASE_URL` 是否正确配置了主机地址、端口、用户名、密码与数据库名。确认 PostgreSQL 服务是否已在对应端口监听，可使用 `pg_isready` 命令测试连通性。若使用 Docker Compose 启动，请确保 `docker-compose.yml` 中定义的服务名称与配置文件中的主机名一致。如果数据库尚未初始化，请执行 `python manage.py migrate` 或项目对应的迁移命令创建数据表结构。

**Q：导入包含大量链接的 CSV 文件时发生超时，如何解决？**

A：对于超过 1000 条记录的批量导入操作，系统默认的单次请求超时限制可能导致中途中断。解决方案分为两种：一种是调整 Web 服务器或反向代理的超时阈值，例如在 Nginx 配置中增加 `proxy_read_timeout` 至 300 秒；另一种是使用命令行导入工具 `./cmd/crawler/main.go --batch-import`，该工具独立于 HTTP 请求生命周期运行，不受超时限制影响，且支持断点续传与失败重试。

**Q：搜索功能无法匹配到预期结果，索引是否需要手动重建？**

A：ResourceBridge 的搜索索引采用增量更新策略，在资源新增或修改后自动刷新。若您发现新添加的资源无法被搜索到，请检查该资源是否处于「已发布」状态而非「草稿」或「待审核」。此外，索引更新存在约 5 秒的异步延迟，稍后重试即可。如需强制全量重建索引，可执行 `python manage.py rebuild_index` 命令，该操作会消耗较多系统资源，建议在低峰时段进行。
