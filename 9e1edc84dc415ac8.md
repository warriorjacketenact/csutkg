# Tancheng Legal Resource Aggregator

Tancheng Legal Resource Aggregator is a curated collection of legal case episodes, judicial interpretations, and regulatory references sourced from the Tancheng legal domain. This project serves as a structured knowledge base for legal professionals, researchers, and policy analysts who require rapid access to a wide spectrum of case-oriented legal materials. The aggregator indexes over 250 distinct episode entries, each representing a discrete legal document, case summary, or regulatory annotation.

The primary goal of this project is to eliminate the fragmentation of legal information by providing a unified, machine-readable index of episode identifiers. Rather than navigating disparate court portals or regulatory databases, users can leverage this aggregator to obtain consistent references, perform batch queries, and integrate legal metadata into automated workflows. The project is designed for legal tech developers, compliance officers, and academic researchers who need reproducible and traceable citation paths.

## 功能概览

**批量引用生成** – Generates standardized citation keys for each episode, enabling consistent cross-referencing across legal documents and research papers.

**元数据解析管道** – Extracts and normalizes embedded metadata from episode URL patterns, including episode codes, category hints, and sequential identifiers.

**全文检索接口** – Provides a lightweight query interface for filtering episodes by code prefix, date range, or keyword annotations extracted from document summaries.

**变更追踪机制** – Monitors episode URLs for availability and content modification, alerting administrators to broken links or updated document versions.

**导出与互操作** – Supports export of episode lists in JSON, CSV, and plain-text formats for integration with external case management systems or data analysis tools.

**访问频率统计** – Tracks access patterns per episode to identify high-usage documents and inform caching strategies for performance optimization.

**权限分级框架** – Implements role-based access controls to restrict certain episodes to authorized legal practitioners or institutional subscribers.

**审计日志记录** – Maintains comprehensive audit trails of all query operations, satisfying compliance requirements for legal data handling.

## 应用场景

**法律文献综述撰写** – Researchers compiling systematic reviews of case law can use the aggregator to rapidly enumerate relevant episodes, verify citation completeness, and export reference lists without manual copying of URLs.

**合规性自动化测试** – Enterprise compliance teams can integrate the episode index into automated screening pipelines, cross-referencing internal policy documents against the aggregated legal resources to identify potential regulatory gaps.

**法律知识图谱构建** – Knowledge engineers can leverage the structured episode list as a seed set for building ontological relationships between legal concepts, court decisions, and statutory provisions, with each episode serving as a distinct node.

**教育案例库维护** – Law school faculty and curriculum designers can maintain a curated subset of episodes for teaching purposes, using the aggregator's filtering capabilities to select relevant materials for specific course modules.

**司法数据分析** – Data scientists can apply statistical methods to the episode collection, analyzing temporal distribution, topic clustering, and citation networks to uncover patterns in legal decision-making.

## 快速开始

```bash
# Clone the repository
git clone https://github.com/tancheng-legal/aggregator.git
cd aggregator

# Install dependencies
pip install -r requirements.txt

# Run the initial indexing pipeline
python scripts/index.py --source data/episodes.lst --output index.db

# Start the local query server
python server.py --port 8080
```

## 项目结构

```
aggregator/
├── data/                           # 原始数据存储目录
│   ├── episodes.lst                # 主索引列表，每行一个 episode 标识
│   ├── metadata.yaml               # 各 episode 的自定义标签与分类映射
│   └── cache/                      # 本地缓存目录，存储已抓取的文档摘要
├── src/                            # 核心源代码
│   ├── indexer/                    # 索引构建模块
│   │   ├── parser.py               # URL 解析与 episode 代码提取
│   │   ├── builder.py              # 数据库表结构与索引构建逻辑
│   │   └── validator.py            # URL 可达性与内容类型校验
│   ├── query/                      # 查询处理模块
│   │   ├── engine.py               # 查询执行引擎，支持布尔与模糊匹配
│   │   ├── filters.py              # 过滤条件构造器（按前缀、日期、标签）
│   │   └── formatter.py            # 结果格式化（JSON, CSV, 表格输出）
│   ├── server/                     # Web 服务模块
│   │   ├── app.py                  # Flask 应用入口
│   │   ├── routes.py               # REST 端点定义
│   │   └── auth.py                 # 基于 JWT 的权限验证
│   └── monitor/                    # 监控与审计模块
│       ├── checker.py              # 定时 URL 可用性检查
│       ├── logger.py               # 审计日志写入
│       └── reporter.py             # 状态报告生成器
├── tests/                          # 单元测试与集成测试
│   ├── test_parser.py              # 解析器功能测试
│   ├── test_engine.py              # 查询引擎性能与正确性测试
│   └── test_routes.py              # API 端点行为测试
```

## 贡献指南

**提交新 episode 条目** – 通过拉取请求将新的 URL 添加到 data/episodes.lst 文件中。每个条目必须包含完整的 episode URL，并附上简要的变更说明，解释该条目的来源和类别归属。

**完善元数据标注** – 在 data/metadata.yaml 中为现有 episode 补充或修正分类标签、日期信息和关键词。标注应基于文档实际内容，避免主观臆断。提交前请运行 scripts/validate.py 验证 YAML 格式正确性。

**改进查询引擎性能** – 针对 src/query/engine.py 中的查询逻辑提出优化方案。建议包含基准测试数据，证明改进后的查询响应时间至少降低 15%。所有性能改进需通过 tests/test_engine.py 中的回归测试。

**完善文档与示例** – 修正 docs/ 目录中的拼写错误、逻辑不清晰之处，或补充新的使用示例。文档类贡献需同时更新对应的中文版本（如果存在）或同步维护多语言内容。

**报告问题与建议** – 使用 GitHub Issues 提交错误报告或功能请求。报告应包含可复现的步骤、预期行为和实际行为描述。功能建议需说明使用场景和预期受益范围。

## 常见问题

**问：如何验证某个 episode URL 是否仍然有效？**  
答：项目内置了监控模块 src/monitor/checker.py，可通过命令行运行 python scripts/check.py --episode <code> 单独检查指定条目，或运行 python scripts/check.py --all 执行全量检查。检查结果会更新到 data/cache/status.json 中，并可通过 API 端点 /api/status 查询。

**问：索引数据库能否支持多用户并发查询？**  
答：默认 SQLite 配置支持读并发，但写操作（如更新元数据）会锁定数据库。对于生产环境的高并发场景，建议迁移至 PostgreSQL 并启用连接池。项目提供了迁移脚本 scripts/migrate.py 用于转换数据，具体步骤参考 docs/deployment.md 中的数据库迁移章节。

**问：如何自定义 episode 的元数据分类体系？**  
答：编辑 data/metadata.yaml 文件，在对应 episode 条目下添加或修改 category、tags、date 等字段。分类体系完全可定制，项目不对元数据结构施加任何强制约束。修改后需重新运行索引脚本 python scripts/index.py --refresh 以更新数据库中的元数据记录。
