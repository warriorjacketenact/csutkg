# LinkVault Resource Aggregator

LinkVault Resource Aggregator 是一个面向开发者、技术研究人员及内容创作者的轻量级外链资源汇总与导航系统。该项目定位于解决分散化技术资源难以统一管理、快速检索与持续追踪的问题，通过结构化索引与分类标签机制，将大量分散的在线工具、文档、社区及参考资料整合为单一可维护的知识库体系。

目标用户包括需要频繁查阅外部技术资料的全栈工程师、从事竞品分析与市场调研的产品经理、以及需要系统化整理学习路径的自学者。LinkVault 不依赖复杂后端服务，采用纯静态站点生成策略，可低成本部署于任意 Web 服务器或对象存储平台，同时保留完整的资源元数据与分类层级，便于团队内部共享与版本控制。

## 功能概览

- **批量资源导入与去重**：支持从 CSV、JSON 及 Markdown 列表批量导入 URL，自动识别重复条目并生成冲突报告，保障资源库的整洁性。

- **多维度分类标签系统**：每个资源条目可关联多个自定义标签（如 "API 文档"、"视频教程"、"社区论坛"），支持层级标签树与组合过滤，满足复杂检索需求。

- **全文检索与快速过滤**：基于倒排索引实现资源标题、描述、标签及来源域名的全文搜索，配合按字母序、导入时间、访问频率等排序方式，提升信息定位效率。

- **健康状态巡检**：定期对已收录 URL 进行可达性检测与 SSL 证书有效期检查，标记失效链接并生成告警日志，保证资源库的长期可用性。

- **自定义元数据扩展**：每条资源支持附加备注、评级（1-5 星）、阅读状态（未读/在读/已读）及个人笔记，满足个性化知识管理需求。

- **静态站点导出**：一键生成完整的静态 HTML 站点，包含资源列表、分类索引、搜索页面及 sitemap，可直接部署至 GitHub Pages、Cloudflare Pages 或本地 Web 服务器。

- **RESTful API 接口**：提供基于 JSON 的只读 API，支持按 ID、标签、关键词查询资源列表，便于与其他工具（如 Alfred、Raycast、Zapier）集成。

- **数据备份与迁移**：内置导出为便携式 SQLite 或 JSON 格式的功能，支持跨设备同步与版本回滚，避免供应商锁定。

## 应用场景

- **技术团队内部知识库构建**：团队可将日常开发中遇到的第三方库、在线调试工具、规范文档等统一收录至 LinkVault，新成员入职时可快速访问团队认可的优质资源，减少重复搜索成本。

- **开源项目依赖与参考整理**：维护者可将项目所依赖的底层框架、协议规范、社区讨论帖等外链集中管理，在 README 或贡献指南中引用 LinkVault 生成的资源列表，提升项目文档的完整性与可追溯性。

- **个人学习路径系统化管理**：学习者可按技术领域（如机器学习、前端框架、数据库原理）建立专属资源库，标记学习进度与重要程度，形成可持续迭代的个人知识地图，避免收藏夹的杂乱无序。

- **技术资讯与周报素材采集**：内容创作者可利用标签与备注功能快速归类每周发现的优质文章、视频与工具，生成周报草稿或话题索引，显著提高内容产出的组织效率。

- **离线环境下的资源镜像准备**：通过健康巡检与批量导出功能，运维人员可预先筛选可用资源列表，配合离线下载工具为内网环境准备完整的文档镜像，满足隔离网络下的技术参考需求。

## 快速开始

以下步骤将帮助您在本地环境快速启动 LinkVault 服务。

```bash
# 克隆项目仓库
git clone https://github.com/linkvault/linkvault-core.git
cd linkvault-core

# 安装项目依赖（使用 pnpm 或 npm）
pnpm install
# 或者 npm install

# 构建静态站点并启动开发预览服务器
pnpm run build
pnpm run preview
# 默认访问地址为 http://localhost:4173
```

若需导入用户提供的初始资源列表，请将包含 URL 的文本文件放置于 `./data/raw/` 目录下，执行以下命令：

```bash
pnpm run import -- --source ./data/raw/urls.txt --format plain
```

导入完成后，可通过 `pnpm run serve` 启动带热重载的开发环境进行本地调试。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | >= 20.0.0 | 运行时环境，需支持 ES2022 特性与原生 fetch API |
| pnpm | >= 8.0.0 | 推荐包管理器，亦可使用 npm 9.x 或 yarn 4.x 替代 |
| SQLite3 | >= 3.40.0 | 用于本地资源元数据存储与查询索引，生产环境可换用 PostgreSQL |

| jq | >= 1.6 | 可选，用于 API 响应的命令行格式化与调试 |
| Docker | >= 24.0.0 | 可选，用于容器化部署与隔离环境测试 |

## 贡献指南

LinkVault 遵循开源社区协作规范，欢迎所有形式的贡献，包括但不限于新增资源条目、改进文档、提交缺陷修复与功能增强。请遵循以下步骤：

1.  **查阅现有议题与项目看板**：在提交 Pull Request 前，请先浏览 GitHub Issues 与 Projects 看板，确认当前是否有类似工作正在进行，避免重复劳动。对于较大规模的功能变更，建议先创建讨论议题，与维护者达成共识后再编写代码。

2.  **派生仓库并创建功能分支**：将主仓库派生至个人账户，基于 `main` 分支创建以 `feature/` 或 `fix/` 为前缀的新分支。分支命名应简明描述变更内容，例如 `feature/add-json-import-plugin` 或 `fix/health-check-timeout`。

3.  **编写或更新测试用例**：所有代码变更必须附带对应的单元测试或集成测试，确保新功能的正确性且不破坏现有行为。测试覆盖率不低于 85%。文档类变更需在 `docs/` 目录下同步更新相关章节。

4.  **遵循编码与提交规范**：TypeScript 代码须通过 ESLint 与 Prettier 检查，提交信息采用 Conventional Commits 格式（如 `feat:`, `fix:`, `docs:`, `chore:`）。提交前执行 `pnpm run lint` 与 `pnpm run test` 确保全部通过。

5.  **发起 Pull Request 并参与评审**：将分支推送至派生仓库后，向主仓库的 `main` 分支发起 Pull Request。填写 PR 模板中的检查清单，详细描述变更动机与实现方式。评审过程中积极回应反馈意见，直至代码合并。

## 常见问题

**问：导入包含大量 URL 的文本文件时，如何避免内存溢出？**

答：LinkVault 的导入模块采用流式处理策略，默认每批处理 500 条记录并自动提交事务，因此可安全处理数万级别的资源列表。若仍需进一步降低内存占用，可在 `./configs/import.config.json` 中调整 `batchSize` 参数。同时建议将输入文件编码统一为 UTF-8，避免特殊字符导致的解析异常。

**问：健康巡检模块如何区分临时性不可达与永久失效链接？**

答：巡检模块对每个 URL 执行三次重试（间隔 5 秒、30 秒、120 秒），若全部超时或返回 5xx 状态码则标记为 "异常"。对于返回 404 的状态码，由于语义明确，立即标记为 "失效" 并记录。所有异常记录会汇总至每日告警邮件中，用户可在 Web 界面手动重新验证或调整超时阈值（默认 10 秒）。

**问：能否在纯内网环境中部署 LinkVault，且不使用任何外部 CDN 资源？**

答：可以。LinkVault 的静态站点生成模式默认将所有依赖（包括 CSS、JavaScript 及字体文件）内联打包至输出目录，不发起任何外部请求。健康巡检模块也可通过配置 `health_checker.offlineMode=true` 关闭外部探测，仅依赖本地缓存的元数据提供服务。部署时只需将 `public/` 目录完整复制至内网 Web 服务器即可。
