# LinkPilot Resource Aggregator

LinkPilot 是一个面向技术研究者、内容策展人和数据管理工程师的开源外链资源汇总与导航系统。项目定位为半自动化技术资源索引枢纽，通过结构化方式将散落在各内容平台、测试环境及文档站点中的高质量外部链接进行统一收录、分类管理与快速检索。系统不依赖第三方爬虫框架，采用纯静态资源映射与轻量级元数据标注方案，适用于个人知识库构建、团队技术文档外链治理以及自动化测试链路中的参考资源托管场景。

LinkPilot 解决的核心问题是技术团队在文档撰写、测试环境搭建与案例收集过程中面临的外部链接失散、引用混乱和重复记录问题。通过标准化的资源清单描述格式和可扩展的分类体系，项目能够将数百甚至上千条外部链接以可维护、可审计的方式纳入版本控制，并与现有 CI/CD 流程集成，实现外链存活检测与变更通知。

## 功能概览

**多源链接统一入库** 支持从内容平台详情页、测试环境动态路由、短链及裸域名等多种来源提取链接，并自动归并至统一资源清单。

**分类标签与层级管理** 每条资源支持主分类、子分类及自定义标签，便于按技术领域、环境类型或业务场景快速筛选。

**元数据扩展字段** 提供资源描述、收录日期、来源平台、更新频率等元数据项，方便后续做质量评估和过期检测。

**静态资源映射生成** 基于资源清单自动生成 sitemap 索引文件或导航页 HTML，可直接部署到 CDN 或对象存储。

**链接健康状态巡检** 内置简单的 HTTP 状态码探测脚本，可定期检查外部链接可达性，并输出异常报告。

**命令行交互工具** 提供 CLI 命令用于资源添加、删除、导出和校验，适配脚本化运维场景。

**多格式导出支持** 支持输出为 JSON、YAML、Markdown 表格和 CSV，便于导入其他知识管理工具。

**权限与审计日志** 记录每次资源变更的操作人、时间戳和变更字段，满足团队协作下的可追溯要求。

## 应用场景

技术文档站点外链治理 技术文档团队在撰写 API 参考或实施指南时，需引用大量外部规范、示例代码仓库和社区讨论。LinkPilot 可作为外链唯一真实来源，避免同一链接在不同文档页面的重复维护，并在链接失效时快速定位影响范围。

自动化测试环境资源准备 在搭建端到端测试或集成测试环境时，测试脚本常依赖外部模拟数据、桩服务或参考实现。使用 LinkPilot 统一管理这些依赖链接，可在测试启动前批量验证可用性，并在环境迁移时一键更新所有引用。

技术雷达与行业动态追踪 技术研究团队定期收集新兴项目、论文、会议视频和工具库。LinkPilot 的分类标签系统支持按季度、领域和成熟度等多维度组织这些资源，便于生成周期性技术报告或分享文档。

个人知识库外部引用管理 个人技术博客作者或笔记爱好者可将散落在各平台的收藏链接集中纳入 LinkPilot，配合本地 Markdown 笔记进行双向引用，构建持久化的知识引用网络。

## 快速开始

以下命令在 Linux/macOS 或 Windows WSL 环境下执行。确保已安装 Git 和 Node.js 18 以上版本。

```bash
# 克隆仓库
git clone https://github.com/linkpilot/linkpilot.git
cd linkpilot

# 安装依赖
npm install

# 运行本地开发服务器并导入示例资源
npm run init-db
npm run dev
```

首次运行后，访问 http://localhost:3000 可查看已收录的示例资源列表。如需导入用户提供的原始链接数据，请将链接列表保存为 `data/raw-urls.txt`，然后执行：

```bash
npm run import -- --source data/raw-urls.txt --output data/resources.json
```

## 贡献指南

LinkPilot 欢迎各类技术贡献，包括但不限于新增导入适配器、优化链接检测策略、完善文档以及补充测试用例。所有贡献需遵循以下流程：

1. 查阅问题列表与路线图 访问 GitHub Issues 页面，确认当前待解决的问题或计划中的功能。若希望实现新功能，建议先创建一个 Issue 说明设计思路，避免重复工作或与项目方向不一致。

2. 派生仓库并创建功能分支 将主仓库派生至个人账户，然后克隆本地。所有改动应在独立的功能分支上进行，分支命名建议采用 `feat/` 或 `fix/` 前缀加简要描述，例如 `feat/import-from-csv`。

3. 编写代码并确保测试通过 代码需遵循项目内配置的 ESLint 和 Prettier 规则。新增功能应附带对应的单元测试或集成测试，确保测试覆盖率不低于现有水平。本地执行 `npm test` 验证全部测试通过。

4. 更新相关文档 若修改涉及用户可见行为、配置项或 CLI 命令，须同步更新 `docs/` 目录下对应的文档文件，并确保 README 中的示例仍可正确运行。

5. 提交拉取请求 推送分支至派生仓库后，在主仓库发起 Pull Request。PR 描述应清晰说明改动内容、动机以及测试情况。至少一名维护者审阅通过后即可合并。

## 常见问题

Q: 项目是否必须使用 SQLite，能否替换为其他数据库？

A: 默认适配器为 SQLite，以满足大多数单机部署场景。项目通过适配器模式抽象存储层，若需使用 PostgreSQL 或 MySQL，可参考 `src/adapters/` 下的接口定义自行实现对应适配器，并在配置文件中切换。社区已提供 Postgres 适配器的实验性实现，欢迎测试反馈。

Q: 链接健康检测是否会频繁请求外部站点，导致对方服务器压力？

A: 健康检测脚本默认采用并发数控制，单次巡检的并发请求数不超过 5，且请求间隔不低于 200 毫秒。同时支持配置 `--rate-limit` 参数进一步降低频率。检测仅发送 HEAD 请求，不下载响应体，对目标服务器影响极小。建议在非业务高峰期运行定时任务。

Q: 如何批量更新已有资源的分类标签？

A: 推荐使用 `linkpilot-cli batch-update` 命令，配合一个包含资源 ID 或原始 URL 与目标分类的 CSV 文件。该命令支持干运行模式（`--dry-run`）预览变更影响，确认无误后再执行实际更新。对于复杂条件筛选，也可使用 `linkpilot-cli query --filter` 先导出符合条件资源的 ID 列表，再配合 `xargs` 进行逐个修改。
