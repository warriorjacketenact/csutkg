# SiteBloom Playbook

SiteBloom Playbook 是一个面向技术团队与内容运营人员的交互式外链资源管理与导航系统。本项目并非传统意义上的静态链接仓库，而是一个具备动态编排能力、状态追踪机制与访问审计功能的轻量级技术资源汇总平台。其核心定位在于帮助中小型开发团队、独立开发者以及技术内容创作者，将散落在日常浏览、项目调研和技术文档编写过程中积累的大量外链资源，进行结构化存储、分类索引与团队共享。

本项目解决的核心问题包括：技术调研过程中的链接遗忘与丢失、团队内部知识共享时外链传递的混乱与不可控、以及项目文档中引用外部资源无法统一更新和维护的痛点。SiteBloom Playbook 通过内置的资源生命周期标记、访问频次统计与失效链接检测机制，使得外链资源从一个静态的“收藏夹”变为具备可操作性的“技术资产”。

## 功能概览

- **资源快照抓取**：系统在录入外链时自动抓取目标页面的标题、元描述与关键词信息，形成基础索引数据，便于后续检索与分类。

- **分类标签体系**：支持用户自定义多级标签与分类命名空间，可按照技术领域、项目归属、阅读状态、重要程度等多个维度对资源进行交叉标记。

- **访问状态监控**：定期对已收录的外链进行可达性检查，自动标记响应超时、状态码异常或内容变更的链接，并提供状态变更通知。

- **全文检索与筛选**：基于标题、描述、标签、收录时间与访问状态等字段提供组合筛选与全文搜索能力，支持模糊匹配与精确过滤。

- **团队协作与注释**：支持多用户环境下的资源注释与批注功能，团队成员可针对特定链接添加技术评价、使用心得或警告提示。

- **导入导出与迁移**：提供标准格式的批量导入接口与导出功能，支持从主流书签系统或文档格式迁移数据，也支持将资源列表输出为结构化文档。

- **访问统计与热度分析**：记录每个资源的点击次数、最后访问时间与访问来源，生成简单热度排行，辅助团队识别高频使用的核心资源。

## 应用场景

- **技术选型调研**：在进行技术选型或框架评估时，开发人员通常需要同时查阅数十篇对比文章、官方文档与社区讨论。SiteBloom Playbook 可以帮助团队统一收集这些链接，并通过标签标记不同框架的优缺点对比，最终形成可回溯的选型依据文档。

- **项目文档维护**：项目 README、设计文档或 API 文档中经常需要引用外部规范、教程或依赖项目地址。使用本系统管理这些引用链接，可以在外部资源发生变动时快速定位并更新文档中的对应条目，避免文档中出现死链。

- **新人入职指引**：团队可以将内部常用工具地址、编码规范、部署文档、日志平台入口等资源统一录入系统，并为新人分配只读访问权限。配合注释功能，资深成员还可以在关键链接旁添加操作注意事项，大幅降低新人上手阶段的沟通成本。

- **技术周报与知识沉淀**：技术负责人或内容运营人员可以将每周浏览到的优质文章、开源项目发布、行业动态等链接集中收录，并利用分类标签按主题归档。一段时间后，这些数据自然形成团队的知识索引库，便于后续回顾与整理。

## 快速开始

以下指令演示了从代码仓库克隆项目、安装依赖并启动开发服务的基本流程。

```bash
# 克隆项目仓库
git clone https://github.com/sitebloom/playbook.git

# 进入项目目录
cd playbook

# 安装项目依赖
npm install

# 初始化本地配置文件
cp .env.example .env

# 运行开发服务器
npm run dev
```

执行上述命令后，开发服务器默认会在本地的 3000 端口启动。打开浏览器访问 `http://localhost:3000` 即可进入 SiteBloom Playbook 的主界面。首次启动时，系统会自动创建示例数据并引导用户完成管理员账号的初始化设置。

## 安装要求

本项目基于 Node.js 运行时环境构建，数据库层使用 SQLite 作为默认存储引擎，同时也支持 PostgreSQL 作为生产环境备选方案。以下表格列出了运行本项目的所有必要依赖与建议配置。

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | 18.x 或 20.x LTS | 项目运行时环境，建议使用最新的 LTS 版本以确保兼容性 |
| npm | 9.x 或以上 | 包管理器，用于安装项目依赖与执行脚本命令 |
| SQLite | 3.x 内置 | 开发与测试环境的默认数据库，无需额外安装 |
| PostgreSQL | 14.x 或以上 | 生产环境推荐数据库，需单独安装并配置连接字符串 |
| Redis | 7.x 或以上 | 可选组件，用于缓存与会话存储，提升高并发场景下的性能 |
| Git | 2.x 或以上 | 用于版本控制与项目克隆操作 |

## 项目结构

项目采用模块化的分层架构设计，核心代码与资源文件按照功能职责组织到不同的顶层目录中。以下为项目主要目录结构及其功能说明。

```
playbook/
├── apps/                                # 应用程序入口与路由配置
│   ├── api/                             # RESTful API 路由处理函数
│   │   ├── resources/                   # 资源增删改查接口
│   │   ├── tags/                        # 标签管理接口
│   │   └── monitoring/                  # 监控与状态检查接口
│   └── web/                             # Web 界面路由与页面组件
│       ├── dashboard/                   # 仪表盘与统计视图
│       └── settings/                    # 系统配置与用户设置界面
├── core/                                # 核心业务逻辑与数据模型
│   ├── models/                          # 数据库模型定义（资源、标签、用户）
│   ├── services/                        # 业务服务层（索引、抓取、监控）
│   └── adapters/                        # 数据库适配器与连接管理
├── libs/                                # 通用工具库与辅助函数
│   ├── crawler/                         # 页面抓取与元数据提取工具
│   ├── checker/                         # 链接可达性检查器
│   └── utils/                           # 字符串、日期、加密等基础工具
├── config/                              # 环境配置文件与默认参数
│   ├── default.yaml                     # 默认配置项
│   └── schema.yaml                      # 配置项校验规则
├── tests/                               # 单元测试与集成测试用例
│   ├── unit/                            # 各模块单元测试
│   └── integration/                     # API 与数据库集成测试
├── docs/                                # 项目文档源文件
│   ├── user-guide/                      # 用户手册
│   ├── admin-guide/                     # 管理指南
│   └── developer/                       # 开发者文档
├── scripts/                             # 运维与工具脚本
│   ├── migrate.js                       # 数据库迁移脚本
│   └── seed.js                          # 初始数据填充脚本
├── public/                              # 静态资源文件
```

## 贡献指南

SiteBloom Playbook 遵循开源社区协作规范，欢迎各类形式的贡献，包括但不限于功能建议、缺陷报告、文档改进与代码提交。所有贡献者请遵循以下流程：

1. 在提交代码或文档变更之前，请先在 GitHub Issues 中查找是否存在相关的讨论或已认领的任务。若没有，请新建一个 Issue 描述您希望处理的问题或功能，等待维护者确认后再开始工作。

2. 从项目主分支签出新的功能分支，分支命名建议遵循 `feat/`、`fix/`、`docs/` 前缀加简要描述的方式，例如 `feat/batch-import`。所有提交信息请使用英文书写，并遵循 Conventional Commits 规范。

3. 开发完成后，请确保所有现有测试用例通过，并为新增功能或修复补充对应的单元测试。提交前运行 `npm run lint` 与 `npm run test` 进行代码风格检查与测试验证。

4. 推送分支到远程仓库后，通过 GitHub 界面发起 Pull Request 到主分支。PR 描述中请关联对应的 Issue 编号，并简述变更内容与测试覆盖情况。

5. 项目维护者将在 PR 发起后的 7 个工作日内进行审查。审查过程中可能会提出修改意见，贡献者需及时响应并更新提交。合并后，贡献者将自动列入项目的贡献者名单。

## 常见问题

**系统支持同时管理多少条外链资源？性能是否会下降？**

系统在设计时充分考虑了资源数量的扩展性。在默认的 SQLite 配置下，单实例可稳定管理约 5 万条资源记录，检索响应时间保持在 200 毫秒以内。当资源数量超过 10 万条时，建议切换到 PostgreSQL 数据库并启用 Redis 缓存层，以维持相同的查询性能。实际的性能瓶颈主要取决于服务器内存与数据库查询优化，项目文档中提供了详细的性能调优参数说明。

**监控功能会对外部目标站点造成请求压力吗？**

监控模块默认采用指数退避策略进行状态检查，每个资源的检查间隔不低于 24 小时，且所有检查请求均携带标准的 User-Agent 头与缓存控制指令。系统不会对单个目标域名发起高频请求，也不会模拟浏览器行为加载页面中的二次资源。用户可以在配置文件中调整检查频率与超时时间，以满足自身对监控实时性与目标站点友好度之间的平衡需求。

**如何将已有的书签数据批量导入系统？**

系统支持导入多种主流书签导出格式，包括 HTML 书签文件、JSON 格式的结构化数据以及 CSV 表格文件。用户可以在后台设置的“导入”功能中上传文件，系统会自动解析并映射到对应的标题、URL 与标签字段。对于自定义格式的数据，项目提供了导入模板与字段映射配置文档，用户可根据实际需求调整映射规则后执行导入。
