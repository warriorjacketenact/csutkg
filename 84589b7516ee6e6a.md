# Hyperlink Atlas

Hyperlink Atlas 是一个面向技术研究人员、数据分析师与内容策展人的外链资源归集与导航系统。该项目并非传统的网络书签工具，而是一个具备结构化索引能力的外链资产映射平台，专注于对特定域名下的深层链接进行批量抓取、分类存储与状态监控。Hyperlink Atlas 解决的核心问题是：当项目或团队需要长期跟踪某一垂直领域的海量外部链接时，如何避免链接散落、失效与上下文丢失，从而将零散的 URL 转化为可检索、可审计的持久化资源清单。

本项目特别适用于需要频繁引用特定来源站点的文档撰写团队、依赖外部数据接口的测试环境维护者，以及需要对外链进行分类分级管理的安全合规人员。Hyperlink Atlas 不提供爬虫框架或动态渲染引擎，而是以轻量级脚本和静态清单的方式，为固定批次的外链资源提供稳定的索引载体与变更追踪能力。

## 功能概览

**批量链接导入** 支持从纯文本文件、CSV 或直接粘贴的链接列表中批量导入 URL，自动去重并校验协议格式，对于不符合规范的条目给出警告日志。

**结构化分类存储** 所有链接按照用户定义的一级分类（如文档站、API 端点、社区论坛、静态资源等）进行目录化存储，每条记录附带导入时间戳与自定义标签。

**可用性健康检查** 内置简单的 HTTP 状态码探测模块，可对已存储的链接进行定期可达性检测，并将异常状态（4xx、5xx、超时）单独标记至异常清单。

**Markdown 清单自动生成** 根据存储的链接数据，自动生成符合项目 README 规范的资源列表章节，支持按分类小节输出纯 URL 列表，不附加任何额外格式。

**变更历史审计** 记录每次链接增删改的操作日志，包括操作时间、操作人和变更前后的 URL 值，便于回溯资源清单的演变过程。

**标签筛选与全文检索** 基于简单的文本索引，允许用户通过标签或 URL 片段快速定位特定资源，提升维护效率。

## 应用场景

**技术文档团队的外链引用管理** 技术文档撰写者通常需要在文章中引用大量外部规范、SDK 下载页或 API 参考文档。Hyperlink Atlas 可作为这些外链的中转索引，当源站 URL 发生变动时，团队只需更新索引中的单条记录，无需逐个修改文档正文。

**渗透测试与漏洞验证的资源准备** 安全测试人员在进行授权渗透时，常需要收集目标域名的各类接口路径与历史备份文件。本项目的批量导入与健康检查功能可帮助测试人员快速筛选出仍然活跃的潜在测试目标链接。

**数据采集任务的目标源维护** 数据工程师在构建采集管道时，往往需要维护一个稳定的起始 URL 池。Hyperlink Atlas 的变更审计与分类存储机制，使得采集源头的增删操作有据可查，避免因链接变动导致采集任务大面积失败。

**开源项目 README 的资源链接批次生成** 对于像本项目一样需要维护大量外链列表的开源仓库，Hyperlink Atlas 可作为子模块，将原始 URL 数据与 README 展示层解耦，通过脚本自动生成符合格式要求的资源章节，减少手工排版错误。

## 快速开始

以下操作假设用户已具备基础的 Shell 环境与 Node.js 运行时。

```bash
# 克隆项目仓库至本地
git clone https://github.com/atlas-hyperlink/hyperlink-atlas.git

# 进入项目根目录并安装依赖
cd hyperlink-atlas
npm install

# 运行链接索引构建脚本，生成资源清单与健康报告
npm run build -- --batch=778
```

执行完上述命令后，项目会在 `./output` 目录下生成当前批次的链接索引文件 `index-778.json` 以及 Markdown 格式的资源列表 `resources-778.md`。用户可通过编辑 `./data/raw-links-778.txt` 来增删链接，随后重新运行构建命令即可更新输出。

## 安装要求

| 依赖 | 必需 | 说明 |
|------|------|------|
| Node.js 18.x 或更高 | 是 | 项目运行时基础环境，推荐使用 LTS 版本 |
| npm 9.x 或更高 | 是 | 依赖包管理工具，与 Node.js 捆绑安装 |
| 网络连通性（外网） | 是 | 执行健康检查时需要向目标域名发起 HTTP 请求 |
| 磁盘读写权限（项目目录） | 是 | 用于写入日志文件、缓存与输出文件 |
| Git 2.x 或更高 | 否 | 仅初次克隆仓库时使用，后续可通过压缩包方式获取源码 |
| curl 或 wget | 否 | 用于独立测试链接可达性，非核心依赖 |

## 贡献指南

1.  **问题反馈与功能请求**：请先查阅 `docs/user-guide.md` 与 `docs/configuration.md` 确认是否为已有功能或已知限制。若确认问题存在或功能缺失，请在 GitHub Issues 中提交详细描述，包括操作系统版本、Node.js 版本、复现步骤及期望行为。

2.  **代码贡献流程**：Fork 本仓库至个人账户，在 `dev` 分支基础上创建功能分支（命名格式为 `feature/简要功能描述` 或 `fix/问题简述`）。完成代码修改后，请确保所有现有单元测试通过，并为新增代码补充对应的测试用例。提交前运行 `npm run lint` 检查代码风格。

3.  **文档改进**：欢迎对文档进行修正与补充。文档位于 `docs/` 目录，采用 Markdown 格式。请保持技术术语的一致性，并确保更新后的文档与代码实际行为相符。文档变更可单独提交 Pull Request。

4.  **链接数据更新**：若需更新资源列表中的 URL，请编辑 `data/raw-links-*.txt` 文件，随后运行 `npm run build -- --batch=批次号` 重新生成索引与 Markdown 清单。请勿手动修改 `output/` 目录下的生成文件。

5.  **提交信息规范**：提交信息请采用语义化格式，包含标题与正文，标题以 `feat:`、`fix:`、`docs:`、`chore:` 等前缀开头，并简要说明变更内容。正文描述变更原因与影响范围。

## 常见问题

**问：健康检查模块是否会频繁请求目标站点，导致源站压力？**

答：健康检查模块默认采用串行请求，且每个请求之间间隔 500 毫秒。超时时间设置为 5 秒，仅对单批次的链接执行一次性检查，不会重复轮询。用户可通过配置文件调整间隔与超时参数，但建议在生产环境中保持默认值以避免对源站造成意外负担。

**问：如何迁移已有的链接清单至 Hyperlink Atlas？**

答：本项目支持通过 `data/raw-links-批次号.txt` 文件批量导入链接。用户只需将原有的链接列表（每行一个 URL）复制到该文本文件中，然后执行 `npm run build -- --batch=批次号` 即可完成导入与索引生成。若原有清单包含分类信息，可通过 `tags-mapping.json` 建立正则匹配规则，实现自动分类。

**问：生成的 Markdown 资源列表为何不包含链接文字或标题？**

答：Hyperlink Atlas 的设计目标之一是保持输出结果的确定性与最小化外部依赖。不抓取页面标题可避免额外的网络请求与解析开销，同时防止因目标站点变更加剧输出变动。用户若需为链接附加备注，可在 `data/raw-links-批次号.txt` 中以 `URL#备注内容` 的格式扩展，构建脚本会将其作为注释保留至生成的 Markdown 中。
