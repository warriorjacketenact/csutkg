# ResourceHub

ResourceHub 是一个面向开发者、技术研究人员与内容创作者的综合性技术资源导航与外部链接管理平台。该项目旨在解决个人与团队在技术学习与项目调研过程中面临的信息分散、优质资源难以沉淀、外链缺乏统一管理入口等痛点，通过结构化的链接分类与轻量级的元数据标注，帮助用户高效地构建、维护和共享属于自己的技术资源索引库。

项目定位为轻量级、可自托管的技术外链汇总站，特别适合技术团队内部知识库建设、开源项目参考资料整理、技术写作素材管理以及个人学习路径规划等使用场景。ResourceHub 不依赖复杂的前端框架，以纯静态页面为核心，兼顾内容管理的便捷性与访问速度，同时提供清晰的数据组织规范，便于二次开发与数据迁移。

## 功能概览

**多维度链接分类体系** 支持按照技术领域、内容类型、来源站点等维度对收录的 URL 进行标签化分类，满足不同视角下的资源检索需求。

**批量链接导入与解析** 提供基于文本列表的批量链接导入能力，自动识别 URL 并提取页面标题与摘要信息，大幅降低手动录入成本。

**自定义标签与备注系统** 允许用户为每一条链接添加自定义标签、重要性评级和文字备注，便于记录个人对资源的理解与使用心得。

**全文检索与多字段筛选** 支持对链接标题、描述、标签和备注内容进行全文关键词检索，同时可按分类、标签、来源域名进行组合筛选。

**链接状态健康检查** 内置定时链接可用性检测机制，自动标记失效或响应超时的链接，帮助维护资源库的准确性与可用性。

**数据导入导出与备份恢复** 提供 JSON、CSV、Markdown 表格等多种数据格式的导入导出功能，支持完整的数据备份与跨实例迁移。

**响应式布局与移动端适配** 前端页面采用响应式设计，在桌面端、平板与手机设备上均能获得良好的浏览与操作体验。

**暗色主题与阅读模式** 内置明暗两套配色方案，并提供专注阅读模式，在浏览长文档或技术文章时降低视觉疲劳。

## 应用场景

技术团队内部知识库建设。团队可以将日常开发中遇到的高质量技术博客、官方文档、API 参考、问题排查记录等外部链接统一收录到 ResourceHub 中，并添加团队内部使用的标签与备注，形成结构化的团队技术知识沉淀。

开源项目参考资料整理。开源项目维护者可以使用 ResourceHub 整理项目相关的依赖库文档、社区讨论帖、扩展工具列表、部署教程等外部资源，作为项目 README 或官方文档的补充导航页，降低新贡献者的上手门槛。

个人技术学习路径规划。技术学习者可以按照学习主题创建链接集合，例如 Go 语言入门、分布式系统设计、云原生运维等，在长期学习过程中持续积累优质资料，形成个人专属的技术成长路线图。

技术写作与内容策划素材管理。技术博主、专栏作者和内容运营人员可以使用 ResourceHub 收集写作灵感来源、数据统计页面、参考文献链接和行业报告地址，在策划系列内容时快速调取相关素材。

## 快速开始

以下命令演示了从 GitHub 克隆项目、安装依赖并启动开发服务器的完整流程。

```bash
git clone https://github.com/resourcehub/resourcehub.git
cd resourcehub
npm install
npm run dev
```

执行上述命令后，开发服务器默认启动在 http://localhost:3000。访问该地址即可进入 ResourceHub 的本地实例，开始链接的添加与管理操作。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | 18.x 或 20.x LTS | 项目运行时的 JavaScript 运行时环境 |
| npm | 9.x 或 10.x | Node.js 包管理器，用于安装项目依赖 |
| SQLite | 3.x（内置，无需额外安装） | 默认轻量级嵌入式数据库，适用于单机部署 |
| PostgreSQL（可选） | 14.x 或 15.x | 生产环境推荐使用，支持高并发与集群部署 |
| Redis（可选） | 7.x | 用于缓存与会话存储，提升多实例部署性能 |
| Nginx（可选） | 1.22+ | 推荐用作反向代理服务器，提供静态资源缓存与负载均衡 |

## 项目结构

```
resourcehub/
├── backend/                    # 后端服务代码目录
│   ├── src/
│   │   ├── api/               # RESTful API 路由定义与请求处理
│   │   ├── core/              # 核心业务逻辑：链接管理、分类、标签
│   │   ├── models/            # 数据模型定义（SQLAlchemy / Prisma）
│   │   ├── services/          # 外部服务集成：链接健康检查、页面元数据抓取
│   │   └── utils/             # 通用工具函数：日志、配置、数据验证
│   ├── tests/                 # 后端单元测试与集成测试用例
│   └── package.json           # 后端项目依赖声明
├── frontend/                   # 前端页面代码目录
│   ├── src/
│   │   ├── components/        # Vue / React 可复用组件
│   │   ├── pages/             # 路由级页面组件
│   │   ├── composables/       # 组合式逻辑复用（Vue Composition API）
│   │   ├── stores/            # Pinia / Redux 状态管理
│   │   └── styles/            # 全局样式变量与主题配置
│   ├── public/                # 静态资源：图标、字体、无需构建的图片
```

## 贡献指南

ResourceHub 持续欢迎社区贡献者参与项目建设。无论您是想报告问题、提交代码改进、完善文档还是提出新功能建议，都请遵循以下协作流程。

第一步：提交 Issue 进行需求沟通。在提交 Pull Request 之前，请先在 GitHub Issues 中创建一个新议题，清晰描述您发现的问题或希望实现的功能，并与维护者达成共识，避免无效的工作投入。

第二步：Fork 项目并创建特性分支。从主仓库 Fork 代码到个人账号下，然后在本地基于 main 分支创建一个描述性的新分支，例如 feature/improve-search-performance 或 fix/link-checker-timeout。

第三步：编码与自测。在本地开发环境中完成代码编写，确保代码风格符合项目 ESLint 与 Prettier 配置要求，并且所有新增或修改的功能都包含相应的单元测试和集成测试用例，保证测试通过率不低于原有水平。

第四步：提交 Pull Request。将本地分支推送至个人 Fork 仓库后，在 GitHub 上向主仓库的 main 分支发起 Pull Request。PR 标题应简洁概括变更内容，正文中需引用相关的 Issue 编号，并逐条列出主要改动点。

第五步：参与 Code Review 并完成合并。维护者会对 PR 进行审查，可能提出修改意见。请及时响应反馈并更新代码。经过至少一名维护者批准且所有 CI 检查通过后，PR 将被合并进入主分支。

## 常见问题

Q：ResourceHub 必须使用 PostgreSQL 吗？能否直接用 SQLite 运行在生产环境？

A：ResourceHub 默认内置 SQLite 支持，无需额外配置即可运行，适合个人使用、本地测试或小型团队部署。但如果您的链接数量超过一万条，或者有多个用户同时读写，建议切换到 PostgreSQL 以获得更好的并发性能和稳定性。通过修改环境变量中的 DATABASE_URL 即可切换数据库类型，无需改动代码。

Q：链接健康检查功能会影响页面访问速度吗？

A：链接健康检查通过独立的后台定时任务执行，默认每 24 小时运行一次，完全异步执行，不会阻塞前端页面渲染或 API 请求。您也可以在生产环境中关闭自动检查，改用手动触发方式，以避免在业务高峰期产生额外的网络开销。

Q：如何将已有的书签或收藏夹数据导入 ResourceHub？

A：ResourceHub 支持导入 Chrome 书签导出的 HTML 文件、标准 CSV 格式表格以及纯文本 URL 列表。您可以在数据管理页面选择对应的导入方式，系统会自动解析并创建链接条目。对于大规模数据（超过 500 条），建议使用命令行脚本 scripts/import.js 进行批量导入，速度更快且支持断点续传。
