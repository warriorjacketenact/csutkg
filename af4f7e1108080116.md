# LinkSphere

LinkSphere 是一个面向开发者与技术研究人员的结构化外链资源聚合平台。该项目不属于传统的代码库或框架，而是一个精心编排的技术资源索引系统，旨在解决信息碎片化时代下高质量技术内容难以检索与归类的问题。LinkSphere 通过人工筛选与机器辅助分类，将散布于网络各处的深度技术文章、工具文档、架构设计案例以及运维最佳实践整合为统一的检索与导航体系。

本项目服务于以下三类核心用户：第一，需要快速查阅特定技术栈底层实现细节的研发工程师；第二，负责技术选型与架构评审的团队技术负责人；第三，希望建立系统化学习路径的计算机专业学生与初级开发者。LinkSphere 不生成原创技术内容，而是充当一个可靠的、持续维护的信息路由枢纽，帮助用户从海量低质量信息中脱离，直达具有实际参考价值的原始出处。

## 功能概览

- **按技术领域分类索引**：资源依据编程语言、中间件、云原生、数据库、操作系统等一级分类进行归档，每个大类下细分二级标签体系，方便用户按技术栈快速定位。

- **全文元数据检索**：每个收录的资源条目均包含标题摘要、关键词标签、发布日期、原始域名可信度评分等元数据字段，用户可通过关键词组合检索精确定位到具体页面。

- **历史版本归档记录**：对于已收录的资源链接，系统定期检测目标页面的可用性与内容变更，提供历史快照日期标记，帮助用户判断信息时效性。

- **社区热度排序**：基于外部引用次数与内部用户点击行为数据，对资源进行热度排序，突出当前行业关注度较高的技术话题与解决方案文章。

- **技术栈依赖关系图**：针对涉及多技术组件协作的架构类资源，系统生成可视化的依赖关系图谱，展示各组件间的数据流与控制流关系。

- **阅读进度追踪**：为注册用户提供已读/未读标记与进度百分比记录功能，支持在多设备间同步阅读状态，便于长期跟进系列文章或长篇技术手册。

- **自定义收藏夹与标签**：用户可创建多个独立收藏夹并按项目或学习主题命名，同时支持为收藏的资源打上自定义标签，实现个人维度的二次分类。

- **RSS 订阅源生成**：每个分类与标签页面均提供 RSS 订阅链接，用户可在 RSS 阅读器中接收新增资源的自动推送，无需频繁手动访问平台。

## 应用场景

**技术选型调研**：架构师在引入新的消息队列或数据库组件时，可通过 LinkSphere 检索该技术领域的对比评测文章、性能压测报告以及生产环境落地案例，集中获取多视角参考信息，缩短调研周期。

**故障排查参考**：运维工程师在遇到内核参数调优、JVM 内存溢出或分布式事务超时等复杂问题时，可使用平台查找对应关键词下的深度排查文章与社区问题讨论串，借鉴他人经验加速根因定位。

**技术学习路径规划**：刚接触 Kubernetes 或 Rust 语言的开发者，可以利用本平台按难度标签从入门教程到高级实践进行筛选，聚合高质量学习材料，避免在低质量重复内容上耗费时间。

**技术文档撰写参考**：技术作者在撰写组件使用手册或架构设计文档时，可通过 LinkSphere 查阅同类文档的结构组织方式、术语使用习惯以及图表表达风格，作为自身文档写作的格式参考基准。

## 快速开始

以下指令演示如何获取 LinkSphere 项目源码并在本地启动开发环境。项目基于 Node.js 与 React 构建，前端采用 Next.js 框架，后端数据服务依赖 Elasticsearch 与 PostgreSQL。

```bash
# 克隆项目仓库
git clone https://github.com/linksphere/linksphere-core.git

# 进入项目根目录
cd linksphere-core

# 安装项目依赖（使用 npm）
npm install

# 复制环境变量模板并填充配置参数
cp .env.example .env.local

# 启动开发服务器（默认监听 3000 端口）
npm run dev
```

生产环境部署时，请参考 `docs/deployment.md` 文档配置 Nginx 反向代理与 PM2 进程管理。前端静态资源通过 CDN 分发，需在 `.env.production` 中正确填写 `NEXT_PUBLIC_CDN_BASE_URL` 变量。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | 18.17.0 或更高 | 项目运行时基础环境，推荐使用 LTS 版本 |
| npm | 9.0.0 或更高 | 包管理器，用于安装项目依赖与执行脚本 |
| PostgreSQL | 14.0 或更高 | 关系数据库，存储用户数据、收藏夹、标签与阅读状态 |
| Elasticsearch | 8.8.0 或更高 | 搜索引擎，提供全文检索与资源分类索引能力 |
| Redis | 7.0.0 或更高 | 缓存中间件，用于会话存储与热点数据缓存加速 |
| PM2 | 5.0.0 或更高（生产环境） | Node.js 进程守护工具，保障服务持续运行 |
| Nginx | 1.24.0 或更高（生产环境） | Web 服务器与反向代理，处理静态资源与负载均衡 |

开发环境最低配置要求为 8GB 内存与 4 核 CPU，存储空间至少 20GB 用于存放依赖与构建产物。生产环境建议使用 16GB 以上内存，并为 PostgreSQL 与 Elasticsearch 配置独立的 SSD 数据盘以提高 I/O 性能。

## 贡献指南

欢迎社区成员参与 LinkSphere 资源库的建设与维护。本项目遵循开源社区协作规范，所有贡献者需遵守行为准则并按照既定流程提交变更。具体参与方式如下：

第一步，在 GitHub 仓库中查看 `issues` 列表，寻找带有 `help-wanted` 或 `good-first-issue` 标签的任务。现有任务包括新增资源分类、更新失效链接、补充资源摘要说明等。若无合适任务，可自行创建 Issue 描述建议新增的资源链接或改进方向，等待维护者确认后开始工作。

第二步，从主分支 `main` 签出新的功能分支，分支命名遵循 `feat/描述` 或 `fix/描述` 格式。在本地完成资源条目新增或元数据修改后，运行测试套件确保现有功能未被破坏。新增资源链接需包含完整的 URL、标题摘要、关键词以及至少一个分类标签，格式可参考 `.github/resource-template.json` 模板文件。

第三步，提交代码变更时遵循 Conventional Commits 规范，提交信息首行使用 `feat:`、`fix:`、`docs:`、`chore:` 等前缀。提交后推送至远程仓库并发起 Pull Request，PR 描述中需链接对应的 Issue 编号，并填写变更摘要与测试覆盖说明。PR 需要至少两名项目维护者审核通过后方可合并。

## 常见问题

Q: 如何报告一个资源链接已经失效或内容被移除？

A: 可通过两种方式报告：第一，在 GitHub 仓库中提交 Issue，标题注明 `[Broken Link]` 并附上具体 URL 与页面 ID；第二，在平台内使用页面底部的「报告问题」按钮提交反馈。维护团队会在 48 小时内验证链接状态，若确认失效则会从索引中移除或标记为不可用，同时查找替代来源。

Q: 平台收录资源的选择标准是什么？是否可以提交自己写的博客文章？

A: 收录标准主要包含三个维度：内容的技术深度（需超越基础入门级别）、原始来源的可信度（偏好官方文档、知名社区或经过验证的个人技术博客）、时效性（优先收录最近 18 个月内发布的内容）。欢迎提交原创技术文章，提交前请确保文章内容客观、数据准确且未过度包含商业推广成分。提交入口位于平台首页底部的「推荐资源」表单页面。
