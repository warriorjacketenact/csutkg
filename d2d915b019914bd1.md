# LinkVault Resource Aggregator

LinkVault 是一个面向技术研究者、内容创作者和数字化运维人员的结构化外链资源归集与健康度监控平台。本项目不对资源内容本身做二次分发，而是提供一套标准化的引用登记、元数据提取与可用性探测框架，帮助团队将散落在各处的第三方链接纳入统一的资产管理视图。

目标用户包括：需要维护大量外部引用链接的技术文档编写者、负责第三方服务商信息归集的运维工程师、以及需要定期核验合作方页面有效性的商务运营人员。LinkVault 通过声明式的链接配置与可插拔的检测器，将原本零散的“人工点检”升级为“定期自动巡检”，并输出结构化的变更报告。

## 功能概览

**批量链接导入**：支持从 CSV、JSON 以及原始文本列表中批量导入待管理的 URL，自动解析协议与域名，生成内部唯一标识。

**可用性周期检测**：基于可配置的 cron 表达式，对每一个登记链接发起 HEAD 与 GET 请求，记录状态码、响应时间与内容哈希，识别页面改版或失效。

**元数据智能补全**：对可访问的链接自动拉取标题、描述与关键词，支持覆盖人工标注，便于后续检索与分类。

**变更差异告警**：当检测到链接响应码从 2xx 变为非 2xx，或页面标题发生显著变化时，通过 Webhook 输出差异载荷，可对接飞书、钉钉或自建通知服务。

**多租户数据隔离**：每个项目（Project）拥有独立的链接空间与检测历史，适合多团队共用同一套部署实例。

**只读只写 API**：提供 RESTful 接口用于链接的增删改查与检测结果查询，所有写操作均记录操作日志，支持审计追溯。

**快照比对视图**：在 Web 管理面板中以时间轴形式展示单条链接的历次检测结果，直观对比内容变迁。

## 应用场景

**技术文档站外链维护**：技术博客或开源项目文档中常引用大量外部规范、SDK 下载页与社区讨论帖。LinkVault 可定期检测这些外链是否仍然有效，避免读者遇到 404 页面，维护文档的专业形象。

**第三方供应商信息看板**：企业运营团队需要跟踪多家服务商的官网公告、状态页与合同续约入口。LinkVault 将相关链接统一登记后，通过检测响应内容的变化，辅助判断供应商是否有版本更新或服务条款调整。

**历史活动资源归档核查**：线上峰会或黑客松活动结束后，活动页、报名表单、资料下载链接等常常迁移或下线。LinkVault 的持续检测能力可帮助归档人员及时发现断链，并在必要时向参与者提供备用地址。

**合规性外链备案管理**：金融、医疗等领域对对外发布内容中的跳转链接有备案与审查要求。LinkVault 输出的检测日志可作为链接存续状态的辅助证明材料，降低合规风险。

## 快速开始

以下命令将完成 LinkVault 源码克隆、依赖安装与服务启动。默认使用 SQLite 作为存储引擎，无需额外安装数据库软件。

```bash
git clone https://github.com/linkvault/linkvault-core.git
cd linkvault-core
pip install -r requirements.txt
python manage.py runserver --host 0.0.0.0 --port 8080
```

启动成功后，访问 http://localhost:8080 进入初始化向导，创建管理员账户与默认项目空间。首次启动会自动执行数据库迁移，并生成示例配置文件 config/example.yaml。

## 贡献指南

1. 查阅 issue 列表，认领未被指派的 feature 或 bug 任务，或提交新的 issue 描述你发现的问题与改进建议。建议在开发前先通过 issue 与维护者确认方向，避免重复劳动。

2. 派生本项目至你的个人仓库，在本地创建功能分支（如 feat/add-telegram-notifier），遵循项目已有的代码风格与目录结构。Go 代码需通过 golangci-lint 检查，Python 脚本需符合 PEP8。

3. 针对新增功能编写单元测试与集成测试，确保测试覆盖率达到 80% 以上。对于链接检测器的改动，需在 test/fixture 中准备对应的模拟服务（mock server）以验证不同响应场景。

4. 提交 pull request 至主仓库的 develop 分支，在 PR 描述中清晰关联对应的 issue 编号，并附上手动测试的截图或日志片段。维护者将在 3 个工作日内进行 review。

5. 更新 docs/ 目录下的相关文档，若新增了配置项或 API 字段，必须同步修改配置手册与 API 参考文档。文档变更与代码变更应在同一个 PR 中提交。

## 常见问题

**问：检测任务是否会因为网络抖动产生大量误报？如何配置重试机制？**

答：LinkVault 对每一次检测采用指数退避重试策略，默认重试 2 次，首次重试延迟 1 秒，第二次延迟 2 秒。若所有重试均失败，才标记为不可用。你可以在 config/default.yaml 中调整 retry.max_attempts 与 retry.backoff_seconds 参数来适配网络环境。对于已知的维护窗口，可以使用 pause_detection 接口按项目维度暂停检测。

**问：如何从其它系统迁移已有的链接列表到 LinkVault？**

答：支持通过 RESTful API 的 /api/v1/links/batch 端点批量提交，请求体为 JSON 数组，每个元素包含 url、project_id、tags 与 custom_metadata 字段。你也可以使用 scripts/seed_links.py 脚本读取 CSV 文件，该脚本会自动处理 URL 格式校验与重复项去重。迁移完成后，系统会返回每个链接的导入状态，失败条目会记录具体错误原因。

**问：检测历史数据会无限增长吗？有没有自动清理策略？**

答：系统默认保留每条链接最近 90 天的检测快照，超出保留期的数据会在每日凌晨 2 点由 cleanup 任务自动归档至压缩包并删除。保留时长可在 config/default.yaml 的 retention.days 字段中修改。若需永久保留，可配置 archive.enabled 为 true，将历史数据同步至外部对象存储。
