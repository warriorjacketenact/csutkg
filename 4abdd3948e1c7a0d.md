# Tangalooma Resource Aggregator

Tangalooma Resource Aggregator 是一个面向技术研究者、内容策展人与信息分析团队的高密度外链资源归集系统。本项目定位于将分散于 Tangalooma House 内容网络中的电影与媒体相关资源条目进行结构化整理与统一呈现，形成可检索、可维护、可扩展的静态资源索引层。

本项目不提供具体的视频播放或文件下载功能，而是作为元数据与入口链接的集中管理仓库，帮助用户在大量动态生成的资源页面之间建立清晰的导航路径。项目内置了完整的 URL 归集模块、路径规范检查工具以及静态站点生成所需的目录模板，适用于需要批量维护外链列表、定期同步第三方内容源、或构建轻量级资源导航站的技术团队。

Tangalooma Resource Aggregator 采用纯静态 Markdown 与 Shell 脚本驱动，无需数据库支持，开箱即用，可无缝集成至现有的 CI/CD 工作流或文档站点生成管道中。

## 功能概览

**资源链接归集**：自动从指定数据源提取并规整全部外链 URL，保留原始协议、域名与路径结构，确保链接在迁移与分发过程中不发生任何改写。

**路径规范校验**：内置链接格式检查器，可识别并报告不符合预期格式的 URL 条目，防止因人为编辑导致的协议缺失或路径截断问题。

**批量元数据标注**：支持为每个资源链接附加来源批次编号、采集时间戳与内容类型标签，便于进行多维度筛选与审计追踪。

**分层目录结构生成**：根据链接路径中的分类字段自动生成多级目录树，将同类资源聚合在同一逻辑分组下，降低人工维护成本。

**静态站点预览模式**：提供轻量级本地开发服务器，可在不部署生产环境的情况下预览资源列表的渲染效果与导航层级。

**增量更新支持**：通过版本化记录文件实现仅同步新增或变更链接的增量更新机制，避免全量重建导致的资源消耗。

## 应用场景

内容聚合平台运维：适用于需要从多个来源持续采集外部链接并统一展示的导航站或资源库，定期运行归集脚本即可完成列表同步。

技术文档站外链管理：适用于大型技术文档项目中对外部参考资料链接的集中管控，将分散在各章节的外链统一纳入版本控制。

数据审计与合规检查：在需要对外链进行定期合规审查的场景中，可利用本项目的结构化输出快速比对链接状态与内容类别。

离线资源备份规划：通过本项目的完整 URL 清单，运维团队可制定针对性的离线缓存策略或 CDN 预热方案。

## 快速开始

```bash
# 克隆项目仓库至本地
git clone https://github.com/tangalooma/resource-aggregator.git

# 进入项目根目录并安装基础依赖（需 Python 3.8+ 及 pip）
cd resource-aggregator
pip install -r requirements.txt

# 执行资源归集主程序，生成完整的资源列表文档
python aggregate.py --input ./data/raw_urls.txt --output ./docs/resources.md
```

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.8 或更高 | 核心归集脚本与校验工具运行环境 |
| pip | 20.0 或更高 | Python 包管理工具，用于安装依赖库 |
| Git | 2.25 或更高 | 用于克隆仓库及版本管理操作 |
| Markdown 解析器 | Python-markdown 3.3+ | 用于生成预览页面的 HTML 渲染 |
| Shell 环境 | Bash 4.0 或更高 | 用于执行自动化构建脚本与路径检查 |
| 文件系统权限 | 读/写 | 需要创建输出目录及写入生成的 Markdown 文件 |
| 网络连接 | 出站可达 | 用于采集阶段验证 URL 可达性（可选） |
| 内存 | 512 MB 以上 | 处理包含 250 个条目以上的批次时建议 |
| 磁盘空间 | 100 MB 以上 | 存放源码、输出文档及临时缓存 |

## 贡献指南

提交资源更新请求：通过 GitHub Issues 提交新增或失效链接的报告，需附带原始 URL 与建议的分类标签。提交前请确认链接符合项目保留原格式的硬性规定。

执行本地预校验：在提交 Pull Request 前，请运行 scripts/validate_urls.sh 对新增或修改的链接进行格式检查，确保无协议改写或路径截断问题。

编写变更日志：对于涉及批次号更新或目录结构调整的改动，需在 docs/changelog.md 中记录修改内容、影响范围及操作时间。

遵循目录命名规范：新增子目录或分类时，必须使用小写字母与连字符组合，并在对应目录下创建 index.md 作为导航页。

## 常见问题

Q: 为什么资源列表中的 URL 必须一字不差原样输出，不能添加或修改协议前缀？

A: 原始链接的协议与域名结构直接关联第三方内容源的路由规则与鉴权机制。任何改动均可能导致链接失效或访问权限错误。项目设计原则强调对外链的绝对忠实记录，确保用户通过项目文档访问时能获得与源站完全一致的行为表现。

Q: 如何处理资源链接失效或目标页面内容变更的情况？

A: 项目本身不负责验证目标页面的可访问性或内容有效性。用户可通过提交 Issue 报告失效链接，维护团队将在下一个同步周期中标记并移除确认无效的条目。项目提供链接格式校验工具，但该工具仅检查 URL 字符串的语法正确性，不发起实际网络请求。

Q: 是否支持自动从远程数据源拉取 URL 列表？

A: 当前稳定版本仅支持从本地 data/raw_urls.txt 文件读取输入。对于需要周期性自动采集的场景，建议结合外部 CI 任务（如 GitHub Actions）定期替换 raw_urls.txt 内容并触发构建流程，生成新的 resources.md 文档。
