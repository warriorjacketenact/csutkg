# Video Resource Aggregator VRA

Video Resource Aggregator (VRA) 是一个面向技术内容创作者、在线教育平台运营者以及数字媒体研究人员的视频资源外链管理与聚合系统。该项目并非视频托管平台，而是专注于对分散于互联网各处的优质视频内容进行结构化梳理、分类索引与元数据提取，帮助用户从海量视频链接中快速定位高价值资源。

VRA 的核心定位在于解决视频资源链接失散、缺乏上下文描述、无法批量校验可用性等常见问题。项目内置了链接健康度检测、自动标签生成、访问频率统计等工具链，可作为个人知识库的视频组件，也可作为团队协作场景下的资源共享中台。当前批次收录资源共计 250 条，全部为经过初步筛选的视频外链，涵盖技术讲座、产品演示、学术报告等多种类型。

## 功能概览

批量链接导入与解析 支持从纯文本、CSV 以及 JSON 格式批量导入视频链接，自动识别链接来源域名并提取路径参数，生成唯一资源标识符。

链接可达性主动检测 定时对收录链接发起 HEAD 请求，检测目标服务器响应状态码，标记失效链接并记录重定向链路，提供可视化健康报告。

元数据自动补全 对可访问链接尝试获取页面标题、描述字段以及 Open Graph 协议中的视频信息，填充资源摘要与分类标签。

自定义标签与分级体系 允许用户为每个链接添加自定义标签（如"前端工程"、"系统设计"、"入门教程"），并设定重要程度等级，支持多维度筛选。

资源使用统计面板 记录每个链接的访问次数、最后访问时间以及引用来源，生成热度趋势图，辅助判断资源长期价值。

数据导入导出标准化 支持将整个资源库导出为 JSON Schema 定义的标准格式，便于与其他知识管理工具（如 Obsidian、Notion）对接。

命令行交互与 API 服务 提供 CLI 工具完成日常管理操作，同时暴露 RESTful API 供第三方应用集成，支持 OAuth 简单鉴权。

多用户隔离与权限控制 区分管理员、编辑者、访客三种角色，团队协作场景下可控制资源的新增、编辑与删除权限。

## 应用场景

技术团队内部知识库建设 开发团队可将日常遇到的优秀技术演讲、框架发布会录像、架构设计复盘视频统一收录至 VRA，为新成员提供体系化的学习路径参考。团队管理者可通过访问统计了解哪些资源被高频使用，据此调整培训计划。

在线教育课程辅助素材管理 教育机构在制作 MOOC 或企业内训课程时，需要引用大量外部视频作为延伸阅读材料。VRA 可帮助课程设计者批量维护这些外链，定期检测链接有效性，避免课程上线后出现大量 404 空链。

个人技术博主资源积累 独立技术博客作者在撰写教程或行业分析文章时，常需引用多个视频来源。使用 VRA 可建立个人引用库，写作时快速检索相关视频链接，同时通过标签体系对资源进行精细分类，提升创作效率。

学术研究文献视频资料整理 人文社科或传播学领域的研究人员在分析网络视频内容生态时，需要收集特定主题下的大量视频样本。VRA 的批量导入与元数据提取功能可显著降低手动整理成本，并提供结构化数据输出便于后续定量分析。

## 快速开始

以下命令演示了如何在 Linux 或 macOS 环境中完成 VRA 项目的克隆、依赖安装与服务启动。

```bash
git clone https://github.com/vra-open/vra-core.git
cd vra-core
pip install -r requirements.txt
python -m vra.server --host 127.0.0.1 --port 8080
```

若需要使用 Docker 容器化部署，可执行以下替代方案：

```bash
docker build -t vra-core .
docker run -d -p 8080:8080 --name vra-instance vra-core
```

首次启动后，系统将在当前目录下生成 config.yaml 配置文件与 data/ 数据目录，请根据实际环境修改数据库连接字符串与缓存路径。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.10 及以上 | 核心运行环境，建议使用 pyenv 或 conda 管理版本 |
| SQLite | 3.35 及以上 | 默认嵌入式数据库，用于存储链接元数据与访问日志 |
| Redis | 7.0 及以上 | 可选依赖，用于高频访问资源的缓存加速与分布式锁 |
| curl | 7.68 及以上 | 链接健康检测底层工具，系统需预装 libcurl 开发库 |
| Node.js | 18 LTS 及以上 | 仅在前端管理面板构建时需要，后端服务不依赖 |
| Docker | 20.10 及以上 | 若采用容器化部署则必需，否则可跳过 |
| git | 2.25 及以上 | 用于克隆仓库及后续拉取更新 |

## 项目结构

项目采用分层架构设计，核心模块与辅助工具分离，便于独立维护与扩展。

```
vra-core/
├── cmd/                                 # 命令行入口与子命令实现
│   ├── cli.py                           # 主 CLI 调度器，解析子命令参数
│   └── commands/                        # 各子命令模块，如 add、check、export
├── internal/                            # 内部私有包，不对外暴露
│   ├── core/                            # 核心业务逻辑：资源管理、标签引擎
│   │   ├── resource.py                  # Resource 数据类与状态机
│   │   ├── tagger.py                    # 基于规则的自动打标签器
│   │   └── checker.py                   # 链接健康度检查器，含重试策略
│   ├── storage/                         # 数据持久化层
│   │   ├── sqlite_store.py              # SQLite 实现，含表结构定义
│   │   ├── redis_cache.py               # Redis 缓存装饰器与连接池
│   │   └── schema.sql                   # 初始化 DDL 脚本
│   └── server/                          # HTTP API 服务实现
│       ├── app.py                       # Flask/FastAPI 应用工厂
│       ├── routes/                      # 路由分组：资源、标签、统计
│       └── middlewares/                 # 鉴权、日志、跨域中间件
├── pkg/                                 # 可复用的公共库
│   ├── http/                            # HTTP 客户端封装，含超时与代理
│   ├── utils/                           # 通用工具函数：时间、哈希、字符串
│   └── validators/                      # URL 验证、输入清洗
├── web/                                 # 前端管理面板源码（Vue/React）
│   ├── src/                             # 组件、视图、状态管理
│   ├── public/                          # 静态资源入口
│   └── dist/                            # 构建输出目录（gitignore）
├── configs/                             # 配置文件模板与环境变量示例
│   ├── config.dev.yaml                  # 开发环境配置
│   ├── config.prod.yaml                 # 生产环境配置
│   └── logging.conf                     # Python logging 配置
```

## 常见问题

问：VRA 是否支持对私有视频源（如企业内网或需要登录的平台）进行链接检测？

答：当前版本仅支持对公开可访问的 URL 进行 HEAD 请求与元数据提取。对于需要 Cookie 或 Bearer Token 认证的资源，您可以在 config.yaml 中配置全局请求头或针对特定域名设置白名单策略。未来版本将考虑引入自定义鉴权钩子机制，允许用户注入登录逻辑。

问：导入大量链接时，系统如何处理重复项？

答：系统以 URL 的标准化形式（去除末尾斜杠、统一为小写 scheme）作为唯一键。导入过程中若发现重复，默认跳过并记录警告日志，您可通过命令行参数 --dedup=overwrite 强制更新已有记录的元数据。建议在批量导入前使用 export 命令备份现有数据。

问：如何迁移数据至生产环境使用的 MySQL 或 PostgreSQL？

答：项目默认使用 SQLite 作为快速启动的存储方案。如需迁移至外部关系型数据库，请参考 docs/operations/database-migration.md 中的详细步骤。我们提供了通用的 SQLAlchemy ORM 适配层，您只需修改 config.yaml 中的连接字符串并执行 migrate 命令即可完成表结构同步与数据转移。
