# ResourceBridge

ResourceBridge 是一个面向开发者、技术研究人员与内容创作者的轻量级外链资源归集与导航工具。该项目旨在解决个人或团队在浏览、收集、整理大量分散外部链接时缺乏统一入口与持久化记录手段的问题，通过结构化的数据组织方式与简洁的本地运行环境，帮助用户建立属于自己的可维护资源索引。

ResourceBridge 定位于小型项目与技术资源导航站点，适用于需要快速启动本地链接收藏服务、维护在线资源列表、或构建轻量级知识库索引的场景。项目本身不依赖数据库，基于静态数据与模板渲染，可便捷部署至主流托管服务。

## 功能概览

- **链接分类管理**：支持按主题、领域、用途将收录的 URL 划分至不同类别，便于按需浏览与维护。
- **批量导入与校验**：提供批量链接添加接口，自动校验 URL 格式并过滤重复条目，降低人工整理成本。
- **标签与全文检索**：为每一条资源赋予可自定义的标签组合，并支持基于标题、描述、标签的模糊搜索。
- **纯静态输出模式**：支持生成完全静态的 HTML 页面，无需后端服务即可运行，适配各类托管平台。
- **元数据扩展支持**：每条资源可额外记录作者、发布日期、所属组织、存档状态等自定义元数据字段。
- **访问统计埋点接口**：提供可选的轻量级访问计数钩子，便于了解高频资源的利用情况。
- **导入导出互通**：支持 JSON 与 CSV 格式的完整数据导入与导出，方便与其他工具进行数据交换。

## 应用场景

1. **个人技术书签库**：开发者可将日常浏览中发现的优质技术文章、API 文档、开源项目地址统一收录至 ResourceBridge，通过分类与搜索快速回溯，避免浏览器书签的杂乱无序。
2. **团队知识共享入口**：小型开发团队可利用 ResourceBridge 搭建内部常用工具链、设计规范、部署文档的导航页面，新成员入职时可快速了解团队所依赖的外部资源体系。
3. **开源项目外链汇总页**：开源项目维护者可将项目依赖的参考材料、社区扩展、相关讨论帖等外链通过 ResourceBridge 集中呈现，方便用户一站式获取周边信息。
4. **课程或技术专栏参考资料**：技术讲师或专栏作者可为每一期内容建立对应的链接集合，学员通过统一入口访问拓展阅读材料，降低信息分散带来的学习门槛。

## 快速开始

以下命令可在本地快速启动 ResourceBridge 服务。

```bash
# 克隆项目仓库至本地
git clone https://github.com/resourcebridge/resourcebridge.git

# 进入项目目录并安装依赖
cd resourcebridge
pip install -r requirements.txt

# 运行本地开发服务器，默认监听 8000 端口
python app.py --port 8000
```

启动后，在浏览器中访问 `http://localhost:8000` 即可进入资源管理界面。首次运行将自动生成示例数据文件 `data/resources.json`，用户可基于该文件结构进行增删改。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.8 及以上 | 核心运行环境，用于提供后端服务及数据处理逻辑 |
| pip | 21.0 及以上 | Python 包管理工具，用于安装项目依赖项 |
| Flask | 2.2.x | Web 服务框架，负责路由、请求处理及模板渲染 |
| Jinja2 | 3.1.x | 模板引擎，用于生成动态页面内容，支持静态输出模式 |
| requests | 2.28.x | 用于链接可用性校验与元数据自动抓取（可选功能） |
| pytest | 7.2.x | 开发测试依赖，仅在运行单元测试时需要（非生产环境必需） |

## 项目结构

项目目录采用模块化分层设计，核心逻辑与用户数据、页面模板、静态资源相互隔离，便于二次开发与部署调整。

├── config.py                      # 全局配置项，包含端口、数据路径、缓存开关等
├── data/                          # 数据存储目录（用户数据与系统索引）
│   ├── resources.json             # 主资源列表，包含所有 URL 及元数据

## 贡献指南

ResourceBridge 欢迎社区提交问题反馈、功能建议与代码贡献。为保证协作效率，请遵循以下步骤：

1. 查阅 Issue 列表与项目看板，确认当前待处理任务或尚未覆盖的使用场景。对于新功能提议，请先创建 Issue 描述需求背景与预期行为，获得维护者反馈后再着手开发。
2. 派生项目仓库至个人账户，在派生副本中创建以 `feature/` 或 `fix/` 为前缀的分支，并确保代码变更仅涉及相关模块，避免无关文件改动。
3. 所有代码提交需遵循约定式提交规范，提交信息应清晰说明变更类型、作用范围与简要描述。在推送前请在本地执行 `pytest` 确保全部测试用例通过，并为新增功能补充对应测试。
4. 提交 Pull Request 至主仓库的 `main` 分支，PR 描述中需引用关联 Issue 编号，并附上变更摘要与测试结果截图（如涉及界面改动）。维护者将在三个工作日内进行审核。
5. 对于文档类贡献（如修正错别字、补充示例、完善翻译），可直接提交 PR 并在标题中标注 `[docs]` 前缀，无需创建 Issue。

## 常见问题

**Q：ResourceBridge 是否必须部署为 Web 服务？能否仅生成静态页面用于托管？**  
A：ResourceBridge 支持两种运行模式。默认模式为轻量级 Web 服务，提供管理界面与实时数据操作能力；同时，用户可通过 `python app.py --export-static` 命令将当前所有资源数据渲染为完整的静态 HTML 页面，直接放置于 Nginx、GitHub Pages 或对象存储中访问，无需任何后端服务。

**Q：如何确保收录的 URL 长期有效？项目是否提供链接可用性检测？**  
A：ResourceBridge 内置了一个可选的链接校验模块，可通过 `python app.py --check-links` 触发，对当前资源列表中的所有 URL 发起 HEAD 请求并记录响应状态码。校验结果将生成独立报告文件 `data/broken_links.json`。该功能需网络环境支持，且校验频率建议由用户根据自身需求通过定时任务调度。

**Q：数据文件 `resources.json` 是否可以手动编辑？是否存在格式风险？**  
A：用户可以手动编辑 `resources.json`，但需严格遵循 JSON 格式规范，并确保每条记录包含 `id`、`url`、`title`、`category` 四个必需字段。项目在启动时会自动对数据文件进行格式校验，若检测到格式错误或缺失字段，将在控制台输出错误提示并拒绝加载该条目，同时保留原始文件备份于 `data/backup/` 目录下。建议使用项目提供的管理界面进行数据操作以避免格式问题。
