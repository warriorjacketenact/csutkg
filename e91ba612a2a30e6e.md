# LinkVault — 技术资源外链聚合与规范化管理平台

LinkVault 是一个面向技术团队、内容创作者与开源项目维护者的外链资源聚合与规范化管理工具。本项目定位于解决分散在各类技术文章、博客、文档中的外部链接难以统一归档、版本追踪与可用性监控的问题，帮助用户建立可维护、可追溯、可共享的外链资源库。

项目目标用户包括：需要长期维护大量外链引用的开源项目文档维护者、技术博客作者、在线教育平台课程编辑团队，以及企业内部知识库管理员。LinkVault 通过结构化的数据存储、自动化链接状态检测与灵活的导出机制，降低外链管理的人力成本，提升资源引用的长期可靠性。

## 功能概览

**链接批量导入与去重**：支持从 CSV、JSON 及 Markdown 表格中批量导入外链，自动识别并合并重复条目，保留首次导入时间与来源标记。

**自动化可用性检测**：周期性对已收录链接发起 HEAD 与 GET 请求，检测 HTTP 状态码变化、响应时间波动及页面标题变更，生成可用性报告。

**分类标签与全文检索**：允许为每个链接添加多级标签与自定义备注，基于 SQLite 全文索引实现毫秒级检索，支持按域名、状态码、更新时间筛选。

**变更历史追踪**：记录每个链接的属性修改记录，包括 URL 变更、标签调整、备注更新，支持回滚至任意历史版本。

**数据导出与集成**：支持将链接列表导出为 Markdown 列表、JSON 结构数据或 HTML 目录页，便于嵌入现有文档体系或静态站点生成器。

**访问统计与热度分析**：统计每个链接的点击次数、最后访问时间与来源页面，生成访问热度趋势图，辅助识别高价值资源。

**多用户协作文档**：基于角色权限控制，支持团队内成员共同编辑链接备注、审核新提交链接、分配维护责任人。

**Webhook 事件通知**：当链接状态变更（如变为 404、域名过期、内容重大更新）时，通过 Webhook 向飞书、钉钉或 Slack 推送告警消息。

## 应用场景

**开源项目文档的外链维护**：开源项目的 README、Wiki 或用户指南中常引用大量外部资源链接。随着时间推移，部分链接可能失效或内容迁移。LinkVault 可周期性检测这些链接的可访问性，并自动标记异常，帮助维护者及时发现并更新文档中的引用。

**技术博客的参考资料归档**：技术博客作者在撰写文章时需引用多篇论文、官方文档或社区讨论。使用 LinkVault 管理这些参考资料链接，可在文章发布后持续监控链接存活状态，当链接失效时快速定位并替换为存档版本或替代资源。

**企业内部知识库的资源治理**：大型企业的内部技术知识库通常包含数千条外部链接，涉及供应商文档、行业标准、开源组件仓库等。LinkVault 提供统一的链接登记与审核流程，确保所有外链经过合规审查，并定期生成链接健康度报表供知识库管理员审阅。

**在线课程平台的资源更新**：在线技术课程中的练习环境、示例代码仓库、延伸阅读链接需要随外部变化同步更新。LinkVault 的变更追踪功能可记录每次链接调整的原因与时间，方便课程维护团队追溯修改历史，确保不同期次的学员看到的内容一致且有效。

## 快速开始

以下命令演示了如何从 GitHub 克隆 LinkVault 源码、安装依赖并启动开发服务器。

```bash
git clone https://github.com/linkvault/linkvault.git
cd linkvault
pip install -r requirements.txt
python manage.py migrate
python manage.py runserver
```

生产环境部署请参考 `deploy/production.md` 文档，其中包含使用 Gunicorn + Nginx + PostgreSQL 的完整配置示例。

## 安装要求

| 依赖 | 必需版本 | 说明 |
|------|----------|------|
| Python | 3.10 及以上 | 核心运行环境，低于此版本将无法解析类型注解与异步语法 |
| PostgreSQL | 14.0 及以上 | 主数据库，用于存储链接元数据、用户信息与审计日志 |
| Redis | 7.0 及以上 | 缓存与任务队列后端，用于异步执行链接状态检测任务 |
| Node.js | 18.0 及以上 | 仅用于前端资源构建，生产环境中若使用预构建静态文件则可选 |
| SQLite | 3.35 及以上 | 用于本地开发与单元测试，生产环境建议切换至 PostgreSQL |
| Docker | 20.10 及以上 | 可选，用于容器化部署，提供一键启动开发环境的 Compose 文件 |

## 贡献指南

1. 阅读开发者指南文档（`docs/developer/` 目录），了解项目整体架构、代码风格规范（PEP 8 + Black）与 Git 提交信息格式（Conventional Commits）。所有新增功能需附带相应的单元测试与 API 文档更新。

2. 在 GitHub Issues 中查找标记为 `good-first-issue` 或 `help-wanted` 的任务，或提交新的 Issue 描述你发现的问题或建议的新特性。重大变更请先通过 Issue 与维护者讨论设计方案，避免无效开发。

3. Fork 本仓库并创建新的功能分支，分支命名格式为 `feature/功能简述` 或 `fix/问题简述`。开发过程中请保持本地主分支与上游同步，定期 rebase 以减少合并冲突。

4. 提交代码前运行本地测试套件（`pytest tests/`）与 lint 检查（`ruff check .`），确保所有测试通过且无新增 lint 警告。提交信息请使用英文，首行不超过 72 字符，详细说明变更动机与影响范围。

5. 发起 Pull Request 至主仓库的 `main` 分支，PR 描述中需关联相关 Issue 编号，并勾选自检清单（测试覆盖、文档更新、变更日志条目）。PR 通过 CI 检查后将由维护者进行 Code Review，必要时请配合修改直至合并。

## 常见问题

Q: 检测引擎是否支持需要登录或带 Cookie 的受限链接？

A: 当前版本仅支持公开可访问的链接检测。对于需登录的页面，可在链接备注中标记为"受限资源"，系统将跳过自动检测并记录手动检查提醒。我们计划在 v2.0 中引入可配置的请求头注入机制，以支持 Bearer Token 或 Basic Auth 认证。

Q: 批量导入时如何处理已存在相同 URL 的数据？

A: 系统以 URL 字符串作为唯一键进行去重。导入时若检测到相同 URL，默认执行"跳过"操作，保留原有记录的所有字段。您也可以在导入配置中选择"覆盖"模式，将旧记录的标题、标签、备注更新为新导入文件中的值，同时保留创建时间与历史变更记录。

Q: 生产环境中检测任务对目标服务器造成过大压力怎么办？

A: 检测任务默认并发数为 10，间隔时间为 24 小时。您可以通过环境变量 `CHECKER_CONCURRENCY` 和 `CHECKER_INTERVAL_HOURS` 调整并发度与执行频率。此外，支持配置 `CHECKER_WHITELIST_DOMAINS` 白名单，仅对特定域名进行高频检测，其余域名降低检测频率或仅检测响应状态码不下载完整页面内容。
