# Fooklam Resource Aggregator

Fooklam Resource Aggregator 是一个面向技术研究人员、数据挖掘工程师和内容聚合平台运维人员的高性能外链资源汇总系统。该项目定位于将分散于各类技术博客、官方文档、社区讨论和开源仓库中的高质量外部链接进行统一采集、分类存储、状态监控和快速检索，帮助团队和个人在信息过载的环境中高效定位有价值的技术资料。

本项目不生产内容，而是通过结构化方式组织已有网络资源，提供标准化的访问入口、可用性检测和标签化过滤能力。适用于需要持续跟踪多个技术领域动态、维护团队知识库或构建垂直领域导航站的场景。第 426/1241 批资源集包含 250 个经过初步筛选的外部链接，全部收录于本系统的核心索引中。

## 功能概览

**统一外链入库与去重**：基于 URL 指纹和域名规则自动过滤重复提交，确保同一资源仅保留唯一入口，避免索引膨胀。

**多维度标签分类**：每个链接可标记技术栈、内容类型、更新频率、语言等自定义标签，支持组合筛选和布尔查询。

**可用性健康检查**：周期性对已收录链接发起 HEAD 请求，记录响应状态码、响应时间与重定向链，自动标记不可用资源。

**批量导入导出接口**：支持 CSV、JSON Lines 格式的批量链接导入，也提供按条件筛选后的数据导出功能，便于迁移或二次处理。

**全文元数据检索**：基于标题、描述、标签和来源域名的轻量级全文搜索，支持模糊匹配和前缀搜索。

**访问统计与热度排序**：记录每个链接的被查询次数和点击通过量，可按热度、新增时间或字母序排列输出。

**权限与多用户协作**：内置基于角色的访问控制，区分管理员、编辑员和只读观察员，适合团队共同维护资源库。

**RESTful API 全量暴露**：所有核心能力均通过 JSON API 开放，便于集成至现有运维看板或自动化流水线。

## 应用场景

技术团队内部知识库的链接底座。团队可以将日常阅读中发现的优质官方文档、调试案例和性能分析文章统一收录至 Fooklam Resource Aggregator，并利用标签按项目或技术领域归类，新成员入职时即可通过标签组合快速获取学习路径，减少信息检索耗时。

垂直领域导航站的后端索引引擎。开源社区或技术媒体可以利用本项目的分类能力和健康检查接口，构建面向特定技术栈的导航页面，例如 Rust 生态工具链导航或云原生可观测性工具集合，前端只需调用 API 渲染分类目录即可，无需关心链接的可用性维护。

数据挖掘样本源的长期存储与更新。进行技术趋势分析或学术研究时，需要固定一批稳定的网络来源作为样本集。本项目提供批量导入和定期健康检查功能，研究者可一次性导入上千个候选源，定期导出存活列表用于后续爬取或统计分析，确保样本集的持续可访问性。

自动化运维巡检的告警输入源。运维团队可将基础监控文档、SRE 最佳实践和故障排查手册链接集中管理，结合可用性检查结果，当关键文档链接失效时通过 Webhook 触发告警，避免依赖链断裂。

## 快速开始

以下命令演示如何从 GitHub 克隆项目、安装依赖并启动开发服务器。确保已安装 Git、Node.js 18.x 及以上版本和 pnpm 8.x。

```bash
git clone https://github.com/fooklam/resource-aggregator.git
cd resource-aggregator
pnpm install
pnpm run init:db
pnpm run dev
```

生产环境部署请参考 docs/deployment.md，使用 Docker Compose 一键启动全套服务（包含 PostgreSQL、Redis 和 Nginx 代理）。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | 18.x 或 20.x LTS | 运行时环境，建议使用最新的 LTS 版本以获得长期支持 |
| pnpm | 8.x 及以上 | 包管理器，用于安装依赖和管理 monorepo 工作区 |
| PostgreSQL | 14.x 及以上 | 主数据库，存储链接元数据、标签、用户信息及访问日志 |

| Docker Compose | 2.20 及以上 | 可选但推荐，用于快速拉起全部基础服务依赖 |
| 系统内存 | 至少 2 GB 可用 | 开发环境建议 4 GB，生产环境根据并发量调整 |

## 项目结构

```
fooklam-resource-aggregator/
├── apps/
│   ├── api/                           # RESTful API 服务 (Fastify)
│   │   ├── src/
│   │   │   ├── routes/                # 按资源划分的路由定义
│   │   │   ├── controllers/           # 请求处理与响应格式化
│   │   │   ├── services/              # 核心业务逻辑：入库、查询、健康检查
│   │   │   ├── models/                # PostgreSQL 数据模型 (Prisma ORM)
│   │   │   ├── middlewares/           # 认证、限流、日志中间件
│   │   │   └── utils/                 # 通用工具函数（URL 归一化、指纹计算）
│   │   └── tests/                     # 单元测试与集成测试
│   ├── web/                           # 管理控制台前端 (React + Vite)
│   │   ├── src/
│   │   │   ├── pages/                 # 仪表盘、链接列表、标签管理、系统设置
│   │   │   ├── components/            # 可复用 UI 组件（表格、筛选器、图表）
│   │   │   ├── hooks/                 # 自定义 React Hooks（API 调用、分页）
│   │   │   └── stores/                # Zustand 状态管理
```

## 贡献指南

我们欢迎各类贡献，包括但不限于新增链接源、改进健康检查策略、优化查询性能和修复文档错误。请遵循以下流程：

1. 查阅 issue 列表，确认是否有正在进行的相关讨论。若无，请新建一个 issue 描述你希望解决的问题或新增的功能，等待核心维护者反馈。

2. 从 main 分支拉出新的功能分支，分支命名遵循 feat/xxx、fix/xxx 或 docs/xxx 格式。确保本地开发环境通过 pnpm install 和 pnpm run test 可完整运行。

3. 提交代码时遵循 Conventional Commits 规范，每个 commit 应包含清晰的作用域和描述。提交前运行 pnpm run lint 和 pnpm run format 保证代码风格统一。

4. 创建 Pull Request 并填写标准模板，说明变更动机、实现方式和测试覆盖情况。PR 至少需要一位维护者 approve 后方可合并。

5. 若涉及新增外部链接源，请确保链接内容符合项目收录标准（技术相关、内容稳定、非广告落地页），并在 PR 中附带链接的简要描述和推荐标签。

## 常见问题

**问：系统如何处理已失效的外部链接？**

健康检查服务会以可配置的周期（默认每 24 小时）对所有已收录链接发起 HEAD 请求。连续三次检查均返回非 2xx 状态码或超时的链接将被标记为不可用。不可用链接不会在 API 的默认查询结果中返回，但管理员可通过显式参数查询并决定是否删除或替换。系统保留历史状态变更日志，便于追溯可用性波动。

**问：是否可以导入自定义标签体系？**

可以。项目支持通过管理后台或 API 批量创建标签，每个标签包含键值对形式（如 language:rust、source:official）。导入外部数据时，可在 CSV 或 JSON 文件中为每个条目预先指定标签数组，系统会在入库时自动匹配已有标签或新建未识别的标签。所有标签名称区分大小写，建议统一使用小写加连字符的命名约定。

**问：性能方面，单实例能支撑多少链接的定期检查？**

在默认配置下（4 核 CPU、8 GB 内存），单 API 实例搭配单 Redis 和 PostgreSQL 可稳定管理约 5 万条链接的元数据存储和每日健康检查。健康检查采用并发控制，同时最多发起 50 个请求，超时设置为 10 秒。对于超过 10 万链接的规模，建议启用 worker 多实例部署，并调整 Redis 队列的并发消费数。
