# ResourceBridge 技术外链索引系统

ResourceBridge 是一个面向技术研究者、开源贡献者和知识工作者的高密度外链资源汇总与导航系统。项目定位于解决信息分散、优质技术内容难以系统化检索的问题，通过结构化分类与持续维护，将散落于网络各处的深度技术文章、系列教程与领域专题整合为可查询、可回溯的知识索引库。

本项目并不直接生产原创技术内容，而是以外部资源为收录对象，通过严格的筛选标准和分类体系，为特定技术方向提供一站式的入口导航。目标用户包括需要快速建立某领域知识地图的初学者、需要横向对比多种技术方案的系统架构师、以及希望追踪特定技术序列演进路径的研究人员。当前批次（第 19/1241 批）收录了 250 个来自 wei-ke.net 的技术系列资源，涵盖系统设计、编程语言、运维实践、算法与数据结构等多个子领域。

## 功能概览

**系列化资源组织** 将零散的独立文章按主题归入技术系列，每个系列包含完整的多篇连载内容，形成从入门到深入的知识路径。

**外链纯净收录** 严格遵循原始 URL 收录原则，不添加跳转追踪参数，不修改协议与域名，保证链接的可访问性与原始指向。

**多维度分类索引** 每个资源条目按所属子领域、技术栈、难度等级和适用场景进行标记，支持多条件组合筛选。

**批量批次管理** 采用批次导入机制，每批资源均有独立编号与管理记录，便于追溯新增内容与更新历史。

**只读镜像架构** 项目本身不存储任何外部内容副本，仅维护链接元数据与分类信息，从根本上规避版权与内容过期问题。

**命令行检索接口** 提供轻量级 CLI 工具，支持关键词搜索、系列列表查看和随机资源抽取等操作。

**元数据校验机制** 每次启动时自动校验已收录链接的 HTTP 状态，标记异常链接并生成报告。

**静态站点生成支持** 内置模板引擎，可将资源索引导出为纯静态 HTML 文档，方便部署为内部知识门户。

## 应用场景

技术团队内部知识库建设。团队技术负责人可使用 ResourceBridge 将团队长期积累的外部参考链接统一纳入索引体系，配合项目结构中的分类规则，使新人入职后能按图索骥地学习团队所依赖的技术栈与设计范式。

个人技术阅读的队列管理。独立开发者或工程师可将日常发现的优质系列文章通过提交 PR 的方式汇入资源池，后续通过 CLI 检索接口随机抽取或按主题浏览，避免收藏夹随时间膨胀而彻底失效。

技术社区的内容推荐辅助。技术博客运营者或社区管理员可基于 ResourceBridge 的系列分类快速定位某一方向的热门或深度内容，作为选题策划或问答引用的外部依据。

离线文档与镜像站点的前置筛选。在需要搭建内部技术镜像的环境下，ResourceBridge 提供的链接清单可直接作为爬虫采集的种子列表，其分类标签同时可用于定义采集深度与更新频率策略。

## 快速开始

以下操作以 Linux / macOS 环境为例，Windows 用户可使用 Git Bash 或 WSL2 执行。

```bash
# 克隆项目仓库
git clone https://github.com/resourcebridge/resourcebridge.git
cd resourcebridge

# 安装依赖（Python 3.10+ 与 uv 包管理器）
curl -LsSf https://astral.sh/uv/install.sh | sh
uv venv
source .venv/bin/activate
uv pip install -r requirements.txt

# 初始化本地索引并导入当前批次资源
python -m rbctl import --batch 19 --source data/batch_19_manifest.json
python -m rbctl index rebuild
python -m rbctl serve --port 8080
```

执行完毕后，访问 http://localhost:8080 可查看当前批次的资源导航页面。如需使用 CLI 检索，可运行 `python -m rbctl search --keyword "系统设计"` 进行测试。

## 安装要求

| 依赖 | 必需 | 说明 |
|---|---|---|
| Python 3.10 或更高版本 | 是 | 核心运行环境，低于此版本将无法解析类型注解与异步语法 |
| uv 0.4.0 或更高版本 | 是 | 用于虚拟环境管理与依赖锁定，替代传统 pip + venv 组合 |
| Git 2.30 或更高版本 | 是 | 用于克隆仓库和后续拉取上游更新 |
| SQLite 3.35 或更高版本 | 是 | 内置索引存储引擎，支持 JSON 函数以处理元数据字段 |
| curl / wget | 否 | 仅在校验外部链接可用性时需要，若缺失将跳过网络检测步骤 |
| make 4.0 或更高版本 | 否 | 用于执行 Makefile 中定义的自动化任务（如批量导入与导出） |
| Node.js 18.x | 否 | 仅在启用静态站点生成功能时需要，用于渲染前端模板 |
| 网络连接（出站 443/80） | 是 | 运行时需访问外部资源以完成链接校验，内网环境需配置代理 |

## 贡献指南

**提交新资源批次**。通过 GitHub Issue 提交新批次申请，需附上 manifest JSON 文件，其中包含每条链接的原始 URL、建议分类及简短摘要。项目维护者将在五个工作日内完成审核与合并。

**修正元数据错误**。若发现某条资源的分类标签、系列名称或摘要信息有误，请提交 Pull Request 直接修改 `data/manifests/` 下对应批次的 JSON 文件，并在 PR 描述中注明修改依据。

**开发新功能模块**。在开始较大规模的功能开发前，建议先通过 Issue 与维护者沟通设计思路，避免重复劳动。开发完成后需确保新增代码通过所有单元测试，并提供必要的文档更新。

**完善项目文档**。文档缺失或表述不清之处欢迎随时提交 PR。文档风格要求技术化、精确，避免模糊用语。涉及命令行操作的改动需同步更新对应的 help text。

**报告链接失效**。使用 `rbctl validate --batch 19` 可生成本批次链接状态报告，如发现大量失效，可通过 Issue 上传报告文件，维护者将定期批量处理失效链接的标记与替换。

## 常见问题

**Q: 收录的链接如果完全无法访问，项目会如何处理？**

A: 项目本身不托管外部内容，因此无法修复源站故障。但项目提供了定期校验机制，每次运行 `rbctl validate` 时会生成详细的状态报告。对于连续三次校验均返回非 2xx 状态码的链接，将被标记为 `deprecated` 并在导航页面中置灰显示。用户可通过提交 Issue 的方式报告失效链接，维护者将视情况联系源站或从索引中移除。

**Q: 为什么资源列表中的 URL 不带 https:// 前缀，有些却带了？**

A: 本项目坚持「原始收录」原则，所有链接均按照用户提交时的原始形式原样存储与展示，不进行任何协议补全、域名规范化或大小写转换。这样做是为了保证链接的指向与提交者预期完全一致，避免因自动改写引入重定向或访问错误。使用时应直接复制列表中的完整 URL 字符串进行访问。

**Q: 如何将本项目部署为团队内部的服务？**

A: 推荐使用 `rbctl serve --port 8080` 启动内置 Web 服务，配合 systemd 或 supervisor 实现进程守护。若需更高并发，可使用 `rbctl export static --output /var/www/html` 生成纯静态 HTML 文件，然后交由 Nginx 或 Caddy 托管。静态导出模式无需数据库连接且性能最优，适合内网大规模分发场景。
