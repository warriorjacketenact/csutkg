# ResourceBridge 索引聚合系统

ResourceBridge 是一个面向技术文档工程师、开源项目维护者以及知识库管理者的外部资源索引与元数据聚合工具。该项目不直接托管任何第三方内容，而是通过结构化方式将分散于不同域名下的信息条目进行统一收录、分类标注与可检索化呈现，帮助团队在文档撰写、技术选型或竞品分析过程中快速定位高价值参考链接。

项目定位为“外链资源的轻量级中台”，适用于需要长期维护大量外部引用链接的任何规模的技术内容项目。ResourceBridge 提供标准化的条目导入接口、基于标签的快速筛选能力以及完整的资源生命周期状态跟踪，确保从收录到归档的每一个环节均可追溯、可审计。

## 功能概览

**批量条目导入** 支持通过 CSV、JSON 或直接 Markdown 列表方式将大量 URL 连同其元数据（标题、摘要、所属类别、收录批次）一次性录入系统。

**自动元数据补全** 对已收录的每个资源条目自动发起可配置的元数据抓取请求，提取页面标题、描述文本以及最后修改时间，减少人工填写负担。

**多级分类标记** 每个资源条目可同时归属于多个分类树节点，支持按项目维度、技术领域维度、文档类型维度进行交叉索引，满足不同角色的查找习惯。

**状态流转管理** 为每个资源条目定义“待审核”、“已发布”、“已过时”、“已归档”四种状态，支持状态变更历史记录与过期提醒。

**全文检索与过滤** 基于条目编号、原始 URL 关键字、分类标签、状态以及收录时间范围进行组合检索，结果按相关度或时间排序输出。

**批次追踪与统计** 每一批导入的资源自动绑定批次号（如第 743/1241 批），系统记录该批次的导入时间、条目数量、失败条目数以及当前有效条目占比。

## 应用场景

技术文档团队在撰写产品新版本文档时，需要引用大量外部技术标准、参考实现和社区讨论帖。ResourceBridge 可帮助团队成员提前将所有待引用链接录入系统，分配审核任务，确保最终发布文档中的每一个外链均经过可用性验证。

开源项目维护者在处理 Issue 或 Pull Request 时，常需要贴出相关技术讨论、修复方案或上游依赖的变更记录。通过 ResourceBridge 预先维护常用参考链接库，维护者可以在评论中快速插入标准格式的引用，提升协作效率。

知识库管理员对内部 Wiki 或 Confluence 空间进行定期清理时，可利用 ResourceBridge 的过期状态自动标记功能，批量识别长期未访问或返回 HTTP 错误状态码的外部链接，生成待处理报告。

## 快速开始

以下步骤帮助你在本地环境中启动 ResourceBridge 开发实例并进行基础功能验证。

```bash
# 克隆代码仓库至本地
git clone https://github.com/resourcebridge/resourcebridge.git
cd resourcebridge

# 安装项目依赖（使用 pip 与虚拟环境）
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 执行数据库迁移并启动开发服务器
python manage.py migrate
python manage.py runserver --host 0.0.0.0 --port 8080
```

## 安装要求

ResourceBridge 依赖以下运行环境与第三方库，部署前请确保所有必需组件均已就绪。

| 依赖 | 必需 | 说明 |
|------|------|------|
| Python | 3.9 或更高版本 | 核心运行环境，低于此版本将无法解析类型注解与异步语法 |
| PostgreSQL | 14.0 或更高版本 | 主数据库引擎，用于存储条目元数据、分类与状态历史 |
| Redis | 6.0 或更高版本 | 缓存与任务队列后端，用于元数据补全任务的异步调度 |
| Celery | 5.2 或更高版本 | 分布式任务队列实现，处理外部资源抓取与状态探测 |
| BeautifulSoup4 | 4.11 或更高版本 | HTML 解析库，用于从资源页面提取标题与描述信息 |
| Requests | 2.28 或更高版本 | HTTP 客户端，用于执行所有对外部资源的探测请求 |
| Pytest | 7.0 或更高版本 | 单元测试框架，仅在开发环境中用于运行测试套件 |
| Docker | 20.10 或更高版本 | 容器化部署选项，生产环境推荐使用 Docker Compose 编排 |

## 贡献指南

ResourceBridge 遵循开源社区协作规范，所有贡献均通过 GitHub Pull Request 流程提交。请确保在发起较大变更前先创建 Issue 进行讨论，避免重复劳动或方向偏离。

1. 派生项目仓库至个人账户，并将派生仓库克隆至本地开发环境。配置上游远程分支以便同步主仓库的最新变更。

2. 为新功能或缺陷修复创建独立的主题分支，分支名称遵循 `feature/` 或 `fix/` 前缀加简要描述的形式。在提交代码前运行全部单元测试与代码规范检查。

3. 提交信息采用约定式提交格式，包含 `type(scope): subject` 头部，正文详细描述变更原因与影响范围。每个 Pull Request 应仅解决一个独立问题或实现一个完整功能。

4. 发起 Pull Request 时填写标准模板中的所有检查项，包括测试覆盖率说明、文档更新情况以及是否向后兼容。至少一位核心维护者审阅通过后方可合并。

5. 文档类贡献（包括 README 更新、示例代码补充、错别字修正）无需创建 Issue，可直接以 Pull Request 提交，但需在描述中明确标注 `[docs]` 前缀。

## 常见问题

问：导入大批量链接时，系统是否会立即对每个链接发起请求进行元数据补全？
答：不会。导入流程仅将链接及其基础元数据写入数据库，并将每个链接的元数据补全任务推入 Celery 队列。补全任务由后台 worker 按可配置的并发度与速率限制逐步执行，避免对目标服务器造成过大压力。任务执行进度可在管理控制台的批次详情页面查看。

问：如果某个外部资源链接失效或返回 404 状态码，系统如何处理？
答：在每次元数据补全或定期状态探测任务中，系统会记录 HTTP 响应状态码。连续三次探测失败（间隔为 24 小时）的条目，其状态将自动转为“已过时”。管理员可配置 Webhook 在状态变更时接收通知，也可手动将条目重新置为“待审核”并触发重新探测。

问：ResourceBridge 是否支持多用户角色与权限控制？
答：当前版本提供基础的基于 API Token 的认证机制，支持管理员与只读访客两种角色。管理员可执行导入、删除、状态变更等写操作；只读访客仅可进行检索与导出。未来版本计划引入基于团队的分类访问控制策略。
