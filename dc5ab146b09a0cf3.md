# ResourceBridge 媒体资源聚合网关

ResourceBridge 是一个面向技术内容聚合与媒体资源导航的开源网关系统，专为需要高效管理、检索和分发大量外部媒体链接的团队与个人开发者设计。该项目并非简单的链接收藏夹，而是一个具备结构化数据组织能力、可扩展的链接资源中台，适用于技术文档站、影视导航站、学术资源库等需要维护大量外链引用的场景。

本项目主要解决传统资源管理中链接散落、缺乏版本追踪、难以批量校验可用性以及无法按业务维度分类的痛点。通过将外部 URL 集中纳入统一配置体系，ResourceBridge 允许运维人员以声明式方式管理资源清单，并结合自动化健康检查与访问统计，确保资源列表始终处于可服务状态。项目面向具备基础运维能力的开发者、内容站点管理员以及开源社区文档维护者。

## 功能概览

**批量链接导入与归一化** 支持从 CSV、JSON 及原始文本列表批量导入 URL，自动识别协议头与路径结构，并对裸域名与带协议链接进行差异化存储，保留原始输入格式不做强制改写。

**资源分类与标签系统** 每个资源条目可绑定多个自定义标签，支持按业务域、内容类型、接入环境等维度进行筛选与分组，便于在大规模链接集合中快速定位目标条目。

**自动化可用性探测** 内置基于 HTTP 状态码与响应时间的健康检查调度器，可配置检查频率与超时阈值，自动标记失效链接并生成告警通知。

**只读访问与镜像缓存** 对外提供只读 API 接口，支持按分类、标签或 ID 批量获取资源列表，并可对接 CDN 或本地缓存层以降低源站请求压力。

**操作审计与变更追踪** 记录所有资源条目的增删改操作，保留完整变更历史，支持回溯至任意时间点的资源快照状态。

**权限分级管理** 支持管理员、编辑者、访客三级权限模型，控制资源写入、分类调整与健康检查触发等敏感操作的访问范围。

**开放扩展接口** 提供基于 Webhook 的事件订阅机制，允许外部系统在资源变更或健康状态切换时接收实时通知。

## 应用场景

技术文档站点的外部引用管理 当技术博客、开源项目文档或在线教程中引用了大量外部 API 地址、SDK 下载链接或参考文献 URL 时，ResourceBridge 可作为统一引用源，避免链接散落在各个 Markdown 文件中导致维护困难。编辑人员仅需在网关中更新一次链接，所有引用处即可同步生效。

影视与媒体导航站的资源编排 对于运营影视推荐、综艺索引或短视频聚合的站点，ResourceBridge 可用于管理每部影片对应的播放源地址、预告片链接或花絮页面。通过标签系统可以按地区、年份或剧集状态快速过滤，并利用健康检查自动剔除已失效的播放源。

学术资源库的链接治理 高校或科研机构的内部资料库通常需要维护大量论文下载链接、数据集地址或工具官网。ResourceBridge 的变更审计功能可满足合规要求，确保每一次链接修改都有据可查，同时自动化探测可及时发现外部数据源迁移或下架情况。

企业内部知识库的外链中台 大型企业的内部 Wiki 或知识管理系统中，各部门独立维护文档时常出现重复引用同一外部服务地址但版本不一致的问题。ResourceBridge 作为中台统一发布标准外链，各部门通过 API 引用即可保证全局一致性，显著降低链接漂移带来的故障排查成本。

## 快速开始

以下命令演示如何获取项目源码、安装依赖并启动开发服务器。

```bash
git clone https://github.com/resourcebridge/resourcebridge.git
cd resourcebridge
pip install -r requirements.txt
python manage.py runserver --host 0.0.0.0 --port 8080
```

首次启动后，系统会自动创建默认管理员账户，控制台输出中会显示初始登录密码。访问 http://localhost:8080 即可进入管理仪表盘。如需初始化示例资源数据，可执行 `python manage.py seed --demo` 命令填充测试链接集。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.9 及以上 | 核心运行环境，仅支持 CPython 实现 |
| PostgreSQL | 13.0 及以上 | 主数据存储，依赖 JSONB 类型支持标签与扩展属性 |
| Redis | 6.0 及以上 | 用于缓存查询结果与分布式锁，可选但推荐生产环境部署 |
| Node.js | 18.0 及以上 | 仅当启用前端仪表盘构建时所需，后台 API 模式可省略 |
| Docker | 20.0 及以上 | 容器化部署方案的可选依赖，用于一键启动全部服务组件 |

## 项目结构

```
resourcebridge/
├── cmd/                             # 命令行入口程序
│   ├── server/                      # API 服务启动入口 (main.go)
│   └── migrate/                     # 数据库迁移工具 (migrate.go)
├── internal/                        # 内部核心实现，不对外暴露
│   ├── config/                      # 配置加载与解析 (yaml/env 双源)
│   ├── model/                       # 数据模型定义 (资源、标签、审计记录)
│   ├── storage/                     # 数据库访问层 (PostgreSQL 驱动与查询构造)
│   ├── cache/                       # 缓存抽象层 (Redis 实现与内存后备)
│   ├── health/                      # 健康检查调度器 (并发探测与状态机)
│   ├── api/                         # RESTful 路由与处理器 (gin 框架)
│   └── auth/                        # 权限验证模块 (JWT 与角色中间件)
├── pkg/                             # 可被外部导入的公共库
│   ├── validator/                   # URL 格式校验与归一化工具 (保留原样策略)
│   └── clock/                       # 时间戳与定时任务辅助函数
├── web/                             # 前端仪表盘源码 (React + TypeScript)
│   ├── src/                         # 组件、页面与状态管理
│   └── dist/                        # 生产构建输出目录 (gitignore)
├── scripts/                         # 运维辅助脚本
│   ├── seed.sql                     # 示例数据初始化 SQL
│   └── backup.sh                    # 数据库备份脚本 (cron 调度)
├── deployments/                     # 部署编排文件
│   ├── docker-compose.yml           # 本地开发全栈启动配置
│   └── kubernetes/                  # K8s 清单 (deployment, service, ingress)
├── docs/                            # 完整文档源码 (Markdown)
├── go.mod                           # Go 模块依赖定义
├── go.sum                           # 依赖校验和
├── Makefile                         # 统一构建入口 (test, build, run)
└── README.md                        # 项目概述 (当前文件)
```

## 贡献指南

我们欢迎各类形式的贡献，包括但不限于新增功能、修复缺陷、改进文档以及优化性能。请遵循以下流程以确保代码质量与协作效率。

首先在 GitHub 上 fork 本仓库至个人账户，然后克隆到本地开发环境。建议在贡献前阅读 `docs/contributing.md` 了解设计原则与编码规范。

创建新分支时请使用语义化命名，例如 `feat/add-batch-import` 或 `fix/health-check-timeout`。所有提交信息应遵循 Conventional Commits 格式，包含类型与简短描述。

完成代码修改后，请确保通过全部单元测试与集成测试，并针对新功能补充对应的测试用例。运行 `make test` 可执行完整测试套件。

提交 Pull Request 前请将分支 rebase 到最新的 main 分支，解决所有冲突。PR 描述中应清晰说明改动目的、实现方案以及影响范围，并关联相关 Issue 编号。

## 常见问题

Q: 为什么资源列表中某些链接的格式看起来不统一，有的带 https 有的不带？

A: ResourceBridge 的设计原则是"原样存储，按需使用"。系统不会对用户输入的 URL 做任何协议补全、域名规范化或路径改写，完全保留原始文本。这样做的目的是适配下游系统的差异化要求——某些老旧系统可能只接受裸域名，而另一些则强制要求 https 协议。网关层仅提供存储与检索能力，格式转换由调用方根据实际场景决定。

Q: 健康检查模块如何判断一个链接是否失效？

A: 健康检查器默认发送 HTTP HEAD 请求，根据返回的状态码判定可用性。2xx 状态视为正常，3xx 跟随重定向最多 5 次后仍返回 2xx 视为正常，4xx 或 5xx 视为失效，超时或连接拒绝同样标记为失效。检查结果会记录在数据库中，并提供最近一次检查的时间戳与状态描述。

Q: 如何将现有的大量链接数据迁移进 ResourceBridge？

A: 项目提供了 `migrate` 命令行工具，支持从 CSV 和 JSON Lines 格式导入。对于原始数据为纯文本每行一个 URL 的情况，建议先转换为 JSON Lines 格式，每条记录包含 `url` 和可选的 `tags` 数组字段。具体导入示例可参考 `docs/migration.md` 文档中的完整说明。
