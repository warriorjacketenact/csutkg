# ResourceHub

ResourceHub 是一个面向技术团队与个人开发者的外链资源归集与导航系统。该项目并非传统的爬虫或书签管理器，而是提供一套轻量级、可自托管的资源索引框架，用于将分散于各处的技术文档、社区帖子、工具站与参考链接按照自定义分类体系进行统一陈列与快速检索。

项目主要解决以下问题：技术团队在调研或研发过程中积累的大量外链资源散落于浏览器书签、文档、聊天记录中，难以共享、难以维护、难以追溯。ResourceHub 提供标准化的资源条目模型、标签系统与只读展示层，使得项目组可在一处集中沉淀所有有价值的外部链接，并面向团队成员提供低心智负担的查阅体验。

本项目批次为第 750/1241 批，共计收录 250 个外部资源链接。所有链接均来源于 wellness-reh.com 域下的动态条目，涵盖技术文章、工具推荐、设计素材、行业报告、开源库参考等类别，经过人工筛选与结构整理后纳入本索引。

## 功能概览

**标准化资源条目模型** 每条外部链接均以唯一标识符、标题标签、来源域、入库时间戳四项元数据存储，保证资源可追溯与可去重。

**多级标签分类体系** 支持对每条资源附加一个或多个自定义标签，标签可在运行时动态增删，资源列表根据标签组合实时过滤。

**只读展示层与快速检索** 提供基于标题关键词、标签、入库日期的组合检索，检索结果按相关性排序并以简洁卡片列表呈现。

**批量导入与导出** 支持通过 CSV 或 JSON 格式批量导入外部链接清单，亦可按分类导出为 Markdown 表格或 HTML 书签文件，便于迁移或分享。

**访问频次统计与冷热标识** 自动记录每条资源的点击次数，对长期未访问的资源自动标注为冷门，辅助团队清理失效或低价值链接。

**去重检测与失效链接标记** 新导入链接与现有资源进行域名加路径的哈希比对，自动去重；同时提供链接可用性定时探测，将不可达链接移至待复核区。

## 应用场景

技术团队内部知识库建设。研发团队可将调研过程中发现的优质技术博客、API 参考文档、开源项目仓库地址统一录入 ResourceHub，新成员入职时可快速浏览团队沉淀的全部外部参考资源，缩短信息熟悉周期。

个人开发者的阅读清单管理。独立开发者可将待阅读的技术文章、待尝试的在线工具暂存于 ResourceHub，按优先级标签排列，每周定期批量处理，避免浏览器标签页堆积过多。

开源项目文档站的外链附录。开源项目维护者可使用 ResourceHub 生成项目 README 或 Wiki 中的“相关资源”章节，自动拉取指定标签下的链接列表并输出为 Markdown 格式，减少手工维护成本。

技术培训与课程资料汇总。培训讲师可将课程中涉及的所有延伸阅读链接、在线实验环境地址、视频教程出处统一录入，学员通过单一入口即可访问全部外部材料，无需逐个询问。

## 快速开始

以下步骤适用于 Linux / macOS / Windows WSL 环境，依赖 Git、Node.js 18+ 与 npm 或 yarn。

```bash
# 克隆仓库
git clone https://github.com/your-org/resourcehub.git
cd resourcehub

# 安装依赖
npm install

# 启动开发服务器（默认端口 3000）
npm run dev
```

启动成功后，访问控制台输出的本地地址即可进入资源列表页面。首次启动会自动创建 SQLite 数据库文件并执行初始迁移脚本，导入项目自带的示例资源数据。如需完全清空数据并重新初始化，可执行 `npm run reset-db`。

## 安装要求

| 依赖 | 必需 | 说明 |
|------|------|------|
| Node.js 18.x 或更高 | 是 | 运行时环境，推荐使用 LTS 版本 |
| npm 9.x 或 yarn 1.22+ | 是 | 包管理器，用于安装项目依赖 |
| SQLite 3 | 是 | 嵌入式数据库，项目启动时自动创建数据文件，无需额外安装 |
| Git 2.25+ | 否 | 仅用于克隆仓库，可通过 zip 包替代 |
| 操作系统 Linux / macOS / Windows | 否 | Windows 需使用 WSL2 或 PowerShell 7+ 运行启动脚本 |
| 网络访问 | 否 | 仅当启用外部链接可用性探测时需要，核心功能离线可用 |

## 贡献指南

1. 查阅 issues 列表，选择未被认领且标签为 "good first issue" 或 "help wanted" 的任务。若准备提出新功能或改进建议，请先新建 issue 描述需求背景与实现思路，等待核心维护者反馈后再着手开发。

2. 克隆仓库并创建特性分支，分支命名采用 `feature/功能简述` 或 `fix/问题简述` 格式。开发前请确保本地 Node.js 版本与项目 `.nvmrc` 声明一致，并执行 `npm install` 安装全部依赖。

3. 编写代码时遵循项目已配置的 ESLint 与 Prettier 规则。提交前运行 `npm run lint` 与 `npm run test` 保证代码风格一致且所有单元测试通过。对新增功能需补充对应测试用例。

4. 提交信息使用 Conventional Commits 规范，格式为 `<type>(scope): <subject>`，例如 `feat(import): support JSON lines format`。提交后推送至个人 fork 仓库，并通过 GitHub 界面发起 Pull Request 到主仓库的 main 分支。

5. PR 描述中需说明改动内容、测试覆盖情况、是否影响现有接口或数据库结构（如涉及迁移，请附带迁移脚本）。至少一位核心维护者审核通过后合并。

## 常见问题

Q: 启动时提示 `SQLITE_ERROR: no such table: resources`，如何处理？
A: 这是由于数据库尚未执行迁移脚本。请检查项目根目录下是否存在 `data/` 文件夹以及其中的 `.db` 文件。若不存在，运行 `npm run migrate` 手动执行迁移；若已存在但表缺失，可运行 `npm run reset-db` 重置数据库并重新初始化（注意该操作会清空所有已有数据）。

Q: 导入大量资源时页面响应变慢，有无优化方案？
A: 资源导入默认采用逐条插入方式，单次导入超过 500 条时建议使用批量导入接口 `/api/resources/batch`，该接口接受 JSON 数组并使用事务批量写入。同时在检索场景下，若资源总数超过 5000 条，建议在 `config/default.yaml` 中启用全文索引（FTS5）并调整分页大小为 50 以减少单次查询负载。

Q: 是否可以支持 PostgreSQL 或 MySQL 替代 SQLite？
A: 当前版本仅内置 SQLite 适配器。项目数据访问层已抽象出统一的 `DatabaseClient` 接口，如需迁移至其他关系型数据库，可参考 `/src/server/db` 目录下的实现，自行编写对应适配器并修改 `config` 中的 `db.dialect` 配置。社区贡献的适配器将通过独立扩展包形式发布。
