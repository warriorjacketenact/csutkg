# TangChengLvShi Episode Gateway

TangChengLvShi Episode Gateway 是一个面向法律从业者、研究学者与公共政策分析人员的技术资源导航工具，专注于聚合与法律实务、裁判文书、合规审查及政策解读相关的外部参考资料。该项目不生产原始内容，而是通过系统化的链接索引机制，帮助用户快速定位分布在多个数据源中的案件分析、法规释义与行业观察文章。

项目定位为法律科技生态中的信息中间层，以轻量化、可扩展的目录结构组织大量外部 URL，适用于个人研究者批量导入参考文献、律所知识管理部门构建内部知识库，以及法律科技产品团队在原型阶段快速集成第三方内容源。通过统一的条目标识符体系，每个外部资源均获得唯一的内部引用代号，便于在团队协作、报告撰写或系统对接中精确追溯。

## 功能概览

**批量链接导入** 支持一次性载入数百条外部 URL，自动解析域名与路径结构，生成带唯一编号的条目索引。

**分类标签系统** 每条链接可标记多级分类标签，涵盖民事、刑事、行政、知识产权、公司合规等常见法律领域。

**全文检索过滤** 基于条目编号、来源域名、路径关键词进行快速过滤，支持正则表达式与通配符模式。

**元数据提取** 自动抓取目标页面的标题、发布时间、内容摘要等基础元数据，用于生成预览卡片。

**导出与集成** 将索引数据导出为 JSON、CSV 或 Markdown 表格格式，便于导入 Notion、飞书文档或自建 CMS。

**状态监控** 定时检测链接可用性，标记失效或重定向条目，生成健康报告。

## 应用场景

**法律研究文献整理** 研究者在撰写论文或报告时，需要引用大量裁判文书解读与法规分析文章。通过本项目的批量导入与分类功能，可将分散在数十个页面中的参考资料统一归集，并为每条记录生成内部引用编号，减少手动整理错误。

**律所知识库建设** 律所的知识管理团队定期收集行业监管动态与典型案例。利用本项目的元数据提取与状态监控能力，可自动化完成外部资源的采集与更新，确保内部知识库中的链接始终有效且信息完整。

**法律科技产品原型开发** 创业团队在构建法律问答系统或智能检索工具时，需要预先准备大量高质量的外部数据源作为测试语料。本项目提供标准化的导出接口，可快速将数百条参考链接转换为结构化数据，供原型系统调用。

**合规培训材料准备** 企业法务部门为员工制作合规培训材料时，需引用最新的法规解读与处罚案例。通过本项目的分类标签与检索功能，可针对不同培训主题快速筛选相关资源，生成参考资料清单。

## 快速开始

以下命令将在本地克隆项目仓库、安装依赖并启动开发服务器。

```bash
git clone https://github.com/tanchenglvshi/episode-gateway.git
cd episode-gateway
npm install
npm run dev
```

执行完毕后，访问控制台输出的本地地址即可进入条目管理界面。默认使用 3000 端口，如有冲突可修改 .env 文件中的 PORT 变量。

## 贡献指南

1. 在 GitHub 上 fork 本仓库，并克隆到本地开发环境。建议使用 Node.js 18 及以上版本，并严格按照 .env.example 创建本地环境变量文件。

2. 选择或新建一个 issue 描述你希望修复的问题或新增的功能。对于较大规模的改动，建议先在 issue 中阐述设计方案，获得维护者反馈后再开始编码。

3. 编写代码时遵循项目内预设的 ESLint 与 Prettier 规则。所有新增或修改的功能必须附带对应的单元测试，测试文件放置在 tests/ 目录下对应子目录中。

4. 提交 commit 时使用语义化的提交信息格式，例如 `feat: 添加批量导入进度条` 或 `fix: 修复元数据提取时编码错误`。提交前确保所有测试通过，执行 `npm run test` 验证。

5. 发起 Pull Request 到主仓库的 main 分支。PR 描述中应清晰说明改动内容、测试覆盖情况以及是否涉及文档更新。维护者将在 3 个工作日内进行评审。

## 常见问题

**问：项目最多支持管理多少条链接？性能瓶颈在哪里？**

答：项目本身不设硬性上限。在 SQLite 默认配置下，实测可稳定管理 5000 至 10000 条条目，检索响应时间保持在 200 毫秒以内。性能瓶颈主要出现在元数据提取阶段，当同时抓取大量外部页面时，网络 I/O 与 HTML 解析会占用较多 CPU 与内存资源。建议通过配置 `maxConcurrentFetches` 参数控制并发数，或部署 Redis 队列实现分布式抓取。

**问：如何更新已有条目的元数据？**

答：在管理界面中点击单条条目右侧的「刷新元数据」按钮，系统将重新发起 HTTP 请求并覆盖原有的标题、摘要与发布时间字段。如需批量刷新，可在命令行执行 `npm run refresh -- --batch --limit=100`，该命令会按照条目 ID 顺序分批处理。注意频繁刷新可能触发目标服务器的反爬策略，建议设置合理的请求间隔。

**问：项目是否支持 PostgreSQL 替代 SQLite？**

答：项目内置的数据库适配层基于 Knex.js 构建，理论上可切换至 PostgreSQL、MySQL 等主流关系型数据库。但需注意，部分查询语句使用了 SQLite 特有的 JSON 函数，迁移前需重写对应的查询构造器。项目暂未提供官方迁移指南，但社区已有用户成功部署在 PostgreSQL 环境，相关配置可参考 docs/deployment.md 中的非官方章节。
