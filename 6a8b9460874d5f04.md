# ResourceLink Aggregate Registry (RLAR)

ResourceLink Aggregate Registry (RLAR) 是一个面向技术文档工程师、知识库维护者、外链管理人员以及自动化资源采集系统的开源外链汇总与规范化管理工具。该项目并非一个传统的网页应用或前端界面，而是一套严格遵循条目化、可追溯原则的URL资源清单存储库，附带结构化的元数据描述、分类索引与运维脚本。

RLAR 的核心目标用户包括：需要维护大量外部参考链接的技术文档团队、从事网络资源归档的数字图书馆员、搭建垂直领域导航站点的开发者，以及希望将散乱外链纳入版本控制体系进行变更追踪的运维人员。本项目解决的核心问题是：外链散落在笔记、邮件、书签或即时通讯记录中，导致链接失效、来源模糊、用途不明，且无法进行批量检查与自动化处理。RLAR 将全部外链以纯文本条目形式固化于仓库中，配合自定义脚本实现链接可达性检测、状态码记录、分类标签提取与变更审计，使外链管理从“随手收藏”升级为“工程化治理”。

本项目当前批次为第 744/1241 批，累计收录资源链接 250 条。所有链接均来自生产环境采集管道，经过去重与格式规范化处理，形成本批次稳定发布版本。

## 功能概览

**条目化外链存储**：每条URL以独立行记录，保留原始采集格式，禁止任何自动补全或改写，确保与上游数据源完全一致。

**分类标签索引**：根据URL路径特征、域名归属及查询参数模式，自动为每条链接分配一级分类与二级标签，支持按技术领域、内容类型、地域或语种快速筛选。

**批量可达性检测**：内置Python检测脚本，支持并发HEAD请求验证每条链接的HTTP状态码，输出可达、重定向、超时或失效等状态报告。

**变更历史审计**：所有增删改操作通过Git提交记录关联，每次发布批次附带变更摘要文件，记录新增、移除与URL修改详情。

**元数据扩展支持**：允许用户为每条链接附加自定义键值对元数据（如采集时间、来源节点、重要等级），存储于同目录下的JSON sidecar文件中。

**导出与集成适配**：支持将条目列表导出为JSON、CSV、OPML或纯文本格式，便于导入其他导航系统、RSS阅读器或浏览器书签工具。

**定时巡检任务模板**：提供GitHub Actions工作流模板，可配置每周自动运行可达性检测，并将报告提交至仓库Issues或指定邮件列表。

## 应用场景

技术文档团队维护API参考手册的外部链接附录。当撰写集成指南或SDK说明文档时，需要引用大量第三方服务官网、SDK仓库地址、社区论坛帖子。RLAR 可作为官方外链附录的数据源，每次文档发版前运行检测脚本，确保所有引用链接有效，避免用户遭遇404页面。

数字图书馆或专题知识库建设者进行网络资源批量归档。针对特定主题（如开源数据库工具、前端UI框架、机器学习数据集）采集相关资源链接后，通过RLAR进行分类存储，并定期重新检测资源可用性，自动标记已迁移或已下线的资源，维持归档质量。

独立开发者构建垂直领域导航站点。需要管理数百个精品资源条目，但不愿使用复杂的数据库后台。RLAR 以纯文本形式维护所有链接，通过简单的脚本生成静态导航页面的HTML或Markdown，将内容与展示分离，同时保留完整的版本历史。

自动化运维团队监控外部依赖服务的可用性。将第三方API端点、健康检查页面、状态页地址纳入RLAR管理，结合定时巡检任务，当检测到大量链接失效时触发告警，辅助判断第三方服务故障或自身网络策略变更的影响范围。

## 快速开始

以下命令序列演示如何获取本仓库、安装基础依赖并执行首次链接状态检测。

```bash
git clone https://github.com/resourcelink/rlar.git
cd rlar
pip install -r requirements.txt
python scripts/check_links.py --input data/batch_744_1241.txt --output reports/status_744_1241.json
```

如需生成分类索引摘要，可运行：

```bash
python scripts/index_links.py --source data/batch_744_1241.txt --format markdown --output docs/index_744_1241.md
```

执行完毕后，将在 `reports/` 目录下生成包含每条链接状态码、响应时间和最终目标URL的JSON报告，并在 `docs/` 目录下生成人类可读的分类索引文档。

## 安装要求

| 依赖项 | 必需版本 | 说明 |
|--------|----------|------|
| Python | 3.8 及以上 | 所有脚本基于Python解释器运行，不支持Python 2.x |
| requests | 2.25.0 及以上 | 用于发送HTTP请求进行链接状态检测，支持重定向跟踪与超时设置 |
| click | 8.0.0 及以上 | 命令行接口解析器，为各脚本提供统一的子命令和参数风格 |
| pytest | 7.0.0 及以上 | 仅开发环境需要，用于运行单元测试和集成测试用例 |
| git | 2.20.0 及以上 | 用于克隆仓库、提交变更、查看历史记录，以及触发pre-commit钩子 |
| curl | 7.68.0 及以上 | 可选依赖，部分备用检测脚本使用curl作为降级方案 |

所有必需依赖均可通过 `pip install -r requirements.txt` 一次性安装。生产环境建议使用虚拟环境或容器化部署。测试环境需额外安装 `pytest` 与 `pytest-cov`。

## 贡献指南

欢迎各类贡献，包括但不限于新增链接批次、改进检测脚本、完善文档、增加导出格式适配器。请遵循以下步骤。

提交新增批次数据：在 `data/` 目录下创建新文件，命名为 `batch_<批次号>_<条目数>.txt`，每行一条原始URL。同时提交对应的 `batch_<批次号>_<条目数>.meta.json` 文件，记录采集时间、来源、采集工具版本等信息。Pull Request 标题请注明批次号。

改进检测逻辑：若发现检测脚本对某些特殊链接（如需要特定User-Agent、支持IPv6、返回非标准状态码）处理不当，请附带目标链接的示例和期望行为，提交修复代码并增加对应测试用例至 `tests/` 目录。所有修改需通过现有测试套件。

完善分类索引规则：分类规则定义于 `scripts/utils/parsers.py` 中的 `CLASSIFY_RULES` 字典。若要新增分类或调整现有分类匹配模式，请提供至少10个样本链接及其预期分类结果，以验证规则有效性。

文档更新：对于用户手册或运维指南的修订，请确保新增内容与现有章节风格一致，代码示例需可实际运行。开发者文档的变更需同步更新函数签名和类图描述。

报告问题：使用GitHub Issues提交Bug报告或功能请求。Bug报告需包含：运行环境（操作系统、Python版本、依赖版本）、复现步骤、实际输出与预期输出的对比。功能请求需阐明使用场景和预期收益。

## 常见问题

Q: 为什么每条链接都强制保留原始格式，包括混合了http和https、带或不带www前缀？是否应该统一规范化？

A: 本项目设计原则之一是“原始数据不可变性”。统一规范化虽然有利于去重和比对，但会丢失上游数据源的真实形态，例如某些链接在采集时就是裸域名，而在其他记录中可能带www，这反映了不同采集时间点的网络配置差异。RLAR 在检测脚本中会实际访问链接并跟踪重定向，最终记录可达的规范URL，但原始条目保持原样，以便审计采集过程。用户可通过 `--normalize` 参数在导出时进行规范化，但不修改源文件。

Q: 检测脚本报告某链接返回403或429，但浏览器可以正常访问，如何解决？

A: 此类情况通常由服务器对非浏览器User-Agent或高频请求进行限制。解决方法：1) 在 `check_links.py` 中通过 `--headers` 参数自定义User-Agent，模拟主流浏览器；2) 使用 `--delay` 参数增加请求间隔，降低并发数；3) 若链接需要特定Cookie或Referer，可在 `scripts/utils/http_client.py` 中配置全局会话默认值。建议优先使用 `--delay`，避免被目标服务器封禁。

Q: 如何只检测新增或变更的链接，而非全量检测？

A: 使用 `--diff` 模式，指定上一批次文件路径，脚本会自动计算两个批次文件的差集，仅对新链接或URL发生变更的条目执行检测。变更检测基于整行完全匹配，若同一链接在前后批次中查询参数不同，视为新条目。检测完成后，报告会同时包含全量统计和增量统计两个部分。
