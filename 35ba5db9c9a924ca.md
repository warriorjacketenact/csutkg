# ResourceHub

ResourceHub 是一个面向开发者、技术研究人员与开源项目维护者的结构化技术资源导航与元数据聚合工具。该项目并非传统的内容管理系统，而是一套轻量级的外链资源汇集与分类框架，专注于将分散在互联网各处的技术文档、教程、工具站点、参考手册与社区讨论帖按照统一的索引规范进行归集，并提供可插拔的解析与展示能力。

ResourceHub 定位为技术团队内部知识库的前置采集层，亦可供独立开发者快速搭建个人技术阅读清单。项目不依赖数据库，核心数据以 Markdown 与 YAML Frontmatter 形式存储，支持静态站点生成器直接消费，亦可作为 VS Code 插件的数据源。通过 ResourceHub，用户可以将原本零散的书签、阅读列表、项目依赖文档整合为可检索、可版本化、可协作的资产目录。

## 功能概览

- **外链资源注册与管理** 支持通过 YAML 模板注册任意 HTTP/HTTPS 资源，自动提取域名、路径层级、响应状态码等元数据，并提供资源存活期健康检查脚本。

- **标签与分类体系** 内置技术栈、难度等级、资源类型（文档/工具/视频/社区）三个维度的分类标签，支持用户自定义扩展，每个资源可归属多个分类。

- **全文检索与过滤** 基于 MiniSearch 或 Lunr.js 实现静态索引构建，支持标题、描述、标签、来源站点的模糊查询与组合筛选。

- **资源关系图谱** 自动分析资源之间的引用关系与域名共现频率，生成关联网络视图，辅助发现技术选型中的隐性依赖。

- **快照与版本对比** 对每个外链资源记录注册时间与最后校验时间，支持 Markdown 格式的变更日志输出，便于团队 Review 资源更新。

- **批量导入与导出** 提供从浏览器书签 HTML、Pinboard 导出 JSON、Feedly OPML 等格式的转换脚本，同时支持将资源列表导出为 CSV 或结构化 Markdown 表格。

- **访问统计看板** 基于本地访问日志或简单的点击计数器，展示热门资源排行与分类活跃度，帮助团队识别高频参考资料。

## 应用场景

**技术团队内部知识库索引**：研发团队可将日常使用的 API 文档、设计规范、部署手册、故障排查案例等外链统一录入 ResourceHub，配合 CI/CD 流程每日校验链接可用性，避免团队书签失效或依赖过时文档。

**开源项目 README 外链管理**：开源项目维护者常需在 README 中维护大量引用链接，ResourceHub 提供链接分组、别名引用与自动格式化功能，可一键生成符合各大托管平台规范的 Markdown 链接列表，减少手工排版错误。

**技术雷达与选型调研**：架构师在调研新框架或中间件时，可将候选项目的官网、GitHub 仓库、性能测试报告、社区讨论帖集中收录，利用标签体系标记评估维度，输出结构化的技术选型对比报告。

**个人开发者的阅读工作流**：独立开发者可将技术博客、播客单集、在线课程、电子书等学习资源按主题归集，结合快照功能记录阅读进度与笔记引用，构建长期积累的个人知识图谱。

## 快速开始

以下命令适用于 Linux 与 macOS 环境，Windows 用户建议通过 WSL2 或 Git Bash 执行。

```bash
# 克隆项目仓库
git clone https://github.com/resourcehub/resourcehub.git
cd resourcehub

# 安装依赖（项目使用 Node.js 18+ 与 pnpm）
pnpm install

# 复制示例配置文件并调整本地端口与资源存储路径
cp .env.example .env

# 启动开发服务器，访问 http://localhost:4321 查看资源导航界面
pnpm run dev
```

生产环境构建命令：

```bash
pnpm run build
pnpm run preview
```

若仅需使用命令行工具进行资源列表校验与格式转换，可执行：

```bash
pnpm run cli -- --help
```

## 安装要求

| 依赖 | 必需 | 说明 |
|---|---|---|
| Node.js 18.x 或 20.x LTS | 是 | 运行时环境，推荐使用最新的 LTS 版本，低版本可能不支持原生 Fetch API |
| pnpm 8.x 或 9.x | 是 | 包管理器，暂不支持 npm 或 yarn 直接安装，因使用了 pnpm workspace 特性 |
| Git 2.30+ | 是 | 用于克隆仓库、版本管理及自动生成变更日志 |
| Python 3.9+（可选） | 否 | 仅在运行资源健康检查脚本（检查 301/404 等状态）时需要 |
| Nginx 或 Caddy（可选） | 否 | 生产环境部署推荐的反向代理，用于处理缓存与 SSL 终止 |
| shellcheck（可选） | 否 | 用于校验项目内辅助 shell 脚本的语法正确性 |

## 项目结构

```
resourcehub/
├── .github/                         # GitHub 社区配置文件
│   ├── workflows/                   # CI 流水线（单元测试、链接检查）
│   └── ISSUE_TEMPLATE/              # 问题与资源申请模板
├── packages/                        # Monorepo 子包目录
│   ├── core/                        # 核心数据模型与资源解析器
│   │   ├── src/models/              # Resource, Tag, Category 实体定义
│   │   ├── src/parsers/             # 针对不同来源的 HTML/JSON 解析适配器
│   │   └── tests/                   # 单元测试覆盖
│   ├── cli/                         # 命令行工具入口（资源校验、导入导出）
│   │   ├── commands/                # 各子命令实现（check, import, export）
│   │   └── bin/                     # 可执行脚本
│   ├── web/                         # 基于 Astro 构建的静态导航站点
│   │   ├── src/pages/               # 路由页面（首页、分类页、详情页）
│   │   ├── src/components/          # UI 组件（搜索框、标签云、资源卡片）
│   │   └── public/                  # 静态资源与 favicon
│   └── shared/                      # 跨包共享的类型定义与工具函数
├── data/                            # 资源数据存储目录（YAML + Markdown）
```

## 贡献指南

ResourceHub 欢迎各类贡献，包括但不限于新增资源索引、完善解析规则、改进文档与修复缺陷。请遵循以下步骤：

1. 查阅项目 Issue 列表与 Project Board，确认当前正在进行的工作，避免重复劳动。对于较大功能或结构变更，建议先创建 Discussion 议题进行方案沟通。

2. 从 main 分支创建功能分支，分支命名采用 feat/、fix/、docs/ 前缀加简要描述，例如 feat/add-python-parser。提交信息遵循 Conventional Commits 规范，便于自动生成变更日志。

3. 新增资源索引时，需在 data/sources/ 下按照模板创建 YAML 文件，并确保所有必填字段（title、url、tags、category）完整无误。运行 pnpm run validate 校验数据格式是否通过。

4. 提交 Pull Request 前，请确保本地通过全部单元测试与链接检查（pnpm run test 与 pnpm run check:links），并在 PR 描述中引用相关 Issue 编号或讨论链接。

5. 文档类贡献可直接编辑 docs/ 目录下的 Markdown 文件，无需通过功能分支，但需确保中文表述清晰、术语统一，且不包含外链失效或过时信息。

## 常见问题

**Q: ResourceHub 是否必须部署为 Web 服务？能否仅作为命令行工具使用？**

A: 不必。ResourceHub 的核心数据处理能力完全封装在 @resourcehub/core 包中，CLI 工具提供了完整的资源注册、校验与导出功能，用户可以在无 Web 界面的环境下运行，例如作为 Git Hook 或定时任务。Web 站点仅为可选的可视化层，方便团队协作浏览。

**Q: 如何处理目标外链资源变更 URL 或彻底下线的情况？**

A: ResourceHub 维护每个资源的永久 ID，当目标链接变更时，用户可在 YAML 文件中更新 url 字段并增加 changelog 条目，系统保留历史记录。若资源彻底不可访问且无替代链接，建议将 status 标记为 archived，并在备注中说明原因。健康检查脚本默认将 410 和 404 状态视为失效，可根据策略自动发送通知。

**Q: 项目是否支持私有化部署与权限控制？**

A: ResourceHub 本身不内置用户认证与权限管理。对于需要私有访问的场景，推荐部署在内部网络，并通过 Nginx 基础认证、OAuth2 Proxy 或 Cloudflare Access 等网关层方案实现访问控制。数据目录可通过 Git 私有仓库托管，确保资源元数据不外泄。
