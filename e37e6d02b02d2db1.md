# Tangalooma House Film Resource Index

Tangalooma House Film Resource Index 是一个面向影视研究、媒体分析和内容策展场景的技术资源汇总系统。该项目并非传统的电影数据库或流媒体聚合器，而是一个结构化的外部链接索引库，专注于收集和分类来自 tangaloomahouse.com 域下的深度电影专题页面、幕后制作解析、影评人专栏以及行业技术文档。

本项目的主要目标用户包括影视专业学生、内容策展人、媒体研究者以及需要批量处理影视元数据的开发者。通过提供稳定、可机读的 URL 资源列表，Tangalooma House Film Resource Index 帮助用户快速定位分散在大量子目录中的高价值影评与制作笔记，避免了人工爬取和手动整理的低效流程。项目本身不存储任何侵权内容，仅作为公开资源的导航层存在，严格遵守内容聚合的道德与法律边界。

## 功能概览

- **结构化资源导航**：提供超过 250 条精确到单部电影子页面的深度链接，所有链接均按原始路径完整保留，确保用户可直接访问源站内容。

- **多维度分类索引**：资源按照电影编号、类型标签和发布批次进行逻辑分组，支持用户按特定编码（如 xwv-2045、lft-8924）快速筛选目标条目。

- **批量外链管理**：专为需要处理大量影视链接的开发者设计，支持将列表导出为 CSV、JSON 或纯文本格式，便于集成到外部内容管理系统中。

- **状态监控与可用性检测**：内置链接有效性检查机制，可定期扫描资源列表中的死链或重定向，并生成健康度报告，帮助维护资源库的长期可用性。

- **版本化更新记录**：每一批资源导入均带有批次号标记（当前为第 467/1241 批），支持回滚和历史变更追踪，确保团队协作时的数据一致性。

- **轻量化快速部署**：项目无复杂后端依赖，基于静态 Markdown 文档和脚本工具即可运行，适合在低资源环境下快速搭建内部知识库导航页。

## 应用场景

- **影视学术研究的数据准备**：研究人员在分析特定导演或制片公司的创作风格时，可通过本索引快速获取该域名下收录的所有相关影评与访谈链接，批量导入文献管理工具，极大缩短前期资料收集时间。

- **内容策展与专题汇编**：电影节策展人或流媒体编辑需要围绕某个主题（如「女性主义电影」「赛博朋克视觉美学」）整理推荐片单时，可利用本项目的分类链接体系，高效生成带原始出处的推荐阅读列表。

- **自动化工作流中的链接源**：开发者编写爬虫或数据管道时，可直接将本项目的资源列表作为种子 URL 集合，避免手动构造正则表达式去匹配 tangaloomahouse.com 的 URL 结构，降低解析代码的维护成本。

## 快速开始

以下指令帮助您在本地环境中快速部署并运行 Tangalooma House Film Resource Index 的基础功能，包括资源列表的拉取、格式校验和静态页面生成。

```bash
# 克隆项目仓库至本地
git clone https://github.com/tangalooma/film-resource-index.git

# 进入项目根目录
cd film-resource-index

# 安装必要的 Python 依赖（用于链接格式校验和统计报告生成）
pip install -r requirements.txt

# 运行索引生成脚本，输出当前批次的资源导航页面
python build_index.py --batch 467 --output ./docs/index.html
```

## 安装要求

为确保项目脚本和工具链正常运行，请确保您的开发环境满足以下依赖条件。建议使用 Python 3.9 及以上版本，并提前配置好虚拟环境。

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.9 或更高 | 用于运行链接解析、格式校验和静态生成脚本 |
| pip | 21.0 或更高 | Python 包管理工具，用于安装 requirements.txt 中的第三方库 |
| Git | 2.25 或更高 | 用于克隆仓库和管理版本更新 |
| Markdown 解析器 | Python-Markdown 3.3.6 | 用于将本 README 格式的索引文档转换为 HTML 预览页 |
| 网络连接 | 稳定访问外网 | 用于在生成报告时验证资源链接的可达性（可选，可离线运行） |
| 操作系统 | Linux / macOS / Windows WSL2 | 脚本在 Unix-like 环境下测试最充分，Windows 用户推荐使用 WSL |

## 项目结构

项目采用模块化目录设计，将核心脚本、资源数据、文档和输出目录清晰分离，便于长期维护和自动化部署。

```bash
.
├── README.md                     # 项目概述与资源索引主文档（本文件）
├── CONTRIBUTING.md               # 贡献者指南，包含 PR 流程与编码规范
├── LICENSE                       # MIT 许可证全文
├── requirements.txt              # Python 依赖声明文件
├── .gitignore                    # Git 版本控制忽略规则
│
├── build_index.py                # 核心脚本：根据资源列表生成静态导航页
├── check_links.py                # 辅助脚本：批量检测资源链接的 HTTP 状态码
├── export_csv.py                 # 导出工具：将链接列表转换为 CSV 格式
│
├── data/                         # 数据存储目录
│   ├── raw/                      # 原始输入数据（按批次命名的 txt 文件）
│   │   └── batch_467.txt
│   ├── parsed/                   # 解析后的结构化 JSON 数据
│   │   └── batch_467.json
│   └── reports/                  # 链接健康度报告与统计信息
│       └── health_467.log
│
├── docs/                         # 用户文档与生成的 HTML 页面
│   ├── index.html                # 当前批次的可视化导航页
│   ├── user-guide.md
│   ├── developer-guide.md
│   └── operations.md
│
├── templates/                    # 页面生成所用的 Jinja2 模板
│   └── nav_template.html
│
└── tests/                        # 单元测试与集成测试脚本
    ├── test_parser.py
    └── test_exporter.py
```

## 贡献指南

我们欢迎社区贡献者提交新的资源链接批次、改进脚本工具或完善文档。请遵循以下步骤确保您的贡献能够被顺利审核和合并。

1.  **派生仓库并创建特性分支**：从主仓库派生副本到您的个人账户下，然后基于 `main` 分支创建一个以 `feature/` 或 `fix/` 为前缀的新分支，避免直接在主分支上修改。

2.  **更新资源数据或脚本**：如果您提交的是新批次链接，请在 `data/raw/` 目录下创建对应批次的文本文件，并确保每行一个 URL，格式与现有批次保持一致。若提交的是代码改进，请同步更新相应的单元测试。

3.  **运行本地校验**：在提交 Pull Request 前，请执行 `python check_links.py --batch 467` 验证新链接的有效性，并运行 `pytest tests/` 确保所有测试用例通过。项目维护者将拒绝未通过自动校验的贡献。

4.  **撰写清晰的提交信息**：提交信息应遵循常规提交规范，简要说明变更类型（feat/fix/docs）和具体内容。Pull Request 描述中请注明本次添加的资源总数或修复的问题编号。

5.  **等待 Code Review**：项目维护者将在 48 小时内审阅您的贡献，可能会提出修改建议。请积极回应反馈，并根据需要更新您的分支。
