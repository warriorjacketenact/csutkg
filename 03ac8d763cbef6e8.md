# GBDLS Link Aggregator

GBDLS Link Aggregator 是一个面向技术研究、数据分析和数字取证领域的结构化外链资源汇总工具。项目致力于将分散在网络各处的深层页面、数据快照和原始素材按照主题与用途进行归类整理，为研究人员、开发者和情报分析人员提供可复用的数据入口集合。本项目不生产内容，仅对现有公开网络资源进行索引与描述性标注。

项目当前批次为第 1235/1241 批，共收录 250 个资源链接。所有链接均来自 gbdls.net 域下的数字路径节点，这些节点指向不同类型的静态数据、动态页面、API 测试端点或存档资料。通过本项目的分类索引，用户可以快速定位到与自身工作领域相关的资源集合，避免在海量无结构数据中重复检索。

## 功能概览

批量链接导入与解析：支持将大量形如 gbdls.net/x/{id} 的短链地址批量导入系统，自动识别链接状态（有效、跳转、失效）并记录元数据。

多维度标签分类：每个链接可被打上多个自定义标签，标签体系包括但不限于：技术文档、日志样本、数据快照、测试端点、参考素材、历史存档、API 响应样例等。

链接状态健康监测：定时对已收录链接进行可达性检查，输出可用的 HTTP 状态码、响应时间、内容类型及内容长度摘要，便于用户筛选有效资源。

全文检索与过滤：支持按 URL 路径、标签组合、状态码、内容类型及收录批次进行精确检索与过滤，快速定位特定类型资源。

自定义数据集导出：允许用户将筛选后的链接集合导出为 JSON、CSV 或纯文本列表格式，便于集成到其他自动化工具或工作流中。

可视化目录树浏览：提供基于目录结构的资源导航界面，用户可按照预设的分类层级逐级浏览，直观查看项目整体资源分布。

协作标注与备注：支持多用户协作场景下的备注添加功能，每位用户可对特定链接添加个人备注或研究笔记，并标记重要程度。

RESTful API 接口：提供完整的只读 API 接口，支持外部程序通过 JSON 格式获取链接列表、分类信息及状态数据，便于二次开发。

## 应用场景

技术研究文献回溯：研究者在撰写技术报告或论文时，需要引用特定时间点的网络资源快照。本项目收录的 gbdls.net 链接指向大量静态数据页面，可作为引用素材库使用。研究者可通过标签快速筛选出与主题相关的链接集合，统一进行内容核查与引用登记。

数据工程测试样本采集：数据工程师在构建 ETL 管道或测试数据解析逻辑时，需要多样化的原始数据样本。本项目涵盖多种响应格式的资源链接，包括 HTML、JSON、纯文本及二进制数据端点，可作为测试用例来源。工程师可导出链接列表后编写自动化拉取脚本，批量获取样本数据。

网络安全取证分析：安全分析人员在事件响应或威胁狩猎过程中，经常需要比对历史数据特征或模拟攻击载荷响应。本项目中的部分链接指向历史日志片段、错误页面及特殊状态码响应，可用于验证检测规则或复现分析环境。

前端开发 Mock 数据源：前端开发人员在原型开发阶段需要模拟后端接口响应。本项目收录的若干链接可作为临时 Mock 数据来源，开发者可从中获取真实世界的响应结构，加快原型迭代速度。

数据归档与去重参照：数字档案管理员在整理大规模数据集合时，需要参照已有的公开数据索引来避免重复收录。本项目提供的批次化链接列表可以作为数据去重的参照依据，管理人员可通过比对链接路径快速识别已有数据范围。

## 快速开始

以下步骤帮助您在三分钟内完成项目克隆、依赖安装与服务启动。

```bash
# 克隆代码仓库
git clone https://github.com/gbdls/link-aggregator.git
cd link-aggregator

# 安装项目依赖（使用 pip 和虚拟环境）
python3 -m venv venv
source venv/bin/activate  # Windows 下使用 venv\Scripts\activate
pip install -r requirements.txt

# 初始化本地数据库并导入当前批次链接列表
python scripts/init_db.py --batch 1235 --count 250
python scripts/import_links.py --batch 1235 --source data/batch_1235.txt

# 启动本地开发服务器
python app.py run --host 127.0.0.1 --port 8080
```

启动成功后，访问 http://127.0.0.1:8080 即可进入项目本地实例的控制台界面。控制台提供链接查询、标签过滤、状态监测及数据导出等核心功能。

## 贡献指南

项目欢迎社区贡献者参与链接分类校准、代码优化及文档完善工作。所有贡献者需遵守以下流程以确保项目质量。

第一，在 GitHub 上 fork 本仓库至个人账户，然后克隆至本地开发环境。创建新的功能分支时，分支名称应遵循 feature/{功能简述} 或 fix/{问题简述} 的命名规范，例如 feature/tag-autocomplete 或 fix/status-check-timeout。

第二，若贡献内容涉及新增链接分类或修订现有标签体系，请在 data/schemas/tag_schema.json 中同步更新分类定义，并提交至少三个样例链接以证明分类规则的普适性。代码层面的修改需通过现有单元测试，新增功能需附带对应的测试用例。

第三，提交代码前运行 black 与 pytest 进行格式检查和测试验证。本项目使用 pre-commit 自动执行基础检查，请确保本地已安装 pre-commit 并执行 pre-commit install 安装钩子。提交信息应使用英语书写，采用简洁的祈使语气，例如 Add batch import retry mechanism 或 Fix filter pagination error。

第四，发起 Pull Request 后，项目维护者将在三个工作日内进行审查。审查意见将直接在 PR 页面中以逐行评论形式给出。若审查通过，PR 将被合并至主分支；若需要修改，贡献者应根据意见补充提交，直至满足合并标准。

第五，对于重大变更，例如数据库结构迁移、核心 API 接口调整或标签体系重构，贡献者需在 docs/ 目录下补充相应的迁移指导文档，并在 PR 描述中明确标注为破坏性变更，以便项目维护者评估影响范围。

## 常见问题

Q: 项目中的链接出现 404 或超时，应该如何处理？

A: 链接状态检测服务每日凌晨 2:00 自动运行一次，更新所有链接的最近状态。用户在 Web 控制台可手动点击「立即检测」按钮对单个链接发起即时检查。如果链接持续不可用，用户可在备注中标记「已失效」并记录最后有效时间。项目维护者会定期清理连续 30 天不可用的链接，但不会主动删除历史记录，仅将其移出默认视图。

Q: 如何将本项目集成到自己的数据分析流水线中？

A: 项目提供只读 RESTful API，基础地址为 /api/v1。主要端点包括 /api/v1/links（支持分页与标签过滤）、/api/v1/batches（批次列表）及 /api/v1/tags（标签树）。所有 API 响应均为 JSON 格式。用户可编写简单的 HTTP 客户端轮询 API，获取链接列表后传入下游处理程序。对于大批量导出需求，建议使用 Web 控制台的「导出数据集」功能，系统将异步生成 CSV 或 JSON 文件并提供下载链接。

Q: 数据库文件会随着链接数量增长而膨胀，如何维护？

A: 项目默认使用 SQLite 数据库，单批次 250 条链接的元数据存储空间约为 2-3 MB。在链接数量超过 10 万条时，建议迁移至 PostgreSQL。项目已在 models/ 层使用 SQLAlchemy ORM，切换数据库仅需修改配置文件中 SQLALCHEMY_DATABASE_URI 变量。此外，项目提供 scripts/vacuum_db.py 脚本，可定期执行数据库压缩与索引重建，建议每月运行一次以维持查询性能。
