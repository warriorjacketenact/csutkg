# LinkVault Core

LinkVault Core 是一个面向技术团队与内容策展人的轻量级外链资源聚合与导航系统。该项目定位于解决多源技术文档、API 参考、社区教程及内部知识库链接分散、难以统一检索与版本追踪的问题，帮助开发者、技术写作人员与运维工程师在单一视图中管理数百个外部资源条目。LinkVault Core 不提供爬虫或自动采集功能，而是通过结构化的 YAML 配置文件与标签体系，将原始链接列表转化为可分类、可注释、可导出为静态站点的导航数据源。本项目适用于需要定期同步大量外部链接、并按批次组织发布的技术内容运营场景。

## 功能概览

**批量链接导入与清洗** 支持从 CSV、JSON 及纯文本列表中批量导入 URL，自动去除重复条目、检测失效协议头并补全缺失的 scheme，同时保留原始 URL 字符串的完整形态。

**多级标签分类引擎** 允许对每条资源赋予最多五级标签（如 `docs/database/redis` 或 `tutorial/security/oauth`），并基于标签树生成自动聚合视图。

**链接状态健康检查** 集成异步 HTTP 探活机制，可配置超时与重试策略，周期性检测资源可达性，并将异常状态标记至管理面板。

**静态站点生成器** 内置 Handlebars 模板引擎，支持将当前链接库一键渲染为响应式 HTML 导航页面，便于部署至 Nginx、GitHub Pages 或对象存储。

**批次版本管理** 每条记录自动记录导入批次号（如 `913/1241`）与时间戳，支持按批次回滚、差异对比与增量发布。

**只读 API 端点** 提供 JSON:API 规范的只读接口，支持按标签、关键词或批次号查询资源列表，方便与其他内部系统集成。

**权限分级预览** 支持 editor、viewer、admin 三级角色，控制链接新增、修改、删除及发布操作的可见范围。

## 应用场景

技术文档团队在每月版本更新前，需将新发现的第三方依赖文档、迁移指南及兼容性公告统一收录至团队知识库。LinkVault Core 的批次版本管理功能允许团队为每个版本创建独立链接集合，并在发布前进行全量健康检查，确保所有引用资源可访问。

企业内部培训部门定期整理面向新员工的入门技术资源包，包含视频教程、交互式 playground 与官方手册。通过多级标签分类引擎，培训负责人可按技能领域（后端、前端、运维）与难度等级（初级、中级、高级）双重维度组织链接，并生成专属静态导航页供学员使用。

开源项目维护者在 issue 或讨论区中频繁收到关于环境搭建、依赖安装与常见编译错误的重复提问。维护者可将已验证有效的社区解答、第三方镜像源地址与排障博文汇总为单一资源列表，通过 LinkVault Core 导出为 Markdown 表格并直接嵌入项目 Wiki，减少重复性回复工作。

## 快速开始

以下命令演示如何获取项目源码、安装依赖并启动开发服务器。

```bash
git clone https://github.com/linkvault/core.git linkvault-core
cd linkvault-core
npm install
npm run dev
```

执行完上述步骤后，访问控制台输出的本地地址（默认为 http://127.0.0.1:5173）即可进入管理界面。首次启动将自动生成示例数据文件 `data/example.yml`，您可在此基础上修改或替换为自己的链接列表。生产环境部署请参考 `docs/deployment.md` 使用 `npm run build` 构建静态输出。

## 安装要求

| 依赖 | 必需 | 说明 |
|------|------|------|
| Node.js 18.x 或 20.x LTS | 是 | 运行时环境，需包含 npm 或 pnpm 包管理器 |
| SQLite 3.35+ | 是 | 内置链接存储、标签映射与批次记录的持久化引擎 |
| Redis 7.x | 否 | 启用会话缓存与探活任务队列时推荐配置，非必需但可提升性能 |
| Nginx 1.22+ | 否 | 生产环境反向代理与静态资源托管参考方案 |
| Git 2.30+ | 是 | 用于版本追踪及贡献者提交记录关联 |
| curl / wget | 是 | 健康检查模块依赖系统工具执行 HTTP 探测 |
| 磁盘空间 200MB | 是 | 存放 SQLite 数据库文件、静态导出产物及临时日志 |
| 内存 512MB 以上 | 是 | 开发模式与中小规模链接管理的最低运行内存建议 |

```
│   │   └── batch.ts        # 批次号生成与版本差异计算

│   ├── render/
│   │   ├── template.ts     # Handlebars 视图引擎封装
│   │   ├── static.ts       # 静态 HTML 与 assets 生成管线

├── data/
│   ├── seeds/              # 初始化示例数据（YAML）
│   └── exports/            # 静态站点输出目录（gitignored）
├── tests/
│   ├── unit/               # 标签解析、批次号格式化单元测试
│   └── integration/        # API 与探活模块集成测试
├── docs/                   # 完整文档站源码（Markdown + VuePress）
├── scripts/                # 构建、发布与数据库迁移辅助脚本
├── .env.example            # 环境变量模板（端口、Redis URL、JWT secret）
├── Dockerfile              # 多阶段构建镜像定义
├── docker-compose.yml      # 本地开发栈（含 Redis 与 SQLite 持久化卷）
├── package.json            # npm 依赖清单与 script 快捷命令
└── README.md               # 本文件
```

## 贡献指南

1. 查阅 `issues` 列表确认当前待办事项或提议的新功能，避免重复工作。对于较大改动（如新增导入格式支持或修改探活策略），建议先创建讨论议题征询维护者意见。

2. Fork 本仓库并创建功能分支，分支命名遵循 `feature/描述` 或 `fix/描述` 格式。所有提交必须通过 ESLint 与 Prettier 检查，并附带必要的单元测试覆盖。

3. 若新增外部依赖，需在 `docs/installation.md` 中同步更新安装要求表格，并确保 Dockerfile 及 docker-compose.yml 中包含对应服务声明。

4. 提交 Pull Request 前请确保本地 `npm run test` 与 `npm run build` 全部通过。PR 描述中应清晰说明改动目的、影响范围及手动测试步骤。

5. 文档类变更（如修正 README 或 API 示例中的错误）可直接提交 PR，无需关联 issue，但需在提交信息中添加 `[docs]` 前缀以便分类。

## 常见问题

**问：导入的 URL 中包含非标准协议头（如 ftp 或 gopher），系统如何处理？**

答：LinkVault Core 的清洗模块默认仅保留 http 与 https 协议条目，并会在导入日志中输出警告。对于非标准协议，系统将其归类为「未分类」并跳过健康检查，但不会丢弃记录，用户可在管理界面手动补充协议头或调整类型标签。

**问：静态站点生成后，能否自定义页面标题、导航栏顺序或页脚信息？**

答：可以。所有页面元数据均从 `data/site-config.yml` 中读取，包括站点名称、描述、导航菜单排序、页脚备案信息等。修改该文件后重新执行 `npm run build` 即可生效，无需修改模板源码。

**问：健康检查探活对目标服务器是否会造成压力？**

答：探活模块默认采用并发数 5 的队列，且每个目标仅发送一次 HEAD 请求，超时设为 3 秒。对于大型列表（超过 1000 条），系统会自动将检查任务分拆为多个批次，每批间隔 10 秒，避免瞬时流量高峰。您可通过环境变量 `PROBE_CONCURRENCY` 与 `PROBE_INTERVAL_MS` 调整这些参数。
