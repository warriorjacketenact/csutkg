# ResourceBridge

ResourceBridge 是一个面向开发者和技术研究人员的结构化外链与资源聚合工具。该项目并不生产内容，而是通过人工筛选与自动化校验相结合的方式，将分散在网络各角落的高质量技术文档、工具链页面、数据集入口以及社区讨论串进行集中编录与分类索引。其核心目标在于解决个人收藏夹混乱、链接失效无法感知、以及跨项目资源复用时的查找效率低下问题。

ResourceBridge 适用于需要频繁查阅外部参考材料的技术决策者、架构师、一线研发人员以及开源贡献者。它既可以被视为一个只读的静态资源导航站点，也可以作为团队内部知识库的外挂参考层，通过开放的数据格式与约定的字段结构，与现有的 CI/CD 流程或文档生成系统进行集成。项目本身不依赖外部数据库，所有资源映射关系以 Markdown 和 YAML 形式存储，确保版本可追溯且易于二次加工。

## 功能概览

**多级标签分类体系** 每个收录的 URL 均附带领域、用途、质量等级三个维度的标签，支持快速筛选与排除。

**自动可用性探测** 通过内置的轻量级探测脚本，定期对已收录链接进行 HEAD 请求检查，标记异常状态并生成变更报告。

**原始数据透传视图** 针对需要保持链接原始形态的场景，提供无修饰的纯文本导出模式，避免浏览器或 Markdown 渲染引擎的自动补全干扰。

**资源关系图谱** 基于链接之间的引用频率与上下文共现关系，生成简单的关联图谱，辅助发现潜在的知识盲区。

**自定义元数据扩展** 每条记录支持附加 notes 字段，用于记录个人心得、注意事项或与之关联的内部工单编号。

**批量导入与去重** 支持从纯文本列表、浏览器书签导出文件以及 CSV 表格中批量导入新链接，并基于 URL 归一化规则自动去重。

**版本化快照** 每次更新均生成变更集，允许回退至任意历史状态，避免误删除或批量替换造成的不可逆损失。

## 应用场景

**技术选型调研** 当团队需要评估多个候选中间件或框架时，可利用本项目的分类标签快速定位官方文档、性能测评文章及社区活跃讨论帖，将分散的参考资料统一拉取至本地进行对比分析。

**文档站外链治理** 对于维护产品文档或 API 参考手册的技术写作人员，可将项目中的链接列表作为文档的外部引用基线，通过探测功能提前发现即将失效的参考链接，并在发布前进行修复或替换。

**新人入职培训路径规划** 导师可以依据资源列表中的难度标签与领域划分，为新人定制阅读清单。通过版本化快照功能，不同批次的培训路径均能被完整保留，便于后续迭代优化。

**个人知识库去中心化备份** 独立开发者或研究人员可将浏览器中散落的数百个技术书签导出后导入本项目，形成一份脱离特定浏览器厂商的、可长期维护的纯文本资产，避免因设备更换或账户问题导致收藏丢失。

## 快速开始

以下命令演示了如何将 ResourceBridge 仓库克隆至本地，安装基础依赖，并执行一次完整的资源索引构建。

```bash
# 克隆项目仓库至当前目录下的 resource-bridge 文件夹
git clone https://github.com/resource-bridge/resource-bridge.git

# 进入项目根目录
cd resource-bridge

# 安装 Python 3.9+ 虚拟环境及依赖包（包含 requests、pyyaml、click）
python -m venv venv
source venv/bin/activate  # Windows 系统请使用 venv\Scripts\activate
pip install -r requirements.txt

# 执行索引构建脚本，生成资源汇总表
python build_index.py --input ./data/raw_links.txt --output ./dist/index.md
```

## 贡献指南

ResourceBridge 的核心数据源完全开放，欢迎以链接推荐、标签修正或代码改进等形式参与共建。请遵循以下标准流程，以确保变更的平滑合并。

**一、提交新的链接推荐**  
通过 Fork 仓库，在 data/raw_links.txt 文件末尾追加新链接，每条链接占据独立一行。同时，在对应的 data/metadata.yaml 文件中为新链接补充至少两个分类标签与一段简要说明。提交 Pull Request 时，请在描述中标注推荐理由或来源上下文。

**二、修正失效链接或更新标签**  
若探测脚本标记某链接为不可达，请先访问原地址确认是否临时波动。若确认永久失效，则从 raw_links.txt 中移除该行，并在 metadata.yaml 中将其状态字段置为 deprecated。对于标签体系，如发现分类错误，可直接修改 YAML 文件中的 tags 列表，并在 PR 说明中给出修改依据。

**三、改进探测脚本或构建工具**  
代码层面的改动需确保现有单元测试通过，并为新增功能提供对应的测试用例。请使用 mypy 进行静态类型检查，并遵循 PEP 8 编码风格。所有 PR 均需通过 CI 中的 lint 与 test 两道检查门禁。

**四、完善文档与示例**  
文档修正或扩充同样重要。若在 README 或用户指南中发现表述不清或缺少示例的情况，可直接修改对应的 Markdown 文件。对于涉及操作步骤的变更，建议附带操作前后的终端输出对比。

**五、问题反馈与讨论**  
日常问题或改进建议优先通过 GitHub Issues 进行追踪。在新建 Issue 时，请选择对应的模板类型（Bug Report / Feature Request / Link Suggestion），并尽可能提供复现步骤或参考链接。

## 常见问题

**问：为什么所有链接都指向同一个主域名，而不是分散的技术站点？**  
本项目作为外链汇总层，不对源站点的内容属性做主观筛选。当前批次资源集中来自于特定内容平台，这属于正常的批次分布现象。我们鼓励用户通过 metadata.yaml 中的标签进行二次分类，或使用项目提供的分组导出功能按域名进行过滤。后续批次会包含更广泛的域名来源。

**问：如何批量导入我自己的书签文件？**  
ResourceBridge 提供了 convert_bookmarks 子命令，可接收 Netscape 格式的 HTML 书签导出文件（大多数浏览器支持此格式）。转换工具会尝试解析书签文件夹结构，并将其映射为标签体系。执行命令示例：python build_index.py convert --input bookmarks.html --output ./data/raw_links.txt

**问：探测脚本是否会对我所提交的链接造成访问压力？**  
探测模块采用单线程间歇请求策略，默认并发数限制为 3，且每个请求间隔不低于 1 秒。超时时间设定为 5 秒，仅发送 HEAD 请求，不会下载响应体内容，对目标服务器的负载影响微乎其微。用户亦可手动调整 probe_config.yaml 中的并发数和超时阈值。
