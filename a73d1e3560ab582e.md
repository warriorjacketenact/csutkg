# LinkVault Core

LinkVault Core 是一个面向技术团队与内容运营者的轻量级外链资产管理与监控系统。该项目并非简单的书签收藏工具，而是针对批量 URL 的健康状态巡检、元数据抽取与访问链路追踪设计的自动化处理框架。项目主要解决大规模外链资源在内容聚合、安全审查与可用性验证场景下的管理难题，适用于需要对海量分散链接进行集中化状态评估的技术运维人员、安全审计工程师以及知识库维护者。

## 功能概览

**批量链接健康检查**：支持对数千个 URL 进行并发探活，自动识别 HTTP 状态码异常、证书过期与响应超时等故障类型，并生成结构化诊断报告。

**元数据智能抽取**：自动解析目标页面的标题、描述、关键词以及 Open Graph 协议信息，将非结构化的网页内容转化为可检索的字段数据。

**自定义标签归类**：允许用户为每个链接创建多维度标签体系，支持基于标签组合的快速筛选与统计，便于构建垂直领域的资源图谱。

**变更差异比对**：通过定时任务记录链接状态的历史快照，对比两次巡检周期内的状态变化，以增量视图呈现新增失效或恢复的链接。

**访问链路追踪**：记录链接的中间重定向路径，检测最终落地页与原始 URL 之间的域名偏离度，识别潜在的反向代理或劫持行为。

**数据导出集成**：支持将巡检结果以 JSON、CSV 及 Markdown 表格格式导出，便于对接上游监控告警系统或下游数据分析流水线。

**轻量级任务编排**：内置基于时间轮的调度引擎，支持按小时、每日或每周粒度执行自动化巡检任务，并支持失败重试与超时熔断策略。

## 应用场景

**内容聚合平台链接巡检**：面向资讯聚合或资源导航类网站，使用 LinkVault Core 每日定时扫描收录的全部外链，自动标记响应码非 200 的异常链接，并将失效链接列表推送至管理员邮箱，保证对外输出内容的可用性。

**开源项目文档链接受控审查**：在开源项目的 README 或 Wiki 页面中常包含大量外部参考链接。维护团队在版本发布前运行 LinkVault Core 对即将上线文档中的所有 URL 进行可达性验证，确保用户访问时不会遭遇死链或被重定向至恶意站点。

**企业知识库资源归档验证**：企业内部知识库沉淀了大量历史技术方案与外部参考资料。利用本项目的差异比对功能，按月扫描归档链接状态，识别大量已迁移或下线的外部资源，辅助知识库管理员决定是否保留或更新条目。

**安全研究数据源筛选**：安全研究人员收集了大量威胁情报源或 C2 域名列表。通过 LinkVault Core 的元数据抽取与重定向追踪，过滤掉无法解析或频繁变更域名的无效源，提升威胁狩猎的数据质量和分析效率。

## 快速开始

以下指令适用于 Linux/macOS 系统，假设已安装 Git 与 Python 3.9 以上版本。

```bash
# 克隆项目仓库至本地
git clone https://github.com/example/linkvault-core.git
cd linkvault-core

# 安装依赖项（推荐使用虚拟环境）
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 执行基础巡检命令，扫描 links.txt 文件中的全部 URL
python cli.py scan --input links.txt --output report.json
```

若需要启动周期调度服务，请将 `cron.yaml` 中配置的巡检任务注册至系统定时任务，或使用内置 `scheduler` 命令直接运行：

```bash
python cli.py scheduler --config config/scheduler.yaml
```

## 项目结构

LinkVault Core 采用分层模块化设计，源码目录遵循约定优于配置原则。以下为项目主要目录及核心文件注释：

```
linkvault-core/
├── cli.py                       # 命令行入口，注册 scan/scheduler/export 子命令
├── config/
│   ├── scheduler.yaml           # 调度任务定义（频率、目标列表、重试上限）
│   ├── user_agents.yaml         # 请求头池，用于轮换 User-Agent 降低封禁风险
│   └── log_config.yaml          # 日志级别、滚动策略与输出格式配置
├── core/
│   ├── __init__.py
│   ├── scanner.py               # 核心扫描引擎，管理异步会话与请求分发
│   ├── parser.py                # HTML 元数据抽取模块，基于 BeautifulSoup
│   ├── tracker.py               # 重定向链追踪与最终 URL 解析器
│   └── health.py                # 健康状态评估器，判定各类异常码与超时
├── storage/
│   ├── __init__.py
│   ├── snapshot.py              # 状态快照写入与读取，支持 JSON 序列化
│   ├── differ.py                # 两次快照的差异对比生成器
│   └── exports.py               # 输出格式适配器（CSV, JSON, Markdown）
├── scheduler/
│   ├── __init__.py
│   ├── engine.py                # 时间轮调度引擎，触发周期性任务
│   ├── retry.py                 # 可重入的失败重试与退避策略实现
│   └── lock.py                  # 基于文件锁的防重复执行保护
├── tests/
```

## 常见问题

**问：执行扫描任务时出现大量 SSL 证书验证错误，应如何处理？**

答：目标服务器若使用自签名证书或已过期证书，默认的 SSL 上下文会拒绝连接。解决方法为在配置文件中将 `ssl_verify` 参数置为 `false`，或通过命令行 `--no-ssl-verify` 开关临时禁用验证。需要明确此操作会降低安全性，仅建议在内部网络或测试环境中使用。

**问：程序运行期间内存占用持续增长，最终导致进程被系统终止，如何排查？**

答：内存增长通常源于异步请求的响应体未正确释放或快照对象累积过多。建议检查 `scanner.py` 中的会话超时设置，确保每次请求后均显式关闭响应对象。同时调整 `storage/snapshot.py` 中的最大快照保留数量，通过 `max_snapshots` 配置项限制历史记录存储上限。若问题仍然存在，可开启 `gc` 模块的调试输出以定位引用循环。

**问：调度器按配置执行巡检，但日志中显示某些任务被跳过，原因是什么？**

答：调度器内置了防重叠执行保护，若上一轮巡检尚未完成而下一轮触发时间已到，新任务将自动跳过并在日志中记录 `Task skipped due to previous run still in progress`。此机制用于避免资源竞争。若希望强制并发执行，请在 `scheduler.yaml` 中设置 `allow_concurrent: true`，但需注意同时运行多个扫描任务可能导致网络端口耗尽或触发目标方的限流策略。
