# LinkVault

LinkVault 是一个面向技术社区与内容创作者的轻量级外链资源聚合与导航系统。该项目定位于为开发者、研究员以及运维人员提供结构化的外部链接收藏、分类、检索与监控能力，帮助用户从零散的书签与临时记录中解脱出来，建立可持续维护的知识资产库。

LinkVault 并非传统意义上的简单链接列表，而是一个具备元数据扩展、健康状态检测、访问统计与标签体系的最小化资源管理后端。系统设计强调简洁部署、低依赖与高可读性，适合个人使用、小团队协作或作为更复杂知识管理系统的链接预处理层。

## 功能概览

- **链接结构化入库**：支持手动录入与批量导入，每条链接可附带标题、描述、标签、所属类别、添加时间与最后访问时间。

- **自动健康检查**：定期对存储的链接发起 HEAD 请求，检测资源是否可访问、响应状态码是否正常、重定向链路是否合法。

- **标签与全文检索**：基于标签与描述字段提供简单但有效的搜索能力，帮助用户在大量外链中快速定位所需内容。

- **访问统计与热度排序**：记录每条链接的点击次数与最近访问时间，支持按热度、添加时间、更新时间多种方式排序展示。

- **数据导入导出**：支持 JSON 与 CSV 格式的批量导入导出，便于与其他工具（如浏览器书签、Notion、Airtable）进行数据交换。

- **只读 API 接口**：提供 RESTful 风格的查询接口，允许第三方工具或脚本以编程方式获取链接数据，便于集成到现有工作流中。

## 应用场景

- **技术文档与博客外部链接管理**：技术写作者在撰写文章或维护文档时，常需要引用大量外部资料。LinkVault 可集中存储这些引用链接，并在发布前自动检查链接有效性，避免死链影响读者体验。

- **开源项目资源导航页**：开源项目维护者可使用 LinkVault 构建项目相关的生态资源导航页面，将周边工具、教程、社区论坛、镜像站等外链统一整理，方便新用户快速上手。

- **研究资料与论文参考文献追踪**：研究人员在文献调研阶段积累大量网络资料。LinkVault 的标签与检索能力可协助按主题、作者、发布时间等维度对资料进行粗粒度分类，提高文献回顾效率。

- **运维监控仪表盘辅助数据源**：运维人员可将内部监控面板、日志系统、报警管理平台的入口链接统一纳入 LinkVault，配合健康检查功能快速发现服务入口不可用的情况。

## 快速开始

以下步骤适用于 Linux / macOS / Windows WSL 环境，依赖 Git、Node.js 18+ 与 npm。

```bash
# 克隆项目仓库
git clone https://github.com/your-org/linkvault.git
cd linkvault

# 安装依赖
npm install

# 初始化配置并启动开发服务器
cp .env.example .env
npm run migrate
npm run dev
```

启动成功后，访问 http://localhost:3000 即可使用 Web 界面。默认管理员账户为 admin@linkvault.local，初始密码在首次启动时打印于终端日志中。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | 18.x 或 20.x LTS | 运行时环境，推荐使用官方预编译二进制或 nvm 管理 |
| npm | 9.x 或 10.x | 包管理器，随 Node.js 一同安装 |
| SQLite3 | 3.35+（内嵌于 better-sqlite3） | 默认嵌入式数据库，无需额外安装；生产环境可切换至 PostgreSQL 14+ |
| Git | 2.30+ | 用于克隆仓库及版本管理 |
| 系统内存 | 最低 512MB，推荐 1GB | 运行内存要求，实际占用取决于链接数量与检查任务并发数 |
| 磁盘空间 | 最低 100MB | 用于存放数据库文件、日志及临时缓存，实际需求随链接数量线性增长 |
| 操作系统 | Linux / macOS / Windows 10+ (WSL2) | 支持主流操作系统，Windows 原生环境需额外配置 node-gyp 编译工具链 |

## 项目结构

```
linkvault/
├── src/
│   ├── core/                     # 核心业务逻辑模块
│   │   ├── link.service.js       # 链接增删改查与标签管理
│   │   ├── health.check.js       # 定时健康检查任务调度与状态更新
│   │   └── stats.collector.js    # 点击统计与热度计算
│   ├── api/                      # HTTP 路由与控制器
│   │   ├── routes.js             # 路由注册与中间件挂载
│   │   └── handlers/             # 各端点请求处理函数
│   ├── db/                       # 数据库相关
│   │   ├── schema.sql            # 表结构定义（SQLite/PostgreSQL 兼容）
│   │   ├── migrations/           # 版本迁移脚本
│   │   └── seed.js               # 初始测试数据填充
│   ├── lib/                      # 通用工具库
│   │   ├── logger.js             # 日志封装（支持文件与控制台输出）
│   │   ├── validator.js          # URL 格式校验与规范化
│   │   └── fetcher.js            # 封装 axios 的健康探测客户端
│   └── web/                      # Web UI 静态资源
│       ├── index.html            # 单页应用入口
│       ├── assets/               # 样式表、前端脚本、图标
│       └── components/           # 可复用 UI 组件模板
├── tests/                        # 单元测试与集成测试
│   ├── unit/                     # 各模块独立测试用例
│   └── integration/              # 端到端 API 测试与数据库交互测试
├── config/                       # 环境配置文件
│   ├── default.json              # 默认配置（端口、检查间隔、分页大小）
│   ├── production.json           # 生产环境覆盖配置
│   └── development.json          # 开发环境覆盖配置
├── scripts/                      # 运维与辅助脚本

├── docker-compose.yml            # 本地开发容器编排（含 Postgres 与 Redis 可选）
└── README.md                     # 本文件
```

## 贡献指南

1. 阅读贡献者手册（docs/contributing.md），了解代码风格、测试要求与提交规范。项目使用 ESLint + Prettier 统一格式，Commit Message 遵循 Conventional Commits 标准。

2. 从 issues 列表中选择一个未被认领的任务，或提交新的 issue 描述你发现的问题或建议的新功能。建议先通过 issue 与维护者沟通，避免重复工作或不匹配的设计方向。

3. Fork 本仓库，在本地新建功能分支（命名格式为 feat/xxx 或 fix/xxx），完成代码实现后确保所有现有测试通过，并为新增代码编写相应的单元测试或集成测试。

4. 提交 Pull Request 到主仓库的 develop 分支，PR 描述中需写明解决的问题、实现方案以及可能的破坏性变更。PR 需要通过 CI 检查（包括 lint、test 与构建）。

5. 代码审查通过后，由维护者合并至 develop 分支，并定期随版本发布合并至 main 分支。所有贡献者将自动列入 CONTRIBUTORS 文件。

## 常见问题

**Q: LinkVault 能否直接导入浏览器书签导出文件？**

A: 当前版本不支持直接解析浏览器书签 HTML 或 JSON 格式。但你可以将书签导出为 CSV 文件（包含 URL 与标题列），然后通过 scripts/import-csv.js 脚本进行导入。未来版本计划增加对 Chrome 和 Firefox 书签导出格式的原生支持。

**Q: 健康检查会对外部目标造成压力吗？**

A: LinkVault 仅发起 HEAD 请求，不下载响应体，且每个检查周期内对同一域名下的多个链接自动增加 200ms 间隔，避免触发目标服务器的限流机制。默认检查周期为每 24 小时一次，可在配置文件中调整间隔或完全关闭自动检查。

**Q: 生产环境下如何切换至 PostgreSQL？**

A: 在 .env 文件中设置 DATABASE_URL 为 PostgreSQL 连接字符串，并安装 pg 驱动（npm install pg）。项目会在启动时自动检测数据库类型并加载对应的适配器。注意 PostgreSQL 需要使用 14 及以上版本，且需提前创建好数据库实例。
