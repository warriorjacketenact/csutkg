# WebArchive Navigator

WebArchive Navigator 是一个面向技术研究、内容考古与数据挖掘场景的轻量级外链资源导航系统。该项目定位于对分散在网络各处的历史页面快照、技术文档存档、论坛讨论记录以及代码仓库镜像进行集中索引与分类管理，帮助开发者、研究人员与内容策展人高效定位特定时间节点的技术资料。

项目本身不存储任何页面内容，仅维护一个结构化的外链索引库。每个条目包含原始 URL、采集时间、内容类型标签与简要描述。通过本系统提供的分类视图与过滤机制，用户可以从数以千计的链接中快速筛选出与当前研究主题相关的历史资源，避免在通用搜索引擎中反复尝试不同关键词组合所带来的低效体验。

WebArchive Navigator 的目标用户包括技术文档维护者、开源社区贡献者、网络安全分析师、学术研究者以及任何需要追溯互联网内容演变过程的个人或团队。项目采用纯静态架构，索引数据以 JSON 格式存储，可被托管在任意 Web 服务器或对象存储服务上，无需数据库或后端运行时依赖。

## 功能概览

- **多维度分类索引**：每个资源条目支持归属多个分类标签，如"技术文档"、"社区讨论"、"代码示例"、"安全公告"等，用户可按标签快速缩小浏览范围。

- **时间范围过滤器**：基于采集时间戳提供滑动时间窗口过滤功能，支持按年、月、日粒度筛选特定时段的外链资源，适用于追溯技术演进过程。

- **关键词检索与高亮**：在标题、描述和自定义标签字段中执行子串匹配检索，匹配项在结果列表中予以高亮显示，便于快速定位相关条目。

- **外链健康状态检测**：系统周期性对已收录的 URL 执行可达性探测，并在索引列表中标记异常状态（如返回 4xx/5xx 状态码或连接超时），辅助用户判断资源可用性。

- **批量导入与导出**：支持以 CSV 或 JSON Lines 格式批量导入外链记录，亦可将筛选结果导出为结构化数据文件，便于二次分析或迁移至其他工具。

- **自定义标签体系**：用户可根据自身研究需要为条目添加自定义标签，标签数据保存在客户端本地存储中，实现个性化分类而不影响公共索引。

- **响应式布局与暗色主题**：界面适配桌面端与移动端浏览器，并提供暗色与亮色两套视觉主题，满足不同使用环境下的阅读舒适度需求。

## 应用场景

**技术文档版本追溯**  
当某开源项目的官方文档发生重大更新后，研究人员可能需要查阅旧版本中的 API 说明或配置参数。通过 WebArchive Navigator 中收录的历史文档链接，可快速访问存档页面，对比不同版本间的差异。

**安全漏洞分析与溯源**  
安全团队在分析历史漏洞时，常需查阅过往的安全公告、论坛讨论或补丁提交记录。本系统提供的按时间范围过滤功能可帮助分析师精准定位漏洞披露时间窗口内的相关外链资源。

**开源项目演进研究**  
社区管理者或技术决策者可通过本系统检索某技术栈在不同年份的社区讨论热帖、教程文章和代码仓库快照，从宏观层面观察该技术的采纳曲线与关注度变化。

**内容策展与知识库构建**  
内容策展人在整理特定领域的学习路径或参考资料时，可利用本系统的分类索引和标签功能快速筛选高质量外链，并将筛选结果导出为结构化清单，用于构建对外公开的知识库页面。

## 快速开始

以下步骤适用于在本地环境中启动 WebArchive Navigator 的开发实例或生产静态构建。

```bash
# 克隆代码仓库
git clone https://github.com/webarchive-navigator/webarchive-navigator.git

# 进入项目根目录
cd webarchive-navigator

# 安装项目依赖（使用 npm）
npm install

# 启动开发服务器，默认监听端口 3000
npm run dev
```

执行上述命令后，在浏览器中访问控制台输出的本地地址（通常为 http://localhost:3000）即可开始使用。若需构建用于生产环境部署的静态文件，请执行 `npm run build`，构建产物将输出至 `dist` 目录。

## 贡献指南

1. 在 GitHub Issues 中查找或创建与您要解决的问题或新增功能对应的议题，等待维护者回复确认后再开始编码工作，避免重复劳动或不必要的 Pull Request 被关闭。

2. 从 `main` 分支创建新的特性分支，分支命名遵循 `feature/描述` 或 `fix/描述` 格式（例如 `feature/tag-export`）。确保本地开发环境已通过 `npm install` 安装全部依赖。

3. 编写代码时遵循项目已配置的 ESLint 与 Prettier 规则。提交前执行 `npm run lint` 与 `npm run test` 确保无风格错误且所有测试用例通过。若新增功能，请同步添加对应的单元测试。

4. 提交 Commit 时使用语义化消息格式：`<类型>: <简短描述>`，类型可选 `feat`、`fix`、`docs`、`style`、`refactor`、`test`、`chore`。描述使用英文祈使句，首字母小写，不超过 72 个字符。

5. 推送分支后通过 GitHub 界面创建 Pull Request，目标分支指向 `main`。PR 描述中需引用关联的 Issue 编号，并逐项列出本次变更的主要内容、测试覆盖情况以及可能影响现有功能的兼容性说明。等待至少一位维护者 Code Review 通过后即可合并。

## 常见问题

**Q：系统是否会自动抓取并存储每个外链指向的页面内容？**  
A：不会。WebArchive Navigator 仅存储 URL 地址及其元数据（标题、标签、采集时间等），不代理、不缓存、不抓取任何第三方页面内容。外链健康检测仅通过发送 HEAD 或 GET 请求验证目标资源是否可访问，不解析或存储响应体。用户点击外链时将直接跳转至原始目标服务器，所有内容版权与可用性均归属原始提供方。

**Q：索引数据由谁维护？我能否自行添加或修改条目？**  
A：本项目公共索引由核心维护团队定期从可信数据源导入并审核后发布。普通用户可通过本地自定义标签功能对条目进行个人化分类，但无法直接修改公共索引中的元数据。若您发现某条链接失效或元数据存在明显错误，请在 GitHub Issues 中提交反馈，维护团队将在后续批次更新中处理。对于有批量导入需求的机构用户，建议使用项目提供的 CSV 导入功能创建私有索引视图。

**Q：项目是否支持多用户协同与权限管理？**  
A：当前版本定位为个人或小团队使用的轻量级工具，不内置用户系统与权限控制。如需在多用户环境中共享索引数据，建议将构建后的静态文件部署在团队内部服务器上，并配合 HTTP 基本认证或反向代理的访问控制策略。多用户协同编辑功能已在后续迭代规划中，具体发布时间待定。
