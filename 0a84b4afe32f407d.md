# LinkVault Resource Aggregator

LinkVault 是一个面向开发者、技术研究人员与内容策展人的高性能外链资源归集与导航系统。本项目不生产原始内容，而是通过结构化组织、标签化索引与可用性监控，将散落在互联网各处的优质技术文档、工具站点、参考手册与社区讨论入口整合为统一访问入口，解决个人书签分散、团队知识断层与资源失效不可知三大痛点。

项目定位于中大型研发团队的技术基础设施建设单元，亦适用于独立开发者构建个人知识工作台。LinkVault 不依赖特定云厂商，可部署于内网或公网环境，提供只读镜像、定期健康检查与自定义分类视图，确保团队始终从稳定、可溯源的路径获取外部技术资源。

## 功能概览

**多源资源导入** 支持从 CSV、Markdown 列表及浏览器书签 HTML 批量导入原始 URL，自动解析标题与元数据，降低初始化成本。

**分类标签体系** 允许用户为每条链接赋予多级标签（如 language/python、cloud/aws、database/postgresql），并基于标签组合生成动态聚合视图。

**可用性拨测** 内置轻量级 HTTP 探测器，按可配置周期（默认 24 小时）对存储的 URL 执行 HEAD/GET 请求，标记异常状态并记录响应时间趋势。

**自定义导航页** 提供模板引擎，可将指定标签集合渲染为静态门户页面，适合作为团队浏览器的默认新标签页或内部知识库入口。

**访问统计与热度排序** 记录各资源链接的点击次数与最后访问时间，支持按热度、新鲜度或字母序排列输出。

**API 优先设计** 所有核心能力通过 RESTful API 暴露，便于与现有自动化流水线（如 CI/CD 文档生成、监控告警）集成。

**全文检索** 基于资源标题、描述与自定义备注构建倒排索引，支持模糊匹配与前缀建议，快速定位目标链接。

**角色权限控制** 提供管理员、编辑者、只读用户三级权限，适用于团队协作场景，敏感资源可标记为内部可见。

## 应用场景

研发团队文档沉淀：技术 Leader 可将团队长期积累的外部参考链接导入 LinkVault，按项目或技术栈分类，新成员入职时直接访问统一导航页，替代口口相传或零散邮件。

个人知识工作流：独立开发者或研究员在日常浏览中发现有用资源，一键提交至 LinkVault 的“待整理”队列，每周集中打标与归档，形成个人外脑索引系统。

离线文档镜像代理：对于网络受限的内网环境，LinkVault 可配合离线下载服务，将频繁引用的外部文档缓存至本地对象存储，并通过内部 URL 重写实现无缝访问切换。

技术社区资源聚合：开源社区运营者可使用 LinkVault 收集生态内周边工具、教程与案例，生成公开的生态资源门户，降低社区成员寻找信息的门槛。

## 快速开始

以下指令适用于 Linux / macOS 环境，Windows 用户可通过 WSL 或 Git Bash 执行。

```bash
# 克隆项目仓库
git clone https://github.com/linkvault/linkvault-core.git
cd linkvault-core

# 安装依赖（使用 Python 3.9+ 虚拟环境）
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 初始化数据库并启动开发服务（默认监听 8000 端口）
python manage.py migrate
python manage.py runserver 0.0.0.0:8000
```

生产环境部署请参考 `docs/deployment.md`，推荐使用 Gunicorn + Nginx 组合。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---------|---------|------|
| Python | 3.9 至 3.11 | 核心运行环境，3.12 暂未进行完整兼容测试 |
| PostgreSQL | 13.0 及以上 | 主数据库，用于存储资源元数据、标签与访问日志 |
| Redis | 6.0 及以上 | 缓存与临时任务队列后端，用于拨测状态暂存 |
| Node.js | 16.0 及以上 | 仅用于前端资源构建（管理界面），后端运行可不安装 |
| Nginx | 1.18 及以上 | 生产环境推荐反向代理，静态文件服务与负载均衡 |
| 磁盘空间 | 至少 1 GB | 用于存储日志、缓存与可能的本地页面渲染输出 |

## 项目结构

```
linkvault-core/
├── cmd/                            # 命令行入口与辅助工具
│   ├── server/                     # 主服务启动入口 (server.go)
│   └── probe/                      # 独立拨测工具 (probe.go)
├── internal/                       # 内部包，不对外暴露
│   ├── storage/                    # 数据库抽象层 (postgres.go, interface.go)
│   ├── cache/                      # Redis 缓存操作 (redis.go, memory.go)
│   ├── probe/                      # HTTP 拨测引擎 (checker.go, scheduler.go)
│   ├── index/                      # 全文索引构建 (tokenizer.go, inverted.go)
│   └── auth/                       # 角色权限中间件 (jwt.go, policy.go)
├── pkg/                            # 可被外部导入的公共库
│   ├── api/                        # RESTful API 路由定义 (v1/)
│   └── model/                      # 数据实体与 DTO (resource.go, tag.go)
├── web/                            # 前端管理界面 (React + TypeScript)
│   ├── src/                        # 组件与页面源码
│   └── dist/                       # 构建输出目录 (gitignore)
├── configs/                        # 环境配置文件 (dev.yaml, prod.yaml)
├── migrations/                     # 数据库迁移脚本 (sql 文件按版本号)
├── docs/                           # 完整文档目录
│   ├── quickstart.md
│   ├── configuration.md
│   ├── api/                        # OpenAPI 规范与示例
│   └── operations/                 # 运维相关指南
```

## 常见问题

**Q: 拨测功能会对目标站点造成压力吗？**  
A: 拨测默认采用 HEAD 请求，仅获取响应头部，不下载完整页面内容。频率可配置，默认每个 URL 每 24 小时探测一次，且支持设置随机延迟分散请求，对绝大多数站点影响可忽略不计。

**Q: 是否支持私有化部署与离线环境？**  
A: 完全支持。LinkVault 所有依赖（PostgreSQL、Redis）均可内网自建，前端资源可预先构建为静态文件嵌入二进制或挂载至 Nginx。离线环境下拨测功能可选择性关闭，仅作为静态导航使用。

**Q: 如何从其他书签管理工具迁移数据？**  
A: 项目提供 `scripts/import_bookmarks.py` 脚本，支持 Chrome 书签 HTML、Raindrop.io CSV 导出及通用 URL 列表文件格式。导入时可保留原始标签结构或映射为新标签体系，具体用法请参考 `docs/migration.md`。
