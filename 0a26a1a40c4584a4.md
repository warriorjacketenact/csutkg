# LinkVault 技术资源索引系统

LinkVault 是一个面向开发者、技术研究人员与开源爱好者的结构化外链资源汇总平台。项目定位于对散落于互联网各处的优质技术文章、工具文档、社区讨论与工程实践进行系统性采集、分类与长期保存，解决个人书签难以检索、团队知识碎片化、优质内容易失效等痛点。通过统一的元数据描述与版本化跟踪机制，LinkVault 使技术团队能够以可复现的方式管理外部知识依赖，并将外链资源作为项目文档体系的一等公民予以维护。

目标用户包括需要维护技术选型文档的架构师、编写教程与博客的开发者、进行文献调研的研究人员，以及希望建立团队共享知识库的工程管理者。项目不依赖任何特定云服务，可完全离线运行，所有资源元数据以纯文本形式存储，便于纳入现有版本控制系统进行协同编辑。

## 功能概览

**批量链接导入**：支持从 CSV、JSON 及 Markdown 表格批量导入外链，自动解析 URL、标题与上下文标签，减少手工录入成本。

**元数据增强引擎**：对每条链接自动补充来源域名、响应状态码、内容摘要哈希值及最后访问时间，为链接质量评估提供基础数据。

**分类与标签体系**：允许为每条资源赋予多个自定义标签，并支持按标签树进行层级浏览，适应不同团队的知识组织习惯。

**全文检索与过滤**：基于标题、描述、标签、域名及导入批次的多条件组合检索，支持正则表达式匹配与结果排序。

**状态监控与告警**：定期对已收录链接进行可用性检查，自动标记失效链接并生成变更报告，协助维护资源健康度。

**版本化导出**：可将指定标签或批次下的链接列表导出为 Markdown、HTML 或结构化 JSON 格式，便于嵌入项目文档或生成周报。

**权限与协同**：提供基于角色的访问控制，支持多用户同时编辑元数据，记录每次修改的操作人与时间戳。

**开放 API 接口**：提供 RESTful API 供第三方工具查询与更新资源，支持与现有 CI/CD 流水线或文档生成工具集成。

## 应用场景

技术选型与方案评审：团队在进行中间件选型或架构评审时，可将候选产品文档、性能对比报告与社区讨论链接统一收录至 LinkVault，并标注决策结论与适用条件，形成可追溯的决策依据库。

教程编写与学习路径设计：技术培训负责人可为不同岗位收集系列化学习资料，按难度层级与技能领域组织标签，为新员工或转岗工程师提供清晰的学习导航，避免信息过载。

项目文档外链治理：开源项目维护者可将依赖的第三方库主页、API 参考手册、问题跟踪地址集中管理，当依赖升级或迁移时，通过状态监控功能快速定位受影响的链接并批量更新。

故障排查与运维知识沉淀：运维团队可将历史故障的根因分析博客、修复补丁链接、监控面板地址按故障类型归档，后续发生类似问题时能快速检索既往处理方案，缩短平均修复时间。

## 快速开始

以下命令演示如何在本地环境中获取 LinkVault 源码、安装依赖并启动开发服务。

```bash
git clone https://github.com/linkvault/linkvault-core.git
cd linkvault-core
pip install -r requirements.txt
python manage.py migrate
python manage.py runserver --host 0.0.0.0 --port 8080
```

首次启动后，可通过浏览器访问 http://localhost:8080 进入初始化向导，完成管理员账户创建与基础配置。生产环境部署请参考 `deploy` 目录下的容器化与 systemd 配置示例。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.9 至 3.12 | 核心运行环境，推荐使用 3.11 版本以获取最佳性能 |
| PostgreSQL | 13.0 及以上 | 主数据库，用于存储链接元数据、标签与用户信息 |
| Redis | 6.2 及以上 | 缓存与任务队列后端，用于异步状态监控与告警调度 |
| Node.js | 18.0 及以上 | 仅用于前端资源构建，生产环境可仅使用预编译静态文件 |
| Nginx | 1.20 及以上 | 可选，用于反向代理与静态资源缓存加速 |
| Git | 2.30 及以上 | 用于版本化导出功能与元数据仓库同步 |
| OpenSSL | 1.1.1 及以上 | 用于 API 签名与安全传输层配置 |
| Supervisor | 4.2 及以上 | 可选，用于守护 Celery 工作进程与定时任务 |

## 项目结构

```
linkvault-core/
├── cmd/                            # 命令行入口与工具脚本
│   ├── server/                     # 服务启动入口 (main.go)
│   ├── migrate/                    # 数据库迁移工具
│   └── inspector/                  # 链接状态检查命令行工具
├── internal/                       # 内部包，不对外暴露
│   ├── core/                       # 核心业务逻辑（链接管理、标签、批次）
│   │   ├── importer/               # 导入器实现（CSV, JSON, Markdown）
│   │   ├── exporter/               # 导出器实现（Markdown, HTML, JSON）
│   │   └── validator/              # URL 校验与归一化
│   ├── storage/                    # 存储层抽象（PostgreSQL 驱动）
│   │   ├── migrations/             # 数据库版本迁移脚本
│   │   └── queries/                # 预编译 SQL 查询模板
│   ├── cache/                      # 缓存层（Redis 驱动）
│   ├── scheduler/                  # 任务调度（周期监控、报告生成）
│   └── api/                        # RESTful API 路由与中间件
│       ├── handlers/               # 请求处理器
│       ├── middleware/             # 认证、日志、限流中间件
│       └── responses/              # 统一响应格式与错误码
├── pkg/                            # 可复用的公共库
│   ├── config/                     # 配置解析（YAML + 环境变量）
│   ├── logger/                     # 结构化日志封装（基于 slog）
```

## 贡献指南

贡献者需遵循以下流程以确保项目质量与协作效率。

第一，在 GitHub 上 fork 本项目至个人账户，并 clone 到本地开发环境。建议使用分支命名规范 `feature/功能简述` 或 `fix/问题编号`，避免在主分支直接提交。

第二，运行 `make test` 确保本地环境已通过全部单元测试与集成测试。新增功能或修复缺陷时，需同步编写对应的测试用例，测试覆盖率不得低于百分之八十。

第三，提交代码前执行 `make lint` 进行静态代码检查与格式化，确保代码风格符合项目既定的 Go 编码规范。提交信息应遵循约定式提交格式，包含类型、范围与描述，例如 `feat(importer): add support for YAML batch import`。

第四，向主仓库的 `dev` 分支发起 Pull Request，并在描述中清晰说明变更目的、影响范围及测试情况。至少需获得一位核心维护者的 Approve 方可合并。对于重大功能变更或架构调整，需先通过 Issue 发起设计讨论，达成共识后再进行开发。

第五，文档类变更（包括 README、用户指南、API 文档）需与代码变更同步更新。所有新增配置项或 API 端点必须在对应文档中提供完整说明与示例。

## 常见问题

问：导入大量链接时页面响应缓慢，应如何优化？

答：当单批次导入链接超过 5000 条时，建议使用异步导入模式。在导入请求头中添加 `X-Async: true`，系统将返回任务 ID 并转入后台处理。您可通过 `/api/tasks/{task_id}` 轮询进度。同时，请确保 PostgreSQL 的 `work_mem` 和 `maintenance_work_mem` 参数已根据硬件调整，推荐设置为 256MB 以上以加速批量写入。

问：如何迁移 LinkVault 数据至另一台服务器？

答：数据迁移需同时处理 PostgreSQL 数据库内容与 Redis 缓存中的任务队列。推荐使用 `pg_dump` 导出数据库快照，并在目标服务器使用 `pg_restore` 恢复。Redis 数据可通过 `SAVE` 命令生成 RDB 文件后复制至新实例。迁移完成后，请运行 `./inspector check --all` 对所有链接进行状态重新校验，确保监控数据与当前网络环境一致。

问：能否自定义链接状态监控的间隔与重试策略？

答：可以。在 `config.yaml` 中的 `monitor` 段可配置 `interval_minutes` 控制检查频率，`retry_times` 与 `retry_delay_seconds` 控制失败重试策略。此外，支持为不同域名设置独立的超时时间，通过 `domain_overrides` 映射表覆盖全局默认值，避免因个别慢速站点阻塞整体监控流程。
