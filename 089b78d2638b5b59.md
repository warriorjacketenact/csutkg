# Hyperlink Index Database System (HIDS)

Hyperlink Index Database System 是一个面向技术研究、数据分析与信息管理领域的结构化外链资源整理与快速检索平台。本项目定位于为开发者、研究员、数据工程师以及技术决策者提供一套可本地部署、可扩展、高度规范化的 URL 资源索引系统，用于解决大规模分散链接难以分类、难以追溯、难以共享的痛点。

与传统的浏览器书签或在线收藏夹不同，HIDS 将资源视为一级数据对象，围绕其来源、类型、状态、标签、批次与校验信息建立关系型数据模型，并提供命令行交互工具与轻量级 Web 浏览界面，使得用户在数万条链接规模下仍能保持高效的筛选与定位能力。本项目当前批次为第 137/1241 批，共计收录 250 个经过初步筛选的外部资源链接，所有链接均保持原始交付格式，不做任何协议补全或域名规范化处理，以确保数据溯源的真实性与可复现性。

## 功能概览

**批量导入与去重校验**：支持从纯文本文件、CSV 以及标准输入流中批量导入 URL 列表，自动执行基于完整 URL 字符串的精确去重与基于域名和路径结构的模糊去重，并生成导入报告。

**多维度标签分类体系**：允许用户为每个资源链接赋予多个自定义标签，支持层级标签（如 技术/前端/构建工具）与自由标签两种模式，并提供标签使用频率统计与未使用标签清理功能。

**链接可用性健康检查**：内置异步 HTTP 探测模块，可配置超时时间、重试次数与用户代理，周期性检测资源链接的可访问状态、响应时间与 HTTP 状态码变化，标记失效或重定向链接。

**结构化查询与过滤语言**：提供类 SQL 的轻量查询语法，支持按域名、路径关键字、标签组合、导入批次、可用状态、最后检查时间等条件进行复合过滤，查询结果可导出为 JSON、CSV 或纯文本列表。

**批次管理与增量更新**：以批次（Batch）为逻辑单元组织资源，每个批次记录导入时间、来源说明、链接数量与校验和，支持批次整体删除、导出与比对，便于多批次数据的增量合并与差异分析。

**数据快照与回滚机制**：每次执行写入操作（导入、删除、更新标签）前自动创建数据快照，用户可通过命令行回退至任意历史快照，防止误操作导致的数据丢失。

## 应用场景

技术博客与文档站点的外链归档管理：技术作者或文档维护者可使用 HIDS 整理文章中引用的外部参考链接，按主题、年份或可信度打标，并在文章更新时快速检查所有外链是否仍然有效，避免读者遭遇死链。

开源项目依赖与参考资源追踪：开源项目维护者可通过 HIDS 记录项目所依赖的第三方库主页、规范文档、参考实现等资源链接，将链接管理与代码仓库解耦，便于在团队内部分享和同步外部参考资料。

数据采集管道中的 URL 治理中间件：在数据采集或爬虫系统中，HIDS 可作为 URL 治理中间件，对采集到的原始链接进行规范化存储、去重、健康检查和标签补全，为下游的数据清洗和分析提供干净、可追溯的 URL 数据集。

技术调研与竞品分析的信息整理：技术决策者或产品经理在进行技术选型或竞品分析时，可将收集到的各类产品官网、技术白皮书、社区讨论帖、性能测试报告等链接统一纳入 HIDS，通过标签和查询功能快速提取特定维度的信息集合。

## 快速开始

以下步骤将在本地环境中完成 HIDS 的克隆、安装与初次运行。执行前请确保系统已安装 Git 与 Python 3.9 及以上版本。

```bash
# 克隆项目仓库至本地
git clone https://github.com/hids-org/hyperlink-index-db.git

# 进入项目根目录
cd hyperlink-index-db

# 创建并激活 Python 虚拟环境（推荐）
python3 -m venv venv
source venv/bin/activate

# 安装项目依赖
pip install -r requirements.txt

# 执行初始化数据库迁移
python manage.py migrate

# 导入当前批次资源列表（示例数据）
python manage.py import --batch 137 --source resources/batch_137.txt

# 启动内置 Web 服务，默认监听 8000 端口
python manage.py runserver
```

执行完成后，打开浏览器访问 http://127.0.0.1:8000 即可进入 HIDS 的 Web 浏览界面。首次启动将自动生成管理员账户，初始密码将输出在终端日志中，请及时修改。

## 安装要求

| 依赖 | 必需 | 说明 |
|---|---|---|
| Python 3.9 或更高版本 | 是 | 核心运行环境，用于执行所有后台任务与 Web 服务 |
| SQLite 3.35 或更高版本 | 是 | 默认内置数据库，用于存储链接元数据、标签与批次信息 |
| requests 库 2.28 或更高版本 | 是 | 用于链接健康检查中的 HTTP 探测与状态码解析 |
| jinja2 库 3.1 或更高版本 | 是 | Web 界面的模板渲染引擎，提供动态页面生成 |
| pytest 7.0 或更高版本 | 否 | 仅当需要运行单元测试或贡献代码时安装 |
| redis-py 4.5 或更高版本 | 否 | 可选，用于分布式部署场景下的缓存与任务队列 |
| gunicorn 20.0 或更高版本 | 否 | 可选，用于生产环境的 WSGI 服务部署 |
| nodejs 18 或更高版本 | 否 | 仅当需要重新编译前端静态资源时使用 |

安装上述依赖时，建议使用 pip 配合 requirements.txt 进行统一安装。对于可选依赖，可通过 pip install -r requirements-optional.txt 单独安装。

## 常见问题

问：导入大量链接时出现数据库锁定错误，如何解决？

答：SQLite 在并发写入时会产生数据库锁定错误。解决方案有三种：一是将数据库后端切换为 PostgreSQL（参考 docs/admin/postgres-setup.md）；二是在导入命令中增加 --batch-size 参数，将单次事务提交的条数调小（例如 100 条）；三是确保导入过程中没有其他进程（如 Web 服务或健康检查调度器）正在执行写入操作。如果问题仍然存在，可尝试将数据库文件迁移至读写性能更高的存储介质。

问：健康检查模块对某些 HTTPS 链接返回 SSL 证书错误，但浏览器访问正常，如何处理？

答：这是因为健康检查模块默认启用证书验证。对于内网或自签名证书的环境，可通过配置文件将 verify_ssl 参数设置为 false。但请注意关闭证书验证会降低安全性，建议将自签名证书的 CA 文件路径添加到系统信任链中，或在配置中指定 ca_bundle 路径。对于企业内网环境，也可将检查超时时间适当延长，避免因网络代理延迟导致的误报。

问：如何将当前批次的链接数据完整迁移到另一台服务器？

答：迁移数据需要同时转移数据库文件和资源目录中的原始批次文件。首先在源服务器上执行 python manage.py snapshot create 创建完整快照，然后将快照文件（位于 snapshots/ 目录）以及 resources/ 目录下的所有批次文件打包传输至目标服务器。在目标服务器上执行 python manage.py snapshot restore --file 快照文件名 即可恢复完整数据。若仅需迁移特定批次，可使用 python manage.py export --batch 编号 --format json 导出为 JSON 格式，再在目标服务器通过 python manage.py import --json 文件路径 进行导入。
