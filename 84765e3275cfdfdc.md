# ResourceBridge

ResourceBridge 是一个面向技术研究者、内容策展人与基础设施运维团队的高密度外链聚合与导航系统。本项目并非传统的网页爬虫或书签管理器，而是一套以静态资源索引为核心的元数据目录服务，专用于对大规模分散式网络资源进行结构化收录、分类标注与状态追踪。

项目定位为“资源的中转层”而非“资源的存储层”。ResourceBridge 本身不托管任何视频、文档或二进制文件，而是通过稳定可校验的 URL 收录机制，为特定主题域（如技术讲座、产品发布记录、版本更新日志、研讨会录像等）提供统一访问入口。目标用户包括技术文档撰写者、在线教育内容运营、企业知识库管理员以及需要长期维护外部参考链接的研究团队。

ResourceBridge 解决的核心问题在于：当外部资源链接分散于数十个不同子域名与路径深度的页面中时，人工维护的书签或本地文档极易出现链接失效、命名混乱、状态不明等情况。本项目提供一套基于 YAML 前端数据标记的链接登记规范，配合自动化检查脚本，可定期输出链接可达性报告，帮助用户在大规模外链集合中快速定位异常资源。

本批次为项目第 676/1241 批资源收录批次，共包含 250 个剧场类活动记录页面链接，全部来自 i-fights.net 域下的 theater 目录。所有链接均按原始形态原样收录，未做任何协议升级、域名规范化或路径改写处理。

## 功能概览

批量链接收录：支持一次性导入大批量 URL 条目，自动解析路径参数与页面元数据，生成标准化索引记录。每一条收录均保留原始协议头与域名形式，不进行任何隐性跳转或改写。

结构化目录输出：根据 URL 路径模式自动生成分类树，支持按一级目录、二级目录、文件名前缀等多维度进行筛选与排序，便于用户按活动编号或时间线查阅。

链接状态巡检：内置轻量级 HTTP 状态检测模块，可对已收录链接进行定期 HEAD 请求检查，标记返回码异常（4xx、5xx）或超时条目，输出异常清单供维护参考。

自定义标签系统：允许用户为每条链接附加多个业务标签，例如“产品发布”、“季度回顾”、“嘉宾分享”等，实现跨目录的横向主题聚合。

只读镜像导出：支持将当前索引数据导出为静态 JSON 或 CSV 格式，便于下游系统（如静态站点生成器、数据可视化面板）消费，无需依赖运行时数据库。

访问日志摘要：记录每条链接的最近查阅时间与引用次数（本地计数），帮助识别高频访问资源与长期未触及的归档链接。

冲突检测与去重：在新增批次收录过程中自动比对现有索引，发现重复 URL 或高度相似路径时发出警告，防止冗余条目污染数据集。

## 应用场景

技术团队内部知识库的外链规范化管理。团队在日常开发中会积累大量指向外部会议录像、设计文档草稿、第三方库发布公告的临时链接。通过 ResourceBridge 集中收录并按版本号或模块名分类，可避免链接散落在 Wiki 页面或聊天记录中，提升检索效率。

在线课程或技术专栏的参考资料维护。教育内容创作者在撰写系列教程时，常需引用多个外部演示页面。利用 ResourceBridge 按课程章节或周次组织链接，并在每学期开始前统一执行状态巡检，可及时发现失效资源并提供备选方案。

大型开源项目的版本发布记录归档。开源社区在每次 Major 或 Minor 版本发布时，通常伴随线上直播或技术讲解录像。将相关 theater 页面链接纳入 ResourceBridge 索引，按版本号标签归档，可为后续社区回顾或合规审计提供稳定入口。

个人研究者的文献式链接收藏。研究人员在调研特定领域（例如分布式系统演进、编程语言设计决策）过程中，需要长期追踪数十个会议演讲页面。ResourceBridge 的标签与状态追踪功能可替代传统的浏览器书签，提供更结构化的管理视角。

## 快速开始

以下指令适用于 Linux/macOS 环境，Windows 用户可使用 WSL 或 Git Bash 执行。

git clone https://github.com/resource-bridge/resource-bridge.git
cd resource-bridge
pip install -r requirements.txt
python bridge.py --ingest --batch 676 --source ./data/batch_676.txt

其中 batch_676.txt 为包含全部 250 个 URL 的纯文本文件，每行一个链接。执行后系统将自动生成索引文件并输出收录统计摘要。

## 项目结构

resource-bridge/
├── bridge.py                 # 主入口脚本，整合收录、检查、导出子命令
├── config.yaml               # 运行时配置，包含巡检间隔、超时阈值、索引存储路径
├── data/
│   ├── index.db              # SQLite 索引数据库，存储所有链接记录与标签
│   ├── batches/              # 批次原始输入文件存档目录
│   │   └── batch_676.txt     # 本批次 250 条链接的原始输入
│   └── reports/              # 巡检报告输出目录（HTML 与 JSON 格式）
├── src/

## 常见问题

Q: 收录过程中提示 "URL already exists in index" 是否会导致收录中断？
A: 不会。该提示为警告级别信息，表示当前批次中存在与历史索引重复的链接。系统会跳过重复条目并继续处理后续 URL，最终统计中会单独列出重复数量。如需强制覆盖已有记录，可在收录命令中添加 --force 参数。

Q: 状态巡检返回 403 或 429 状态码时，系统如何处理？
A: 系统将 403（禁止访问）和 429（请求过多）视为可恢复异常，不会立即标记为失效。巡检模块会记录该状态码并在下一次巡检周期中重试。若连续三次巡检均返回相同异常码，则将该链接标记为 "blocked" 状态，并在报告中高亮提示。用户可以手动调整巡检间隔或配置代理池以规避访问限制。

Q: 项目是否支持 Windows 文件路径中的反斜杠分隔符？
A: 支持。项目内部使用 pathlib 库处理所有文件系统路径，该库会自动适配当前操作系统的路径分隔符。但建议在配置文件中仍使用正斜杠（/）以保证跨平台兼容性，系统在解析时会自动转换。
