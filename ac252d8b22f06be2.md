# LinkVault Core

LinkVault Core 是一个面向技术团队与独立开发者的外链资源聚合与导航系统。该项目并非简单的书签管理器，而是一个具备可观测性、健康检查与访问审计能力的资源网关。其目标用户包括运维工程师、技术文档维护者、API 集成测试人员以及需要系统化管理大量外部依赖 URL 的开发团队。LinkVault Core 通过统一的入口对分散的外部资源进行标准化封装，解决因链接失效、来源不明或访问策略不一致导致的信息碎片化问题，从而提升技术资源的可维护性与可追溯性。

## 功能概览

- **资源分级索引**：支持按技术领域、服务商、内部项目等多维度对链接进行标签化分类，并提供层级清晰的目录视图。

- **健康状态探测**：内置异步 HTTP 探针，可定期检测每个资源 URL 的可达性与响应码，自动标记异常链接并生成告警日志。

- **访问审计追踪**：记录每次通过网关访问外部资源的请求元数据（时间、来源 IP、用户代理），满足企业内部审计合规要求。

- **元数据提取增强**：对目标 URL 进行智能抓取，自动提取页面标题、描述、内容摘要与最后修改时间，丰富资源索引信息。

- **批量导入与导出**：支持通过 YAML 或 JSON 格式批量导入链接清单，并支持将索引数据导出为 CSV 或 Markdown 表格，便于离线分发。

- **缓存回源策略**：对高频访问的资源链接提供本地缓存机制，降低对外部服务的重复请求压力，同时支持手动强制刷新。

- **权限分级管理**：基于角色的访问控制体系，区分管理员、编辑者与只读访客，适用于团队协作场景。

- **搜索与过滤引擎**：提供基于关键词、标签、状态码、更新时间等多条件组合的过滤与全文检索能力，快速定位目标资源。

## 应用场景

**技术文档中心外链治理**  
大型项目的技术文档往往包含数百个外部引用链接。LinkVault Core 可定期扫描文档仓库中的所有外链，识别失效或变更的资源，并生成报告供文档维护者批量修正。

**微服务依赖端点管理**  
在微服务架构中，各个服务依赖的外部 API 地址、配置中心端点、监控数据上报地址等均可纳入 LinkVault Core 统一管理。当上游服务变更地址时，管理员只需在网关处更新一次映射，无需修改每个服务的配置文件。

**渗透测试与安全审计前置检查**  
安全团队在实施渗透测试前，需收集目标系统的所有外部交互地址。LinkVault Core 可快速聚合目标域名的所有关联资源，并按照协议类型、端口、路径特征进行分类，辅助攻击面梳理工作。

**离线环境资源同步规划**  
在政务云或金融内网等离线隔离环境中，部署人员需提前梳理所有需要同步的外部资源清单。LinkVault Core 导出的完整资源列表与元数据可作为同步脚本的输入源，显著降低遗漏风险。

## 快速开始

以下指令可在 Linux 或 macOS 环境中完成 LinkVault Core 的初始部署。Windows 用户建议通过 WSL2 或 Git Bash 执行。

```bash
# 克隆项目仓库至本地
git clone https://github.com/linkvault/core.git linkvault-core

# 进入项目根目录并安装依赖
cd linkvault-core
pip install -r requirements.txt

# 执行数据库初始化与内置探针编译
python scripts/bootstrap.py
python -m linkvault.gateway --port 8080
```

启动成功后，访问本机 8080 端口即可进入 Web 管理界面。首次登录默认管理员账号为 `admin`，初始密码为 `linkvault2024`，登录后请及时修改。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.9 及以上 | 核心后端运行环境，推荐使用 3.11 |
| Redis | 7.0 及以上 | 用于缓存与分布式锁，单机或集群模式均可 |
| PostgreSQL | 14 及以上 | 主数据库，存储资源索引、审计日志与用户数据 |
| Node.js | 18 LTS 及以上 | 仅用于前端静态资源构建，生产环境可复用已构建产物 |
| Nginx | 1.24 及以上 | 推荐作为反向代理前端静态文件与 API 路由 |
| Celery Worker | 5.3 及以上 | 用于异步健康探测与元数据抓取任务队列 |
| Supervisor | 4.2 及以上 | 建议用于守护 Celery 与 Gateway 进程（非强制） |
| Git | 2.30 及以上 | 用于版本管理与补丁更新 |

## 贡献指南

1.  **查阅问题跟踪器**  
    访问 GitHub Issues 页面，查找标记为 `help-wanted` 或 `good-first-issue` 的任务。在开始实现之前，请在对应 issue 下留言说明认领意向，避免重复工作。

2.  **派生仓库并创建特性分支**  
    将主仓库派生至个人账户下，然后基于 `main` 分支创建新的分支，分支命名遵循 `feature/功能简述` 或 `fix/问题简述` 格式。

3.  **编写代码与单元测试**  
    所有新增功能或修复必须包含对应的单元测试用例，确保测试覆盖率达到 80% 以上。测试文件存放于 `tests/` 目录，命名与模块保持对应。

4.  **提交前执行本地检查**  
    运行 `make lint` 进行代码风格检查（基于 Black 和 Flake8），运行 `make test` 确保全部用例通过。提交信息需符合 Conventional Commits 规范，例如 `feat(probe): add retry mechanism for timeout errors`。

5.  **发起 Pull Request**  
    将分支推送至派生仓库后，向主仓库的 `main` 分支发起 Pull Request。在描述中关联对应的 Issue 编号，并简要说明改动点与测试结果。PR 合并前需要至少一名维护者进行 Code Review。

## 常见问题

**Q：健康探测任务积压严重，如何优化？**  
A：首先检查 `config/production.yaml` 中的 `probe.interval` 参数，适当增大探测间隔（例如从 300 秒调整为 600 秒）。其次，确保 Celery Worker 的并发数（`--concurrency` 参数）与 CPU 核心数匹配，避免过多协程竞争。对于大量超时链接，可在 `probe.timeout` 中设置更短的单次请求超时时间，并启用失败快速跳过机制。

**Q：如何对接企业内部的 LDAP 认证系统？**  
A：在 `config/production.yaml` 中配置 `auth.ldap` 段落，填写 LDAP 服务器地址、Base DN、绑定账号与搜索过滤器。重启 Gateway 服务后，登录界面会自动出现 LDAP 登录选项。LDAP 用户默认映射为只读访客角色，如需提升权限，需由管理员在后台手动关联角色。

**Q：批量导入大量链接时出现内存溢出？**  
A：建议将导入文件拆分为多个较小批次（例如每批 500 条），使用 `scripts/import_links.py --batch-size 500` 参数执行导入。同时，检查 PostgreSQL 的 `work_mem` 和 `maintenance_work_mem` 参数，适当调大以加速批量写入。如果使用 Docker 部署，请确保分配给容器的内存上限不低于 2GB。
