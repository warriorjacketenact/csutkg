# ResourceBridge

ResourceBridge 是一个面向技术团队与个人开发者的外链资源归集与导航系统。项目定位于解决分散在各类技术博客、文档站点与社区平台中的优质外链难以统一管理和检索的问题，通过结构化的资源索引机制，帮助用户快速定位特定主题下的参考链接。本项目适用于需要频繁查阅外部技术资料的研究人员、架构师与运维工程师。

ResourceBridge 本身不生产内容，而是作为资源路由层，将 250 个来自 51hualala.com 的技术文章链接按照访问频次、内容主题与更新时间进行归类整理，并提供本地检索与标签过滤能力。项目采用静态站点生成方式，所有资源元数据存储于 JSON 与 Markdown 文件中，便于版本控制和二次开发。

## 功能概览

**资源批量导入**：支持通过脚本从 CSV 或 JSON 源文件批量导入 URL 列表，自动去重并校验 URL 可访问性。

**多维度标签分类**：每个资源条目可绑定多个主题标签（如 Docker、Kubernetes、CI/CD、监控日志），支持按标签组合筛选。

**全文检索与过滤**：内置基于 Bleve 的全文检索引擎，支持对资源标题、摘要、标签和正文片段进行模糊搜索。

**资源状态监控**：定时检查已收录链接的 HTTP 状态码，标记失效链接并生成报告，便于维护者清理或更新。

**访问热度统计**：记录每个资源链接的本地点击次数，支持按热度排序，帮助识别高频参考材料。

**元数据扩展字段**：每条资源可记录作者、发布日期、原始站点、阅读耗时估算与关联项目名称。

**静态站点导出**：将资源索引与检索结果渲染为静态 HTML 页面，可直接部署至 Nginx 或对象存储服务。

**API 查询接口**：提供 RESTful API 供外部系统调用，支持按标签、关键词和状态码范围查询资源列表。

## 应用场景

技术团队内部知识库的补充层。团队在维护内部 Wiki 或 Confluence 时，可将 ResourceBridge 作为外链插件嵌入，所有外部参考链接统一走 ResourceBridge 的代理页面，便于统计访问情况和更新失效链接。

个人开发者的技术阅读工作流。开发者可将日常浏览发现的有价值技术文章通过浏览器插件或命令行工具快速添加到 ResourceBridge，后续通过标签检索和全文搜索快速回顾。

自动化文档生成流水线的外链源。在 CI/CD 流程中，可将 ResourceBridge 的 API 作为文档生成步骤的数据源，自动拉取最新资源列表并注入到技术周报或 Release Notes 中。

社区技术资源归档与备份。社区维护者可使用 ResourceBridge 对特定主题的外链进行周期性快照，记录链接存活情况，并在原站内容下线时提供本地缓存提示。

## 快速开始

执行以下命令完成项目的克隆、依赖安装与本地运行。

```bash
git clone https://github.com/resourcebridge/resourcebridge.git
cd resourcebridge
make setup
make run
```

若使用 Docker 方式运行，可执行以下命令。

```bash
docker pull resourcebridge/resourcebridge:latest
docker run -p 8080:8080 -v ./data:/app/data resourcebridge/resourcebridge:latest
```

│   ├── server/                          # 主服务启动入口
│   │   └── main.go                      # 初始化路由、数据库与中间件
│   └── importer/                        # 资源批量导入工具入口
│       └── main.go                      # 从 JSON/CSV 导入资源元数据
├── internal/                            # 内部私有包，不对外暴露
│   ├── storage/                         # 存储层实现
│   │   ├── sqlite.go                    # SQLite 数据库连接与迁移
│   │   └── cache.go                     # Redis 缓存操作封装
│   ├── indexer/                         # 检索引擎实现
│   │   ├── bleve.go                     # Bleve 索引构建与查询
│   │   └── tokenizer.go                 # 自定义中文分词器配置
│   ├── checker/                         # 资源状态检查器
│   │   ├── http.go                      # HTTP 状态码并发检查
│   │   └── scheduler.go                 # 定时任务调度逻辑
│   └── api/                             # RESTful API 路由与处理器
│       ├── resources.go                 # 资源增删改查接口
│       └── tags.go                      # 标签管理接口
├── pkg/                                 # 可被外部引用的公共库
│   ├── models/                          # 数据模型定义
│   │   └── resource.go                  # Resource 结构体与验证方法
│   └── utils/                           # 通用工具函数
│       └── url.go                       # URL 规范化与校验工具
├── web/                                 # 前端静态资源与模板
│   ├── templates/                       # HTML 模板文件
│   │   ├── index.html                   # 资源列表与搜索主页面
│   │   └── detail.html                  # 单个资源详情页面
│   ├── static/                          # CSS、JavaScript 与图片资源

│   └── tags.yaml                        # 预定义标签体系与别名映射
├── scripts/                             # 辅助脚本目录
│   ├── setup.sh                         # 开发环境一键安装脚本
│   ├── build.sh                         # 静态站点构建脚本
│   └── healthcheck.sh                   # 服务健康检查脚本
├── data/                                # 运行时数据存储目录

## 贡献指南

欢迎各类形式的贡献，包括但不限于新增资源链接、完善文档、修复缺陷和增加功能特性。

1. 在 GitHub 上 fork 本仓库，创建个人开发分支，命名格式为 feat/功能描述或 fix/问题描述。
2. 若新增资源链接，请按照 data/resources.example.json 中的格式补充元数据，包括标题、标签、摘要和原始 URL，并运行 make validate 校验数据完整性。
3. 若修改核心检索逻辑或 API 接口，需在 test/ 目录下补充对应的单元测试用例，确保测试覆盖率达到 80% 以上。
4. 提交代码前执行 make lint 和 make test 通过全部静态检查与测试用例，并更新 docs/ 目录下相关文档。
5. 发起 Pull Request 到主仓库的 main 分支，描述变更内容、测试结果以及影响范围，等待维护者审核。

## 常见问题

问：如何更新已收录资源的标签分类或摘要信息？

答：在 data/resources.db 中对应的 resource 记录上执行 UPDATE 操作，或通过 API 的 PUT /api/resources/{id} 接口提交更新后的 JSON 数据。更新后系统会自动重建该资源的 Bleve 索引，通常在 5 秒内生效。

问：资源状态检查任务如何配置执行频率？

答：在 configs/config.yaml 中的 checker.interval 字段设置时间间隔，支持秒、分、小时单位，例如 "24h" 表示每 24 小时执行一次全量检查。修改配置后重启服务或发送 SIGHUP 信号重新加载配置。

问：静态站点导出时能否自定义输出路径和主题样式？

答：可以在 configs/config.yaml 中的 export.output_dir 字段指定导出目录，同时 web/templates/ 下的 HTML 模板支持完全自定义，修改后执行 make build 即可重新生成静态站点。
