# TechResource Hub

TechResource Hub 是一个面向开发者与技术研究人员的结构化外链资源聚合平台。本项目不直接托管任何教程或文档内容，而是通过人工筛选与社区提交的方式，整理互联网上优质的技术文章、工具站点、学术论文与工程实践参考链接，并以清晰的主题分类与可检索的元数据格式对外提供。

本项目解决的核心问题是：技术从业者在日常学习与项目开发中，面对海量分散的网络资源，缺乏一个轻量级、可版本化、可协作维护的集中式入口。TechResource Hub 提供了一套基于纯文本文件与静态站点生成机制的链接管理方案，使团队或个人能够高效地建立私有或公开的技术外链库，并支持通过 Git 工作流进行协同更新。

本项目的目标用户包括：需要维护团队知识库的架构师、撰写技术周报的编辑人员、搭建个人导航页面的开发者，以及希望从社区获取高质量技术资料推荐的学习者。

## 功能概览

**批量链接导入**：支持从 CSV、JSON 或纯文本列表批量导入 URL，自动解析标题与来源域名，并生成初始的目录条目。

**智能分类引擎**：根据 URL 路径关键词、页面标题与人工标注的标签，将链接自动归类至后端、前端、运维、算法、数据库等预设分类体系。

**全文元数据检索**：为每条链接生成包括标题、摘要、关键词、提交人、提交时间、所属批次在内的完整元数据，并提供多条件组合过滤与全文搜索接口。

**静态站点生成**：内置模板引擎，可将链接库导出为静态 HTML 站点，支持响应式布局与暗色主题，适用于内部文档服务器或 GitHub Pages 部署。

**自动健康检查**：定期对已收录的链接进行可访问性探测与响应时间统计，自动标记失效链接或重定向链接，并生成健康报告。

**协作审核工作流**：基于 Pull Request 的链接提交与审核机制，允许社区成员提议新增或更新链接，由维护者合并后自动更新索引。

## 应用场景

企业内部技术团队的知识沉淀：团队可将项目开发中参考过的官方文档、解决方案博客、API 手册等外链统一收录至 TechResource Hub，形成团队共享的技术外链索引，减少重复查找时间。

个人技术博客的友情链接管理：博主可利用本项目维护一个“好友链”或“推荐阅读”页面，通过静态导出功能生成导航页，并定期运行健康检查避免链接失效。

技术社区的内容聚合与推荐：开源社区或技术论坛可基于本项目搭建一个“每周精选”资源汇总站，由多位编辑共同提交链接，经审核后发布为周报或月刊形式。

高校或培训机构的课程参考资料库：教师可将每门课程涉及的扩展阅读链接按章节分类整理至 Hub 中，学生可通过搜索快速定位特定主题的参考材料。

## 快速开始

以下命令演示了如何在本地环境中克隆项目、安装依赖并启动开发服务器。

```bash
# 克隆项目仓库
git clone https://github.com/techresource-hub/hub-core.git
cd hub-core

# 安装 Python 依赖（推荐使用虚拟环境）
python -m venv venv
source venv/bin/activate  # Windows 下使用 venv\Scripts\activate
pip install -r requirements.txt

# 初始化本地链接数据库并导入示例数据
python manage.py initdb
python manage.py import --source fixtures/sample_links.json

# 启动本地开发服务器（默认监听 8000 端口）
python manage.py runserver
```

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---------|---------|------|
| Python | 3.9 或更高 | 核心运行环境，用于后端服务与命令行工具 |
| Node.js | 18.x 或更高 | 仅当启用前端构建功能时需要，用于编译静态资源 |
| SQLite | 3.35 或更高 | 默认嵌入式数据库，适合小型部署与本地开发 |
| Redis | 6.2 或更高 | 可选依赖，用于缓存与分布式锁，提升高并发查询性能 |
| Git | 2.30 或更高 | 用于版本控制与协作工作流，也是获取项目源码的必要工具 |

## 贡献指南

1. 提交链接建议：通过 GitHub Issues 提交新的外链推荐，需包含标题、完整 URL、所属主题分类以及简短的摘要说明（50-200 字）。建议使用我们提供的 Issue 模板以加快审核速度。

2. 改进分类规则：若发现现有自动分类结果不准确，可修改 configs/rules.json 文件中的关键词映射表，然后发起 Pull Request。请附上修改前后对若干测试链接的分类对比结果。

3. 完善文档：我们欢迎对文档的补充与修正，包括但不限于修复错别字、补充常见问题条目、更新部署指南中的新版本特性说明。文档位于 docs/ 目录下，使用 Markdown 格式编写。

4. 开发新功能或修复缺陷：请先从 Issues 中认领任务，或新建 Issue 描述你打算解决的问题。开发完成后，确保所有单元测试通过（make test），并尽可能为新功能编写对应的测试用例。提交 Pull Request 时请清晰描述改动点与测试覆盖情况。

5. 参与社区讨论：我们也欢迎在 Discussions 板块分享你的使用经验、集成方案或对项目未来演进的建议，这些讨论同样属于贡献的一部分。

## 常见问题

问：项目是否必须依赖 Redis？能否完全离线运行？

答：Redis 是可选依赖。如果不使用缓存与分布式锁功能，项目会降级为内存缓存模式，所有数据仍持久化至 SQLite，可完全离线运行于单机环境。只需在配置文件中将 cache.enabled 设置为 false 即可。

问：如何更新已收录链接的元数据或修正错误的分类？

答：可以通过命令行工具执行更新操作。例如：`manage.py update --id <链接ID> --title "新标题" --category "新分类"`。也支持通过编辑数据库文件的方式批量修改，但不推荐直接操作数据库，建议使用管理命令以保证索引同步更新。

问：健康检查会对目标站点造成压力吗？频率如何控制？

答：健康检查仅发送轻量级 HEAD 请求，不下载页面内容，对目标服务器影响极小。默认检查间隔为 24 小时，且在配置中可调整至最长 7 天。并发请求数默认为 5，可通过命令行参数调低以避免触发目标站点的限流策略。
