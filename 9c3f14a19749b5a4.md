# StreamHub 技术资源导航站

StreamHub 是一个面向开发者和技术研究人员的结构化外链资源聚合平台，专注于收集、分类和索引高质量的技术文档、开发工具、学术资源与基础设施服务。项目定位为"技术资源的入口"，通过扁平化的短码路由体系，将分散于互联网各处的优质内容组织为可检索、可分享、可维护的集中式目录。

本项目适用于需要频繁查阅外部技术资料、维护个人知识库或搭建团队技术导航页的开发者。StreamHub 不存储任何实际内容，仅提供链接索引与分类管理能力，所有资源指向原始第三方站点。项目采用静态生成架构，支持一键部署至主流托管服务，资源列表通过 YAML 配置文件维护，便于版本控制和协作更新。

## 功能概览

**短码路由系统** 每个资源条目分配唯一的三字符短码（如 uhg、ylz），通过 /stream/{code} 端点提供直达访问，便于口头传播与文档引用。

**分类标签体系** 资源按技术领域、内容类型、适用场景等多维度标签组织，支持自由组合筛选。

**自动元数据抓取** 系统在资源入库时自动请求目标站点，提取标题、描述和图标信息，减少手动维护成本。

**状态监控面板** 内置链接可用性检查功能，定时检测资源是否可访问，并在管理界面标注异常状态。

**静态站点生成** 构建时输出纯 HTML 文件，无需后端服务即可运行，兼容 GitHub Pages、Cloudflare Pages、Netlify 等平台。

**全文检索支持** 基于资源标题、描述和标签构建本地索引，提供毫秒级的关键词搜索响应。

**导入导出能力** 支持 CSV 与 JSON 格式的资源批量导入导出，便于与其他工具或团队同步数据。

**响应式布局** 适配桌面端与移动端浏览，确保在任何设备上获得一致的访问体验。

## 应用场景

**个人技术知识库构建** 开发者可将日常高频查阅的官方文档、API 参考、教程文章统一收录于 StreamHub，通过短码快速访问，替代浏览器书签的杂乱管理方式。随着资源积累，个人知识库逐渐形成体系化的技术地图。

**团队内部技术导航** 技术团队可使用 StreamHub 搭建部门级别的资源门户，集中存放内部系统地址、运维面板、代码仓库、CI/CD 链接等，新成员入职时通过导航站快速了解团队工具链全貌，减少反复询问。

**开源项目文档站外链管理** 开源项目维护者将项目依赖的规范标准、参考实现、社区论坛等外部链接整理为 StreamHub 资源列表，置于项目 README 或文档站中，方便贡献者查阅背景资料，降低参与门槛。

**技术社区资源聚合** 技术社区运营方可利用 StreamHub 汇集成员推荐的优质文章、视频课程、开源工具，形成动态更新的社区精选集，通过短码在讨论区中快速引用具体资源。

## 快速开始

以下命令将在本地启动 StreamHub 开发实例，默认监听端口 8080。

```bash
git clone https://github.com/streamhub/streamhub.git
cd streamhub
npm install
npm run build
npm start
```

如需使用开发热重载模式，可执行 `npm run dev`。构建产物位于 `dist/` 目录，可直接部署至任意静态托管服务。

## 安装要求

| 依赖 | 必需版本 | 说明 |
|------|----------|------|
| Node.js | 18.x 或更高 | 运行环境与包管理器基础依赖 |
| npm | 9.x 或更高 | 依赖安装与脚本执行工具 |
| Git | 2.30 或更高 | 代码克隆与版本管理 |
| YAML 解析器 | 内置于项目依赖 | 用于解析 resources.yaml 资源清单文件 |
| 现代浏览器 | 支持 ES2020 | 访问管理界面与资源页面所需客户端环境 |
| 网络连接 | 稳定 | 首次构建时需下载 npm 包，运行监控功能需访问外网 |
| 磁盘空间 | 至少 100 MB | 存放项目代码、依赖及构建产物 |
| 可选：Docker | 20.x 或更高 | 容器化部署方式替代方案 |

## 常见问题

Q: 如何自定义短码映射规则？是否支持修改已存在的短码？

A: 所有短码映射在 config/resources.yaml 文件中定义，采用键值对格式（shortcode: url）。您可以直接编辑该文件增删或修改条目。修改已存在的短码后，需要重新执行 npm run build 触发全量构建，新映射才会在生产环境生效。注意，修改已有短码会导致旧链接失效，建议保留原短码并添加新条目，或通过 301 重定向规则平滑过渡。

Q: 资源链接状态检查显示异常，但目标站点在浏览器中可以正常访问，是什么原因？

A: 链接检查器默认使用 HEAD 请求验证资源可用性，部分站点（如 API 网关、个人博客）可能拒绝 HEAD 请求或要求特定的 User-Agent 头。您可以在 config/default.yaml 中调整 checker.timeout 超时时间、修改 checker.userAgent 字符串，或切换检查方法为 GET。若仍失败，请在管理界面手动标记该资源为"忽略检查"状态。

Q: 是否支持部署到国内云服务商或内网环境？需要注意什么？

A: 支持。StreamHub 生成的是纯静态文件，可部署至任意对象存储或 Web 服务器。部署到内网时，需确保资源列表中的所有 URL 可从内网环境访问，否则重定向会失败。若需在内网使用外部资源，可配置反向代理或修改 resources.yaml 中的 URL 为内部镜像地址。构建过程无需联网，但首次安装依赖需能够访问 npm 公共仓库，可提前配置国内镜像源（如淘宝镜像）。
