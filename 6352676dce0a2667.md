# LinkVault — 技术资源外链聚合与导航系统

LinkVault 是一个面向开发者、技术研究人员与内容策展人的轻量级外链资源聚合与导航平台。该项目定位于将分散于网络各处的技术文章、工具站点、文档参考与社区讨论进行结构化收集、分类存储与快速检索，解决个人或团队在技术研究过程中“链接散落、回忆困难、复用率低”的核心痛点。

LinkVault 不提供爬虫与自动采集功能，而是围绕人工精选与元数据标注构建工作流，适用于搭建内部技术知识库、开源项目推荐墙、个人阅读清单或团队共享资源中心。项目以静态站点生成器为核心，输出纯 HTML 资源索引，可部署于任何 Web 服务器或对象存储服务。

## 功能概览

- **资源条目结构化存储**：每条外链支持标题、来源域名、分类标签、收录日期、摘要描述与优先级标记，以 YAML 前置数据结合 Markdown 正文形式组织。

- **多维度分类与筛选**：内置分类体系涵盖后端工程、前端性能、运维监控、安全审计、算法理论、语言特性、数据库调优、设计模式等二十余个技术子域，支持按分类、标签、收录时间区间进行组合过滤。

- **全文检索与模糊匹配**：基于倒排索引实现标题与摘要的轻量级全文搜索，支持中文分词与拼音首字母缩写检索，满足快速定位需求。

- **静态站点生成与部署**：提供内置构建命令，将源目录下的所有资源条目编译为单页应用形式的静态 HTML 文件，附带分类索引页、标签云页与最近更新页。

- **书签导入与导出**：支持从主流浏览器书签 HTML 格式批量导入现有收藏，同时支持将资源列表导出为 CSV、JSON 或 OPML 格式，便于迁移至其他阅读工具。

- **访问统计与死链检测**：集成可选的链接状态检查模块，周期性探测已收录资源的 HTTP 状态码，标识失效链接并生成报告，辅助维护资源健康度。

- **自定义元数据扩展**：允许用户为每条资源添加键值对形式的自定义字段，例如“阅读耗时”“难度评级”“关联项目仓库地址”，适配不同团队的信息管理规范。

- **响应式浏览界面**：生成的前端界面基于 CSS Grid 与 Flexbox 布局，适配桌面端、平板与移动设备，支持浅色与深色两种主题切换。

## 应用场景

- **技术团队内部知识库建设**：团队可将日常调研中发现的优质技术博客、开源库文档、性能调优案例统一收录至 LinkVault，并为每条资源标注适用的项目模块或技术栈版本，新成员入职时可快速浏览团队积累的外部参考脉络。

- **开源项目推荐页面**：开源维护者可使用 LinkVault 构建项目生态资源导航页，集中陈列社区插件、示例代码仓库、视频教程与相关技术讨论的链接，降低新用户的入门信息搜集成本。

- **个人技术阅读工作流管理**：技术从业者可利用 LinkVault 管理每周的资讯阅读清单，对已读与待读资源进行标记，并结合摘要记录个人思考，形成长期积累的可检索知识档案。

- **技术活动资料归档**：在组织黑客松、技术沙龙或内部培训后，将活动涉及的参考资料、幻灯片链接、示例代码仓库与录播回放地址统一整理为活动专属资源集，方便后续查阅与传播。

## 快速开始

以下步骤适用于 Linux、macOS 及 Windows WSL 环境，需提前安装 Git 与 Node.js 18.x 以上版本。

```bash
# 克隆项目仓库
git clone https://github.com/example/linkvault.git
cd linkvault

# 安装项目依赖
npm install

# 构建静态资源站点
npm run build

# 启动本地开发服务器，默认监听 http://localhost:8080
npm run serve
```

执行上述命令后，访问本地服务地址即可浏览已收录的资源列表。添加新资源时，在 `./resources` 目录下新建 `.md` 文件，按照模板填写前置元数据与正文描述，重新执行构建命令即可更新站点内容。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | 18.x 或 20.x LTS | 运行时环境，用于执行构建脚本与本地服务 |
| npm | 9.x 或 10.x | 包管理器，用于安装项目依赖包 |
| Git | 2.25 以上 | 版本控制工具，用于克隆仓库与提交变更 |
| YAML 解析器 | 项目内置 | 用于解析资源条目前置元数据，无需额外安装 |
| 静态文件服务器 | 项目内置 | 开发环境基于 http-server 提供本地预览 |
| 磁盘空间 | 建议 200 MB 以上 | 存储源码、依赖包及生成的静态文件 |
| 内存 | 建议 1 GB 以上 | 构建过程涉及索引生成与文件聚合，内存不足可能导致构建缓慢 |
| 操作系统 | Linux / macOS / Windows WSL2 | 跨平台支持，但生产部署推荐 Linux 环境 |
| 网络连接 | 可选 | 仅用于周期性死链检测与远程资源导入功能 |
| 浏览器 | 现代浏览器（Chrome / Firefox / Edge / Safari 最新两个版本） | 用于访问生成的资源导航界面 |

## 贡献指南

1. 在 GitHub 仓库中提交 Issue 说明拟贡献的类型，包括但不限于新增功能、修复缺陷、完善文档或新增资源收录。等待维护者确认需求合理性后再着手实现，避免重复劳动。

2. 克隆项目仓库并在本地创建功能分支，分支命名遵循 `feature/描述` 或 `fix/描述` 格式。确保本地构建通过且所有单元测试与集成测试保持通过状态。

3. 若贡献涉及新增资源条目，请在 `resources/` 下对应分类目录中创建 Markdown 文件，严格按照模板填写前置元数据，包括标题、来源域名、分类、标签、收录日期与摘要，并确保摘要长度在 80 至 200 字符之间。

4. 提交代码前执行 `npm run lint` 检查代码风格，项目遵循 ESLint 配置的 Airbnb 风格指南。提交信息使用英文，采用 Conventional Commits 规范，格式为 `<type>(<scope>): <subject>`。

5. 发起 Pull Request 至主仓库的 `main` 分支，在 PR 描述中关联相关 Issue 编号，并简要说明变更内容与测试覆盖情况。PR 需要至少一位项目维护者审阅通过后方可合并。

## 常见问题

**问：LinkVault 是否支持自动爬取网页内容并填充摘要？**
答：不支持。LinkVault 定位为人工精选资源导航工具，不包含任何自动爬虫或内容抓取功能。每一条资源的标题、摘要与分类均需由收录者手动填写，以确保信息质量与语义准确性。如需批量导入现有收藏，可使用浏览器书签导入功能，但导入后仍需人工补充元数据字段。

**问：死链检测模块如何工作？是否会影响线上服务？**
答：死链检测模块独立于主构建流程，通过 `npm run link-check` 命令手动触发，或以定时任务方式在后台运行。检测器会读取 `resources/` 下所有条目的 URL 字段，发起 HEAD 请求检查 HTTP 状态码，超时设定为 5 秒。检测结果输出至 `reports/` 目录下的 JSON 与 HTML 报告文件，不会修改任何资源条目内容，也不影响已部署的静态站点。

**问：资源数据如何备份与迁移？**
答：所有资源数据以纯文本形式存储在 `resources/` 目录下，备份该目录以及 `config/` 目录中的自定义配置文件即可完整保留全部内容与站点设置。迁移至新环境时，在新机器上克隆项目仓库，将备份的 `resources/` 与 `config/` 复制覆盖，重新执行 `npm install` 与 `npm run build` 即可生成与源环境一致的站点。若需跨系统迁移，确保 Node.js 版本一致即可。
