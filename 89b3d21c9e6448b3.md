# LinkBridge Resource Aggregator

LinkBridge 是一个面向技术研究者、内容策展人和开源开发者的外链资源汇总与结构化导航系统。项目定位于将分散在网络各处的优质外部链接按照主题分类、元数据标注和时效性追踪的方式进行统一管理，解决个人收藏夹杂乱、链接失效、缺乏上下文描述等核心痛点。目标用户包括需要维护技术文档列表的工程师、运营技术资讯网站的编辑团队以及希望建立个人知识库的独立开发者。

本项目并非一个传统的爬虫或采集工具，而是一套轻量级的资源索引框架。它提供标准化的 YAML 前置数据模板、链接健康度检查脚本以及静态站点生成管道，使得用户能够以极低的维护成本运营一个包含数千条外链的垂直领域导航站。项目本身不存储任何第三方内容，仅保留指向原始来源的引用关系，符合开源社区的资源共享伦理。

## 功能概览

**标准化资源登记** 提供基于 JSON Schema 的条目模板，强制要求填写标题、来源域名、分类标签、收录日期和简短摘要，确保每条链接具备一致的元数据结构。

**自动化健康检查** 内置基于 Node.js 的链接巡检模块，支持定时检测每条 URL 的 HTTP 状态码，自动标记返回 4xx 或 5xx 的失效链接并生成报告。

**多级分类体系** 支持无限层级的标签树创建，允许同一资源挂载到多个分类节点下，便于从不同维度检索相关内容。

**静态站点生成** 集成 Eleventy 或 VuePress 渲染管道，将机器可读的 YAML 资源库转换为人类可浏览的 HTML 导航页面，支持暗色主题与全文搜索。

**变更审计日志** 每次增删改操作均记录提交哈希、操作时间与变更说明，方便团队协作时追溯资源来源与修改原因。

**导入导出适配器** 提供与 Raindrop.io、Pocket、Chrome 书签等第三方服务的 CSV/JSON 双向转换工具，降低迁移成本。

**自定义插件钩子** 允许开发者编写自定义过滤脚本，例如按域名黑名单屏蔽、按关键词自动打标或按阅读时长排序。

## 应用场景

技术团队内部知识库维护。开发团队可以使用 LinkBridge 整理项目相关的官方文档链接、社区讨论帖、性能分析报告和故障排查案例，通过统一的元数据格式让新成员快速上手。每周自动运行的链接检查可以及时发现文档站迁移或下线的情况，避免团队依赖失效的外部参考。

技术资讯网站的编辑后台。内容编辑在策划每周技术周报时，通过 LinkBridge 的分类筛选快速定位近期收录的高质量博文或 GitHub 仓库。系统生成的静态页面可以直接作为网站的“推荐阅读”栏目发布，无需开发额外的前端展示逻辑。

个人开发者的学习路径管理。正在学习微服务或云原生技术的工程师可以将散落在各处的教程视频、官方示例、最佳实践指南统一纳入 LinkBridge 管理。借助标签系统，学习者可以按“入门”“进阶”“案例”等阶段组织资源，并定期清理已读完或已过时的条目。

开源社区的活动资源归档。技术会议的组织者可以将历届演讲幻灯片、录制视频链接、参会者笔记等外链通过 LinkBridge 进行编目，生成按年份、按主题分类的会议资料库页面，供社区成员长期查阅。

## 快速开始

以下指令适用于 Linux / macOS / Windows WSL 环境。请确保系统已安装 Git 和 Node.js 18.x 以上版本。

```bash
# 克隆项目仓库至本地
git clone https://github.com/linkbridge/linkbridge-core.git
cd linkbridge-core

# 安装项目依赖（使用 npm 或 yarn）
npm install

# 复制示例配置文件并修改数据库连接等参数
cp .env.example .env

# 运行内置的链接健康检查工具，验证示例数据
npm run check:links

# 启动开发服务器，预览静态站点
npm run dev
```

执行上述步骤后，访问控制台输出的本地地址（通常为 http://localhost:8080）即可浏览示例资源列表。如需导入用户自定义的链接数据，请将符合格式规范的 YAML 文件放入 `./data/entries` 目录后重新执行 `npm run build` 生成新页面。

| Python（可选） | 3.8 或更高 | 仅当启用高级数据分析插件时需要 |

## 项目结构

```
linkbridge-core/
├── bin/                                # 可执行命令行入口
│   ├── cli.js                          # 主命令调度器，解析 process.argv
│   └── commands/                       # 各子命令实现模块
│       ├── check.js                    # 链接健康度检查命令
│       ├── build.js                    # 静态站点构建命令
│       └── import.js                   # 外部书签导入命令
├── lib/                                # 核心业务逻辑库
│   ├── resolver/                       # URL 解析与规范化工具
│   │   ├── normalize.js                # 去除追踪参数、协议补全等
│   │   └── validator.js                # 域名黑名单与格式校验
```

## 贡献指南

本项目的成长依赖于社区成员的资源补充与代码改进。请遵循以下流程提交贡献：

第一，在 GitHub 上 fork 本项目仓库至个人账户，然后 clone 到本地开发环境。建议在贡献前阅读 `docs/data-specification.md` 了解 YAML 条目的字段约束，确保新增的链接数据能够通过 JSON Schema 校验。

第二，新增资源条目时，在 `data/entries` 目录下创建以 `YYYY-MM-DD-<slug>.yaml` 格式命名的文件，其中 slug 为链接标题的简短英文标识。填写所有必填字段（title、url、categories、tags、description、source_accessed_at），并确保 url 字段的值与用户原始数据完全一致，不添加任何协议或路径前缀。

第三，若提交代码层面的改进（例如新增检查策略或优化渲染性能），请同时在 `test/unit` 下补充对应的单元测试用例，并确保执行 `npm run test` 后所有现有测试均保持通过状态。提交信息请遵循 Conventional Commits 规范，使用 `feat:`、`fix:`、`docs:` 等明确前缀。

第四，发起 Pull Request 之前，请先执行 `npm run build` 验证本地构建过程无报错，并且 `npm run check:links -- --all` 能够正常运行。如果新增了环境变量依赖，记得同步更新 `.env.example` 文件。

第五，PR 描述中请清晰说明本次贡献解决了什么问题或增加了什么价值，并关联相关的 Issue 编号（如有）。维护者会在 48 小时内进行 Code Review，可能要求修改或补充文档。

## 常见问题

问：运行链接检查时遇到大量超时错误，如何调整？

答：默认的链接检查超时时间为 5000 毫秒。对于网络状况较差或目标服务器响应较慢的环境，可以在项目根目录下的 `.env` 文件中增加 `CHECK_TIMEOUT=10000` 配置项，将超时阈值调整为 10 秒。同时建议使用 `--concurrency` 参数降低并发请求数量，例如 `npm run check:links -- --concurrency 5` 限制同时只发起 5 个请求。

问：如何将现有的 Chrome 书签或 Raindrop.io 收藏批量导入到 LinkBridge？

答：首先从浏览器书签管理器中导出 HTML 格式的书签文件，或者从 Raindrop.io 导出 CSV 格式数据。然后将该文件放置在项目根目录下的 `./imports` 文件夹中，执行 `npm run import -- --source=chrome --file=bookmarks.html` 命令。系统会自动解析并转换为符合规范的 YAML 条目存入 `data/entries` 目录。导入完成后，建议人工抽查部分条目，手动补充缺失的分类标签或摘要描述。

问：静态站点生成后，部分链接页面显示为空白或缺少样式，如何排查？

答：这类问题通常与资源路径配置有关。请检查 `.env` 中的 `BASE_URL` 环境变量是否设置为正确的站点根路径。如果使用 `npm run dev` 开发模式预览，默认路径为 `/`。若部署到子目录（例如 `https://example.com/linkbridge/`），则需要将 `BASE_URL` 设置为 `/linkbridge/`。此外，确认 `./lib/render/templates` 目录下的 Liquid 模板文件未被修改过语法，可以尝试删除 `./.cache` 和 `./dist` 目录后重新执行构建命令。
