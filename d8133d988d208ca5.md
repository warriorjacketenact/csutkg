# LinkHub Registry

LinkHub Registry 是一个面向技术团队与内容运营者的外链资产统一管理平台，提供结构化的 URL 索引、批量链接校验、快照存档与访问状态监控能力。项目定位为技术资源外链汇总站，适用于需要长期维护大量外部引用链接的文档库、知识库或合规审计场景。

项目核心解决以下问题：外链分散在多个文档中难以统一盘点，第三方页面变更或下线导致引用失效无法及时发现，以及多批次外链导入导出缺乏标准化工具。LinkHub Registry 通过声明式配置、批处理流水线与可扩展的存储后端，帮助用户建立可维护的外链资产清单。

## 功能概览

批量链接导入 支持从 CSV、JSON 或纯文本列表批量导入 URL，自动去重并生成内部唯一标识符。

存活状态探测 基于 HTTP 状态码与响应时间对外链进行周期性可达性检测，标记异常链接。

快照归档存储 对每个外链页面内容计算哈希指纹，支持版本差异比对，便于追溯内容变更。

标签分类管理 支持用户自定义标签体系，对链接按项目、领域、优先级或审核状态进行多维分组。

查询过滤与排序 提供灵活的命令行与 Web 查询接口，可按域名、状态码、最后检测时间等条件筛选。

导出报告生成 支持导出 Markdown、HTML 或 JSON 格式的外链清单报告，包含状态摘要与变更日志。

权限与审计日志 内置基于角色的访问控制，记录所有导入、删除与修改操作，满足合规追溯要求。

## 应用场景

文档站点外链生命周期管理 技术文档团队可使用 LinkHub Registry 维护 API 参考、教程引用及第三方工具链接，每月自动扫描一次全部外链，及时更新或替换失效引用，避免文档读者遇到死链。

开源项目 README 与官网资源整合 开源项目维护者可将项目依赖的生态资源、社区教程、镜像站点等外链统一纳入 LinkHub Registry，在发布新版本前批量校验所有链接的可访问性，提升项目页面的专业度与可用性。

合规审计与法务存档 法务或合规部门需要定期检查对外引用的政策文件、监管公告或合作伙伴声明是否仍然有效且内容未被篡改。LinkHub Registry 的快照哈希功能可为每次审计提供可验证的证据链。

多环境配置管理 运维团队可在开发、测试、预发布环境中分别部署 LinkHub Registry 实例，管理不同环境依赖的外部服务地址，确保配置变更不会引入不可用的外部依赖。

## 快速开始

以下操作步骤适用于 Linux 与 macOS 环境，Windows 用户可使用 WSL 或 Git Bash 执行。

```bash
# 克隆代码仓库
git clone https://github.com/linkhub/linkhub-registry.git
cd linkhub-registry

# 安装项目依赖（使用 Python 虚拟环境）
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 初始化数据库并导入示例链接
python manage.py migrate
python manage.py load_links --input sample_links.json

# 启动开发服务器
python manage.py runserver --host 0.0.0.0 --port 8080
```

访问 http://localhost:8080 即可查看外链管理面板。首次登录默认管理员账号为 admin，密码在启动日志中输出，请及时修改。

## 安装要求

| 依赖 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.10 - 3.12 | 核心运行时，建议使用 pyenv 管理版本 |
| PostgreSQL | 15.x 或 16.x | 主数据库，存储链接元数据与审计日志 |
| Redis | 7.x | 缓存与任务队列后端，用于异步探测任务 |
| Node.js | 20.x LTS | 仅用于前端资源构建，后端可独立运行 |
| Nginx | 1.24+ | 生产环境推荐反向代理与静态文件服务 |
| Docker Compose | 2.20+ | 可选，用于本地快速启动全套服务栈 |

## 项目结构

```
linkhub-registry/
├── cmd/                            # 命令行入口
│   ├── server/                     # Web 服务启动入口
│   │   └── main.go                 # 主服务进程
│   └── worker/                     # 异步任务工作进程
│       └── main.go                 # 探测任务消费者
├── internal/                       # 内部核心包，不对外暴露
│   ├── config/                     # 配置加载与校验（支持 YAML + 环境变量）
```

## 常见问题

Q: 导入大量链接后，首次状态探测非常耗时，如何加快速度？

A: 系统默认使用 Redis 任务队列并发执行探测任务。您可以通过配置文件调整 worker 数量（probe.concurrency 参数）来增加并发度。建议根据网络带宽和目标服务器负载合理设置，通常 20-50 个并发进程即可获得较优吞吐量。同时，系统支持仅检测状态码（不获取完整内容）的轻量模式，可通过 --head-only 参数启用。

Q: 外链页面内容发生变化时，系统如何通知我？

A: 系统在每次探测时会计算页面内容的哈希值并与上次记录比对。当哈希值不一致时，会在审计日志中记录 content_changed 事件。您可以通过配置 Webhook 或邮件通知插件（需自行安装对应扩展）接收变更告警。默认情况下，变更信息会显示在管理后台的链接详情页的变更历史标签中。

Q: 生产环境部署后，数据库连接频繁超时怎么办？

A: 请检查 PostgreSQL 的 max_connections 配置以及应用层的连接池设置。推荐使用 pgbouncer 作为连接池代理，并在 application.yaml 中调整 pool.max_open 和 pool.max_idle 参数。同时确保 Redis 和数据库部署在同一内网区域以减少网络延迟。详细调优参数请参考 docs/ops/deployment.md 中的性能调优章节。
