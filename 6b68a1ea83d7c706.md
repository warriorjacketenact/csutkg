# ResourceBridge

ResourceBridge 是一个面向开发者和技术研究人员的开源外链资源聚合与导航系统。该项目并非传统的搜索引擎或书签管理器，而是一个以“资源关系图谱”为核心的元数据索引中间件。它通过结构化的方式将散落在互联网各处的技术文档、工具链、学习资料和社区讨论进行归类、标注和关联，帮助用户在特定技术领域内快速建立知识坐标系。

项目定位为“技术资源的引水渠”，目标用户包括正在学习新技术栈的初级开发者、需要频繁查阅碎片化资料的高级工程师，以及希望系统化整理团队知识库的技术管理者。ResourceBridge 不存储任何实际文件内容，仅提供资源定位、分类标签、摘要提取和关联推荐功能，通过用户自定义的标签体系和内置的智能相似度算法，将孤立的外链转化为可检索、可追溯、可扩展的个人或团队知识网络。

## 功能概览

- **多级分类与标签引擎**：支持为每条外链分配无限数量的自定义标签，并创建多级分类目录。系统提供标签云和分类树两种可视化导航方式，便于从宏观主题逐步下钻至具体资源。

- **全文摘要与元数据提取**：自动抓取目标页面的标题、描述、关键词及主要正文段落，生成标准化摘要。支持手动编辑摘要内容，并记录资源类型（文档、视频、仓库、工具等）、语种、作者或发布机构等元数据字段。

- **资源关系图谱**：基于标签共现、分类归属和用户手动设置的“相关链接”字段，构建资源之间的双向关联。图谱支持力导向布局可视化，帮助发现跨领域、跨主题的隐性连接。

- **检索与过滤系统**：提供基于标题、摘要、标签、分类、域名和资源类型的多条件组合检索。检索结果支持按相关性、更新时间或热度（点击量）排序，并具备拼写容错和同义词扩展能力。

- **批量导入与导出**：支持通过 CSV、JSON 和 OPML 格式批量导入现有书签或外链列表。导出功能支持生成 Markdown 表格、结构化 JSON 以及适用于静态站点生成器的数据文件，便于迁移或公开分享。

- **协作与审核工作流**：内置多用户权限管理，支持创建“资源提交池”。团队成员可提交新外链，由管理员或高级编辑进行审核、分类和标签修正后正式入库，保证资源质量。

- **定时健康检查**：后台调度任务定期对已存储的外链发起 HEAD 请求，检测链接可用性（HTTP 状态码）。对于返回 4xx 或 5xx 状态的链接，自动标记为“失效”并发送通知，支持导出失效链接列表以供人工处理。

## 应用场景

- **技术选型调研**：当团队需要评估多个中间件或框架时，可快速建立对比资源集合。例如，在调研消息队列（MQ）方案时，将 Kafka、RabbitMQ、Pulsar 的官方文档、性能测试报告、社区最佳实践和故障排查案例归入同一分类，通过标签区分“稳定性”“吞吐量”“运维复杂度”等维度，辅助决策。

- **新员工技术 onboarding**：企业可将内部技术栈相关的入门教程、规范文档、常用工具站点和典型问题排查记录整合为“新人必读”分类。新员工通过浏览分类树和标签云，能够系统性地了解团队技术全貌，减少零散搜索时间，加速融入开发流程。

- **开源项目依赖追踪**：维护者可使用 ResourceBridge 记录项目所依赖的库、服务、工具链及其官方网站、文档入口和社区讨论区。当依赖项版本升级或出现安全漏洞时，可通过关联标签快速筛选出受影响的所有内部项目或模块，提升响应效率。

- **学术研究文献管理**：研究人员可围绕特定课题（如分布式共识算法、深度学习模型量化）收集论文预印本、开源实现仓库、实验数据集和评测榜单。通过关系图谱可视化，能够直观识别该领域内的核心工作、衍生工作以及前沿方向，辅助撰写文献综述。

## 快速开始

以下步骤指导您在本地环境中快速启动 ResourceBridge 开发实例。请确保已安装 Git、Node.js（v18 或以上）和 pnpm（或 npm）。

```bash
# 克隆代码仓库
git clone https://github.com/resourcebridge/resourcebridge.git

# 进入项目目录
cd resourcebridge

# 安装项目依赖
pnpm install

# 复制环境变量配置文件模板，并根据您的数据库配置进行修改
cp .env.example .env

# 执行数据库迁移脚本，初始化数据表结构
pnpm run migrate

# 启动开发服务器（默认监听 http://localhost:3000）
pnpm run dev
```

启动成功后，访问 http://localhost:3000 即可进入 ResourceBridge 的仪表盘界面。首次运行会自动创建默认管理员账户（用户名: admin，密码: admin123），请务必在首次登录后通过个人设置页面修改密码。

## 安装要求

生产环境部署前，请确认以下依赖均已正确安装并满足版本要求。开发环境可酌情放宽部分版本限制，但不保证所有功能正常运行。

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | v18.17.0 或更高 | 运行时环境，推荐使用 LTS 版本。v20 及以上版本同样受支持。 |
| PostgreSQL | v14.x 或 v15.x | 主数据库，用于存储资源元数据、用户信息、标签和分类关系。不支持 SQLite 或 MySQL。 |
| Redis | v7.x 或更高 | 缓存与会话存储，用于提升检索响应速度并管理用户登录状态。如仅单机开发，可暂时禁用缓存模块。 |
| pnpm | v8.x 或更高 | 包管理器，用于安装依赖和管理 monorepo 工作空间。npm 或 yarn 未经过测试，不推荐使用。 |
| Git | v2.30 或更高 | 版本控制系统，用于克隆仓库和管理代码更新。 |
| 环境变量配置文件 | 必须存在 | 项目根目录下的 .env 文件需包含数据库连接串、Redis 连接串、JWT 密钥等关键配置项，缺失会导致启动失败。 |

## 项目结构

ResourceBridge 采用 monorepo 组织方式，核心代码位于 `packages/` 目录下，辅以部署脚本和文档。以下为项目主要目录及文件说明。

```
resourcebridge/
├── apps/
│   ├── web/                         # 主 Web 应用（Next.js 仪表盘）
│   │   ├── src/
│   │   │   ├── app/                 # Next.js App Router 页面路由
│   │   │   ├── components/          # 可复用的 UI 组件（Shadcn/ui）
│   │   │   ├── lib/                 # 工具函数、API 客户端封装
│   │   │   └── types/               # TypeScript 类型定义
│   │   ├── public/                  # 静态资源（图标、字体）
│   │   └── next.config.js           # Next.js 构建配置
│   └── api/                         # 后端 API 服务（Express + Prisma）
│       ├── src/
│       │   ├── routes/              # RESTful 路由定义（按资源实体划分）
│       │   ├── controllers/         # 请求处理逻辑
│       │   ├── services/            # 业务服务层（标签、检索、健康检查）
│       │   ├── middlewares/         # 鉴权、日志、限流中间件
│       │   ├── models/              # Prisma ORM 数据模型映射
│       │   └── utils/               # 通用工具（摘要抓取、链接校验）
│       └── prisma/
│           ├── schema.prisma        # 数据库表结构定义
│           └── migrations/          # 迁移历史文件

│   ├── shared-types/                # 前后端共享的 TypeScript 类型
│   ├── crawler/                     # 独立的外链元数据抓取微服务
```

## 贡献指南

ResourceBridge 欢迎并感谢任何形式的贡献，包括但不限于提交新资源索引、改进文档、报告缺陷和开发新功能。请遵循以下步骤参与项目：

1. 查阅问题追踪器：访问 GitHub Issues 页面，筛选带有 `help wanted` 或 `good first issue` 标签的任务。如果您发现未报告的缺陷或有新功能建议，请先搜索是否已存在相似议题，避免重复创建。

2. 派生仓库并创建特性分支：将主仓库派生至您的个人账户，然后克隆至本地。请基于 `main` 分支创建新的特性分支，分支命名建议使用 `feature/功能简述` 或 `fix/缺陷编号` 格式。

3. 编写代码并添加测试：所有新功能或缺陷修复应包含相应的单元测试或集成测试。请确保现有测试套件全部通过，并在提交前运行 `pnpm run lint` 和 `pnpm run format` 以统一代码风格。

4. 提交变更并签署开发者原创声明：提交信息请遵循 Conventional Commits 规范（如 `feat: 增加批量导入 OPML 功能`）。提交时需签署 Developer Certificate of Origin，表示您有权并同意将代码贡献至本项目。

5. 发起拉取请求：推送到您的派生仓库后，向主仓库的 `main` 分支发起拉取请求。请在请求描述中清晰说明变更内容、关联的议题编号以及测试覆盖情况。至少一名项目维护者将审核您的代码，审核通过后即可合并。

## 常见问题

**问：ResourceBridge 是否会对收录的外链进行内容缓存或代理？**

答：不会。ResourceBridge 仅存储外链的元数据（标题、摘要、标签、分类等），从不缓存或代理目标页面的完整内容。用户点击资源链接时，浏览器将直接访问原始 URL。健康检查功能仅发送 HEAD 请求验证链接可达性，不会下载响应体。如需存档功能，请考虑配合第三方网页归档服务使用。

**问：如何迁移已录入的资源数据至另一套 ResourceBridge 实例？**

答：您可以使用内置的导出功能，选择 JSON 格式导出全部资源及其元数据。导出的 JSON 文件包含完整的分类、标签和关联信息。在目标实例中，通过管理后台的“批量导入”功能上传该文件即可完成迁移。请注意，导出文件不包含用户账户信息和访问统计，这两类数据需通过数据库原生工具（如 pg_dump）进行迁移。

**问：标签系统支持层级关系吗？**

答：标签本身是扁平的字符串，不支持父子层级。但您可以通过“分类”功能实现层级结构，分类支持无限级子分类嵌套。我们建议将分类用于宽泛的主题划分（如“数据库”“消息队列”），将标签用于更细粒度的属性标记（如“高可用”“低延迟”“金融级”），两者配合使用可获得最佳的检索和组织体验。
