# LinkCatalog

LinkCatalog 是一个面向技术团队与内容创作者的轻量级外链资源聚合与导航系统。该项目定位于解决多源技术文档、教程、API 参考与社区文章分散管理的问题，通过结构化的资源索引机制，帮助用户快速定位、分类与复用高质量的外部技术内容。

LinkCatalog 不提供内容存储服务，而是作为智能资源路由层，将分散于不同平台的系列技术文章、版本更新日志与专题讨论统一收录，并提供标签过滤、全文检索与时效性监控能力。项目适用于技术文档站点、内部开发者门户、开源项目推荐位以及个人知识库的外链管理。

## 功能概览

**资源批量导入** 支持通过 CSV、JSON 或 URL 列表批量导入外部链接，自动解析页面标题、描述与元数据。

**智能分类与标签** 基于页面内容与 URL 特征自动生成分类标签，支持用户自定义标签体系与多级目录映射。

**时效性监控** 定期检查已收录链接的可用性与内容更新状态，对失效链接或重大版本变更发出提醒。

**全文检索** 基于倒排索引实现对资源标题、描述、标签及页面摘要的快速检索，支持布尔查询与模糊匹配。

**访问统计与分析** 记录每个资源的点击次数、引用来源与访问趋势，提供可视化面板用于评估资源热度。

**公开 API 接口** 提供 RESTful API 用于外部系统集成，支持资源的增删改查与分类树遍历，便于嵌入现有管理后台。

**数据快照与回滚** 每日自动生成资源索引快照，支持按时间点回滚至历史状态，防止误操作导致数据丢失。

## 应用场景

技术文档站点外链管理 技术写作团队可使用 LinkCatalog 统一管理文档中引用的外部 API 参考、SDK 下载页与社区教程，确保所有外链在版本发布前均经过可用性检查。

开发者门户资源导航 企业内部开发者门户可将 LinkCatalog 作为外链聚合层，集中展示各类中间件文档、监控面板地址与运维手册，提升团队查找效率。

开源项目推荐位聚合 开源项目维护者可使用 LinkCatalog 收集社区中与项目相关的优秀博文、视频教程与衍生项目，形成生态资源列表，降低新贡献者的学习门槛。

个人知识库外链索引 知识管理爱好者可将浏览器收藏夹中的技术文章按主题批量导入，配合标签与检索功能构建个人外链知识图谱。

## 快速开始

以下指令适用于 Linux / macOS 与 Windows WSL 环境。请确保系统已安装 Git 与 Node.js 18 以上版本。

```bash
git clone https://github.com/linkcatalog/linkcatalog.git
cd linkcatalog
npm install
npm run build
npm start
```

执行完毕后，服务默认监听 3000 端口。访问 http://localhost:3000 即可进入 LinkCatalog 管理控制台。首次启动将自动创建默认管理员账户，用户名与密码输出于控制台日志中。

如需使用 Docker 部署，可执行以下命令：

```bash
docker build -t linkcatalog:latest .
docker run -p 3000:3000 -v ./data:/app/data linkcatalog:latest
```

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | 18.x 或 20.x LTS | 运行时环境，建议使用官方预编译二进制 |
| npm | 9.x 或 10.x | 包管理器，用于安装项目依赖 |
| SQLite3 | 系统自带或通过 npm 安装 | 嵌入式数据库，用于存储资源索引与元数据 |
| Git | 2.30 以上 | 用于克隆仓库及版本管理 |
| 系统内存 | 最低 512 MB，推荐 2 GB | 内存不足时检索性能将显著下降 |
| 磁盘空间 | 最低 1 GB | 用于存储数据库文件、日志与快照备份 |
| 网络连接 | 出站 443 端口开放 | 用于首次解析资源页面标题与描述 |
| 操作系统 | Linux / macOS / Windows WSL2 | 生产环境建议使用 Debian 11 或 Ubuntu 22.04 |

我们欢迎并鼓励社区贡献。请遵循以下步骤提交代码或文档改进：

1. 查阅问题列表 访问 GitHub Issues 页面，筛选带有 help-wanted 或 good-first-issue 标签的任务。如计划实现新功能，请先创建 Issue 说明设计思路，避免重复劳动。

2. 派生仓库并创建功能分支 将主仓库派生至个人账户，随后克隆至本地。创建以 feature/ 或 fix/ 为前缀的分支，例如 feature/add-import-export。

3. 编写测试与代码 所有新增功能必须包含对应的单元测试。代码风格遵循 ESLint 配置（基于 Airbnb 规范）。提交前执行 npm run lint 与 npm run test 确保无报错。

4. 提交变更说明 提交信息采用 Conventional Commits 格式，即 type(scope): description。类型包括 feat、fix、docs、style、refactor、perf、test、chore。

5. 发起拉取请求 将分支推送至派生仓库后，在主仓库发起 Pull Request。描述中需关联相关 Issue 编号，并附上变更摘要与测试结果截图（如涉及 UI 改动）。

## 常见问题

问题一：首次启动后无法访问管理控制台，页面显示 502 错误。

解答：该问题通常由端口冲突或数据库初始化失败引起。请检查 3000 端口是否被其他进程占用（可使用 lsof -i :3000 命令排查）。若端口空闲，请查看控制台日志中的错误堆栈，确认 data/db 目录具有写入权限。SQLite 在权限不足时会抛出构建错误。

问题二：导入包含 200 个以上链接的 CSV 文件时，部分链接解析失败并标记为 unreachable。

解答：LinkCatalog 默认对每个链接设置 5 秒超时与 3 次重试。若您的网络环境访问外部站点较慢，可在 config/default.yaml 中调整 crawler.timeout 与 crawler.retries 参数。此外，部分站点可能启用反爬机制，请确保 User-Agent 配置合法。

问题三：如何将现有数据库迁移至另一台服务器？

解答：直接复制 data/db 目录下的 SQLite 文件至新服务器的相同相对路径即可。若需切换数据库类型（例如 PostgreSQL），请参考 docs/operations.md 中的迁移章节，使用内置的迁移脚本导出为 SQL 转储文件再导入。
