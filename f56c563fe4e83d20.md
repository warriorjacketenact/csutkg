# WebResource Nexus

WebResource Nexus 是一个面向技术研究人员、网络数据分析师以及数字内容管理者的外链资源归集与导航系统。该项目不生产原始内容，而是通过结构化的方式对分散在网络各处的技术文档、案例页面和参考资料进行系统性收录、分类和索引，帮助用户快速定位所需信息，减少在分散来源中反复检索的时间成本。

本项目定位为纯粹的技术资源外链汇总站，适用于需要频繁查阅外部参考材料、维护个人知识库或构建团队共享资源导航的场景。通过统一的条目管理和分类体系，用户可以基于既定的目录结构快速理解资源分布，并借助本项目提供的辅助工具实现链接有效性检测、元数据提取和批量导出等操作。

## 功能概览

**链接归集与分类管理**：支持按照自定义标签和层级目录对海量外链进行归类，每个条目可关联备注、优先级和状态标记，便于后续筛选和整理。

**批量导入与去重检测**：提供从文本文件、CSV 表格和浏览器书签导出文件批量导入链接的功能，自动检测重复条目并生成合并建议报告。

**有效性巡检与状态监控**：内置异步 HTTP 探测模块，可定期检查每条链接的可访问性，自动标记失效、重定向或响应超时的条目，并生成巡检日志。

**元数据自动提取**：对支持 Open Graph 和 JSON-LD 的页面，自动抓取标题、描述和主要图片，减少手工录入工作量，提升资源描述的准确性。

**多格式导出与分享**：支持将选中的资源列表导出为 HTML 书签文件、Markdown 列表、JSON 结构化数据以及 CSV 表格，便于嵌入其他文档或与团队协作。

**检索与过滤面板**：提供基于关键词、所属分类、添加时间和状态的多维度筛选接口，支持模糊匹配和精确匹配两种模式，结果可排序和分页展示。

**自定义视图与仪表盘**：用户可根据使用习惯配置首页展示模块，包括最近新增、待检查链接、高频访问分类等，所有配置保存在本地，不依赖服务端存储。

## 应用场景

技术文档聚合与查阅：研发团队可将日常开发中涉及的 API 文档、框架指南、故障排查案例等外链统一收录至本项目，按技术栈分类后，新成员可快速了解团队常用的参考资源体系，缩短上手周期。

数据采集与信息监控：数据分析师可将需要定期查看的数据报表平台、行业资讯源和公开数据集链接集中管理，利用本项目的有效性巡检功能及时发现访问异常，确保数据采集链路稳定。

个人知识库外链整理：独立研究者或内容创作者在阅读文献、观看技术分享或调研竞品时，将散落在浏览器收藏夹和笔记软件中的链接导入本项目，通过补充备注和分类标签形成可检索的外部知识索引。

团队共享资源导航：部门或项目组可部署本项目作为内部导航页的底层管理端，由专人维护常用工具、设计规范、测试环境和内部系统的入口链接，其他成员通过只读视图访问，减少口头询问和文档查找时间。

## 快速开始

以下命令适用于 Linux 和 macOS 环境，Windows 用户建议使用 WSL2 或 Git Bash 执行。

```bash
# 克隆代码仓库
git clone https://github.com/webresource-nexus/nexus-core.git
cd nexus-core

# 安装项目依赖（需要 Node.js 18+ 和 npm 9+）
npm install

# 以开发模式启动本地服务，默认监听 127.0.0.1:3400
npm run dev
```

启动后，在浏览器中访问控制台首页，首次运行将自动创建示例分类和演示数据。如需重置数据，可执行 `npm run reset-db`。

## 常见问题

问：巡检功能报告大量链接超时，但手动访问浏览器中可正常打开，如何解决？

答：此问题通常由服务器端的反爬策略或速率限制引起。建议在配置文件中调整探测并发数（默认值为 5）和单次请求超时时间（默认值为 8000 毫秒），同时启用 `respect-robots` 选项以遵循目标站点的爬虫协议。若目标站点有严格的访问控制，可在 `probe-engine` 中配置自定义 User-Agent 和 Referer 头。

问：导入浏览器书签 HTML 文件时，部分链接的分类和标签未能正确映射，应如何处理？

答：本项目默认按照书签文件夹名称作为分类依据，但嵌套层级超过两层的文件夹结构会被扁平化处理。若需要保留完整的层级关系，建议在导入前将书签整理为平铺结构，或使用 CSV 模板手动填写分类列。导入后，可通过批量编辑功能对未分类条目进行统一调整。

问：开发模式下启动服务正常，但构建生产版本后静态资源返回 404，是什么原因？

答：生产构建会启用静态资源指纹哈希和路径重写，若反向代理或 Web 服务器未正确配置 `public` 目录的访问路径，可能导致资源定位失败。请检查 `config/production.yaml` 中的 `staticBase` 配置项，确保其值与实际的 URL 前缀一致。若使用 Docker 部署，需确认容器内构建产物已完整复制至镜像的 `/app/dist` 目录。
