# WebLink Navigator

WebLink Navigator 是一个面向技术研究者、数据采集工程师和内容聚合平台的开源外链资源导航系统。该项目旨在解决海量分散 URL 的归集、分类、可用性监控与快速检索问题，帮助用户从无序的链接堆叠中构建出具备语义结构的可访问资源图谱。

本项目并非一个简单的书签管理工具，而是一套完整的资源链接生命周期管理方案。它包含链接抓取调度、状态探测、元数据提取、分类标签生成以及前端检索界面等模块，适用于需要长期维护大量外链资源的个人知识库或团队协作场景。

## 功能概览

**批量链接导入** 支持从纯文本、CSV 或 JSON 格式批量导入 URL 列表，自动去重并识别域名归属。

**可用性主动探测** 内置异步 HTTP 探测器，可配置超时与重试策略，定期对资源链接进行可达性检查，标记失效节点。

**元数据智能提取** 从目标页面自动解析标题、描述、关键词及主要内容类型，为后续检索提供结构化字段。

**多级分类标签** 允许用户自定义分类体系，支持为每个链接打上多个标签，并基于标签组合进行快速筛选。

**全文检索与过滤** 基于标题、描述、标签和 URL 路径的轻量级全文检索引擎，支持模糊匹配与布尔操作。

**链接状态历史追踪** 记录每次探测的时间戳与响应状态码，生成链接健康度变化趋势图，便于运维决策。

**数据导入导出** 支持将全量资源数据导出为 Markdown 表格、JSON 或 SQLite 数据库文件，便于迁移或备份。

## 应用场景

**技术文档归档** 技术团队可将日常阅读的 API 文档、教程文章、官方公告等链接统一录入系统，通过标签区分技术栈与优先级，避免知识碎片化。

**数据采集任务管理** 数据工程师在编写爬虫前，可使用本系统整理目标站点的入口 URL、分页规则和反爬策略备注，形成结构化的采集清单，提高任务可复现性。

**内容聚合站外链库** 内容运营人员可维护一份高质量的外部引用链接库，在撰写稿件或编辑推荐时快速检索匹配的参考资料，提升内容产出效率。

**开源项目依赖追踪** 开源维护者可将项目依赖的第三方库文档、社区论坛、问题跟踪地址统一登记，当依赖库版本升级或官网迁移时，通过探测功能及时发现链接变动。

## 快速开始

以下指令适用于 Linux / macOS / Windows WSL 环境，需提前安装 Git 与 Node.js 16+。

```bash
# 克隆项目仓库
git clone https://github.com/weblink-navigator/weblink-navigator.git

# 进入项目目录
cd weblink-navigator

# 安装依赖
npm install

# 启动开发服务器
npm run dev
```

访问控制台输出的本地地址（默认为 http://127.0.0.1:3000）即可进入系统首页。首次启动会自动创建示例数据并执行初始化迁移。

## 贡献指南

项目采用 GitHub Flow 协作模型，所有贡献均通过 Pull Request 合并至 main 分支。请遵循以下步骤：

1. 在 GitHub 上 fork 本仓库至个人账户，随后克隆到本地开发环境。确保本地安装的 Node.js 版本与项目要求一致。

2. 创建新的功能分支，分支命名采用 `feature/功能简述` 或 `fix/问题简述` 格式。在本地进行代码修改时，请保持代码风格与 ESLint 配置一致，并为新增功能编写对应的单元测试。

3. 提交代码前运行 `npm run test` 确保所有现有测试通过，运行 `npm run lint` 修复代码风格问题。提交信息请使用英文，采用 imperative mood，首字母大写，长度不超过 72 字符。

4. 推送分支至个人远程仓库，随后在 GitHub 上向本仓库的 main 分支发起 Pull Request。请求描述中需明确说明变更目的、影响范围及测试情况，至少一位项目维护者审核通过后方可合并。

5. 若新增依赖或修改了数据库 Schema，请在 PR 中同步更新 docs 目录下的对应文档，并确保 `package.json` 中的依赖版本锁定为具体数值。

## 常见问题

Q: 探测引擎是否会频繁触发目标服务器的限流机制？

A: 系统默认采用指数退避重试策略，初始并发数设为 5，且每个目标域名在 60 秒窗口内最多发起 30 次请求。用户可在配置文件中调整 `concurrency` 与 `rateLimit` 参数以适应不同目标站点的承受能力。对于已知的严格限流站点，建议手动标记为低优先级探测组。

Q: 导入大量链接后，界面检索响应变慢，如何优化？

A: 检索性能主要受限于 SQLite 的全文索引。建议在导入完成后执行 `npm run build-index` 命令重建 FTS 虚拟表。对于超过 10 万条记录的场景，可考虑切换至 PostgreSQL 作为生产数据库，项目已提供适配器接口，相关迁移指南参见 docs/ops/scaling.md。

Q: 能否与现有的监控告警系统（如 Prometheus）集成？

A: 项目在 `/metrics` 端点暴露了 Prometheus 格式的统计数据，包括总链接数、失效链接数、平均探测耗时等指标。运维人员可在部署时将该端点接入 Prometheus 的 scrape 配置，配合 Grafana 实现可视化监控面板。
