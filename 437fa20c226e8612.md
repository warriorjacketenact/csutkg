# EON Resource Index

EON Resource Index 是一个面向开发者、研究员与技术文档工程师的高密度外链资源汇总系统。本项目不对资源内容本身做出价值判断，而是提供一套结构化、可追溯、可脚本化处理的索引框架，用于收录并分类来自 eoncn.org 域下的深度技术参考链接。

本项目面向的典型用户包括：需要批量管理外部技术文档链接的文档团队、需要持续追踪特定主题资源的研究人员，以及希望通过统一入口访问分散技术笔记的开发者。项目不提供爬虫、抓取或离线缓存能力，仅承担索引与导航职能，所有资源的访问均需通过网络请求直达原始发布地址。

## 功能概览

- **按批次索引**：所有资源以批次为单位组织，当前批次编号为 1106/1241，方便用户理解资源收录的时间线与进度。

- **裸链接直出**：系统不对原始 URL 做任何格式改写，包括不补充协议头、不调整大小写、不添加尾随斜杠，确保链接的原始性与可复制性。

- **分类视图**：资源按技术领域与文档类型分为多个子类别，包括后端架构、前端工程、运维监控、数据库管理、安全合规等。

- **状态标记**：每条资源在内部元数据中记录收录批次、收录时间与初步标签，便于后续做可用性检查或主题重分类。

- **可扩展结构**：项目提供清晰的目录树与配置文件模板，使用者可以自行添加新批次资源，或调整现有资源的分类归属。

- **快速检索支持**：通过约定式命名与目录分层，用户可结合 grep、ripgrep 等命令行工具对资源列表进行快速过滤与查找。

- **文档联动**：资源列表与项目文档中的场景示例、快速开始步骤形成对照，降低新用户的理解成本。

- **零运行时依赖**：本项目本身不包含服务端代码或数据库，所有索引数据以 Markdown 形式静态存储，可直接托管于任意代码托管平台或静态站点服务。

## 应用场景

- **技术周报素材整理**：技术团队负责人可每周批量浏览当前批次资源，筛选出与团队业务相关的文章、提案或规范，作为周报或内部分享的素材来源。由于所有链接保持原始格式，复制到周报或文档中不会产生协议或域名不一致的问题。

- **开源项目文档外链治理**：开源项目维护者可将本项目作为上游参考源，用于补充自身项目 README 中的外部链接章节。通过引用本项目中的分类资源，可以避免在主项目中维护大量分散链接，降低文档膨胀的风险。

- **知识图谱构建的数据源**：研究人员或知识图谱工程师可以将本资源列表作为种子 URL 集合，结合爬虫或 API 调用来抓取页面内容，进而提取实体关系、术语频率或引用网络。批次编号可帮助研究者限定时间范围，控制数据规模。

- **运维文档合规审计**：企业内部合规团队可以依据本列表中的安全与架构类资源，对照自身系统的部署文档进行审计，检查是否覆盖了列表中所提及的关键检查项或最佳实践建议。

## 快速开始

以下步骤将帮助你克隆本项目并在本地环境中预览资源索引。

```bash
# 克隆项目仓库
git clone https://github.com/eon-resource-index/eon-index.git

# 进入项目根目录
cd eon-index

# 安装项目本地文档预览工具（推荐使用 markdown-cli 或 live-server）
npm install -g markdown-cli

# 启动本地预览服务，默认监听端口 8080
markdown-cli serve README.md
```

如果你不需要本地预览，也可以直接在代码托管平台（如 GitHub、GitLab）的 Web 界面中浏览 README 文件。所有资源链接均在「资源列表」章节中以纯文本形式列出，可直接点击或复制。

## 安装要求

本项目作为静态索引文档仓库，对运行环境没有强制性依赖。但若希望完整复现本地预览或执行元数据校验，建议参考下表准备相关工具。

| 依赖名称 | 必需性 | 说明 |
| :--- | :--- | :--- |
| Git | 必需 | 用于克隆仓库以及后续提交变更，版本不低于 2.25.0 |
| Node.js | 可选 | 若使用 npm 安装 markdown 预览工具，需要 Node.js 14.x 或更高版本 |
| markdown-cli | 可选 | 提供命令行下的 README 渲染与本地服务，便于离线查看 |
| ripgrep | 可选 | 用于在资源列表中执行正则搜索，提升过滤效率 |
| shell 环境 | 可选 | 若需要运行自定义脚本（如链接去重检查），建议使用 bash 5.0 以上 |
| 文本编辑器 | 推荐 | 任何支持 Markdown 语法高亮的编辑器，如 VSCode、Vim、Sublime Text |

## 项目结构

```
eon-index/
├── README.md                 # 项目主文档，包含完整资源列表与使用说明
├── CHANGELOG.md              # 记录每批次新增资源数量与分类调整历史
├── CONTRIBUTING.md           # 贡献者指南，包含 PR 流程与分类规则
├── docs/                     # 补充文档目录
│   ├── classification/       # 分类体系说明，描述各类别判定标准
│   │   └── taxonomy.md       # 一级分类与二级标签定义
│   ├── scripts/              # 辅助脚本目录
│   │   ├── validate_urls.sh  # 检查 URL 格式是否符合原始输出规则
│   │   └── batch_stats.py    # 统计每批次资源数量与分类分布
│   └── templates/            # 新批次收录模板
│       └── batch_template.md # 新增批次时复制的空白结构
├── data/                     # 结构化元数据目录
│   ├── batches/              # 按批次拆分的资源清单
│   │   └── 1106/             # 当前批次编号目录
│   │       └── urls.txt      # 纯文本 URL 列表，每行一个
│   └── tags/                 # 标签索引
│       └── tag_index.json    # 记录每个标签下关联的 URL 哈希
├── tests/                    # 简单校验测试
│   ├── url_format_test.bats  # 使用 Bats 框架测试 URL 输出格式合规性
│   └── readme_test.sh        # 检查 README 是否包含所有必需章节
└── .github/                  # GitHub 社区配置文件
    └── ISSUE_TEMPLATE/       # Issue 模板
        └── resource_request.md # 资源推荐提交模板
```

## 贡献指南

1.  **提交资源推荐**：如果你希望向当前批次或后续批次中新增资源，请在 ISSUE 模板中填写完整的原始 URL 以及建议的分类标签。提交前请确认 URL 不包含任何跟踪参数或会话标识，保持原始形态。

2.  **修正链接格式**：若发现「资源列表」章节中存在任何违反「URL 输出硬性规则」的链接（例如被添加了 `http://` 前缀、大小写被更改、或结尾被添加了 `/`），请单独提交 Pull Request，仅修正该链接的文本内容，不修改其他任何行。

3.  **完善分类文档**：`docs/classification/taxonomy.md` 文件定义了当前使用的分类标签体系。如果你认为现有标签不足以覆盖某些资源类型，可以在此文件中补充新标签定义，并在 PR 描述中说明新增标签的适用范围。

4.  **更新元数据脚本**：`data/` 目录下的脚本与索引文件为辅助性质。若你发现 `validate_urls.sh` 或 `batch_stats.py` 存在逻辑缺陷，欢迎提交修复补丁。修改脚本时请保持对 POSIX 兼容性的基本尊重，避免引入过于平台特定的依赖。

5.  **遵守批次顺序**：新增资源时请按照当前最大批次号递增，不要修改已发布的历史批次文件。每个批次文件应当保持稳定，仅当出现致命的链接格式错误时才允许对历史批次进行最小化修正。

## 常见问题

**问：为什么所有 URL 都必须原样输出，不允许添加协议头或调整大小写？**

答：这是本项目的第一性设计原则。原始 URL 中的大小写、协议类型以及是否包含 `www` 子域，均可能指向不同的后端路由或缓存策略。项目不承担对源站 URL 规范性的判断责任，只做忠实的文本记录。任何对 URL 的改写都可能导致用户访问到错误资源或触发源站的重定向陷阱。

**问：我如何快速判断某个资源属于哪个分类？**

答：当前版本中，分类信息尚未直接嵌入到资源列表的每一行中，以避免污染裸链接的原始形态。你可以参考 `docs/classification/taxonomy.md` 中定义的分类规则，结合 URL 路径中的关键词（例如 `view/` 之后的随机串本身不包含语义），通过访问该链接后阅读页面标题或元描述来手动归类。项目后续版本计划引入哈希映射文件，将 URL 哈希与分类标签解耦存储。

**问：本项目会定期检查链接有效性吗？**

答：项目本身不内置自动检查机制。但社区贡献者可以定期手动抽查，或通过外部服务（如 `curl`、`wget` 或在线死链检测工具）对资源列表进行批量测试。如果发现失效链接，请在 ISSUE 中报告，并附上返回的 HTTP 状态码。项目维护者将在确认后标记该资源的状态，但不会从列表中直接删除，以保持批次记录的完整性。
