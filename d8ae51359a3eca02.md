# HyperLink Collective

HyperLink Collective 是一个面向技术研究者、内容策展人与数据管道开发者的高性能外链聚合与规范化管理工具。项目定位为“链接的链接”，不生产内容，不渲染页面，仅提供结构化的 URL 收录、分类、校验与导出能力，适用于需要批量处理非结构化 URL 数据的自动化工作流。

本项目并非传统爬虫或书签管理器，而是一套围绕 URL 原始字符串的轻量级治理框架。核心目标用户包括：维护大型导航站点的站长、构建知识图谱的语义网工程师、以及需要定期同步第三方资源列表的 DevSecOps 团队。HyperLink Collective 默认不对任何 URL 发起主动请求，仅提供基于规则的分类、去重、格式校验与批次标记，确保原始数据在流转过程中不被污染。

## 功能概览

**原始 URL 透传存储** 所有输入的 URL 字符串以完全原始形态存入数据库，不进行任何协议补全、大小写转换或末尾斜杠修剪，保留用户提交时的每一个字符。

**基于路径模式的自动分类** 根据 URL 路径中的关键词（如 /vod/、/episode/）自动划分资源类型，支持自定义规则引擎扩展，便于后续按业务域检索。

**批次与序号标记** 每个 URL 自动关联所属批次编号（当前为第 1021/1241 批）及行内序号，便于追溯数据来源与执行增量更新。

**格式合规性校验** 检测裸域名与带协议 URL 的混合输入，标记不符合预期格式的条目，但不强制修改，仅生成警告日志。

**Markdown 清单导出** 将收录的 URL 按类别分组，生成符合项目 README 规范的纯文本列表，直接用于文档发布或静态站点生成。

**多源合并去重** 支持从不同渠道导入的 URL 列表进行并集运算，基于字符串完全匹配剔除重复项，保留首次出现的批次信息。

## 应用场景

**技术导航站点的资源库初始化** 站点维护者可通过本项目批量导入零散的参考链接，自动完成分类与编号，生成统一的资源列表章节，避免手工整理导致的遗漏或格式错误。

**数据管道中的 URL 规范化前置处理** ETL 工程师在抓取第三方数据前，使用本项目对来源 URL 进行格式预检和批次标记，确保进入下游系统的每条链接均符合约定的字符串规范。

**文档项目的外链审计与备份** 开源文档撰写者可将分散在各章节中的外部链接集中提交至本项目，生成独立的资源清单，便于定期检查链接有效性或迁移至新域名。

## 快速开始

执行以下命令在本地环境完成项目克隆、依赖安装与示例数据运行。

```bash
git clone https://github.com/your-org/hyperlink-collective.git
cd hyperlink-collective
python -m pip install -r requirements.txt
python cli.py process --batch 1021 --input ./data/raw_urls.txt --output ./dist/resource_list.md
```

## 项目结构

项目采用模块化分层设计，核心逻辑与用户配置分离，便于维护和扩展。

```
hyperlink-collective/
├── cli.py                     # 命令行入口，解析用户参数并调度核心流程
├── config/
│   ├── default.yaml           # 默认分类规则与输出模板配置
│   └── custom.yaml            # 用户自定义规则，覆盖默认设置
├── core/
│   ├── parser.py              # URL 字符串解析器，提取协议、域名、路径
│   ├── classifier.py          # 基于路径模式和正则表达式的分类引擎
│   ├── validator.py           # 格式合规性检查，生成警告和错误码
│   └── deduper.py             # 完全匹配去重与批次优先级合并
├── output/
│   ├── markdown.py            # 生成 GFM 兼容的列表和表格
│   └── json_exporter.py       # 导出结构化 JSON 供其他系统消费
├── tests/
│   ├── test_parser.py         # 解析器单元测试，覆盖边缘输入情况
│   ├── test_classifier.py     # 分类规则测试，验证路径匹配准确性
│   └── fixtures/              # 测试用的示例 URL 文件
├── docs/
│   ├── usage.md               # 详细使用说明与命令行示例
│   └── rules.md               # 分类规则编写指南与最佳实践
└── data/
    ├── raw/                   # 存放用户提交的原始输入文件
    └── processed/             # 存放处理后的输出文件
```

## 常见问题

**Q：输入文件中混合了裸域名和带协议的 URL，项目会如何处理？**
项目默认透传所有原始字符串，不做任何自动补全。用户可在配置中开启校验模式，此时 validator 会标记不符合预期格式的条目，但不会修改原始数据。建议用户在输入前自行统一格式，或通过预处理脚本完成转换。

**Q：如何添加自定义分类规则？**
编辑 config/custom.yaml 文件，在 rules 字段下新增条目，格式为 "路径关键词: 类别名称"。例如添加 "/docs: 文档资源" 即可将所有包含 /docs 的 URL 归入该类别。修改后重新运行 cli.py 即可生效，无需重启服务。

**Q：输出 Markdown 中的 URL 顺序是否可以调整？**
默认按输入顺序输出。若需按域名或类别排序，可在 cli.py 中通过 --sort 参数指定排序键，目前支持 domain、path 和 input_order 三种选项。
