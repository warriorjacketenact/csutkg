# EON Resource Aggregator

EON Resource Aggregator 是一个面向技术研究与信息检索领域的结构化外部资源导航系统。项目定位于为开发者、数据分析师、技术决策者以及信息调研人员提供高质量的参考链接汇总服务，通过人工筛选与主题分类，将分散于多个域名的技术文档、研究笔记、案例分析和行业报告整合为统一的访问入口。

本项目并非搜索引擎，也不直接托管任何原始内容，而是充当索引层与导航层，帮助用户快速定位到特定主题下的可用资源。通过对资源 URL 的批量整理与语义归类，项目显著降低了信息查找的时间成本，特别适用于需要跨站点、跨领域查阅辅助材料的技术调研场景。

## 功能概览

**批量外链索引**：支持一次性录入和管理数百条外部 URL，自动保持原始地址的完整性，避免改写或重定向造成的引用失效。

**资源分类标签**：依据 URL 路径特征和来源域名，为每个资源自动打上主题标签，便于后续按类别筛选与检索。

**快速导航表格**：提供至少四个维度的资源导航表格，分别对应入门指南、使用手册、API 参考和常见问题，方便不同经验层级的用户按需查找。

**项目结构可视化**：通过 ASCII 目录树清晰展示系统内部模块划分，使贡献者和维护者能够快速理解代码组织逻辑。

**依赖与环境清单**：以表格形式列出所有必需的运行环境、外部工具和版本要求，确保部署过程透明可控。

**多场景适配**：覆盖开发调试、生产部署、文档编写和数据分析四种典型使用场景，每个场景均有对应的资源推荐路径。

**纯静态部署支持**：系统设计为纯静态站点，可托管于任何支持 HTTP 服务的 Web 服务器或对象存储平台，无需数据库或后端运行时依赖。

**贡献者友好流程**：提供标准化的资源提交流程，包括链接校验、分类建议和描述撰写规范，降低社区参与门槛。

## 应用场景

技术调研阶段的外部佐证查阅。在撰写技术方案或评估第三方库时，研发人员可以通过本项目快速访问预筛选的相关讨论、案例和官方文档，避免在海量搜索结果中反复试错，提高调研效率。

数据科学项目的背景材料组织。数据分析师在进行行业分析或建模之前，需要收集大量背景报告和数据集说明。本项目可作为这些外部材料的索引看板，帮助团队统一信息来源，减少沟通成本。

开源社区的知识库共建。社区维护者可以将本项目作为社区文档的补充层，将分散在论坛、博客和 issue 中的有用链接集中管理，形成结构化的知识外延，方便新成员上手。

自动化流程中的资源依赖声明。在 CI/CD 或自动化测试流水线中，本项目输出的资源清单可作为外部依赖的元数据源，供脚本程序读取并执行相应的下载或校验操作，实现资源引用的自动化管理。

## 快速开始

以下操作步骤适用于在本地环境中搭建 EON Resource Aggregator 的开发实例。

```bash
git clone https://github.com/eon-resource/eon-aggregator.git
cd eon-aggregator
npm install
npm run build
```

执行上述命令后，系统将在 `dist` 目录下生成完整的静态文件。如需启动本地预览服务，可运行 `npm run serve`，默认监听端口 8080。

## 安装要求

| 依赖项 | 必需版本 | 说明 |
|--------|----------|------|
| Node.js | 18.0.0 或更高 | 用于执行构建脚本和依赖管理 |
| npm | 8.0.0 或更高 | 包管理器，用于安装项目依赖 |
| Git | 2.30.0 或更高 | 用于克隆仓库和管理版本历史 |
| 现代 Web 浏览器 | Chrome 90+ / Firefox 88+ / Edge 90+ | 用于预览和测试生成的静态站点 |
| HTTP 服务器 | 任意支持静态文件的服务器（如 Nginx、Apache、Caddy） | 用于生产环境部署 |
| 磁盘空间 | 至少 200 MB | 用于存放源码、依赖包和构建产物 |
| 操作系统 | Linux / macOS / Windows（WSL 推荐） | 开发环境支持所有主流平台 |

## 贡献指南

提交新资源链接。贡献者需在 `src/config/constants.js` 的分类标签列表中确认是否存在适配的主题分类。若无合适分类，应先在 issue 中提议新增分类并获得维护者确认。提交时需提供原始 URL、建议的分类标签以及一段不超过 80 字的中文描述，说明该资源的核心内容或适用场景。

运行本地校验流程。所有新增或修改的资源条目必须通过 `npm run validate` 命令的检查，该命令会验证 URL 格式、去重性以及分类标签的有效性。校验失败的项目将无法通过持续集成流水线。

发起拉取请求。贡献者应在个人复刻仓库中创建特性分支，提交变更后向主仓库的 `main` 分支发起拉取请求。请求描述中应列出新增的资源数量、分类分布情况以及任何可能影响构建流程的配置改动。

参与分类规则讨论。当现有分类体系无法准确描述某类资源时，鼓励贡献者在 issue 区域发起讨论，说明分类调整的理据和预期收益。重大分类变更需获得至少两位维护者的同意后方可合并。

## 常见问题

**问：项目会定期检查资源链接的有效性吗？**

答：项目维护团队每月通过 GitHub Actions 运行一次全量链接检查，对返回 HTTP 状态码非 200 或超时的链接进行标记。标记后的链接会在下一版本中移出主列表，转入待复核清单。贡献者也可随时运行 `npm run validate` 手动触发检查。

**问：我可以将本项目用于商业产品或内部系统吗？**

答：可以。本项目采用 MIT 许可证，允许自由使用、修改、分发和商业化，只需保留原始版权声明即可。但请注意，本项目的资源列表仅提供导航服务，不担保外部链接内容的准确性、合法性和可用性，使用者应自行评估风险。

**问：如何更新已收录资源的描述或分类？**

答：更新资源元数据的方式与新增资源相同。在 `src/config/constants.js` 中修改对应分类定义，或在 `src/core/classifier.js` 中调整分类逻辑，然后提交拉取请求。若仅需修正单一资源的描述，可在 `docs/usage/navigation.md` 的对应小节中直接编辑并提交变更。
