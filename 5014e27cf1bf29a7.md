# LinkVault Resource Aggregator

LinkVault 是一个面向技术研究与开发人员的结构化外部资源导航系统，专注于对分散在网络中的高质量技术文档、教程、工具页面及参考材料进行统一的索引、分类与快速访问。该项目不存储任何第三方内容，仅提供元数据组织与链接跳转服务，旨在帮助开发者、技术决策者与研究人员在特定技术领域内高效定位关键资源，减少信息检索过程中的时间损耗与认知负担。

本项目定位于中大型技术团队的知识库辅助工具，以及个人开发者构建自身技术学习路径的导航起点。通过对批量导入的 URL 进行目录化展示与标签关联，LinkVault 使得原本孤立的外部页面能够在统一的项目框架下被集中查阅，适用于需要频繁查阅大量外部参考资料的开发场景。

## 功能概览

- 批量链接导入与结构化存储：支持通过命令行工具或配置文件批量导入外部 URL，自动解析并提取页面标题，将原始链接列表转化为可浏览的目录树结构。

- 多维度分类与标签过滤：每个资源条目可关联多个分类标签，用户可通过标签组合快速筛选出特定主题或技术栈的相关链接。

- 本地 Web 服务预览：内置轻量级 HTTP 服务器，启动后可通过浏览器访问本地导航页面，所有外部链接在新标签页中打开，不干扰当前浏览上下文。

- 链接可用性检查：提供周期性或按需的链接存活检测功能，对返回 4xx 或 5xx 状态码的资源进行标记，辅助维护者清理失效条目。

- 元数据缓存与更新：自动缓存每个链接的页面标题与摘要信息，减少重复请求；同时支持手动刷新缓存以获取最新页面信息。

- 导出与集成支持：支持将已整理的链接列表导出为 JSON、CSV 或静态 HTML 格式，便于集成至现有文档系统或知识管理平台。

- 命令行交互界面：提供完整的 CLI 工具，涵盖链接添加、删除、列表查询、分类管理、服务启动等常用操作，适合脚本化与自动化任务。

- 访问日志与统计：记录每个外部链接的点击次数与最后访问时间，帮助识别高频资源与冷门条目，为后续资源维护提供数据依据。

## 应用场景

- 技术团队内部知识库构建：团队技术负责人可将项目相关的 API 文档、设计规范、部署指南等外部链接统一纳入 LinkVault，新成员入职时仅需访问本地导航页即可获取全部参考材料，无需逐一询问或翻阅散落的收藏夹。

- 开源项目依赖文档整理：开源项目的维护者可以将项目所依赖的第三方库文档、社区讨论帖、问题跟踪链接、构建工具指南等集中管理，并将 LinkVault 作为项目 README 中的“外部资源”章节的数据源，保持文档整洁且易于更新。

- 技术学习路线规划：正在学习特定技术栈（如分布式系统、机器学习、前端框架）的开发者，可将收集到的教程、案例、视频地址、习题集等按学习阶段分类，通过 LinkVault 形成清晰的学习路径导航，避免学习过程中因频繁查找资料而中断思路。

- 技术调研与竞品分析：在进行技术选型或竞品分析时，研究人员可批量导入相关产品的官方网站、技术白皮书、性能测试报告、社区评价等链接，通过标签区分不同产品线，快速切换对比维度，提高调研效率。

## 快速开始

以下操作基于 Linux / macOS 环境，Windows 用户建议使用 WSL2 或 Git Bash 执行。

```bash
# 克隆项目仓库至本地
git clone https://github.com/your-org/linkvault.git
cd linkvault

# 安装项目依赖（Python 3.9+ 环境）
pip install -r requirements.txt

# 初始化数据库并启动本地服务（默认监听 127.0.0.1:8080）
python linkvault.py init
python linkvault.py serve --port 8080
```

执行上述命令后，在浏览器中访问 `http://127.0.0.1:8080` 即可看到链接导航首页。若需导入用户提供的原始链接列表，可将 URL 按行存入 `links.txt` 文件，然后执行：

```bash
python linkvault.py import --file links.txt --category external
```

## 贡献指南

1. 查阅 issue 列表与项目看板，选取未被指派的 issue 或提出新的改进建议。对于较大功能改动，建议先创建讨论 issue 与维护者沟通设计思路，避免无效开发。

2. 克隆项目至本地，创建以 `feature/` 或 `fix/` 为前缀的分支，确保分支名称与 issue 编号相关联，例如 `feature/issue-123-add-json-export`。

3. 编写代码时遵循项目根目录下的 `.flake8` 与 `.black` 配置文件所约定的代码风格，提交前执行 `make lint` 与 `make test` 确保静态检查与单元测试全部通过。

4. 提交 commit 时使用语义化消息格式，即 `<type>(<scope>): <subject>`，其中 type 包括 feat、fix、docs、style、refactor、test、chore 等，scope 为受影响的模块名称。

5. 推送分支后创建 Pull Request，描述中须明确说明改动内容、测试覆盖情况以及是否影响现有接口或数据库结构，等待至少一位维护者审核通过后合并。

## 常见问题

Q: 导入大量链接时如何处理重复条目？是否会自动去重？

A: 系统在导入时以 URL 完整字符串作为唯一键进行去重。若数据库已存在相同 URL，则不会重复插入，但会更新该条目的最后导入时间和标签关联信息。可通过命令行 `linkvault.py list --duplicates` 查看所有重复记录。

Q: 本地 Web 服务启动后，外部链接点击统计功能如何工作？是否会影响隐私？

A: 所有外部链接均通过本地 `/redirect/` 端点进行中转，记录点击时间与来源 IP（仅用于统计，不持久化存储用户身份信息），随后返回 302 重定向至目标 URL。该机制仅记录聚合数据，不追踪用户后续行为，符合基本隐私保护要求。

Q: 如何迁移数据库至其他存储后端，例如 PostgreSQL？

A: 当前版本仅内置 SQLite 支持。如需迁移至 PostgreSQL，请在 `config.yaml` 中修改 `database_url` 为 PostgreSQL 连接字符串，并确保已安装 `psycopg2` 依赖。随后执行 `linkvault.py db upgrade` 以应用所有迁移脚本。数据迁移工具将在后续版本中提供。
