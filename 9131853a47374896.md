# LinkVault Core

LinkVault Core 是一个面向技术团队与内容创作者的轻量级外链资源管理与导航系统。项目定位为技术资源汇总枢纽，用于统一收录、分类展示和快速检索来自不同来源的高价值外部链接。目标用户包括开发者、技术文档撰写者、运维工程师以及项目维护者，解决他们在多项目、多文档环境中分散收藏、难以共享和频繁失效的外链管理痛点。系统基于静态站点生成机制，无需数据库，所有链接数据以结构化文本存储，支持版本控制与自动化构建，便于集成到现有 CI/CD 流程中。

## 功能概览

**批量外链导入**：支持从 CSV、JSON 或纯文本列表批量导入 URL，自动解析元数据并生成结构化条目。

**多维度分类标签**：每个链接可关联多个分类标签和自定义属性，支持按项目、领域、文件类型或紧急程度进行筛选。

**链接健康检查**：内置定时巡检模块，自动探测链接可达性，标记失效或重定向状态，并生成报告。

**静态站点生成**：基于模板引擎输出完整的 HTML 导航页面，可直接部署至 Nginx、CDN 或对象存储服务。

**全文检索与过滤**：针对链接标题、描述、标签和备注字段提供快速全文搜索，支持正则表达式过滤。

**访问统计看板**：记录链接被点击的次数、最后访问时间和来源 IP 聚合，生成简单的热度分析视图。

**权限分级控制**：支持多用户只读、编辑和管理员三级权限，适用于团队内部协作场景。

**数据导出与迁移**：支持将链接库完整导出为 Markdown 表格、JSON 或 SQLite 文件，便于迁移至其他平台。

## 应用场景

1. 技术文档站的外链附录管理。项目维护者可以在每个版本文档中嵌入 LinkVault Core 生成的链接列表，确保所有外部引用均经过统一校验和格式规范化，避免文档中出现孤立或过期的参考链接。

2. 研发团队内部资源聚合页。团队可将常用的内部工具面板、监控系统、日志平台、代码仓库和 CI 流水线地址统一收录，配合标签分类，新成员入职时即可通过导航页快速访问所有基础设施。

3. 开源项目的外部贡献者指南。开源项目维护者可以将依赖库、参考规范、社区论坛和下载镜像站整理为外链集合，随项目源码一同发布，降低贡献者查找资料的认知成本。

4. 运维巡检自动化流程。运维团队定期运行链接健康检查任务，输出失效链接清单，结合告警系统通知责任人更新或移除异常记录，保持运维文档的可信度。

5. 个人知识库的外链备份中心。知识管理爱好者可将散落在笔记中的网页链接集中托管，配合标签和备注记录上下文，避免因原网页变更导致信息丢失。

## 快速开始

以下步骤适用于 Linux / macOS 环境，Windows 用户建议使用 WSL 或 Git Bash。

```bash
git clone https://github.com/your-org/linkvault-core.git
cd linkvault-core
pip install -r requirements.txt
python linkvault.py --import links.txt --output ./public
```

其中 `links.txt` 为每行一个 URL 的纯文本文件，导入后默认生成静态站点到 `./public` 目录。若需要启动本地预览服务，可执行：

```bash
python serve.py --dir ./public --port 8080
```

访问 `http://127.0.0.1:8080` 即可查看导航界面。

## 安装要求

| 依赖 | 必需 | 说明 |
|------|------|------|
| Python 3.8 或更高版本 | 是 | 主程序运行环境，建议使用 3.10 以上获得性能优化 |
| pip 21.0 以上 | 是 | 用于安装依赖包，确保与新版 PyPI 兼容 |
| requests 库 (>=2.25.0) | 是 | 发起 HTTP 请求进行链接健康检查与元数据抓取 |
| beautifulsoup4 (>=4.9.0) | 否 | 启用 HTML 解析时自动提取页面标题和描述，无则仅存储原始 URL |
| markdown 库 (>=3.3.0) | 否 | 支持将 Markdown 格式的备注渲染为 HTML，无则保留纯文本 |
| watchdog (>=2.0.0) | 否 | 开启文件监控模式，实现导入目录的自动重载 |
| PyYAML (>=5.4.0) | 否 | 支持 YAML 格式的配置文件，无则仅使用 JSON 或环境变量 |
| SQLite3 系统库 | 是 | 用于本地缓存链接状态和访问日志，Python 内置 |
| Nginx 或任何静态 HTTP 服务器 | 否 | 生产环境部署静态页面所需，开发环境使用内置 serve 即可 |

## 项目结构

```
linkvault-core/
├── linkvault.py                 # 主入口脚本，解析命令行参数并调度核心流程
├── serve.py                     # 内置静态文件开发服务器，支持热加载
├── requirements.txt             # Python 依赖清单，固定版本号以保证环境一致性
├── config/
│   ├── default.yaml             # 默认配置项，包含巡检间隔、输出目录、缓存策略
│   ├── custom.yaml.example      # 用户自定义配置模板，可覆盖默认参数
│   └── schema.json              # 配置项 JSON Schema，用于校验用户输入
├── core/
│   ├── __init__.py
│   ├── parser.py                # URL 解析器，支持纯文本、CSV、JSON 多种输入格式
│   ├── checker.py               # 链接健康检查模块，并发探测 HTTP 状态码
│   ├── indexer.py               # 构建倒排索引，支持标题、标签和备注的全文搜索
│   ├── renderer.py              # 模板渲染引擎，生成静态 HTML 页面
│   └── watcher.py               # 文件系统监控器，监听导入目录变更自动触发重建
├── templates/
│   ├── base.html                # 基础 HTML 骨架，包含全局导航和页脚
│   ├── index.html               # 首页模板，展示分类导航和热门链接
│   ├── detail.html              # 链接详情页模板，展示完整元数据和访问统计
│   └── search.html              # 搜索结果页模板，支持分页和排序
├── static/
```

## 贡献指南

1. 阅读开发文档与 API 参考，了解核心模块的设计思路和接口约定。建议先从标注有 `TODO` 或 `good first issue` 的简单任务入手。

2. 在 GitHub 上 Fork 本仓库，创建新的特性分支，分支命名遵循 `feature/功能描述` 或 `fix/问题编号` 格式。提交信息请使用约定式提交规范，如 `feat: 添加 JSON 导入对嵌套数组的支持`。

3. 编写代码时保持与现有风格一致，Python 代码遵循 PEP 8 规范，JavaScript 采用 ESLint 推荐配置。新增功能需包含对应的单元测试，测试覆盖率不得低于现有基线。

4. 提交 Pull Request 之前，请确保本地所有测试通过（执行 `pytest tests/`），并且文档已同步更新。若涉及配置项变更，需同时修改 `schema.json` 和默认配置文件。

5. PR 描述中请清晰说明解决的问题、实现方案以及测试结果。至少需要一位维护者审核通过后方可合并。重大变更需在 PR 中标注 `breaking-change` 标签，并附迁移指南。

## 常见问题

问：导入的链接数量很大时，性能表现如何？

答：系统针对 10 万条级别的链接进行了优化。首次导入时解析和索引构建耗时约 0.5 秒/千条，健康检查模块采用异步并发机制，默认并发数为 50，扫描 1 万条链接的可用性约需 2 至 3 分钟。SQLite 数据库启用 WAL 模式和合适的索引后，查询和筛选操作响应时间保持在 200 毫秒以内。若数据量超过百万级，建议分批次导入并调整并发参数。

问：如何自定义前端页面的样式和布局？

答：所有模板文件位于 `templates/` 目录，基于 Jinja2 模板引擎。您可以直接修改 `base.html` 中的导航结构，或覆盖 `main.css` 中的 CSS 变量。系统支持通过 `config/custom.yaml` 中的 `theme.primary_color`、`theme.logo_text` 等字段快速调整配色和品牌标识，无需改动源码。若需要全新的页面布局，建议复制现有模板后修改，并将 `renderer.py` 中的模板查找路径指向自定义目录。

问：链接健康检查的误报如何处理？

答：部分网站可能返回 403 或 429 状态码，但实际内容仍可访问。系统允许在配置文件中设置 `checker.ignore_status_codes` 列表忽略特定状态码。同时支持对单个链接添加 `verify: false` 元数据标签，强制跳过检查。对于临时性网络波动，可通过 `checker.retry_times` 和 `checker.timeout` 参数调整重试策略。所有被标记为异常的链接会记录在 `data/exports/failed_links.csv` 中，方便人工复核。
