# CDJinniu Resource Aggregator

CDJinniu Resource Aggregator 是一个面向技术内容聚合与外部资源索引的开源工具集，专注于将分散在互联网各处的视频、文档、数据流等外链资源进行统一采集、分类存储与快速检索。该项目主要服务于需要构建自有资源导航站、内部知识库或媒体资产管理系统的小型团队与个人开发者，通过标准化的 URL 解析与元数据抓取流程，降低人工整理外链资源的成本。

项目采用模块化插件架构，核心引擎支持对用户提交的资源链接进行存活检测、内容类型识别、标签自动生成以及周期性更新检查，并提供 RESTful API 与轻量级 Web 管理界面，便于运维人员实时监控资源库状态。当前版本已稳定处理超过 1200 批次、累计 25 万条外链资源，被多家技术社区和数字档案馆采用作为底层数据采集组件。

## 功能概览

- **批量链接导入与解析**：支持从纯文本、CSV、JSON 等格式批量导入 URL 列表，自动去除重复项并规范化 URL 格式，同时提取域名、路径层级、查询参数等结构化信息。

- **智能资源分类与标签系统**：基于 URL 路径模式、内容类型响应头及页面元数据，自动为每个资源分配分类标签（如 video、document、image、api 等），并支持自定义标签规则。

- **资源可用性监控与告警**：定时对已收录的链接发起 HEAD 请求检测 HTTP 状态码，记录响应时间变化，当链接返回 4xx 或 5xx 状态时通过 Webhook 发送告警通知。

- **全文检索与过滤查询**：内置倒排索引引擎，支持对资源标题、描述、标签、来源域名等多字段进行组合查询，并提供分页排序与结果高亮显示。

- **数据导出与备份机制**：支持将资源库全量或按条件筛选后导出为 CSV、JSON、Markdown 表格等格式，每日自动生成增量备份文件，保留最近 30 天历史版本。

- **用户权限与操作审计**：提供基于角色的访问控制（RBAC），区分管理员、编辑者、访客三级权限，所有增删改操作均记录操作日志，支持按时间范围回溯。

- **自定义元数据扩展字段**：允许用户为每个资源添加自定义键值对元数据，例如来源项目编号、负责人、过期时间、关联资源 ID 等，满足不同业务场景的个性化需求。

## 应用场景

1. **技术文档导航站构建**：技术团队可将日常积累的 API 文档、运维手册、架构设计图等外部链接统一录入系统，通过标签分类与全文检索快速定位所需资料，减少在书签或即时通讯工具中反复查找的时间。

2. **媒体素材资源库管理**：视频制作团队或内容运营人员使用本系统整理分散在多个云存储平台上的素材链接，为每条素材标注拍摄日期、使用状态、版权信息等扩展字段，实现跨项目素材复用。

3. **开源项目依赖链跟踪**：开源项目维护者将项目依赖的第三方库文档、镜像源地址、社区论坛链接纳入监控范围，当上游资源发生变更或不可用时及时获得通知，降低依赖失效带来的构建风险。

4. **学术文献与数据集索引**：研究人员可将论文预印本、公开数据集、实验数据备份链接等资源集中管理，通过自定义元数据记录论文标题、作者、发表年份，便于撰写综述时快速引用。

5. **企业内部知识库外链整合**：企业可将各部门沉淀的 Sharepoint 链接、Confluence 页面、工单系统地址统一采集，配合权限审计功能控制不同部门员工的访问范围，避免敏感链接泄露。

## 快速开始

以下步骤将帮助您在本地环境中快速启动 CDJinniu Resource Aggregator 服务。

```bash
# 克隆项目仓库
git clone https://github.com/cdjinniu/resource-aggregator.git
cd resource-aggregator

# 安装项目依赖（使用 pip 与虚拟环境）
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 初始化数据库结构并启动开发服务器
python manage.py migrate
python manage.py loaddata initial_tags.json
python manage.py runserver --host 0.0.0.0 --port 8080
```

上述命令执行完毕后，访问 http://localhost:8080 即可进入 Web 管理界面，默认管理员账号为 admin，密码在首次启动时打印于终端输出。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.9 及以上 | 核心运行环境，建议使用 3.10 LTS 版本以获得更好的性能与类型提示支持 |
| PostgreSQL | 14.0 及以上 | 主数据库引擎，用于存储资源元数据、标签映射及操作审计日志 |
| Redis | 6.2 及以上 | 缓存与消息队列中间件，用于提升检索响应速度及异步任务调度 |
| Node.js | 18.0 及以上 | 仅在前端构建时需要，用于编译 Web 管理界面的静态资源文件 |
| Nginx | 1.20 及以上 | 生产环境推荐反向代理服务器，用于负载均衡与静态文件缓存 |
| Celery Worker | 5.2 及以上 | 分布式任务队列执行器，独立运行于后台处理链接监控与备份任务 |
| Elasticsearch | 7.17 及以上 | 可选依赖，启用后将替代内置倒排索引，提供更强大的全文检索能力 |

## 项目结构

```
resource-aggregator/
├── cmd/                                  # 命令行入口与启动脚本
│   ├── server/                           # Web 服务启动入口 (main.go)
│   └── worker/                           # 后台任务执行器入口 (celery 启动脚本)
├── internal/                             # 内部核心包，不对外暴露
│   ├── collector/                        # 资源采集模块，包含 HTTP 客户端与解析器
│   │   ├── fetcher.go                    # 并发获取 URL 内容，支持超时与重试策略
│   │   ├── parser.go                     # 解析 HTML 元数据与 Open Graph 标签
│   │   └── classifier.go                 # 基于规则与机器学习的资源分类器
│   ├── storage/                          # 数据持久化层，包含数据库 ORM 与缓存接口
```

## 常见问题

**问：系统能处理多少条资源链接？性能瓶颈在哪里？**

答：在推荐的硬件配置（4 核 CPU、16 GB 内存、SSD 存储）下，单实例可稳定管理 50 万条资源记录，检索响应时间保持在 200 毫秒以内。性能瓶颈通常出现在数据库查询层面，建议对标签字段和创建时间字段建立联合索引，并定期清理 90 天前的操作审计日志以控制表大小。

**问：如何将已有的书签文件（如 Chrome 导出的 HTML）批量导入？**

答：项目提供 import_bookmark.py 辅助脚本，支持解析 Netscape 书签格式。您可以将导出的书签文件放置于 scripts/imports/ 目录下，执行 python scripts/import_bookmark.py --file bookmarks.html --tag chrome 即可完成导入，系统会自动提取 URL 和标题并打上指定标签。

**问：监控模块检测到死链后会如何处理？是否支持自动重试？**

答：监控模块默认在检测到 4xx 或 5xx 状态后不会立即标记为死链，而是进入重试队列，分别在 5 分钟、30 分钟、2 小时后进行三次重试。若三次重试均失败，则将该资源状态标记为 unreachable 并触发告警。您可通过配置文件修改重试间隔与最大重试次数。
