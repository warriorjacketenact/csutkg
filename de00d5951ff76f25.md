# ResourceHub

ResourceHub 是一个面向开发者与技术研究人员的结构化外链资源聚合平台。该项目并非传统的内容管理系统，而是一个高度规范化的资源索引中间件，旨在将分散于网络各处的优质技术文档、工具站点、数据源与参考材料，通过统一的元数据模型进行整理、分类与持久化存储。项目定位为技术资源导航枢纽，解决开发者在信息检索过程中面临的内容碎片化、链接失效、分类混乱以及重复造轮子等核心痛点。

ResourceHub 的目标用户包括运维工程师、全栈开发人员、技术文档撰写者、数据科学家以及任何需要系统化管理外部技术参考链接的团队或个人。通过本项目提供的标准化录入流程与分类体系，用户可以快速建立私有或团队共享的外部资源库，并借助项目内置的索引机制实现高效检索与回溯。

## 功能概览

资源条目标准化录入 提供基于 Markdown 前端元数据的结构化录入模板，支持标题、标签、所属领域、作者、更新日期等字段的规范化填写。

多层级标签分类系统 内置覆盖编程语言、框架、中间件、云原生、数据库、算法等六大技术领域的预置标签体系，并支持用户自定义扩展。

链接健康状态检测 集成外部链接可达性检查模块，可定期对已收录的 URL 执行 HTTP 状态码验证，自动标记失效链接并生成报告。

全文与元数据联合检索 支持针对资源标题、描述、标签、域名等字段的组合查询，并提供基于 Levenshtein 距离的模糊匹配能力。

资源版本快照记录 对每条外链资源记录关联的元数据变更历史进行持久化存储，支持回溯任意历史版本的资源描述与分类信息。

批量导入与导出接口 提供 JSON 与 CSV 格式的批量数据导入导出功能，便于与其他知识管理工具进行数据迁移与同步。

访问统计与热度排序 记录每条资源的点击次数与最近访问时间，支持按热度、新增时间、字母序等多种维度对资源列表进行排序展示。

## 应用场景

技术团队内部知识库建设 研发团队可将本项目中收录的第三方依赖文档、内部运维手册链接、API 参考站点等资源统一纳管，形成团队共享的技术外链知识图谱，有效降低新成员上手阶段的信息检索成本。

开源项目文档站外链管理 开源项目维护者可以在项目文档中引用 ResourceHub 生成的资源列表，将项目依赖的参考材料、延伸阅读、社区讨论帖等外链集中管理，避免在主文档中插入大量冗长的外部链接。

技术博客与教程素材整理 技术内容创作者在撰写系列教程时，可利用 ResourceHub 分类归档每篇文章涉及的外部引用链接，确保所有引用材料可追溯、可验证，提升教程的严谨性与可维护性。

企业合规性外部引用审计 对于需要对外部技术引用进行合规审查的行业场景，ResourceHub 的资源变更历史与健康检测功能可提供完整的引用链路审计日志，满足内部风控与外部监管要求。

## 快速开始

以下步骤将在本地环境中完成 ResourceHub 项目的克隆、依赖安装与开发服务启动。

```bash
git clone https://github.com/resourcehub/resourcehub.git
cd resourcehub
npm install
npm run dev
```

执行上述命令后，开发服务器默认监听 3000 端口。用户可通过浏览器访问 http://localhost:3000 进入资源管理面板。首次启动时，系统会自动创建基于 SQLite 的本地数据库文件并初始化预置标签数据。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | 18.x 或更高 LTS 版本 | 项目运行时环境，需包含 npm 或 yarn 包管理器 |
| npm | 8.x 或更高版本 | 用于安装项目依赖包，随 Node.js 一并安装 |
| SQLite3 | 3.35.0 或更高版本 | 默认本地数据库引擎，用于存储资源元数据及标签关系 |
| Python | 3.9 或更高版本 | 仅当启用链接健康检测脚本时必需，用于运行异步 HTTP 探测任务 |
| Git | 2.30 或更高版本 | 用于克隆项目仓库及后续版本更新 |
| 内存 | 最低 512 MB，推荐 1 GB 以上 | 开发环境内存要求，生产环境视并发量相应扩展 |

## 项目结构

项目采用分层架构设计，前端基于 React 与 TypeScript 构建，后端基于 Node.js 与 Express 框架实现 RESTful API，数据层使用 SQLite 与 Knex.js 查询构建器。以下为项目核心目录结构及其职责说明：

```
resourcehub/
├── packages/                           # 多包管理根目录
│   ├── frontend/                       # 前端单页应用模块
│   │   ├── src/
│   │   │   ├── components/             # 可复用 UI 组件库 (ResourceCard, TagFilter, SearchBar)
│   │   │   ├── hooks/                  # 自定义 React Hooks (useResourceQuery, useHealthCheck)
│   │   │   ├── stores/                 # Zustand 状态管理 (resourceStore, filterStore)
│   │   │   └── services/               # API 调用封装层 (resourceService, tagService)
│   │   └── public/                     # 静态资源文件 (favicon, manifest)
│   ├── backend/                        # 后端服务模块
│   │   ├── src/
│   │   │   ├── routes/                 # 路由定义 (resourceRoutes, tagRoutes, healthRoutes)
│   │   │   ├── controllers/            # 请求控制器与业务逻辑聚合
│   │   │   ├── models/                 # 数据模型定义 (Resource, Tag, VisitLog)
│   │   │   ├── middleware/             # 中间件 (cors, rateLimit, errorHandler)
│   │   │   └── utils/                  # 工具函数 (urlValidator, markdownParser)
│   │   └── migrations/                 # 数据库迁移脚本 (初始表结构及索引)
│   └── shared/                         # 前后端共享类型定义与常量
│       ├── types/                      # TypeScript 接口与枚举定义
│       └── constants/                  # 预置标签列表、分类映射表
├── scripts/                            # 运维与辅助脚本
│   ├── health-check.js                 # 链接健康状态探测脚本 (定时任务)
```

## 常见问题

问：首次启动后数据库连接失败，提示 "SQLITE_CANTOPEN" 错误，如何解决？

答：该错误通常由数据库文件写入权限不足引起。请检查项目根目录下 data 文件夹是否存在且具有读写权限。若该目录不存在，请手动创建 mkdir data，然后重新执行 npm run dev。若问题依旧，请检查系统是否安装了 SQLite3 库，Linux 系统可执行 sudo apt-get install sqlite3，macOS 可执行 brew install sqlite3。

问：链接健康检测脚本报告大量超时，是否代表所有资源均不可达？

答：健康检测脚本默认超时时间为 5000 毫秒，且并发请求数限制为 10。若目标服务器响应较慢或存在网络抖动，可能出现超时误报。建议通过 scripts/health-check.js 中的配置项调整 timeout 与 concurrency 参数。同时，检测结果仅反映脚本执行瞬间的网络可达性，不排除目标服务器临时维护或区域网络限制等情况。

问：如何将现有浏览器书签批量导入 ResourceHub？

答：项目提供了 CSV 格式批量导入功能。用户需先将浏览器书签导出为 HTML 格式，再通过第三方工具（如 Bookmarks Clean Up）将 HTML 转换为包含 title、url、tags 三列的 CSV 文件。随后执行 npm run import-csv -- --file=./bookmarks.csv 即可完成导入。导入前请确认 CSV 文件编码为 UTF-8，避免中文乱码问题。
