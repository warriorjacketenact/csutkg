# SiteBloom Playbook 资源导航系统

SiteBloom Playbook 是一个面向技术团队、内容创作者和项目策划者的外链资源汇总与导航工具。项目定位于将分散在各类技术博客、演示站点、测试环境和临时项目中的零散链接进行集中化管理，通过结构化的索引机制降低链接遗忘与丢失的风险。目标用户包括前端开发人员、全栈工程师、技术文档写作者以及需要频繁管理外部引用资源的产品经理。系统本身不存储任何实际内容，仅提供链接的录入、分类、检索和状态监控能力，以轻量级方式解决资源碎片化问题。

## 功能概览

**链接批量导入与去重**：支持通过文本文件或 API 接口一次性导入大量 URL，系统自动识别重复条目并生成导入报告。

**分类标签体系**：每个链接可绑定多个自定义标签，支持按标签层级筛选和聚合，便于构建多维度的资源视图。

**可用性健康检查**：后台定时任务对已录入链接发起 HEAD 请求，标记失效或响应超时的资源，输出异常状态列表。

**全文检索与模糊匹配**：基于链接路径关键词和自定义备注字段提供快速检索，支持拼音首字母模糊搜索。

**访问热度统计**：记录每个链接的点击次数和最后访问时间，按日、周、月维度生成热度排行。

**数据导入导出**：支持 JSON、CSV、Markdown 表格三种格式的批量导出，便于迁移至其他文档系统或进行离线备份。

**只读镜像发布**：可将选定分类下的链接生成静态只读页面，适合公开发布为团队知识库入口。

**操作日志审计**：记录所有新增、删除、修改操作的时间与操作者身份，满足团队协作场景下的追溯需求。

## 应用场景

技术团队内部知识库聚合：团队在日常开发中会产生大量临时测试链接、第三方依赖文档地址和内部演示站点，使用 SiteBloom Playbook 可将这些分散资源统一收录，新成员入职时通过分类标签快速获取所需环境入口。

项目交接与文档归档：项目结项时，将期间使用到的所有外部参考链接、API 文档地址和设计稿预览链接导入系统，生成完整的资源清单随项目文档一并移交，避免后续维护时遗落关键引用。

内容创作与外部引用管理：技术博客作者或开源文档维护者在撰写文章时，需要频繁引用外部规范、标准草案和社区讨论帖。通过本系统为每篇文章建立独立的链接集合，发布前统一校验所有链接的有效性，降低死链对读者体验的影响。

多环境测试入口组织：开发团队同时维护开发、测试、预发布、生产等多套环境，每套环境均有独立的管理后台和监控面板地址。利用标签体系按环境维度分类，一键切换视图，避免在浏览器书签中混乱堆叠。

## 快速开始

```bash
# 克隆项目仓库
git clone https://github.com/sitebloom/playbook-navigator.git
cd playbook-navigator

# 安装依赖（使用 pnpm 或 npm）
pnpm install
# 或
npm install

# 初始化数据库并创建管理员账户
npm run db:init

# 启动开发服务器（默认监听 3000 端口）
npm run dev
```

访问 http://localhost:3000 后，使用初始化生成的管理员凭证登录，进入控制台后点击「批量导入」即可开始录入链接数据。生产环境部署建议使用 `npm run build` 构建静态文件，并通过 Nginx 反向代理。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | 18.x 或 20.x LTS | 运行时环境，推荐使用 20.11.0 及以上版本 |
| pnpm | 8.x 或 9.x | 包管理器，若使用 npm 则需 9.x 以上 |
| PostgreSQL | 14.x 或 15.x | 主数据库，用于存储链接元数据与用户信息 |
| Redis | 7.x | 缓存与会话存储，可选但强烈建议生产环境启用 |
| Nginx | 1.24 及以上 | 反向代理与静态资源服务，生产环境部署必需 |
| Systemd | 249 及以上 | 用于进程守护，Linux 发行版默认包含 |

## 项目结构

```

├── apps/
│   ├── web/                          # 前端 Next.js 应用（App Router）

│       ├── src/

│       │   ├── plugins/              # Fastify 插件（数据库、Redis、JWT）
│       │   ├── services/             # 业务逻辑层（导入、检查、统计）
│       │   ├── models/               # Knex.js 数据模型与迁移

│   ├── Dockerfile.api                # 后端服务镜像

├── scripts/
│   ├── seed-links.js                 # 测试数据填充脚本
│   └── migrate-db.sh                 # 数据库迁移快捷命令

├── .env.example                      # 环境变量模板
├── pnpm-workspace.yaml               # pnpm 工作空间配置
└── README.md                         # 本文件
```

## 贡献指南

贡献者需先阅读项目行为准则，并签署贡献者许可协议。所有代码提交均需通过持续集成流水线检查，具体步骤如下：

1. 在 GitHub 上 fork 本仓库，将 fork 后的仓库克隆至本地开发环境。确保本地 Node.js 版本与 .nvmrc 中声明的版本一致，推荐使用 nvm 进行版本管理。

2. 创建新的功能分支，分支命名需遵循 `feat/描述`、`fix/描述` 或 `docs/描述` 格式。提交前运行 `pnpm lint` 和 `pnpm test` 确保代码风格与单元测试通过，新增功能需附带对应测试用例。

3. 若涉及数据库结构变更，需在 `packages/api/src/models/migrations` 目录下新增迁移文件，并同时更新对应的模型定义。迁移文件需同时包含 `up` 和 `down` 方法，便于回滚。

4. 完成开发后，向主仓库的 `main` 分支发起 Pull Request，在 PR 描述中清晰说明改动内容、关联 Issue 编号以及测试覆盖情况。PR 至少需要一名维护者审核通过后方可合并。

## 常见问题

**问：系统支持的最大链接数量是多少？单次导入有无上限？**

答：系统设计上不设硬性数量上限，实际承载能力取决于部署环境的数据库性能。PostgreSQL 单表在千万级记录下仍可保持正常查询响应，建议单次导入不超过 5 万条，超过此数量可分批导入。健康检查任务默认每 24 小时扫描所有链接，若链接总数超过 50 万，建议调整检查频率为每周一次并增加超时时间配置。

**问：如何迁移已有书签或浏览器收藏夹数据？**

答：系统提供了 `npm run import:bookmarks` 命令，支持导入 Chrome 导出的 HTML 书签文件、Firefox JSON 备份以及 Raindrop.io 的 CSV 格式。导入时会自动识别书签文件夹层级并转换为标签体系。若需要从其他格式迁移，可先将数据整理为 `URL, 标题, 标签1|标签2` 的 CSV 格式，通过管理后台的批量导入界面完成。

**问：私有部署时如何配置外部访问与 HTTPS？**

答：生产环境推荐将前端构建为静态文件后由 Nginx 托管，Nginx 同时负责 SSL 终止。项目根目录提供了 `nginx/nginx.conf.example` 示例配置，包含 Let‘s Encrypt 自动续签的配置片段。若使用 Docker Compose 部署，可通过 `docker-compose.yml` 中的 `environment` 字段设置 `PUBLIC_URL` 环境变量，系统生成的所有绝对链接将以此值为基础。
