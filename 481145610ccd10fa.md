# RY MFS External Resource Aggregator

RY MFS External Resource Aggregator 是一个面向技术团队与个人开发者的轻量级外部资源索引与导航系统。项目定位于将分散在不同服务商、不同域名下的技术文档、工具链接、参考手册与社区讨论进行集中式管理与快速检索，帮助用户减少在多个标签页之间反复切换的成本，提升信息获取效率。

本项目不存储任何实际内容，仅提供 URL 元数据与分类标签。目标用户包括需要维护大量外部依赖文档的运维工程师、需要频繁查阅多份技术规范的后端开发者，以及希望建立团队共享知识索引的技术管理者。通过统一的资源录入与分类机制，项目能够将零散的链接组织为可维护的结构化数据集，并支持后续扩展为带检索与过滤功能的前端界面。

## 功能概览

**集中式链接入库** 提供标准化的 URL 录入流程，支持单条添加与批量导入，自动校验链接可达性并记录添加时间戳。

**分类标签管理** 每条资源可关联多个自定义标签，支持按标签过滤、按标签聚合统计，便于构建多维度导航视图。

**快速模糊搜索** 基于内存索引的字符串匹配检索，支持对 URL、标题、描述、标签进行跨字段查询，响应时间控制在 200 毫秒以内。

**外部状态检测** 定时向已收录的 URL 发送 HEAD 请求，检测服务可用性与响应码，标记异常链接并生成健康报告。

**导入导出兼容性** 支持 JSON 与 CSV 两种数据交换格式，便于与其他工具链集成，也支持从浏览器书签文件导入链接。

**访问统计看板** 记录每条链接的点击次数与最后访问时间，提供热门资源排序，辅助团队识别高频使用的外部依赖。

**权限分级控制** 内置管理员、编辑者、只读用户三级权限，适用于团队协作场景，避免误删或误改生产环境链接。

**Webhook 变更通知** 当资源列表发生增删改操作时，可配置向指定 URL 发送变更通知，便于下游系统同步更新。

## 应用场景

**技术团队内部文档导航** 研发团队通常维护数十个外部依赖的文档地址，例如数据库官方手册、消息队列客户端指南、监控系统 API 文档等。本项目可将这些分散链接统一收录，按项目或按技术栈分类，新成员入职时仅需访问本项目即可获得全部必要外部参考。

**开源项目 README 外链管理** 开源项目在 README 中引用大量第三方资源时，直接罗列长 URL 会使文档冗长且难以维护。本项目可作为独立的外链索引服务，在 README 中仅放置本项目的地址，由本项目集中管理所有外部链接的增删改，避免频繁修改 README 本身。

**技术调研资料收集** 在进行技术选型或性能调优时，工程师需要同时参考多份博客、官方文档、性能测试报告与社区讨论帖。本项目可用于临时收集和分类这些调研资料，并支持按调研主题打标签，后续复盘时能够快速定位到原始材料。

**自动化巡检依赖健康度** 运维团队可以利用本项目的外部状态检测功能，对生产环境依赖的外部 API 或第三方服务端点进行定时可达性检查。当某个外部服务出现异常时，系统可输出告警信息，帮助运维人员提前发现上游故障。

## 快速开始

以下指令适用于 Linux 与 macOS 环境，Windows 用户建议使用 WSL2 或 Git Bash。

```bash
# 克隆项目仓库
git clone https://github.com/your-org/rymfs-agg.git
cd rymfs-agg

# 安装项目依赖（基于 Python 3.10+）
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 初始化本地数据库并启动开发服务
python manage.py migrate
python manage.py loaddata resources.json
python manage.py runserver --host 0.0.0.0 --port 8000
```

启动成功后，访问 http://localhost:8000 可进入资源列表主页。默认管理员账号为 admin，密码在首次启动时输出于终端日志中，请及时修改。

## 项目结构

```
rymfs-agg/
├── cmd/
│   └── server/                   # 服务入口，包含 main.go 与路由初始化
├── internal/
│   ├── storage/                  # 数据库连接池与 CRUD 操作封装
│   ├── index/                    # 内存倒排索引构建与查询逻辑
│   ├── checker/                  # 外部状态检测调度器与 worker 池
│   ├── webhook/                  # Webhook 定义、队列与重试机制
│   └── auth/                     # 基于 JWT 的认证与权限中间件
├── pkg/
│   ├── models/                   # 资源、标签、访问日志的数据结构定义
│   └── utils/                    # 通用工具函数，如 URL 规范化与时间处理
├── api/
│   └── openapi.yaml              # OpenAPI 3.0 规范文档，用于生成客户端 SDK
├── web/
│   ├── templates/                # Go 模板引擎的 HTML 视图文件
│   └── static/                   # CSS、JavaScript 与图标等前端静态资源
├── configs/
│   ├── dev.yaml                  # 开发环境配置，启用调试日志与热加载
│   └── prod.yaml                 # 生产环境配置，关闭调试并启用压缩
├── scripts/
│   ├── migrate.sh                # 数据库迁移脚本，自动执行 schema 更新
│   └── seed.sh                   # 初始化测试数据与默认标签分类
├── test/
│   ├── integration/              # 集成测试用例，需要真实数据库环境
│   └── e2e/                      # 端到端测试，模拟浏览器交互行为
├── go.mod                        # Go 模块依赖定义文件
├── go.sum                        # 依赖包哈希校验文件
├── Makefile                      # 构建任务编排，包含 build、test、lint 等目标
└── README.md                     # 当前文档
```

## 贡献指南

项目遵循标准开源贡献流程，所有提交需经过代码审查与持续集成检查。以下步骤适用于希望参与本项目开发的贡献者。

第一，在 GitHub 上 fork 本仓库到个人账户，然后 clone 到本地开发环境。建议在 dev 分支上创建新的特性分支，分支命名采用 feat/ 或 fix/ 前缀加简要描述。

第二，运行本地开发环境并编写测试用例。新增功能必须包含至少一个正向测试与一个异常测试。修复缺陷时需先编写可复现该缺陷的测试用例，再实施修复，确保测试通过。

第三，提交代码前执行 make lint 与 make test 命令，确保代码风格符合 golangci-lint 规范且所有测试通过。提交信息遵循 Conventional Commits 格式，即 type(scope): subject 的结构，type 取 feat、fix、docs、refactor、perf 等。

第四，向主仓库的 main 分支发起 Pull Request，并在 PR 描述中清晰说明变更目的、影响范围以及测试覆盖情况。至少需要一名项目维护者批准后方可合并。

## 常见问题

**问：资源列表中的链接如果失效怎么办？**

答：系统内置的状态检测模块会每小时对所有链接执行一次可达性检查。当连续三次检测到 4xx 或 5xx 状态码时，该资源会被标记为「异常」并在管理后台高亮显示。管理员可以手动编辑资源状态，将其置为「已失效」或「临时维护」。失效链接不会被自动删除，以便保留历史记录用于追溯。

**问：如何批量导入我自己的链接列表？**

答：项目支持通过命令行脚本进行批量导入。将链接按每行一个 URL 的格式保存在 plaintext 文件中，然后执行 `python manage.py import --file links.txt --tag custom`。导入前系统会进行去重校验，若发现与现有链接完全相同的 URL 则自动跳过并输出警告日志。也支持 CSV 格式，其中标题行需包含 url、title、description、tags 四列。

**问：能否将本项目部署为无数据库的纯静态站点？**

答：当前版本依赖 SQLite 存储元数据，无法直接编译为纯静态 HTML。但项目提供了 `export static` 命令，该命令会读取当前数据库中的所有资源并生成一组静态 HTML 页面，包含分类索引与搜索功能。生成的静态文件可部署到任何 HTTP 服务器上，但此时状态检测和 Webhook 等动态功能将不可用。
