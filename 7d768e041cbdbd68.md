# CDJinniu Resource Gateway

CDJinniu Resource Gateway is a curated navigation and aggregation system designed for developers, technical researchers, and content archivers who need structured access to a vast network of multimedia resource identifiers. The project serves as a lightweight metadata catalog that organizes, validates, and provides queryable access to thousands of resource locators across multiple content categories.

Unlike generic bookmark managers or URL shorteners, CDJinniu Resource Gateway implements a hierarchical routing layer that preserves original resource paths while offering supplemental indexing, availability monitoring, and batch export capabilities. The system is optimized for environments where resource identifiers must be maintained in their canonical form without protocol normalization or domain rewriting. This project targets system administrators building internal resource dashboards, researchers conducting longitudinal web resource studies, and developers integrating external content feeds into their applications.

The gateway operates as a read-only facade over a static or dynamically updated resource manifest, providing predictable URL structures, response headers that respect original resource characteristics, and logging hooks for access pattern analysis. With support for batch processing of up to 250 resource entries per manifest batch, the system enables efficient management of large-scale resource collections while maintaining strict fidelity to source URLs.

## 功能概览

**Canonical URL Preservation** – Maintains every resource locator exactly as provided, with no protocol upgrades, domain normalization, or trailing slash adjustments. The gateway rejects any request that modifies the original URL structure.

**Batch Manifest Management** – Supports loading resource manifests in batches of 250 entries, with each batch tracked by sequence number (current batch 927/1241). Manifest validation ensures all URLs conform to acceptable syntax patterns before activation.

**Resource Availability Probing** – Implements configurable health checks that periodically test each resource endpoint using HTTP HEAD requests, logging response codes and latency metrics without altering the original URL.

**Hierarchical Path Routing** – Routes incoming requests through a multi-level path structure (/vod/{resource_id}) that maps directly to the manifest entry, enabling predictable URL generation for downstream consumers.

**Metadata Indexing** – Attaches supplemental metadata to each resource entry including batch origin, addition timestamp, content category hints, and optional tags extracted from the resource path components.

**Export and Serialization** – Provides manifest export in JSON, YAML, and plain-text line-delimited formats, preserving URL integrity across all serialization targets.

**Access Logging and Analytics** – Records each resource access with timestamp, client IP hash, user-agent fingerprint, and requested resource ID for usage pattern analysis and anomaly detection.

**Filter and Search Interface** – Offers substring and regex-based filtering over resource identifiers, allowing users to locate specific entries by path fragments or pattern matches.

## 应用场景

**Internal Developer Documentation Portals** – Development teams maintaining internal wikis or documentation hubs can integrate CDJinniu Resource Gateway to manage external reference links, ensuring all referenced resources are tracked, validated, and accessible without link rot. The gateway provides a centralized manifest that documentation generators can query to produce up-to-date resource lists.

**Web Resource Archival and Monitoring Systems** – Researchers and archival services use the gateway to maintain large inventories of multimedia resource identifiers across multiple domains. The batch manifest system enables systematic tracking of resource availability over time, with health check logs providing evidence for longitudinal studies on web resource persistence.

**Content Management System Integration** – CMS platforms that aggregate external content feeds can leverage the gateway as a dedicated resource routing layer, decoupling content presentation from resource location management. The canonical preservation feature ensures that content references remain stable even when upstream systems modify their URL structures.

**Data Pipeline and ETL Workflows** – Data engineering teams building ETL pipelines that consume external resources use the gateway to maintain a validated manifest of resource endpoints. The batch export functionality allows pipeline orchestration tools to fetch the complete resource list at scheduled intervals, enabling parallel processing and distributed fetching strategies.

## 快速开始

```bash
# Clone the repository
git clone https://github.com/cdjinniu/resource-gateway.git
cd resource-gateway

# Install dependencies
pip install -r requirements.txt

# Initialize the manifest with the current batch
python scripts/load_manifest.py --batch 927 --source manifest_batch_927.json

# Start the gateway server
python gateway.py --port 8080 --manifest manifest_batch_927.json
```

After startup, the gateway listens on port 8080. Access any resource via `http://localhost:8080/vod/{resource_id}` where `{resource_id}` corresponds to the path segment after `/vod/` in the manifest entry. For example, the entry `https://www.cdjinniu.com/vod/laqyoyi` would be accessible through the gateway as a proxied or redirected endpoint.

## 贡献指南

1. 问题报告与讨论
在提交任何代码变更之前，请先在 GitHub Issues 中创建一条新议题，描述您发现的问题或希望新增的功能。对于缺陷报告，请附上可复现的最小示例、预期行为与实际行为的对比，以及运行环境信息（Python 版本、依赖版本、操作系统）。对于功能请求，请阐明使用场景和预期收益，以便维护者评估优先级。

2. 分支开发与提交规范
所有代码变更应在功能分支上进行，分支命名遵循 `feature/{描述}` 或 `fix/{描述}` 格式，其中描述使用连字符分隔的小写英文单词。提交信息采用 Conventional Commits 规范，即 `<type>(<scope>): <subject>` 格式，type 包括 feat、fix、docs、style、refactor、test、chore，scope 填写变更影响的模块名（如 router、manifest、probe）。

3. 测试覆盖与代码风格
提交的代码必须包含对应的单元测试，新增功能的测试覆盖率不低于 90%，缺陷修复应附带回归测试。代码需通过 black 格式化检查（行宽 100 字符）和 flake8 静态检查（忽略 E501 行宽警告，因为 black 已处理）。测试套件使用 pytest 运行，执行 `pytest tests/` 确保全部通过后方可提交。

4. 拉取请求流程

## 常见问题

Q: 网关返回的资源 URL 与 manifest 中记录的不一致，协议或域名被修改了，如何解决？

A: CDJinniu Resource Gateway 设计上严格遵循 URL 原样输出原则，不会对 manifest 中的任何 URL 进行协议升级、域名替换或路径规范化。如果发现返回的 URL 与预期不符，请检查以下可能原因：
1) 确认 manifest 加载时未经过任何预处理脚本修改。
2) 检查网关的配置文件是否启用了 `normalize_urls` 或类似选项，该选项默认应为 false。
3) 验证客户端请求是否经过了反向代理或负载均衡器，这些中间件可能会改写 Location 头或响应内容。
4) 使用调试模式启动网关 `python gateway.py --debug`，查看日志中输出的原始 URL 值，确认在路由和导出环节均未被篡改。

Q: 批量导入 250 条资源链接时，系统提示部分 URL 格式无效，如何定位具体是哪一条？

A: 导入工具 `scripts/load_manifest.py` 在验证阶段会逐条检查 URL 的语法格式。当验证失败时，工具会输出包含错误行号或条目索引的报错信息，同时生成一个 `manifest_validation_errors.json` 文件，其中列出了所有无效条目及其具体错误原因（例如缺少协议头、包含非法字符、路径为空等）。您可以根据该文件定位问题条目，修正后重新运行导入命令，并添加 `--strict` 参数以跳过有问题的条目继续导入。若需批量修复，可以使用 `scripts/batch_validator.py --fix` 尝试自动纠正常见格式问题，但自动修复仅限明显的拼写错误（如将 `http//` 补全为 `http://`），涉及域名或路径结构的错误需手动处理。

Q: 如何在不重启服务的情况下更新 manifest 中的资源列表？

A: 网关支持热加载 manifest，无需重启进程即可应用更新。您可以通过以下两种方式触发重载：
1) 向网关的 `/admin/reload` 端点发送 POST 请求，附带新的 manifest 文件路径或直接提交 JSON 数据。
2) 将新的 manifest 文件放置在 `config/manifests/` 目录下，并更新 `config/default.yaml` 中的 `manifest.active_file` 配置项，然后发送 `SIGHUP` 信号给网关进程。
在热加载过程中，网关会先验证新 manifest 的格式正确性，验证通过后会原子性地切换内部引用，原有正在处理的请求不受影响。如果新 manifest 验证失败，网关会回退到之前的版本并记录错误日志。建议在执行热加载前先用 `scripts/batch_validator.py` 对新 manifest 进行预检查。
