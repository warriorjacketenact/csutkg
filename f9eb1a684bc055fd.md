# Fooklam Resource Index

Fooklam Resource Index 是一个面向开发者、技术研究人员与信息分析师的轻量级外链资源归集与导航系统。该项目不对资源内容进行二次存储或深度加工，而是以结构化索引的方式，将分散于网络各处的技术文档、工具页面、参考手册与数据面板统一收束至可检索、可分类、可版本追踪的目录体系之下。

项目定位为“技术外链的静态汇编层”，适用于需要频繁引用外部参考资料、维护团队知识地图或构建个人研究入口的场景。Fooklam Resource Index 不依赖数据库，不采集用户隐私，仅通过纯静态索引表与约定式目录结构，为使用者提供稳定、可审计、可离线浏览的外链引用基底。

## 功能概览

**多级分类索引**：支持按技术领域、文档类型、来源机构等维度对资源链接进行分层归置，每个链接可挂载多个标签，便于交叉筛选。

**原始链接直出**：所有收录链接均以原始字符串形式呈现，不附加跳转中间页、不进行短链包装、不添加追踪参数，确保链接的原始语义与访问路径完全保留。

**批量导入与校验**：支持通过 CSV 或纯文本列表批量导入链接，系统自动执行协议一致性检查、重复项检测与域名可达性预检，降低人工整理成本。

**版本化快照记录**：每次索引更新均生成时间戳标记的变更日志，记录新增、移除与分类调整操作，便于回溯资源收录历史。

**静态站点生成**：基于模板引擎将索引数据渲染为纯静态 HTML 页面，可直接部署至任意 Web 服务器或对象存储服务，无需运行时环境支持。

**全文检索与过滤**：内置标题、标签与描述字段的轻量级全文搜索功能，支持按分类、按添加时间、按访问频次等多维度过滤排序。

## 应用场景

技术团队内部知识库外链整合。当团队使用 Wiki 或文档平台维护内部技术资料时，可将 Fooklam Resource Index 作为独立的外链管理模块，集中存放所有对外部文档、API 参考、社区讨论的引用链接，避免链接散落于各处造成维护混乱。

个人技术研究者的阅读清单管理。研究人员在阅读技术博客、论文预印本或开源项目文档时，可使用该项目快速收录待读或已读链接，并按主题分类构建个人学习路径。

开源项目文档的外部参考附录。开源项目维护者可在项目文档中嵌入 Fooklam Resource Index 生成的链接列表，统一呈现依赖库、相关工具与延伸阅读的外部资源，提升文档的完整性与可追溯性。

技术资讯聚合与筛选。内容编辑或社区运营人员可通过该项目批量整理技术峰会演讲录像、发布会回放、产品公告等时效性资源，形成按时间线排列的资讯索引页。

## 快速开始

以下指令适用于 Linux 与 macOS 环境，Windows 用户可使用 WSL 或 Git Bash 执行。

```bash
git clone https://github.com/fooklam/resource-index.git
cd resource-index
pip install -r requirements.txt
python build.py --input data/links.csv --output dist/
```

执行完成后，dist 目录下将生成完整的静态索引页面，可使用任何 HTTP 服务器进行本地预览，例如：

```bash
python -m http.server 8000 --directory dist/
```

## 项目结构

```
resource-index/
├── build.py                 # 主构建脚本，负责读取索引数据并渲染静态页面
├── config.yaml              # 全局配置文件，包含站点标题、输出路径、分页大小等参数
├── requirements.txt         # Python 依赖列表
├── README.md                # 项目说明文档
├── LICENSE                  # MIT 许可证文件
├── data/                    # 数据目录，存放所有索引源文件
│   ├── links.csv            # 主链接索引表，字段包含 id, title, url, tags, description, date_added
│   ├── categories.csv       # 分类定义表，定义分类名称、父级分类与显示顺序
│   └── changelog.json       # 变更日志，记录每次构建的链接增减与分类调整
├── src/                     # 源码目录
│   ├── parser.py            # 解析器模块，负责校验与规范化原始链接数据
│   ├── renderer.py          # 渲染器模块，调用模板引擎生成 HTML 页面
│   ├── checker.py           # 链接可达性检查模块，发送 HEAD 请求验证域名有效性
│   └── utils.py             # 通用工具函数，包括日期格式化、字符串处理与哈希计算
├── templates/               # Jinja2 模板目录
│   ├── base.html            # 基础布局模板，包含头部导航与底部版权信息
│   ├── index.html           # 首页模板，展示分类概览与最近更新
│   ├── category.html        # 分类页模板，展示该分类下的所有链接条目
│   └── detail.html          # 详情页模板，展示单个链接的完整描述与标签
├── static/                  # 静态资源目录
│   ├── css/
│   │   └── style.css        # 自定义样式表，基于无框架纯 CSS 实现响应式布局
│   ├── js/
│   │   └── search.js        # 前端搜索逻辑，实现即时过滤与关键词高亮
│   └── assets/              # 图片与字体等公共资源
├── tests/                   # 单元测试目录
│   ├── test_parser.py       # 解析器功能测试
│   ├── test_checker.py      # 链接检查器测试
│   └── test_utils.py        # 工具函数测试
├── docs/                    # 文档目录
│   ├── user-guide.md        # 用户手册
│   ├── configuration.md     # 配置参数详解
│   ├── developer-guide.md   # 开发者指南
│   └── design.md            # 设计文档
└── dist/                    # 构建输出目录（由 build.py 生成，不纳入版本控制）
    ├── index.html
    ├── category/
    ├── detail/
    └── static/
```

第一，在 GitHub 上 fork 本仓库，并克隆至本地开发环境。创建新的功能分支时，请使用 feature/ 或 fix/ 前缀，例如 feature/support-json-import。

第二，对 data/ 目录下的 CSV 文件进行修改时，需严格遵守列定义与字段格式要求。新增链接必须包含有效的 id、title、url 与 tags 字段，其中 tags 为管道符分隔的多标签字符串，例如 "python|data-science|tutorial"。

第三，提交代码前需执行 flake8 与 pytest 进行代码风格检查与单元测试，确保所有测试用例通过且无新增警告。测试用例应覆盖新增功能或修复的缺陷。

第四，提交 pull request 时请在描述中清晰说明变更目的、影响范围与测试结果。若新增分类或修改索引结构，需同步更新 docs/ 目录下的对应文档。

## 常见问题

问：为什么项目不直接存储资源内容，而只保留外链？

答：本项目设计初衷为导航与索引工具，而非内容托管平台。仅保留外链可避免版权争议、降低存储成本，并确保用户始终访问资源来源的最新版本。同时，这种做法使得项目的维护重心聚焦于链接的组织质量与可用性校验。

问：链接可达性预检会不会影响构建速度？

答：预检功能默认关闭，仅在配置文件中显式启用时才会执行。对于数千条链接的索引，预检会增加数秒至数十秒的构建时间，具体取决于网络状况与目标服务器的响应速度。建议在定期维护时手动启用预检，日常构建时保持关闭。

问：能否将索引部署为动态网站，支持在线编辑链接？

答：当前版本定位为静态站点生成器，不提供在线编辑接口。若需要动态管理功能，建议结合第三方 CMS 或自建后台，将数据写入 CSV 后由构建脚本重新生成静态页面。后续版本将考虑提供 RESTful API 接口。
