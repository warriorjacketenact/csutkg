# Tangalooma Resource Index

Tangalooma Resource Index 是一个面向技术研究者和内容聚合者的高密度外链资源索引系统。本项目不对索引内容本身进行加工或二次分发，而是提供一个结构化的、可审计的、机器可读的资源映射清单，用于快速定位分散在多个路径节点下的目标文档。项目定位为技术资源导航工具，适用于需要批量管理外部链接、构建自动化采集管道或进行链接生命周期追踪的工程场景。目标用户包括数据工程师、爬虫开发者、学术资源整理人员以及企业知识库维护者。

本项目不提供代理、破解、绕过访问控制或侵犯任何第三方权益的功能。所有索引链接均来自公开可访问的路径，项目仅对这些路径进行目录化记录与分类展示，不改变原始资源的访问方式和内容状态。

## 功能概览

批量链接聚合：支持将数千条外部 URL 按照目录结构批量导入索引系统，自动提取路径参数与文件名特征，生成可查询的映射表。

多维度分类检索：基于 URL 路径中的目录分段（如 film 类型标识码）和文件名哈希特征，提供按前缀、按目录、按文件扩展名等多种检索方式。

快速部署与离线使用：项目完全基于静态 Markdown 与纯文本结构，无需数据库依赖，克隆后即可在本地环境完成索引构建与查询。

链接状态审计：内置链接可访问性检查脚本，可定期对索引中的每条记录进行 HTTP 状态码验证，输出失效链接报告。

扩展性接口：提供 JSON 和 CSV 格式的索引导出功能，便于与其他数据处理工具（如 Apache Airflow、Scrapy、Logstash）集成。

访问日志记录：记录每条链接的查询时间与来源 IP（可选），为后续访问频率分析提供原始数据。

## 应用场景

自动化采集任务配置：数据采集工程师可将本索引作为种子 URL 清单，直接导入 Scrapy 或 Puppeteer 采集框架，按照链接分类并发抓取目标页面内容，无需手动整理 URL 列表。

资源链路健康度监控：运维团队可配置定时任务，通过项目提供的链接状态检查脚本，对所有索引链接进行每日可用性探测，自动告警失效或响应超时的资源。

学术文献与参考资料归档：研究人员可将本项目作为外部参考文献的外部索引层，将分散在不同章节和项目中的外部链接统一登记在案，避免链接散落导致的引用丢失。

企业知识库外部依赖管理：企业文档团队可利用本索引建立外部资源依赖清单，当外部链接发生变更或迁移时，通过索引版本管理快速定位受影响的知识条目。

## 快速开始

以下命令可在任何支持 Bash 和 Git 的环境中完成项目的初始部署与运行。请确保系统已安装 Git、Python 3.8 及以上版本以及 pip 包管理工具。

```bash
git clone https://github.com/tangalooma/resource-index.git
cd resource-index
pip install -r requirements.txt
python index_builder.py --input ./resources/links.txt --output ./output/index.json
```

上述操作依次完成代码仓库克隆、依赖安装以及索引构建。其中 links.txt 为原始链接清单文件，每行一个 URL。构建完成后，index.json 为生成的索引映射文件，可供后续查询工具使用。

## 贡献指南

我们欢迎任何形式的贡献，包括但不限于新增链接索引、改进解析脚本、完善文档和修复缺陷。请遵循以下步骤提交贡献。

第一，在 GitHub 上 Fork 本仓库至您的个人账户，然后克隆到本地开发环境。请确保您的本地分支与主仓库的 main 分支保持同步。

第二，新增或修改链接时，请将 URL 按行追加到 resources/links.txt 文件末尾，或按照批次目录结构放置在相应的 batch_*.txt 文件中。每个 URL 必须单独占一行，且不允许包含尾部空白字符。提交前请运行 scripts/validator.py 进行格式校验。

第三，若您新增了脚本或修改了现有功能，请同步更新 tests/ 目录下的对应单元测试用例，并确保所有测试用例通过。执行 make test 可运行完整测试套件。

第四，提交 Pull Request 时，请填写清晰的标题和描述，说明本次变更的目的、影响范围以及测试结果。若变更为链接列表更新，请在描述中注明新增链接的来源或依据。

第五，项目维护者将在 3 个工作日内对 Pull Request 进行审阅。若审阅通过，您的变更将被合并至 main 分支，并自动触发索引重建流水线。

## 常见问题

Q：项目是否会定期自动检查链接可用性？
A：项目本身不包含内置的自动调度功能，但提供了 check_links.sh 脚本。用户可通过操作系统的 cron 或 systemd timer 配置定时任务，例如设置每日凌晨 2 点执行检查并将结果输出到 output/reports/ 目录。示例 cron 配置：0 2 * * * cd /path/to/resource-index && ./bin/check_links.sh。

Q：如果某个索引链接失效，项目会删除该链接吗？
A：项目不会自动删除任何链接。失效链接会被记录在当天的可用性日志中，但索引条目保持不动。用户可根据日志报告手动决定是否保留或移除特定链接。这种设计避免了自动化误删导致的索引不可逆丢失。

Q：如何导入新批次的链接而不覆盖现有索引？
A：您可以将新批次链接追加到 resources/links.txt 末尾，或创建新的 batch_*.txt 文件。运行 index_builder.py 时使用 --merge 参数，构建器会读取所有批次文件并合并生成新的 index.json，原有索引条目不会被覆盖，仅新增或更新变化部分。
