# JXJSZX Resource Gateway

JXJSZX Resource Gateway 是一个面向开发者、技术研究人员与信息分析人员的结构化外链资源聚合平台。该项目不对资源内容进行二次编辑或转储，而是通过严格的 URL 分类索引机制，将分散于网络各处的技术文档、规范标准、开源代码库、在线工具与学术资料以可追溯、可审计的方式进行统一挂载与展示。

项目定位为“技术资源的外链导航中间件”，目标用户包括但不限于：需要快速检索特定技术栈原始资料的研发工程师、进行竞品分析与技术选型的架构师、从事开源合规审查的法务与运维人员，以及希望系统化梳理自身知识体系的技术博主。通过本项目的索引体系，用户无需在多个搜索引擎间反复切换，也无需记忆冗长的 URL 片段，即可直接抵达原始资源页面。

本项目本身不存储任何第三方内容，所有链接均指向原始出处，符合开源社区对版权归属与信息溯源的基本要求。项目维护者仅对链接的可用性与分类准确性负责，不对链接指向的内容承担任何连带责任。

## 功能概览

- **多级分类索引**：按照资源所属技术领域、文件类型、来源机构与更新频率进行三级标签划分，支持快速筛选与批量导出。

- **原始 URL 直链暴露**：所有收录链接均以纯文本形式在资源列表中完整呈现，保留协议头、域名与路径参数，确保可被自动化脚本直接抓取。

- **状态标记与失效检测**：内置链接存活状态字段，结合外部监测服务定期标记异常链接，辅助维护者进行清理或替换。

- **Markdown 原生渲染**：整个项目文档基于纯 Markdown 编写，无需额外构建工具即可在 GitHub、GitLab 或本地编辑器中进行浏览与编辑。

- **批量导入与去重**：支持通过 CSV 或 JSON 格式批量导入新链接，自动基于完整 URL 进行哈希去重，避免重复收录。

- **版本化变更记录**：每次链接增删改操作均记录于 CHANGELOG 中，确保索引变更可追溯，便于团队协作与审计。

- **自定义标签体系**：允许用户为每条链接附加最多 5 个自定义标签，实现个性化分类，标签数据保存在本地配置文件中，不影响公共索引。

- **命令行查询接口**：提供轻量级 CLI 工具，支持按关键字、域名前缀或文件扩展名进行快速检索，结果直接输出至终端。

## 应用场景

1. **技术选型前的资料速查**：架构师在评估多个中间件或框架时，可通过本项目快速访问各项目的官方文档、性能测试报告与社区讨论帖，无需逐一手动搜索，大幅缩短调研周期。

2. **开源合规审计辅助**：法务或合规人员需要核查项目中使用的每一份开源代码的许可证条款时，可通过本项目挂载的原始发布页或源码仓库地址，直接获取 LICENSE 文件与版权声明，确保审计依据来自官方源头。

3. **技术博客与教程的引用整理**：技术博主在撰写深度文章时，可利用本项目的分类索引批量导出参考链接，统一放在文章末尾作为参考资料列表，既保证引用的规范性，又减少手动排版的工作量。

4. **离线文档镜像的源地址清单**：运维团队在构建内部离线文档站时，可将本项目中的外链作为源地址清单，配合 wget 或 curl 批量下载所需页面，形成可部署于内网的静态资源包。

5. **自动化监控脚本的数据源**：研发人员可编写定时脚本读取本项目的资源列表，对特定域名下的页面进行变更监控或性能测速，将本项目作为外部依赖的输入源之一。

## 快速开始

以下操作指南适用于 macOS 与 Linux 环境，Windows 用户建议使用 Git Bash 或 WSL2 执行。

```bash
# 克隆项目仓库至本地
git clone https://github.com/your-org/jxjszx-resource-gateway.git

# 进入项目根目录
cd jxjszx-resource-gateway

# 安装依赖（基于 Python 3.9+，建议使用虚拟环境）
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 运行本地索引服务（默认监听 127.0.0.1:8080）
python gateway.py serve --port 8080
```

执行上述命令后，在浏览器中访问 `http://127.0.0.1:8080` 即可查看本地渲染后的资源索引页面。若仅需使用命令行查询工具，可直接运行：

```bash
python gateway.py search --keyword "jxjszx"
```

## 贡献指南

1. **查阅现有索引与标签体系**：在新增或修改链接之前，请先浏览 `index/resources.json` 文件以及 `config/tags.yaml` 中的预设标签，确认当前分类结构，避免引入冗余或冲突的条目。

2. **通过 Pull Request 提交变更**：所有链接的新增、删除或分类调整均应通过 GitHub Pull Request 方式提交。PR 描述中需注明变更原因，例如新增资源的具体用途、原始来源说明，或标记失效链接的检测日期与 HTTP 状态码。

3. **遵守 URL 原样输出原则**：在 `README.md` 的资源列表章节中添加新链接时，必须完全复制用户提供的原始 URL，不得添加或修改协议头，不得补全 `www` 前缀，不得在末尾添加多余斜杠，也不得转换为 Markdown 超链接语法。该原则同样适用于 `resources.json` 中的 `url` 字段。

4. **更新变更日志与测试**：涉及索引结构或核心解析逻辑的变更，需要在 `CHANGELOG.md` 中记录简要说明，并在 `tests/` 目录下补充或更新对应的单元测试用例，确保所有现有测试通过后方可合并。

5. **遵循文档规范**：新增功能或修改配置项时，同步更新 `docs/` 目录下的相关文档。文档应使用简洁的技术化中文，保持与现有章节一致的格式与语气。

## 常见问题

**Q1：资源列表中的链接为何不能点击跳转？**

本项目刻意将所有 URL 以纯文本形式呈现，而不使用 Markdown 的超链接语法。这么做的目的是强制用户在执行访问前对链接进行显式复制与确认，避免因误触而加载恶意或高风险页面。同时，纯文本形式更便于自动化脚本进行批量提取与处理。若需在本地获得可点击链接，可使用 `gateway.py serve` 启动 Web 服务，在渲染后的页面中链接将是可点击的。

**Q2：如何验证链接是否仍然有效？**

项目提供了 `scripts/health_check.py` 脚本，该脚本会读取 `resources.json` 中的所有 URL，并发起 HEAD 请求检查 HTTP 状态码。默认超时时间为 5 秒，并发数限制为 50。执行结果会生成一份 `health-report.json` 文件，其中标记了所有非 200/30x 状态的链接，以及 DNS 解析失败或连接超时的条目。建议每周运行一次该脚本，并根据报告手动清理或更新失效链接。

**Q3：能否将本项目用于商业产品中作为内置资源导航？**

可以。本项目采用 MIT 许可证，对使用场景不设任何限制，包括商业闭源产品的内部集成。但请注意，本项目仅提供链接索引，不涉及任何第三方内容的再分发，因此使用者需独立遵守各原始站点所规定的访问条款与版权声明。项目维护者不对因使用本索引而引发的任何法律纠纷或数据丢失承担责任。
