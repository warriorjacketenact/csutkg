# Tangalooma Resource Aggregator

Tangalooma Resource Aggregator 是一个面向技术研究、内容归档与数字资源管理场景的开源外链汇总系统。项目定位于为开发者、数据分析师、内容策展人以及学术研究者提供一套结构化的外部资源索引框架，能够将分散在多个来源的原始 URL 按照自定义分类、标签体系和元数据模型进行统一管理与展示。与通用书签工具或收藏夹不同，本系统强调资源的可追溯性、批处理能力以及与自动化流水线的集成能力，适用于需要定期同步、校验和重新分发大量外链信息的运维级场景。

本项目并非简单的链接列表，而是一个完整的资源生命周期管理工具。它内置了链接可用性检测、元数据抓取、访问频率统计以及分类视图生成等模块，能够帮助用户从海量的原始外链中提取出具备长期价值的资料节点。项目默认集成了一批经过初步整理的影片类资源外链作为示例数据集，用户也可以完全替换为自己的 URL 列表，快速构建专属的资源导航站点或内部知识库入口。

## 功能概览

批量链接导入与解析：支持从纯文本文件、CSV 表格或直接粘贴的原始 URL 列表中批量导入外部链接，自动解析协议、域名、路径参数及查询字符串，并提取可读性较高的文件名或路径摘要作为默认标题。

自定义分类与标签体系：允许用户为每个资源链接分配一个主分类和多个自由标签，分类层级支持无限嵌套，标签支持模糊搜索与交集筛选，便于构建符合自身业务逻辑的资源组织视图。

资源元数据自动补全：内置基于 HTTP 头部解析与 HTML 元标签提取的元数据抓取引擎，能够自动补全资源的标题、描述、内容类型、最后修改时间等字段，减少手动录入工作。

链接健康状态监控：周期性对已收录的所有外链发起 HEAD 请求，检测返回状态码，识别失效链接、重定向链过长或响应超时的资源，并以仪表盘形式展示整体健康度。

多维度检索与过滤：提供基于关键词、分类、标签、域名、文件扩展名、添加时间范围等多条件组合的检索接口，支持搜索结果以 JSON 或 CSV 格式导出，便于二次处理。

访问统计与热度排序：记录每个资源链接的点击次数、最近访问时间以及来源渠道，支持按热度、新鲜度或字母序对资源列表进行动态排序，帮助用户发现高频使用的核心资料。

视图模板自定义：提供基于 Jinja2 模板引擎的前端展示层，用户可以自由修改列表页、详情页和分类页的 HTML 结构与 CSS 样式，满足品牌化部署或内网门户风格统一的需求。

数据导入导出与备份：支持将全部资源数据导出为标准的 JSON Lines 格式或 Markdown 表格格式，也支持从外部备份文件完整恢复整个索引库，便于版本控制与跨环境迁移。

## 应用场景

企业内部技术文档中心导航：技术团队可将本系统部署为内部文档门户的入口层，集中管理分散在 Confluence、GitLab Wiki、S3 存储桶以及外部技术博客中的各类参考资料。通过统一的分类与标签体系，新入职员工可以快速找到环境搭建指南、架构设计提案和故障排查手册等关键资源。

学术研究参考文献索引管理：研究人员在进行文献综述或 meta 分析时，通常需要追踪数十甚至上百个在线数据源、预印本仓库和数据集页面。本系统能够为每个外链记录添加研究主题、数据格式、采集时间等自定义字段，并定期检查链接有效性，避免因源站改版导致引用失效。

内容聚合站点后端数据源管理：面向垂直领域的内容平台（如影视资讯站、技术新闻聚合器）可使用本系统作为后端数据源管理面板，运营人员统一录入待抓取或待引用的外部链接，系统按分类输出结构化的链接清单供前端页面或 RSS 生成器调用，实现内容源与展示层的解耦。

自动化运维监控告警附属信息索引：在运维监控体系中，本系统可作为告警事件附件的索引中心。当监控系统检测到异常时，可通过 API 从本系统中检索与该故障类型相关的历史处理记录、修复脚本链接或外部厂商状态页地址，缩短平均修复时间。

## 快速开始

以下命令演示了从 GitHub 克隆项目、安装依赖并启动开发服务器的完整流程。执行前请确保系统已安装 Python 3.9 及以上版本和 Git。

```bash
git clone https://github.com/tangalooma/resource-aggregator.git
cd resource-aggregator
pip install -r requirements.txt
python app.py --init-db --load-sample
python app.py --serve --port 8080
```

首次运行时，`--init-db` 参数会在本地 SQLite 数据库中创建所需的表结构，`--load-sample` 会将项目内置的示例资源链接（即本 README 资源列表章节所收录的 URL）导入数据库。若希望清空示例数据并导入自己的链接列表，可省略 `--load-sample` 并改用 `--import-file /path/to/your/urls.txt`。

## 安装要求

本项目的运行依赖以下系统组件与 Python 库。生产环境部署时建议使用 Python 虚拟环境，并确保所有依赖版本与表中列出的推荐版本一致或更高。

| 依赖组件 | 必需性 | 说明 |
|----------|--------|------|
| Python 3.9+ | 必需 | 核心运行环境，低于 3.9 版本将无法解析部分类型注解语法 |
| SQLite 3.35+ | 必需 | 默认元数据存储引擎，支持 JSON 字段类型以存放标签与扩展属性 |
| requests 2.28+ | 必需 | 用于发送 HTTP 请求以检测链接状态和抓取元数据 |
| jinja2 3.1+ | 必需 | 模板渲染引擎，用于生成前端展示页面 |
| click 8.1+ | 必需 | 命令行交互框架，解析 `--serve`、`--import-file` 等子命令参数 |
| pytest 7.0+ | 可选 | 仅在运行单元测试套件时需要，生产环境可不安装 |
| gunicorn 20.1+ | 可选 | 生产环境推荐使用 Gunicorn 作为 WSGI 服务器，开发环境可使用内置 werkzeug |
| black 23.0+ | 可选 | 代码格式化工具，仅对项目贡献者开发阶段有用 |

## 项目结构

项目源码的目录组织遵循分层架构原则，核心业务逻辑、数据访问层、展示层和工具函数相互独立，便于单元测试和功能替换。

```

├── requirements.txt            # Python 依赖声明，锁定主要库版本号
├── config/

│   └── exporter.py             # 数据导出模块，支持 JSON Lines 与 Markdown 格式
├── storage/
