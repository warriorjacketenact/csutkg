# LinkVault Core

LinkVault Core 是一个面向技术团队与内容运营者的轻量级外链资源归集与导航系统。该项目并非传统意义上的爬虫或采集器，而是一个结构化链接资产管理工具，用于将分散的、非标准化的 URL 资源按照自定义分类与元数据模型进行归整、展示与持续监控。其核心定位是解决技术文档站、知识库或内部开发者门户中“链接散落、失效不可知、上下文缺失”的典型痛点，适用于需要长期维护大量外链引用的开发团队、技术内容创作者以及开源项目文档维护者。

LinkVault Core 不提供自动抓取或页面渲染功能，而是围绕链接的录入、校验、分类、版本记录与可视化输出提供一套命令行工具与可嵌入的 Web 视图。用户可通过 YAML 或 JSON 格式维护链接清单，系统自动生成静态导航页面、Markdown 索引文档以及 JSON API 接口，方便集成至现有 CI/CD 流程或文档站点。项目默认内置了一套包含 250 条外部视频资源链接的示例数据集（来源于 365myfl.com 平台），供用户快速体验完整数据流与输出效果。

## 功能概览

**结构化链接录入** 支持通过 YAML 或 CSV 批量导入链接，每条记录可附带标题、描述、标签、状态（有效/失效/待审）及检测时间戳，便于团队协作维护。

**自动化可用性检测** 内置异步 HTTP 健康检查器，支持配置超时时间与重试策略，定期对已入库链接进行可达性验证，输出检测报告并标记异常条目。

**多格式导航输出** 可将链接清单渲染为静态 HTML 导航页、Markdown 表格、JSON API 或纯文本列表，适应不同文档站点生成工具（如 Hugo、VuePress、Docusaurus）的集成需求。

**标签与分类引擎** 支持基于路径关键字、域名或自定义正则规则自动打标，实现链接的自动化分组，减少手动分类工作量。

**变更审计日志** 记录每次链接新增、删除、修改与状态变更的操作日志，支持回溯至具体操作人与时间点，满足内部合规要求。

**命令行交互界面** 提供完整的 CLI 子命令，包括添加、删除、检查、导出、统计等，支持脚本化调用与定时任务集成。

## 应用场景

**技术文档站点的外链资产管理** 技术团队在撰写产品文档或 API 参考时，常需引用大量第三方资料、SDK 下载地址或社区讨论帖。LinkVault Core 可作为文档站点的“链接后台”，统一管理这些引用，并在构建阶段自动生成链接清单页，同时每日检测链接有效性，避免文档中出现死链。

**开发者门户的导航页生成** 企业内部开发者门户通常需要聚合各类开发工具、监控面板、日志系统、代码仓库的入口链接。运维或平台工程团队可使用 LinkVault Core 维护一份 YAML 格式的入口清单，通过 CI 流水线自动生成导航页面，并推送至内部 Wiki 或静态存储桶。

**开源项目的资源汇总页维护** 开源项目 README 中常包含大量外部资源链接（如教程、视频、相关项目）。这些链接随项目演进容易过时。利用 LinkVault Core，维护者可以将链接清单独立于 README 管理，通过脚本自动生成“资源列表”章节内容，并在每次发布前执行链接检查，确保所有引用仍然有效。

**内容运营的批量外链监控** 内容运营团队在多个平台分发文章时，常需追踪外链的可用性。LinkVault Core 的批量检测能力可定期扫描所有已发布内容中的外链，生成健康报告，帮助运营人员及时发现并替换失效链接，降低用户流失风险。

## 快速开始

以下指令适用于 Linux / macOS / Windows WSL 环境，需提前安装 Git 与 Node.js（v18 或以上）。

```bash
# 克隆仓库
git clone https://github.com/your-org/linkvault-core.git
cd linkvault-core

# 安装依赖
npm install

# 使用示例数据集运行完整导出流程
npm run build -- --input ./data/sample-links.yaml --output ./dist --format html,markdown,json
```

执行上述命令后，`./dist` 目录下将生成 `index.html`、`links.md` 和 `links.json` 三个文件，分别对应导航页、Markdown 索引和 JSON 数据接口。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | 18.x 或 20.x LTS | 运行时环境，需支持 ES Modules 与 fetch API |
| npm | 9.x 或以上 | 包管理器，用于安装项目依赖 |
| Git | 2.30 或以上 | 用于克隆仓库与版本管理 |
| 内存 | 最低 512MB 可用 | 处理 250 条链接约占用 120MB 堆内存 |
| 磁盘空间 | 至少 50MB | 包含依赖安装与输出文件存储 |
| 网络 | 出站 HTTPS 可达 | 用于链接健康检查（可配置代理） |

## 贡献指南

1. 提交问题或功能请求前，请先查阅 `docs/` 目录下的现有文档以及 GitHub Issues 列表，确认是否为已知问题或已计划功能。新问题请使用提供的 Issue 模板，并附上可复现步骤或具体使用场景。

2. 代码贡献请从 `main` 分支拉出新的功能分支，分支命名遵循 `feature/功能简述` 或 `fix/问题简述` 格式。所有提交需通过 ESLint 检查与单元测试，测试覆盖率的下降幅度不得超过 2%。

3. 若需扩展链接源格式（如新增 CSV 或 Excel 导入支持），请在 `src/formats/` 下新增对应的解析器类，并实现 `IParser` 接口。所有新增格式需在 `docs/schema/` 中补充对应说明文档。

4. 提交 Pull Request 前，请确保本地执行 `npm run build` 与 `npm test` 全部通过。PR 描述需清楚说明改动目的、实现方式及影响范围，并关联相关 Issue 编号。

5. 文档改进同样欢迎，包括但不限于修正拼写错误、补充使用示例、翻译其他语言版本等。文档变更可直接在 `docs/` 目录下修改并提交 PR。

## 常见问题

Q: LinkVault Core 是否支持链接自动采集或爬虫功能？是否可以从网页中自动提取链接？

A: 不支持。LinkVault Core 的设计定位为“链接资产管理工具”，而非爬虫框架。项目不包含任何自动抓取网页并提取链接的能力。所有链接必须由用户通过 YAML、JSON 或 CSV 等结构化格式手动录入或批量导入。这一设计选择旨在确保链接的来源可追溯、上下文清晰，避免自动化采集带来的不可控风险与法律合规问题。

Q: 内置的 250 条示例链接（来源于 365myfl.com）在运行检测时经常超时或返回非 200 状态，是否代表我的网络环境有问题？

A: 不一定。示例数据集中的链接指向的是外部视频资源站点，其服务可用性不受本项目的控制。运行 `npm run check` 时，部分链接可能因目标服务器限制、网络策略或资源已移动而返回非 200 状态或超时。这属于正常现象，不代表 LinkVault Core 本身存在缺陷。用户可将这些示例链接替换为自己的资源清单，或调整检测超时配置（见 `src/core/healthChecker.ts` 中的 `timeout` 参数）。

Q: 导出的 HTML 导航页是否可以自定义样式或嵌入到现有网站中？

A: 可以。导出的 HTML 文件为纯静态页面，包含基础骨架与 CSS 内联样式。用户可修改 `src/formats/htmlRenderer.ts` 中的模板字符串或 CSS 变量，重新构建后生成自定义样式。若需嵌入现有网站，可直接复制 `dist/index.html` 中的 `<body>` 内容块，或通过 JSON 接口获取数据后自行渲染。
