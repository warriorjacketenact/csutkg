# i-Fights Theater Resource Aggregator

i-Fights Theater Resource Aggregator 是一个面向技术内容研究者和数字档案管理者的外链资源汇总系统。该项目专注于对 i-fights.net 平台下 theater 子目录中大量独立资源页面进行结构化收录、分类索引与元数据提取，为后续的数据分析、内容归档和知识图谱构建提供基础数据层支持。

项目定位为技术化的外链治理工具，而非最终用户浏览站点。主要服务于需要批量处理、检索和监控 theater 类型资源链接的开发者、数据工程师和学术研究人员。通过对 URL 模式、响应状态码、页面元信息的持续采集与版本追踪，该项目能够有效降低人工维护数百乃至数千个外链资源时的认知负担与操作成本。

## 功能概览

批量资源链接收录：支持一次性导入大量 theater 资源 URL，自动完成去重、格式校验和基础连通性测试，输出结构化资源清单。

元数据自动提取：对每个收录的 URL 发起 HTTP 请求，解析 HTML 标题、meta description、正文关键词及最后修改时间，生成标准化的资源描述记录。

分类标签体系：基于 URL 数字 ID 段、页面内容特征和响应头信息，为每条资源自动打上内容类型、语言、更新频率等多维度标签，便于后续筛选。

增量更新与变更检测：定期重新抓取已收录资源，比对页面摘要与状态码变化，生成变更日志，标记失效、重定向或内容大幅改动的链接。

查询过滤与导出：支持按标签、状态、更新时间区间等条件过滤资源列表，结果可导出为 JSON、CSV 或 Markdown 表格格式，方便下游工具使用。

RESTful API 接口：提供标准化的 HTTP API，允许外部系统通过 GET /resources、POST /resources/scan、PUT /resources/{id} 等端点进行程序化访问与控制。

任务队列与并发控制：内置基于工作队列的异步抓取调度器，支持配置并发数、请求间隔和超时重试策略，避免对目标服务器造成过大压力。

本地缓存与持久化：使用 SQLite 作为默认存储后端，缓存每个资源的响应摘要和元数据，支持导出为外部 PostgreSQL 或 MySQL 数据库。

## 应用场景

数字档案维护与镜像站建设：团队可利用本项目的批量收录和变更检测功能，定期对 i-fights.net/theater 下的资源进行快照记录，当原始链接发生移动或失效时及时获得通知，从而维持镜像站或参考链接库的可用性。

技术内容聚合与推荐系统研发：数据工程师通过 API 获取全部资源标题和标签信息，构建基于内容的简单推荐原型，或者将元数据导入 Elasticsearch 实现全文检索，方便内部知识库的快速定位。

SEO 监控与反向链接分析：运营人员可将该项目配置为周期性任务，监控特定 ID 段资源的响应码和标题变化，识别可能存在的页面下架、内容替换或重定向链，作为网站健康度评估的辅助数据源。

学术研究与网络文化分析：社会科学或传播学研究者可通过导出全部资源的元数据表格，进行 URL 命名规律、发布时间分布、内容主题聚类等定量分析，无需手工逐个访问页面。

自动化测试数据源供给：开发者在进行爬虫框架或 HTML 解析库的性能测试时，可将本项目收录的 URL 列表作为稳定且多样化的测试输入，用于验证解析逻辑的健壮性和请求模块的并发能力。

## 快速开始

以下步骤适用于 Linux 与 macOS 环境，Windows 用户建议使用 WSL2 或 Git Bash。

```bash
# 克隆仓库
git clone https://github.com/example/i-fights-aggregator.git
cd i-fights-aggregator

# 安装依赖（推荐使用 Python 3.10 及以上版本）
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 初始化数据库并执行首次收录（默认读取 resources.txt 中的 URL 列表）
python cli.py init
python cli.py scan --input resources.txt --output data/initial_manifest.json
python cli.py serve --port 8000
```

执行上述命令后，项目会在本地 8000 端口启动 Web API 服务，可通过 http://127.0.0.1:8000/docs 查看自动生成的 Swagger 接口文档。

## 项目结构

```
i-fights-aggregator/
├── cli.py                     # 命令行入口，提供 init / scan / serve / export 子命令
├── requirements.txt           # Python 依赖清单，锁定主要库版本
├── .env.example               # 环境变量模板，配置数据库路径、并发数、超时阈值
├── resources.txt              # 待收录的原始 URL 列表，每行一个，支持 # 注释
├── src/                       # 核心源代码目录
│   ├── __init__.py
│   ├── fetcher.py             # 异步 HTTP 请求模块，封装 retry、timeout 和 user-agent 轮换
│   ├── parser.py              # HTML 解析模块，提取 title、meta、h1 及正文前 200 字符
│   ├── model.py               # Pydantic 数据模型，定义 Resource, Manifest, Tag 等实体
│   ├── storage.py             # SQLite 存储层，包含建表语句、插入更新和查询方法
│   ├── scheduler.py           # 任务队列调度器，控制并发抓取与速率限制
│   └── api.py                 # FastAPI 应用，注册所有 RESTful 路由与异常处理器
├── tests/                     # 单元测试与集成测试目录
```

## 常见问题

Q: 扫描过程中部分 URL 返回超时或 5xx 错误，项目会如何处理？

A: fetcher 模块内置了指数退避重试机制，默认最多重试 3 次，间隔分别为 2 秒、4 秒和 8 秒。若全部重试后仍失败，该资源的状态将被标记为 unreachable，并记录最后一次错误原因。用户可通过 cli.py scan --retry-failed 命令单独重试所有失败项，或通过 API 的 PUT /resources/{id}/retry 端点进行单条重试。

Q: 如何修改并发抓取速率以防止对目标服务器造成过大压力？

A: 可在项目根目录的 .env 文件中设置 MAX_CONCURRENT 和 REQUEST_INTERVAL 两个环境变量。MAX_CONCURRENT 控制同时进行的请求数量，默认值为 10；REQUEST_INTERVAL 控制每个请求完成后的等待秒数，默认值为 0.5。建议对 i-fights.net 域名将并发数调低至 5 以下，并设置间隔为 1 秒以上，以避免触发服务端限流策略。

Q: 数据库文件逐渐增大，是否有推荐的维护方式？

A: 项目使用 SQLite 作为存储后端，建议每月执行一次 VACUUM 命令回收已删除记录占用的空间。用户可通过 sqlite3 data/aggregator.db "VACUUM;" 手动执行，或使用 cli.py db --vacuum 命令触发。此外，可通过 cli.py export --archive 将 180 天前已确认稳定的资源记录导出为归档文件并从主表中移除，以保持主表查询性能。
