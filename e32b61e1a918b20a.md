# LinkVault

LinkVault 是一个面向技术团队与内容运营者的轻量级外链资源聚合与分发系统。该项目定位于解决分散在各类文档、笔记、社交消息中的外部链接难以统一管理、无法批量校验可用性、缺乏访问统计与分类检索的问题。LinkVault 提供基于目录树与标签双维度的链接组织方式，支持链接状态监控、访问频次分析以及快照备份，适用于技术文档站、个人知识库、社区导航页等多种场景。

目标用户包括开发者、技术博主、文档维护者、项目经理以及任何需要长期维护大量外部引用链接的知识工作者。LinkVault 不依赖外部数据库，以纯文本与 JSON 格式存储链接元数据，便于版本控制与脚本化处理，同时提供可扩展的插件接口用于对接监控告警、定时任务等周边系统。

## 功能概览

**链接条目管理**：支持添加、编辑、删除单条链接，每条链接可记录标题、分类标签、备注说明、添加时间与最后访问时间。链接条目以扁平化 JSON 数组存储，便于脚本批量导入导出。

**目录树组织**：允许用户创建多级目录结构，将链接按主题或项目层级归类。目录配置独立存储，支持拖拽式调整层级关系，目录变更不影响链接条目本身。

**批量导入与校验**：支持从 CSV 文件、纯文本列表或浏览器书签导出文件批量导入链接。导入后自动发起 HEAD 请求校验链接可达性，返回 HTTP 状态码与响应时间，标记失效链接。

**访问统计与快照**：每次访问链接时记录时间戳与来源 IP 摘要，提供链接点击频次热力图与趋势折线图。支持配置定时任务对重要链接生成网页快照（MHTML 格式），防止内容下线后丢失参考信息。

**标签过滤与全文检索**：每条链接可附加多个标签，支持标签组合过滤。内置基于 Bleve 的全文检索引擎，可对链接标题、备注、域名进行模糊匹配搜索。

**开放 API 与 Webhook**：提供 RESTful API 用于链接增删改查、目录结构读取、统计信息获取。支持配置 Webhook，在链接状态变更或新增链接时触发外部通知（如企业微信、Slack、邮件）。

## 应用场景

**技术文档站的外部参考链接库**：技术文档中往往包含大量外部参考链接，如 RFC 文档、开源项目地址、规范说明。LinkVault 可作为文档站的后台服务，文档中仅嵌入短码或 ID，由 LinkVault 统一渲染为可点击链接并提供失效检测，避免文档因外链失效而降低参考价值。

**个人知识库的网址收藏夹**：知识工作者通过浏览器收藏夹或笔记软件保存网址，但缺乏分类检索与失效提醒。LinkVault 提供结构化目录与标签体系，配合定时校验任务，可每周自动报告失效链接，帮助维护知识库的完整性。

**社区导航页的后端数据源**：开源社区、技术论坛常设置导航页面汇集常用工具与资源。LinkVault 提供 API 供前端页面拉取链接列表，导航页的增删改操作通过 LinkVault 管理后台完成，无需修改前端代码或重新部署。

**项目交接时的外部依赖清单**：项目交接文档中常包含大量外部依赖链接（第三方库主页、API 文档、运维面板地址）。LinkVault 可导出结构化清单，支持 Markdown 表格、JSON、YAML 多种格式，便于交接双方核对依赖项的可用性与负责人信息。

## 快速开始

以下步骤适用于 Linux / macOS / Windows WSL 环境，确保系统已安装 Git 与 Go 1.21 及以上版本。

```bash
# 克隆代码仓库
git clone https://github.com/yourorg/linkvault.git
cd linkvault

# 安装依赖模块（使用 Go Modules）
go mod download
go mod verify

# 编译并运行服务（默认监听 8080 端口）
go run cmd/linkvault/main.go --port 8080 --config ./configs/config.yaml
```

启动后访问 http://localhost:8080 进入管理界面。首次启动会自动生成示例配置与空数据文件，数据目录位于 ./data 下。如需自定义数据存储路径，可通过 --data-dir 参数指定。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Go 编译器 | 1.21 或更高 | 项目使用 Go 编写，依赖泛型特性与标准库增强 |
| Git | 2.25 或更高 | 用于克隆仓库及版本管理，部分自动化脚本依赖 Git 钩子 |
| Node.js | 18.x 或 20.x LTS | 前端管理界面使用 Vue 3 + Vite 构建，仅开发构建时需要 |
| Make | 3.81 或更高 | 用于执行 Makefile 中的构建、测试、打包命令 |
| curl / wget | 任意版本 | 用于健康检查脚本及链接校验功能的外部请求工具 |
| sqlite3 客户端 | 3.35 或更高（可选） | 若使用 SQLite 作为后端存储，需安装客户端工具用于调试 |

## 常见问题

**Q: LinkVault 是否支持分布式部署或多实例负载均衡？**

A: LinkVault 默认基于文件存储，不支持多实例同时读写同一数据目录。若需分布式部署，建议使用 SQLite 存储驱动并配合共享存储卷（如 NFS），或使用外部 SQL 数据库（PostgreSQL / MySQL）驱动（开发中）。多实例场景下需注意定时校验任务的重复执行问题，建议通过分布式锁（如 Redis）或仅主节点启用校验调度器。

**Q: 导入大量链接（超过 10000 条）时性能如何？**

A: 链接数据以 JSON 数组形式存储，单次读取全部数据至内存。实测 50000 条链接（含标题、备注、标签）的 JSON 文件约为 15 MB，加载时间约 200 ms，检索与过滤操作依赖内存中的 Bleve 索引。建议定期执行数据压缩与索引重建（可通过 API 触发），若链接量持续增长可切换至 SQLite 驱动以减轻内存压力。

**Q: 如何迁移 LinkVault 的数据至另一台服务器？**

A: 停止服务后，将整个 data/ 目录（或 SQLite 数据库文件）复制至新服务器对应路径，同时携带 configs/config.yaml 配置文件。若链接中包含本地快照文件（位于 snapshots/ 子目录），需一并复制。启动新服务时通过 --data-dir 参数指定数据目录位置。跨版本迁移请先查看 docs/ops/migration.md 了解版本兼容性说明。
