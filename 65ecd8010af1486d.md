# EON Resource Gateway

EON Resource Gateway 是一个面向技术研究人员、开源开发者与信息分析人员的结构化外链资源汇总系统。该项目并非传统的内容管理系统或导航站，而是一套以可维护性、可审计性与可扩展性为核心的资源索引工程方案。它解决了分散在网络各处的碎片化参考链接缺乏统一元数据描述、版本追踪与质量评估机制的问题，帮助团队或个人建立私有的、高信噪比的外部知识引用库。

项目定位为轻量级资源网关层，不依赖数据库，不存储任何副本内容，仅通过清单文件与静态标记管理资源的准入、分类与失效检测。适用于需要长期维护大量外部引用链接的文档站点、技术周报工具、合规性审计追踪系统以及研究笔记工程。

## 功能概览

**清单驱动的资源注册机制** 通过 YAML 清单文件定义每个外部链接的唯一标识、来源域名、收录批次与标签体系，所有资源记录以纯文本形式存储于项目仓库中，支持版本控制与变更追溯。

**多维度资源分类视图** 内置按域名、按主题标签、按收录批次三种分类索引方式，可快速筛选出特定来源或特定领域的链接集合，便于进行集中审查或批量更新。

**自动化可达性检测管道** 提供基于 curl 与 shell 脚本的周期性检测任务，自动标记返回非 200 状态码的资源记录，并生成失效报告，辅助维护者及时清理或替换不可用链接。

**元数据扩展字段支持** 每条资源记录预留 description、keywords、priority 三个可选扩展字段，允许维护者为链接附加业务含义，如标注参考价值等级、关联内部文档编号或所属项目阶段。

**纯静态导出能力** 资源索引可被构建为独立的静态 HTML 目录页，无需动态后端即可部署到任意 HTTP 服务器，适合作为内部团队的共享知识入口或公开的技术参考墙。

**权限分级草案** 支持通过清单文件中的 access_level 字段标记资源可见性范围（公开、内部、受限），为后续接入认证系统提供数据基础，避免敏感外链被误公开。

## 应用场景

技术文档团队的外链引用仓库。团队在编写系统设计文档或 API 说明时，需要频繁引用外部标准规范、社区讨论帖或相关项目主页。EON Resource Gateway 提供统一的引用登记与校验流程，确保文档中的每一个外部链接在发布前都经过可达性与内容相关性复核，避免文档发布后出现大量死链。

开源项目维护者的依赖参考追踪。开源项目往往依赖多个上游库、工具链文档或第三方示例代码。维护者可将这些依赖关系以资源清单形式纳入项目仓库，当上游地址发生变更或失效时，通过检测管道快速定位受影响的依赖，提前做出调整。

研究人员的文献参考管理。从事计算机科学或工程技术领域的研究人员需要整理大量 arXiv 预印本、技术报告、演讲视频链接等资料。本项目提供的标签与批次分类机制，能够按照研究课题、时间周期或会议名称对链接进行分组，并支持与笔记工具或文献管理软件进行数据交换。

合规审计部门的外部引用备案。金融、医疗或政务系统在对外发布材料时，需要记录每一个被引用的外部信息来源，以备后续审查。EON Resource Gateway 的清单化存储与版本历史，能够完整呈现某一时间段内所有引用链接的准入记录与变更日志，满足审计追溯要求。

个人知识库的碎片信息归档。独立研究者或高级工程师在长期学习过程中会积累大量临时保存的参考链接。通过本项目的批次收录功能（如第 1118/1241 批），用户可以按照时间批次管理这些链接，避免收藏夹无序膨胀，同时利用失效检测定期清理已失效资源。

## 快速开始

以下指令适用于 Linux 与 macOS 环境，Windows 用户建议通过 WSL 或 Git Bash 执行。

```bash
# 克隆项目仓库
git clone https://github.com/eon-resource/eon-gateway.git
cd eon-gateway

# 安装依赖工具（需要 curl、yq 与 jq）
./scripts/install_deps.sh

# 运行资源清单检测，生成当前批次的状态报告
./scripts/check_resources.sh --batch 1118 --output report_1118.md
```

首次运行检测管道时，脚本会自动读取 `manifests/batch_1118.yaml` 中的资源列表，依次发送 HEAD 请求并记录响应状态。执行完毕后，报告文件将输出至 `reports/` 目录。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| curl | 7.68.0 或更高 | 用于发送 HTTP 请求检测资源可达性 |
| yq | 4.25.0 或更高 | 解析 YAML 格式的资源清单文件 |
| jq | 1.6 或更高 | 处理 JSON 格式的检测结果数据 |
| bash | 4.4 或更高 | 运行所有检测与报告生成脚本 |
| git | 2.25.0 或更高 | 克隆仓库及提交清单变更 |

## 贡献指南

提交新的资源批次或改进现有功能前，请先阅读 docs/design.md 了解项目核心约定。所有贡献均通过 Pull Request 流程进行，需确保清单格式通过 schema 校验，并且新提交的链接至少经过一次本地可达性检测。

新增资源清单时，请在 manifests/ 目录下创建以 batch_ 为前缀的 YAML 文件，参照 batch_1118.yaml 的结构填写每条记录的 id、url、tags 字段。若链接包含业务上下文，建议一并填写 description 字段，便于其他维护者理解收录意图。

修改检测脚本或报告生成逻辑时，需在 tests/ 目录下补充对应的单元测试用例，并确保现有测试用例全部通过。测试执行命令为 `./tests/run_all_tests.sh`，该脚本会在 CI 环境中自动运行。

文档更新必须与代码变更同步。若新增了命令行参数或环境变量，必须同步更新 docs/user_guide.md 中的对应章节。文档格式采用标准 Markdown，代码示例需标注语言类型。

提交 Pull Request 时，请使用约定式提交标题格式，例如 `feat(checker): add retry mechanism for timeout` 或 `fix(report): correct batch number in header`。PR 描述中应包含变更动机、测试结果截图或日志摘要，以及是否影响现有清单的兼容性说明。

## 常见问题

检测脚本报告大量链接超时，应如何调整？

超时通常由网络环境或目标服务器响应速度引起。您可以在项目根目录创建 .env 文件，设置 `CURL_TIMEOUT=10` 与 `CURL_RETRY=3` 来增加单次请求等待时间与重试次数。若仍存在大量超时，建议检查网络代理配置或更换检测执行时段，避免目标服务器的高峰期。检测结果中的超时记录会单独标记为 TIMEOUT，不影响其他链接的检测执行。

如何将已有的一批链接快速转换为 YAML 清单格式？

项目提供了转换辅助脚本 `scripts/import_from_list.sh`，该脚本接受一个每行一个 URL 的纯文本文件作为输入，自动生成包含 id 与 url 字段的基础 YAML 条目，并输出到标准输出。您可以将输出重定向到 manifests/ 目录下的新文件中，然后手动补充 tags 与 description 字段。注意该脚本不会自动去重，请在导入前自行清理重复条目。

静态导出的 index.html 页面能否自定义样式或增加搜索功能？

静态导出页面位于 static/index.html，您可以直接修改该文件内的 CSS 样式或嵌入 JavaScript 实现本地搜索。项目本身不限制对导出页面的二次定制。若需要更复杂的过滤或排序功能，建议基于 reports/ 目录下的结构化 Markdown 报告自行构建前端应用，或将清单数据导入到支持静态搜索的生成器工具中。
