# Stream Resources Aggregator (SRA)

Stream Resources Aggregator (SRA) 是一个面向开发者和技术研究人员的流式资源导航与元数据索引项目。该项目定位于收集、整理并对外提供结构化的流媒体技术相关外部链接，涵盖视频流处理、实时数据传输、低延迟通信协议以及边缘计算节点状态查询等多个技术方向。SRA 本身不存储任何实际流内容，仅提供公开可用的资源定位符（URL）及其分类索引，便于技术选型参考、网络质量测试以及协议兼容性验证。

目标用户包括流媒体后端工程师、网络运维人员、协议分析研究者以及需要批量获取外部流服务端点用于自动化测试的持续集成系统。SRA 通过统一的资源清单对外暴露一批稳定的流服务探测路径，每个路径均指向经过基础连通性验证的外部端点。项目采用静态清单加外部健康检查脚本的方式运行，保证每个收录的 URL 均处于可访问状态，并定期通过自动化任务更新清单的有效性标记。

## 功能概览

**批量资源索引**：提供超过两百条流服务端点 URL，按路径标识符进行唯一编号，支持按前缀或类别批量筛选。

**端点可达性探测**：集成轻量级 HTTP 探针，可对每个收录的 URL 执行定期连通性检查，输出可用性状态报告。

**分类标签体系**：每个资源路径附带协议类型、预期内容格式、地理位置提示等元数据标签，便于快速定位特定场景下的可用端点。

**只读静态清单输出**：所有资源以纯文本或结构化 JSON 形式对外发布，无需额外后端服务即可集成到其他监控或测试框架中。

**自动化更新流水线**：通过 GitHub Actions 或等效 CI 工具每周自动执行端点重验证，标记失效链接并生成变更日志。

**命令行查询接口**：提供轻量级 CLI 工具，支持按关键字搜索、按状态过滤、按延迟排序输出可用资源列表。

**导出兼容性**：支持将资源列表导出为 Prometheus 目标配置、Nginx 上游服务器列表或通用的 CSV 格式。

## 应用场景

**流媒体服务选型测试**：开发者在选择第三方流媒体中继或边缘节点时，可使用 SRA 提供的端点列表进行并行请求测试，比较各路径的首字节时间、吞吐量及连接稳定性，从而筛选出最符合业务需求的接入点。

**网络质量监控探针部署**：运维团队可将 SRA 资源列表作为 Prometheus Blackbox Exporter 的目标配置源，定期从多个地理位置同时对列表中的 URL 发起 HTTP/HTTPS 请求，监控各端点的可用率及响应时延变化趋势。

**协议兼容性验证套件**：在开发新的流媒体客户端或网关时，工程师可利用 SRA 提供的多样化端点集合进行回归测试，验证程序对不同的传输编码、重定向策略、TLS 版本以及 HTTP 头部处理逻辑的兼容程度。

**自动化数据采集管道初始化**：数据工程师在搭建流式数据消费管道时，可将 SRA 清单作为初始引导配置，批量注册外部数据源地址，配合自定义解析器完成字段映射和格式转换，加速管道原型搭建。

## 快速开始

以下命令演示如何获取 SRA 项目源码、安装基础依赖并执行首次资源列表导出。

```bash
# 克隆项目仓库
git clone https://github.com/stream-resources/sra.git
cd sra

# 安装依赖（需要 Python 3.9+ 及 pip）
pip install -r requirements.txt

# 执行资源列表导出（默认输出 JSON 格式至 ./output/endpoints.json）
python src/export.py --format json --output ./output/endpoints.json
```

导出后的 JSON 文件包含所有收录 URL 的完整列表及其元数据字段，可直接被其他脚本或监控工具加载使用。

## 项目结构

```
sra/
├── src/                                # 核心源码目录
│   ├── export.py                       # 主导出脚本，支持 json/csv/prometheus 格式
│   ├── probe.py                        # 端点健康检查模块，基于 asyncio 并发请求
│   ├── metadata.py                     # 加载并解析 data/metadata.yaml 中的标签信息
│   └── cli/                            # 命令行子命令包
│       ├── __init__.py
│       ├── list_cmd.py                 # 实现 list 子命令，支持过滤和排序
│       └── validate_cmd.py             # 实现 validate 子命令，执行全量连通性验证
├── data/                               # 静态数据目录
│   ├── endpoints.json                  # 导出的完整端点列表（自动生成，勿手动编辑）
│   ├── metadata.yaml                   # 人工维护的端点元数据标签（协议/区域/备注）
│   └── schema/                         # JSON Schema 校验文件
│       └── endpoint_schema.json        # 定义 endpoints.json 的字段结构与类型约束
├── tests/                              # 单元测试与集成测试目录
│   ├── test_export.py                  # 测试不同格式导出的正确性
│   ├── test_probe.py                   # 模拟 HTTP 请求测试探测逻辑
│   └── fixtures/                       # 测试用固定样本数据
│       └── sample_endpoints.json
├── scripts/                            # 辅助运维脚本
│   ├── update.sh                       # 拉取最新代码并重新生成 endpoints.json
│   ├── validate_all.sh                 # 调用 probe.py 对所有端点执行一次检查
│   └── generate_report.py              # 从验证日志生成 Markdown 状态报告
├── .github/                            # GitHub 专用配置
│   └── workflows/
│       ├── weekly_validate.yml         # 每周定时执行全量验证
│       └── pr_validate.yml             # PR 提交时自动检查 endpoints.json 格式
├── requirements.txt                    # Python 运行时依赖列表
├── Makefile                            # 常用任务封装（install, update, test）
├── LICENSE                             # MIT 许可证全文
└── README.md                           # 项目说明文件（当前文档）
```

## 贡献指南

SRA 项目欢迎社区贡献，无论是新增资源链接、更新元数据标签，还是改进探测脚本性能，均可通过标准流程参与。请遵循以下步骤提交变更。

1. 复刻项目仓库并创建本地功能分支：从主仓库复刻后，在本地执行 git checkout -b feature/your-change-id，确保分支名称简洁描述变更主题。

2. 更新资源元数据文件：如需新增或修改外部链接，请编辑 data/metadata.yaml 文件，按照已有条目的格式添加路径标识符、协议类型、预期状态码及可选的备注字段。暂不支持直接修改自动生成的 endpoints.json。

3. 执行本地验证流程：在项目根目录运行 make test 以执行全部单元测试，运行 make validate 以对待提交的元数据变更进行连通性预检。确保所有测试通过且无新增警告。

4. 提交变更并推送至远程复刻仓库：使用 git add 添加修改文件，提交时请附上清晰的提交信息，说明变更原因及影响的资源条目数量。随后推送至个人复刻仓库。

5. 创建拉取请求：在 GitHub 上向主仓库的 main 分支提交 PR，填写 PR 模板中的检查项，包括是否通过测试、是否更新文档、是否影响现有输出格式等。等待维护者审核与合并。

## 常见问题

**问：为什么部分 URL 返回 404 或 5xx 状态码，是否意味着资源已失效？**

答：SRA 收录的 URL 均指向外部独立服务，其可用性不受本项目建设控制。返回非 200 状态码可能由目标服务临时过载、路径变更或访问频率限制导致。建议在首次使用前通过 probe.py 脚本对目标端点执行多轮重试探测，并参考 metadata.yaml 中记录的预期响应类型。如果连续多次验证均失败，可在 GitHub Issues 中提交反馈，维护团队将定期核实并清理长期不可用的条目。

**问：如何获取资源列表的 JSON 格式以便集成到自己的监控系统中？**

答：在项目根目录执行 python src/export.py --format json --output ./my_endpoints.json 即可生成标准 JSON 文件。此外，Makefile 中提供了快捷命令 make export-json 直接执行导出。导出文件包含每个 URL 的完整路径、最近验证时间戳、历史可用率及标签数组，可直接被 Prometheus 的 file_sd_config 或 Blackbox Exporter 的 module 配置引用。

**问：资源列表的更新频率是多少，我是否需要手动同步最新清单？**

答：SRA 项目通过 GitHub Actions 每周自动执行一次全量验证并更新 endpoints.json 中的状态字段，但 URL 列表本身（即路径标识符集合）仅在 metadata.yaml 有社区提交变更时才会调整。建议用户通过 git pull 定期同步主仓库最新内容，或利用脚本 scripts/update.sh 自动拉取更新并重新生成本地输出文件。对于生产环境，可考虑将每周同步任务写入 cron 或 CI 流水线。
