# SiteBloom Resource Gateway

SiteBloom Resource Gateway 是一个面向技术研究人员、开发者与内容策展人的轻量级外链资源聚合与管理平台。该项目定位于解决分散链接难以归类、检索与共享的问题，通过结构化的目录体系与可插拔的解析引擎，将大量原始 URL 转化为可维护、可扩展的知识索引。

本项目目标用户包括开源社区维护者、技术文档编写团队、在线教育内容运营方以及任何需要系统化管理外部参考链接的个体或组织。项目本身不存储第三方内容，仅提供元数据组织与快速跳转能力，所有原始链接均以只读方式呈现，确保引用路径的完整性与可追溯性。

## 功能概览

**批量链接注入与解析** 支持一次性导入大批量 URL，自动识别协议头、域名层级与路径参数，生成标准化的内部记录对象。

**动态分类标签系统** 允许用户为每一条链接分配自定义标签与分类层级，支持多对多关联，便于后续按主题、领域或用途筛选。

**只读镜像访问模式** 所有外链均通过网关层进行 302 重定向，不缓存页面内容，同时记录点击频次与最后访问时间，供运维审计使用。

**结构化的目录树输出** 内置目录生成器可将链接集合映射为嵌套目录结构，便于集成到静态站点生成器或文档框架中。

**资源状态健康检查** 周期性对已收录链接进行可用性探测，标记失效或重定向链接，并生成异常报告。

**全文检索与过滤** 基于路径关键词与自定义标签提供轻量级搜索能力，支持精确匹配与前缀模糊查询。

**导入导出兼容性** 支持 CSV 与 JSON 格式的批量导入导出，方便与其他工具链（如书签管理器、知识库软件）进行数据交换。

**权限分级与只读发布** 内置简单的角色区分，支持将整个资源库发布为公开只读视图，适用于团队对外共享链接集。

## 应用场景

**技术文档外部参考索引** 团队在编写系统设计文档或 API 说明时，需引用大量外部规范、博客与标准。SiteBloom Resource Gateway 可作为统一的参考链接仓库，避免在文档正文中堆砌长 URL，同时确保链接变更时可集中更新。

**在线课程资料包管理** 教育机构或培训讲师在开设技术课程时，常需提供大量课外阅读链接。本项目的分类与检索能力可帮助学员按周次或主题快速定位相关资源，减少信息过载。

**开源项目 README 资源整合** 开源项目维护者往往需要在 README 中列出相关生态工具、镜像源或社区论坛。使用本网关可生成稳定的内部短链，同时避免频繁修改 README 文件本身。

**个人知识库外链去重** 研究员或博客作者在日常阅读中积累大量浏览器书签，通过本项目可进行去重、标注与定期健康检查，将零散书签转化为结构化知识资产。

## 快速开始

以下操作以 Linux/macOS 环境为例，假定已安装 Git、Node.js（v18 以上）与 npm。

```bash
# 克隆代码仓库
git clone https://github.com/sitebloom/resource-gateway.git
cd resource-gateway

# 安装项目依赖
npm install

# 以开发模式运行服务（默认监听 3000 端口）
npm run dev
```

服务启动后，访问 http://localhost:3000 可看到网关首页。首次运行时，系统会自动生成示例数据并导入本批次资源清单（第 349/1241 批，共 250 条链接）。若需清空示例数据并重新导入，可执行 `npm run reset`。

## 项目结构

```
resource-gateway/
├── src/                               # 核心源代码目录
│   ├── core/                          # 核心引擎模块
│   │   ├── linker.js                  # 链接解析器，负责提取协议、域名与路径
│   │   └── registry.js                # 注册表管理，维护内存索引与数据库同步
│   ├── routes/                        # HTTP 路由层
│   │   ├── api.js                     # REST API 端点（增删改查、健康检查）
│   │   └── web.js                     # Web 界面路由（首页、分类视图）
│   ├── middleware/                    # 中间件链
│   │   ├── cors.js                    # 跨域资源共享策略
│   │   ├── logger.js                  # 访问日志与错误捕获
│   │   └── ratelimit.js               # 基于 IP 的请求限流
│   ├── models/                        # 数据模型定义
│   │   ├── link.js                    # 链接实体（URL、标签、创建时间、最后检查）
│   │   └── tag.js                     # 标签实体（名称、颜色、关联计数）
│   └── utils/                         # 工具函数集
│       ├── validator.js               # URL 格式校验与规范化
│       └── health.js                  # 链接可达性探测（HEAD 请求）
├── config/                            # 配置文件目录
│   ├── default.json                   # 默认配置（端口、数据库路径、检查间隔）
│   └── production.json                # 生产环境覆盖配置
├── data/                              # 数据存储目录
│   └── gateway.sqlite                 # SQLite 数据库文件（首次启动自动生成）
├── scripts/                           # 辅助脚本
│   ├── import-batch.js                # 批量导入本批次链接清单
│   ├── export-csv.js                  # 导出全部数据为 CSV 格式
│   └── health-report.js               # 生成链接健康状态报告
├── docs/                              # 文档目录（内容见上文导航）
├── test/                              # 单元测试与集成测试
│   ├── unit/                          # 单元测试用例
│   └── integration/                   # 接口测试脚本
├── package.json                       # 项目依赖与脚本声明
├── README.md                          # 本文件
└── LICENSE                            # MIT 许可证文本
```

## 贡献指南

欢迎社区贡献者参与项目改进。请遵循以下步骤以确保协作顺畅：

1. 查阅 issue 列表，选择未被认领的任务，或提交新的功能请求与缺陷报告。对于较大的架构调整，建议先通过 issue 与维护者沟通设计思路。

2. 派生本仓库至个人账户，在派生副本中创建功能分支。分支命名建议采用 `feature/功能简述` 或 `fix/问题简述` 格式，避免直接在 main 分支上修改。

3. 编写代码时遵守项目既定的 ESLint 规则与 Prettier 格式标准。所有新增功能需附带对应的单元测试，测试覆盖率不应低于现有基线。

4. 提交代码前执行 `npm run test` 与 `npm run lint`，确保全部测试通过且无格式警告。提交信息请采用简洁的英文描述，首行不超过 72 字符。

5. 从个人派生仓库向本仓库的 main 分支发起 Pull Request，在描述中清晰说明改动内容、影响范围以及是否涉及数据库迁移。PR 需要至少一位维护者审阅通过后方可合并。

## 常见问题

**Q: 项目中收录的链接如果失效了怎么办？**

A: 系统内置的健康检查脚本（`scripts/health-report.js`）会以可配置的周期（默认每 24 小时）对所有链接发送 HEAD 请求。若某链接连续三次检查失败，其状态将被标记为「异常」，并在 Web 界面与导出报告中体现。运维人员可定期查看报告，手动验证并决定是否移除或更新链接。

**Q: 我能将本网关部署到公网供他人使用吗？安全方面有何建议？**

A: 可以。项目本身不存储第三方内容，仅做重定向，因此内容安全风险较低。但建议在生产环境中启用 HTTPS（通过 Nginx 或 Caddy 做反向代理），并配置 `config/production.json` 中的 `rateLimit` 参数以防止恶意爬取。此外，可通过环境变量 `ADMIN_IPS` 限制管理端口的访问来源。

**Q: 如何迁移现有书签或链接集到本项目中？**

A: 项目支持 CSV 与 JSON 两种导入格式。CSV 需包含 `url` 与 `tags` 两列；JSON 格式参考 `data/example-import.json`。您可使用 `scripts/import-batch.js` 脚本进行导入，例如 `npm run import -- --file ./my-bookmarks.csv`。系统会自动去重，并在导入完成后输出统计报告。
