# VODLink 聚合索引系统

VODLink 是一个面向多媒体技术研究人员、网络资源管理者和内容分发网络运维人员的统一视频资源定位符索引与解析系统。本项目提供标准化的外链汇集与元数据映射能力，允许用户通过稳定的标识符体系访问分布式的视频内容节点，解决因资源散落、链接失效或路径变更导致的内容寻址困难问题。

本系统定位于技术资源聚合与链接生命周期管理，不直接托管或存储任何视频文件，仅维护资源定位符的索引与校验逻辑。目标用户包括视频平台开发者、数据迁移工程师、自动化测试脚本编写者以及需要批量处理媒体资源引用的技术团队。

## 功能概览

统一资源标识映射：为每个视频资源分配唯一内部编码，并建立与原始定位符之间的稳定映射关系，支持单向与双向查询。

定位符健康度检测：自动对已收录的定位符执行可用性探测，识别响应状态码异常、超时或重定向循环，输出健康报告。

批量导入与导出：支持通过 CSV 或 JSON 格式批量导入外部链接列表，并可按照指定条件筛选后导出为结构化数据文件。

分类标签管理：允许用户为每个资源添加自定义标签与备注字段，便于按项目、地区、清晰度或内容类型进行二次归类。

访问频次统计：记录每个定位符的查询次数与最后访问时间，辅助运维人员识别热点资源与低频僵尸链接。

定位符规范化校验：自动检测并修复常见的 URL 格式错误，包括协议缺失、大小写不一致、尾部斜杠冗余等问题。

增量更新机制：支持基于时间戳或版本号的增量同步，确保多端部署环境下的资源列表一致性。

## 应用场景

大规模视频迁移工程中的链接映射：当视频源站计划更换域名或存储路径时，运维团队可利用本系统维护新旧定位符之间的对应关系，通过修改映射表实现零感知切换，避免硬编码链接散落在代码库各处。

自动化测试环境中的资源依赖管理：测试脚本往往需要引用固定的视频样本片段，使用本系统的标识符进行间接引用后，测试人员只需在索引表中替换实际定位符即可切换不同测试数据集，无需修改脚本逻辑。

多区域内容分发网络节点对比分析：通过同时收录同一内容在不同 CDN 节点上的定位符，对比响应时间、可用率等指标，为调度策略优化提供数据支撑。

资源失效主动告警与自动清理：配置定时健康检查任务后，系统可将连续多次不可达的定位符标记为异常状态，并通过 Webhook 通知管理人员介入处理，维持资源池的整洁性。

## 快速开始

以下步骤指导用户在 Linux 或 macOS 环境下完成本项目的克隆、依赖安装与服务启动。

```bash
# 克隆代码仓库至本地
git clone https://github.com/example/vodlink-index.git
cd vodlink-index

# 安装 Python 依赖包（推荐使用虚拟环境）
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 初始化数据库表结构并导入示例索引数据
python manage.py migrate
python manage.py loaddata initial_index.json

# 启动开发服务器，默认监听 8000 端口
python manage.py runserver 0.0.0.0:8000
```

访问 http://localhost:8000/admin 可使用默认管理员账户（admin / admin123）进入后台管理界面，开始录入或导入资源定位符。

## 安装要求

本系统依赖以下运行环境与第三方库，请务必在部署前确认版本兼容性。

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.9 及以上 | 核心运行环境，低于此版本将导致语法错误 |
| Django | 4.2.x | Web 框架，用于提供管理界面与 REST API |
| PostgreSQL | 14.x 及以上 | 生产环境推荐数据库，支持并发读写与事务 |
| Redis | 6.x 及以上 | 缓存与分布式锁，用于健康检查任务队列 |
| Celery | 5.3.x | 异步任务编排，处理定时探测与统计更新 |
| gunicorn | 21.x | WSGI 服务器，生产环境部署必备 |
| httpx | 0.25.x | 异步 HTTP 客户端，用于定位符可用性探测 |
| python-dotenv | 1.0.x | 环境变量加载，用于敏感配置分离 |

## 项目结构

```
vodlink-index/
├── manage.py                # Django 项目管理入口，用于启动、迁移、shell 等操作
├── requirements.txt         # Python 依赖清单，锁定所有第三方库版本号
├── .env.example             # 环境变量配置模板，包含数据库连接串与密钥占位
├── src/                     # 项目核心源码目录
│   ├── settings/            # 分环境配置文件（开发、测试、生产）
│   │   ├── base.py          # 基础公共配置，所有环境共享
│   │   ├── development.py   # 开发环境配置，启用调试与本地缓存
│   │   └── production.py    # 生产环境配置，关闭调试，使用 PostgreSQL
│   ├── urls.py              # 全局路由分发器，定义 API 端点与后台路径
│   ├── wsgi.py              # WSGI 网关接口，用于 gunicorn 部署
│   └── celery.py            # Celery 应用实例，声明定时任务与队列
├── apps/                    # 功能模块集合（Django 子应用）
│   ├── index/               # 核心索引引擎模块
│   │   ├── models.py        # 资源映射表、健康记录、标签类定义
│   │   ├── views.py         # 查询、导入、统计等视图函数
│   │   ├── serializers.py   # REST API 序列化器，控制输入输出格式
│   │   └── validators.py    # URL 规范校验、黑名单过滤逻辑
```

## 贡献指南

我们欢迎社区开发者提交问题报告、功能建议与代码改进。请遵循以下流程以确保贡献过程的顺畅与代码质量。

首先，在 GitHub 仓库的 Issue 区搜索是否已有相同或类似的问题讨论，避免重复提交。若未找到，请新建一个 Issue 并选择对应的模板（缺陷报告或功能请求），清晰描述当前行为与期望行为，并附上必要的日志或截图。

其次，Fork 本仓库至个人账户，并在本地新建一个描述性的分支名称，例如 fix-health-check-timeout 或 add-export-format。请确保所有代码变更均通过现有的单元测试，并为新增功能编写对应的测试用例。

第三，遵循项目代码风格规范，Python 代码使用 Black 格式化工具，导入语句按标准库、第三方库、本地模块顺序分组。提交信息应使用简洁的英文祈使句，首行不超过 72 字符，详细说明可追加在第二段。

第四，完成开发后，推送分支到远程仓库并发起 Pull Request 至主分支。PR 描述中请关联对应的 Issue 编号，并勾选检查清单中的自测项。项目维护者将在三个工作日内进行 Code Review，提出修改意见或合并。

## 常见问题

问：系统如何保证所收录定位符的长期有效性？如果原始链接发生永久迁移，应该如何处理？

答：本系统内置的 Celery 定时任务会按照可配置的周期（默认为每 6 小时）对所有活跃定位符执行 HEAD 请求探测，记录状态码与响应时间。当检测到 301 或 308 永久重定向时，系统会自动记录新的目标 URL 并建议管理人员更新映射。若连续三次探测均返回 4xx 或 5xx 状态，该条目将被标记为「异常」并移出默认查询视图。用户可在管理后台手动更新定位符字段，或通过导入接口批量修正。

问：本项目与普通的书签管理工具或网络收藏夹有何本质区别？

答：书签工具通常面向个人用户，侧重于本地存储和浏览器集成，缺乏多端同步、健康监控和批量操作能力。VODLink 定位为团队级基础设施组件，提供统一的数据模型、RESTful API 以及任务编排框架，支持自动化运维流程。例如，运维团队可编写脚本调用 API 每日同步最新定位符列表至 CDN 配置中心，或与监控告警系统联动，实现资源链路的全生命周期治理。

问：部署后如何扩展支持自定义的定位符解析规则，例如处理私有协议或特殊签名参数？

答：系统在 apps/index/validators.py 中提供了可插拔的校验器基类。开发者只需继承 BaseValidator 并实现 validate 与 normalize 方法，然后在 settings 配置中将新类的导入路径添加到 CUSTOM_VALIDATORS 列表中即可。系统在录入和查询阶段会自动调用所有已注册的校验器，对定位符进行预处理和合规性检查，不影响核心索引逻辑。
