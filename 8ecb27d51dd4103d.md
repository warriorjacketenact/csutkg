# LinkVault Core

LinkVault Core 是一个面向技术团队与独立开发者的外链资源归集与结构化导航系统。该项目并非简单的书签管理器，而是一个将分散、异构的外部知识链接转化为可检索、可分类、可版本化的内部知识资产的数据处理管道。它主要解决技术文档碎片化、外部参考链接失效、以及团队协作中知识传递对个人浏览器收藏夹过度依赖的问题。目标用户包括架构师、技术负责人、运维工程师以及需要维护大量外部依赖文档的开发人员。

## 功能概览

**批量链接导入与标准化清洗**：支持从文本、CSV及部分浏览器导出文件中批量摄入原始URL，自动去除跟踪参数、识别协议缺失并执行格式规整，确保入库数据符合稳定引用规范。

**基于主题标签的柔性分类引擎**：用户可为每条链接赋予多个自定义标签，系统同时根据URL路径特征与关键词库自动生成辅助分类建议，实现层级目录与多维标签的混合管理。

**链接存活与元数据探针**：后台异步任务定期检测所有外链的可访问性，并自动抓取页面标题、描述及更新时间，在不打开页面的前提下提供丰富预览信息。

**全文检索与高级过滤视图**：基于URL、标题、标签、备注及关联主题编号构建倒排索引，支持按状态（有效/失效）、批次、时间范围等多条件组合筛选。

**可导出结构化知识图谱**：所有链接数据支持导出为JSON、Markdown表格或结构化YAML清单，便于集成至静态站点生成器、Wiki或内部开发者门户。

**变更审计与快照回溯**：记录每条链接的每次属性修改，支持按时间点查看资源集合状态，便于追溯问题根因或回退至稳定版本。

## 应用场景

技术团队在启动新项目时，需要快速汇集相关框架文档、API参考、社区讨论及示例仓库。LinkVault Core允许团队负责人创建项目专属批次，批量导入候选链接并分配主题编号，新成员通过一次同步即可获取完整知识图谱，无需逐个询问。

运维工程师定期巡检外部依赖服务状态。LinkVault Core的存活探针可配置为定时任务，当依赖的文档站点或镜像仓库出现访问异常时，自动标记并通知相关责任人，替代手工点击验证的低效流程。

技术博主或开源作者维护自己的推荐资源列表时，经常遇到链接分散在各处的问题。利用LinkVault Core的分类与导出功能，可一键生成格式规范的资源清单Markdown文档，直接嵌入博客文章或项目README中，保持内容同步更新。

## 快速开始

以下步骤将指导您在本地环境完成LinkVault Core的部署与初始运行。

```bash
# 克隆项目仓库至本地
git clone https://github.com/linkvault/linkvault-core.git

# 进入项目根目录并安装依赖
cd linkvault-core
pip install -r requirements.txt

# 执行数据库初始化并启动内置开发服务器
python manage.py migrate
python manage.py runserver --host 0.0.0.0 --port 8080
```

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---------|---------|------|
| Python | 3.10 或更高 | 核心运行环境，低于此版本将导致类型注解解析异常 |
| SQLite | 3.35.0 或更高 | 默认内置数据库，支持JSON扩展及全文搜索功能 |
| Redis | 6.2 或更高 | 用于缓存探针结果及异步任务队列（可选，生产环境推荐） |
| Node.js | 18.0 或更高 | 仅用于前端静态资源构建，后端运行可不安装 |
| Git | 2.25 或更高 | 用于版本跟踪及后续热更新补丁应用 |
| 系统内存 | 1GB 以上 | 建议值，实际占用取决于链接总数及并发任务数 |
| 磁盘空间 | 500MB 以上 | 用于存储数据库、日志及缓存页面元数据 |

## 贡献指南

首先在GitHub上fork本仓库至个人账户，然后克隆fork后的仓库到本地开发环境。建议在开发前阅读docs/dev/architecture.md了解整体设计，避免引入与现有架构冲突的实现。

创建新的功能分支时，请使用`feature/`或`fix/`前缀，例如`feature/add-export-csv`或`fix/probe-timeout`，确保分支名简洁描述变更内容。所有代码提交前必须执行`make lint`和`make test`，确保通过静态检查且单元测试覆盖率达到90%以上。

对于新增的探针类型或导出格式，需要在docs/api/或docs/usage/下补充对应的文档说明，并在test/integration/中添加至少一个端到端验证用例。提交Pull Request时请详细描述变更动机、实现方案及测试结果，维护者将在三个工作日内进行审查。

## 常见问题

Q: 导入包含数百个URL的CSV文件时，页面长时间无响应或超时，应如何处理？

A: 这是浏览器单次请求超时的正常限制。对于大批量导入，推荐使用命令行工具`./cmd/batch-import`，该工具支持分块提交与进度条显示。若仍希望使用Web界面，可将文件拆分为每份50条以内分批提交，或调整`application.yaml`中的`http.read_timeout`参数至300秒以上。

Q: 探针任务报告大量链接为失效，但手动浏览器访问正常，原因是什么？

A: 此类情况通常由目标站点的反爬机制、请求头校验或JavaScript动态加载导致。请检查`probe-rules.yaml`中的`user_agent`配置是否模拟了主流浏览器，并尝试将`probe.http.follow_redirect`设置为`true`。若目标站点为SPA应用，请考虑将其标记为“需人工验证”类型，绕过自动探针。

Q: 如何将LinkVault Core的资源数据迁移至另一台服务器？

A: 仅需迁移SQLite数据库文件（默认位于`data/linkvault.db`）以及`configs/`目录下自定义的配置文件。停止服务后复制这两个目录至新服务器，重新运行`./cmd/server`并指定相同的`--config`路径即可。若启用了Redis缓存，建议在新环境重新构建缓存，避免键冲突。
