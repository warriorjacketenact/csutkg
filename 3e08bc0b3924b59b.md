# LinkVault Resource Aggregator

LinkVault is a curated, machine-readable resource index designed for developers, researchers, and content curators who need programmatic access to a structured directory of external technical references, educational materials, and multimedia assets. Unlike traditional bookmarking tools or manual web directories, LinkVault treats resource links as first-class data entities, enabling automated fetching, validation, metadata extraction, and dependency resolution for downstream pipelines.

This project addresses the fundamental challenge of managing large, heterogeneous link collections across team environments and CI/CD workflows. Instead of scattering URLs across documents, spreadsheets, or ad-hoc text files, LinkVault provides a unified manifest format, validation tooling, and export adapters for common data science and web development ecosystems. The current release includes the complete batch 590/1241, totaling 250 verified resource endpoints from the 6chuang theater content network, with ongoing support for incremental batch updates, dead-link detection, and content-type classification.

## 功能概览

- **Manifest-Based Resource Indexing** – All resources are declared in a single YAML manifest with support for tags, content type hints, and optional metadata fields such as source domain and expected availability status.

- **Dead-Link Validation Pipeline** – A built-in CLI tool periodically checks each URL for HTTP 200/403/404 responses, generating a health report with retry policies and exponential backoff for rate-limited domains.

- **Export Adapters for Multiple Formats** – Convert the resource manifest to JSON Lines for data ingestion, Markdown table for human-readable documentation, CSV for spreadsheet analysis, or RSS feed for subscription-based update monitoring.

- **Content-Type Guessing and Categorization** – Automatically infer MIME types from URL patterns and response headers to classify resources into video, document, interactive, or archive categories without manual annotation.

- **Versioned Snapshot Support** – Each batch update creates a timestamped snapshot of the resource list, enabling rollback, diff comparison, and change-tracking across releases.

- **Dependency-Aware Filtering** – Resources can declare dependencies on external libraries or runtime environments, allowing users to filter the manifest for subsets relevant to their specific stack.

- **Webhook Notification System** – Configure Slack or generic HTTP webhooks to receive alerts when resources change status (new dead links, redirects, or content-type shifts).

- **RESTful Query API** – A lightweight FastAPI wrapper exposes the resource manifest via search endpoints with regex filtering, domain whitelisting, and random sampling for testing purposes.

## 应用场景

**Automated Documentation Generation for Technical Wikis** – When maintaining a team internal knowledge base, LinkVault can be used as the source-of-truth for external reference links. The export adapter generates a formatted Markdown section that is merged into the wiki build pipeline, ensuring that all cited resources are verified before documentation deployment.

**Data Pipeline Input Curation for Web Scraping** – Data engineering teams can use LinkVault to manage seed URLs for crawlers. The manifest provides a controlled list of entry points, and the validation pipeline prevents the scraper from attempting to fetch already-dead endpoints, reducing task failures and log noise.

**Educational Curriculum Resource Distribution** – Instructors distributing reading lists or video lecture series can publish the LinkVault manifest alongside course materials. Students then use the CLI tool to fetch the latest resource list, verify accessibility, and optionally download metadata for offline reference, all without manual copy-paste errors.

**Multi-Environment Configuration Management** – DevOps practitioners can treat LinkVault manifests as environment-specific configuration artifacts. Staging and production pipelines consume different snapshots or filtered subsets, allowing controlled rollout of new resources before they are exposed to end-user-facing applications.

**Research Paper Supplementary Data Indexing** – Academic projects that reference online datasets, code repositories, or interactive demos can package their references as a LinkVault manifest. Reviewers and readers can then run the validation tool to verify that all supplementary URLs remain accessible at the time of reading, enhancing reproducibility.

## 快速开始

```bash
# Clone the repository
git clone https://github.com/your-org/linkvault.git
cd linkvault

# Install dependencies using pipenv or standard pip
pip install -r requirements.txt

# Verify the installation by running the validation CLI on the bundled manifest
python -m linkvault validate --manifest manifests/batch_590.yaml

# Generate a Markdown report of all resources
python -m linkvault export --format markdown --output docs/resources.md --manifest manifests/batch_590.yaml
```

For first-time users, the `quickstart.sh` script automates the above steps and also runs a sample health check against a subset of the resource list to demonstrate the live validation feature.

## 贡献指南

1.  **Issue 驱动开发** – 在提交 Pull Request 之前，请先在 GitHub Issues 中创建一个条目，描述你打算修复的缺陷或新增的功能。核心维护者将在 48 小时内给出反馈并分配标签。

2.  **遵循代码风格与类型检查** – 所有 Python 代码必须通过 black 格式化，并通过 mypy 的严格模式检查（`--strict` 参数）。建议在提交前运行 `pre-commit run --all-files` 自动执行这些检查。

3.  **新增资源批次流程** – 若需要添加新的资源批次，请将 URL 列表放入 `manifests/` 目录下的新 YAML 文件中，并运行 `python -m linkvault validate --manifest your_new_batch.yaml` 确保格式正确且无重复条目。通过后更新 `docs/batch_processing.md` 记录该批次信息。

4.  **测试覆盖要求** – 所有新增功能或对现有逻辑的修改，必须附带相应的单元测试。测试用例需覆盖正常路径和至少两种异常路径（例如网络超时、无效 URL 格式）。运行 `pytest --cov=src/linkvault` 检查覆盖率，目标不低于 85%。

5.  **文档同步更新** – 如果修改了 CLI 命令参数、API 响应结构或配置项，必须同步更新 `docs/` 目录下对应的用户文档。文档变更与代码变更应在同一个 Pull Request 中提交。

## 常见问题

**Q: 验证工具报告大量 403 状态码，是否意味着资源已经失效？**

A: 并非所有 403 都代表资源永久不可用。许多资源服务器会实施访问频率限制或基于 User-Agent 的过滤策略。LinkVault 的验证工具默认使用 `LinkVault/1.0` 的 User-Agent，并通过 `--retry` 参数自动进行三次重试。若仍然返回 403，建议手动通过浏览器访问该 URL 以确认是否需要特定的 Cookie 或 Referer 头。你也可以在 manifest 中为特定条目添加 `custom_headers` 字段来覆盖默认请求头。

**Q: 如何将 LinkVault 集成到我的 CI/CD 流水线中？**

A: 推荐使用 Docker 镜像方式运行。项目提供了官方的 `Dockerfile`，你可以构建镜像并在 GitLab CI、GitHub Actions 或 Jenkins 中调用容器。在流水线中，你可以先执行验证步骤（`validate` 命令）来阻断构建，如果发现死链接则使流水线失败；然后执行导出步骤（`export` 命令）将最新的资源列表生成到构建产物中，供后续部署阶段使用。

**Q: 批次编号的规则是什么？我如何确定下一个批次的编号？**

A: 批次编号采用全局递增序号，从 1 开始每新增一个批次加 1。当前已发布到第 1241 批。新增批次时，请查看 `manifests/` 目录下最新的 YAML 文件名，提取最大编号，然后加一即可。每个批次建议包含 200-300 个资源条目，以确保验证任务的运行时间在可控范围内。如果需要频繁更新少量资源，建议使用快照机制（`snapshot` 子命令）而不是每次都创建新批次。
