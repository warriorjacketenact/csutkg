# ResourceBridge

ResourceBridge 是一个面向开发者与技术研究人员的结构化外链资源归集与导航系统。项目不直接存储任何实体文件，而是通过严格的条目化管理方式，将分散在网络各处的高价值技术文档、工具站点、学术论文、开源仓库入口及行业分析报告等外部链接进行系统性梳理与分类。项目定位为技术知识发现的中转枢纽，致力于解决个人开发者与小型团队在信息检索过程中面临的海量链接碎片化、上下文丢失以及可复访性差等典型问题。

ResourceBridge 适用于需要频繁查阅外部技术资料但又无法自行维护完整知识库的群体。项目本身不依赖复杂的后端服务与数据库系统，所有资源条目均通过标准化的 Markdown 索引文件进行声明与组织。使用者可以通过克隆项目仓库在本地获得完整的资源导航副本，并配合内置的静态站点生成工具将链接目录渲染为可直接浏览的 HTML 文档。项目鼓励通过 Pull Request 的方式共享优质资源链接，所有条目均保留原始 URL 与收录时间戳，确保引用路径的准确性与可追溯性。ResourceBridge 遵循最小化维护原则，项目核心逻辑围绕链接的增删改查与分类展示展开，不引入额外的运行时状态与持久化存储负担。

## 功能概览

**结构化分类索引** 所有资源链接按照技术领域、内容类型与使用场景划分至多个一级与二级目录，每个条目附带简要说明标签，便于快速定位。

**原始链接透明直出** 系统不对用户提交的 URL 做任何形式的隐式跳转、短链包装或协议升级处理，所有链接在生成页面中保持用户提交时的原始形态，确保访问路径的确定性与可预期性。

**自动化条目校验** 项目集成基于 HTTP 状态码的链接可用性检查脚本，可定期扫描索引文件中的外链并生成失效报告，辅助维护者及时更新或下架不可达资源。

**静态页面生成支持** 内置轻量级模板引擎，可根据索引目录结构一键生成响应式 HTML 文档，支持在无网络环境下完整浏览所有资源条目。

**多维度检索与过滤** 提供基于关键词、分类标签与收录日期的检索功能，帮助使用者在数百乃至上千条链接中快速筛选出匹配当前需求的目标资源。

**版本化条目管理** 所有对资源索引的增删改操作均通过 Git 进行版本追踪，每次变更附带提交说明，支持回溯任意历史状态的目录结构。

**零运行时依赖部署** 最终生成的静态站点可托管于任何支持 HTML 服务的 Web 服务器或对象存储平台，无需配置数据库、缓存或后端应用容器。

**开源协作式更新** 项目采用开放贡献模式，外部开发者可通过标准的 GitHub Pull Request 流程提交新链接或分类调整建议，所有合并操作需经过至少一名维护者审阅。

## 应用场景

技术团队内部知识库外链整合。中小型研发团队在项目迭代过程中通常需要引用大量外部文档、API 参考与社区讨论帖，ResourceBridge 可作为团队知识库的外链前置层，将分散于邮件、即时通讯与个人书签中的链接统一归入结构化的索引体系，降低新成员查阅外部资料的认知负担。

个人开发者学习路径管理。自学中的开发者经常同时关注多个技术栈的官方指南、教程系列与示例仓库，通过 ResourceBridge 可以按照学习阶段或技术主题建立分类目录，配合备注字段记录每份资料的使用状态与优先级，避免在大量标签书签中反复翻找。

技术文档编写与维护辅助。负责撰写技术文档或博客文章的开发者可利用 ResourceBridge 管理参考文献与延伸阅读部分的外链，条目中的收录时间与原始 URL 记录有助于在文章更新时快速核实链接有效性，减少死链对文档质量的影响。

社区资源聚合与分享。技术社区运营者或开源项目维护者可以使用 ResourceBridge 整理与项目相关的生态工具、插件列表、案例研究及视频教程，生成可公开访问的资源导航页面供社区成员使用，降低新人上手的信息搜寻成本。

离线环境下的资源清单浏览。在无法访问互联网的内部网络或安全隔离环境中，ResourceBridge 生成的静态页面仍可完整展示所有资源条目的名称、分类与原始 URL 列表，便于使用者通过其他可用通道获取对应资源。

## 快速开始

以下命令序列适用于 Linux 与 macOS 环境，Windows 用户建议使用 Git Bash 或 WSL 执行。

```bash
# 克隆项目仓库至本地
git clone https://github.com/resourcebridge/resourcebridge.git

# 进入项目根目录
cd resourcebridge

# 安装 Python 依赖（用于运行本地预览服务与链接校验工具）
pip install -r requirements.txt

# 构建静态页面并启动本地预览服务器
python build.py --serve
```

执行上述命令后，在浏览器中访问 http://127.0.0.1:8000 即可查看生成的资源导航页面。若仅需生成静态文件而不启动服务，可执行 `python build.py`，所有输出文件默认写入 `_site` 目录。

## 贡献指南

贡献者应当先阅读 docs/contributing/ 目录下的详细指引文档。以下为主要的贡献流程与规范要点。

第一，Fork 本仓库并在本地克隆 Fork 后的副本。新建一个以 `add-` 或 `update-` 为前缀的功能分支，避免在主分支上直接提交变更。

第二，在 indexes/ 目录下找到与新增资源匹配的分类文件，按照既定的 YAML 或 Markdown 格式追加条目。每个条目必须包含原始 URL、标题、简短描述以及收录日期。禁止修改他人已提交的条目内容，除非同时修复明显的拼写错误或失效链接。

第三，本地执行 `python scripts/check_links.py --new` 仅检查本次新增或修改的链接，确保所有新加入的 URL 返回有效的 HTTP 成功状态码。若链接在三次重试后仍不可达，则不应提交该条目。

第四，提交变更时使用规范的 Commit Message 格式：`[类型] 简短描述`，类型可选 `add`、`update`、`remove`、`fix`。提交说明应使用英文或中文均可，但需保持整个项目历史中的语言一致性。提交前运行 `pytest tests/` 确保所有单元测试通过。

第五，向主仓库的 `main` 分支发起 Pull Request。PR 标题需简要概括本次变更内容，正文中需列出新增或修改的链接数量以及任何可能影响现有分类结构的说明。至少一名项目维护者审阅通过后方可合并。合并后 CI 流水线将自动触发站点重新构建并部署至预览环境。

## 常见问题

问：提交的链接在本地校验时通过，但合并后定期检查报告该链接变为不可达，应该如何处理？

答：ResourceBridge 的链接校验脚本每天凌晨执行全量扫描。当某条链接连续三次（即连续三天）均返回非 200 状态码时，系统会自动生成失效报告并分配给最后一次修改该条目的贡献者。贡献者需要在收到通知后的五个工作日内确认链接状态：若链接已永久迁移则应更新 URL，若资源已下架则应删除条目，若为临时故障则可在备注字段中标记忽略检查直至恢复。逾期未处理的条目将被自动移至待归档目录。

问：是否可以批量导入大量链接，而非逐条手动添加？

答：项目提供了 `scripts/import_csv.py` 工具，支持从 UTF-8 编码的 CSV 文件批量导入链接。CSV 文件需包含 url、title、description、category 和 tags 五列。导入前建议先使用 `--dry-run` 参数执行试运行，检查分类映射与字段格式是否符合预期。批量导入的条目在提交 Pull Request 时建议按逻辑分组，避免单次 PR 包含过多不相关的资源链接，以便于维护者逐项审阅。
