# ResourceBridge

ResourceBridge 是一个面向开发者与技术研究人员的开源外链资源聚合与导航系统。该项目不直接托管任何文件或数据内容，而是通过结构化索引与分类标签体系，将分布在网络各处的优质技术文档、工具站点、学习资料与社区讨论串进行集中管理与快速检索。

ResourceBridge 适用于需要频繁查阅外部技术资料但缺乏统一入口的团队或个人用户。通过本项目提供的索引结构与本地预览服务，用户可以在自托管环境中维护一套私有的外链资源库，避免重复收藏、链接失效或信息散落于浏览器书签中难以整理的问题。项目定位为轻量级、可扩展的链接聚合层，而非内容存储仓库。

## 功能概览

**多级分类索引**：支持按技术领域、资源类型、适用水平等维度对链接进行标记与筛选，便于构建符合自身知识体系的分类树。

**原始链接直出**：所有收录链接在资源列表中保持用户提交时的原始格式，不自动补全协议或域名前缀，确保跳转路径与预期一致。

**本地预览服务**：通过内置的静态服务器，用户可在本地启动 Web 界面，以卡片或列表方式浏览全部资源条目，并支持关键字模糊搜索。

**标签过滤系统**：每条资源可关联多个自定义标签，系统提供标签云与组合过滤功能，帮助用户在海量链接中快速定位目标内容。

**导入导出机制**：支持将资源列表导出为 JSON 或 CSV 格式，便于与其他工具（如书签管理器、笔记软件）进行数据交换或备份。

**结构化管理视图**：以目录树与表格双重视图展示资源层级关系，兼顾技术文档的严谨性与浏览效率。

**轻量化部署**：无需数据库或外部依赖，仅通过静态文件即可运行，适合嵌入现有文档站点或个人开发环境。

**开源可扩展**：核心逻辑与界面模板完全开源，开发者可根据需要新增分类规则、修改展示样式或集成外部 API。

## 应用场景

技术团队内部知识库构建。团队可以将项目部署在内网服务器上，作为统一的技术外链入口，收录常用的官方文档、API 参考、调试工具与最佳实践文章，新成员入职时可直接通过该入口快速了解团队常用的技术资源池。

个人开发环境中的资料聚合。开发者可将项目克隆至本地工作站，将散落在浏览器书签、笔记文件中的数百个技术链接统一录入系统，并通过标签体系按语言、框架、用途等进行精细化管理，减少切换上下文的时间成本。

开源项目文档的外链附录。开源项目的维护者可以使用 ResourceBridge 作为项目 README 或 Wiki 的补充，将大量参考链接、社区讨论、衍生工具等外部资源以结构化表格形式呈现，避免主文档过于冗长，同时保持链接的可维护性。

技术文章或教程的参考文献管理。技术博主或课程讲师可以将本项目的资源列表作为配套材料，按照章节或知识点组织外链，学员可以自行克隆并启动本地服务，在阅读教程时同步查阅外部资料，无需手动输入大量 URL。

## 快速开始

以下命令序列适用于 Linux 与 macOS 环境，Windows 用户建议使用 Git Bash 或 WSL 执行。

```bash
# 克隆项目仓库至本地
git clone https://github.com/resource-bridge/resource-bridge.git

# 进入项目根目录
cd resource-bridge

# 安装依赖（项目基于 Node.js，需要 npm 环境）
npm install

# 启动本地预览服务，默认监听端口 8080
npm start
```

执行完成后，打开浏览器访问 http://localhost:8080 即可查看资源列表与分类导航界面。如需修改端口号，可在项目根目录下的 config.json 文件中调整 server.port 字段。

│   └── middleware/          # 中间件目录，含日志与缓存处理
├── data/
│   ├── resources.json       # 主资源索引文件，存储所有链接与标签

│   └── tags.json            # 标签库及使用频次统计

│   ├── index.html           # 主界面入口页面
│   ├── css/
│   │   ├── main.css         # 全局样式

## 贡献指南

1. 在 GitHub 上 Fork 本仓库，并在本地克隆 Fork 后的副本。创建新的功能分支时，请使用 `feature/` 或 `fix/` 前缀，并附上简要说明，例如 `feature/add-export-format`。

2. 对代码或文档进行修改后，确保通过现有的单元测试与集成测试。若新增功能，请同步补充对应的测试用例与文档说明，测试文件存放于 `tests/` 目录下。

3. 提交变更时，请使用语义化的提交信息格式，即 `<type>(<scope>): <subject>`，其中 type 可选 `feat`、`fix`、`docs`、`style`、`refactor`、`test`、`chore`。

4. 推送分支至个人远程仓库后，通过 GitHub 界面发起 Pull Request 至主仓库的 `main` 分支。PR 描述中请清晰说明变更动机、实现方式及影响范围，并关联相关的 Issue 编号（若有）。

5. 项目维护者会在收到 PR 后的 7 个工作日内进行审核。审核通过后，PR 将被合并至主分支并随下一个版本发布。若需要进一步修改，维护者会在 PR 中留言说明具体要求。

## 常见问题

**问：启动服务时提示端口 8080 已被占用，应如何解决？**

答：可以在项目根目录下的 `config.json` 文件中修改 `server.port` 字段的值，例如改为 `3000`。保存文件后重新执行 `npm start` 即可。也可以在命令行启动时通过环境变量指定端口，如 `PORT=3000 npm start`。

**问：添加新链接后，前端界面没有立即显示更新，是什么原因？**

答：项目在启动时会一次性读取 `data/resources.json` 文件并缓存在内存中。若在服务运行期间手动修改了该文件，需要重启服务（即重新执行 `npm start`）才能加载最新数据。未来的版本计划增加热加载功能，目前暂需手动重启。

**问：能否将资源列表部署到 GitHub Pages 或类似的静态托管服务上？**

答：可以，但需要注意项目当前包含 Node.js 服务端逻辑，无法直接作为纯静态站点部署。若仅需要展示资源列表的快照版本，可以使用 `npm run build` 命令生成静态 HTML 文件，再将 `public/` 目录下的内容上传至静态托管服务。动态搜索与过滤功能在静态模式下会受到限制。
