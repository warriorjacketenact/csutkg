# Tokyo Mirai Resource Indexer

Tokyo Mirai Resource Indexer (TMRI) 是一个面向技术研究、内容归档与跨域资源管理的高效外链索引系统。本项目并非传统意义上的爬虫或采集器，而是一个半自动化的资源映射框架，专为需要高频次、多维度整理分散在网络各处的技术视频、文档与实验数据的研究者与工程师设计。

TMRI 定位于技术运营与知识工程之间的工具层。它帮助用户将松散的外部链接转化为带有上下文、标签与状态标记的内部索引条目，从而降低信息遗失与重复查找的成本。目标用户包括技术文档撰写者、数据分析师、架构师以及需要长期维护外部参考资料的开发团队。

本项目当前批次为第 27 批次，共包含 250 个来自东京未来网络实验室的视频资源外链，涵盖系统架构、性能测试、协议分析与运维自动化等多个方向。TMRI 提供统一的索引模板与命令行工具，便于用户快速导入、分类和导出这批资源。

## 功能概览

**批量链接导入** 支持从纯文本文件或标准输入流中一次性导入大量 URL，自动去重并生成唯一索引标识。

**元数据提取存根** 对每个导入的链接生成元数据存根，包含导入时间戳、批次号、来源标注和自定义标签字段，便于后续筛选与聚合。

**标签系统与多级分类** 允许用户为每个索引条目添加多个层级标签，例如协议类型/分析维度/数据来源，支持基于标签的快速过滤与统计。

**状态追踪与生命周期管理** 每个资源条目可标记为待审阅、已归档、已失效或待更新等状态，支持状态变更日志记录。

**导出适配器机制** 提供 JSON、CSV、Markdown 表格三种导出格式，并可扩展自定义适配器以对接外部文档系统或知识库平台。

**命令行交互与脚本化支持** 所有核心功能均通过命令行子命令暴露，便于嵌入自动化脚本或 CI/CD 流水线。

## 应用场景

**技术文档团队批量引用外部视频资源** 当文档团队需要在一份系统设计说明书中引用数十个外部演示视频时，可使用 TMRI 导入所有链接，统一添加“架构分析”与“性能演示”标签，随后导出为 Markdown 表格直接插入文档，确保引用格式一致且可追溯。

**运维工程师定期归档故障排查录像** 运维人员可将每次故障排查过程中产生的屏幕录制视频链接集中导入 TMRI，标记故障类型、发生时间与处理状态，形成可检索的故障案例库，便于事后回顾与知识传承。

**研究机构整理开放数据集配套资料** 研究机构在发布数据集时，常附带大量实验环境搭建与操作演示视频。TMRI 可帮助整理这些外链资源，生成带索引编号与校验信息的资源清单，作为数据集附录发布。

**个人开发者构建学习路径资源索引** 开发者可收集某一技术方向（如 WebAssembly 或 eBPF）的线上会议视频链接，通过 TMRI 建立分章节、分难度的索引，并定期更新状态，形成动态成长的知识地图。

## 快速开始

以下命令序列演示如何克隆项目仓库、安装依赖并运行一次完整的资源导入与导出流程。

```bash
git clone https://github.com/tokyo-mirai/tmri.git
cd tmri
pip install -r requirements.txt
python tmri.py import --batch 27 --source ./resources_27.txt --output ./index.json
python tmri.py tag --input ./index.json --add "protocol:http" --add "source:tokyo-mirai"
python tmri.py export --format markdown --input ./index.json --output ./catalog_27.md
```

其中 resources_27.txt 为存放原始 URL 列表的纯文本文件，每行一个链接。导入后生成的 index.json 包含所有条目的完整索引信息，后续标签操作与导出均基于该索引文件进行。

## 贡献指南

欢迎以多种方式参与本项目。无论是报告问题、提交代码还是完善文档，均需遵循以下流程。

第一，在 GitHub 仓库的 Issues 区域搜索已有话题，确认没有重复提交相同或高度相似的问题。若为新问题，请使用提供的模板详细描述当前行为、预期行为以及可复现的最小步骤。

第二，对于代码贡献，请先从仓库的 main 分支创建新的特性分支，命名格式为 feature/简要描述 或 fix/问题编号。所有代码提交前需运行 black 与 mypy 进行格式与类型检查，并通过 pytest 测试套件。

第三，提交 Pull Request 时，需在描述中关联相关的 Issue 编号，并说明改动范围与测试覆盖情况。PR 至少需要一位项目维护者审阅通过后方可合并。

第四，文档类贡献（包括 README 更新、用户手册补充或注释完善）可直接提交 Pull Request，无需事先创建 Issue，但需在 PR 描述中明确说明文档改动的动机与受众。

## 常见问题

问：导入的 URL 如果包含重复条目，系统如何处理？

答：TMRI 在导入阶段会基于完整 URL 字符串进行去重。若检测到重复，系统会记录一条警告日志并将该条目标记为“已跳过”，同时保留首次导入时生成的索引 ID 与时间戳。用户可通过 status 子命令查看去重统计信息。

问：标签系统是否支持层级结构，例如 parent/child 形式？

答：当前版本支持以斜杠分隔的扁平标签体系，例如 protocol/http 与 protocol/websocket 被视为两个独立标签，但可通过前缀进行逻辑分组。未来版本计划引入原生层级标签支持，目前可通过命名约定实现类似效果。

问：如何将本项目的索引数据迁移到其他知识管理工具？

答：建议使用 export 命令生成 CSV 或 JSON 格式的导出文件，再通过目标工具提供的导入功能进行迁移。对于 Confluence、Notion 等平台，可基于 Markdown 适配器输出的表格进行二次复制粘贴。若需要定制化迁移脚本，可参考 adapters 目录中的实现编写自定义适配器。
