# LinkSphere Catalog System

LinkSphere Catalog System 是一个面向技术内容聚合与外部资源管理的开源外链汇总平台。项目定位于为开发者、技术博主、研究机构以及知识管理团队提供结构化的外部链接存储、分类、检索与可视化能力，帮助用户将分散在多个来源的参考链接、文档地址、工具站点与案例页面进行集中化管理，并通过标准化的目录结构提升信息复用效率。

本项目并非一个通用的书签管理工具，而是专注于技术资源场景下的外链治理方案。它解决的核心问题是：当团队或个人积累了大量外部 URL 后，如何通过版本控制、标签体系、场景映射与自动化校验来维持资源集合的可用性与可维护性。LinkSphere 适用于需要长期维护技术周报、知识库、项目文档或学习路径的群体，默认以静态站点方式运行，亦可集成至现有文档流水线。

## 功能概览

**结构化外链存储**：所有资源链接以 Markdown 列表形式按类别分节存放，支持层级目录与自定义标签前缀，便于人工审核与脚本解析。

**多维度分类视图**：内置按技术领域、使用频率、资源类型与收录批次四种分类维度，用户可通过修改元数据头快速调整归类逻辑。

**自动链接可用性检查**：集成基于 HTTP 状态码的链接存活探测脚本，可定期输出失效链接报告，帮助维护资源质量。

**快速启动脚手架**：提供一键克隆、依赖安装与本地预览服务，三分钟内完成从拉取到运行的全流程，降低上手门槛。

**文档导航表格生成**：根据目录结构与元数据自动生成面向不同读者角色（终端用户、贡献者、维护者、管理者）的导航表格，提升信息可达性。

**资源批次追踪**：每批收录的资源均附带批次编号与收录日期，支持按批次回滚、对比与增量更新，适合长期运营场景。

**无外部数据库依赖**：全部数据存储于 Markdown 与 YAML 文件中，无需配置独立数据库服务，兼容绝大多数静态托管平台。

## 应用场景

技术周报与月刊编辑：编辑团队可将分散在各大技术社区、博客与官方文档中的参考链接统一收录至 LinkSphere，按栏目分类后生成预览页面，减少手动整理成本并确保链接可追溯。

项目技术选型评估：架构师在调研中间件、框架或云服务时，将候选产品的官网、基准测试报告、社区案例与安全公告汇总至单一目录，便于团队内部评审与决策记录。

知识库外链治理：企业知识管理团队在使用 Confluence 或 Notion 等平台时，可利用 LinkSphere 作为外部链接的权威来源层，通过定期校验避免知识库中出现大量死链，提升文档可信度。

开源项目 README 资源区维护：开源项目维护者可将相关教程、插件生态、视频讲解与迁移案例统一存放于 LinkSphere 管理的资源列表中，并在 README 中引用，减轻主文档的膨胀压力。

个人技术学习路径管理：学习者可按主题（如分布式系统、前端工程化、机器学习基础）建立独立的链接集合，配合备注字段记录阅读进度与心得，形成可版本化的个人学习档案。

## 快速开始

以下命令适用于 Linux 与 macOS 环境，Windows 用户建议使用 WSL 或 Git Bash 执行。

```bash
# 克隆项目仓库
git clone https://github.com/linksphere/linksphere-catalog.git
cd linksphere-catalog

# 安装依赖（Python 3.9+ 与 pip）
pip install -r requirements.txt

# 运行本地预览服务（默认监听 8080 端口）
python serve.py --port 8080
```

执行上述命令后，打开浏览器访问 `http://localhost:8080` 即可查看资源总览页面。如需生成静态 HTML 文件，可运行 `python build.py --output ./dist`，生成的内容可直接部署至任意 HTTP 服务器。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.9 或更高 | 核心运行环境，用于执行构建脚本与链接检查工具 |
| pip | 21.0 或更高 | Python 包管理工具，用于安装第三方依赖库 |
| Markdown 解析库 | markdown 3.4.1+ | 用于将资源列表渲染为 HTML 预览视图 |
| Requests 库 | 2.28.0+ | 用于发送 HTTP 请求以校验链接可用性 |
| PyYAML | 6.0+ | 用于解析分类配置与元数据头信息 |
| Git | 2.30 或更高 | 版本控制工具，用于克隆仓库与提交贡献变更 |
| 操作系统 | Linux / macOS / Windows（WSL） | 开发与运行环境，Windows 原生 PowerShell 未经充分测试 |

## 项目结构

```
linksphere-catalog/
├── README.md                     # 项目概述、安装说明与资源列表索引
├── serve.py                      # 本地开发预览服务入口，基于 http.server 扩展
├── build.py                      # 静态站点生成脚本，输出 HTML 至 dist 目录
├── requirements.txt              # Python 依赖声明文件，锁定主要库版本
├── .github/                      # GitHub 相关配置目录
│   ├── workflows/                # CI 工作流定义
│   │   └── link-check.yml        # 定时运行链接可用性检查的 GitHub Actions 配置
│   └── PULL_REQUEST_TEMPLATE.md  # 拉取请求模板，引导贡献者填写链接变更信息
├── config/                       # 项目运行时配置目录
│   ├── categories.yaml           # 资源分类映射表，定义每个类别对应的目录名与标签
│   ├── metadata.yaml             # 全局元数据配置，包含项目名称、批次编号与默认作者
│   └── filters.yaml              # 链接过滤规则，用于黑名单与特殊协议处理
├── data/                         # 核心资源数据存储目录
│   ├── 174/                      # 第 174 批次原始数据，按收录时间归档
│   │   └── links.md              # 该批次全部链接的 Markdown 列表，带分类标题
│   └── manifests/                # 批次清单文件，记录每批次的收录日期与链接总数
```

## 贡献指南

感谢您对 LinkSphere Catalog System 的关注。本项目遵循以下贡献流程，请仔细阅读并严格按照步骤操作。

第一步：阅读贡献者手册。在提交任何变更之前，请完整阅读 `docs/contributor-guide.md` 文档，确保您已理解链接增删改的格式规范、分类标签的使用约束以及提交信息的撰写要求。

第二步：复刻仓库并创建分支。将本项目复刻至个人账号下，然后从 `main` 分支创建新的功能分支，分支命名建议采用 `feature/add-batch-xxx` 或 `fix/link-xxx` 格式，避免直接在 main 分支上操作。

第三步：修改数据文件并本地验证。根据您的目标，编辑 `data/` 目录下对应批次的 `links.md` 文件，或调整 `config/categories.yaml` 中的分类映射。修改后请运行 `python scripts/check_links.py --path ./data` 进行链接可用性检查，并执行 `python build.py` 确保构建过程无报错。

第四步：提交拉取请求。推送分支至您的复刻仓库后，通过 GitHub 界面提交拉取请求至本项目的 `main` 分支。请完整填写拉取请求模板中的所有字段，包括变更类型、影响的链接列表、验证结果摘要以及相关文档更新情况。

第五步：等待评审与合并。项目维护者将在三个工作日内评审您的拉取请求，可能会提出修改建议。请保持关注并积极回应反馈。合并后您的变更将随下一次批次发布生效。

## 常见问题

问：新增链接时，如何确定应该放入哪个分类目录？

答：请参考 `config/categories.yaml` 中定义的分类映射表。每个分类包含关键词列表和匹配规则。如果新链接的主题与现有分类均不匹配，您可以在拉取请求中建议新增分类，并在 `docs/contributor-guide.md` 中说明理由。项目维护者会根据资源整体分布情况决定是否采纳新分类。

问：链接检查脚本报告某个链接失效，但我确认该链接仍可访问，应如何处理？

答：链接检查脚本默认使用超时 10 秒与重试一次的策略。如果您的链接响应较慢或具有反爬机制，请在 `scripts/check_links.py` 中调整 `TIMEOUT` 与 `MAX_RETRIES` 变量值。若调整后仍失效，请检查该站点是否屏蔽了 `Requests` 库的默认 User-Agent，您可以在脚本中自定义请求头。对于持续误报的链接，可在 `config/filters.yaml` 中将其加入忽略白名单。

问：我可以批量导入我自己的链接集合吗？

答：可以。本项目支持通过 `scripts/migrate_legacy.py` 导入 CSV 或 JSON 格式的链接列表。导入前请确保您的数据包含 URL、标题、分类和备注字段。建议先在测试分支上执行导入并运行构建脚本以验证结果，确认无误后再提交拉取请求。大规模导入（超过 100 条）时请提前在拉取请求中说明，以便维护者安排评审时间。
