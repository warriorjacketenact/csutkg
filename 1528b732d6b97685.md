# ResourceLink Collective

ResourceLink Collective 是一个面向技术文档工程师、开源项目维护者以及知识库管理者的外链资源聚合与规范化管理工具。该项目并非简单的链接收藏夹，而是一套具备元数据提取、链接可用性检测、分类标签生成和 Markdown 文档自动编排能力的轻量级命令行工具集。

项目定位为“技术资源外链的入口治理中间件”，主要解决以下问题：技术写作中反复手动整理外部参考链接效率低下；大量链接缺乏结构化描述，难以检索和维护；不同来源的 URL 格式不统一，导入导出容易出错；团队协作时缺乏统一的链接引用规范。ResourceLink Collective 通过一组约定优于配置的规则，将原始链接列表转化为可维护、可扩展、可审计的资源清单，并自动生成符合开源社区标准的 README 文档骨架。

本项目的典型使用者包括：需要管理数百个外部参考链接的技术文档团队、构建开发者导航站的开源社区运营者、以及希望将零散书签转化为项目文档一部分的独立开发者。

## 功能概览

**批量 URL 导入与校验**：支持从纯文本文件、CSV 或标准输入流中读取大量 URL，自动过滤无效协议、去重并检查域名可达性，输出结构化 JSON 中间文件。

**元数据自动提取**：对每个链接发起轻量级 HEAD 请求，提取响应头中的 Content-Type、Last-Modified 和 Content-Length，并依据路径关键词自动打上初步分类标签。

**Markdown 文档生成引擎**：将结构化链接数据与项目内置的章节模板融合，生成符合开源社区 README 规范的 Markdown 文件，支持自定义章节顺序和标题层级。

**链接分类与标签系统**：内置基于 URL 路径特征的关键词匹配规则，可自动将链接归入技术博客、官方文档、视频教程、工具站点、社区讨论等类别，并支持用户自定义标签覆盖。

**可用性监控与报告输出**：定期或按需对所有已收录链接执行批量连通性检查，生成可用性报告，标记失效链接并记录 HTTP 状态码变化历史。

**模板变量替换与国际化占位**：在文档生成阶段支持 ${VAR_NAME} 形式的变量替换，便于针对不同输出平台或语言环境生成定制化版本。

## 应用场景

技术文档团队的外部参考链接库维护：文档编写者每天引用大量外部资源，使用 ResourceLink Collective 可以将所有参考链接统一录入并自动生成链接清单章节，确保文档中的外部引用可追溯、可验证。

开源项目导航页的自动构建：社区运营者收集了大量与项目生态相关的第三方工具、插件和教程链接，通过本工具可以将原始链接列表一键转换为格式规范、分类清晰的导航 Markdown 页面，直接用于项目仓库的 README 或 Wiki。

个人知识库外链的批量整理与归档：研究员或开发者积累了数百个浏览器书签，需要将其迁移为纯文本格式纳入版本管理。本工具提供导入、清洗、分类和导出全流程，帮助用户将零散书签转化为结构化的知识资产。

## 快速开始

以下命令演示了从克隆仓库到生成一份完整 README 文档的标准流程。

```bash
# 克隆项目仓库
git clone https://github.com/resourcelink/collective.git
cd collective

# 安装依赖（使用 pip 与虚拟环境）
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 准备原始链接文件 urls.txt，每行一个 URL
# 运行核心生成命令，指定输出文件名为 README_generated.md
python -m rlc generate --input urls.txt --output README_generated.md --template template_default.json
```

## 贡献指南

贡献者需遵循以下流程以确保项目质量和链接数据的准确性。

提交链接增删改请求时，需同时附上变更理由及至少一条可验证的来源说明。对于新增链接，建议预先使用本工具的 fetcher 模块测试可达性。

修改核心分类规则或模板结构前，请先在本地运行完整的单元测试套件，确保已有功能不受影响。涉及输出格式变更时，需同时更新 docs 目录下的对应规范文档。

提交的代码或文档变更应保持与现有代码风格一致。Python 代码遵循 PEP 8 规范，Markdown 文档遵循 CommonMark 标准，且每行不超过 120 个字符。

对于链接分类标签的新增或修改，需在 config/custom_rules.yaml 中提供正则表达式匹配示例和至少三个典型 URL 样本，以验证规则的有效性。

提交 Pull Request 时，请填写 PR 模板中的检查清单，包括是否通过测试、是否更新文档、是否添加变更日志条目。所有 PR 需至少一位项目维护者审阅后方可合并。

## 常见问题

问：生成的 Markdown 文档中，链接列表的顺序与输入文件不一致，如何固定顺序？

答：生成器默认按输入顺序保留原始排序。若出现顺序变化，请检查输入文件是否包含 BOM 头或不可见字符导致解析偏移。可在命令行中添加 --preserve-order 参数强制使用原始行序，该参数在 2.0 版本后为默认行为。

问：部分链接无法通过 HEAD 请求获取元数据，导致生成过程中断，如何处理？

答：fetcher 模块默认启用超时重试和降级策略，对 HEAD 失败的链接会自动回退至 GET 请求并仅读取前 512 字节。若仍失败，该链接会被标记为 "metadata_unavailable" 并继续处理后续链接，不会中断整体流程。可在配置文件中调整 max_retries 和 timeout 值。

问：输出文档中的分类标签不符合预期，如何自定义？

答：分类引擎基于 config/default.yaml 中的正则规则进行匹配，优先级按顺序执行。用户可在 config/custom_rules.yaml 中添加自己的规则，规则格式为 "标签名: 正则表达式"。自定义规则会合并到默认规则之后，且优先级高于默认规则。修改后重新运行 generate 命令即可生效。
