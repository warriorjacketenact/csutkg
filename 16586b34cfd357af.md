# LinkVault Resource Aggregator

LinkVault 是一个面向技术团队、内容创作者和研究人员的高效外部资源链接管理与导航系统。本项目并非传统意义上的爬虫或采集器，而是一个结构化的外链索引仓库，旨在解决分散在各类技术文档、团队 Wiki 与个人书签中的资源丢失、分类混乱与检索困难问题。

项目核心定位为“技术资源的元目录”，通过预定义的分类维度与固定的 URL 命名空间，将大量外部视频、文档、工具页面映射为本地可维护的引用列表。LinkVault 不对目标资源进行二次分发或代理转发，仅提供定位服务与访问指引。

目标用户包括：
- 需要维护团队技术周报、月刊的外部链接合集的 DevOps 与文档工程师；
- 定期追踪特定领域（如 Web 开发、系统运维、AI 应用）视频教程与案例分析的研究者；
- 希望在项目 README 或内部知识库中嵌入稳定、可审计的外部引用列表的技术管理者。

LinkVault 采用纯静态 Markdown 作为数据存储格式，无需数据库或后端服务，可直接托管于 Git 仓库、对象存储或静态站点生成器中。每一批次资源均以独立目录隔离，支持增量更新与版本回滚，保障引用链路的长期可追溯性。

## 功能概览

- **批次化资源索引**：每个资源批次拥有唯一编号与归档日期，支持按批次维度浏览全部链接，便于周期性审查与更新。

- **分类导航视图**：根据 URL 路径中的模块前缀（如 vod、doc、api）自动归入不同分类视图，提供按技术领域、内容类型或团队归属的筛选入口。

- **链接状态检测**：集成基于 HTTP HEAD 请求的轻量级存活检查，可标记异常链接（404、超时、证书错误），输出检测报告供维护者参考。

- **自定义字段注解**：每条链接支持附加最多 5 个键值对注解字段，例如“主讲人”“时长”“难度等级”“适用版本”“关联议题”，满足精细化管理需求。

- **Markdown 原生渲染**：所有链接列表、表格与树形结构均使用标准 Markdown 语法，兼容 GitHub、GitLab、Gitea 等主流代码托管平台的渲染引擎。

- **批量导入与导出**：支持从 CSV、JSON Lines 格式批量导入链接清单，导出为结构化 YAML 或 SQLite 数据库文件，便于与其他工具链集成。

- **变更审计日志**：每次增删改操作自动记录操作人、时间戳与变更摘要，日志存储于 .changes 子目录，支持 diff 对比与回滚。

- **模板生成器**：提供命令行工具一键生成新批次的目录骨架、占位 README 与示例注解文件，降低初始化成本。

## 应用场景

**团队周报外链归档**  
技术负责人每周汇总团队成员的分享视频、设计文档与第三方工具评测链接，按批次录入 LinkVault 后，可直接嵌入周报邮件或 Confluence 页面，避免链接散落在聊天记录中。

**开源项目外部依赖索引**  
开源项目维护者可将项目依赖的参考实现、协议说明、性能测试基准等外部资源统一纳入 LinkVault 管理，随代码仓库一同发布，方便贡献者快速了解项目背景与设计依据。

**技术培训课程资源包**  
企业培训部门为每期训练营准备的外部阅读材料、实验环境入口与案例演示视频，通过 LinkVault 按批次组织，学员可通过统一的资源列表访问全部材料，无需反复询问讲师。

**研究文献与数据溯源**  
学术研究团队在撰写论文或技术报告时，将引用的在线数据集、工具仓库、预印本链接集中存放于 LinkVault 批次目录中，确保评审与复现过程中引用路径清晰可查。

## 快速开始

以下指令适用于 Linux / macOS / Windows WSL 环境，需预先安装 Git 与 Node.js 18+ 或 Python 3.10+（视具体实现而定）。

```bash
# 克隆仓库
git clone https://github.com/your-org/linkvault.git
cd linkvault

# 安装依赖（以 Node.js 实现为例）
npm install

# 初始化当前批次（第 419 批）
npm run init -- --batch 419

# 导入本批次资源列表
npm run import -- --batch 419 --source ./data/419_raw_links.txt

# 生成本批次 README 预览
npm run build -- --batch 419 --output ./dist/README_419.md
```

若使用 Python 实现版本：

```bash
# 创建虚拟环境
python -m venv venv
source venv/bin/activate

# 安装依赖
pip install -r requirements.txt

# 初始化批次并导入
python vault.py init --batch 419
python vault.py import --batch 419 --file ./data/419_raw_links.txt

# 生成静态页面
python vault.py render --batch 419 --output ./dist/
```

执行完成后，可在 ./dist/ 目录下找到当前批次的完整资源列表页面，直接使用浏览器打开或交由 CI/CD 流水线发布。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | 18.x 或 20.x LTS | 运行时环境，用于执行导入、检测与渲染脚本 |
| npm | 9.x 或更高 | 包管理器，用于安装项目依赖库 |
| Git | 2.30 或更高 | 版本控制，用于克隆仓库与提交变更日志 |
| curl / wget | 任意稳定版本 | 链接存活检测工具，用于发送 HEAD 请求 |
| Python 3 | 3.10+（可选） | 仅在使用 Python 实现时必需，Node 实现无需此项 |
| sqlite3 | 3.35+（可选） | 仅在启用 SQLite 导出功能时需要 |

## 贡献指南

1. 提交 Issue 说明需求或缺陷  
   在 GitHub Issues 页面新建议题，使用提供的模板填写批次编号、链接数量、期望功能或异常现象。对于新增链接建议，请附带来源说明与分类推荐。

2. 创建功能分支并克隆仓库  
   使用 git checkout -b feature/your-feature-name 创建新分支，避免直接在主分支上操作。确保本地环境满足安装要求中列出的所有依赖版本。

3. 导入新批次或更新现有批次  
   若需添加新资源批次，运行 npm run init -- --batch <编号> 生成骨架，随后将原始链接列表放入 data/ 目录并执行导入命令。若更新现有批次，请先在 .changes/ 对应批次目录下记录变更说明。

4. 运行测试与检测  
   执行 npm test 运行全部单元测试，确保覆盖率不低于 85%。运行 npm run check -- --batch <编号> 对批次内链接进行存活检测，将报告归档至 logs/ 目录。

5. 提交 Pull Request 并等待审核  
   推送分支至远程仓库后，创建 Pull Request 并填写变更摘要。维护者将在 3 个工作日内完成审核，通过后合并至主分支并触发自动渲染流水线。

## 常见问题

**问：为什么需要固定 URL 原样输出，禁止添加协议或修改域名？**  
答：LinkVault 作为纯引用索引，不承担代理或重定向功能。保持 URL 原样输出可确保用户直接访问目标资源时不受中间层干扰，同时避免因协议不一致（如 http 与 https）导致的混合内容警告或 CORS 错误。任何对 URL 的修改都将破坏引用链路的确定性，因此项目明确规定不得补全协议前缀、不得移除 www 子域、不得改变大小写或末尾斜杠。

**问：如何处理链接失效或目标页面内容变更的情况？**  
答：项目提供 health-check 命令，可定期扫描所有链接的 HTTP 状态码与响应头。对于返回 4xx 或 5xx 的链接，系统会在检测报告中标记为“异常”，并建议维护者人工核实。若目标页面内容发生重大变更（如原视频被替换为无关内容），维护者应通过变更记录流程将该链接移入“已归档”分类，并附加说明。LinkVault 不自动删除失效链接，而是保留历史记录以便追踪资源生命周期。

**问：能否将本批次资源列表与其它批次的资源合并生成统一索引？**  
答：可以。项目提供 render --merge 命令，支持指定多个批次编号，合并生成一份包含全部链接的完整索引页面。合并后的列表会按批次号降序排列，并在每条链接旁标注所属批次，方便用户快速定位来源。合并索引的输出格式同样为 Markdown 或 HTML，可直接用于项目 Wiki 或静态站点。
