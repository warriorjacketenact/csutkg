# ResourceBridge

ResourceBridge 是一个面向技术团队与个人开发者的高质量外链资源聚合与导航工具。项目定位为“技术资源的统一入口”，旨在解决开发者在日常工作中频繁搜索文档、工具、教程、依赖库、API 参考等碎片化信息所带来的效率损耗。通过结构化分类与轻量级检索机制，ResourceBridge 帮助用户将分散在数百个域名的优质资源整合为可复用、可共享的知识索引体系。

目标用户包括运维工程师、全栈开发人员、技术架构师、开源项目维护者以及技术研究团队。项目本身不存储任何第三方内容，仅作为 URL 元数据管理平台，提供资源分类、标签过滤、状态检测与快速跳转能力。ResourceBridge 适用于需要长期维护技术选型资料库、团队 onboarding 知识库、或个人学习路径管理体系的场景。

## 功能概览

**资源分类目录** 支持按技术领域、使用频率、适用人群等多维度创建自定义分类，每个资源条目可归属多个分类标签，实现灵活检索。

**链接健康检测** 定时对收录的 URL 进行 HTTP 状态码检查，自动标记失效或重定向链接，确保索引库的可用性与准确性。

**元数据扩展字段** 每条资源记录支持记录摘要描述、维护人、更新日期、版本兼容性、相关项目等额外信息，便于团队协作。

**快速模糊搜索** 基于资源标题、域名关键词、分类标签进行实时匹配，支持拼音首字母检索，降低查找成本。

**导入导出机制** 支持批量导入 URL 列表（纯文本/CSV），并可导出为 JSON/YAML 格式，便于与其他工具链集成或版本控制。

**访问统计分析** 记录每个链接的点击次数与最近访问时间，辅助判断资源热度，为目录排序与归档提供数据参考。

## 应用场景

技术团队知识库构建 新成员入职时，团队可将长期积累的 API 文档、内部工具、监控面板、日志平台等链接通过 ResourceBridge 统一整理，形成可检索的团队资源门户，显著缩短上手周期。

个人开发环境配置同步 开发者可将本地开发常用的镜像源、依赖查询、代码生成、在线调试工具等链接集中托管，在不同工作设备间同步资源列表，保持环境一致性。

开源项目外部依赖索引 开源项目维护者可使用 ResourceBridge 管理项目依赖的第三方库主页、补丁追溯地址、版本发布说明页、安全公告源等外链，替代散落在 README 中的无序超长链接列表。

技术调研与选型记录 在进行技术方案对比时，可临时创建调研分类，收录各候选方案的官网、性能基准测试报告、社区讨论帖、替代品对比文章等，形成结构化的决策依据档案。

## 快速开始

以下命令适用于 Linux / macOS / Windows WSL 环境，Go 1.21+ 或 Node.js 18+ 均可运行。

```bash
# 克隆仓库
git clone https://github.com/resourcebridge/resourcebridge.git
cd resourcebridge

# 安装依赖（Go 版本）
go mod download

# 构建并运行
go build -o resourcebridge ./cmd/server
./resourcebridge --config ./configs/default.yaml

# 或使用 Node.js 版本
npm install
npm run build
npm start
```

服务启动后将监听 8080 端口，访问 http://localhost:8080 即可进入资源管理面板。首次运行会自动生成示例数据与默认管理员账户，请查阅控制台输出的初始密码。

## 项目结构

```
resourcebridge/
├── cmd/
│   ├── server/                     # 主服务入口，初始化路由与中间件
│   └── migrate/                    # 数据库迁移命令行工具
├── internal/
│   ├── api/                        # RESTful API 处理器，按资源/分类/统计划分
│   ├── core/                       # 核心业务逻辑：资源索引、健康检测调度
│   ├── models/                     # 数据模型定义 (GORM/Ent) 与 DTO 转换
│   ├── repository/                 # 数据访问层，封装 PostgreSQL 操作
│   └── cache/                      # Redis 缓存策略实现，含分类树序列化
├── pkg/
│   ├── checker/                    # HTTP 健康检测客户端，支持重试与超时配置
│   ├── parser/                     # 批量导入解析器 (CSV/JSON/Plaintext)
│   └── utils/                      # 通用工具：ID 生成、时间处理、字符串匹配
├── configs/
│   ├── default.yaml                # 默认配置（端口、日志级别、数据库连接）
│   └── production.yaml.example     # 生产环境配置示例，含安全参数
├── web/
│   ├── static/                     # 前端静态资源 (CSS/JS/图标)
│   ├── templates/                  # 服务端渲染页面 (Go template)
│   └── dist/                       # 前端构建产物 (React/Vue 打包)
├── scripts/
│   ├── seed.sql                    # 初始分类与演示数据
│   └── healthcheck.sh              # 外部健康探测脚本，由 cron 触发
├── docs/                           # 完整文档目录，见上方文档导航
├── docker-compose.yml              # 本地开发依赖：PostgreSQL + Redis + Adminer
├── Dockerfile                      # 多阶段构建文件，最终镜像约 25MB
├── Makefile                        # 统一构建入口：make build / test / run
├── go.mod                          # Go 模块依赖定义
└── README.md                       # 本文件
```

## 贡献指南

1. 阅读设计概述文档 docs/design/architecture.md 理解模块边界与数据流向，在 GitHub Issues 中认领或提交新的功能提案或缺陷报告。

2. 派生项目仓库至个人账号，创建以 fix/ 或 feat/ 前缀命名的特性分支，保持分支名称与关联 Issue 编号对应。

3. 编写代码时遵循项目根目录 .golangci.yml 或 .eslintrc 中定义的代码规范，所有导出函数与结构体需包含英文注释，并补充对应的单元测试文件。

4. 提交前执行 make test 与 make lint 通过全部检查，确保测试覆盖率不低于 80%，并更新 docs/ 下相关文档中的示例或配置说明。

5. 发起 Pull Request 至 main 分支，描述中需引用关联 Issue 编号，并附上手动测试截图或日志片段，等待至少一位维护者审核。

## 常见问题

Q: 健康检测模块是否会因为目标网站响应缓慢而阻塞主服务？
A: 不会。健康检测采用独立 goroutine 池 + 超时控制（默认 5 秒），检测结果通过 channel 异步写入数据库，主请求路径不受影响。超时与重试次数可在 configs/default.yaml 中的 checker 段落调整。

Q: 如何迁移已有的散乱书签或收藏夹数据到 ResourceBridge？
A: 支持导入 Chrome 书签导出的 HTML 文件（需转换为 CSV 格式）、标准 Netscape 书签格式、以及每行一个 URL 的纯文本列表。导入时可通过 --tags 参数为所有链接统一附加分类，导入后可在管理界面进行细粒度编辑。

Q: 项目是否支持多用户与权限隔离？
A: 当前版本仅提供单用户模式，所有资源记录共享同一命名空间。多用户与团队空间功能已在 roadmap 中规划，预计在 v2.0 版本引入基于角色的访问控制（RBAC），届时将支持团队内只读/编辑角色划分。
