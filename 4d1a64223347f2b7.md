# StreamHub 技术资源导航站

StreamHub 是一个面向开发者、运维工程师与技术研究人员的结构化外链资源聚合平台。本项目不对任何外部资源进行二次存储或镜像，而是以目录索引与分类导航的形式，将分散于互联网各处的高质量技术文档、社区讨论、工具链入口与协议参考进行系统化归集。

本项目的核心定位是降低技术信息检索的边际成本。在日常研发工作中，开发者常常需要在数十个书签、历史记录与搜索引擎结果之间反复切换，以寻找某个特定协议的实现细节、某次社区讨论中的解决方案或某个依赖库的官方文档。StreamHub 通过预设的流式分类规则与稳定的标识符映射，将上述资源入口统一收敛至可预测的 URL 路径体系之下，从而帮助用户建立长期、稳定的外部知识获取管道。

目标用户包括但不限于：需要频繁查阅第三方参考资料的底层系统开发者、从事协议分析与网络编程的研究人员、负责技术选型与架构评审的团队负责人，以及希望系统化构建自身技术知识图谱的自学者。

## 功能概览

**结构化资源索引**：按照技术领域、协议类型与主题标签对每一枚外部链接进行三级分类，所有分类规则记录于项目根目录的分类映射表中。

**永久标识符映射**：每个资源入口均分配一个固定长度的流标识符（Stream ID），该标识符与上游 URL 形成稳定映射关系，确保在外部链接发生迁移时能够通过更新映射表保持导航连续性。

**轻量级元数据标注**：为每个资源条目标注内容类型（官方文档、社区讨论、工具页面、协议草案、实现参考等）、最后校验时间与内容摘要。

**多维度检索支持**：支持按 Stream ID、按域名前缀、按内容类型与按更新时间范围进行组合过滤查询。

**自动健康检查**：内置基于 HTTP 状态码与响应超时的链接可用性探测任务，可定期输出失效链接报告。

**纯静态生成架构**：项目构建产物为纯 HTML 与 Markdown 文件，无需后端服务或数据库即可完整运行，适配 GitHub Pages、Cloudflare Pages 等任意静态托管环境。

## 应用场景

场景一：技术调研期间的快速参考索引。当工程师需要调研某一种网络协议或某一类中间件的社区生态时，可直接通过 StreamHub 的协议分类浏览相关资源，避免在海量搜索引擎结果中重复筛选。

场景二：团队内部知识库的外链治理。技术团队可以将 StreamHub 作为团队文档体系的外链前置层，所有指向外部站点的链接统一通过 StreamHub 的标识符进行引用，当外部资源变动时只需在 StreamHub 中更新映射，无需修改所有内部文档。

场景三：个人开发环境的起始页定制。开发者可以将 StreamHub 部署为本地浏览器的起始页或新标签页替换插件的数据源，每天开始工作时通过浏览更新列表获取关注领域的最新社区讨论与技术文章。

场景四：离线文档体系的入口备份。对于需要在内网环境或断网条件下工作的场景，StreamHub 的资源清单可作为离线文档收集计划的任务列表，按清单逐一下载并归档至内部存储。

## 快速开始

以下命令适用于 Linux / macOS / Windows WSL 环境，要求已安装 Git 与 Node.js 18.0 及以上版本。

```bash
# 克隆项目仓库
git clone https://github.com/streamhub/streamhub.git
cd streamhub

# 安装项目依赖
npm install

# 执行构建流程，生成静态资源索引页面
npm run build
```

构建完成后，所有生成的静态文件位于 `dist/` 目录下，可使用任何静态文件服务器进行预览，例如：

```bash
npx serve dist
```

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | 18.0.0 或更高 | 运行时环境，用于执行构建脚本与依赖管理 |
| npm | 8.0.0 或更高 | 包管理器，用于安装项目依赖项 |
| Git | 2.25.0 或更高 | 版本控制工具，用于克隆仓库与提交更新 |

| 磁盘空间 | 至少 500 MB | 用于存储源代码、依赖包与构建产物 |
| 操作系统 | Linux / macOS / Windows（WSL2） | 支持主流 POSIX 环境，Windows 用户建议使用 WSL2 以避免路径兼容性问题 |

## 贡献指南

第一，阅读项目行为准则与贡献者许可协议。所有贡献者需确保所提交的资源链接不包含任何违法、侵权或恶意内容，并同意将其贡献内容以与项目相同的许可证进行分发。

第二，从资源列表中选择需要新增或更新的条目。新增资源需提供完整的 URL、建议的分类标签以及简短的内容描述。对于已有条目的更新，需说明更新理由，例如链接失效、内容迁移或分类调整。

第三，通过 Fork 仓库并提交 Pull Request 的方式贡献。在提交前请运行 `npm run test` 确保所有现有测试用例通过，并运行 `npm run build` 确认构建流程能够正常完成。新增资源建议附带对应的单元测试用例。

第四，等待项目维护者的代码审查。审查过程中维护者可能会就资源分类的合理性、描述的准确性以及链接的长期可用性提出问题，贡献者需在规定时间内回应并完成相应修改。

## 常见问题

问：StreamHub 是否会对引用的外部资源进行内容缓存或快照保存？

答：不会。本项目仅存储外部资源的 URL 及其元数据描述，不缓存任何来自外部站点的页面内容、文件或数据。用户访问资源列表中的链接时，将直接与原始站点建立连接。

问：如果我发现的某个外部链接已经失效，应该如何处理？

答：可以通过 GitHub Issues 提交链接失效报告，或按照贡献指南中的流程提交 Pull Request 将该链接标记为失效状态。项目内置的健康检查任务也会定期扫描所有链接，并将失效链接汇总至 `data/unreachable.json` 文件中以供查阅。

问：能否将 StreamHub 部署在完全离线或内网环境中？

答：可以。首次构建时需要外网访问权限以抓取必要的元数据，完成构建后所有静态文件均可迁移至离线环境运行。对于内网部署场景，建议在配置文件中将 `externalFetch.enabled` 设置为 `false`，以禁用构建时的实时网络请求。
