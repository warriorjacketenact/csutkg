# ResourceBridge 资源桥接系统

ResourceBridge 是一个面向技术内容聚合与外部资源导航的开源基础设施项目，定位于为开发者、技术研究人员及信息整理者提供高效、结构化、可扩展的外链资源管理与分发方案。本项目并非简单的链接收藏工具，而是一套完整的资源元数据采集、分类索引、健康检查与快速访问管道，适用于需要长期维护大量外部参考链接的技术团队或个人知识库。

目标用户包括技术文档编写者、开源项目维护人、在线教育内容策划者以及任何需要将分散于网络各处的优质资源进行统一归口管理的专业人士。ResourceBridge 通过标准化 URL 处理流程、自动化可用性检测与友好的目录导航结构，显著降低资源链接失效风险，提升团队协作中外部参考信息的复用效率。

## 功能概览

批量链接导入解析：支持从纯文本、CSV 及 JSON 格式批量导入 URL 列表，自动完成协议归一化与去重校验，保留原始链接完整字面量。

自动化健康检查：后台定时任务对已收录链接进行 HTTP 状态码探测，标记异常链接并生成报告，辅助维护人员及时更新或下架失效资源。

多维度分类索引：基于链接路径、域名及自定义标签体系，为每条资源生成多级分类索引，支持快速筛选与定位。

元数据扩展字段：为每个链接附加描述、关键词、收录时间、最后检查时间及归属批次（如第 559/1241 批）等元数据，便于长期跟踪。

只读访问模式：生产环境默认开启只读接口，确保资源列表的稳定输出，避免误操作污染原始数据。

标签过滤与搜索：提供基于标签组合的过滤语法，支持与或非逻辑查询，快速定位特定主题或来源的资源条目。

导入导出互操作：支持将资源列表导出为 Markdown 表格、JSON 结构或纯文本列表，便于嵌入其他文档系统或迁移至第三方平台。

## 应用场景

技术文档团队外链管理：技术文档中常引用大量外部规范、教程或工具站点。使用 ResourceBridge 可集中管理这些引用链接，并在文档构建流程中动态生成最新的外链附录，避免文档发布后出现死链。

开源项目 README 资源附录生成：开源项目维护者需要定期整理社区贡献的教程、视频或衍生项目链接。ResourceBridge 可按批次导入这些链接，自动生成符合规范的 Markdown 资源列表章节，直接粘贴至 README 中。

个人知识库外部参考归档：研究员或工程师在阅读技术资料时积累大量收藏链接。通过 ResourceBridge 的标签与分类功能，可建立个人化的外部知识索引，配合健康检查机制持续维护链接有效性。

离线文档打包前置处理：在构建离线文档包（如 Docker 镜像内置文档、电子书附录）时，使用 ResourceBridge 预处理所有外链，生成静态资源映射表，供文档渲染引擎使用。

## 快速开始

以下指令适用于 Linux/macOS/WSL 环境，需预先安装 Git、Node.js 20.x 及以上版本与 npm。

```bash
git clone https://github.com/resourcebridge/resourcebridge.git
cd resourcebridge
npm install
npm run start
```

启动成功后，服务默认监听 3000 端口，可通过 http://localhost:3000 访问管理界面。资源列表数据存储于 `data/registry.json`，可直接替换或追加条目。

## 贡献指南

提交 Issue 报告链接失效或建议新增分类标签时，请附上具体的资源 URL 及预期分类，并说明该资源的用途或来源背景。

如需新增导入格式支持（如 Excel、RSS 源），请先查阅 `docs/development.md` 中关于 Importer 接口的定义，提交 Pull Request 时需包含对应的单元测试用例。

对健康检查模块的调整（如增加超时重试策略、自定义 User-Agent）应确保不影响默认配置下的稳定性，修改后需通过 `npm run test` 全量测试套件。

文档更新类贡献请直接修改 `docs/` 目录下对应的 Markdown 文件，并保持中文表述的技术准确性与简洁性，提交时在 PR 描述中标注为文档变更。

## 常见问题

服务启动后访问管理界面空白，控制台提示端口占用。解决方式：修改 `config/default.yaml` 中的 `port` 字段为其他可用端口，或使用 `npm run start -- --port 3001` 指定端口启动。

健康检查模块报告大量链接超时，但浏览器可正常访问。原因：默认检查超时时间为 3 秒，部分境外站点响应较慢。可在配置文件中将 `checker.timeout` 调整为 10000（毫秒），并适当增加 `checker.retry` 次数。

如何将本项目的资源列表迁移至另一台服务器。方式：直接复制 `data/registry.json` 文件至新服务器的相同相对路径下，重启服务即可。若需保留历史检查记录，一并迁移 `data/history.db` 文件。
