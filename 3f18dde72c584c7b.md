# ResourceBridge

ResourceBridge 是一个面向技术研究、内容聚合与知识管理场景的轻量化外链资源汇总平台。项目定位为结构化技术资源导航工具，核心目标用户包括技术文档撰写者、开源社区维护者、研究机构信息专员以及需要系统化管理外部参考链接的开发团队。

ResourceBridge 解决的核心问题是：当项目或团队需要维护大量外部 URL 资源（如参考文献、技术规范、数据源、演示素材等）时，如何以可版本控制、可协作、可快速检索的方式统一收纳、分类展示和持续更新。项目本身不存储任何实际内容，仅提供清晰的目录结构与标准化收录规范，使资源管理行为与代码仓库工作流完全融合。


## 功能概览

- **结构化资源收录**：提供统一的资源条目格式，支持按主题、来源、文件类型等多维度对链接进行分类整理，确保每条资源均有明确的归属层级。

- **Markdown 原生渲染**：所有资源列表与说明文档均采用标准 Markdown 语法编写，可直接在 GitHub、GitLab 或本地编辑器中进行查看与编辑，无需额外解析工具。

- **自动目录树生成辅助**：项目维护了完整的虚拟目录树结构，可直观反映资源分类逻辑，便于新贡献者快速理解资源组织方式。

- **批次化管理机制**：支持将大量资源拆分为多个收录批次，每批次独立编号，便于追踪新增资源的入库时间与责任归属。

- **外部链接状态检测接口**：项目提供简单的链接可达性检测脚本，可定期扫描资源列表中的 URL，标记可能失效或变更的地址。

- **多场景适配输出**：资源数据可被上游工具链消费，支持生成纯文本清单、HTML 导航页或 JSON 结构数据，适应不同发布环境的需求。

- **协作友好型贡献流程**：通过标准化的 Issue 模板与 Pull Request 检查清单，确保资源增删改操作经过必要审核，维护整体质量。


## 应用场景

**技术文档外部引用管理**：技术团队在撰写产品白皮书或 API 文档时，需引用大量外部规范、标准协议或第三方评测报告。ResourceBridge 提供统一的收录入口，使引用来源清晰可溯，避免链接散落在文档正文中难以维护。

**开源项目 README 资源导航**：开源项目维护者通常需要在 README 中陈列社区教程、视频讲解、相关工具等外链。ResourceBridge 可作为一个独立的资源清单子模块，通过子模块方式引入主项目，降低主文档的冗余度。

**研究机构文献索引共享**：研究团队在开展文献调研时，需整理大量 PDF 下载地址、数据集主页、工具仓库等。ResourceBridge 支持按研究课题建立独立资源集，并可通过版本管理跟踪链接的增删记录，方便团队成员同步信息。

**企业内部知识库外链治理**：企业内部 Wiki 常包含大量指向外部供应商、技术社区或云服务控制台的链接。ResourceBridge 提供集中化管理方案，配合定时检测脚本，可及时发现失效链接并提醒维护人员更新。


## 快速开始

以下命令演示了如何将 ResourceBridge 项目克隆至本地、安装基础依赖并启动本地预览服务。

```bash
git clone https://github.com/resourcebridge/resourcebridge.git
cd resourcebridge
npm install
npm run dev
```

执行上述命令后，本地开发服务器将在 3000 端口启动，访问 http://localhost:3000 即可查看资源导航页面的预览效果。若需仅生成静态资源清单，可执行 `npm run build`，产物将输出至 `dist` 目录。


## 安装要求

| 依赖 | 必需 | 说明 |
|------|------|------|
| Node.js 18.x 或更高版本 | 是 | 项目构建工具与本地服务器运行环境 |
| npm 9.x 或更高版本 | 是 | 用于安装项目依赖及执行构建脚本 |
| Git 2.30 或更高版本 | 是 | 克隆仓库及版本控制操作 |
| 现代网页浏览器 | 否 | 仅用于本地预览资源导航页面，非运行必需 |
| shell 环境 (bash/zsh) | 否 | 仅当使用自动化链接检测脚本时需要 |

## 贡献指南

1. **查阅贡献者指南**：在提交任何资源变更之前，请先阅读 `docs/contributing/` 目录下的完整贡献规范，了解资源条目的标准格式、分类原则以及批次编号规则。

2. **创建新分支并更新资源文件**：根据要修改的资源分类，在 `data/batches/` 下定位对应的 YAML 文件，按照既定格式增删或修改资源条目。建议每次提交仅处理一个批次或一个分类的变更。

3. **运行本地校验**：执行 `npm run validate` 命令对修改后的资源数据进行格式校验和链接可达性检查，确保所有新增 URL 无语法错误且目标地址可正常访问。

4. **提交 Pull Request**：将变更推送至远程仓库后，通过 GitHub 提交 Pull Request。PR 描述中需注明本次变更的批次编号、变更类型（新增/修改/删除）以及必要的变更理由。

5. **等待审核与合并**：项目维护者将在 2 个工作日内审核 PR。若审核通过，资源变更将被合并至主分支；若需修改，维护者将在 PR 中给出具体反馈意见。


## 常见问题

**Q：ResourceBridge 是否存储外部资源的内容副本？**

ResourceBridge 仅收录外部资源的 URL 地址和必要的元数据描述，不缓存、不代理、不存储任何外部资源的内容副本。所有资源访问均跳转至原始地址，项目本身不承担内容可用性的保证责任。

**Q：如何批量导入大量已有链接？**

项目提供了 `scripts/import-csv.sh` 辅助脚本，可将符合格式的 CSV 文件批量转换为 YAML 资源条目。CSV 需包含 url、title、category、description 四列。执行脚本时需指定目标批次编号和分类名称。

**Q：链接检测脚本的检测频率如何设置？**

`npm run check` 命令执行单次全量检测。若需定期自动检测，建议通过系统 cron 任务或 CI 定时任务调用该命令，并将检测报告输出至指定日志文件。检测超时时间可在 `src/core/validator.ts` 中调整。
