# ResourceVault

ResourceVault 是一个面向技术团队与内容聚合场景的开源外链资源管理工具。该项目并非传统意义上的爬虫或采集器，而是一个专门用于对批量 URL 进行结构化清洗、分类归档、可用性检测与只读视图渲染的轻量级中间件。其核心定位是帮助运维人员、技术文档工程师与数据治理团队，将散落在各类日志、监控看板、第三方平台中的异构链接，转化为可维护、可检索、可定期验证的内部资源清单。

本项目不提供数据库持久层，不绑定任何特定前端框架，以纯静态资源生成器为核心工作流，配合定时任务或 CI 流水线，即可在数分钟内完成对数千条链接的状态预检与目录索引生成。ResourceVault 不对原始资源内容做任何实质性修改，仅通过元数据提取与分类标记，为下游系统提供标准化的 JSON Schema 输出与 Markdown 文档呈现。该项目适用于需要长期维护外部依赖引用列表、技术文档外链库、或第三方服务监控入口的中大型研发组织。

## 功能概览

批量 URL 注入与解析：支持从纯文本文件、CSV 或标准输入流中一次性导入大量链接，自动识别协议头与路径结构，并对畸形 URL 给出预警日志。

分类标签自动推断：基于 URL 中的二级目录、文件名后缀及路径关键词，通过可配置的正则规则集为每条链接分配初步分类标签，例如文档、视频、API 端点或第三方工具台。

可用性基线探测：内置基于 HTTP 状态码与响应超时的轻量探测模块，可配置重试策略与超时阈值，在资源清单生成前标记出失效或高延迟链接，支持输出健康报告。

结构化目录树输出：基于 URL 的层级关系自动生成 ASCII 目录树视图，便于快速理解资源归属与路径分布，该视图可直接嵌入 README 或 Wiki 页面。

只读资源列表渲染：将经过清洗与分类的链接集合渲染为固定格式的 Markdown 列表，支持按域名、按目录深度或按自定义标签进行分组展示，确保输出结果稳定且可读。

配置化忽略规则：支持通过 .resourceignore 文件定义需要排除的路径模式（如测试环境域名、特定前缀或文件类型），避免无关资源混入最终清单。

CI/CD 原生集成：提供无状态的命令行接口，所有输入输出均通过标准文件或环境变量传递，可无缝接入 Jenkins、GitLab CI 或 GitHub Actions 工作流，实现资源清单的自动化更新。

## 应用场景

技术文档站点外链审计：研发团队在维护 API 文档或开发者指南时，通常会引用大量外部 SDK 地址、规范标准链接或社区讨论帖。随着时间推移，其中部分链接可能失效或被重定向。ResourceVault 可被配置为文档构建流程的前置检查步骤，每次构建时自动拉取最新外链列表并输出可用性状态报告，帮助文档维护者及时发现并替换失效引用。

微服务治理中的依赖映射：在微服务架构中，各个服务之间通过 HTTP 端点进行通信，运维团队往往需要维护一份服务发现地址清单。ResourceVault 能够定期读取注册中心或配置中心导出的 URL 列表，进行可达性探测与路径规范化，最终生成一份带状态标记的资源地图，供监控系统或故障排查手册使用。

第三方服务商入口聚合管理：企业通常会采购多家云服务商或 SaaS 工具，其管理控制台地址分散在不同域名下。通过 ResourceVault，可将这些入口统一收录并按业务线分类，结合可用性探测功能，在每周自动化报告中高亮显示无法正常访问的控制台地址，从而降低因平台故障导致的管理盲区。

数据治理中的链接标准化前置处理：在数据中台或数据湖建设项目中，原始数据往往包含大量来自不同业务系统的 URL 字段，这些 URL 格式不统一、包含冗余参数或大小写混用。ResourceVault 可作为 ETL 管道的前置过滤器，对原始 URL 进行规整化处理，剔除无效协议或非法字符，输出清洗后的标准链接集合供后续分析任务使用。

## 快速开始

以下指令适用于 Linux 与 macOS 环境，Windows 用户可通过 WSL 或 Git Bash 执行。

```bash
# 克隆代码仓库
git clone https://github.com/resourcevault/resourcevault.git
cd resourcevault

# 安装依赖（基于 Python 3.10+）
pip install -r requirements.txt

# 准备输入文件 urls.txt，每行一个 URL，随后执行清洗与输出
python vault.py process --input urls.txt --output report.md
```

上述命令执行完毕后，控制台将打印本次处理的统计信息，包括总链接数、成功分类数、失效链接数及输出文件路径。若需查看详细探测日志，可追加 --verbose 参数。

## 项目结构

```text
resourcevault/
├── vault.py                      # 主入口程序，协调各模块执行流程，负责解析命令行参数
├── requirements.txt              # 生产环境依赖清单，记录所有必需的第三方库及版本约束
├── .resourceignore               # 忽略规则配置文件，定义需要排除的 URL 正则模式与路径前缀
├── .env.example                  # 环境变量示例文件，包含代理设置与探测超时参数模板
├── core/
```

## 常见问题

问：在执行探测步骤时，程序对部分内部域名返回超时错误，但这些域名在浏览器中可正常访问。应如何调整？

答：该现象通常是由于探测模块默认使用严格的超时设置（连接超时 3 秒，读取超时 5 秒）所致。对于响应较慢的内部服务或需经过多层网关的地址，您可以通过 .env 文件中的 PROBE_CONNECT_TIMEOUT 和 PROBE_READ_TIMEOUT 变量分别调大这两个阈值。此外，若目标站点启用了访问控制或需要特定 User-Agent，可在 .env 中配置 PROBE_HEADERS 键值对进行伪装。调整后请重启程序使配置生效。

问：分类器无法识别我自定义的业务路径，输出结果全部归为 "other" 类别，该如何扩展规则？

答：默认分类规则基于 rules/default_patterns.json 中的正则表达式匹配。您可以创建 rules/custom_patterns.json 文件，按照相同的 JSON 结构添加您的业务规则。程序启动时会自动合并两个文件中的规则，且自定义规则的优先级更高。建议您在编写正则时使用在线工具先行验证匹配效果，避免因转义字符错误导致规则失效。若仍需帮助，可参考 docs/config-rules.md 中的调试章节。

问：资源列表中的部分链接包含动态参数（如时间戳或 sessionid），这导致重复条目被误判为不同资源。能否规范化这类 URL？

答：您可以通过配置 .resourceignore 文件中的 normalize_rules 段，定义需要移除的查询参数名称列表。例如添加 "timestamp", "sid", "nonce" 等常见动态参数名称，程序在加载阶段会自动剔除这些键值对后再进行去重与分类。注意该操作仅影响内部处理逻辑，原始 URL 仍会保留在最终输出的原始字段中供追溯。
