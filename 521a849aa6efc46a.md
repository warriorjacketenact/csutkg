# ResourceBridge 媒体资源聚合网关

ResourceBridge 是一个面向技术内容聚合、媒体资源索引与外部链接治理的开源网关工具。它并非一个传统的播放器或下载器，而是一个结构化的外链资源映射与校验系统，专门用于对大批量（千级规模）的外部媒体资源 ID 进行标准化抓取、可用性探测与分类归档。

本项目的核心目标用户是从事数据治理、内容管理系统（CMS）开发、自动化测试脚本编写以及个人知识库构建的技术人员。它解决的核心痛点是：当面对数百个由不同业务线产生的异构资源链接时，如何通过统一的接口规范、稳定的命令行工具以及清晰的目录结构，快速完成这批链接的连通性测试、元数据提取与状态监控，从而避免人工逐条点击验证的低效与遗漏。

ResourceBridge 不依赖特定的图形界面，所有操作均可通过 Shell 脚本与 Python 运行时完成，适合嵌入 CI/CD 流水线或定时任务集群中运行。

## 功能概览

- **批量资源导入与解析**：支持从 CSV、JSON 或纯文本列表批量注入外部资源 URL，自动识别 URL 路径中的资源 ID 与分类标识。

- **HTTP 状态码与可达性探测**：并发异步请求检测每个资源链接的响应状态，支持自定义超时时间与重试策略，输出结构化探测报告。

- **元数据指纹提取**：对可访问的资源页面提取标题、内容摘要、最后修改时间等基础元数据，用于后续检索与展示。

- **分级日志与审计追踪**：完整记录每次扫描任务的开始时间、结束时间、成功/失败计数及详细错误栈，便于运维人员回溯。

- **目录树自动生成器**：依据扫描结果自动在本地生成带注释的 ASCII 目录树结构，方便快速了解当前资源库的分类层次。

- **阈值告警钩子**：当某一批次资源失败率超过预设百分比时，可触发外部 Webhook 或邮件通知，实现被动监控。

## 应用场景

- **数据中台资源校验**：数据团队每周需同步一批来自上游业务方的媒体资源 ID。通过 ResourceBridge 定时运行，可自动验证所有链接是否仍处于有效状态，并将失效链接列表推送至修复工单系统。

- **CMS 迁移前资源盘点**：在将旧版内容管理系统迁移至新架构前，运维人员使用本工具遍历全部历史资源外链，确认哪些链接在迁移后依然可解析，哪些需要做 301 重定向映射。

- **自动化测试用例补充**：质量保障工程师将资源链接集合作为测试数据源，利用本工具提供的 CLI 接口批量验证测试环境与生产环境的配置一致性，确保灰度发布后资源路由无误。

- **个人知识库死链清理**：博客作者或笔记维护者定期导出 Markdown 文档中的所有外链，通过本工具批量检测，自动生成死链修复建议表。

## 快速开始

以下指令适用于 Linux 与 macOS 环境，Windows 用户建议通过 WSL2 或 Git Bash 执行。

```bash
# 克隆项目仓库至本地
git clone https://github.com/resourcebridge/resourcebridge.git
cd resourcebridge

# 安装 Python 虚拟环境与核心依赖
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 运行示例扫描任务（使用内置测试链接列表）
python bridge.py scan --input samples/url_list_sample.txt --output reports/
```

## 项目结构

```

├── bridge.py                  # CLI 入口，解析子命令并调度扫描引擎

├── .env.example               # 环境变量模板（代理、超时、并发数）

│   ├── core/

│   ├── utils/
│   │   ├── logger.py          # 分级日志配置（DEBUG/INFO/ERROR）
│   │   └── validators.py      # URL 规范化与格式校验函数
│   └── hooks/
│       ├── webhook.py         # 失败率告警 Webhook 发送器

│   ├── unit/
│   │   ├── test_fetcher.py    # 异步获取器单元测试
│   │   └── test_parser.py     # 解析器边界条件测试
│   └── integration/
│       └── test_scan_flow.py  # 端到端扫描流程测试
├── samples/
│   ├── url_list_sample.txt    # 示例 URL 列表（用于快速体验）

│   ├── quickstart.md          # 快速入门指南

│   ├── development.md         # 开发环境搭建与贡献规范
```

## 贡献指南

1. 阅读开发文档与代码风格规范
   在提交任何代码前，请先完整阅读 docs/development.md 文档，确保理解本项目的异步编程模型、异常处理约定以及 PEP 8 代码风格要求。

2. 创建功能分支并编写测试用例
   从 main 分支 checkout 出一个新的功能分支，命名格式为 feature/简短描述。对于新增的任何解析器逻辑或 HTTP 行为，必须在 tests/unit 下补充对应的单元测试，且覆盖率不低于 90%。

3. 运行完整测试套件与静态检查

## 常见问题

**Q：扫描大量链接时出现 "Too many open files" 错误，如何解决？**

A：该错误通常由系统文件描述符限制引起。建议在运行前通过 ulimit -n 4096 提高当前会话的限制值，同时可在配置文件或 .env 中调低 max_concurrent 参数（例如从 100 降至 50），以减少同时打开的 socket 数量。对于长期运行的任务，推荐使用分批扫描模式（--batch-size 选项）。

**Q：部分资源链接返回 403 或 429 状态码，扫描任务是否会终止？**

A：不会终止。ResourceBridge 默认将 403 与 429 视为可重试的临时性错误，会依据配置的重试次数（默认为 3 次）进行指数退避重试。若最终仍然失败，该链接会被标记为 "blocked" 或 "rate_limited" 状态，并记录在报告的 failed_list 字段中，扫描任务会继续处理后续链接。如需调整重试策略，请修改配置文件中的 retry_settings 段落。

**Q：扫描结果报告中的 "fingerprint" 字段具体代表什么？**

A：fingerprint 字段是基于资源页面标题、正文前 200 个字符以及最后修改时间拼接后计算的 SHA-256 哈希值。该字段主要用于快速判断同一 URL 在不同时间点扫描时内容是否发生变化，便于内容更新追踪。若页面无法解析出足够的文本内容，该字段将回退为对 URL 路径本身的哈希值。
