# LinkMatrix

LinkMatrix 是一个面向技术团队与独立开发者的外链资源归集与导航系统。该项目将分散于各处的优质技术文档、工具链入口、社区讨论与内部知识库链接统一纳入可检索、可分类、可版本追踪的管理体系，解决团队中“书签杂乱、链接失效、知识孤岛”的普遍痛点。LinkMatrix 不提供内容存储，而是作为外链的中转与组织层，通过结构化元数据为每个资源链接赋予上下文，使技术决策与学习路径可追溯、可复用。

LinkMatrix 的目标用户包括：需要维护团队技术知识库的架构师、负责新人 onboarding 的工程主管、以及希望建立个人技术阅读清单的开发者。项目本身以静态站点形式交付，可部署于任意 Web 服务器或对象存储，无需数据库依赖，所有链接数据以 YAML 文件管理，便于纳入现有 CI/CD 流程。

## 功能概览

- 链接分类与标签系统：每个资源链接可归属多个分类，支持自定义标签，便于按技术栈、难度等级、来源部门等维度筛选。

- 元数据扩展字段：支持记录链接的维护人、添加日期、最后验证时间、关联项目代号，解决链接长期无人维护的问题。

- 全文检索与过滤：基于标题、描述、标签、分类的轻量级搜索，支持多条件组合过滤，快速定位目标资源。

- 链接健康检查：内置链接状态检测脚本，定期输出失效链接报告，标记需要更新的条目。

- 导入导出机制：支持 CSV、JSON、YAML 格式的批量导入导出，兼容主流书签管理工具的数据迁移。

- 访问统计看板：基于访问日志生成链接点击热度排行，辅助识别高频使用的核心资源。

- 权限分级视图：支持公开链接与内部链接分离展示，未登录用户仅可见公开部分，登录用户可查看完整列表。

## 应用场景

1. 团队新人技术 onboarding：新入职开发者在 LinkMatrix 中按“后端基础”“部署规范”“代码评审指南”等分类浏览，可在一周内系统性地接触团队所有关键基础设施与文档入口，大幅减少零散询问。

2. 技术选型调研存档：架构组在评估新中间件时，将候选产品的官方文档、性能测试报告、社区案例、竞品对比文章统一收录至 LinkMatrix 的特定项目目录，后续复盘或向管理层汇报时可一键导出完整链接清单。

3. 个人知识体系构建：开发者将日常阅读的博客、论文、API 参考、视频教程按学习路线分类存放，配合标签记录阅读状态与笔记索引，构建长期可维护的个人技术知识图谱。

4. 离线文档镜像导航：对于部署在内网环境无法直接访问外网的研发环境，LinkMatrix 可配置为内网镜像站点的导航页，列出所有已同步的离线文档入口，解决内网环境下“知道有文档但找不到地址”的问题。

5. 技术分享会材料管理：每次内部技术分享后，演讲者将相关参考资料、延伸阅读链接提交至 LinkMatrix 的活动分类下，形成持续积累的分享档案库，新成员可回看历史分享的全部材料。

## 快速开始

以下步骤适用于 Linux / macOS 开发环境，Windows 用户建议使用 WSL2 或 Git Bash。

```bash
# 克隆仓库
git clone https://github.com/your-org/linkmatrix.git
cd linkmatrix

# 安装依赖（Python 3.9+ 环境）
pip install -r requirements.txt

# 启动开发服务器
python manage.py runserver --host 0.0.0.0 --port 8080
```

部署生产环境时，可使用 `build` 命令生成静态文件，然后通过 nginx 或 S3 托管：`python manage.py build --output ./dist`。

## 安装要求

| 依赖 | 必需 | 说明 |
|------|------|------|
| Python >= 3.9 | 是 | 核心运行环境，低于此版本可能导致语法错误或标准库缺失 |
| PyYAML >= 6.0 | 是 | 用于解析链接元数据配置文件，支持 YAML 1.2 规范 |
| Jinja2 >= 3.1 | 是 | 模板渲染引擎，用于生成静态页面与报告 |
| requests >= 2.28 | 是 | 链接健康检查模块依赖，用于发送 HTTP 探测请求 |
| markdown >= 3.4 | 否 | 仅在渲染包含 Markdown 格式的描述字段时需要，默认使用纯文本 |
| pytest >= 7.0 | 否 | 开发测试依赖，运行单元测试套件时使用 |
| gunicorn >= 20.1 | 否 | 生产环境 WSGI 服务器，非必需但推荐用于高并发场景 |
| watchdog >= 2.0 | 否 | 开发模式下热重载辅助，提升本地调试效率 |

## 贡献指南

1. 查阅 issue 列表与 projects 看板，找到未分配的链接归类、模板优化或健康检查增强任务，在任务下留言表明认领意向，避免重复工作。

2. 从 main 分支创建新的功能分支，分支命名遵循 `feature/描述` 或 `fix/描述` 格式，例如 `feature/add-timeout-config`，确保分支名称简洁体现变更内容。

3. 提交代码前运行本地测试套件 `pytest tests/` 确保所有已有用例通过，新增功能需同步添加对应单元测试，测试覆盖率不低于 80%。

4. 提交 pull request 至 main 分支，PR 描述中附上变更摘要、测试结果截图或日志，以及关联的 issue 编号。若有链接数据变更，需同时更新 `data/links.yaml` 中的 `last_verified` 字段。

5. PR 通过 CI 检查（包括测试、lint、构建）后，至少一位项目维护者进行 code review，review 通过后由维护者 squash merge 并关闭关联 issue。

## 常见问题

Q: LinkMatrix 是否必须使用 Python 3.9 以上版本？能否在 Python 3.8 环境运行？

A: 项目使用了 `dict` 合并运算符 `|` 与 `str.removeprefix` 方法，这两个特性在 Python 3.9 中引入。若强行在 3.8 下运行，会抛出 `SyntaxError`。建议升级 Python 版本，或使用 `pyenv` 管理多版本环境。对于无法升级的环境，可考虑使用 `2to3` 工具自行向下移植，但官方不提供支持。

Q: 链接健康检查会对目标服务器造成压力吗？如何调整检查频率？

A: 健康检查模块默认使用 10 个并发线程，每个请求超时设为 5 秒，且仅对每个链接发送一次 HEAD 请求（若 HEAD 不被支持则降级为 GET 并只读取响应头）。默认每 24 小时执行一次全量检查。如需调整并发数或超时时间，可在 `src/health/checker.py` 中修改 `CONCURRENCY` 与 `TIMEOUT` 常量，或通过环境变量 `LINKMATRIX_CHECK_CONCURRENCY` 与 `LINKMATRIX_CHECK_TIMEOUT` 覆盖。

Q: 如何将 LinkMatrix 部署到内网环境且不暴露外网访问？

A: 构建静态文件后，将 `./dist` 目录复制到内网 Web 服务器根目录即可。由于静态生成阶段已完成所有数据渲染，运行时无需访问外部网络，也无需数据库连接。内网用户通过 HTTP 访问该服务器即可浏览全部链接。若需要动态搜索功能，可将搜索索引文件一并托管，前端 JavaScript 在浏览器端完成过滤，不依赖后端 API。
