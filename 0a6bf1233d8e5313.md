# ResourceLink Hub

ResourceLink Hub 是一个面向技术研究者、内容策展人和开源开发者的高性能外链资源聚合与导航系统。该项目并非传统的网页爬虫或简单的书签管理器，而是一个结构化的资源索引框架，旨在解决海量分散链接的归类、元数据标注、可用性监控与快速检索问题。项目本身不存储资源实体，而是提供一套标准化的外链描述规范（Resource Description Specification, RDS）和配套的验证、查询工具集，帮助团队或个人将零散的 URL 转化为可维护、可共享的知识资产。

目标用户包括需要管理大量技术文档链接的研发团队、运营技术博客或知识库的内容创作者，以及对特定领域（如能源数据、开源镜像、学术预印本）进行系统性跟踪的研究人员。通过 ResourceLink Hub，用户能够以极低的心智成本对数百乃至数千个外链进行批次导入、标签化分类、连通性检查和使用统计，从而避免链接失效、上下文丢失和重复劳动。

## 功能概览

批量资源导入与解析：支持通过命令行或 API 接口批量提交 URL 列表，自动识别协议头、域名路径和查询参数，提取文件名或路径片段作为初始元数据。

基于规则的标签引擎：允许用户自定义正则匹配或路径前缀规则，为链接自动打上分类标签（如 "documentation", "api_reference", "dataset"），支持标签继承与合并。

链接可用性主动探测：内置异步 HTTP 健康检查模块，可配置超时与重试策略，定期对资源列表执行 HEAD 或 GET 请求，记录状态码与响应时间，标记失效或重定向链接。

结构化元数据附件：每个资源条目支持附加键值对元数据，包括但不限于来源项目、收录批次、维护人、最后验证时间，元数据以 JSON 格式存储于独立索引文件中。

多维度检索与过滤：提供基于标签、域名、路径关键词、状态码、更新时间等条件的组合查询接口，支持输出为 JSON、CSV 或 Markdown 表格格式。

批次管理与差异比对：针对第 909/1241 批此类大规模导入场景，系统自动记录每次批次的资源快照，支持批次间的新增、删除与变更比对，输出变更报告。

静态站点生成器集成：内置模板引擎可将当前资源索引渲染为静态 HTML 导航页面，便于部署至 GitHub Pages 或对象存储服务，实现资源目录的公开分享。

## 应用场景

技术文档站点的外链生命周期管理：某开源社区维护了一份包含数百个第三方库文档链接的 wiki 页面，维护者使用 ResourceLink Hub 定期扫描所有链接，自动发现 404 页面和域名过期情况，并在每次社区会议前生成一份可用性报告，极大减少了手动点击验证的时间。

研究课题的文献资源追踪：一位能源领域的研究人员需要跟踪大量国际机构发布的报告与数据集，这些资源的 URL 分散在邮件、笔记和 PDF 中。研究人员将所有链接导入 ResourceLink Hub，按数据来源（IEA、EIA、IRENA）和数据类型（统计年鉴、技术报告、原始数据）进行标注，并通过标签筛选快速定位特定机构的最新数据链接。

企业内部的知识库外链审计：某公司技术文档团队使用 ResourceLink Hub 对公司内部 Confluence 知识库中引用的所有外部链接进行统一审计。系统每周自动运行，将失效链接列表发送至团队邮箱，并标记出引用了非 HTTPS 协议或包含过期域名的链接，辅助团队进行安全与合规整改。

开源项目的依赖文档镜像策展：某开源中间件项目的维护者需要确保其 README 中引用的所有性能对比报告、依赖项源码仓库和 benchmark 工具链接长期有效。维护者将这些链接纳入 ResourceLink Hub 的资源池，配置自定义元数据记录每个链接的用途和最后确认版本，并在每次 release 前运行一次完整检查。

## 快速开始

以下指令适用于 Linux/macOS 环境，Windows 用户可使用 WSL2 或 Git Bash 执行。

```bash
# 克隆项目仓库并进入根目录
git clone https://github.com/resourcelink-hub/resourcelink-hub.git
cd resourcelink-hub

# 安装依赖项（项目使用 Python 3.10+ 与 Poetry 管理）
pip install poetry
poetry install --no-dev

# 执行示例资源导入与健康检查
# 使用内置的示例批次文件（包含 250 条链接）进行功能验证
poetry run rlh-cli import --batch ./samples/batch_909_1241.csv
poetry run rlh-cli check --batch 909 --concurrency 50
poetry run rlh-cli report --batch 909 --format markdown --output ./reports/status_909.md
```

## 项目结构

```
resourcelink-hub/
├── src/                                # 核心源代码目录
│   ├── rlh_cli/                        # 命令行接口主程序
│   │   ├── __init__.py                 # 包初始化与版本声明
│   │   ├── main.py                     # CLI 入口，解析子命令（import/check/report/diff）
│   │   └── commands/                   # 各子命令的具体实现模块
│   │       ├── import_batch.py         # 解析 CSV 并写入索引数据库
│   │       ├── run_health_check.py     # 异步并发探测链接可用性
│   │       └── generate_report.py      # 汇总状态并输出指定格式
│   ├── core/                           # 业务逻辑与数据模型
│   │   ├── models.py                   # Pydantic 定义的 Resource, Batch, Metadata 类
│   │   ├── indexer.py                  # 资源索引的增删改查与版本管理
│   │   ├── checker.py                  # 封装 aiohttp 的健康检查会话与策略
│   │   └── rule_engine.py              # 基于路径和域名的标签自动匹配逻辑
│   └── utils/                          # 通用工具函数
│       ├── file_io.py                  # 读写 CSV、JSON、Markdown 文件的辅助
│       ├── logger.py                   # 带颜色和等级过滤的日志配置
│       └── validator.py                # URL 规范化与格式校验函数
├── config/                             # 项目配置文件目录
│   ├── default.toml                    # 默认参数（并发数、超时、重试、输出路径）
│   ├── rules.yaml                      # 用户可编辑的标签规则示例（正则匹配 -> 标签列表）
│   └── schema.json                     # 资源元数据附加字段的 JSON Schema 约束
├── data/                               # 数据持久化存储（不纳入 Git 仓库，运行时生成）
│   ├── indices/                        # 资源主索引文件，按批次分目录
│   │   └── batch_909/                  # 第 909 批次的完整索引快照
│   │       ├── resources.json          # 资源列表与元数据
│   │       └── status.json             # 最近一次健康检查的汇总结果
│   ├── reports/                        # 生成的报告输出目录
│   │   └── status_909.md               # 示例报告文件
```

## 常见问题

问：系统能处理多少条链接？是否支持超过 250 条的大批次？
答：系统的单批次设计容量为 2000 条链接，超过此数量建议拆分为多个批次以避免内存占用过高。对于第 909 批次的 250 条链接，默认并发数 50 可在约 2 至 5 秒内完成首次健康检查（视网络状况而定）。若需处理数万条链接，可调整 config/default.toml 中的 chunk_size 参数启用分块处理。

问：如果目标服务器拒绝 HEAD 请求或响应缓慢，检查机制如何处理？
答：健康检查模块首先尝试发送 HEAD 请求，若服务器返回 405 Method Not Allowed 或超时，则自动降级为 GET 请求并只读取响应头部分，不下载完整响应体。超时时间（默认为 10 秒）和重试次数（默认为 2 次）均可通过配置文件或命令行参数调整。对于明显不可达的域名，系统会记录为 FAILED 状态并跳过后续重试。

问：项目是否依赖外部的数据库或缓存服务？
答：不依赖。所有索引数据和缓存均以 JSON 和 SQLite 文件的形式存储在 data/ 目录下，完全自包含。这种设计保证了项目的可移植性和低运维成本，用户只需确保 data/ 目录具有读写权限即可。若需要进行团队协作，可将 data/ 目录置于同步网盘或 Git LFS 中，但需注意避免冲突。
