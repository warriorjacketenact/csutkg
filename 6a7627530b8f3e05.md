# WebLink Navigator

WebLink Navigator 是一个面向技术研究、数据挖掘和内容聚合场景的轻量级外链资源汇总与导航系统。该项目定位于帮助开发者、数据分析师和技术内容创作者高效组织、检索和访问分散在多个域名下的结构化资源链接，尤其适用于处理大批量、规律性强的 URL 集合场景。

项目核心解决传统书签管理或散落文档在面临数百个关联链接时出现的检索效率低下、分类混乱和无法批量操作的问题。WebLink Navigator 提供基于规则的分组引擎、元数据提取工具和静态站点生成能力，可将原始链接列表转化为具有多维筛选和全文检索能力的可维护知识库。当前批次为第 892/1241 批，共处理 250 个资源链接，所有链接均来源于 hslcgj.com 域下的电视剧相关资源路径，项目已内置该批次的完整索引。

## 功能概览

**批量链接导入与解析**：支持从纯文本、CSV 和 JSON 格式批量导入 URL 列表，自动识别 URL 结构中的路径层级、查询参数和文件名模式，提取资源标识符与类型标签。

**多维度分类引擎**：基于 URL 路径模式、域名特征和自定义规则集对链接进行自动归类，支持用户定义分类标签并持久化存储分类方案，便于后续批次复用。

**全文检索与过滤**：为所有链接资源建立倒排索引，支持按资源 ID、路径关键词、分类标签和导入批次进行组合筛选，检索结果可按相关度或导入时间排序。

**资源状态监控**：定期对已收录链接进行可用性检查，标记失效或重定向链接，生成状态报告并支持导出异常链接列表，便于维护者及时更新。

**静态站点导出**：将当前批次及历史批次的全部链接资源生成为静态 HTML 站点，包含分类索引页、标签云和按字母排序的链接总表，可直接部署到任意 Web 服务器或对象存储服务。

**元数据扩展字段**：每条链接支持附加标题、描述、备注和自定义键值对元数据，元数据独立存储于 SQLite 数据库中，与 URL 引用分离，避免重复爬取造成的资源浪费。

## 应用场景

**技术文档站外链接归档**：技术博客或开源项目文档中常引用大量外部资源链接，使用 WebLink Navigator 可对这些链接进行统一管理、定期检查可用性，并在文档站点中嵌入动态链接列表，避免文档发布后链接失效无法追溯。

**数据挖掘项目资源索引**：数据采集或爬虫项目中需要维护目标 URL 清单，本系统可将不同批次、不同来源的 URL 整合为单一索引库，支持按采集状态、域名或时间范围快速筛选特定子集，提升数据管道的可维护性。

**内容聚合站点的底层数据源**：构建影视、资讯或学术资源聚合站点时，需批量管理外链数据。WebLink Navigator 的静态站点导出功能可直接生成面向最终用户的内容导航页，减少从数据到展示的中间开发成本。

**开源项目 README 资源链接维护**：开源项目维护者可将项目依赖的教程、工具、参考文档等外链统一录入系统，自动生成结构化的资源列表章节，避免手写 Markdown 时的格式不一致和遗漏问题。

## 快速开始

以下步骤将指导您在本地环境完成 WebLink Navigator 的克隆、安装和初次运行，导入当前批次的 250 个链接资源。

```bash
# 克隆项目仓库
git clone https://github.com/weblink-navigator/weblink-navigator.git
cd weblink-navigator

# 安装依赖（使用 Python 虚拟环境）
python -m venv venv
source venv/bin/activate  # Windows 下使用 venv\Scripts\activate
pip install -r requirements.txt

# 运行初始导入流程，导入第 892/1241 批次的链接数据
python cli.py import --batch 892 --source data/batch_892_urls.txt
python cli.py index --batch 892
python cli.py serve  # 启动本地 Web 服务，默认端口 8080
```

## 项目结构

项目采用分层架构设计，核心模块位于 src 目录下，数据存储与配置分离，测试与文档独立维护。以下为项目主要目录及文件说明。

```
weblink-navigator/
├── src/                                # 核心源代码目录
│   ├── core/                           # 核心业务逻辑层
│   │   ├── importer.py                 # 批量导入引擎，处理 CSV/JSON/TXT 解析
│   │   ├── indexer.py                  # 倒排索引构建与检索接口
│   │   ├── classifier.py               # 基于规则的多维度分类器
│   │   └── models.py                   # 数据模型定义（Link, Batch, Metadata）
│   ├── storage/                        # 数据持久化层
│   │   ├── database.py                 # SQLite 连接管理与基础 CRUD
│   │   ├── migrations/                 # 数据库版本迁移脚本
│   │   └── repositories/               # 各实体的 Repository 实现
│   ├── monitor/                        # 资源监控模块
│   │   ├── checker.py                  # HTTP 可用性检查与超时重试策略
│   │   └── reporter.py                 # 异常链接报告生成器
│   ├── export/                         # 导出与渲染模块
│   │   ├── static_generator.py         # 静态站点生成主流程
│   │   ├── markdown_renderer.py        # Markdown 格式资源列表导出
│   │   └── templates/                  # Jinja2 HTML 模板目录
```

## 贡献指南

WebLink Navigator 欢迎社区贡献，无论是新增功能、修复缺陷还是改进文档，均可按照以下流程参与项目协作。

第一，查阅 issue 列表与项目看板。访问 GitHub Issues 页面查看当前待处理的任务和已知缺陷，选择未被认领且与自身技能匹配的条目。若计划新增功能，建议先创建讨论类 issue 与维护者沟通设计思路，避免开发方向偏离项目规划。

第二，派生项目仓库并创建功能分支。将主仓库派生至个人账号下，克隆至本地后基于 main 分支创建新的功能分支，分支命名遵循 feat/功能描述 或 fix/问题描述 的格式。本地开发时确保代码通过 black 格式化检查并补充对应的单元测试。

第三，提交变更并签署开发者原产地证书。提交信息采用约定式提交格式，包含类型和简短描述。所有贡献者需在 Pull Request 描述中确认已阅读并同意开发者原产地证书，即保证所提交代码为原创或已获适当授权。

第四，发起 Pull Request 并参与代码评审。将功能分支推送至派生仓库后向主仓库的 main 分支发起 Pull Request，填写 PR 模板中的变更摘要、测试覆盖和影响范围等信息。评审过程中及时响应评审意见，通过所有 CI 检查后由维护者合并。

## 常见问题

**问：导入的链接数量较多时，系统性能表现如何？**

系统设计目标为单批次支持 5000 条以内的链接导入与索引构建，在当前批次 250 条规模下，导入与索引全流程耗时不超过 3 秒。数据库使用 SQLite 并针对 url_hash 和 batch_id 建立了复合索引，全文检索查询响应时间控制在 50 毫秒以内。若单批次超过 5000 条，建议分批导入并使用 --batch 参数区分批次编号。

**问：静态站点导出后的资源链接如何保持与数据库同步？**

静态站点在导出时刻生成 HTML 文件，其中的链接列表为快照数据，不会自动随数据库更新而刷新。若需保持同步，可配置定时任务周期执行导出命令，或通过 Webhook 在每次数据变更后触发增量导出。动态更新场景建议使用项目内置的 serve 命令启动本地 Web 服务，该模式直接从数据库读取实时数据。

**问：如何添加自定义分类规则而不修改核心代码？**

分类规则采用 JSON 配置文件存储，位于 data/classifications.json。规则格式为规则名称与正则表达式匹配组的映射，系统启动时自动加载。用户可在此文件中新增或调整规则，无需重启服务即可通过 reload 子命令热加载配置。详细规则语法参考 docs/data-import.md 中的分类器配置章节。
