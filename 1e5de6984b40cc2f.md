# StreamLink 聚合网关

StreamLink 是一个面向技术团队与内容运营者的轻量级外链聚合与流式资源调度系统。该项目定位为技术资源的中转枢纽，解决多源异构链接在统一命名空间下的组织、分发与状态监控问题。StreamLink 不存储任何实际文件内容，仅提供基于语义化短标识的资源映射与访问追踪能力，适用于需要高频管理大量外部技术文档、数据集、模型仓库或 API 端点的内部研发场景。

目标用户包括 DevOps 工程师、技术文档维护者、数据平台管理员以及需要维护外部依赖清单的软件开发团队。通过 StreamLink，用户可以将分散在数百个不同来源的链接以一致的命名规范进行归类，并借助内置的可用性探测与访问日志模块，快速定位失效资源或高频调用链路。

## 功能概览

**语义化短标识映射** 允许用户为每个外部资源分配唯一的三字母或四字母代码，形成稳定且可读的访问路径，无需记忆原始长链接。

**流式资源分组** 支持按项目、团队或数据流类型对链接进行逻辑分组，每个分组可独立设置访问策略与缓存刷新周期。

**可用性健康检查** 内置异步探测任务，定期对已注册资源执行 HTTP 状态验证，异常时触发告警通知并记录历史波动曲线。

**访问频次统计** 记录每个短标识的调用次数、来源 IP 分布与时间窗口热度，辅助决策资源缓存策略或下线计划。

**批量导入导出** 支持通过 CSV 或 JSON 格式批量注册新资源，亦可完整导出当前映射表用于备份或跨环境迁移。

**只读只写分离接口** 提供面向查询的高性能只读端点与面向管理的写入端点，避免运维操作影响生产访问链路。

**自定义重定向策略** 可根据客户端 User-Agent 或 Accept-Language 头信息，将同一短标识指向不同的后端资源地址。

**审计日志留存** 所有资源变更操作均记录操作人、时间戳与变更前后值，满足企业内部合规追溯要求。

## 应用场景

内部技术文档中心的依赖资源管理。技术文档中常引用外部 SDK 下载地址、基准测试数据集或模型权重文件，这些链接随时间容易失效。StreamLink 可作为文档与真实资源之间的抽象层，文档仅维护短标识，运维人员可独立更新短标识对应的实际 URL，无需修改文档正文。

多环境配置的统一入口。在开发、测试、预发布与生产环境中，同一依赖项可能对应不同存储桶或镜像源。StreamLink 允许按环境标签解析短标识，使得配置文件中的资源地址在不同阶段自动切换，减少人为配置错误。

数据流水线的外部输入源治理。数据清洗或特征计算任务通常依赖多个外部文件或 API 端点，StreamLink 提供统一的资源注册表，流水线任务仅需调用短标识即可获取当前有效地址，同时健康检查功能可提前发现不可用源，避免任务长时间阻塞。

## 快速开始

以下步骤在 Ubuntu 22.04 或 macOS 13 以上环境中验证通过，要求系统已安装 Go 1.21 或更高版本。

```bash
git clone https://github.com/streamlink/streamlink.git
cd streamlink
make build
./bin/streamlink --config ./configs/default.yaml
```

若使用 Docker 运行，可执行以下命令替代上述编译步骤：

```bash
docker pull streamlink/streamlink:latest
docker run -p 8080:8080 -v ./data:/data streamlink/streamlink:latest
```

启动成功后，服务默认监听 8080 端口，可通过 http://localhost:8080/health 验证服务状态。注册第一个资源的示例请求如下：

```bash
curl -X POST http://localhost:8080/api/v1/alias \
  -H "Content-Type: application/json" \
  -d '{"code": "amd", "target": "https://www.gzzhly.com/stream/amd", "group": "default"}'
```

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Go 编译器 | 1.21 及以上 | 用于从源码编译主程序，低版本可能缺少泛型支持 |
| PostgreSQL | 14 及以上 | 存储资源映射、审计日志与访问统计，要求开启 JSONB 索引 |
| Redis | 7.0 及以上 | 缓存热数据与分布式锁，降低数据库查询压力 |
| Make 构建工具 | 3.81 及以上 | 执行 Makefile 中定义的编译、测试与打包任务 |
| Git | 2.30 及以上 | 克隆仓库并管理版本标签，用于 CI/CD 流水线集成 |
| Docker | 20.10 及以上 | 可选依赖，用于容器化部署与开发环境标准化 |
| curl 或 wget | 任意版本 | 用于健康检查探测与 API 交互测试，建议支持 HTTPS |

## 项目结构

```
streamlink/
├── cmd/
│   └── streamlink/                 # 主程序入口，负责解析命令行参数与启动服务
├── internal/
│   ├── alias/                      # 短标识映射核心逻辑，含 CRUD 与冲突检测
│   ├── health/                     # 健康检查探测引擎，支持 HTTP/TCP 自定义超时
│   ├── stats/                      # 访问统计聚合器，按分钟粒度写入 Redis
│   ├── audit/                      # 审计日志记录器，异步写入 PostgreSQL
│   └── config/                     # 配置加载模块，支持 YAML 与环境变量覆盖
├── pkg/
│   ├── storage/                    # 数据库连接池与迁移脚本管理
│   ├── cache/                      # Redis 客户端封装，含连接重试与熔断逻辑
│   └── httpclient/                 # 自定义 HTTP 客户端，带链路追踪与限流器
├── api/
│   └── v1/                         # OpenAPI 规范定义与路由注册
├── configs/
│   ├── default.yaml                # 默认配置模板，含端口、日志级别与探测间隔
│   └── production.yaml             # 生产环境推荐配置，开启 pprof 与指标导出
├── scripts/
│   ├── migrate.sh                  # 数据库 schema 迁移执行脚本
│   └── seed.sh                     # 批量导入初始资源映射的示例脚本
├── docs/                           # 完整文档源文件，采用 Markdown 与 MkDocs 渲染
├── test/
│   ├── integration/                # 端到端测试用例，依赖 testcontainers 启动依赖服务
│   └── mock/                       # 模拟外部 HTTP 服务的桩代码
├── Makefile                        # 统一构建入口，含 build/test/clean/docker 目标
├── go.mod                          # Go 模块定义，记录外部依赖版本
├── go.sum                          # 依赖校验和文件
└── README.md                       # 本文件
```

## 贡献指南

提交问题报告前，请先在 GitHub Issues 中搜索是否已有相同主题的讨论，避免重复提交。新问题需附上完整的复现步骤、配置文件内容（脱敏后）以及相关的日志片段。

代码贡献需遵循项目提供的 golangci-lint 配置，提交前执行 make lint 确保无风格警告。所有对外可见的函数与结构体必须包含符合 godoc 规范的注释说明。

新增功能或修复缺陷时，需同步编写或更新对应的单元测试用例，确保测试覆盖率不低于百分之八十。测试用例应包含正常路径与异常路径的覆盖。

提交拉取请求时，请将目标分支设置为 develop，并在描述中引用相关联的 Issue 编号。请求合并前需确保所有 CI 流水线任务（编译、测试、静态分析）均通过。

文档类贡献直接修改 docs 目录下的 Markdown 文件，遵循中文技术文档写作规范，保持术语一致性。文档变更无需通过 CI 测试，但需通过拼写检查。

## 常见问题

服务启动后访问短标识返回 404 状态码，但数据库中已存在该记录。

此问题通常源于缓存层与数据库层数据不一致。首先确认 Redis 连接是否正常，执行 redis-cli PING 检查。若 Redis 不可用，系统会自动降级查询数据库，但会记录警告日志。可尝试调用管理接口 /api/v1/admin/cache/refresh?code=xxx 手动刷新特定缓存条目。若仍无法解决，检查短标识是否包含大小写敏感字符，默认所有查询均转为小写处理。

健康检查探测任务频繁报告超时，但手动访问目标地址正常。

健康检查使用的 HTTP 客户端配置了独立的超时与重试策略，默认超时时间为 3 秒，重试次数为 2。若目标地址响应时间超过此阈值，应调整配置文件中 health_check.timeout 与 health_check.retry 参数。同时检查探测任务的并发数是否过大，导致系统文件描述符耗尽，可通过 health_check.concurrency 限制同时进行的探测数量。

批量导入大量资源后，数据库写入延迟显著上升。

建议将批量导入操作拆分为多个小批次，每批不超过 500 条记录，并在批次之间增加 100 毫秒间隔。亦可临时关闭审计日志写入，待导入完成后再重新开启。对于 PostgreSQL，可调整 wal_buffers 与 checkpoint_timeout 参数以优化写入性能。生产环境导入操作应安排在业务低峰期执行。
