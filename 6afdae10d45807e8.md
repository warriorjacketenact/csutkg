# ResourceHub

ResourceHub 是一个面向开发者与技术研究者的结构化技术资源聚合与导航系统。该项目定位于对分散在网络各处的优质技术文档、工具链、学习材料及社区讨论进行系统性收录与分类管理，解决技术从业者在信息检索过程中面临的海量数据筛选效率低下、优质资源分散难以追踪、技术决策缺乏可靠参考依据等核心问题。ResourceHub 并非简单的书签集合，而是一个带有元数据标签、质量评估与版本追踪能力的资源索引层，适用于个人知识体系建设与团队技术标准制定等场景。

## 功能概览

**多维度资源分类体系** 内置超过二十个技术领域分类标签，涵盖编程语言、框架生态、基础设施、算法理论、工程实践等方向，支持按技术栈版本与成熟度进行筛选。

**自动化健康度检测** 定期对收录的 URL 进行可访问性与响应时间检测，自动标记失效链接与性能劣化资源，保障索引列表的时效性与可靠性。

**结构化元数据编辑** 每个资源条目均支持维护包括但不限于技术栈标签、适用版本、推荐指数、收录理由、关联资源等结构化字段，便于团队内部知识传承。

**快速全文检索与过滤** 基于倒排索引实现毫秒级资源标题、描述、标签与内容的全文检索，支持多条件组合过滤，显著降低信息定位时间。

**资源版本追踪与变更日志** 记录每个资源条目的新增、更新与状态变更历史，支持回溯特定时间点的资源列表状态，满足审计与项目管理需求。

**批量导入与导出接口** 提供标准 JSON 与 CSV 格式的批量数据导入导出功能，支持与其他知识管理工具（如 Notion、Obsidian）进行数据同步与迁移。

**团队协作与权限管理** 支持多用户环境下的资源编辑权限划分，可设定管理员、编辑者与只读观察员三种角色，适用于企业技术文档团队与开源社区维护场景。

## 应用场景

技术团队内部知识库建设：技术负责人可使用 ResourceHub 整理团队日常开发中依赖的第三方库文档、内部工具地址、代码规范参考与故障排查手册，形成统一的技术入口，降低新成员上手成本并减少重复性技术决策讨论。

个人开发者学习路径规划：开发者可围绕自身技术成长目标，利用 ResourceHub 收集优质教程、API 参考、开源项目示例与性能调优案例，通过标签体系构建个性化学习图谱，实现从碎片化浏览到系统性输入的转变。

开源项目依赖资源管理：开源项目维护者可将项目依赖的文档站点、构建工具链、持续集成配置参考与社区讨论帖集中托管于 ResourceHub，方便贡献者快速获取上下文信息，提升协作效率。

技术选型评估支持：架构师在进行中间件、框架或云服务选型时，利用 ResourceHub 汇总各类产品的官方文档、性能测试报告、社区评价与迁移案例，形成可横向对比的信息面板，辅助客观决策。

## 快速开始

以下指令适用于 Linux 与 macOS 环境，Windows 用户建议通过 WSL2 或 Git Bash 执行。

```bash
# 克隆项目仓库至本地
git clone https://github.com/resourcehub/resourcehub.git

# 进入项目根目录并安装依赖
cd resourcehub
pip install -r requirements.txt

# 初始化本地数据库并启动开发服务
python manage.py migrate
python manage.py runserver --host 0.0.0.0 --port 8080
```

执行完成后，访问 http://localhost:8080 即可进入 ResourceHub 仪表板。默认管理员账户为 admin，初始密码在首次启动时由控制台输出，请及时修改。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---------|---------|------|
| Python | 3.10 及以上 | 核心运行时环境，推荐使用 3.11 或 3.12 以获得性能优化 |
| Node.js | 18.x LTS 或 20.x LTS | 仅用于前端资产构建与静态资源编译，不参与运行时逻辑 |
| PostgreSQL | 14.x 及以上 | 主数据库，存储资源元数据、用户信息与操作日志 |
| Redis | 6.x 及以上 | 缓存层与会话存储，用于提升查询响应速度与分布式锁管理 |
| Elasticsearch | 7.x 或 8.x | 可选组件，用于全文检索增强，若未安装系统将回退至 SQL 模糊匹配 |
| Nginx | 1.20 及以上 | 生产环境推荐使用的反向代理与静态文件服务组件 |
| Docker | 20.10 及以上 | 容器化部署方式下的必要依赖，开发环境亦可使用 SQLite 替代 |

## 贡献指南

我们欢迎并感谢所有形式的贡献，包括但不限于新增资源索引、修复文档错误、提出功能建议与代码实现。请遵循以下步骤参与贡献。

第一，查阅 issue 列表与项目看板，确认当前正在进行的工作与待办事项。建议在着手较大改动前先创建 issue 与维护团队沟通，避免重复劳动或方向偏差。

第二，从主分支 fork 项目至个人仓库，并基于 develop 分支创建特性分支。分支命名建议采用 feat/功能描述 或 fix/问题描述 格式，便于自动化流程识别。

第三，完成代码或文档改动后，请确保通过全部单元测试与代码风格检查。提交信息请遵循 Conventional Commits 规范，使用 feat、fix、docs、style 等前缀，并附上清晰简洁的描述。

## 常见问题

问：ResourceHub 与普通浏览器书签或 Notion 数据库有何本质区别？

答：ResourceHub 的核心定位在于结构化与自动化。它不仅仅存储 URL 字符串，而是为每个资源维护了包含技术栈、版本适配、收录理由、健康状态、关联关系在内的完整元数据模型。同时内置的健康检测与版本追踪能力是静态书签与通用笔记工具不具备的，这使得 ResourceHub 特别适合需要长期维护技术决策记录与团队标准参考的严肃场景。

问：系统是否支持私有化部署与离线环境使用？

答：完全支持。ResourceHub 从设计之初即考虑企业内网与离线环境需求。全部依赖组件均可内网镜像化，数据库与检索服务支持无外网运行。前端静态资源完全本地化加载，不依赖任何外部 CDN 或公共 API 服务。唯一需要注意的是，健康检测功能在离线环境下仅能检测内网可访问资源，对外网资源的检测将被跳过并标记为不可达。

问：如何将现有的大量书签或资源列表批量导入 ResourceHub？

答：系统提供了两种批量导入方式。一是通过管理后台的 CSV 上传界面，按照模板格式整理数据后直接导入；二是使用源码目录 src/scripts/batch_import.py 脚本，支持从标准 JSON、浏览器书签 HTML 导出文件、Markdown 列表等多种格式进行解析转换。具体操作步骤与字段映射说明请参考管理手册中的批量操作章节。
