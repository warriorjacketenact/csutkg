# LinkCatalyst

LinkCatalyst 是一个面向技术研究、数据挖掘与互联网档案馆维护场景的高密度外链资源归集与结构化整理工具。该项目不对任何外部资源的内容进行二次存储或代理转发，仅提供原始链接的批量导入、分类标注、有效性预检与按批次导出能力。项目定位为离线外链治理工作流中的标准化输入中间件，适用于需要定期将大批量分散链接纳入统一目录体系的技术运维人员、开源情报分析研究者以及静态站点内容聚合器开发者。

LinkCatalyst 不提供爬虫、索引或全文检索功能，专注于解决外链清单的版本管理、元数据补全与合规性标记问题。项目内置基于文件系统的轻量级目录结构模板，支持自定义标签注入与批次追溯，能够显著降低维护数百至数千条外部链接时的认知负担与人工遗漏风险。

## 功能概览

- **批量链接导入与去重** 支持按行或按 JSON 数组格式批量注入原始 URL，自动识别协议头与域名主体，剔除重复条目并生成导入报告。
- **批次元数据绑定** 每一批链接均可绑定批次编号、导入时间、来源描述与自定义标签，便于后续按批次进行整体归档或迁移。
- **链接协议与格式规整检测** 自动识别裸域名、缺失协议头、大小写混用以及尾部斜杠不一致等常见格式问题，输出格式审查警告日志。
- **分类子目录自动映射** 根据链接域名或路径前缀特征，将条目自动归入预设的分类子目录（如 /domains、/paths、/misc），降低人工分类成本。
- **结构化清单导出** 支持将当前批次的全部链接导出为纯文本列表、Markdown 无序列表或 CSV 格式，供下游静态站点生成器或监控脚本使用。
- **变更追踪与差异对比** 记录每次导入操作的新增、删除与变更条目数，支持对比两个批次之间的链接差异，便于审查外链流动情况。
- **本地轻量运行** 完全基于 Node.js 或 Python 运行时，无需数据库或外部网络依赖，所有元数据存储于本地 JSON 与文本文件中。

## 应用场景

- **技术博客外链附录整理** 技术作者在撰写年度资源汇总文章时，可使用 LinkCatalyst 对数百条参考链接进行去重、分类与格式检查，确保最终发布的链接列表无断裂或重复项。
- **开源项目 README 资源章节维护** 开源项目维护者需要定期更新文档中的外部链接列表时，可通过 LinkCatalyst 管理不同批次的外部资源链接，避免手动编辑带来的遗漏与格式错误。
- **互联网档案馆批量提交前预处理** 在向互联网档案馆或第三方网页存档服务提交批量 URL 之前，使用 LinkCatalyst 对链接进行协议统一与基础可达性预检，提高提交成功率。
- **安全情报链接聚合** 安全研究人员将每日收集的威胁情报域名或报告链接通过 LinkCatalyst 集中管理，按日期与来源打标，生成可追溯的链接数据集。
- **静态网站生成器数据源准备** 静态站点开发者可将 LinkCatalyst 导出的 CSV 或 JSON 链接清单直接作为 Hugo、Jekyll 或 Gatsby 的数据文件，自动渲染为友情链接或推荐阅读页面。

## 快速开始

以下命令演示了如何通过 Git 克隆项目仓库、安装依赖并执行一次完整的链接导入与导出流程。

```bash
git clone https://github.com/your-org/linkcatalyst.git
cd linkcatalyst

# 使用 npm 安装 Node.js 依赖（如使用 Python 版本，请参考 docs/python-setup.md）
npm install

# 将原始链接列表放入 data/raw/batch_168.txt（每行一个 URL）
# 执行导入与格式检查
npm run import -- --batch 168 --file data/raw/batch_168.txt

# 执行分类映射与导出
npm run export -- --batch 168 --format markdown --output docs/links_batch_168.md
```

## 项目结构

```
linkcatalyst/
├── bin/                                 # 命令行入口脚本
│   └── linkcatalyst.js                  # 主 CLI 程序，解析参数并调用核心模块
├── config/                              # 项目配置文件目录
│   ├── default.json                     # 默认分类规则与导出格式配置
│   └── custom-tags.yaml                 # 用户自定义标签映射文件示例
├── data/                                # 数据存储目录（所有用户数据存放于此）
│   ├── raw/                             # 原始导入文件存放区
│   │   └── batch_168.txt                # 第 168 批原始链接列表文件
│   ├── parsed/                          # 解析后结构化元数据缓存
│   │   └── batch_168.meta.json          # 批次元数据（去重、错误标记等）
│   └── exports/                         # 导出文件输出目录
│       └── batch_168_links.md           # 最终生成的 Markdown 链接清单
├── lib/                                 # 核心逻辑库
│   ├── importer.js                      # 链接导入、去重与格式解析模块
│   ├── classifier.js                    # 基于域名与路径的分类映射引擎
│   ├── validator.js                     # 协议头与格式合规性检查器
│   ├── differ.js                        # 批次间差异对比算法实现
│   └── exporter.js                      # 多格式导出生成器（Markdown / CSV / JSON）
├── test/                                # 单元测试与集成测试脚本
│   ├── importer.test.js                 # 导入模块测试用例
│   ├── validator.test.js                # 格式验证模块测试
│   └── fixtures/                        # 测试固定样本数据
│       └── sample_links.txt             # 用于单元测试的链接样本
├── docs/                                # 用户与开发者文档
│   ├── user-guide.md                    # 用户完整操作手册
│   ├── batch-management.md              # 批次管理策略说明
│   ├── format-spec.md                   # 链接格式规范与错误码参考
│   └── development.md                   # 开发者贡献指南与环境搭建说明
├── package.json                         # Node.js 项目依赖与脚本定义
├── README.md                            # 项目首页文档（当前文件）
└── LICENSE                              # MIT 许可证文本
```

## 常见问题

**问：LinkCatalyst 是否会主动访问或爬取列表中外部链接的内容？**

答：不会。LinkCatalyst 仅在导入阶段对链接格式进行静态字符串解析，不会发起任何 HTTP/HTTPS 网络请求去验证链接的可达性或内容类型。所有操作均在本地离线完成，项目本身不构成网络爬虫或数据采集工具。

**问：如何迁移已有批次数据到新版本的 LinkCatalyst？**

答：项目始终保持 data/ 目录结构向后兼容。升级版本时，请先备份整个 data/ 目录，然后在新版本代码中直接运行 npm run migrate 命令，该脚本会自动检测旧版元数据格式并执行必要的字段迁移。迁移完成后，所有历史批次仍然可以通过现有导出命令正常访问。

**问：能否在 Windows 系统上使用 LinkCatalyst？**

答：可以。LinkCatalyst 基于 Node.js 编写，运行时不依赖 Unix 特有的系统调用。Windows 用户建议使用 Git Bash 或 Windows Terminal 配合 PowerShell 执行命令，文件路径请使用正斜杠 / 或双反斜杠 \\ 以避免转义问题。如果遇到文件权限错误，请确保项目目录具有写入权限。
