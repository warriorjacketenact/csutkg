# Zohao Resource Aggregator

Zohao Resource Aggregator 是一个面向技术研究者、开发者和信息分析人员的外链资源汇总工具。该项目通过对海量外部分布式资源链接进行结构化整理与分类索引，帮助用户快速定位特定领域的技术文档、数据快照、历史存档及参考材料。

本项目定位为技术资源导航层，不对链接指向的具体内容进行修改、转存或代理。其核心价值在于提供稳定、可维护的链接清单，并通过元数据分类体系降低用户在海量信息中的筛选成本。目标用户包括需要频繁查阅外部参考资料的系统架构师、运维工程师、算法研究员以及开源项目维护者。

## 功能概览

自动分类索引：根据链接路径中的分类标识符自动将资源归入预设的技术领域类别，支持用户按类别浏览全部外链资源。

链接状态检测：周期性对收录的链接进行可用性检查，标记异常链接并生成状态报告，确保资源清单的可访问性。

多维度检索：支持按资源编号、分类代码、发布时间范围以及关键词组合进行精确筛选，满足复杂查询需求。

批量导入导出：提供标准化的链接清单导入导出接口，支持 CSV 与 JSON 格式，便于与其他数据管道系统集成。

元数据提取：对每个链接自动提取页面标题、内容摘要、文件类型及最后修改时间，增强链接信息的可读性。

访问统计看板：内置轻量级访问计数模块，按日、周、月维度统计各资源的被访问频率，辅助用户识别高频使用资源。

标签体系管理：支持用户自定义标签并绑定至特定链接，构建个性化分类树，适用于团队协作场景下的知识组织。

## 应用场景

技术文档归档与查阅：开发团队可将项目开发过程中参考的外部技术规范、API 文档链接统一纳入本系统，通过分类检索快速回溯历史参考材料，减少重复搜索时间。

数据源备查与溯源：数据分析师在处理外部数据集时，可将数据来源链接集中存储，便于后续数据质量审查或溯源验证，确保分析流程的可重复性。

开源项目外部依赖索引：开源项目维护者可将项目依赖的第三方资源、镜像站点、补丁文件链接纳入清单，在项目迁移或环境重建时能够快速获取所有外部依赖。

安全公告与漏洞库跟踪：安全研究人员可将各厂商安全公告、漏洞数据库链接统一归类，通过链接状态检测功能及时发现失效或变更的公告页面，保持情报渠道的持续有效。

## 快速开始

以下步骤适用于 Linux 及 macOS 环境，Windows 用户可使用 WSL 或 Git Bash 执行。

```bash
git clone https://github.com/zohao/resource-aggregator.git
cd resource-aggregator
./scripts/install.sh
./scripts/start.sh
```

若使用 Docker 运行，可执行以下命令：

```bash
docker build -t zohao-aggregator .
docker run -p 8080:8080 -v ./data:/app/data zohao-aggregator
```

## 安装要求

| Python 3.9 及以上 | 是 | 核心运行环境，用于执行链接解析、分类索引与状态检测逻辑 |

| Node.js 16 及以上 | 否 | 仅当需要构建前端管理面板时依赖，后端服务可不安装 |
| Docker 20.10 及以上 | 否 | 容器化部署方式的可选依赖，便于环境隔离与快速交付 |
| make 4.0 及以上 | 否 | 用于执行自动化构建脚本及测试套件，开发环境推荐安装 |

## 常见问题

问：链接状态检测显示失败，但浏览器中可以正常访问该链接，如何解决？

答：此种情况通常是由于目标服务器对 User-Agent 或请求头有特定要求所致。请检查 conf/aggregator.yaml 中的 http_headers 配置段，调整检测请求的 User-Agent 字段。若目标站点有反爬机制，可在配置中增加检测间隔时间，避免过于频繁的请求被拦截。另外，部分站点对 HEAD 请求不支持，检测模块会自动回退至 GET 请求并仅读取响应头，该行为可通过 config 中的 method_fallback 开关控制。

问：导入大量链接时系统响应变慢，应如何优化？

答：建议首先检查 SQLite 数据库的 journal_mode 是否设置为 WAL，该模式可显著提升并发写入性能。对于超过一万条记录的导入任务，可使用 scripts/import_csv.py 提供的 --batch 参数分批提交，每批建议 500 条。若仍存在性能瓶颈，可考虑启用 Redis 缓存并调整 conf/aggregator.yaml 中的 cache_ttl 值，减少重复查询的数据库开销。

问：如何将系统迁移至新的服务器环境？

答：迁移时需保留 data/links.db 数据库文件以及 conf/aggregator.yaml 配置文件。在新服务器上克隆项目代码后，将上述文件覆盖至对应目录。若使用了 Redis，需同步导出 Redis 数据或重新构建缓存。启动服务后执行 ./bin/healthcheck.sh 验证所有依赖连通性及数据完整性。若迁移前后操作系统不一致，建议使用 Docker 镜像方式打包整个运行环境，避免库依赖差异导致的兼容性问题。
