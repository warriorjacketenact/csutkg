# ResourceBridge

ResourceBridge 是一个面向技术社区与开源开发者的外链资源汇总与导航工具。该项目不直接托管任何文章或媒体内容，而是以结构化方式收录来自互联网各领域的深度技术文章、分析报告与案例研究，帮助开发者、研究人员与技术决策者快速发现、检索与保存高价值的外部信息资源。

ResourceBridge 的核心定位是“技术资源的索引中枢”。它通过轻量级的数据采集与分类机制，将分散在多个内容源、不同域名下的优质外链统一归集，并提供标签过滤、全文检索与快照预览功能。项目本身不涉及爬虫抓取或内容转发，所有收录的链接均为人工筛选与社区提交，确保引用质量与来源可溯性。目标用户包括需要持续跟踪技术动态的架构师、进行竞品分析的产品经理、撰写技术博客的开发者，以及希望从大量信息中快速定位关键资料的研究人员。

本项目以 Markdown 驱动的静态站点方式发布，兼容 GitHub Pages、Vercel 等主流托管平台，支持一键部署与离线浏览。收录的外链覆盖软件工程、云计算、人工智能、信息安全、开源治理等十余个技术子领域，并按照时效性、热度与专业深度进行分级标注。ResourceBridge 不依赖任何第三方 API，所有元数据均以 YAML Front Matter 形式嵌入 Markdown 文件，便于版本控制与协作编辑。

## 功能概览

- **多维度分类索引**：每条外链按技术领域、内容类型（教程、案例、白皮书、访谈）、适用读者层级（入门、进阶、专家）进行三级标签标注，支持组合筛选。

- **全文元数据检索**：基于标题、摘要、标签与来源域名的本地搜索，无需联网即可在已收录数据范围内快速定位目标链接。

- **时效性状态标记**：对每条外链记录采集时间、更新时间与内容有效性状态（有效、失效、迁移），帮助用户识别资源可用性。

- **社区提交与审核流**：注册用户可通过 Web 表单提交新链接，提交后进入审核队列，审核通过后自动纳入索引并生成变更日志。

- **快照描述与摘要缓存**：每条外链附带人工撰写的 150-300 字内容摘要，说明文章核心观点与适用场景，减少用户点开链接前的决策成本。

- **批量导入与导出**：支持 CSV 与 JSON 格式的链接批量导入，以及按筛选条件导出链接列表，适用于团队内部共享或二次加工。

- **浏览历史与收藏夹**：基于 LocalStorage 的本地浏览记录与收藏功能，方便用户标记重要资源并跨会话保留。

- **RSS 订阅源生成**：按分类或标签生成 RSS Feed，用户可通过阅读器订阅特定领域的新增资源，无需频繁访问站点。

## 应用场景

- **技术团队内部知识库建设**：团队 Leader 可使用 ResourceBridge 整理每周技术周报的外链素材，将分散在邮件、IM 群组中的链接统一归集，并为每条链接添加内部备注与优先级标记，减少重复沟通成本。

- **开源项目文档的外链引用管理**：开源项目维护者可将项目文档中引用的第三方参考链接集中托管于 ResourceBridge，当外部链接发生迁移或失效时，只需在 ResourceBridge 中更新映射，无需修改项目各处的文档源文件。

- **技术博主选题与素材整理**：技术博主可借助分类筛选快速浏览某一领域近期的热门文章，通过摘要判断内容价值，大幅缩短选题调研与资料收集阶段的时间消耗。

- **培训机构课程大纲外链配套**：技术培训机构可为每期课程生成专属链接合集，学员通过统一入口访问课外阅读材料，机构亦可统计各链接的访问热度，优化课程内容配置。

- **个人技术阅读流重排**：个人开发者可将日常订阅的十余个技术博客、Newsletter 中的外链集中导入 ResourceBridge，通过自定义标签构建个人知识图谱，避免被社交媒体的信息流淹没。

## 快速开始

以下指令适用于 macOS / Linux 及 Windows WSL 环境，要求系统已安装 Git 与 Node.js 18.x 及以上版本。

```bash
git clone https://github.com/resourcebridge/resourcebridge.git
cd resourcebridge
npm install
npm run build
npm start
```

执行完成后，访问本地 3000 端口即可预览站点。生产环境部署建议使用 `npm run export` 导出静态文件，随后将 `out/` 目录内容上传至目标托管服务。

## 贡献指南

ResourceBridge 欢迎社区贡献者参与外链资源提交、元数据校对、功能开发与文档完善。所有贡献需遵守项目行为准则。

1. 提交新链接或更新现有链接元数据时，请先阅读 `docs/maintainer/metadata-spec.md`，确保标签与摘要符合规范。随后在 `src/data/sources/` 目录下找到对应分类的 YAML 文件，按照既定格式追加或修改条目，并提交 Pull Request。

2. 功能开发或 Bug 修复前，建议先在 Issues 中创建讨论帖，说明意图与实施方案，避免重复劳动或方案偏离。小规模修复（文档错别字、单行样式调整）可直接提交 PR，无需预先创建 Issue。

3. 所有 PR 必须通过持续集成检查，包括 TypeScript 类型校验、单元测试覆盖率不低于 80% 以及 ESLint 规则检测。PR 描述中需明确说明变更类型（新增功能、修复缺陷、文档更新）与影响范围。

4. 对于外链资源提交类的 PR，维护者将在 5 个工作日内完成审核。审核标准包括链接可访问性、内容专业度、与现有分类体系的匹配度以及摘要质量。审核通过后合并，并自动生成更新日志。

## 常见问题

**Q: ResourceBridge 与普通浏览器书签或 Pocket 等稍后读工具的核心区别是什么？**

A: 普通书签管理以个人收藏为导向，缺乏结构化元数据与协作能力；Pocket 等工具侧重于阅读体验与离线保存。ResourceBridge 定位为团队可共享、可分类索引、可审计变更的外链资源目录，每条链接均附带多维度标签、摘要与状态跟踪，适合作为组织内部的知识索引层，而非个人阅读工具。

**Q: 如果我发现收录的链接已经失效或内容发生重大变更，该如何处理？**

A: 您可以通过两种方式报告：一是在项目仓库的 Issues 中提交“链接失效”类型的工单，附上链接地址与失效现象描述；二是直接按照贡献指南修改对应的 YAML 元数据文件，将 `status` 字段更新为 `offline` 或 `moved` 并提交 PR。维护者会定期运行自动化检查脚本，但社区主动报告能显著提升数据时效性。

**Q: 项目是否支持私有化部署并与团队内部认证系统集成？**

A: ResourceBridge 核心版本为静态站点，不内置用户认证模块。但您可以通过前置反向代理（如 nginx 或 Cloudflare Access）实现基础认证，或通过部署在 Vercel 等平台时启用平台提供的访问控制功能。对于需要完整用户权限管理的团队，建议在二次开发时集成 OAuth 中间件，相关接口预留已在 `src/core/auth.ts` 中定义占位方法。
