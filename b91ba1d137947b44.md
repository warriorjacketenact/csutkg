# EMHA Show Link Catalog

EMHA Show Link Catalog 是一个面向技术研究者、内容策展人和开源社区维护者的结构化外链资源汇总系统。该项目将分散于 EMHA Shows 节目单中的大量深层链接进行统一索引、分类归档与状态监控，帮助用户快速定位特定活动、话题或技术主题对应的原始节目页面，避免在海量扁平 URL 中低效检索。

项目定位为技术资源导航工具，不爬取页面内容，仅对链接本身进行组织、标注与可用性检查。目标用户包括技术文档撰写者、社区活动组织者、开源布道师以及需要定期跟踪 EMHA Shows 节目更新的研究者。通过本项目的索引机制，用户可批量获取节目单链接的元信息（如所属期数、话题标签、嘉宾信息），并基于这些元数据执行筛选、导出或二次分发操作。

## 功能概览

批量链接导入与结构化存储 支持一次性导入 EMHA Shows 节目单目录下的大量 URL，自动解析文件名哈希并生成唯一记录标识。

链接状态健康检查 周期性对已收录链接发起 HEAD 请求，检测返回码是否为 200，标记失效或重定向链接以便人工复核。

多维度标签分类 允许用户为每个链接添加自定义标签（如主题、语言、嘉宾姓名），并基于标签组合进行快速过滤。

元数据扩展字段 每条记录预留注释字段，可记录页面标题、摘要或收录原因，支持后续全文检索。

列表导出与分享 将筛选后的链接列表导出为纯文本、CSV 或 JSON 格式，便于嵌入其他文档或工具链。

命令行交互界面 提供轻量级 CLI 工具，支持按关键词、状态、标签等条件查询链接，并实时输出结果。

增量更新机制 支持新增链接的追加导入，自动去重并提示重复条目，适合长期维护场景。

## 应用场景

技术社区周报撰写 社区维护者每周需要汇总 EMHA Shows 中与特定技术栈（如 Rust、Kubernetes）相关的节目链接。通过本项目的标签过滤功能，可快速提取本周所有匹配标签的链接，并导出为周报附录。

开源项目资源归档 开源项目 README 或文档中常需要引用外部活动链接。策展人可使用本项目将 EMHA Shows 的节目链接按年份或季度归档，生成稳定的参考列表，避免原始节目单变动导致引用失效。

活动内容回溯分析 研究者可对已收录链接的响应状态进行长期追踪，分析哪些历史节目页面仍保持可访问，哪些已迁移或下架，从而评估内容的持久性。

批量链接迁移辅助 当 EMHA Shows 节目单目录结构发生变化时，本项目提供的映射表可帮助用户快速将旧链接映射至新地址，减少手工替换工作量。

## 快速开始

以下操作假设已安装 Git 和 Node.js（v18 及以上版本）。

```bash
git clone https://github.com/emha-tools/emha-link-catalog.git
cd emha-link-catalog
npm install
npm run build
./bin/catalog.js import --file ./samples/emha-urls.txt
```

导入完成后，可使用 `list` 命令查看所有已收录链接：

```bash
./bin/catalog.js list --status active --limit 20
```

若需导出为 CSV 文件：

```bash
./bin/catalog.js export --format csv --output emha-export.csv
```

## 项目结构

```
emha-link-catalog/
├── bin/                                 CLI 可执行入口文件目录
│   └── catalog.js                       # 主命令行工具，注册所有子命令
├── src/                                 核心源代码目录
│   ├── core/                            领域逻辑模块
│   │   ├── linkRegistry.js              # 链接注册、去重与状态管理
│   │   ├── healthChecker.js             # 基于 HTTP HEAD 的健康检查实现
│   │   └── tagEngine.js                 # 标签增删改查及组合过滤逻辑
│   ├── adapters/                        数据持久化适配器
│   │   ├── memoryStore.js               # 内存存储，用于测试或临时运行
│   │   ├── jsonStore.js                 # JSON 文件读写，默认持久化方案
│   │   └── sqliteStore.js               # SQLite 存储，用于大规模数据场景
```

## 常见问题

**Q：项目最多可以收录多少条链接？性能会受影响吗？**

A：本项目无硬性数量上限。使用 JSON 存储时，实测 5000 条以内的链接导入和列表查询响应时间在 1 秒以内。超过 5000 条建议切换至 SQLite 存储模式，并通过 `--limit` 分页查询。健康检查并发数可通过环境变量 `CHECK_CONCURRENCY` 调节，默认值为 10。

**Q：健康检查会访问页面内容吗？是否会触发反爬机制？**

A：健康检查仅发送 HTTP HEAD 请求，不下载页面正文，因此对目标服务器造成的负载极小，通常不会被识别为爬虫。HEAD 请求返回的状态码足以判断链接是否有效。若目标站点对 HEAD 方法有限制，可配置为 GET 请求并设置 `--no-body` 标志，但会消耗更多带宽。

**Q：如何将已收录链接的标签信息迁移到新版本？**

A：标签数据与链接记录一同存储在存储层（JSON 文件或 SQLite 数据库）。升级项目版本时，只需备份原存储文件，新版本启动时会自动兼容旧格式。若存储格式发生重大变更，发布说明中会明确提供迁移脚本，执行 `npm run migrate` 即可完成转换。
