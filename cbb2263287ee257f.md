# Tokyo Mirai Resource Aggregator

Tokyo Mirai Resource Aggregator 是一个面向技术研究者、内容开发者与数字化档案管理者的高性能外链资源汇总工具。该项目定位于对分散于网络各处的技术视频、文档与媒体资源进行结构化收集、分类存储与快速检索，帮助用户在特定领域（如人工智能、软件开发、数字媒体制作）中建立可复用的外部知识索引。

项目并非传统的网页爬虫或下载器，而是一个轻量级的资源引用与管理框架。它提供统一的资源标识符映射机制，允许用户通过本地配置文件或 RESTful API 将外部资源 URL 注册至项目索引中，并依据主题、日期、版本等维度进行自动归类。适用于需要长期维护大量外部链接的个人知识库构建者、技术社区内容运营团队以及教育培训机构。

## 功能概览

**统一资源索引注册**：提供基于 YAML 与 JSON 格式的批量资源导入接口，支持单次注册数百条外部链接，并自动校验 URL 可访问性。

**多维度分类标签系统**：每条资源可绑定多个自定义标签（如 "backend"、"frontend"、"ml"、"devops"），支持按标签组合进行快速筛选与视图切换。

**快照与元数据缓存**：自动抓取目标资源的标题、描述与内容摘要，生成本地元数据缓存，避免因源站内容变更导致索引信息失效。

**定时健康检查**：内置后台任务调度器，可配置定时策略对已注册资源进行可用性探测，生成异常报告并发送通知。

**只读 API 端点**：提供基于 RESTful 设计的只读查询接口，支持分页、排序与字段过滤，便于与其他内部系统集成。

**数据导入导出**：支持将整个资源索引导出为 CSV 或 Markdown 表格格式，便于离线存档或在其他文档平台发布。

## 应用场景

技术团队内部知识库维护：开发团队可使用该项目汇总日常遇到的优秀技术演讲视频、官方文档链接与开源项目仓库地址，统一存放于团队内部的索引服务器，新成员入职时可快速获取完整的学习材料清单。

在线教育平台课程资源管理：教育机构在制作在线课程时，需要引用大量外部补充阅读材料与案例演示视频。使用本项目的标签系统可按课程章节、难度级别或语言对资源进行划分，方便学员按需访问。

个人研究人员的文献与媒体档案建设：独立研究员在进行文献综述或技术调研时，可将收集到的会议录播、技术报告 PDF 链接与演示文稿地址全部注册至本地索引，配合定时健康检查功能及时发现失效链接并补充替代来源。

## 快速开始

以下步骤演示如何在本地环境中获取项目源码、安装依赖并启动开发服务器。

```bash
# 克隆项目仓库至本地
git clone https://github.com/tokyo-mirai/resource-aggregator.git
cd resource-aggregator

# 安装项目依赖（使用 pip 与虚拟环境）
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 初始化本地数据库与配置文件
python manage.py initdb
python manage.py load-fixtures initial_tags

# 启动开发服务器（默认监听 127.0.0.1:8000）
python manage.py runserver
```

访问 http://127.0.0.1:8000/api/v1/resources 即可查看已注册资源的 JSON 响应。默认配置中已包含一组示例数据，可供测试 API 分页与过滤功能。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.10 或更高 | 项目主要运行时环境，用于执行 API 服务与后台任务 |
| SQLite | 3.35.0 或更高 | 默认嵌入式数据库，用于存储资源索引与元数据缓存 |
| pip | 22.0 或更高 | Python 包管理工具，用于安装 requirements.txt 中列出的依赖项 |
| Git | 2.30.0 或更高 | 用于克隆仓库以及未来通过 git pull 获取更新 |
| curl 或 wget | 任意稳定版本 | 用于元数据抓取模块中的 HTTP 请求，非必需但强烈建议安装 |
| cron 或 systemd-timer | 任意版本 | 用于定时健康检查功能的系统级调度，生产环境部署时必需 |

## 贡献指南

我们欢迎并鼓励开发者以多种形式参与本项目贡献。请遵循以下标准化流程：

第一，在 GitHub 仓库的 Issue 页面中查找带有 "help wanted" 或 "good first issue" 标记的任务。若无适合的任务，可自行创建 Issue 描述您希望新增的功能或修复的问题，并等待维护者确认需求合理性。

第二，将项目仓库复刻至个人账户，并在本地基于 main 分支创建新的功能分支。分支命名应遵循 feat/xxx 或 fix/xxx 的模式，其中 xxx 为关联的 Issue 编号或简短描述。

第三，完成代码变更后，确保所有现有单元测试通过，并为新增功能编写至少一个对应的测试用例。运行 pytest 命令检查测试覆盖率，若覆盖率低于 85%，需补充测试代码。

第四，提交变更时使用规范的提交信息格式，即 `<type>(<scope>): <subject>`，例如 `feat(api): add sort_by parameter to resource list endpoint`。提交前运行 pre-commit 钩子进行代码风格检查。

第五，向主仓库的 main 分支发起合并请求，并在描述中关联相关 Issue 编号。合并请求至少需要一名项目维护者审核通过后方可合入。

## 常见问题

**问：项目是否支持 PostgreSQL 或 MySQL 作为生产数据库？**

答：项目基于 SQLAlchemy ORM 构建，理论上支持所有 SQLAlchemy 兼容的关系型数据库。在生产部署时，只需将配置中的 SQLALCHEMY_DATABASE_URI 修改为对应的 PostgreSQL 或 MySQL 连接字符串即可。但请注意，项目默认的初始化脚本与迁移文件中包含部分 SQLite 特定的日期处理函数，迁移至其他数据库时可能需要手动调整这些函数。

**问：元数据抓取模块是否会因为目标网站的反爬机制而失效？**

答：元数据抓取模块默认使用常见的 User-Agent 请求头，并尊重 robots.txt 中的爬虫规则。对于具有严格反爬策略的网站，用户可通过配置文件自定义请求头与代理设置。如果某资源持续抓取失败，系统会自动将该资源标记为 "metadata_unavailable" 状态，并在下次健康检查时重试，但不会影响该资源的索引注册与 API 查询功能。

**问：如何批量更新已注册资源的标签或分类信息？**

答：项目未提供内置的批量更新 UI，但支持通过 API 或脚本方式进行批量操作。推荐的做法是导出当前资源的完整列表为 CSV 文件，在表格软件中修改标签列后，使用 scripts/import_batch.py 脚本以 --update 模式重新导入，脚本会依据资源 ID 或 URL 匹配并更新对应字段。对于超过 1000 条资源的大规模更新，建议在维护窗口期间执行，并提前备份数据库文件。
