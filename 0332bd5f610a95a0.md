# WebIndex

WebIndex 是一个面向技术研究、数据分析与数字取证领域的高密度外链资源汇总系统。项目定位于将分散于互联网各处的动态数据源、直播流接口与实时信息页面进行结构化收录，为研究人员、爬虫开发者与安全分析人员提供可复用的入口索引。本项目不存储任何内容，仅提供公开 URL 的整理与分类，所有资源均指向第三方服务。

本项目的目标用户包括网络爬虫工程师、舆情分析人员、CDN 节点监测开发者以及信息安全方向的学术研究者。通过统一的索引体系与标准化的资源分类，用户可以快速定位特定业务场景下的外部数据源，大幅降低信息发现与入口维护的时间成本。项目采用纯静态 Markdown 与 JSON 双格式输出，可无缝对接各类自动化采集框架与监控告警系统。

## 功能概览

**多层级资源分类索引**：依据来源域名、内容类型与访问协议对全部外链进行三级分类，支持按业务场景快速筛选。

**自动化链接可用性检测**：内置基于 HTTP 状态码与响应时间的活性检测脚本，可定期输出失效链接报告。

**批量导入与去重管理**：支持 CSV 与纯文本列表批量导入新链接，自动识别并移除重复条目，保留最新添加时间戳。

**标签系统与全文检索**：每条资源可附加多维度标签，配合简单的 grep 与 jq 查询接口，实现毫秒级关键词定位。

**版本化快照与变更日志**：每次索引更新均生成带时间戳的 JSON 快照，变更日志记录新增、移除与分类调整操作。

**元数据增强字段**：每条记录包含响应内容类型预估、SSL 证书签发者信息与目标服务器地理位置粗略标注。

**自定义输出模板**：支持将索引导出为 HTML 导航页、Nginx 重写规则文件或 Prometheus 服务发现目标配置。

**私有化部署就绪**：项目提供 Docker 镜像与一键安装脚本，可于内网环境完整部署，不依赖任何外部 API 服务。

## 应用场景

**舆情监测系统的入口管理**：舆情分析团队通常需要同时监控数十个不同来源的动态页面。WebIndex 提供了统一的链接登记与变更通知机制，当某个数据源入口失效或响应异常时，维护人员可在数分钟内定位并替换，避免采集任务大面积失败。

**CDN 节点质量对比分析**：通过收录不同服务商提供的直播流测试页面，网络工程师可以利用本项目索引快速构建节点延迟测试矩阵，周期性探测各线路的可用带宽与丢包率，为调度策略优化提供数据支撑。

**学术研究中的可复现性保障**：社会科学与网络测量领域的论文常需要引用特定的公开数据页面。WebIndex 为每个收录链接分配稳定的内部编号，并记录首次收录时间与最后验证时间，研究者可在论文中引用项目编号而非直接 URL，以便后续版本追踪。

**渗透测试与信息收集热身**：安全测试人员在启动正式漏洞挖掘前，往往需要快速收集目标周边的公开信息。本项目提供的分类索引可以作为信息收集阶段的起点，帮助测试人员快速构建目标关联资源清单。

## 快速开始

以下指令适用于 Linux 与 macOS 环境，Windows 用户建议通过 WSL 2 或 Git Bash 执行。项目基于 Python 3.9+ 开发，依赖标准库与少量第三方包。

```bash
git clone https://github.com/webindex/webindex.git
cd webindex
pip install -r requirements.txt
python webindex.py --build --output ./dist
```

执行完成后，所有索引文件将输出至 ./dist 目录，包含 index.md、index.json 与分类子目录。如需启用活性检测，可增加 --check 参数。

## 项目结构

```
webindex/
├── config/                                 # 配置目录，存放全局设置与分类映射
│   ├── default.yaml                        # 默认配置，含检测超时与输出路径
│   └── categories.yaml                     # 分类映射表，定义一级与二级分类规则
├── core/                                   # 核心逻辑模块
│   ├── indexer.py                          # 索引构建主程序，负责解析与合并
│   ├── checker.py                          # 活性检测实现，支持并发与重试
│   └── exporter.py                         # 输出适配器，生成 MD、JSON 等格式
├── data/                                   # 数据存储目录，存放原始列表与快照
│   ├── raw/                                # 原始输入文件，按批次存放
│   │   └── batch_555.txt                   # 第 555 批次原始链接清单
│   ├── snapshots/                          # 历史快照，按日期归档
│   │   └── 2026-06-23.json                 # 每日完整索引快照
│   └── cache/                              # 活性检测缓存，减少重复网络请求
│       └── status_cache.db                 # SQLite 缓存数据库
├── docs/                                   # 用户文档与设计说明
│   ├── user-guide.md                       # 用户手册，含常见操作示例
│   ├── configuration.md                    # 配置参数完整参考
│   └── data-schema.md                      # JSON 输出字段定义与示例
├── scripts/                                # 辅助运维脚本
│   ├── import_csv.py                       # CSV 批量导入工具
│   └── cron_check.sh                       # 周期性活性检测 Cron 任务模板
├── tests/                                  # 单元测试与集成测试
│   ├── test_indexer.py                     # 索引构建逻辑测试用例
│   └── test_checker.py                     # 检测模块模拟测试
├── dist/                                   # 构建输出目录，生成的文档与 JSON 存放于此
│   ├── index.md                            # 汇总 Markdown 文档
│   └── index.json                          # 结构化 JSON 数据

├── setup.py                                # 安装脚本，支持 pip install -e .
└── webindex.py                             # 命令行入口，接收 build、check 等子命令
```

## 贡献指南

1. 在 GitHub 仓库页面点击 Fork 按钮，将项目复制至个人账户，随后克隆至本地开发环境。建议在 dev 分支上进行所有修改操作。

2. 新增资源链接时，请先运行 python webindex.py --dedup 检查是否已收录，避免重复添加。确认无重复后，将链接追加至 data/raw/ 目录下对应批次的文本文件中，并按现有格式补充分类标签。

3. 修改分类映射或调整输出模板时，需同步更新 config/categories.yaml 与 core/exporter.py 中的对应逻辑，确保不同输出格式之间的分类一致性。修改后执行 python -m pytest tests/ 验证核心功能未被破坏。

4. 提交 Pull Request 前，请确保本地已运行 python webindex.py --build --check --output ./dist 完整构建一次，并检查 dist/index.json 中的新增条目元数据字段是否完整。提交信息请遵循 `type(scope): description` 格式，例如 `feat(indexer): add batch import retry mechanism`。

5. 对于失效链接的移除或替换，请在 Pull Request 描述中提供检测时间与 HTTP 状态码截图，以便维护者核实。合并后系统将自动更新变更日志。

## 常见问题

**问：活性检测是否会频繁访问目标服务器，造成对方压力？**
答：检测模块默认采用单线程顺序执行，且每个请求间隔不低于 500 毫秒。用户可通过配置文件中的 check_interval 参数调整请求频率。对于明确声明了 robots.txt 禁止爬取的站点，建议在配置中手动排除检测列表。

**问：项目输出的 JSON 数据结构是否会频繁变动？**
答：项目承诺在 2.x 版本周期内保持主要字段的向后兼容性。新增字段会标记为 optional，并以扩展方式添加，不会移除现有核心字段。具体变更会在 CHANGELOG 中提前至少一个版本声明。

**问：如何在内网环境使用 WebIndex？**
