# LinkVault Resource Aggregator

LinkVault Resource Aggregator 是一个面向技术研究人员、内容审核人员与数据归档工作者的轻量级外链资源汇总与导航系统。该项目定位于将分散在网络各处的媒体资源、文档入口与信息页面进行集中收录，通过结构化的目录索引与稳定的详情页映射机制，解决信息碎片化导致的检索效率低下与链接失散问题。

系统核心功能围绕资源链接的批量导入、分类存储与快速检索展开，支持通过唯一标识符（如qlkxkb、ssdhjfxy）精准定位至对应的资源详情页面。LinkVault 不对上游资源内容进行二次加工或代理转发，仅提供索引与跳转服务，确保资源来源的原始性与可追溯性。目标用户包括需要维护大型外链清单的运维工程师、从事内容聚合的平台运营人员以及需要批量管理参考资料的研究型用户。

## 功能概览

**批量链接导入**：支持通过命令行工具或 API 接口一次性导入数百条资源链接，自动解析 URL 结构与参数，生成内部索引记录。

**分类标签体系**：为每条资源赋予多级分类标签（如技术文档、影视媒体、学术资料），支持按标签组合进行快速过滤与筛选。

**详情页智能映射**：根据资源 ID 自动生成标准化详情页访问路径，无需手动维护复杂的路由映射表。

**链接健康检测**：定时对已收录链接进行可达性探测，自动标记失效链接并生成告警日志，便于运维人员及时清理或更新。

**全文检索支持**：基于资源标题、描述与标签字段构建倒排索引，支持毫秒级的关键词检索响应。

**数据导入导出**：提供 JSON、CSV、Markdown 三种格式的数据导入导出功能，方便与其他系统进行数据交换。

**访问统计看板**：记录每条资源的点击次数与访问来源，生成可视化统计图表，辅助分析资源热度与用户偏好。

**多用户权限管理**：支持管理员、编辑者、访客三级权限体系，不同角色拥有不同的增删改查操作范围。

## 应用场景

**技术文档归档**：研发团队可将散落在各技术博客、官方文档站点的参考资料链接统一录入 LinkVault，为每个项目建立独立的资源清单，方便新成员快速获取学习材料。

**影视资源导航**：内容聚合平台运营者可使用 LinkVault 整理影视详情页入口，通过分类标签区分类型、地区或年份，为用户提供清晰的内容发现路径。

**学术参考文献管理**：研究人员在撰写论文或综述时，可将引用的网络资源链接集中存储于 LinkVault，配合备注字段记录引用上下文，避免参考文献链接丢失或遗忘。

**运维监控巡检**：系统运维人员将需要定期检查的服务状态页、监控看板链接录入系统，利用链接健康检测功能自动巡检，第一时间发现不可达的服务入口。

## 快速开始

以下步骤将指导您在本地环境快速启动 LinkVault 服务。

```bash
# 克隆项目仓库
git clone https://github.com/linkvault/linkvault.git

# 进入项目目录
cd linkvault

# 安装依赖（使用 pip 和 npm 分别安装后端与前端依赖）
pip install -r requirements.txt
npm install --prefix frontend

# 初始化数据库
python manage.py migrate

# 导入示例资源数据
python manage.py loaddata fixtures/initial_links.json

# 启动开发服务器（后端默认运行于 8000 端口，前端开发服务器运行于 3000 端口）
python manage.py runserver &
npm run dev --prefix frontend
```

访问本地 3000 端口即可进入 LinkVault 资源管理界面，默认管理员账号为 admin，密码为 admin123，首次登录后请及时修改密码。

## 安装要求

| 依赖 | 必需 | 说明 |
|------|------|------|
| Python 3.9+ | 是 | 后端运行环境，推荐使用 3.10 或更高版本以获得更好的性能与类型提示支持 |
| Node.js 18+ | 是 | 前端构建与运行环境，需包含 npm 包管理器 |
| PostgreSQL 13+ | 是 | 生产环境推荐使用 PostgreSQL，开发环境可选用 SQLite 作为替代 |
| Redis 6+ | 否 | 用于缓存与会话存储，未安装时将回退至内存缓存模式，但多进程环境下不推荐 |
| Nginx 1.20+ | 否 | 生产环境静态资源代理与反向代理服务器，提高并发处理能力 |
| Docker 20.10+ | 否 | 使用容器化部署时必需，项目提供完整的 Docker Compose 编排文件 |
| Git 2.25+ | 是 | 用于版本管理与代码拉取 |
| make 或 gnumake | 否 | 使用 Makefile 自动化任务时可选，非必需但能简化操作流程 |

## 贡献指南

1. 在 GitHub 上 Fork 本仓库至个人账户，并克隆到本地开发环境。请确保在开发前阅读 docs/developer/ 目录下的架构设计文档，理解各模块之间的依赖关系。

2. 创建新的功能分支，分支命名遵循 feat/ 或 fix/ 前缀加简短描述（如 feat/batch-import、fix/health-check-timeout），避免在 main 分支上直接提交。

3. 完成代码开发后，请确保所有单元测试通过（后端执行 pytest，前端执行 npm run test），并为新增功能补充对应的测试用例。代码风格需符合项目配置的 flake8 与 prettier 规范。

4. 提交 Pull Request 时，请填写标准的 PR 模板，清晰描述本次变更的目的、实现方式以及测试覆盖情况。PR 至少需要一位项目维护者审核通过后方可合并。

5. 文档更新与代码变更同等重要。如果您的修改影响了用户操作流程或部署方式，请同步更新 docs/ 目录下的对应文档，确保文档与代码始终保持一致。

## 常见问题

**Q：系统导入的链接数量达到上限后如何处理？**

A：LinkVault 默认不设硬性上限，实际承载能力取决于服务器硬件配置与数据库性能。当单表记录超过 100 万条时，建议启用分区表功能或将历史数据归档至冷存储。您也可以通过配置 LINK_LIMIT 环境变量在应用层设置软性阈值，超出时系统会发出警告日志但不会阻止导入。

**Q：链接健康检测的结果是否会影响前端展示？**

A：健康检测结果会以状态标签（正常、失效、超时）的形式展示在资源列表与详情页中，供用户参考。但即便检测结果为失效，系统仍保留原始链接的跳转入口，不会自动屏蔽或删除任何资源。所有检测结果仅用于提示与运维告警，不改变资源的可访问性。

**Q：如何迁移 LinkVault 到另一台服务器？**

A：迁移需要同步转移数据库数据与静态文件。数据库层面可使用 pg_dump 导出再导入；静态文件（用户上传的附件等）需整体拷贝 storage/ 目录。完成数据迁移后，请检查 settings.py 中的 DATABASES 与 STATIC_ROOT 配置是否与新环境匹配，最后执行 python manage.py collectstatic 重建静态文件索引。
