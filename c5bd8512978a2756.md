# GXHY-LSMX Resource Aggregator

GXHY-LSMX Resource Aggregator 是一个面向技术研究人员、网络安全分析师、数据挖掘工程师以及开源情报（OSINT）从业者的高性能外链资源汇总与导航系统。该项目不产生原创内容，专注于对特定领域内的高价值网络资源进行结构化整理、分类归档与可用性监控，解决用户在分散数据源中检索效率低下、链接失效频繁以及上下文缺失等核心痛点。

项目定位为技术性中间层工具，而非终端内容平台。用户群体包括但不限于渗透测试人员、威胁情报分析员、学术论文研究者以及企业安全运营中心（SOC）的初级分析师。通过本项目的资源索引能力，用户可快速定位至原始讨论线程或数据页面，大幅缩短信息收集周期。

## 功能概览

- **多源异构资源索引**：系统支持对来自不同顶级域名、不同路径结构（包括动态参数型与静态伪静态型）的 URL 进行统一归一化处理，并提取核心标识符（如线程 ID 或内容哈希），实现跨站点的资源关联。

- **线程级元数据提取**：针对论坛型资源（如 gxhy.net 下的线程），自动解析 URL 中的复合标识符（如 thread-XXXXXX-XXXXXX 格式），并尝试从路径中还原发布时间、作者匿名 ID 或分区标签，为后续过滤提供字段支持。

- **内容节点健康检查**：内置异步 HTTP 探针，可定期对收录的所有外链执行可用性验证（基于状态码与响应体特征词），并生成离线报告，标记失效或重定向资源。

- **资源分类与标签体系**：基于 URL 来源域名、路径关键词以及上下文语义（通过轻量级正则规则库）为每个资源赋予多维度标签，例如「代码审计」、「漏洞讨论」、「工具分享」、「配置示例」或「日志分析」。

- **命令行与 Web 双前端**：提供 Python CLI 工具用于批量导入、导出和查询资源；同时内置 Flask 轻量级 Web 界面，支持按标签、域名、时间范围进行组合过滤与全文检索。

- **增量更新与去重机制**：采用布隆过滤器结合 SQLite 本地缓存，确保同一资源在不同批次导入时不会重复记录，并自动记录首次发现时间与最后更新时间。

## 应用场景

1. **渗透测试前期信息收集**：测试人员在进行外部攻击面梳理时，可通过本项目的标签筛选功能快速定位到以往公开讨论中涉及的子域名、IP 段或第三方服务凭证泄露线索，大幅降低信息盲区。

2. **开源威胁情报关联分析**：安全分析师在调查特定恶意软件家族或攻击组织（APT）时，可利用项目内聚合的多个独立论坛讨论串，交叉验证攻击手法、工具特征及基础设施关联，形成完整的攻击链视图。

3. **学术文献参考文献整理**：高校研究人员在撰写关于网络安全态势感知或社会工程学攻击演进的论文时，可将本项目作为参考文献的源头索引，批量获取原始讨论链接，避免因个人书签丢失导致引用中断。

4. **企业安全合规审计追溯**：企业安全部门在自查内部系统是否暴露已知漏洞时，可依据项目中收录的早期漏洞讨论帖，追溯漏洞披露时间线、修复补丁发布状态以及社区提供的临时缓解措施。

5. **CTF 竞赛训练资源检索**：参与网络安全技能竞赛的队伍可利用本项目的域名聚合功能，快速查找特定类型（如 Web 注入、逆向工程、密码破解）的实战讨论资源，作为日常训练的辅助资料库。

## 快速开始

以下步骤适用于 Linux/macOS 系统及 Windows WSL2 环境。请确保已安装 Git 与 Python 3.9 或更高版本。

```bash
# 克隆项目仓库至本地
git clone https://github.com/your-organization/gxhy-lsmx-aggregator.git
cd gxhy-lsmx-aggregator

# 安装项目依赖（推荐使用虚拟环境）
python3 -m venv venv
source venv/bin/activate
pip install --upgrade pip
pip install -r requirements.txt

# 执行首次资源导入与索引构建（包含样例数据）
python cli.py import --source data/seed_urls.json --output db/index.sqlite
python cli.py health-check --db db/index.sqlite --timeout 5 --retry 2
python web/app.py --db db/index.sqlite --host 127.0.0.1 --port 8080
```

执行上述命令后，Web 服务将在本地 8080 端口启动。打开浏览器访问 `http://127.0.0.1:8080` 即可查看资源列表。

## 贡献指南

1.  **资源提交规范**：外部贡献者需通过 GitHub Issue 提交新增资源链接，每条链接必须附带所属主题标签（至少一个）以及一句话的内容摘要（中文或英文）。Issue 标题格式为 `[Resource] 域名 - 简述`，例如 `[Resource] gxhy.net - 讨论 Windows Kernel 提权技巧`。

2.  **解析器扩展流程**：若需支持新域名或新 URL 结构，请在 `core/parser.py` 中新增对应的解析类，继承基础 `BaseParser`，并实现 `extract_id()` 与 `normalize()` 方法。提交前需确保通过 `tests/test_parser.py` 中新增的单元测试用例。

3.  **文档更新要求**：任何涉及配置变更、CLI 参数调整或数据库结构修改的拉取请求，必须同步更新 `docs/user-guide.md` 与 `docs/api-reference.md` 中的对应章节。文档采用 Markdown 格式，代码示例需保持可运行状态。

4.  **健康检查策略优化**：针对频繁超时或被限流的站点，允许在 `config/default.yaml` 中调整 `timeout`、`retry` 以及 `backoff_factor` 参数。提交优化方案时需附上调整前后的响应时间对比数据（至少 10 个样本）。

5.  **代码审查与合并**：所有拉取请求需至少两位项目维护者进行 Code Review。审查重点包括但不限于：异常处理完整性、SQL 注入防护、并发安全性以及日志输出的结构化程度。合并前需通过全部 CI 流水线检查（包括 Lint、单元测试与覆盖率阈值）。

## 常见问题

**问：导入大量 URL 时出现内存占用过高的情况，应如何优化？**

答：建议采用分批导入策略，每批处理 500 条记录并调用 `gc.collect()` 主动回收内存。同时可调整 `config/default.yaml` 中的 `batch_size` 参数至 200 以降低峰值内存。对于超过 10 万条记录的批量导入，推荐使用 `--no-web` 模式关闭 Web 上下文加载，仅执行核心索引逻辑。

**问：健康检查报告显示大量超时，但浏览器访问对应 URL 正常，是什么原因？**

答：部分站点启用了反爬机制，会针对非浏览器 User-Agent 返回 403 或延迟响应。请在 `config/default.yaml` 的 `user_agents` 池中增加真实浏览器 UA（如 Chrome/Edge 最新版本），并将 `timeout` 调整至 10 秒以上。同时检查网络环境是否对出口 IP 有访问限制，必要时配置 `proxies` 参数使用代理池。

**问：如何迁移数据库至 PostgreSQL 以支持多用户并发查询？**

答：项目默认使用 SQLite 适用于单机部署。如需迁移至 PostgreSQL，请使用 `db/migrations/` 下的迁移脚本，将 `schema.sql` 中的数据类型（如 `TEXT` 改为 `VARCHAR`，`INTEGER` 改为 `BIGINT`）做对应转换，并在 `core/models.py` 中替换数据库驱动为 `psycopg2`。同时需在 `config/default.yaml` 中修改 `database_url` 连接串格式。
