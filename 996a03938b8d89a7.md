# LinkMaster Pro

LinkMaster Pro 是一个面向技术团队与内容运营者的外链资源归集与导航系统。项目定位为轻量级、可自托管的技术文章与工具站点外链汇总平台，帮助用户将散落在各处的优质外部资源按统一规则收录、分类、版本追踪，并提供面向团队内部的快速检索与引用能力。

本项目适用于需要长期维护大量外部链接的文档工程师、技术博主、开源项目维护者以及知识库管理员。通过结构化的数据目录与自动化校验脚本，LinkMaster Pro 能够有效降低外链失效、路径变更、域名迁移带来的维护成本，并将外链资源以标准化格式对外呈现，便于第三方工具抓取与二次分发。

## 功能概览

批量外链导入与去重校验 系统提供基于 CSV 与 JSON 的批量导入接口，自动识别重复 URL 并生成冲突报告，支持人工审核后合并。

链接状态周期探测 内置基于 HTTP HEAD 请求的状态检测模块，可每日定时扫描全部外链，标记 4xx、5xx 异常状态并输出告警日志。

多级分类标签管理 支持用户自定义分类树，每条外链可关联多个标签，分类与标签数据独立存储，便于多维度筛选与统计。

资源元数据扩展字段 每条链接记录支持标题、摘要、作者、发布时间、所属领域等十余个扩展字段，适配不同场景下的信息整理需求。

全文检索与快速跳转 基于倒排索引实现标题与摘要的全文搜索，搜索结果附带直接跳转链接，减少手动复制粘贴操作。

访问热度统计与排序 记录每条外链在本系统内的点击次数与最后访问时间，支持按热度、更新时间、创建时间排序输出列表。

数据快照与回滚机制 每日自动生成全量数据快照，保留最近 30 个版本，支持一键回滚至任意历史状态，降低误操作风险。

开放 API 与 Webhook 通知 提供 RESTful 风格的查询与更新 API，并支持将状态异常、新增链接等事件通过 Webhook 推送至飞书、钉钉或企业微信。

## 应用场景

技术团队内部知识库外链管理 技术团队在日常开发中会积累大量外部参考链接，包括官方文档、社区讨论、博客教程等。LinkMaster Pro 可作为团队知识库的外链补充模块，统一存放这些分散的链接，并为每条链接补充标签与摘要，方便新成员快速了解技术选型背景。

开源项目 README 与文档站外链维护 开源项目的 README 或文档站中常引用大量外部资源，包括依赖库主页、API 参考、示例项目等。当这些外部资源发生迁移或下架时，项目文档中的链接将失效。使用 LinkMaster Pro 将外链集中管理后，可通过自动化检测提前发现失效链接，并在项目文档中统一更新链接地址，避免逐个文件手动修改。

技术博客与资讯周报的素材库 技术博主或资讯周报编辑需要定期收集和整理近期值得关注的技术文章与工具。LinkMaster Pro 的分类标签与热度统计功能可帮助快速筛选近期热门资源，并生成符合 Markdown 格式的链接列表，直接粘贴至周报或博客草稿中。

多项目共享外链池 当同一组织下有多个平行项目或子产品时，各项目可能需要引用相同的外部依赖或参考文档。LinkMaster Pro 支持多项目隔离与共享两种模式，管理员可将通用链接标记为“公共资源”，供所有项目引用，减少重复录入工作。

## 快速开始

以下命令演示了如何从 GitHub 克隆项目、安装依赖并启动开发服务。请确保本地已安装 Git 与 Node.js 18.x 或更高版本。

```bash
git clone https://github.com/linkmaster-pro/linkmaster.git
cd linkmaster
npm install
npm run dev
```

执行上述命令后，开发服务器将默认启动在 127.0.0.1:3000。访问该地址即可进入 LinkMaster Pro 的仪表盘界面，首次启动将自动创建示例数据与管理员账户，初始密码在启动日志中输出，请及时修改。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | 18.x 或 20.x LTS | 运行时环境，推荐使用官方二进制包或 nvm 管理 |
| npm | 9.x 或 10.x | 包管理工具，随 Node.js 一同安装 |
| SQLite | 3.38 及以上 | 默认内置数据库，无需额外安装；生产环境可切换至 PostgreSQL 14+ |
| Redis | 7.0 及以上 | 用于缓存与任务队列，可选依赖，未安装时降级为内存存储 |
| Git | 2.30 及以上 | 用于版本管理与后续热更新补丁拉取 |
| PM2 | 5.x | 生产环境进程守护，可选但强烈建议安装 |

## 贡献指南

我们欢迎并鼓励开发者以多种方式参与 LinkMaster Pro 项目的改进与完善。请遵循以下步骤提交贡献：

第一，在 GitHub 上 fork 本仓库至个人账号，并克隆至本地开发环境。建议在开发前阅读 docs/development.md 文档，了解代码结构、编码规范与测试要求。

第二，新建一个以功能或修复内容命名的分支，例如 feature/batch-import-csv 或 fix/detector-timeout。所有开发工作应在该分支上进行，避免直接修改 main 分支。

第三，完成代码修改后，请确保所有现有单元测试与集成测试通过，并为新增功能或修复补充对应的测试用例。测试命令为 npm test，覆盖率报告生成命令为 npm run test:coverage。

第四，提交代码时请遵循 Conventional Commits 规范，使用 feat、fix、docs、style、refactor、perf、test、chore 等类型前缀，并附上简明清晰的提交说明。

第五，向本仓库的 main 分支发起 Pull Request，并在 PR 描述中详细说明变更目的、实现方案与测试结果。PR 将由项目维护者进行 Code Review，通过后合并入主线。

## 常见问题

Q: 启动时提示 SQLite 版本过低或数据库连接失败，应如何处理？

A: 请先检查本地 SQLite 版本是否符合要求（3.38 及以上）。若版本过低，请通过系统包管理器升级 SQLite。若数据库连接失败，请检查 app/lib/config.js 中 DATABASE_URL 环境变量是否设置正确。默认情况下，系统会在项目根目录下创建 linkmaster.db 文件，若该文件所在目录无写入权限，也会导致连接失败。建议将数据库文件存放在用户目录或挂载卷中。

Q: 链接状态检测任务执行时间过长，是否会影响主服务响应？

A: 检测任务通过独立的 worker 线程执行，不会阻塞主事件循环。若外链数量超过 5000 条，建议将 scheduler/daily-check.js 中的并发数（concurrency 参数）从默认的 10 调整为 5，避免网络 I/O 过载。同时，可在环境变量中配置 CHECK_TIMEOUT（单次检测超时毫秒数）和 CHECK_RETRY（重试次数）以优化检测策略。若仍需进一步优化，可切换至 Redis 任务队列模式，将检测任务分发至多个工作节点并行处理。

Q: 数据快照占用的存储空间持续增长，如何设置自动清理策略？

A: 系统默认保留最近 30 个快照版本。您可在 app/lib/config.js 中修改 SNAPSHOT_MAX_COUNT 参数调整保留数量，例如设置为 14 或 7 以降低存储占用。快照文件存储在项目根目录下的 snapshots/ 子目录中，建议定期将该目录挂载至外部持久化卷，或通过 logrotate 等工具进行归档压缩。若使用 PostgreSQL 作为后端数据库，还可考虑使用 pg_dump 替代内置快照机制，以利用数据库原生的压缩与备份功能。
