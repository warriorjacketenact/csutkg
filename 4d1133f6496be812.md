# ResourceBridge

ResourceBridge 是一个面向技术社区与开发者的外链资源聚合与导航系统。本项目并非传统的内容管理平台，而是一个高度结构化的技术资源索引中间件，旨在解决信息分散、优质外链难以追溯、社区讨论与外部知识无法有效关联等核心问题。ResourceBridge 定位于技术团队、开源项目维护者、技术博主以及研究者，帮助其将散落在论坛、文档站、代码仓库中的高价值外部链接进行统一收录、分类标注与版本化追踪。

ResourceBridge 不对外部资源内容进行二次存储或快照，而是通过元数据建模与链接状态监测，构建一个可靠、可维护、可审计的外部资源引用图谱。项目内置链接健康检查、标签体系、访问计数与失效告警机制，确保资源列表长期可用且具备可操作性。ResourceBridge 适用于搭建技术文档站的外部链接附录、开源项目 README 的引用索引、团队知识库的参考链接管理中心，以及社区论坛精华帖的归档导航层。

## 功能概览

**链接元数据建模** 支持为每条外链附加标题、描述、标签、收录批次、收录时间、访问次数与最后检查时间等字段，形成结构化的资源条目。

**批量导入与去重** 提供基于文本列表的批量链接导入接口，自动识别并剔除重复 URL，支持按域名或路径规则进行批量标签赋值。

**链接健康状态监测** 内置异步 HTTP 探测任务，定期检查每条链接的可访问性，自动标记失效链接并生成告警通知，支持自定义超时与重试策略。

**多维度分类与筛选** 支持按项目批次、资源类型、技术领域、来源域名等多标签组合筛选，提供 RESTful API 与命令行交互界面两种查询方式。

**访问统计与热度排序** 记录每条链接的点击次数与最近访问时间，支持按热度、收录时间、字母序等多种排序方式输出资源列表。

**导出与嵌入** 支持将资源列表导出为 Markdown 表格、JSON 结构化数据、HTML 无序列表等多种格式，便于嵌入到技术文档、博客文章或项目 README 中。

## 应用场景

**技术文档站的外部链接附录** 技术文档或开源项目手册通常需要引用大量外部资源，如规范文档、参考实现、相关工具等。ResourceBridge 可作为文档站的子模块，统一管理这些外链，并提供自动校验功能，避免文档中出现失效链接。

**社区论坛精华帖的归档索引** 技术社区中高价值讨论帖常包含大量外部引用链接，但这些链接分散在各楼层中，难以检索和维护。ResourceBridge 可将这些链接提取出来，按主题批次进行归档，形成可独立访问的链接导航页，方便社区成员快速定位。

**团队知识库的参考链接管理中心** 研发团队内部知识库在编写技术方案、复盘报告或学习笔记时，会引用大量外部资料。ResourceBridge 提供中心化的链接管理服务，支持多团队共享资源标签体系，减少重复引用与信息冗余。

**开源项目 README 的资源清单生成器** 开源项目维护者需要在 README 中维护项目依赖的生态资源列表，例如相关工具、学习资料、社区论坛等。ResourceBridge 可批量生成符合 Markdown 规范的资源列表，并支持定期同步更新，降低维护成本。

## 快速开始

以下指令适用于 Linux / macOS / Windows WSL 环境，要求系统已安装 Git、Node.js 18.x 及以上版本与 npm 或 yarn 包管理器。

```bash
# 克隆项目仓库
git clone https://github.com/resourcebridge/resourcebridge.git
cd resourcebridge

# 安装项目依赖
npm install

# 以开发模式运行服务，默认监听 127.0.0.1:3000
npm run dev
```

执行上述命令后，ResourceBridge 将启动本地服务实例。用户可通过命令行工具或访问本地 API 端点执行链接导入、查询与管理操作。生产环境部署请参考 `docs/deployment.md` 文档，配置环境变量与数据库连接。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---------|---------|------|
| Node.js | 18.x LTS 或更高 | 项目运行时基础环境，建议使用 LTS 版本以获得长期支持 |
| npm | 9.x 或更高 | Node.js 包管理器，用于安装项目依赖及运行脚本 |
| SQLite3 | 系统内置或由 better-sqlite3 驱动提供 | 默认嵌入式数据库，无需额外安装，适用于开发与小型部署 |
| PostgreSQL | 14.x 或更高（可选） | 生产环境推荐使用，需自行安装并配置连接字符串 |
| Redis | 7.x 或更高（可选） | 用于缓存与链接状态探测任务的分布式锁，提升大规模部署性能 |
| Git | 2.30 或更高 | 用于克隆仓库及版本管理，开发环境必需 |

## 贡献指南

欢迎技术开发者与文档贡献者参与 ResourceBridge 项目建设。请遵循以下步骤提交贡献：

1. 阅读项目行为准则与贡献者协议，在提交 Pull Request 前确保已签署 Contributor License Agreement。协议文本位于项目根目录的 `CLA.md` 文件中。

2. 从 GitHub 仓库派生项目副本到个人账户，并在本地创建功能分支。分支命名请遵循 `feature/功能简述` 或 `fix/问题简述` 格式，例如 `feature/add-json-export`。

3. 编写代码或文档变更后，请确保所有现有单元测试通过，并为新增功能补充对应的测试用例。测试覆盖率不得低于 80%。运行 `npm run test` 执行全部测试套件。

4. 提交变更前运行 `npm run lint` 与 `npm run format` 对代码进行静态检查与自动格式化，确保代码风格符合项目 ESLint 与 Prettier 配置要求。

5. 发起 Pull Request 至主仓库的 `main` 分支，在 PR 描述中清晰说明变更目的、实现方式与影响范围。项目维护者将在 7 个工作日内进行评审。

## 常见问题

**问：ResourceBridge 会存储外部链接指向的页面内容或快照吗？**

答：不会。ResourceBridge 仅存储链接本身的元数据信息，包括 URL、标题、标签、收录时间与健康状态。项目不抓取、不缓存、不存储任何外部页面内容。链接健康状态检测仅通过发送 HTTP HEAD 或 GET 请求验证资源是否可访问，不会解析或保存响应正文。

**问：如何批量更新资源列表中的链接标题与描述信息？**

答：ResourceBridge 提供两种更新方式。第一种是通过命令行工具执行 `npm run cli import -- --update`，该命令会重新读取导入源并匹配已有记录进行增量更新。第二种是通过 RESTful API 的批量更新端点 `PATCH /api/links/batch`，提交 JSON 格式的更新载荷。具体参数格式请参考 `docs/user-guide/batch-update.md` 文档。

**问：链接健康状态检测失败的可能原因有哪些？如何排查？**

答：检测失败通常由以下原因造成：目标服务器响应超时、返回 HTTP 状态码 4xx 或 5xx、DNS 解析失败、SSL 证书过期或不受信任。ResourceBridge 会在检测日志中记录具体的 HTTP 状态码与错误信息。用户可通过 `npm run cli check -- --verbose` 命令手动触发单条链接的详细探测，并查看完整的请求响应头信息以定位问题。
