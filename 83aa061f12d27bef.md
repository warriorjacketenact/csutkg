# LinkVault —— 结构化外链资源管理与导航系统

LinkVault 是一个面向技术团队、内容创作者与研究者设计的轻量级外链资源归集与导航平台。项目定位为“可读可写的外链资源库”，通过 URL 结构化记录、标签分类与全文检索，解决个人与团队在收藏、回溯、共享网络资源时面临的链接失效、分类混乱与协作低效问题。

LinkVault 以静态站点生成器为核心，支持 Markdown 驱动的内容管理，将分散的网络链接转化为可维护的知识资产。项目适用于技术文档站点、开源项目附属资源站、个人知识库外链模块以及轻量级企业内部导航页构建。

## 功能概览

- **链接结构化入库**：支持单条与批量导入，自动解析 URL 元信息，生成标准化记录条目。
- **自定义标签与分类树**：用户可为每条链接分配多级标签，构建按主题、领域或项目归类的灵活分类体系。
- **全文检索引擎**：内置基于倒排索引的检索模块，支持按标题、描述、标签、域名及页面摘要内容快速定位链接。
- **链接健康检查**：定时执行 HTTP 状态探测，标记失效或重定向链接，输出可用性报表。
- **Markdown 数据驱动**：所有链接记录以 Markdown 文件形式存储于 `./data/links` 目录，支持版本控制与外部编辑器直接修改。
- **响应式导航页面生成**：基于配置模板自动生成静态 HTML 导航页，适配桌面与移动端浏览。
- **导入导出兼容性**：支持 JSON、CSV 及 HTML 书签格式的导入导出，便于与其他书签管理工具或团队文档系统交换数据。
- **访问统计与热度排序**：记录链接点击次数，支持按热门、最新、随机排序输出。

## 应用场景

- **技术团队内部文档中心**：开发团队可将项目依赖文档、API 参考、设计规范、运维手册等外链统一归集于 LinkVault，通过导航页实现一站式访问，避免在聊天记录与邮件中反复查找。
- **开源项目资源附录站**：开源项目维护者可将教程、视频、社区讨论、扩展工具等外部资源收录为项目附属导航，置于 GitHub Pages 或独立站点，帮助用户快速了解生态全貌。
- **个人知识库外链模块**：个人笔记体系中的“参考资料”部分可通过 LinkVault 独立管理，每条链接附带摘要、阅读状态与评注，与 Obsidian、Logseq 等知识库软件协同使用。
- **企业合规资源登记**：法务、信息安全部门可对外部引用数据源、第三方 SDK 官网、行业标准页面进行登记与定期可用性审查，满足合规审计中的来源追溯需求。

## 快速开始

以下指令适用于 Linux / macOS / Windows WSL 环境。请确保已安装 Git 与 Node.js 18.x 及以上版本。

```bash
# 克隆项目仓库
git clone https://github.com/your-org/linkvault.git
cd linkvault

# 安装项目依赖
npm install

# 以开发模式启动本地服务，默认监听 3000 端口
npm run dev
```

访问 `http://127.0.0.1:3000` 即可看到导航首页。默认数据目录下已包含示例链接条目，用户可直接修改 `./data/links` 下的 Markdown 文件或通过 Web 管理界面上传新链接。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---------|---------|------|
| Node.js | 18.x 或更高 | 运行时环境，用于执行构建脚本与开发服务器 |
| npm | 9.x 或更高 | 包管理器，用于安装依赖与执行脚本命令 |
| Git | 2.30 或更高 | 版本控制工具，用于克隆仓库与拉取更新 |
| 操作系统 | Linux / macOS / Windows (WSL 2) | 支持主流操作系统，Windows 原生环境未经过完整测试 |
| 磁盘空间 | 至少 200 MB | 包含源码、依赖与生成的静态文件，链接数据量增加时按需扩展 |
| 内存 | 推荐 1 GB 以上 | 构建大型站点或执行批量链接检查时内存影响显著 |
| 浏览器 | 支持 ES6 的现代浏览器 | 管理界面使用 React 构建，需支持最新 ECMAScript 特性 |

## 贡献指南

1. 查阅 `./docs` 目录下的设计文档与 API 约定，确认修改范围。对于较大特性或架构调整，建议先在 Issues 中提出 RFC 讨论，避免无效实现。
2. Fork 本仓库并创建特性分支，分支命名格式为 `feat/功能简述` 或 `fix/问题描述`。提交信息需符合 Conventional Commits 规范，便于自动生成变更日志。
3. 在 `./tests` 中为新增功能或修复编写对应的单元测试与集成测试，确保所有测试用例通过。运行 `npm run test` 执行完整测试套件。
4. 更新受影响的相关文档，包括 README、配置示例与用户手册。文档变更应与代码变更保持同步。
5. 提交 Pull Request 至主仓库的 `main` 分支，描述需包含变更动机、实现方式及影响范围。核心维护者将在 3 个工作日内进行 Review。

## 常见问题

**Q：LinkVault 是否支持私有链接或需要登录鉴权的资源？**

A：当前版本仅针对公开可访问的 URL 进行管理和健康检查。对于需要 Cookie、Session 或 OAuth 认证的资源，建议在外层通过反向代理或 VPN 访问，LinkVault 仅作为导航记录层，不处理认证信息的存储与传递。后续版本将考虑集成凭证管理模块以支持内网资源。

**Q：数据目录迁移或更换服务器后，链接健康状态会丢失吗？**

A：健康检查结果存储于 `./data/meta/health-cache.json` 中，该文件随数据目录一并迁移。迁移后首次运行 `npm run check` 会重新探测所有链接，并更新缓存。若需保留历史状态，可一并备份该文件。

**Q：能否在 LinkVault 中为同一 URL 添加多条记录并分别标记不同标签？**

A：可以。数据层不限制同一 URL 的重复录入，每条记录通过 `id` 字段区分。不同记录可拥有独立的标签、摘要和状态标记，适用于同一资源在不同项目或主题下的分类需要。搜索与展示时会分别列出。
