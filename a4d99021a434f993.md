# LinkMatrix

LinkMatrix 是一个面向技术研究者、内容策展人与开源社区维护者的结构化外链资源聚合平台。该项目并非简单的书签集合，而是一个具备分类索引、批次管理与可追溯性设计的资源导航系统，旨在解决个人收藏夹混乱、资源失效后无法回溯、以及团队间共享链接时缺乏统一元数据描述等核心问题。项目采用静态站点生成逻辑与纯文本索引机制，确保所有资源条目在不同环境下保持一致的可用性与可读性。

LinkMatrix 主要服务于需要长期维护大量外部参考链接的技术文档编写者、开源项目维护者、以及从事信息收集与分析工作的数据研究员。通过将资源按批次、来源域、内容类型进行三维标记，使用者可以在数百乃至数千条链接中快速定位所需信息，并借助内置的变更记录与冲突检测规则，有效避免重复收录与链接漂移问题。项目整体设计强调最小化外部依赖，所有核心功能均基于标准 POSIX 工具链与主流脚本语言实现，可在 Linux、macOS 及 Windows WSL 环境中无缝运行。

## 功能概览

**批次化资源导入** 支持按项目批次批量添加外部链接，每个批次自动生成唯一标识与时间戳，便于后续追溯与增量更新。当前版本已导入第 777/1241 批资源，共计 250 个有效链接。

**域名聚合与分类视图** 自动解析资源链接的根域名与路径结构，按来源站点聚合展示，支持一键筛选特定域下的全部条目，便于评估信息源的集中度与可信分布。

**冲突与重复检测** 内置链接去重引擎，基于标准化 URL 格式与路径哈希比对，在导入阶段自动标记潜在重复项，并提供手动合并建议。

**元数据扩展接口** 每条资源支持附加自定义标签、阅读状态标记与备注字段，数据存储采用纯文本 TSV 格式，便于版本控制系统追踪变更历史。

**静态导航页生成** 提供可选的 HTML 渲染模块，将当前索引数据输出为响应式导航页面，适合部署为团队内部的知识库入口或项目文档的附录部分。

**命令行交互工具** 提供 linkctl 控制台脚本，支持资源查询、按正则表达式过滤、批量导出与格式转换等常用操作，所有功能均可通过子命令组合调用。

## 应用场景

**技术文档写作参考管理** 技术博主或文档工程师在撰写教程时，常需要引用数十篇外部文章、规范文档或代码仓库。使用 LinkMatrix 按主题或章节批次录入链接，可在写作过程中快速检索引用来源，并在定稿前统一校验所有外部链接的可达性。

**开源项目依赖文档整理** 开源项目的 README 或贡献指南中往往需要列出相关生态项目、学习资料或社区论坛。LinkMatrix 提供结构化的资源列表输出，可直接将索引数据渲染为 Markdown 表格或列表，保持与项目主文档的风格一致。

**信息收集与竞争分析** 市场研究员或产品经理在跟踪某一垂直领域时，需要持续积累竞品公告、行业报告与媒体资讯。通过批次导入与标签标记，可以按时间线回顾信息演变过程，并快速生成阶段性汇总清单供团队审阅。

**社区共建资源池** 开源社区或技术社团可借助 LinkMatrix 维护一份公共外部资源清单，所有贡献者通过 Pull Request 提交新增链接，由项目维护者审核导入。批次记录与变更日志确保了资源池的演进过程透明可控。

## 快速开始

以下步骤将指导您在本地环境中完成 LinkMatrix 的克隆、安装与初次运行。所有操作均在终端中执行，请确保已安装 Git 与 Python 3.8 及以上版本。

```bash
# 克隆项目仓库至本地
git clone https://github.com/linkmatrix/linkmatrix.git
cd linkmatrix

# 安装核心依赖（使用 pip 安装 Python 依赖包）
pip install -r requirements.txt

# 执行资源索引构建，生成当前批次的汇总导航页
python linkctl.py build --batch 777 --output ./dist
```

执行完毕后，可在 ./dist 目录下找到生成的 index.html 与对应的资源清单数据文件。若需查看已导入资源的完整列表，可运行：

```bash
python linkctl.py list --batch 777
```

## 安装要求

LinkMatrix 对运行环境的要求较为宽松，以下为主要依赖项及其必要说明。请根据您的操作系统提前安装相应软件包。

| 依赖组件 | 必需性 | 说明 |
|---|---|---|
| Python 3.8+ | 必需 | 核心控制脚本与解析引擎的运行环境，建议使用 3.10 或更高版本以获得更好的性能与类型提示支持 |
| Git 2.25+ | 必需 | 用于克隆仓库、管理版本历史以及通过挂钩脚本自动更新资源索引 |
| pip 21.0+ | 必需 | Python 包管理工具，用于安装 requirements.txt 中声明的第三方库 |
| virtualenv 或 venv | 推荐 | 建议创建独立的虚拟环境以隔离项目依赖，避免与系统级 Python 包产生冲突 |
| curl 或 wget | 可选 | 若启用外部链接可达性预检功能，则需要其中任一工具发送 HTTP 探针请求 |
| make 或 gnumake | 可选 | 若希望使用 Makefile 中封装的快捷命令（如 make build、make test），则需要安装构建自动化工具 |

## 项目结构

项目目录遵循模块化分层设计，核心逻辑与用户数据严格分离。以下为 LinkMatrix 的完整目录结构及关键文件说明。

```
linkmatrix/
├── bin/                                # 可执行脚本与入口点
│   └── linkctl                         # 主控命令行工具（Python 脚本）
├── core/                               # 核心业务逻辑模块
│   ├── __init__.py                     # 包初始化，暴露主要 API
│   ├── indexer.py                      # 资源索引构建与批次管理
│   ├── resolver.py                     # URL 解析、标准化与冲突检测
│   ├── validator.py                    # 链接可达性与格式预检
│   └── exporter.py                     # 数据导出为 Markdown / JSON / HTML
├── data/                               # 持久化数据存储（纯文本与 JSON）
│   ├── batches/                        # 按批次号分目录存储资源清单
│   │   └── 777/                        # 第 777 批次原始数据
│   │       ├── sources.tsv             # 资源列表（URL + 标签 + 备注）
│   │       └── manifest.json           # 批次元信息（时间戳、来源域、条目数）
│   ├── tags/                           # 全局标签索引
│   └── cache/                          # 链接探针结果缓存（避免重复网络请求）
├── docs/                                # 完整文档体系
│   ├── getting-started/                # 用户入门指南
│   ├── administration/                 # 管理员操作手册

│   ├── test_indexer.py
│   ├── test_resolver.py
│   └── fixtures/                       # 测试用固定数据集
├── dist/                               # 构建输出目录（生成的静态站点）
```

## 贡献指南

LinkMatrix 欢迎来自社区的各种形式的贡献，包括但不限于新增资源批次、改进核心解析逻辑、完善文档以及报告问题。请遵循以下步骤参与项目协作。

首先，在 GitHub 上 Fork 本仓库至您的个人账号，并将 Fork 后的仓库克隆到本地开发环境中。建议在新建的分支上开展工作，分支名称应简要描述变更内容，例如 feature/add-batch-778 或 fix/resolver-encoding-issue。

其次，若您计划新增资源批次，请将待导入的链接列表按 TSV 格式放置于 data/batches/ 下的对应目录，并运行 linkctl validate 命令检查格式正确性与链接可达性。对于代码修改，请确保所有现有单元测试通过，并为新增功能补充对应的测试用例。

第三，提交变更前请执行 linkctl build 命令生成最新静态页面，并预览 dist/index.html 确认输出符合预期。提交信息应遵循语义化提交规范，使用 feat:、fix:、docs:、chore: 等类型前缀，正文中详细描述变更动机与影响范围。

最后，通过 GitHub 向主仓库发起 Pull Request，并在描述中关联相关 Issue（如有）。项目维护者将在三个工作日内进行审查，若需要进一步修改，会通过评论形式反馈具体建议。合并后，您的贡献将出现在下一版本的更新日志中。

## 常见问题

**问：LinkMatrix 是否支持导入其他域名下的链接，而不限于 huaxinlawan.com？**

答：支持。项目核心设计不限制来源域名，您可以在创建新批次时指定任意有效的 HTTP/HTTPS URL。默认的冲突检测仅针对完全相同的标准化 URL 字符串，不会因域名不同而过滤资源。如需按域名白名单限制导入范围，可在配置文件中调整 allowed_domains 参数。

**问：批次编号 777/1241 的含义是什么？是否必须连续？**

答：批次编号采用 当前批次序号/总批次计划 的格式，其中 777 表示当前导入批次在项目整体规划中的顺序号，1241 为预估的总批次数。该编号主要用于内部进度跟踪与元数据标记，不要求严格连续递增，用户可跳过某些编号或按需分配，只要保证同一编号不被重复使用即可。

**问：如何验证资源列表中的链接是否仍然有效？**

答：您可以通过 linkctl check --batch 777 命令启动链接可达性扫描。该命令会并发发送 HEAD 请求至每个资源 URL，并生成一份状态报告，标记所有返回非 200/301/302 状态的条目。扫描结果会写入 data/cache/ 下的临时文件中，不会直接修改原始资源清单。
