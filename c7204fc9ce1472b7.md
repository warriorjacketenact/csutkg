# ResourceBridge

ResourceBridge 是一个面向技术团队与独立开发者的外链资源归集与导航系统。该项目不生产内容，而是通过结构化方式将分散于网络各处的优质技术文档、工具站点、学习材料与社区讨论进行统一收录、分类与检索，解决个人收藏夹混乱、团队知识传递效率低、以及技术选型时信息分散的问题。

ResourceBridge 适用于需要长期维护技术知识库的研发团队、开源项目维护者、技术博主以及日常需要查阅大量外部资料的全栈工程师。项目以纯静态资源站形态交付，无需后端服务与数据库，可部署于任意 Web 服务器或对象存储平台。

## 功能概览

**批量链接导入与自动归类** 支持通过文本列表或简易 CSV 格式批量录入外部 URL，系统根据预设规则自动分配至对应分类目录。

**多维度标签过滤与全文检索** 每个资源可附加多个标签（如 "docker"、"security"、"frontend"），并支持基于标题、描述与标签组合的快速过滤。

**资源状态监控与失效标记** 内置定时检测任务（可选）对已收录链接进行可达性检查，对返回 4xx/5xx 状态的资源自动标记为待复核。

**团队共享与权限分级** 提供基于团队空间的资源分组能力，成员可被分配为管理员、编辑者或只读访客，便于控制资源库的变更范围。

**自定义分类层级** 允许用户创建多级分类结构（如 "后端 / 微服务 / 服务发现"），满足不同业务场景下的组织习惯。

**导入导出与标准格式兼容** 支持将资源列表导出为 JSON、YAML 或 Markdown 表格格式，便于迁移、备份或嵌入项目文档。

## 应用场景

**技术团队内部知识库建设** 研发团队可将日常调研中发现的第三方库文档、性能分析工具、故障排查案例统一收录至 ResourceBridge，新成员入职时可快速了解团队常用的技术栈与参考资料，减少重复搜索成本。

**开源项目外部依赖索引** 开源项目维护者可使用 ResourceBridge 整理项目所依赖的上下游仓库、协议说明、构建工具链文档以及社区讨论帖，作为项目 README 或 Wiki 的补充扩展，帮助贡献者快速定位相关资源。

**技术博客与资讯聚合** 技术博主或社区运营人员可将分散在不同平台的深度文章、视频教程、幻灯片与代码示例集中管理，形成按主题划分的阅读列表，提升内容策展效率。

**多项目并行时的上下文切换辅助** 同时参与多个产品或服务的开发者可为每个项目建立独立的资源空间，切换项目时一键过滤出该上下文相关的链接，避免信息干扰。

## 快速开始

以下命令适用于 Linux / macOS / Windows WSL 环境。请确保系统已安装 Git 与 Node.js（版本 18 或以上）。

```bash
# 克隆项目仓库
git clone https://github.com/resourcebridge/resourcebridge.git

# 进入项目目录
cd resourcebridge

# 安装依赖
npm install

# 启动开发服务器（默认端口 3000）
npm run dev
```

执行上述命令后，在浏览器中访问 http://localhost:3000 即可进入 ResourceBridge 实例。首次启动将自动生成示例资源数据用于预览。

## 安装要求

| 依赖 | 必需 | 说明 |
| :--- | :--- | :--- |
| Node.js | 是 | 版本 18.17.0 及以上，推荐使用 LTS 版本 |
| npm 或 yarn | 是 | 包管理工具，npm 版本需 9.0.0 以上 |
| Git | 是 | 用于克隆仓库及版本管理，需 2.25.0 以上 |
| 现代浏览器 | 是 | 支持 ES2020 及 CSS Grid/Flexbox，如 Chrome 90+、Firefox 88+、Edge 90+ |
| 操作系统 | 否 | 无特定限制，Linux、macOS、Windows 均受支持，Windows 下建议使用 WSL2 |
| 外部存储（可选） | 否 | 如需启用资源快照备份，可配置 S3 兼容对象存储或本地文件系统路径 |
| 邮件服务（可选） | 否 | 如需发送资源变更通知或失效告警，需配置 SMTP 服务器参数 |
| 反向代理（生产环境） | 否 | 生产部署时推荐使用 Nginx 或 Caddy 作为前置代理，用于 TLS 终止与静态资源缓存 |

## 贡献指南

欢迎社区贡献者参与 ResourceBridge 的开发与完善。请遵循以下步骤提交贡献：

1. 在 GitHub Issues 中查找标记为 "help wanted" 或 "good first issue" 的任务，或提交新 Issue 说明您希望解决的问题或新增功能。等待维护者反馈后再开始编码，以避免重复工作。

2. 从 main 分支创建新的功能分支，分支命名遵循 `feature/功能简述` 或 `fix/问题简述` 格式。确保本地开发环境已通过全部测试用例，且新增代码包含对应的单元测试。

3. 提交 Pull Request 前请运行 `npm run lint` 与 `npm run test` 确保代码风格与功能正确性。PR 描述中需清晰说明变更目的、影响范围以及测试方式，并关联相关 Issue 编号。

4. 维护者将在 5 个工作日内进行 Code Review。如有修改意见，请及时更新分支。合并后您的贡献将被列入下一版本的发布说明中的贡献者名单。

## 常见问题

**问：导入大量链接（超过 1000 条）时界面出现卡顿，如何优化？**

答：ResourceBridge 默认采用前端分页策略，每页加载 50 条。若导入总数较大，建议分批导入（每次不超过 500 条），或使用命令行导入模式（通过 `npm run import -- --file=links.csv` 执行，该模式绕过 UI 渲染直接写入存储）。同时可调整 `config/default.json` 中的 `pagination.pageSize` 参数降低单次渲染数量。

**问：资源状态检测显示链接失效，但我手动访问是正常的，为什么？**

答：检测模块默认使用 HEAD 请求且超时时间为 3 秒。部分站点对 HEAD 请求返回 405 或 403，但 GET 请求正常。此时可在检测配置中将 `healthCheck.method` 改为 `GET`，并适当增加 `timeout` 值。另外，若站点启用了 Cloudflare 等防爬机制，检测 IP 可能被临时限制，建议配置代理或调整检测间隔。

**问：能否将 ResourceBridge 部署为纯静态站点（无 Node.js 服务）？**

答：可以。运行 `npm run build` 后，`dist/` 目录下会生成完整的静态文件（HTML、CSS、JS）。该构建产物包含所有资源数据（基于构建时的存储快照），适合部署至 Vercel、Netlify 或阿里云 OSS 等静态托管平台。但需要注意的是，静态模式不支持后台检测任务和在线编辑保存功能，如需动态管理请使用完整服务模式。
