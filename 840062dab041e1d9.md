# ResourceBridge

ResourceBridge 是一个面向技术团队与个人开发者的外链资源归集与导航系统。项目定位为轻量级的技术资源网关，通过结构化的索引机制，将分散于不同域名的优质技术文档、工具站、数据源与社区内容整合为统一的访问入口。项目目标用户包括运维工程师、全栈开发者、技术决策者以及开源社区贡献者。ResourceBridge 解决的核心问题是技术信息碎片化导致的检索效率低下，帮助用户在无需记忆大量书签的前提下，快速定位到经过分类与描述的高价值外部资源。

## 功能概览

**统一资源索引** 提供基于多维度标签的资源分类体系，每个外链均附带内容摘要与更新频次标记，方便用户快速评估资源时效性。

**按源域名聚合浏览** 支持按资源来源域名进行分组展示，用户可一键查看同一域名下收录的所有项目页面，便于追溯信息源头。

**项目批次管理** 内置批次标识机制，当前版本对应第 758/1241 批资源入库记录，用户可清晰了解资源库的扩充进度与维护节奏。

**链接状态健康检查** 周期性对已收录 URL 进行可达性探测，自动标记异常链接并生成报告，保证资源列表的有效性。

**快速检索与过滤** 支持基于 URL 关键词、页面标题关键词以及资源编号的全文检索，检索结果按相关度排序并高亮匹配片段。

**结构化数据导出** 支持将当前资源列表导出为 JSON、CSV 或纯文本格式，便于用户二次加工或集成到其他文档系统中。

**外链跳转追踪** 所有外部链接均经过内部路由中转并记录点击次数，为后续资源热度分析提供基础数据，但不影响用户最终访问目标地址。

## 应用场景

**技术团队内部知识库外链模块** 技术负责人可以将 ResourceBridge 部署为团队知识库的子模块，专门负责管理所有引用到的外部技术博客、官方文档与依赖库主页。团队成员在查阅内部文档时，可通过 ResourceBridge 统一跳转，避免外链散落于各篇 Markdown 文件中难以维护。

**开源项目 README 资源附录自动生成** 开源项目维护者可以使用 ResourceBridge 管理项目依赖的所有第三方链接，并一键生成格式规范的附录章节。项目批次号与资源列表可直接嵌入项目主页，便于用户回溯信息来源。

**个人开发者的书签替代方案** 前端或后端开发者可将日常高频访问的 API 参考站、CDN 查询工具、正则测试平台等统一录入 ResourceBridge，通过本地部署获得一个无广告、无追踪的纯净导航页，替代浏览器自带书签栏的杂乱管理方式。

**技术调研阶段的信息收集** 在进行技术选型或竞品分析时，调研人员可快速将临时发现的参考链接收录到 ResourceBridge 的指定批次中，并添加备注标签。调研结束后，整个批次的资源可整体导出并移交评审组。

**社区资源共建共享** 开源社区或技术兴趣小组可以公用一个 ResourceBridge 实例，成员各自提交优质外链，由维护者审核后入库。资源列表随项目批次更新而滚动增长，形成社区特有的知识资产。

## 快速开始

以下操作以 Linux/macOS 环境为例，Windows 用户建议使用 WSL2 或 Git Bash 执行。

```bash
# 克隆项目仓库
git clone https://github.com/resourcebridge/resourcebridge.git
cd resourcebridge

# 安装项目依赖（基于 Python 3.10+ 与 pip）
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 初始化本地数据库并启动开发服务器
python manage.py migrate
python manage.py runserver --port 8000
```

启动成功后，在浏览器中访问 http://127.0.0.1:8000 即可进入资源导航主页。首次启动会自动载入当前批次（第 758/1241 批）的种子资源数据。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.10 / 3.11 / 3.12 | 核心运行环境，推荐使用 3.11 以获得最佳性能 |
| Django | 4.2 LTS | Web 框架，用于提供路由、ORM 与管理后台 |
| SQLite | 3.35+ | 默认轻量级数据库，适合单机或小规模部署；生产环境可换用 PostgreSQL |

| gunicorn | 21.2+ | 生产环境 WSGI 服务器，仅在部署到公网时需要 |
| nodejs | 18.x | 用于前端静态资源构建（仅开发模式下需要） |
| git | 2.30+ | 版本控制工具，用于克隆仓库与后续更新拉取 |

## 贡献指南

1. 复刻项目仓库至个人账号，并在本地创建新功能分支。分支命名规范为 feature/功能简述 或 fix/问题编号，避免直接在 main 分支上修改。

2. 运行本地开发环境并执行全部现有单元测试，确保当前基线通过。测试套件位于 apps/*/tests/ 目录下，使用 pytest 驱动。新增功能需附带对应的测试用例，覆盖率不低于 80%。

3. 提交代码前执行代码格式化与静态检查。项目使用 black 作为 Python 格式化工具，flake8 作为语法风格检查器，pre-commit 钩子已配置于仓库根目录。提交信息需遵循 Conventional Commits 格式，即 type(scope): subject 的结构。

4. 发起 Pull Request 至主仓库的 develop 分支，并在描述中注明关联的 Issue 编号或功能需求。PR 至少需要一名项目维护者进行 Code Review，讨论通过后由维护者合并并关闭对应 Issue。

5. 文档更新与贡献同步。若新增功能涉及配置项变更或新增 API 端点，需同步更新 docs/ 目录下的对应手册，并确保 README 中的快速开始步骤仍然有效。

## 常见问题

Q: 启动服务后访问首页看不到任何资源列表，如何处理？
A: 请首先确认是否执行了数据迁移命令 python manage.py migrate 以及是否载入种子数据。默认情况下项目首次启动不会自动导入任何外部链接，需要手动执行 python manage.py import_batch --batch 758 来加载当前批次资源。若导入后依然无显示，检查 logs/error.log 中是否有数据库读写权限相关的报错。

Q: 健康检查任务一直处于 pending 状态，从未更新链接状态，可能是什么原因？
A: 该功能依赖 redis 服务与后台任务进程。请先确认 redis 已在本地或指定地址正常启动，并检查 settings.py 中 CACHES 与 RQ_QUEUES 配置是否匹配实际 redis 连接参数。若 redis 正常但任务未执行，请单独启动任务处理进程：python manage.py rqworker healthcheck，观察终端输出是否有报错。

Q: 我可以在生产环境中使用 SQLite 吗？
A: 对于个人使用或小团队内部部署（并发用户数低于 10），SQLite 完全可以满足需求。但若需要支持多用户并发写入、高频检索或较大的资源列表（超过 5 万条记录），强烈建议切换至 PostgreSQL。切换方式为修改 settings.py 中的 DATABASES 配置，并执行 python manage.py migrate --database=default 重新迁移。
