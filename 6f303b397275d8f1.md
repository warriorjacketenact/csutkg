# TechIndex Resource Aggregator

TechIndex Resource Aggregator is a curated, community-driven technical documentation and external resource indexing system designed for developers, technical writers, and open-source maintainers who need a structured, searchable, and version-controlled repository of high-value technical references. The project addresses the common problem of scattered, unmaintained bookmarks and fragmented learning materials by providing a unified catalog with semantic categorization, automated metadata extraction, and a lightweight static-site generation pipeline.

Target users include backend engineers, DevOps practitioners, frontend architects, and technical team leads who require rapid access to authoritative content across programming languages, system design, cloud infrastructure, and software architecture. Unlike general-purpose bookmark managers, TechIndex applies domain-specific taxonomies and maintains a strict quality filter, ensuring that every listed resource meets documented inclusion criteria.

## 功能概览

**Semantic URL Classification** - Each indexed resource is automatically tagged with a category prefix derived from its content path (e.g., /content/jih/ for distributed systems, /content/gkn/ for cloud-native tooling), enabling faceted browsing without database dependencies.

**Markdown-based Metadata Store** - All resource records are stored as plain Markdown files with YAML frontmatter, allowing full Git history tracking, offline access, and seamless integration with static site generators like Hugo or MkDocs.

**Batch Import Pipeline** - Supports CSV, JSON, and plain-text URL list imports with deduplication, broken link detection, and schema validation against a predefined resource model (title, description, category, tags, submission date).

**Tag-based Relevance Scoring** - Implements a lightweight term frequency-inverse document frequency (TF-IDF) based ranking algorithm to surface the most relevant resources for a given search query, computed entirely at build time to avoid runtime database overhead.

**Automated Health Checks** - A scheduled CI workflow (GitHub Actions) validates all indexed URLs for HTTP status 200, checks SSL certificate validity, and alerts maintainers via issue creation when a resource becomes unreachable.

**Static Export with Search** - Generates a fully static HTML site with client-side search (lunr.js or Pagefind) that works without JavaScript fallback, suitable for hosting on GitHub Pages, Cloudflare Pages, or any static hosting service.

**Community Submission Workflow** - External contributors can propose new resources via pull request against the /drafts directory; a maintainer bot validates format and runs security scans before merging.

## 应用场景

**Technical Onboarding for New Team Members** - Engineering leads can point new hires to the curated "Essential Reading" category within TechIndex, covering coding standards, architecture decision records, and infrastructure runbooks. This reduces ramp-up time by providing a single source of truth for company-aligned technical references.

**Open-Source Project Documentation Enrichment** - Maintainers of open-source projects can embed TechIndex as a submodule or reference in their README to offer users a curated list of complementary tools, tutorials, and case studies without manually maintaining a "Resources" section.

**Offline Knowledge Base for Air-Gapped Environments** - Teams operating in restricted networks can clone the entire TechIndex repository and build the static site locally, ensuring that all referenced external documentation is available as cached HTML snapshots or via local mirrors.

**Academic Research and Literature Review** - Graduate students and researchers in computer science can use TechIndex to track evolving best practices in distributed databases, consensus algorithms, and performance optimization, with timestamped entries that enable historical trend analysis.

**Compliance and Audit Artifact** - Organizations subject to regulatory frameworks (e.g., SOC2, ISO 27001) can treat TechIndex as an auditable log of approved external references, with every addition or removal captured in the Git commit history along with reviewer signatures.

## 快速开始

Below are the minimal steps to clone the repository, install dependencies, and run the indexing pipeline locally.

```bash
git clone https://github.com/techindex-community/techindex-aggregator.git
cd techindex-aggregator

# Install Python dependencies (requires Python 3.9+)
pip install -r requirements.txt

# Run the build script to generate static site from /content directory
python build.py --input ./content --output ./dist --validate-links

# Start local development server
python -m http.server --directory ./dist 8000
```

## 贡献指南

We welcome contributions from the community. Please follow the steps below to ensure a smooth review process.

1. **Fork the repository and create a feature branch** - Fork the main repository to your GitHub account, then create a branch with a descriptive name such as `feature/add-resource-category-jih` or `fix/broken-link-validation`.

2. **Add or update resources in the appropriate category directory** - Place new resource Markdown files under the correct /content/ subdirectory. Each file must include valid YAML frontmatter with at least `title`, `description`, `category`, and `tags` fields. Run `python build.py --validate` locally to check your changes.

3. **Write or update tests for any code changes** - If your contribution involves modifications to the build pipeline or validators, add corresponding unit tests under /tests/unit/. Ensure all existing tests pass by running `pytest` before committing.

4. **Submit a pull request against the main branch** - Open a pull request with a clear title and description referencing the issue number (if applicable). The PR template must be filled out completely, including the checkboxes for DCO sign-off and documentation updates.

5. **Address review feedback and await merge** - Maintainers will review your PR within 5 business days. CI checks must pass before merging. Once approved, a maintainer will squash-merge your changes and update the resource index accordingly.

## 常见问题

**Q: How often are the indexed URLs validated for availability?**

A: The validation workflow runs automatically every 24 hours via GitHub Actions. Additionally, a manual validation can be triggered by commenting `/validate` on any pull request. If a URL returns a status other than 200 or exhibits a redirect chain longer than 3 hops, an issue is created in the repository with the label `broken-link`. Maintainers then attempt to find an updated URL or archive the resource.

**Q: Can I host a private fork of TechIndex with restricted access?**

A: Yes. TechIndex is MIT-licensed and fully self-contained. You can clone the repository and deploy it on an internal Git server or static hosting service. The build pipeline does not require any external network connectivity beyond the optional health-check feature, which can be disabled via the `--skip-validation` flag. For air-gapped environments, we recommend cloning all external resources as offline snapshots using the included `scripts/mirror_resources.sh` utility.

**Q: How are categories determined for new resources, and what happens if no category matches?**

A: Category assignment is based on the first-level path segment in the URL (e.g., /content/jih/). If a resource does not fit any existing category, a maintainer will either create a new subdirectory following the naming convention (three-letter mnemonic) or place the resource in the /content/misc/ directory. Contributors are encouraged to propose new categories via the issue tracker, providing at least three existing resources that would belong to the proposed category to demonstrate sufficient content density.

**Q: What is the backup and disaster recovery strategy for the resource index?**

A: The primary index is stored in Git, which provides distributed redundancy. All contributors have a full clone of the historical data. Additionally, the GitHub repository is backed up nightly to an S3-compatible object store via the `.github/workflows/backup.yml` workflow. The backup includes the entire /content/ directory, the generated /dist/ static site, and a JSON export of all metadata. Restore procedures are documented in /docs/maintainers/disaster-recovery.md.
