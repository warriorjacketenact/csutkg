# LinkMatrix

LinkMatrix 是一个面向技术社区与内容创作者的轻量级外链资源汇总与导航系统。项目定位于为开发者、运维工程师、技术博主以及开源项目维护者提供一套可自托管的、结构清晰且具备分类管理能力的资源聚合平台，帮助用户从分散的浏览器书签、临时文档、即时通讯记录中解脱出来，建立属于自己的高质量技术外链知识库。

LinkMatrix 不依赖任何前端框架，核心基于静态 Markdown 与 Shell 脚本生成索引页，同时提供可选的 Node.js 辅助服务用于实现全文检索与自动分类打标。项目本身可作为个人知识管理工具，也可作为团队内部文档导航入口，甚至可作为中小型开源社区的资源聚合页基础模板。通过标准化 URL 格式与分类标签体系，LinkMatrix 能够将大量未归类的外部链接快速转化为可检索、可分享、可审计的 structured resource catalog。

## 功能概览

**统一资源注册与标识映射** 每条资源使用固定格式的短码与资源 ID 进行注册，确保链接在迁移与重索引过程中的稳定可追溯性。

**自动分类与标签生成** 基于 URL 路径特征、资源 ID 哈希以及用户自定义规则，系统可自动为每条链接生成初步分类建议与标签组合。

**多层级索引视图** 提供按资源 ID 排序、按类别分组、按添加时间倒序等多种索引页面，满足不同使用习惯下的快速定位需求。

**全文检索与模糊匹配** 内置基于倒排索引的轻量级搜索引擎，支持对资源 URL、ID、描述字段进行关键字检索与模糊匹配。

**静态站点生成模式** 所有索引页与详情页均支持导出为纯静态 HTML，可直接部署至对象存储、CDN 或任意 Web 服务器。

**外部链接健康检查** 提供可定时执行的链接可用性检查脚本，自动标记失效或响应异常的条目，辅助资源库维护。

**权限分级与私有标记** 支持为每条资源设置公开、内部、私有三级可见性标记，便于团队协作场景下的内容管理。

**批量导入与导出** 支持从 CSV、JSON、浏览器书签 HTML 文件批量导入外部链接，同时支持按筛选条件导出资源列表。

## 应用场景

个人技术博客作者可利用 LinkMatrix 建立自己的推荐阅读列表与工具链导航，将散落在浏览器收藏夹中的数百个开发工具、文档站点、在线服务链接统一归档，并为每一条链接添加个人备注与使用心得，最终生成可直接嵌入博客页面的资源索引页。

中小型研发团队可使用 LinkMatrix 维护内部文档导航与运维手册引用体系。例如，将 CI/CD 流水线地址、监控面板链接、日志查询入口、数据库管理界面、内部知识库文章等按照业务模块分类归档，配合私有标记功能确保敏感链接仅对团队成员可见。

开源项目维护者可利用 LinkMatrix 整理项目依赖的外部参考资源，包括协议规范、API 参考文档、相关社区讨论、插件生态列表等，并将生成的资源页面作为项目官方文档的补充附录，降低新贡献者的学习成本与技术调研时间。

技术社区运营方可使用 LinkMatrix 构建周报或月刊的外部链接汇编页，每期整理数十条优质技术文章、视频教程、工具更新、会议演讲录像等链接，通过自动分类与标签功能快速生成专题索引，替代传统的手工排版方式。

## 快速开始

以下指令适用于 Linux 与 macOS 环境，Windows 用户建议使用 WSL2 或 Git Bash 执行。

```bash
# 克隆项目仓库
git clone https://github.com/linkmatrix/linkmatrix.git
cd linkmatrix

# 安装依赖（Node.js 辅助工具 + Python 检查脚本）
npm install --production
pip install -r requirements.txt  # 可选，用于链接健康检查

# 执行初始化数据构建与索引生成
./scripts/build.sh --input ./data/raw/urls.txt --output ./dist
```

执行完成后，所有生成的索引页面与静态资源将输出至 `./dist` 目录。若需启动本地预览服务，可运行 `npm run serve`，默认监听 8080 端口。

## 贡献指南

我们欢迎并鼓励社区贡献，无论您是提交问题报告、改进文档、还是贡献核心代码。请遵循以下流程以确保协作顺畅。

首先，在开始任何实质性工作之前，建议先查阅 issue 列表，确认当前是否存在相关讨论或已有进行中的工作。若无重叠，可创建一个新的 issue 描述您计划解决的问题或新增的功能，以便维护者与其他贡献者同步信息。

其次，所有代码变更应基于最新的 main 分支创建特性分支，分支命名建议采用 `feature/描述` 或 `fix/描述` 格式。提交信息请遵循 Conventional Commits 规范，确保提交历史清晰可读。

第三，若您贡献的是代码，请确保新增或修改的部分包含相应的单元测试，并通过现有的全部测试用例。若您贡献的是文档，请确保 Markdown 格式正确且链接有效。所有变更在合并前需要通过持续集成流水线的检查，包括代码风格检查、测试覆盖率与构建验证。

第四，完成变更后，请向 main 分支发起 Pull Request，并在描述中关联对应的 issue 编号，简要说明变更内容、测试情况以及可能存在的兼容性影响。维护者会在合理时间内进行 Review 并给出反馈。

最后，对于非代码类贡献，如反馈使用体验、提出改进建议、分享应用案例或推广项目，我们同样深表感谢。您可以通过 issue 或讨论区参与交流。

## 常见问题

问：LinkMatrix 是否必须使用 Node.js 运行所有功能？能否仅依赖静态构建模式？

答：Node.js 主要用于索引构建、检索服务和模板渲染。如果您仅需要生成静态页面，可以在安装依赖后运行一次构建脚本，然后将 `dist` 目录下的静态文件部署至任意 HTTP 服务器，后续访问无需 Node.js 运行时。但检索功能和实时预览服务仍依赖 Node.js。

问：资源链接的健康检查会对外部站点造成请求压力吗？如何控制检查频率？

答：健康检查脚本默认以串行方式发送 HEAD 请求，且每个请求之间带有 200 毫秒的延迟间隔。您可以通过配置文件的 `health_check.interval` 字段调整检查周期，建议在生产环境中设置为每周执行一次，并避开业务高峰时段。检查结果仅记录在本地缓存中，不会自动重试或并发请求。

问：如何迁移 LinkMatrix 实例到另一台服务器？数据迁移需要注意哪些内容？

答：迁移时需打包整个 `data/` 目录、`config/custom.yaml`（若存在）以及 `dist/` 目录（若需要保留历史构建产物）。在新服务器上完成代码部署与依赖安装后，将上述目录覆盖至对应位置，重新执行构建脚本即可恢复所有资源索引。若检索索引库 `data/registry/index.db` 的 SQLite 版本存在兼容性差异，可删除该文件后通过 `scripts/build.sh --rebuild-index` 重新构建。
