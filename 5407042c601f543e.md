# ResourceBridge

ResourceBridge 是一个面向技术社区的开源外链资源归集与导航系统。该项目并非传统的搜索引擎或网页爬虫，而是一个由人工筛选并经由社区投票验证的高质量技术文章、工具站点与开发文档的索引仓库。ResourceBridge 服务于开发者、技术博主与架构师，帮助其在不依赖商业推荐算法的情况下，快速定位特定领域（如分布式系统、编译原理、数据库内核、前端工程化）的深度阅读材料与参考实现。

ResourceBridge 的核心定位是“技术阅读的慢车道”。在信息碎片化与快餐式内容泛滥的背景下，本项目坚持收录具有长期阅读价值（Evergreen Content）的硬核技术输出。所有收录条目均附带简短的上下文标注，便于使用者判断内容的相关性与时效性。项目本身不存储任何第三方版权内容，仅提供元数据与引用链接。

## 功能概览

- **按技术领域分类索引**：资源按照后端工程、基础设施、前端架构、人工智能、运维监控等一级分类进行组织，每个分类下进一步细化至二级标签，例如“分布式共识算法”、“JVM 字节码操纵”、“React 并发模式”等。

- **资源成熟度标记**：每条外链附带成熟度标记，包括“概念入门”、“工程实践”、“深度剖析”、“官方草案”与“历史归档”，帮助读者根据自身水平筛选阅读材料。

- **社区投票与评论集成**：支持通过 GitHub Issues 或 Discussions 对已收录资源进行投票与点评。项目维护者定期根据社区反馈调整资源的推荐指数与分类归属。

- **全文元数据检索**：基于外链标题、描述、标签与收录编号构建轻量级全文检索能力，支持按关键字、分类、收录批次进行组合过滤。

- **自动化链接健康检查**：通过 GitHub Actions 定时执行外链可用性检查，自动标记失效链接并生成报告，确保资源列表的长期可用性。

- **外链收录批次管理**：所有资源以批次为单位进行收录与发布，便于追踪项目进展与贡献者工作量。

## 应用场景

- **技术选型调研**：当架构团队需要评估多个候选中间件（如消息队列、配置中心、服务网格）时，可通过 ResourceBridge 查阅相关领域的对比分析文章与生产环境案例，节省在搜索结果中筛除低质内容的时间。

- **新人入职培训路径制定**：团队技术负责人可根据 ResourceBridge 中的“入门”与“工程实践”标记，为新入职的开发工程师快速组装一套涵盖编码规范、调试技巧、性能分析、持续集成等方面的阅读清单。

- **技术文章写作参考**：技术博主在撰写深度文章前，可通过本项目查阅同一主题下已有文章的切入角度、论证结构与实验数据，以避免重复造轮子或偏离行业共识。

- **开源项目文档补充**：开源项目维护者可将 ResourceBridge 中的相关外链作为项目 README 或 Wiki 的“延伸阅读”部分来源，降低用户的学习曲线。

## 快速开始

以下步骤帮助您在本地环境快速运行 ResourceBridge 的静态站点生成与检索服务。

```bash
# 克隆项目仓库
git clone https://github.com/resourcebridge/resourcebridge.git

# 进入项目目录
cd resourcebridge

# 安装项目依赖（基于 Node.js 22 LTS 与 pnpm）
pnpm install

# 构建资源索引并启动开发服务器
pnpm run build:index
pnpm run dev
```

执行上述命令后，访问本地端口 3000 即可查看资源导航界面。索引文件生成于 `data/index.json`，可直接被前端检索模块消费。

|------|------|------|
| Node.js 22.x LTS | 是 | 运行时环境，用于执行索引构建脚本与开发服务器 |

| docker 24+ | 否 | 仅当使用容器化部署方案时需要 |
| nginx 1.24+ | 否 | 仅当生产环境作为静态文件服务器时需要 |

## 贡献指南

ResourceBridge 严格遵循开源社区协作规范。所有贡献者需遵守行为准则与提交协议。

1. **外链提名**：通过 GitHub Issue 提交新资源提名，需包含标题、原文链接、所属分类与推荐理由。提名模板位于 `.github/ISSUE_TEMPLATE/resource-nomination.md`。维护者将在 5 个工作日内进行初审。

2. **元数据修正**：若发现已有资源的标题、标签或描述存在错误，欢迎提交 Pull Request 修改 `data/batches/` 下对应的 JSON 文件。修改需附带修改说明，并确保 JSON 格式有效。

3. **分类体系优化**：若您对分类架构有改进建议（如新增标签、合并重叠分类），请先通过 Discussion 发起设计提案，待社区达成初步共识后再提交代码变更。

4. **代码与文档贡献**：任何涉及 `src/` 或 `docs/` 的修改，请确保通过 `pnpm run test` 与 `pnpm run lint` 检查。提交信息遵循 Conventional Commits 规范。

5. **失效链接报告**：若通过正常访问发现某外链已失效，可直接在对应资源详情页下方评论区标记，或通过 GitHub Issue 报告。维护者每月根据健康检查报告统一处理。

## 常见问题

**问：ResourceBridge 与书签管理器或 Pocket 等工具的差异是什么？**

答：书签管理器与稍后读工具侧重于个人化收藏与时间管理，而 ResourceBridge 是一个面向公众的、经过分类与投票筛选的共享资源索引。它不依赖于个人标签习惯，而是采用社区共识的分类法，并明确标记每项资源的阅读门槛与内容类型。此外，ResourceBridge 提供批量导出与 API 查询能力，便于团队集成。

**问：如何确保收录链接的长期有效性？**

答：项目通过 GitHub Actions 每 72 小时执行一次全量链接健康检查。检查结果在 `data/health-reports/` 目录下生成 JSON 报告。对于连续三次检查失败的链接，系统会自动标记为“疑似失效”并在前端界面予以提示。维护者会根据报告决定是否保留该条目或寻找替代镜像。

**问：我能否将 ResourceBridge 的索引数据用于自己的项目中？**

答：可以。索引数据以 JSON 格式开放，并采用 MIT 许可证发布。您可将其用于构建自定义导航站、技术雷达或知识图谱。使用时建议保留原始出处标识，但非强制要求。
