# ResourceBridge

ResourceBridge 是一个面向技术社区与开发者的外链资源归集与导航系统。项目定位为轻量级、可自托管的资源目录中间件，帮助技术团队、内容创作者与开源维护者将散落在多个外部平台的高质量参考链接统一收录、分类、版本化管理，并以结构化方式对外提供访问入口。

项目不提供资源文件存储，不依赖外部数据库，所有资源记录以纯文本形式组织，便于版本控制、协作审校与自动化流水线集成。ResourceBridge 适用于需要长期维护大量外链引用、同时要求低运维成本与高可读性的场景，例如技术文档站点的参考附录、开源项目的外部依赖索引、或社区周报的链接素材库。

## 功能概览

- **多源链接归集** 支持将不同域名、不同路径结构的外部链接统一收录至项目资源清单中，保留原始 URL 不做任何改写，确保引用来源的准确性与可追溯性。

- **分类目录组织** 提供基于资源类型、来源域名或业务场景的目录划分能力，资源列表可按小节分组呈现，便于浏览与定位。

- **纯文本可版本化** 所有资源记录以 Markdown 格式维护于仓库中，原生支持 Git 版本管理，可追溯每次增删改操作，支持代码审查与变更记录。

- **零运行时依赖** 项目本身为静态文档站点，不依赖后端服务、数据库或外部 API，构建后可直接部署至任意静态托管服务。

- **自动化链接检查** 集成 CI 工作流，定期对收录的 URL 进行可达性检测，标记失效链接并生成报告，辅助维护者及时更新。

- **资源条目注释能力** 每条链接支持附加简短说明文字，标注资源用途、所属模块或推荐理由，便于团队协作与后续审阅。

- **多格式导出** 资源清单可导出为纯 Markdown 表格、JSON 结构或 HTML 目录页，满足不同下游系统的数据消费需求。

## 应用场景

**技术文档站的参考附录维护** 技术文档通常需要引用大量外部规范、论文、工具仓库或教程链接。ResourceBridge 可作为文档仓库的子模块，集中管理所有外链，文档正文仅保留引用标识符，由 ResourceBridge 统一渲染附录页面，降低文档正文的冗余程度。

**开源项目的依赖索引与致谢清单** 开源项目往往依赖诸多上游库、灵感来源或参考实现。维护者可使用 ResourceBridge 记录所有外部引用，形成结构化的致谢与依赖说明页面，满足合规要求的同时提升项目的透明度。

**社区周报或月刊的素材库** 技术社区运营人员每周需要整理大量值得关注的文章、发布、工具更新。ResourceBridge 可作为内部素材暂存库，按周或按月分类存放链接，最终一键生成对外发布的阅读清单页面。

**内部团队的知识收藏夹** 企业研发团队可将 ResourceBridge 部署在内网，作为团队共建的技术收藏夹，收录踩坑记录、性能调优案例、架构方案参考等链接，形成长期积累的组织知识资产。

**自动化资源监控的基线配置** 运维或 QA 团队可将 ResourceBridge 作为监控系统的输入源，定期检测所有收录链接的可用性与响应时间，及时发现外部资源变更或下线情况。

## 快速开始

以下命令演示如何在本地环境获取项目代码、安装依赖并启动开发服务。

```bash
# 克隆仓库
git clone https://github.com/resourcebridge/resourcebridge.git
cd resourcebridge

# 安装依赖（项目基于 Node.js 与 npm 构建）
npm install

# 启动本地开发服务，默认监听端口 3000
npm run dev
```

执行上述命令后，可在浏览器中访问 http://localhost:3000 查看资源目录首页。项目默认加载 `resources` 目录下的示例数据，用户可直接编辑或替换该目录下的 Markdown 文件以更新资源内容。

## 安装要求

| 依赖 | 必需 | 说明 |
|------|------|------|
| Node.js 18.x 或更高 | 是 | 项目构建与开发服务运行时环境 |
| npm 9.x 或更高 | 是 | 包管理器，用于安装项目依赖 |
| Git 2.30 或更高 | 是 | 用于克隆仓库与版本管理 |
| 现代浏览器（Chrome / Firefox / Edge 最新两版） | 否 | 仅本地预览时需使用，生产构建为静态 HTML |
| CI / CD 平台（GitHub Actions / GitLab CI） | 否 | 如需启用自动化链接检查功能，需配置 CI 环境 |

## 常见问题

**问：ResourceBridge 是否必须部署为 Web 服务才能使用？**

答：不是。ResourceBridge 的核心能力基于静态 Markdown 文件组织，即使不启动 Web 服务，用户也可以直接在本地编辑 `resources/` 目录下的文档，并通过 Git 进行版本管理。Web 服务仅用于提供可视化目录页面的浏览便利，属于辅助功能。

**问：如何批量导入已有的大量链接？**

答：项目提供了 `scripts/import.js` 辅助脚本（需自行创建或从社区获取），支持从 CSV 文件或 JSON 数组批量生成符合格式规范的 Markdown 索引文件。用户也可直接按照 `resources/` 目录下现有文件的格式手动编写，每行一个链接，支持附加说明文字。

**问：链接检查脚本检测到失效链接后如何处理？**

答：CI 工作流每次运行后会生成检查报告，以 GitHub Actions 的 Job Summary 形式呈现。报告中将列出所有返回非 200 状态码的 URL 及其响应状态。维护者根据报告在本地编辑对应的索引文件，删除或替换失效链接，提交变更后重新触发检查以确认修复。
