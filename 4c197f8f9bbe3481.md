# LinkVault

LinkVault 是一个面向技术团队与内容创作者的轻量化外链资源聚合与导航系统。该项目定位于解决分散在多处的技术文档、工具站点、参考手册与社区讨论链接难以统一管理和快速检索的问题。LinkVault 不提供爬虫或自动采集功能，而是通过结构化目录与标签体系，帮助用户将零散的 URL 资源整理为可维护、可分享、可版本控制的知识库。目标用户包括运维工程师、全栈开发人员、技术写作团队以及内部开发者门户的维护者。

## 功能概览

- **分级目录导航**：支持无限层级的自定义分类，用户可按项目、技术领域或使用频率组织链接，每个分类可单独设置可见范围与描述。
- **批量链接导入**：通过纯文本或 CSV 格式批量添加 URL，系统自动识别重复条目并提示合并，支持手动标签补全。
- **全文检索与过滤**：基于链接标题、描述、标签及备注内容的轻量级全文搜索，同时支持按分类、添加时间或访问热度进行多维度筛选。
- **链接健康检查**：定时对已收录的 URL 发送 HEAD 请求，检测状态码变化，对返回 4xx 或 5xx 的链接生成告警报告，避免资源失效。
- **访问统计看板**：记录每个链接在本地的点击次数与最后访问时间，支持按周或按月汇总热门链接排行，辅助团队识别高频资源。
- **导入导出标准格式**：支持将整个链接库导出为 JSON、YAML 或 HTML 书签文件，也支持从浏览器书签导出文件或 Markdown 列表文件进行导入。
- **团队协作注释**：允许具有编辑权限的成员为任意链接添加内部注释或使用指南，注释内容支持 Markdown 格式，便于记录踩坑经验或版本适配说明。

## 应用场景

- **技术文档中心导航**：开发团队可将内部 Wiki、API 文档、设计稿原型、CI/CD 流水线地址和代码仓库按项目阶段分类，作为开发者门户的快捷入口，减少上下文切换时间。
- **运维监控汇总页**：运维人员将各类监控面板、日志查询系统、告警管理后台和云服务控制台集中管理，配合健康检查功能快速发现因证书过期或配置变更导致的不可用服务。
- **开源项目资源整理**：开源维护者使用 LinkVault 收集相关生态工具、社区论坛、贡献指南和版本发布说明，为贡献者提供统一的起点，降低新成员上手门槛。
- **技术培训与学习路径**：技术培训师可按课程章节整理参考文章、视频教程、在线练习环境和官方手册，形成结构化的学习资源包，方便学员课后自学。
- **多项目切换工作区**：自由职业者或咨询顾问为不同客户建立独立的工作区，每个工作区包含专属的工具链链接和项目文档地址，避免混淆且切换迅速。

## 快速开始

以下步骤适用于 LinkVault 的本地开发或生产部署。请确保系统已安装 Git、Node.js 和 npm。

```bash
# 克隆项目仓库
git clone https://github.com/linkvault/linkvault.git
cd linkvault

# 安装项目依赖
npm install

# 启动开发服务器，默认监听 3000 端口
npm run dev
```

访问控制台后，系统会引导创建管理员账户。完成初始配置后，即可进入仪表盘开始创建分类并添加链接。生产环境部署请参考文档导航中的部署指南。

## 安装要求

| 依赖 | 必需 | 说明 |
|---|---|---|
| Node.js 18.x 或 20.x LTS | 是 | 运行时环境，推荐使用 Active LTS 版本以确保兼容性 |
| npm 9.x 或 yarn 1.22+ | 是 | 包管理器，用于安装项目依赖和执行脚本 |
| SQLite 3 或 PostgreSQL 14+ | 是 | 默认使用 SQLite 进行轻量级存储，生产环境建议切换至 PostgreSQL |
| Redis 6.2+ | 否 | 用于缓存统计数据和会话存储，高并发场景建议启用 |
| Nginx 或 Caddy | 否 | 反向代理服务器，用于生产环境提供 TLS 终止和静态资源缓存 |
| Docker 20.10+ | 否 | 容器化部署方案，可使用提供的 Dockerfile 构建镜像 |
| Git 2.25+ | 开发时必需 | 版本控制工具，用于克隆仓库和管理代码更新 |

## 贡献指南

1. 在 GitHub 上 Fork 本仓库，并克隆到本地开发环境。建议使用 Node.js 20 LTS 版本，并开启 corepack 以使用最新版 yarn 作为包管理器。
2. 创建新的功能分支，分支命名规范为 `feature/功能简述` 或 `fix/问题简述`。提交代码前请确保运行 `npm run lint` 和 `npm run test` 全部通过，且新增功能包含对应的单元测试。
3. 若涉及数据库 Schema 变更，请在 `src/db/migrations` 目录下新建迁移脚本，并提供回滚方案。迁移脚本需同时包含 `up` 和 `down` 方法。
4. 提交 Pull Request 时请参照 PR 模板填写变更摘要、测试覆盖情况以及是否影响已有 API 兼容性。核心贡献者会在 48 小时内进行 Code Review。
5. 文档类贡献（包括修正拼写错误或补充使用示例）同样欢迎，请直接在 `docs/` 目录下修改对应的 Markdown 文件，并确保中文与英文之间保留空格。

## 常见问题

**Q：LinkVault 能否直接导入现有浏览器书签中的大量链接？**

A：可以。系统内置了书签导入功能，支持 Chrome 和 Firefox 导出的 HTML 书签文件。导入时系统会自动尝试解析书签文件夹结构并映射为分类层级。对于无法识别的条目，会统一归入「未分类」区域，用户可在导入后手动整理。建议单次导入链接数量不超过 5000 条，以保证处理速度。

**Q：链接健康检查功能是否会对外部站点造成压力？**

A：健康检查默认采用异步并发请求，并发数限制为 10，且每个链接的检查间隔最短为 24 小时。检查仅发送不带请求体的 HEAD 请求，超时时间设置为 5 秒，不抓取页面内容。该设计旨在最小化对目标服务器的负担，同时避免被误判为攻击流量。用户也可在配置中调整检查频率或完全关闭该功能。

**Q：LinkVault 支持多用户权限隔离吗？**

A：当前版本支持基于角色的访问控制，内置管理员、编辑者、只读访客三种角色。管理员可管理所有分类和链接，编辑者仅能在被授权的分类下增删链接，只读访客只能查看和搜索链接但不能修改。权限粒度以分类为单位，未来版本将增加基于标签的细粒度权限策略。
