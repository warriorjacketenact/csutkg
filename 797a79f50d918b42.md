# LinkVault Core

LinkVault Core 是一个面向技术社区和内容聚合场景的高性能外链资源管理与导航系统。本项目定位于为开发者、技术内容创作者和社区运营者提供一套标准化的外部资源收录、分类、展示与监控方案，解决海量分散链接难以统一管理、缺乏结构化元数据、无法有效追踪可用性与变更的问题。LinkVault Core 不依赖特定前端框架，核心引擎基于 TypeScript 开发，可嵌入现有静态站点生成器、CMS 或云函数环境，适用于从个人书签库到企业级知识库的多种规模部署。

## 功能概览

批量资源收录与结构化存储：支持通过 JSON、YAML 或 CSV 格式批量导入外部链接，自动解析 URL 成分，提取域名、路径层级、查询参数，并生成资源唯一标识。

链接状态健康检查：内置基于 Node.js 的 HTTP 探测模块，支持配置超时、重试次数和状态码校验规则，定期输出可用性报告，标记失效或重定向资源。

自定义标签与分类体系：允许用户创建扁平或树状分类，为每个资源分配多个标签，支持基于标签组合的快速筛选与视图保存。

动态资源导航页生成：提供一套 Handlebars 与 EJS 模板引擎适配器，可将结构化数据渲染为静态 HTML 导航页面，支持响应式布局与暗色主题。

访问统计与点击热力追踪：集成轻量级事件跟踪接口，支持记录资源点击次数、来源页面与时间戳，输出基础统计分析数据。

数据导入导出兼容性：支持与主流书签服务（如 Raindrop.io、Pinboard）的 HTML 导出格式互转，降低迁移成本。

命令行工具与 RESTful API：提供完整的 CLI 操作界面用于资源增删改查，同时暴露 OpenAPI 规范的 HTTP 接口，便于与其他内部系统集成。

## 应用场景

技术团队内部知识库导航：开发团队可使用 LinkVault Core 维护常用开发文档、API 参考、内部工具面板和监控系统入口，通过标签体系按项目或技术栈分类，快速定位关键资源。

开源项目文档站扩展：开源项目维护者可将 LinkVault Core 集成至项目文档站点，作为「社区生态」或「相关资源」页面，集中展示依赖库、插件、教程和案例链接，提升文档的信息密度。

个人技术博客资源聚合：技术博主可将本系统作为博客后端的数据管理工具，定期整理推荐阅读文章、工具列表和视频教程，生成独立的资源推荐页面，丰富博客内容形态。

企业合规资源归档：企业合规或法务团队可利用 LinkVault Core 的批量导入与健康检查功能，定期归档监管政策页面、行业标准文档和第三方审计报告链接，确保关键外部引用始终可访问。

## 快速开始

以下命令演示如何在本地环境中获取源码、安装依赖并启动开发服务。

```bash
git clone https://github.com/linkvault/linkvault-core.git
cd linkvault-core
npm install
npm run dev
```

执行上述操作后，开发服务器默认监听 3000 端口，可通过 http://localhost:3000 访问管理界面。数据存储默认使用 SQLite，生产环境可切换至 PostgreSQL。

## 项目结构

项目采用模块化分层架构，核心代码与配置、文档、测试分离，便于维护和扩展。

```
linkvault-core/
├── src/                                # 核心源代码目录
│   ├── core/                           # 核心数据模型与业务逻辑
│   │   ├── resource.ts                 # 资源实体类，包含校验与序列化方法
│   │   ├── tag.ts                      # 标签树结构及合并算法
│   │   └── collection.ts               # 资源集合的聚合操作与查询构建器
│   ├── http/                           # HTTP 服务层
│   │   ├── server.ts                   # Express 应用入口，中间件与路由挂载
│   │   ├── routes/                     # RESTful 路由定义，按资源类型拆分
│   │   └── middleware/                 # 鉴权、日志、限流等通用中间件
│   ├── cli/                            # 命令行工具实现
│   │   ├── commands/                   # 每个子命令独立文件，如 add, list, check
│   │   └── bootstrap.ts                # CLI 入口解析与命令注册
│   ├── adapters/                       # 外部服务适配器
│   │   ├── storage/                    # 数据库适配器（SQLite/PostgreSQL）
│   │   ├── cache/                      # Redis 缓存适配器接口与实现
│   │   └── notifier/                   # 健康检查告警通知适配器
│   └── utils/                          # 通用工具函数
│       ├── url-parser.ts               # URL 解析与规范化工具
│       ├── retry.ts                    # 指数退避重试装饰器
│       └── logger.ts                   # 结构化日志封装（基于 pino）
├── config/                             # 环境配置文件目录
│   ├── default.yaml                    # 默认配置项（端口、数据库连接池大小）
│   ├── development.yaml                # 开发环境覆盖配置
│   └── production.yaml                 # 生产环境覆盖配置
├── tests/                              # 单元测试与集成测试
│   ├── unit/                           # 核心模块单元测试（Jest）
│   └── integration/                    # API 端到端测试（Supertest）
├── docs/                               # 文档源码
│   ├── getting-started.md
│   ├── core-concepts.md
│   ├── api-reference.md
│   └── deployment.md
├── scripts/                            # 辅助脚本
│   ├── seed-db.ts                      # 初始化测试数据
│   └── migrate-db.ts                   # 数据库迁移管理
├── templates/                          # 导航页模板文件
│   ├── index.hbs                       # 资源总览页模板
│   └── detail.hbs                      # 单个资源详情页模板
├── .env.example                        # 环境变量示例文件
├── package.json                        # npm 依赖清单与脚本定义
├── tsconfig.json                       # TypeScript 编译配置
├── Dockerfile                          # 生产环境容器镜像构建文件
```

## 贡献指南

本项目的成长依赖社区贡献。欢迎提交问题报告、功能请求和代码变更。请遵循以下流程以确保协作顺畅。

提交 Issue 前先查阅现有问题列表，避免重复报告。对于缺陷报告，请提供可复现的最小示例、环境信息和预期行为；对于功能请求，请清晰描述使用场景和收益。

Fork 仓库并创建功能分支，分支命名采用 feat/、fix/ 或 docs/ 前缀加简短描述，如 feat/add-batch-import。确保代码通过所有单元测试，并为新增功能补充对应测试用例。

提交代码前运行 ESLint 和 Prettier 进行代码风格检查与自动格式化，配置已在项目中预设。提交信息遵循 Conventional Commits 规范，使用 feat:、fix:、docs:、chore: 等类型前缀。

创建 Pull Request 时填写提供的模板，说明变更动机、实现方式、测试覆盖范围以及破坏性变更说明。至少需要一位维护者批准后方可合并。

## 常见问题

Q: 导入大量链接时页面出现超时或内存错误，如何解决？

A: 对于超过 5000 条记录的批量导入，建议使用 CLI 命令而非 Web 界面执行，CLI 模式不受 HTTP 超时限制且内存占用更优。同时可在配置文件中调整 batchSize 参数，控制单次事务提交的记录数量，默认值为 1000。若使用 PostgreSQL，建议预先创建索引以加速写入。

Q: 健康检查功能会频繁访问外部资源，是否会对目标站点造成压力？

A: 健康检查模块默认采用增量探测策略，仅对标记为活跃且近 24 小时内未检查的资源发起请求。并发请求数可通过 config 中的 maxConcurrentChecks 限制，默认值为 5。对于高频访问的资源，可手动将其加入排除列表或调整检查间隔至 72 小时以上。

Q: 如何迁移现有书签数据到 LinkVault Core？

A: 项目内置了迁移助手，支持 Netscape HTML 书签导出格式（常见于浏览器和在线书签服务）。将导出文件放置于项目根目录的 data/import 文件夹，执行 npx linkvault migrate --from netscape --input bookmarks.html 即可完成转换与入库。迁移后可通过标签继承保留原文件夹层级结构。
